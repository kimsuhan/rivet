Rivet 프로젝트 문서의 위치와 탐색 기준을 정리한다. 기획·기술 설계·개발 실행 문서는 선행 문서에서 확정한 판단을 입력으로 사용하고, 개발 지침은 변경 범위에 맞는 문서만 선택해 읽는다.

## 읽는 순서

1. `001`에서 제품 원칙, 핵심 사용자 루프와 기능 우선순위를 확인한다.
2. `002`에서 Rivet MVP의 확정 범위와 구현 인수 조건을 확인한다.
3. `003`에서 핵심 사용자 흐름, 화면 구조, 상태·오류 처리와 모바일 범위를 확인한다.
4. `004`에서 화면별 정보 배치, 반응형 구조와 공통 UI 동작을 확인한다.
5. `005`에서 제품 UI의 디자인 토큰, 컴포넌트 상태와 구현·검수 기준을 확인한다.
6. 기술 설계 `001`에서 MVP의 시스템 구조, 팀 책임 경계와 후속 기술 명세의 기준을 확인한다.
7. 기술 설계 `002`에서 도메인 엔터티, 데이터 관계, 무결성 제약과 보존 정책을 확인한다.
8. 기술 설계 `003`에서 프론트엔드와 백엔드가 공유할 REST API, 오류, 파일과 실시간 연결 계약을 확인한다.
9. 기술 설계 `004`에서 계정 인증, 세션·토큰, CSRF, 역할과 워크스페이스 데이터 격리 기준을 확인한다.
10. 기술 설계 `005`에서 실시간 신호, 알림·이메일 수신자, Outbox 재시도와 예약·정기 작업 기준을 확인한다.
11. 기술 설계 `006`에서 계층별 테스트, 자체 서버 배포, 마이그레이션, 프로세스 운영, 관측과 백업·복구 기준을 확인한다.
12. 개발 실행 `001`에서 확정 요구사항을 실제 작업, 백엔드 전달 게이트, 수직 마일스톤과 MVP 출시 판정으로 전환한다.

Linear에서 가져온 시각적 판단의 근거가 필요할 때만 참고 자료 `001`을 추가로 확인한다.

개발 지침은 위 순서에 이어서 전체를 읽지 않는다. 코드 작업을 시작할 때 [개발 지침 색인](./development/index.md)에서 현재 변경 범위에 지정된 문서만 선택한다.

## 파일 규칙

- 폴더명은 문서 단계 또는 역할을 나타내는 영문 소문자로 작성한다.
- 단계 안의 문서는 의존 순서에 따라 `001. 문서명.md` 형식으로 번호를 붙인다.
- 순서가 아니라 코드 변경 범위로 선택하는 `development/` 문서는 `index.md`, `common.md`, 앱·패키지 역할을 나타내는 영문 파일명을 사용한다.
- 문서 순서가 바뀌면 이 색인과 문서 사이의 링크를 함께 수정한다.
- 문서의 순서와 역할은 이 색인에서만 관리하며, 개별 문서에는 후속 문서 목록이나 넘길 계약 섹션을 두지 않는다.
- 검토 과정에서 확정한 값은 해당 본문에 직접 반영하고 `Q1`, `Q2` 같은 결정 요약표를 별도 섹션으로 반복하지 않는다.
