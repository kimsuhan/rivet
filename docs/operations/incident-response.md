# 장애 대응

> 상태: 현행
> 적용 범위: 운영 서비스, 데이터베이스, 비동기 작업, 파일과 외부 공급자 장애
> 정본 책임: 공통 대응 순서와 장애 유형별 안전한 복구 경계
> 갱신 조건: 관측 수단, 실패 분류, 복구 절차 또는 운영 책임 변경

## 공통 순서

1. 사용자 영향, 시작 시각, 환경과 `RELEASE_ID`를 확인한다.
2. 추가 손상 가능성이 있으면 새 쓰기 또는 관련 Worker 처리기를 중단한다.
3. `requestId`, `jobId`, 정제 오류 코드와 내부 리소스 ID로 로그 범위를 좁힌다.
4. 최근 배포, migration, 비밀 교체, 용량과 외부 공급자 상태를 확인한다.
5. 애플리케이션 롤백, 전진 수정, 선택 재시도 또는 백업 복원 중 가장 안전한 방법을 선택한다.
6. 복구 뒤 health, 핵심 smoke test, Outbox와 데이터·파일 정합성을 확인한다.
7. 원인, 영향, 임시 조치, 복구 시각과 재발 방지 작업을 기록한다.

비밀 값, 토큰, 사용자 업무 본문과 공급자 응답 원문을 장애 기록에 복사하지 않는다.

## API·Web 장애

- Nginx 5xx, PM2 상태, 반복 재시작과 `/health/live`, `/health/ready`를 확인한다.
- readiness 실패 원인이 DB, listener, 파일 경로 또는 환경 변수인지 구분한다.
- Web만 실패하면 API를 통한 데이터 쓰기가 안전한지 확인하고 사용자 진입을 제한한다.
- 지속 5xx나 권한·격리 오류가 있으면 새 쓰기를 중단한다.
- 되돌리기는 [릴리스와 되돌리기](./release-and-rollback.md)의 호환 조건을 먼저 확인한다.

## Outbox 영구 실패

1. 이벤트 ID, 유형, 시도 횟수와 정제 오류 코드를 확인한다.
2. payload를 로그에 복사하지 않고 참조 리소스의 현재 상태를 권한 있는 방식으로 조회한다.
3. 설정, 코드, 공급자 또는 데이터 오류 중 원인을 수정한다.
4. 외부 부작용이 이미 발생했는지 event ID와 전달 기록을 확인한다.
5. 승인 뒤 선택한 이벤트만 재시도한다.
6. 성공 또는 재실패와 판단 근거를 기록한다.

이벤트를 삭제하거나 payload를 직접 수정해 해결하지 않는다.

## Resend 장애

- API 키, 발신 주소, 도메인 검증과 공급자 상태를 확인한다.
- 일시 오류는 Outbox 정책에 따라 재시도한다.
- 설정·권한 같은 영구 오류는 반복 재시도하지 않는다.
- 원인 수정 뒤 토큰이 아직 유효한 이벤트만 재시도한다.
- 토큰이 만료됐으면 이벤트를 고치지 않고 사용자에게 새 링크 발급 흐름을 제공한다.

## Web Push 장애

- 인앱 알림이 정상 생성됐는지 먼저 확인한다.
- 세션과 구독 상태, endpoint 만료, VAPID 설정과 공급자 오류 분류를 확인한다.
- 만료된 endpoint는 비활성화하고 키를 제거한다.
- Push 장애 때문에 인앱 알림을 재생성하거나 읽음 상태를 바꾸지 않는다.

## PostgreSQL listener와 SSE 장애

- API의 listener 연결 로그와 PostgreSQL 연결 상태를 확인한다.
- listener가 준비되지 않으면 신규 SSE를 `503`으로 거부하고 가능한 REST API는 유지한다.
- Nginx 버퍼링·압축·timeout과 heartbeat를 함께 확인한다.
- 연결 복구 뒤 브라우저가 재연결하고 REST 재조회로 최신 상태에 수렴하는지 확인한다.
- 과거 SSE 이벤트 재생으로 복구하지 않는다.

## 파일 저장소 장애

- 디스크 용량, 마운트, API·Worker 권한과 양쪽 `FILE_STORAGE_ROOT` 일치를 확인한다.
- 경로 이탈 징후가 있으면 업로드와 파일 정리를 즉시 중단한다.
- 메타데이터만 있거나 바이너리만 있는 상태를 자동 정리하지 않는다.
- 백업 복구 가능성을 확인하고 화면에는 파일을 불러올 수 없음 상태를 유지한다.
- 복구 전 고아 파일 삭제를 실행하지 않는다.

## 보존·정리 장애

- advisory lock, 최근 성공 시각, 실패 단계와 job ID를 확인한다.
- 한 단계 실패가 다른 독립 단계의 성공을 무효화하지 않았는지 확인한다.
- 참조 중인 파일, 활성 세션 또는 아직 보존 기간이 남은 행을 수동 삭제하지 않는다.
- 원인 수정 뒤 같은 작은 batch 기준으로 재실행하고 삭제 건수를 검증한다.

## 종료 조건

- 사용자 영향이 멈추고 핵심 흐름이 복구됐다.
- Outbox 적체와 반복 오류가 정상 범위로 돌아왔다.
- 데이터베이스와 파일 저장소 정합성을 확인했다.
- 임시 쓰기 차단과 우회 설정을 제거했다.
- 남은 위험, 모니터링 항목과 후속 작업의 책임자를 기록했다.
