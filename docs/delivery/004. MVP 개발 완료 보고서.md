## 1. 문서 목적

이 문서는 [MVP 개발 실행 계획서](./001.%20MVP%20개발%20실행%20계획서.md)의 작업 결과를 현재 소스와 검증 근거로 닫는다. 계획서와 결정 기록을 다시 설명하지 않고, 실제 구현된 사용자 결과, 검증 결과, 의도적으로 남긴 비목표와 Alpha 전환 시 보존할 계약을 기록한다.

MVP 기능 개발은 종료한다. 이후 새 기능은 MVP 문서에 소급해 넣지 않고 Alpha 제품 범위와 실행 계획에서 관리한다.

## 3. 최종 제품 계약

### 3.1 업무 원본과 실행 단위

- 이슈가 제목, 설명, 첨부파일, 댓글, 우선순위, 라벨과 프로젝트의 유일한 콘텐츠 원본이다.
- 팀 작업은 이슈에 속한 팀·역할·상태·담당자와 작업 노트의 실행 레코드다.
- 내 작업은 현재 사용자에게 할당된 미완료 팀 작업의 개인 실행 큐다.
- 이슈 상세와 내 작업 상세는 같은 통합 상세를 사용하고 진입 경로와 복귀 문맥만 다르다.
- 작업 전달은 append-only이며 최초·추가 전달의 원문과 수신 작업을 보존한다.

### 3.2 일관성과 권한

- 모든 업무 조회와 변경은 workspace 경계를 강제한다.
- 오래된 `version` 변경은 최신 값을 조용히 덮어쓰지 않는다.
- 생성·작업 시작·claim·배정·완료·전달은 관련 레코드와 Outbox를 같은 트랜잭션으로 처리한다.
- 목록 정렬은 불변 ID 동률 해소 값을 사용하고 cursor는 리소스·workspace·필터·정렬에 묶인다.
- 서버가 최종 데이터 원천이며 낙관적 UI 실패는 이전 값과 입력을 복구한다.

### 3.3 알림과 이메일

- 인앱 알림이 수신자, 읽음 상태, 이동 대상과 중복 제거의 제품 정본이다.
- 행위자 자신, 비활성 멤버와 권한 없는 대상은 알림 수신자에서 제외한다.
- 이메일은 회원가입·초대·이메일 인증·비밀번호 재설정 같은 계정 흐름에만 사용한다.
- 업무 이메일 알림은 Alpha, Beta와 Public까지 제공하지 않는다.
- 운영 발신 도메인의 회원가입·이메일 인증과 비밀번호 초기화 송수신은 2026-07-15 사용자 확인으로 완료됐다.

## 4. 현재 검증 결과

2026-07-15에 Node.js 24 bundled runtime과 같은 HEAD `b45ba9a`의 A0 검증 작업 트리를 사용했다. 사용자 관리 비밀 값은 읽거나 기록하지 않았고 테스트 DB는 루트 `.env.test.local`의 허용된 로컬 `rivet.public`만 사용했다.

| 검증 | 결과 | 근거 |
| --- | --- | --- |
| `pnpm lint` | 통과 | 7개 workspace lint 성공 |
| `pnpm typecheck` | 통과 | Prisma Client 생성 후 7개 workspace typecheck 성공 |
| `pnpm test` | 통과 | scripts 3, event-contracts 25, api-client 7, database 12, worker 98, API 295, web 231개 통과 |
| `pnpm test:integration` | 통과 | 사용자 Rivet dev worker 실행 상태에서 database 13, worker 24, API 28개 통과, 15개 migration 적용 상태 확인 |
| `pnpm api:contract:check` | 통과 | API build, OpenAPI 생성과 Orval 생성 결과 무차이 |
| `pnpm build` | 통과 | 격리 `RELEASE_ID`와 공개 API origin을 주입해 web·api·worker production build 성공 |
| `pnpm test:e2e` | 통과 | 격리 포트·실제 Chromium에서 desktop·compact·mobile 20개 통과, viewport 조건에 따른 4개 의도 스킵, 실패 0개 |
| 빈 DB migration 전체 재현 | 통과 | 명시적 안전 동의 후 로컬 `rivet.public`만 reset, 빈 스키마부터 15개 migration 순차 적용, schema diff 없음, migration 상태 최신 |

### 4.1 E2E 보완 내용

M9 E2E의 팀 작업 담당자 locator를 현재 `combobox` 역할과 `팀 작업 담당자 (...): 담당자 없음` 접근 가능한 이름에 맞췄다. 목록 행 진입도 중복 텍스트가 아니라 실제 상세 링크의 접근 가능한 이름을 사용한다.

웹 workspace가 `tsx`를 직접 개발 의존성으로 선언하고 Playwright CLI를 `tsx`로 실행한다. 기본 E2E 포트는 API `4300`, Web `3300`으로 개발 기본 포트와 분리했다. 따라서 루트 `pnpm test:e2e`만으로 소스 Prisma Client를 읽으며 임시 `NODE_OPTIONS`, 개인 loader 파일, 기존 프로세스 종료가 필요하지 않다.

실제 Chromium의 desktop·compact·mobile 전체 결과는 20개 통과, viewport 조건에 따른 4개 의도 스킵, 실패 0개다.

### 4.2 Migration 재현성 보완 내용

Prisma 경고 이후 사용자가 로컬 `rivet` 데이터베이스의 `public` 스키마 reset에 별도로 동의했다. 그 응답을 안전 동의 변수에 그대로 전달해 허용된 스키마만 초기화했고, 빈 스키마에서 15개 migration이 순서대로 모두 적용됐다.

후속 `prisma migrate diff --from-config-datasource --to-schema prisma --exit-code`는 차이 없음으로 종료됐고 `prisma migrate status`는 최신 상태를 확인했다. 실행 중인 기존 Worker와의 Outbox 선점 경합은 세 관련 spec에서 해소했다. Outbox claim 계약 테스트는 커밋되지 않은 트랜잭션 행을 사용하고, 계정·초대 이메일 테스트는 이벤트를 미래 시각으로 생성한 뒤 공통 헬퍼가 대상 ID만 트랜잭션에서 due·locked 상태로 원자적으로 바꾼다. 전체 행 `FOR UPDATE` 잠금은 사용하지 않는다.

사용자 Rivet dev worker PID `90001`을 종료하지 않은 상태에서 Worker 24개를 3회 연속 통과했고, 같은 상태의 루트 `pnpm test:integration`에서 database 13개, Worker 24개, API 28개가 통과했다. 검증 뒤에도 해당 dev worker가 실행 중임을 확인했다.

## 5. 운영 준비 결과

- PM2의 API → readiness → worker → web 시작 순서와 정상 종료를 로컬 격리 환경에서 검증했다.
- Nginx 문법과 프록시·SSE·업로드 예시를 격리 환경에서 검증했다.
- PostgreSQL custom dump와 파일 tar를 같은 기준 시각의 백업 쌍으로 생성하고 checksum을 확인했다.
- 빈 격리 스키마·파일 루트에 복원해 migration, DB metadata와 파일 binary 정합성을 확인했다.
- PostHog 허용 제품 이벤트·정제 예외와 Slack 운영 경고의 외부 수신을 검증했다.
- 운영 인증 메일의 회원가입·이메일 인증과 비밀번호 초기화 송수신을 확인했다.

실제 릴리스마다 DNS·TLS·운영 비밀, migration, 프로세스, smoke, 백업과 복구 절차를 배포 가이드에 따라 다시 실행한다. 과거 리허설은 새 릴리스의 운영 승인을 대신하지 않는다.

## 6. 의도적으로 완료하지 않은 범위

- CSV 가져오기, 저장된 보기, Web Push, 이슈 템플릿과 Alpha 피드백
- GitHub·Slack 제품 연동과 Issue sync
- 업무 이메일 알림과 digest
- 사이클, 고급 트리아지, 하위 이슈·일반 관계와 수동 순서
- private 팀, 게스트, 이슈 단위 공유와 사용자 정의 역할
- 다중 워크스페이스와 전환
- 프로젝트 마일스톤·타임라인·상태 업데이트와 문서
- 모바일 푸시와 네이티브 앱
- 공개 API·OAuth 앱·범용 웹훅
- Enterprise 보안, 분석 대시보드와 AI 기능
- 자동 CI/CD, 무중단 배포, 다중 인스턴스·리전과 외부 메시지 브로커

위 항목은 미완성 MVP가 아니라 확정된 후속 또는 비목표다.

## 7. 결정 기록 종료

- MVP 결정 기록 19건은 모두 `RESOLVED`다.
- 제품 범위를 막는 `OPEN`과 `DEFAULT_APPLIED` 항목은 없다.
- `MVP-DEC-019`는 별도 안전 동의와 빈 테스트 스키마 재현성 검증을 완료해 종료했다.
- 새 Alpha 판단을 MVP 결정 번호로 추가하지 않는다.
- 기존 기록은 현재 계약의 배경을 설명하는 이력으로 보존한다.

## 8. 완료 판정

MVP의 제품 기능 개발과 M0~M9 구현 범위는 완료됐고 Alpha A0 기준선 검증도 통과했다. 기획·아키텍처·API·생성 클라이언트·데이터 모델·운영 절차의 기준은 현재 구현과 일치한다.

Alpha는 이 기준선에서 새 제품 범위로 진행한다. A0 통과는 개발 기준선 동결이며 실제 Alpha 배포 완료를 뜻하지 않는다. 배포는 A6 출시 판정과 별도 운영 체크리스트를 거쳐야 한다.
