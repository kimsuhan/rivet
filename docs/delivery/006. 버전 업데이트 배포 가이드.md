# 버전 업데이트 배포 가이드

> 적용 범위: [비공개 베타 배포 및 복구 가이드](./003.%20비공개%20베타%20배포%20및%20복구%20가이드.md)로 최초 배포를 이미 마친 단일 Linux 서버에서, 같은 서버에 새 릴리스를 반영하는 반복 절차.

이 문서는 003 문서의 원칙과 안전장치를 그대로 따르되, PM2 사용자·로그 디렉터리·logrotate·Nginx 사이트 설정처럼 최초 배포에서 한 번만 하면 되는 서버 준비 단계를 생략하고 실제로 반복하는 절차만 정리한다. 준비 원칙, 되돌리기, 백업·복구, 완료 기록 기준은 003 문서가 정본이며 이 문서와 충돌하면 003을 따른다.

## 0. 시작 전 확인

- 배포를 실행하는 리눅스 계정이 최초 배포 때 PM2를 띄운 그 계정과 같은지 확인한다. `sudo pm2`를 섞어 쓰지 않는다.
- 운영 env 파일 경로(`RIVET_ENV_FILE`)와 직전 정상 릴리스 위치를 기록해 둔다.
- Nginx 설정(`server_name`, 포트, 프록시 규칙)이 바뀌지 않는 배포라면 3장을 건너뛴다.

## 1. 코드 갱신과 릴리스 게이트

```sh
cd "$RELEASE_DIR"
git fetch
git checkout "$TARGET_COMMIT"

node --version
pnpm --version
pnpm install --frozen-lockfile
pnpm db:validate
pnpm lint
pnpm typecheck
pnpm test
pnpm test:integration
pnpm api:contract:check
pnpm test:e2e:full
pnpm build
```

실패를 무시하거나 `--force`로 진행하지 않는다. `pnpm test:integration`은 루트 `.env.test.local`의 테스트 전용 `DATABASE_URL`이 필요하다([AGENTS.md](../../AGENTS.md) 검증 절 참고). `argon2` 네이티브 빌드가 `-std=gnu++20` 오류로 실패하면 GCC/g++ 10 이상을 설치하고 `CXX=g++-12 pnpm install --frozen-lockfile`로 재시도한다(003 문서 1장 참고).

빌드 산출물을 확인한다.

```sh
test -f apps/api/dist/main.js
test -f apps/worker/dist/main.js
test -f apps/web/.next/BUILD_ID
node --check ecosystem.config.cjs
```

마이그레이션 SQL의 잠금, 기존 행 처리와 직전 앱 호환성을 사람이 검토한다. 최근 PostgreSQL·파일 저장소 백업과 디스크 여유가 확인되지 않으면 여기서 중단한다.

## 2. 운영 환경변수 로드

**PM2 명령을 실행할 그 셸에서** 직접 로드한다. 다른 셸에서 로드해도 이 셸에는 반영되지 않는다.

```sh
export RIVET_ENV_FILE=/secure/path/rivet-production.env
test -r "$RIVET_ENV_FILE"
set -a
. "$RIVET_ENV_FILE"
set +a
test "$NODE_ENV" = production
test -n "$RELEASE_ID"
test -n "$DATABASE_URL"
```

값을 출력하지 않고 필수 키가 실제로 채워졌는지만 확인하려면:

```sh
env | cut -d= -f1 | grep -E '^(DATABASE_URL|FILE_STORAGE_ROOT|ONE_TIME_TOKEN_HMAC_KEY|CSRF_HMAC_KEY|RATE_LIMIT_HMAC_KEY|RESEND_API_KEY|RESEND_FROM|POSTHOG_API_KEY|SLACK_ALERT_WEBHOOK_URL|WEB_ORIGIN|RELEASE_ID)$'
```

## 3. Nginx (설정이 바뀐 배포만)

`WEB_PORT`/`API_PORT`나 프록시 규칙이 바뀌지 않았다면 이 장을 건너뛴다. 바뀌었다면 003 문서 3장의 절차와 [`deploy/nginx/rivet-proxy.conf.example`](../../deploy/nginx/rivet-proxy.conf.example)을 기준으로 운영 설정을 갱신한 뒤 문법 검사와 reload를 수행한다.

```sh
sudo nginx -t
sudo nginx -s reload
```

## 4. 마이그레이션과 PM2 순차 반영

비밀 값이나 필수 환경변수가 이번 배포에서 바뀌었는지 먼저 판단한다.

- **바뀌지 않음**: `pm2 restart --only <app> --update-env`로 충분하다.
- **바뀌었거나(추가·삭제 포함) 판단이 애매함**: `pm2 restart --update-env`는 이전에 프로세스가 들고 있던 환경변수를 완전히 교체하지 않고 병합할 수 있어, 제거한 값이 계속 남아있는 채로 뜰 수 있다. `pm2 delete <app>` 후 `pm2 start`로 프로세스를 새로 만든다.

```sh
cd "$RELEASE_DIR"
pnpm db:migrate:deploy

pm2 delete rivet-api 2>/dev/null
pm2 start ecosystem.config.cjs --only rivet-api
pm2 describe rivet-api
curl --fail --silent --show-error "$RIVET_ORIGIN/api/v1/health/ready"

pm2 delete rivet-worker 2>/dev/null
pm2 start ecosystem.config.cjs --only rivet-worker
pm2 logs rivet-worker --lines 30

pm2 delete rivet-web 2>/dev/null
pm2 start ecosystem.config.cjs --only rivet-web
curl --fail --silent --show-error --location "$RIVET_ORIGIN/" >/dev/null
```

API readiness를 확인한 뒤 워커, 그 뒤 웹 순서를 지킨다. 비호환 Outbox `schemaVersion` 변경이 있으면 워커를 API보다 먼저 배포하는 별도 호환 순서를 쓴다(003 문서 4장 참고).

## 5. 배포 후 점검

003 문서 4장 "배포 후 점검"의 항목을 그대로 수행한다.

1. `$RIVET_ORIGIN/api/v1/health/live`, `/api/v1/health/ready`
2. 점검 계정으로 로그인·로그아웃, 이슈 생성·수정, 알림, SSE, 파일 업로드
3. `pm2 logs rivet-worker`에서 새 반복 오류 부재 확인
4. Outbox 적체 확인

```sh
psql -X -v ON_ERROR_STOP=1 -c \
  "SELECT count(*) AS pending, min(created_at) AS oldest_pending FROM outbox_events WHERE processed_at IS NULL AND canceled_at IS NULL;"
```

모두 통과하면:

```sh
pm2 save
```

readiness 실패, 반복 재시작, 지속 5xx, Outbox 지연 증가나 민감 로그가 하나라도 있으면 성공으로 기록하지 않고 5장(되돌리기)으로 간다.

## 6. 실패 시 되돌리기

[003 문서 5장](./003.%20비공개%20베타%20배포%20및%20복구%20가이드.md#5-애플리케이션-되돌리기)의 조건과 절차를 그대로 따른다. 직전 정상 릴리스 디렉터리를 지우지 않았는지 먼저 확인한다.

## 흔한 실패 원인 체크리스트

| 증상 | 원인 | 조치 |
| --- | --- | --- |
| `pnpm install` 중 `argon2` 빌드 실패, `-std=gnu++20` 오류 | g++가 C++20 미지원 버전 | g++ 10+ 설치 후 `CXX=g++-12 pnpm install` |
| `pnpm test:integration`에서 `.env.test.local` 요구 에러 | 테스트 전용 DB env 파일 없음 | 루트에 테스트용 `DATABASE_URL`을 담은 `.env.test.local` 준비 |
| `pnpm build`에서 `웹 환경 변수 구성이 올바르지 않습니다` | `RIVET_ENV_FILE`을 이 셸에 로드하지 않음 | 2장대로 같은 셸에서 로드 후 재빌드 |
| PM2가 `/var/log/rivet` 폴더를 못 만듦 | 로그 디렉터리 소유자가 실제 운영 계정과 다름 | `chown`으로 실제 계정에 맞춤(최초 배포 1회) |
| `pm2 startOrRestart`에서 `Script not found: .../pnpm` | `sudo pm2`로 실행해 root PATH에 pnpm이 없음 | sudo 없이 운영 계정으로 실행 |
| 웹 로그에 `EPROTO ... wrong version number`, `Failed to proxy https://localhost:.../ko` | Next.js standalone이 nginx의 `X-Forwarded-Proto: https`를 내부 self-proxy에도 써서 자기 자신에 TLS로 접속 시도 | [`rivet-proxy.conf.example`](../../deploy/nginx/rivet-proxy.conf.example)의 `location /`처럼 웹 upstream에는 `X-Forwarded-Proto http` 고정 |
| 값을 바꿔 재배포했는데 옛 값이 계속 적용됨 | `pm2 restart --update-env`가 이전 프로세스 env와 병합됨 | `pm2 delete` 후 `pm2 start`로 재기동 |
