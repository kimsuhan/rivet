> 상태: 확정 v1.18 — A0 완료·A1 독립 재검토 PASS·A2 독립 검토 PASS·A3·A4·A5 독립 재검토 PASS

## 1. 문서 목적

이 문서는 [Alpha 제품 범위 및 검증 계획서](../planning/009.%20Alpha%20제품%20범위%20및%20검증%20계획서.md)를 사용자에게 끝까지 보이는 수직 마일스톤으로 전환한다. 기능별 백엔드와 웹을 따로 완료 처리하지 않고 데이터·권한·API·UI·복구·분석·브라우저 검증이 같은 마일스톤에서 닫히게 한다.

일정과 담당자는 이 문서에 고정하지 않는다. 마일스톤 번호는 의존 순서이며 스프린트나 배포 날짜가 아니다.

## 5. A0 MVP 기준선 동결

> 상태: 완료 — 2026-07-15, 검증 작업 트리 HEAD `b45ba9a`

### 작업

- M9 E2E의 담당자 선택기 locator를 현재 combobox 역할과 접근 가능한 이름에 맞춘다.
- E2E가 소스 Prisma Client를 읽을 때 사용하는 TypeScript 실행 계약을 저장소 명령으로 재현 가능하게 만든다.
- 사용자 실행 중인 기본 포트와 충돌하지 않도록 격리 포트 실행 방법을 검증한다.
- desktop·compact·mobile Chromium 전체를 다시 실행한다.
- 사용자 안전 동의를 받은 뒤 로컬 `rivet.public` 테스트 스키마를 초기화해 전체 migration 재적용과 schema diff를 검증한다.
- lint, typecheck, unit, integration, API contract와 build를 같은 HEAD에서 재확인한다.

### 완료 조건

- E2E의 제품 시나리오 실패가 0건이다.
- viewport 조건에 따른 skip만 남는다.
- 빈 테스트 스키마에서 전체 migration이 순서대로 적용되고 Prisma schema와 실제 스키마의 diff가 없다.
- 검증 명령에 임시 로컬 파일, 사용자 비밀과 실행 중인 사용자 프로세스 종료가 필요하지 않다.
- MVP 완료 보고서의 보완 필요 항목을 현재 증거로 닫을 수 있다.

### 완료 근거

- M9 담당자 선택기를 현재 `combobox` 역할과 접근 가능한 이름으로 찾고, 목록 진입 locator도 실제 상세 링크 계약에 맞췄다.
- 웹 workspace의 직접 `tsx` 의존성과 격리 기본 포트 API `4300`, Web `3300`을 저장소 명령 `pnpm test:e2e`에 고정했다.
- desktop·compact·mobile 실제 Chromium 결과는 20개 통과, viewport 조건에 따른 4개 의도 스킵, 실패 0개다.
- 별도 안전 동의 후 로컬 `rivet.public`만 reset했고 빈 스키마부터 15개 migration이 순차 적용됐으며 schema diff가 없다.
- 계정·초대 이메일을 포함한 세 Worker spec의 Outbox fixture를 공통 트랜잭션 헬퍼로 격리했다. 사용자 Rivet dev worker PID `90001`이 실행 중인 상태에서 Worker 24개를 3회 연속 통과하고 루트 integration의 database 13개, Worker 24개, API 28개를 통과했으며 검증 뒤에도 dev worker가 실행 중이었다.
- lint, typecheck, unit, database·API·Worker integration, API contract와 production build를 같은 HEAD의 최종 작업 트리에서 통과했다.
- 기존 프로세스 종료, 전체 Outbox 테이블 잠금, 임시 `NODE_OPTIONS`, 개인 loader 파일, 검증 제외와 실패 은폐를 사용하지 않았다.

## 6. A1 CSV 가져오기

> 상태: 구현 완료·독립 재검토 PASS — 2026-07-15, 커밋 `674b2bf`

### 백엔드

- 가져오기 실행, source row와 정제된 결과를 저장할 모델을 설계한다.
- 원본 CSV 본문을 장기 보관하지 않는 업로드·파싱·정리 정책을 확정한다.
- CSV 형식·인코딩·행 수·파일 크기와 필수 컬럼을 검증한다.
- 팀·상태·멤버·프로젝트·우선순위·라벨 매핑 후보 API를 제공한다.
- 검증과 실제 저장 사이에 입력·대상 버전이 바뀌면 다시 검증하게 한다.
- 한 실행의 프로젝트·이슈·라벨 연결과 감사 기록을 단일 트랜잭션으로 처리한다.
- 허용 상한 10,000행은 애플리케이션 생성 ID와 500행 단위 bulk insert, 명시적 transaction 시간 예산으로 처리한다.
- source key와 실행 식별자로 중복 실행을 막는다.
- 다른 workspace ID, 비관리자와 보관·비활성 대상 연결을 거부한다.

### 웹

- 파일 선택 → 컬럼 확인 → 값 매핑 → 검증 결과 → 최종 확인 → 결과의 단계형 흐름을 제공한다.
- 오류 행, 미지원 컬럼, 제외 행과 새로 생성될 값을 저장 전에 확인한다.
- 이름 기반 미지원 컬럼은 매핑하지 못하게 하고 그 밖의 모든 미매핑 컬럼도 저장 제외 예정으로 표시한다.
- 네트워크 오류와 충돌에서 파일을 무조건 다시 선택하게 하지 않고 안전한 단계까지 복원한다.
- 완료 뒤 생성된 프로젝트·이슈로 이동하고 다시 가져오기를 명시적으로 시작할 수 있게 한다.

### 검증

- 정상 CSV, 빈 파일, 잘못된 인코딩·헤더·행, 최대 경계와 formula injection 안전성을 검증한다.
- 알 수 없는 팀·상태·멤버, 보관 대상, 중복 source key와 동시 실행을 검증한다.
- 중간 오류 뒤 생성 레코드가 남지 않는지 PostgreSQL 통합 테스트로 확인한다.
- 다른 workspace의 매핑 대상과 결과를 조회할 수 없는지 확인한다.
- desktop에서 전체 단계와 오류 위치 이동을 검증하고, 모바일은 공통 설정 정책에 따라 데스크톱 사용 안내와 업무 복귀 동작을 확인한다.

### 현재 구현·검증 근거

- `ImportRun`, `ImportSourceRow`와 `ImportRunStatus`를 추가한 `20260715000000_add_csv_imports` migration을 적용했다. 실행 식별자·파일 fingerprint·workspace source key hash의 유일성, 요청 관리자와 결과 수·정제 오류를 저장하되 CSV 원본과 업무 본문 사본은 저장하지 않는다.
- 관리자 전용 inspect·mapping-options·validate·execute·run 조회 API와 OpenAPI·생성 클라이언트를 연결했다. 검증 서명은 파일과 활성 팀·상태·멤버·프로젝트·라벨 snapshot을 묶으며 실행 직전에 다시 검사한다.
- execute는 행별 순차 쓰기를 제거하고 애플리케이션 생성 UUID, 팀별 표시 번호 사전 계산과 500행 단위 `createMany`를 사용한다. workspace·팀 잠금, source key 유일 제약과 lock 대기 30초·실행 180초의 interactive transaction 안에서 전체 성공 또는 전체 롤백한다.
- 파일 선택, 컬럼 확인, 팀·상태·멤버·프로젝트·우선순위·라벨 값 매핑, 검증 preview, 최종 원자성 확인, 생성 리소스와 실행 식별자 결과의 6단계 설정 화면을 구현했다. 이름으로 식별한 미지원 컬럼은 매핑할 수 없고 모든 미매핑 컬럼을 저장 제외 예정으로 표시한다.
- 가져온 이슈·팀 작업은 `ISSUE_IMPORTED`, 새 프로젝트는 `PROJECT_IMPORTED` 활동만 남기고 `ISSUE_CREATED`, `TEAM_WORK_CREATED` Outbox와 담당자·멘션 알림을 발행하지 않는다. 댓글·과거 활동·알림·전달 이력·첨부파일·외부 자동화 규칙은 미지원으로 명시한다.
- 파서 단위 17건과 웹 가져오기 모델·화면 6건을 포함한 전체 unit 회귀 694건이 통과했다. 빈 파일, UTF-8·UTF-16, 헤더·행 구조, 5MB·10,000행·50컬럼 경계와 formula injection을 검증한다.
- 실제 PostgreSQL A1 통합 12건을 포함한 전체 integration 77건이 통과했다. 10,000행 execute는 이슈·팀 작업·source row·활동을 각각 10,000건 원자 저장했고 전체 회귀 실행에서 약 6.0초가 걸렸다. 정상 저장, 보관·알 수 없는 매핑, 파일 중복 허용과 기존 source key 제외, 같은 실행과 다른 실행의 동시성, 대상 version 변경 재검증, 본문 참조·미지원 컬럼 거부, 일반 멤버·다른 workspace 거부를 확인했다.
- 의도적으로 이슈 식별자 충돌을 만든 저장 실패에서 프로젝트·이슈·라벨·source row와 workspace 카운터가 모두 이전 값으로 돌아가는 것을 실제 PostgreSQL에서 확인했다.
- desktop Chromium의 A1 6단계 흐름은 최신 변경에서 2.1초로 통과했다. 전체 브라우저 회귀는 사용자의 부하 중단 요청 전 15건 통과·5건 조건부 스킵까지 확인했으며 A1과 무관한 기존 M9 전달 탭 URL 회귀 1건이 실패했다. 중단된 1건과 남은 5건은 최신 작업 트리에서 완료하지 않았다.
- Prisma format·validate·generate, lint, typecheck, unit, integration, API contract와 `RELEASE_ID=local-build` production build가 통과했다. 전체 `format:check`는 A1 범위 밖의 수정되지 않은 기존 파일 25건 때문에 실패했고 그 파일을 일괄 변경하지 않았다. commit, push와 deploy는 실행하지 않았다.

## 7. A2 개인 저장된 보기

### 백엔드

## 8. A3 Web Push

> 상태: 구현 완료·독립 재검토 PASS — 2026-07-16

### 백엔드·워커

- 브라우저 Push subscription, 소유 membership, 활성 상태, 최근 성공·실패와 생성·수정 시각을 저장한다.

- 인앱 Notification 생성 뒤 같은 notification ID를 Push 전달 작업의 정본으로 사용한다.
- `(notification_id, subscription_id)` 중복 전달을 막고 영구 실패 구독을 비활성화한다.
- 테스트 발송은 membership·구독 단위 rate limit으로 공용 Outbox 폭주를 막는다.
- VAPID private key는 사용자 관리 환경에만 두고 저장소·DB·payload에 넣지 않는다.

### 웹

- 사용자의 명시적 동작에서만 브라우저 권한을 요청한다.
- 지원·기본·허용·거절·만료 상태와 해결 방법을 구분한다.
- 서비스 워커 등록과 구독 갱신이 앱 로딩과 인앱 알림을 막지 않게 한다.
- 테스트 알림과 브라우저별 해제를 제공한다.
- 알림 클릭은 기존 인앱 알림과 같은 이슈·팀 작업·댓글·전달 딥 링크를 연다.

### 검증

- 허용·거절·미지원, 다중 브라우저, 만료·영구 실패와 중복 처리를 검증한다.
- 다른 membership의 활성 endpoint 선점·구독 테스트·해제와 다른 workspace 알림 전달을 거부한다.
- 업무 본문·이메일·파일명이 Push payload와 로그에 없는지 확인한다.
- 서비스 워커 업데이트, 로그아웃과 멤버 비활성화 뒤 구독 처리 정책을 검증한다.
- 실제 지원 브라우저에서 background 수신과 알림 클릭을 확인한다.

### 현재 구현·검증 근거

- `20260716000000_add_web_push` migration으로 브라우저 구독과 전달 ledger를 추가했다. membership·workspace·session 소유권, 활성 endpoint hash 유일성, notification 또는 테스트 Outbox 중 하나만 참조하는 source 제약과 `(notification_id, subscription_id)` 멱등 키를 데이터베이스가 보장한다. `20260716010000_detach_web_push_session_on_delete`는 구독의 `session_id`를 nullable과 `ON DELETE SET NULL`로 바꿔 보존 기한이 지난 세션 삭제를 막지 않으며, 남은 구독 기록으로 만료 원인을 확인할 수 있게 한다.
- 현재 세션의 구독 등록·목록·테스트·해제 API, OpenAPI와 생성 클라이언트를 연결했다. 응답에는 endpoint, key material과 provider 오류를 포함하지 않으며 로그아웃은 현재 세션, 멤버 비활성화는 함께 폐기되는 대상 사용자의 모든 세션 구독을 비활성화하고 민감 material을 제거한다.
- 테스트 발송은 Outbox 생성 전에 membership당 10분 10회와 구독당 5분 3회 제한을 모두 적용한다. 다른 membership의 활성 endpoint 등록은 기존 소유 구독을 끄지 않고 409로 거절하며, 타 membership·workspace의 테스트와 해제는 404로 격리한다.
- worker는 먼저 생성된 인앱 `Notification`만 조회해 해당 수신자의 활성 브라우저별 구독으로 전달한다. notification ID와 저장된 앵커로 기존 이슈·팀 작업·댓글·전달 딥 링크를 만들며 별도 수신자 계산이나 읽음 변경을 하지 않는다.
- 전달 payload는 notification ID, 일반화된 사건 유형과 상대 딥 링크만 포함한다. 404·410과 만료 구독은 `EXPIRED`, 그 밖의 영구 `4xx`는 `INACTIVE`로 정리하고, `408`·`425`·`429`·`5xx`와 네트워크 오류만 Outbox 재시도 대상으로 남긴다.
- worker는 5분을 넘긴 `SENDING` lease만 회수하고 유효한 전송이 남아 있으면 Outbox 완료를 막는다. 공급자 성공과 DB 기록 실패를 분리해 성공 응답 뒤 delivery를 다시 `PENDING`으로 돌리지 않으며, 비-HTTP 프로그래밍 오류는 재시도하지 않는다. worker VAPID 미설정은 민감 정보 없는 구조화 경고로 관측한다.
- 보존 작업은 만료·폐기 세션의 활성 구독을 먼저 `EXPIRED`로 바꾸고 endpoint와 key material을 제거한 뒤 세션을 삭제한다. 잠금 경합이나 구독 정리 실패로 활성 구독이 남아 있으면 해당 세션을 삭제하지 않고 다음 보존 주기로 넘겨 `ACTIVE`이면서 `session_id`가 없는 고아 구독을 만들지 않는다. worker도 구독 선택과 delivery 선점 뒤 세션 만료·폐기 여부를 다시 확인해 정리 주기 사이의 만료 세션으로 전달하지 않는다.
- 한 delivery의 영구 실패는 같은 이벤트의 나머지 delivery 처리를 중단하지 않는다. worker는 각 delivery를 독립 처리한 뒤 재시도 오류, 처리 중 lease와 영구 오류를 이벤트 마지막에 집계한다.
- 알림함 헤더의 개인 `알림 설정`에서 지원·기본·허용·거절·만료를 구분하고 사용자 클릭에서만 권한을 요청한다. 알림함 본문에는 조치가 필요한 상태만 짧게 안내하며, 설정 안에서 브라우저별 상태·최근 성공/실패·테스트·해제와 재등록을 제공한다. 다른 멤버십의 기존 브라우저 구독을 자동 귀속하지 않으며, 사용자가 연결을 누른 등록 요청이 `409 WEB_PUSH_SUBSCRIPTION_IN_USE`이면 브라우저의 기존 구독을 해제하고 새 endpoint로 다시 구독한 뒤 복구 결과를 전용 문구로 알린다.
- 서비스 워커는 암호화 payload를 검증해 일반화된 잠금 화면 문구만 표시하고, 같은 notification tag로 중복 표시를 억제한다. 클릭은 정확한 기존 창을 포커스하고, 없으면 현재 창 이동 또는 새 창 열기로 동일 딥 링크를 연다.
- API 단위 테스트는 타 endpoint 선점·타 구독 테스트와 해제·rate limit 이전 Outbox 미생성을, worker 단위 테스트는 workspace 조회 경계·stale `SENDING` 회수·성공 뒤 DB 실패의 무재전송·설정 누락 경고를 고정한다. 서비스 워커 테스트는 OpenAPI의 전체 `NotificationType` 문구 매핑을, web과 worker 경로 테스트는 같은 할당·멘션·댓글·전달 딥 링크 사례를 고정한다.
- 실제 PostgreSQL 보존 통합 테스트는 Push 구독이 붙은 만료 세션이 삭제되고 구독의 `session_id`가 끊기며 material이 제거되는지, 활성 세션 구독은 유지되는지와 두 번째 실행의 멱등성을 함께 확인한다. 별도 트랜잭션이 활성 구독 행을 잠근 경합에서는 세션을 보존하고, 잠금 해제 뒤 다음 정리에서 구독 만료와 세션 삭제를 완료하는지도 검증한다. worker 단위 테스트는 만료 세션 무전송과 한 delivery 영구 실패 뒤 다음 delivery 성공을, web 단위 테스트는 409 뒤 기존 구독 해제·새 endpoint 등록을 고정한다.
- 2026-07-16 최종 보완 뒤 전체 회귀에서 단위 테스트 768건과 실제 PostgreSQL 통합 테스트 82건, lint·typecheck·OpenAPI 생성·Prisma schema 일관성이 통과했다. production build 기반 브라우저 E2E는 40건 통과, viewport·브라우저 조건 20건 스킵, 실패 0건이다.
- Chromium 자동화는 CDP로 합성 Push를 실제 service worker에 전달해 앱 문서가 없는 background 표시를 확인하고, 합성 `NotificationEvent`로 클릭 라우팅을 확인했다. 이 자동화는 VAPID·외부 공급자 왕복과 물리적 사용자 클릭까지 검증했다는 근거로 사용하지 않는다. 실제 구성된 Chrome의 테스트 알림 수신은 2026-07-16 수동 확인했다.
- VAPID public/private key와 subject는 E2E에서도 실행 시점에만 생성·주입했다. 운영 private key는 worker 사용자 관리 환경에만 두며 저장소·데이터베이스·payload·로그·분석·오류 수집에 저장하지 않는다. 업무 이메일, 모바일 네이티브 푸시, Slack 개인 DM과 별도 수신자 계산은 추가하지 않았다.

### 비차단 후속 개선

- `A3-FOLLOWUP-001`: worker의 Push `targetPath()`와 web의 `issueNotificationHref()`는 현재 같은 할당·멘션·댓글·전달 사례와 결과를 각각 테스트하지만 앱 경계 때문에 구현은 분리돼 있다. 새 알림 앵커나 이슈 경로 계약을 추가할 때 두 테스트가 공유하는 계약 fixture 또는 공용 경계 도입을 재검토한다. 현재 경로 일치와 A3 사용자 흐름은 검증됐으므로 A3 종결과 Alpha 출시를 막는 finding으로 보지 않는다.

### 검증

- 템플릿 수정이 과거 이슈를 바꾸지 않는지 확인한다.
- 일반 멤버의 관리 API 접근과 다른 workspace 조회를 거부한다.
- 수정·보관·복구·적용·최종 생성의 stale version 충돌이 최신 템플릿을 조용히 덮어쓰거나 부분 이슈를 남기지 않는지 확인한다.
- 활동·Outbox·분석 payload에 템플릿 설명 Markdown과 복사된 이슈 본문이 없는지 확인한다.
- 키보드만으로 템플릿 선택, 값 수정과 이슈 생성을 완료한다.

### 구현·검증 결과

- A3 구현과 독립 재검토, owner prompt 008의 PASS를 확인한 뒤 A4를 시작했다.
- Prisma migration 23개가 적용된 실제 PostgreSQL `public` 테스트 스키마와 Prisma schema의 diff가 없고 migration status가 최신이다.
- OpenAPI와 생성 클라이언트 일관성, API 구조 검사, 패키지 lint, typecheck와 production build가 통과했다.
- 전체 단위 테스트 842건과 실제 PostgreSQL 통합 테스트 89건이 통과했다.
- production API·web 기반 전체 Playwright 회귀는 51건 통과, 브라우저·viewport 조건 24건 스킵, 실패 0건이다. A4 Chromium 시나리오는 관리·덮어쓰기 확인·version 충돌·재선택·적용 후 수정·과거 이슈 불변성·보관·키보드 흐름을 검증했다.
- owner prompt 010의 독립 검토 finding인 stale 오류 직후 재제출 우회와 생성·설정 UI 읽기 순서를 수정했고, 독립 재검토가 PASS했다.
- 루트 `pnpm lint`와 `pnpm format:check`는 기존 비-A4 기준선 파일 때문에 실패했다. 패키지 전체 `turbo run lint`, A4 변경 파일 대상 ESLint·Prettier와 `git diff --check`는 통과했다.

## 10. A5 Alpha 피드백·계측

> 상태: 구현 완료·독립 재검토 PASS — 2026-07-18

### 피드백

- 피드백 본문, 카테고리, 제출 membership·workspace, 경로, release ID와 상태를 저장한다.
- 인증 사용자 제출과 관리자 목록·상태 변경 API를 제공한다.
- 본문 길이, 제어 문자, pathname-only 경계와 민감 정보 안내를 적용한다. 브라우저는 URL query를 읽지 않고 API도 query·fragment가 포함된 경로를 거부한다.
- 피드백 본문을 PostHog, 운영 로그와 Slack 경고에 넣지 않는다.
- 제출 실패 시 입력을 유지하고 중복 제출을 방지한다.

### 제품 이벤트

- 허용 이벤트와 payload 버전을 event-contracts에 정의한다.
- 초대·가져오기·첫 실행·저장 보기·템플릿·Push·검색·알림·피드백 사건을 기존 Outbox·관측 경계에서 보낸다.
- 업무 본문, 이메일, 토큰, 파일명과 Push endpoint를 payload에서 거부한다.
- 이벤트 전송 실패가 원래 업무 트랜잭션을 되돌리지 않는다.
- 반복 행동은 membership·리소스·UTC 일자 단위 capped ID로, Push 권한은 membership별 상태 전환과 version 기반 ID로 중복·조작을 방어한다.

### 검증

- 사용자 제출·관리자 조회·상태 변경과 workspace 격리를 확인한다.
- PostHog 허용 필드와 금지 필드 회귀 테스트를 추가한다.
- 이벤트 중복·재처리와 외부 공급자 실패가 제품 성공 응답을 바꾸지 않는지 확인한다.
- Alpha 관찰 지표를 실제 이벤트로 계산할 수 있는지 샘플 데이터로 대조한다.

### 구현 결과

- `product_feedback` 정본과 `20260718000000_add_product_feedback` migration을 추가했다. 인증 사용자는 전역 대화상자에서 제출하고, 관리자는 현재 workspace 목록을 필터링해 `RECEIVED`, `IN_REVIEW`, `IMPLEMENTED`, `DEFERRED` 상태를 optimistic version으로 변경한다.
- 브라우저는 대화상자를 닫거나 성공하기 전까지 본문과 `submissionId`를 유지한다. API의 `(workspace_id, submitted_by_membership_id, submission_id)` 유일 계약은 같은 내용 재시도를 기존 접수로 복구하고 다른 내용 재사용은 충돌로 거부한다.
- `event-contracts`가 payload v1 허용 목록과 정확한 속성을 검증한다. PostHog `uuid`에는 사건별 결정적 `eventId`를 사용하고 Outbox 재처리에도 같은 사건을 중복 계산하지 않는다. 브라우저는 event ID·시각을 지정할 수 없고 API가 현재 membership·workspace 리소스를 확인해 envelope을 만든다.
- 허용 사건은 초대 수락, 직접 가입 완료, 로그인, CSV 시작·검증·완료·실패, 이슈 생성, 작업 시작·완료, 저장 보기 생성·열기, 템플릿 생성·적용·기반 이슈 생성, Push 권한·테스트·전달 성공·실패·클릭, 검색 실행·무결과·선택, 알림 생성·읽음, 피드백 제출과 기존 안전 사건이다.
- 피드백 본문은 DB와 관리자 응답에만 존재한다. 이벤트 검증은 업무·댓글·전달·피드백 본문, 이메일, 토큰, 파일명, 첨부 정보와 Push endpoint를 추가 속성으로 거부하며 observability 실패는 제품 응답과 Outbox 완료를 되돌리지 않는다.
- 샘플 사건 계산기는 재처리를 제거한 뒤 membership별 온보딩 단계, CSV 검증 attempt·오류, 같은 issue ID의 cross-actor 작업 시작·완료, 알림 읽음 지연, 저장 보기 생성자, 템플릿 생성·적용·기반 이슈, Push `UNSUPPORTED`·delivery 결과·고유 성공 notification 클릭 비율과 피드백 카테고리를 대조한다. 성공률 목표와 대시보드는 만들지 않았다.
- `20260718010000_add_product_event_states` migration은 Push 권한의 membership별 최신 상태와 전환 version을 저장한다. 저장 보기 열기·템플릿 적용·검색 선택은 같은 membership·리소스·UTC 일자에 같은 ID를 사용하고 알림 클릭은 notification ID로 고정한다. 계산기는 다른 UUID로 다시 들어온 의미상 동일 사건도 제외하며 Push 권한 분포는 최신 상태만 사용한다.
- 같은 작업 트리에서 unit 885건, PostgreSQL integration 91건, production build 기반 전체 Playwright 53건이 통과했고 조건별 32건은 의도적으로 건너뛰었다. Prisma validate·generate·migration 일관성, 패키지 lint, typecheck, OpenAPI 생성·API 계약, API 구조, 환경 값을 명시한 production build와 `git diff --check`가 통과했다. API 피드백 통합 뒤 worker 보존 통합을 연속 두 번 실행해 같은 결과를 확인했다.
- 루트 `pnpm lint`는 수정되지 않은 `scripts/check-api-structure.test.mjs`의 기존 import 정렬 1건 때문에, `pnpm format:check`는 A5 범위 밖 기존 33개 파일 때문에 실패했다. A5 변경 파일 formatting과 모든 workspace lint는 통과했다.
- 첫 독립 검토의 `CHANGES_REQUIRED`에서 지적한 상관 단위, 브라우저 신뢰 경계, 최소 응답·캐시, Outbox 재처리와 스모크 envelope을 수정하고 재검토를 통과했다.
- 011 독립 재검토의 `CHANGES_REQUIRED`에서 지적한 URL query 자동 보존, 클라이언트 사건 반복 제출, PostgreSQL fixture 오염을 수정했다. 민감 query 거부·동시 피드백 유일 충돌·403/409 no-store·Push 권한 반복/상충·의미 중복 계산·E2E 삭제 순서를 회귀 테스트로 고정했고 최종 재검토를 통과했다.

## 11. 조건부 연동 결정

A5에서 다음 자료를 준비한다.

- 같은 문제가 발생한 workspace·사용자 수
- 현재 우회 방법과 반복 비용
- GitHub 또는 Slack 첫 범위로 줄어드는 조작
- 필요한 외부 권한, 비밀, 웹훅·rate limit과 운영 대응
- 데이터 보존·삭제·감사와 공급자 장애 영향
- 구현하지 않을 때 Beta에 미치는 영향

GitHub와 Slack 중 어느 것도 반복 근거가 없으면 구현하지 않고 A6로 진행한다. 하나를 선택하면 Alpha 제품 범위의 제한 계약을 지키며 읽기 표현 또는 공용 알림부터 구현한다.

## 14. 완료 판정

A0~A6의 인수 조건과 전체 출시 게이트가 통과하면 Closed Alpha 개발 완료로 판정한다. 실제 초대 팀 운영은 별도의 배포 승인과 운영 체크리스트를 실행한 뒤 시작한다.

조건부 연동을 구현하지 않았다는 사실만으로 Alpha가 미완료가 되지 않는다. 반대로 필수 기능 일부를 조건부 연동으로 대체할 수 없다.
