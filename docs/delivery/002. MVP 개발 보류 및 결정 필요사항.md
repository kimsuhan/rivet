## MVP-DEC-003 Resend 실제 계정 이메일 발송 검증

- 상태: RESOLVED
- 발견 시각: 2026-07-11 08:05:22 KST
- 마일스톤: M1 인증과 첫 워크스페이스
- 관련 작업: 이메일 인증·비밀번호 재설정·워크스페이스 초대 메일의 실제 Resend 발송과 수신 링크 확인
- 관련 요구사항: ORG-002, ORG-004, ORG-006, AUTH-02, AUTH-04, AUTH-05
- 분류: 외부 권한
- 상황과 확인 근거: 2026-07-11 21:22 KST에 사용자 관리 `.env.local`의 Resend 키·발신 주소·허용 수신 주소 두 개로 API와 Worker 환경 검증을 통과했다. 실제 애플리케이션 흐름에서 관리자 이메일 인증과 비밀번호 재설정 Outbox는 각각 한 번 처리됐고 provider message ID가 생성됐으며, Resend 대시보드에서 두 메일 모두 `delivered`를 확인했다. 사용자가 제공한 Gmail 화면에서도 인증 메일이 실제 수신된 것을 확인했다. 실제 메일의 인증·재설정 링크는 로컬 `WEB_ORIGIN`의 `#token=` fragment를 사용하고 query token이 없었다. 인증, 재설정, 이전 세션 폐기와 사용한 재설정 토큰의 재사용 거부까지 성공했다. 워크스페이스와 두 번째 주소 초대 생성도 성공했지만 시험용 `resend.dev` 발신 제한으로 초대 메일은 거절됐다. [Resend 공식 문서](https://resend.com/docs/knowledge-base/403-error-resend-dev-domain)에 따르면 이 시험용 도메인은 Resend 계정 주소 한 곳에만 보낼 수 있다. 전체 초대 가입·인증·수락 흐름은 Playwright E2E로 통과했다. 사용자는 현재 도메인을 연결하지 않고 실제 초대 메일 수신만 운영 배포 전 점검으로 이관하기로 결정했다.
- 결정이 필요한 이유: 해소됨. 개발 단계에서는 인증·재설정의 실제 외부 수신과 전체 초대 흐름의 자동 검증을 완료 기준으로 채택하고, 검증된 발신 도메인이 필요한 실제 초대 수신은 운영 배포 전 필수 점검으로 남긴다.
- 가능한 선택: 개발 전용 Resend 키와 검증된 발신 주소를 로컬 환경에 주입해 허용 수신 주소로 확인하거나, 운영 배포 준비 시 서버 비밀 설정과 함께 확인한다.
- 추천 방향: 실제 운영 도메인을 연결할 때 그 도메인의 발신 주소를 `RESEND_FROM`에 넣고, 현재 운영 가이드대로 미가입 주소의 초대 메일 수신·가입·인증·수락을 다시 확인한다.
- 임시 처리: 관리자 인증·재설정의 공급자 전달과 일회용 링크 상태 전이는 실제 외부 호출로 검증했고, 미가입 초대의 전체 제품 흐름은 API 통합·Playwright E2E로 검증했다. 비밀 값·수신 주소·토큰 원문과 메일 본문은 기록하지 않았다.
- 건너뛴 범위: 검증된 운영 발신 도메인의 실제 초대 메일 발송·수신. 운영 배포 전에는 건너뛸 수 없다.
- 영향받는 후속 작업: 실제 운영 배포 전 Resend 도메인·초대 수신 점검
- 재개 조건: 개발 완료 기준은 충족됐다. 실제 운영 도메인을 연결하면 배포 가이드의 Resend 실제 수신 점검을 다시 수행한다.
