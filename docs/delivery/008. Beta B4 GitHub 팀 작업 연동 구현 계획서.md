# Beta B4 GitHub 팀 작업 연동 구현 계획서

> 상태: **착수 보류 — B2 팀 단위 관리 권한 완료 후 진행**
>
> 제품 범위: [Beta 임시 계획서](../planning/beta-temp.md)의 `6. B4 GitHub 팀 작업 연동`

## 1. 결정 기록과 착수 게이트

2026-07-20 점검에서 B3 동적 프로젝트 참여 팀은 완료됐지만, B4가 전제하는 B2 팀장 관계와 팀 범위 관리 권한을 구현하지 않은 채 B4 계획으로 넘어간 사실을 확인했다. 다음 구현은 B2로 돌아가며, B4 코드는 아래 게이트를 모두 통과하기 전에는 시작하지 않는다.

1. 한 팀에 한 명 이상의 팀장을 지정할 수 있고 관리자만 지정·해제할 수 있다.
2. 팀장이 맡은 팀의 정보, 멤버, 초대와 워크플로 상태를 관리할 수 있다.
3. 팀장이 다른 팀과 워크스페이스 전체 설정을 변경하지 못하도록 API와 화면 권한이 격리돼 있다.
4. 팀장 지정, 팀 범위 관리, 교차 팀 접근 차단의 해피패스와 권한 엣지 케이스 테스트가 통과한다.
5. B2에서 바뀐 제품·권한·API·데이터 계약과 생성 클라이언트가 현행 문서 및 코드에 반영돼 있다.

B2를 임시 관리자 전용 기능으로 대체해 B4를 먼저 구현하지 않는다. B4의 저장소 허용 목록과 PR 상태 매핑은 팀장이 실제로 관리할 수 있어야 한다.

## 2. 목표와 범위

GitHub Pull Request를 실제 팀의 TeamWork에 연결하고, PR의 작성·리뷰·검사·병합 상태를 팀 워크플로에 반영한다. 한 TeamWork에 여러 PR을 연결할 수 있으며 모든 연결 PR이 병합된 뒤에만 예약된 완료 또는 전달을 실행한다.

- 상위 이슈 상태를 GitHub가 직접 변경하지 않는다. 기존 TeamWork 상태와 진행률 집계 계약을 따른다.
- GitHub 사용 여부나 완료 방식을 팀 이름과 종류에서 추론하지 않는다.
- GitHub는 외부 상태의 원본이고 Rivet은 연결, 최소 스냅샷, 자동화 설정과 처리 이력만 보관한다.
- GitHub Issues, 댓글, diff, 소스 코드, PR 쓰기와 병합 기능은 B4 범위에서 제외한다.

## 3. 선행 기술 검증

### B4-0. GitHub App 계약 스파이크

실제 GitHub App과 테스트 조직·저장소를 사용해 다음 계약을 먼저 검증한다.

- `Any account` 설치와 선택 저장소 접근
- `Metadata`, `Pull requests`, `Checks`, `Commit statuses` 읽기 권한만으로 필요한 상태를 조회할 수 있는지
- 설치 완료 callback에서 installation을 현재 Rivet workspace에 안전하게 귀속할 수 있는지
- `pull_request`, `pull_request_review`, `check_suite`, `status`, `installation`, `installation_repositories` payload로 필요한 식별자를 얻을 수 있는지
- 필수 리뷰, 필수 검사와 mergeable 상태를 조합해 `병합 대기`를 판정할 수 있는지
- App 제거, 저장소 접근 해제, rate limit과 일시적인 GitHub 장애를 재현할 수 있는지

권한이 부족하다는 사실을 실제 App으로 확인하기 전에는 `Administration` 등 추가 권한을 요청하지 않는다. 권한 추가가 필요하면 요청 이유와 노출 범위를 제품·보안 계약에 기록하고 다시 검토한다.

## 4. 아키텍처와 데이터 흐름

```text
GitHub Webhook
  -> API 원본 본문 서명 검증
  -> GitHubWebhookDelivery Inbox 멱등 접수
  -> Worker 비동기 처리
  -> GitHub API 최신 상태 재조회
  -> PR 스냅샷 갱신 및 TeamWork 상태 집계
  -> TeamWork 전환 / 활동 / Outbox / SSE
  -> Web 상세 및 설정 화면 갱신
```

Webhook payload는 처리의 계기일 뿐 최종 상태의 원본으로 사용하지 않는다. Worker가 installation access token을 발급받아 연결된 PR의 최신 상태를 다시 조회한 뒤 결정한다.

기존 `OutboxEvent`는 Rivet 트랜잭션에서 외부 비동기 처리를 내보내는 용도다. 외부에서 들어오는 GitHub delivery의 Inbox로 재사용하지 않고 별도 모델과 처리 루프를 둔다.

## 5. 데이터 모델 계획

구체적인 이름은 구현 시 기존 Prisma 관례에 맞추되 다음 책임을 분리한다.

- `GitHubInstallation`: workspace에 귀속된 App installation, account와 접근 상태
- `GitHubRepository`: installation에서 접근 가능한 저장소의 최소 식별 정보
- `GitHubTeamRepository`: 한 팀에 허용된 저장소 관계
- `GitHubWorkflowMapping`: GitHub 단계와 같은 팀의 워크플로 상태 매핑
- `GitHubPullRequest`: 저장소·번호별 최소 PR 스냅샷과 마지막 동기화 상태
- `GitHubTeamWorkPullRequest`: TeamWork와 PR의 다대다 연결 및 연결 방식
- `GitHubWebhookDelivery`: delivery ID, 이벤트, 처리 상태, 재시도와 오류 분류
- `GitHubTeamWorkAutomation`: 자동화 활성·일시 중지 상태와 중지 이유
- `GitHubMergeReservation`: 모든 PR 병합 뒤 실행할 완료 또는 전달 예약

필수 제약은 다음과 같다.

- 한 installation은 하나의 Rivet workspace에만 귀속한다.
- workspace는 여러 installation을 연결할 수 있다.
- GitHub delivery ID는 전역 또는 App 범위에서 유일하게 저장해 중복 처리를 막는다.
- 모든 repository, team, TeamWork 관계는 같은 workspace인지 DB와 서비스 양쪽에서 검증한다.
- PR 본문, 댓글, diff와 소스 코드는 저장하지 않는다.
- App 접근이 끊겨도 기존 연결, 활동과 예약 입력은 삭제하지 않는다.

## 6. 설치·설정 API 계획

### 설치와 연결 해제

- 관리자가 설치 시작 URL을 요청하면 workspace와 요청자를 연결한 짧은 수명의 서명된 state를 생성한다.
- 설치 callback에서 일회성 사용자 인증으로 installation 접근 권한과 계정을 확인하고 현재 workspace에 귀속한다.
- 검증용 사용자 token은 저장하지 않고 즉시 폐기한다.
- Rivet의 `연결 해제`는 로컬 동기화를 중지하고 이력을 보존한다. GitHub App 자체 제거는 GitHub 설정으로 안내한다.
- `installation`과 `installation_repositories` 이벤트로 App 제거와 저장소 접근 변경을 반영한다.

### 권한

- 워크스페이스 관리자: App 설치·연결 해제, 모든 팀의 저장소·매핑 관리
- 해당 팀의 B2 팀장: 맡은 팀의 허용 저장소와 상태 매핑 관리
- 일반 멤버: 연결 상태와 자신의 접근 가능한 TeamWork PR 정보 조회, 허용된 수동 연결과 자동화 제어
- 다른 workspace 또는 다른 팀 설정에 대한 조회·변경은 거부한다.

OpenAPI, 생성 API client, query key와 캐시 무효화 계약을 같은 변경에서 갱신한다.

## 7. Webhook Inbox와 Worker 계획

### 수신 API

- 파싱 전 원본 요청 본문으로 `X-Hub-Signature-256` HMAC-SHA256을 timing-safe 방식으로 검증한다.
- `X-GitHub-Delivery`를 멱등 키로 저장한다.
- 지원 이벤트의 최소 식별 정보만 Inbox에 기록하고 빠르게 `2xx`를 반환한다.
- 서명 실패, 알 수 없는 installation과 잘못된 payload를 구분해 관측 가능한 오류로 남긴다.

### Worker

- 전용 Inbox poller가 `FOR UPDATE SKIP LOCKED` 방식으로 delivery를 선점한다.
- installation access token을 필요할 때만 발급하고 장기 저장하지 않는다.
- 같은 PR과 연결 TeamWork의 최신 상태를 다시 조회해 순서가 뒤바뀐 delivery도 최종 상태로 수렴시킨다.
- 일시 오류, rate limit, 영구 접근 해제와 잘못된 계약을 구분해 재시도 여부를 결정한다.
- 사용자 `다시 동기화`와 열린 PR 주기 대조를 제공하며, 주기 작업은 advisory lock으로 단일 실행을 보장한다.

## 8. PR 연결과 상태 집계

### 연결

- 브랜치명 또는 PR 제목에서 workspace 내 TeamWork 표시 ID를 정확히 파싱한다.
- PR 저장소가 해당 TeamWork 팀에 허용돼 있고 TeamWork와 프로젝트 참여 관계가 활성일 때만 자동 연결한다.
- 수동 연결은 정규화된 GitHub PR URL을 검증하고 같은 권한·workspace 규칙을 적용한다.
- 삭제되거나 다른 workspace인 TeamWork, 허용되지 않은 저장소와 모호한 표시 ID는 자동 연결하지 않는다.

### 여러 PR의 결정적 집계 우선순위

1. App 또는 저장소 접근 해제, 최신 상태 조회 실패: 현재 TeamWork 상태를 유지하고 동기화 오류 표시
2. 병합되지 않고 닫힌 PR 존재: 자동화 차단 및 연결 확인 요청
3. 변경 요청 PR 존재: `변경 요청` 매핑
4. 실패한 필수 검사 존재: 검토 단계 유지, `병합 대기` 금지
5. Draft 또는 일반 open PR 존재: `진행 중` 매핑
6. 리뷰 대기 또는 필수 승인 미충족 PR 존재: `리뷰 대기` 매핑
7. 모든 미병합 PR이 승인·필수 검사·mergeable 조건 충족: `병합 대기` 매핑
8. 모든 연결 PR 병합: 병합 후 처리 실행 또는 완료 방식 선택 안내

GitHub의 mergeable 값이 계산 중이거나 알 수 없으면 병합 가능으로 간주하지 않는다. 사용자가 TeamWork를 일시 중지·취소·완료하거나 매핑 밖 상태로 옮기면 자동화를 중지하고, 명시적인 `GitHub 자동화 재개` 전까지 Webhook은 PR 스냅샷만 갱신한다.

## 9. 병합 후 처리 예약

- 담당자는 리뷰 대기 또는 병합 대기 단계에서 `이 작업만 완료` 또는 `다른 팀에 전달 후 완료`를 예약한다.
- 전달 예약은 대상 활성 참여 팀, 전달 Markdown과 기존 B3 첨부 계약을 보존한다.
- 모든 PR 병합 시 TeamWork version, 담당자, 프로젝트 참여 관계, 대상 팀, PR 연결과 자동화 상태를 잠금 아래 재검증한다.
- 검증을 통과하면 B3 완료·전달 계약을 재사용해 TeamWork, 전달, 대상 TeamWork, 이슈 진행률, 활동과 Outbox를 한 트랜잭션으로 반영한다.
- 예약이 없거나 충돌하면 자동 완료하지 않는다. 입력을 보존하고 사용자가 다시 확인할 수 있게 한다.
- 사용자가 먼저 TeamWork를 완료하면 예약을 종료하고 이후 GitHub 이벤트가 완료 상태를 덮어쓰지 못하게 한다.

API와 Worker에 완료·전달 로직을 복제하지 않는다. B4 구현 시 두 앱이 같은 트랜잭션 동작을 재사용할 수 있는 가장 좁은 경계를 추출한다. 새 workspace가 필요하다면 같은 변경에서 해당 경로의 `AGENTS.md`와 개발 문서 색인을 추가한다.

## 10. Web 화면 계획

### 워크스페이스·팀 설정

- GitHub App 설치 상태, 연결 계정과 저장소 접근 상태
- 관리자용 설치·연결 해제·재연결
- 관리자와 B2 팀장용 팀별 허용 저장소 및 GitHub 단계별 워크플로 매핑
- 누락된 매핑, 접근 해제와 마지막 동기화 오류 안내

### TeamWork·통합 이슈 상세

- 표시 ID와 제목을 포함한 브랜치명 복사
- 저장소, PR 번호, 제목, 작성자, 상태, GitHub 링크와 마지막 동기화 시각
- PR URL 수동 연결과 연결 해제
- 자동화 일시 중지·재개, 다시 동기화
- 병합 후 완료·전달 예약, 충돌과 입력 보존 상태
- 로딩, 빈 상태, 권한 오류, 접근 해제와 모바일 레이아웃

## 11. 구현 순서

1. **B4-0 계약 스파이크**: 실제 App 권한, 설치, Webhook과 mergeability 검증
2. **B4-1 DB 기반**: installation, repository, mapping, PR, Inbox, automation, reservation migration
3. **B4-2 설치·설정 API**: 설치 callback, 연결 상태, 팀 저장소와 상태 매핑, 권한
4. **B4-3 Webhook·Worker**: 서명 검증, 멱등 Inbox, 최신 상태 재조회, 재시도와 대조
5. **B4-4 PR 연결·상태 자동화**: 자동·수동 연결, 여러 PR 집계, 일시 중지·재개와 활동
6. **B4-5 병합 후 예약**: B3 완료·전달 트랜잭션 재사용, 충돌 보존과 원자성
7. **B4-6 Web UI·운영 검증**: 설정·상세 UX, 관측, 실제 GitHub 시나리오와 단계적 활성화

각 단계는 migration, API 계약과 생성 코드가 필요한 경우 같은 PR 안에서 일관되게 반영한다. B4-0 결과로 계약이 달라지면 구현 전에 이 계획과 Beta 제품 범위를 먼저 갱신한다.

## 12. 테스트와 검증

### 자동 검증

- 단위: TeamWork ID 파서, PR URL 정규화, 서명 검증, 여러 PR 상태 집계
- DB 통합: delivery 중복, workspace 격리, worker 선점, 예약 원자성과 충돌 보존
- API E2E: 관리자·B2 팀장·일반 멤버 권한, 다른 팀·workspace 접근 차단
- Worker: 중복·순서 역전 delivery, mergeable 미확정, rate limit, App 제거와 접근 해제
- Web: 설치·매핑·PR 카드·예약의 로딩, 빈 상태, 오류, 모바일과 접근성

예상 검증 명령은 실제 변경 범위에 맞춰 다음에서 선택한다.

```bash
pnpm db:format
pnpm db:validate
pnpm db:generate
pnpm api:contract:generate
pnpm api:contract:check
pnpm lint
pnpm typecheck
pnpm test
pnpm build
```

### 실제 GitHub 스모크 시나리오

1. 테스트 조직에 App 설치 및 선택 저장소 연결
2. 팀 저장소와 상태 매핑 설정
3. TeamWork 브랜치명으로 PR 생성 및 자동 연결 확인
4. Draft → 리뷰 요청 → 변경 요청 → 승인 전환 확인
5. 필수 검사 실패·성공과 mergeable 상태 확인
6. 여러 PR 중 일부만 병합했을 때 완료되지 않는지 확인
7. 모든 PR 병합 뒤 예약된 완료·전달의 원자적 처리 확인
8. App 제거·저장소 접근 해제·재연결과 누락 이벤트 대조 확인

## 13. 배포·관측·롤백

1. expand migration과 필요한 비밀 환경 변수 이름을 배포 환경에 준비한다.
2. API와 Worker를 GitHub 기능 비활성 상태로 먼저 배포한다.
3. Web을 배포한 뒤 지정 Beta workspace에서만 feature flag를 켠다.
4. 실제 설치·PR 스모크 테스트를 통과한 뒤 대상 workspace를 확대한다.
5. Inbox backlog, 처리 지연·실패율, 중복 delivery, GitHub rate limit, 접근 해제와 예약 충돌을 관측한다.

문제가 생기면 feature flag를 꺼서 새 수신과 자동 상태 변경을 중지한다. installation, PR 연결, 활동과 예약 데이터는 보존해 복구 후 다시 대조할 수 있게 하며, rollback이 데이터 삭제에 의존하지 않게 한다.

필요한 비밀 값은 문서에 실제 값을 기록하지 않고 API의 GitHub client ID·secret·webhook secret·App slug, Worker의 App ID·private key와 기능 활성화 설정 이름만 관리한다.

## 14. 완료 조건

- B2 착수 게이트를 먼저 통과했다.
- GitHub App을 외부 테스트 조직에 최소 권한으로 설치하고 제거·재연결할 수 있다.
- 관리자와 B2 팀장 권한이 팀·workspace 경계 안에서 동작한다.
- 자동·수동 PR 연결과 여러 PR 상태 집계가 중복·순서 역전에도 결정적으로 수렴한다.
- 수동 상태 변경을 자동화가 덮어쓰지 않으며 명시적으로 재개할 수 있다.
- 모든 PR 병합 후 B3 완료·전달 계약이 한 번만 원자적으로 실행된다.
- App 접근 해제와 GitHub 장애가 Rivet 데이터 손실이나 잘못된 자동 완료로 이어지지 않는다.
- Prisma, OpenAPI, 생성 client, 관련 문서와 검증이 같은 계약을 설명한다.
- 실제 GitHub 스모크 시나리오와 단계적 배포·롤백 절차를 확인했다.
