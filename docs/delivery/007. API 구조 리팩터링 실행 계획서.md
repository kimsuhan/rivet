# API 구조 리팩터링 실행 계획서

> 상태: 완료 v1.0
> 적용 범위: `apps/api/src/modules`, API 단위·통합 테스트와 구조 검사

## 1. 목적

이 리팩터링은 기존 HTTP·OpenAPI·데이터 계약을 바꾸지 않고, 대형 Service에 섞인 조회·잠금·변환·입력 규칙과 보조 로직의 소유권을 명확히 한다. 프론트 기능을 다시 설계하는 작업이 아니라 Beta 권한·외부 연동을 안전하게 추가할 수 있는 백엔드 변경 기반을 만드는 작업이다.

## 2. 확정 원칙

- 기능 모듈 구조를 유지하며 기술 계층별 전역 폴더로 재편하지 않는다.
- Controller는 HTTP, Service는 사용 사례와 트랜잭션, Repository는 복잡·재사용 데이터 접근, Mapper는 순수 응답 변환을 담당한다.
- 모듈 루트 파일은 `<대상>.<역할>.ts`로 이름을 지어 소유 책임을 드러낸다.
- `helper`, `utils`, `types`, `interfaces` 같은 범용 수집 폴더를 만들지 않는다.
- DTO는 `dto/`, 순수 도메인 규칙은 `domain/`에 두고 나머지 type은 소유 파일 가까이에 둔다.
- 단위 spec은 소스 옆에 유지한다. `__test__` 이동은 기존 탐색성과 저장소 테스트 계약을 약화하므로 범위에 포함하지 않는다.
- 주석은 동작을 번역하지 않고 잠금·보안·트랜잭션·외부 계약의 이유만 설명한다.
- 공통 함수는 두 호출부의 의미와 변경 이유가 같을 때만 승격한다.

## 3. 단계

| 단계 | 범위 | 완료 기준 | 상태 |
| --- | --- | --- | --- |
| R0 | 구조 계약, 자동 검사, Projects 기준 사례 | 파일 역할·금지 폴더 규칙이 문서와 자동 검사에 고정되고 Projects 조회·명령 책임이 분리된다. | 완료 |
| R1 | Auth·Files의 무역할 파일명 | 입력 정책, 암호화, 요청 문맥과 파일 콘텐츠 정책의 역할명이 명확해진다. | 완료 |
| R2 | Teams·Files·Workspaces | 목록·복잡 조회와 변환을 분리하고, 분리 조건이 없는 작은 Service는 유지하며 권한·잠금 회귀를 통과한다. | 완료 |
| R3 | Auth·Invitations | 세션·토큰·초대 사용 사례 경계와 보안 실패 계약을 유지한다. | 완료 |
| R4 | Issues·TeamWorks·Collaboration·Search | 모듈 내부 구현 참조를 공개 application service 경계로 제한하고 N+1 조회를 제거하며 이슈 트랜잭션 경계를 명시한다. | 완료 |
| R5 | CSV Imports | 파싱·검증·대량 저장·실행 조회를 분리하면서 10,000행 원자성 계약을 유지한다. | 완료 |
| R6 | 전체 회귀와 완료 감사 | 구조 검사, lint, typecheck, unit, integration, API contract와 production build가 같은 작업 트리에서 통과한다. | 완료 |

각 단계는 한 기능 모듈의 공개 provider와 테스트를 함께 변경한다. 여러 대형 Service를 동시에 옮기지 않으며, 단계마다 검증을 통과한 뒤 다음 모듈로 진행한다.

## 4. 분리 판단

다음 중 둘 이상을 한 Service가 함께 소유하면 분리 대상으로 본다.

- 목록 필터·정렬·커서 해석
- 복잡하거나 반복되는 Prisma 조회와 행 잠금
- 응답 DTO 변환과 계산 필드 조립
- 생성·수정·삭제 트랜잭션과 Outbox 발행
- 외부 형식 파싱, 암호화와 순수 입력 정책

단순 Prisma 호출 한두 줄, 한 사용 사례 안에서만 의미가 있는 작은 타입과 값 전달 함수는 별도 계층으로 만들지 않는다.

## 5. 비목표

- 엔드포인트, 응답 DTO, 오류 코드, Prisma schema와 migration 변경
- 제네릭 Repository·Mapper·BaseService 도입
- 모든 함수에 JSDoc 추가
- 모든 작은 함수를 공용 util로 승격
- 단위 spec의 `__test__` 일괄 이동
- API와 무관한 Web·Worker 구조 변경

## 6. 검증과 안전장치

- 구조 검사는 새 범용 폴더와 역할 없는 모듈 루트 파일을 차단한다.
- 모듈별 단위 테스트는 정상 흐름과 권한·workspace·version 충돌·잠금 실패를 유지한다.
- 쿼리와 트랜잭션 변경은 실제 PostgreSQL 통합 테스트로 회귀를 확인한다.
- API 계약이 달라지지 않았음을 OpenAPI 생성 결과와 `api:contract:check`로 확인한다.
- commit, push와 deploy는 별도 사용자 요청이 없으면 실행하지 않는다.

## 7. 완료 결과

- Projects, Teams, Files, Invitations는 명령 Service에서 목록·상세 조회, 응답 변환과 복잡 데이터 접근을 분리했다.
- Auth는 계정 토큰과 프로필 사용 사례를 분리하고 입력 정책·토큰·비밀번호 파일의 역할명을 명시했다.
- Issues와 TeamWorks는 조회, 할당, 상태 계산과 명령 트랜잭션을 분리했다. Search는 이슈 요약을 일괄 조회해 목록 N+1을 제거했다.
- Collaboration은 댓글, 인계, 타임라인 조회와 잠금 책임으로 분리했다.
- CSV Imports는 오케스트레이션, 분석, 매핑 정책, 대상 조회, 실행 조회와 대량 저장 책임으로 분리했다.
- 구조 검사는 역할 없는 모듈 루트 파일, 범용 수집 폴더와 `__test__` 계열 폴더를 차단한다.
- API 공개 계약과 Prisma schema·migration은 변경하지 않았다.
- 구조 검사 3개, API 단위 테스트 346개, PostgreSQL 통합 테스트 43개, lint, typecheck, OpenAPI·생성 클라이언트 일관성 검사와 production build가 통과했다.
