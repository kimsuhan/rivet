# 비공개 베타 배포 및 복구 가이드

> 상태: M7 로컬 격리 리허설·운영 인증 메일 송수신 검증 완료, 실제 릴리스마다 운영 절차 재실행 필요
> 적용 범위: 단일 Linux 서버, PM2, Nginx, 자체 PostgreSQL과 로컬 파일 저장소

이 문서는 [테스트·배포 및 운영 명세서](../architecture/006.%20테스트%C2%B7배포%20및%20운영%20명세서.md)의 수동 절차를 실행 가능한 순서로 정리한다. 서버 접속, DNS·TLS 변경, 비밀 발급과 운영 백업 실행은 저장소 밖에서 운영자가 승인해 수행한다.

## 1. 준비 원칙

- Node.js 24와 루트 `package.json`에 고정된 pnpm을 사용한다.
- 웹·API·워커는 같은 커밋, `RELEASE_ID`, lockfile과 Prisma schema에서 빌드한다.
- 비밀 값과 운영 `.env`는 저장소·릴리스 디렉터리·PM2 Ecosystem 파일에 넣지 않는다. 권한이 제한된 외부 환경 파일을 현재 셸에 주입한다.
- API와 워커는 같은 `DATABASE_URL`과 `FILE_STORAGE_ROOT`를 사용한다. `DATABASE_POOL_MAX`를 공통으로 덮어쓰지 않으면 앱별 기본값 10과 5가 적용된다.
- PM2는 Node.js 24로 설치·실행하고 PM2 사용자 홈과 dump 파일은 해당 운영 사용자만 읽을 수 있게 한다. `pm2 env`, `jlist`와 dump 파일을 작업 기록에 첨부하지 않는다.
- PM2 명령은 항상 릴리스를 실제로 실행할 운영 계정으로만 실행한다. `sudo pm2`로 한 번이라도 실행하면 root 소유의 별도 PM2 데몬이 생겨 프로세스 목록이 분리된다. 잘못 띄웠다면 `sudo pm2 delete <app>`과 `sudo pm2 kill`로 정리하고 운영 계정에서 다시 시작한다.
- 비밀 값이 바뀐 배포에서 `pm2 restart --update-env`는 이전에 프로세스가 들고 있던 환경변수를 완전히 교체하지 않고 병합할 수 있다. 값을 제거하거나 바꾼 배포에서는 `pm2 restart` 대신 `pm2 delete <app>` 뒤 `pm2 start`로 프로세스를 새로 만든다.
- 서버에 `argon2`를 빌드할 수 있는 GCC/g++ 10 이상이 없으면 `pnpm install`의 네이티브 빌드가 `-std=gnu++20` 옵션 오류로 실패한다. `g++ --version`으로 먼저 확인하고, 낮으면 `sudo apt install g++-12`(또는 배포판에서 지원하는 최신 버전) 설치 후 `CXX=g++-12 pnpm install --frozen-lockfile`로 재시도한다.
- 직전 정상 릴리스 디렉터리와 백업을 지우지 않는다. 데이터베이스 down migration은 되돌리기 수단으로 사용하지 않는다.

PM2를 시작하기 전에 앱 로그 디렉터리를 운영 사용자만 쓸 수 있게 준비한다. 실제 운영 사용자·그룹이 `rivet`이 아니면 디렉터리 소유권과 [`deploy/logrotate/rivet.example`](../../deploy/logrotate/rivet.example)의 `su` 값을 함께 바꾼다.

```sh
sudo install -d -o rivet -g rivet -m 0750 /var/log/rivet
sudo install -o root -g root -m 0644 deploy/logrotate/rivet.example /etc/logrotate.d/rivet
sudo logrotate --debug /etc/logrotate.d/rivet
```

예시는 앱의 한 줄 JSON stdout·stderr를 매일 순환하고 압축해 30개를 보존한다. Nginx 접근 로그는 `/var/log/nginx/rivet-access.log`에 한 줄 JSON으로 기록한다. 서버의 기존 Nginx logrotate가 이 파일을 포함하는지, 30일 뒤 삭제하는지와 reload 뒤에도 기록이 이어지는지를 별도로 확인한다.

환경 변수 이름과 검증 기준은 [`.env.example`](../../.env.example)을 따른다. 다음 예시는 값 자체를 출력하지 않고 외부 환경 파일을 주입한다.

```sh
export RIVET_ENV_FILE=/secure/path/rivet-production.env
test -r "$RIVET_ENV_FILE"
set -a
. "$RIVET_ENV_FILE"
set +a
test "$NODE_ENV" = production
test -n "$RELEASE_ID"
```

## 2. 릴리스 준비와 검증

새 디렉터리에서 아래 출시 게이트를 통과시킨다. 실패를 무시하거나 `--force`로 진행하지 않는다.

```sh
node --version
pnpm --version
pnpm install --frozen-lockfile
pnpm db:validate
pnpm lint
pnpm typecheck
pnpm test
pnpm test:integration
pnpm api:contract:check
pnpm test:e2e:full
pnpm build
```

빌드 뒤 PM2가 실행할 산출물과 설정을 확인한다.

```sh
test -f apps/api/dist/main.js
test -f apps/worker/dist/main.js
test -f apps/web/.next/BUILD_ID
node --check ecosystem.config.cjs
```

마이그레이션 SQL의 잠금, 기존 행 처리와 직전 앱 호환성을 사람이 검토한다. 최근 PostgreSQL·파일 저장소 백업과 디스크 여유가 확인되지 않으면 여기서 중단한다.

## 3. Nginx 준비

[`deploy/nginx/rivet-http-log.conf.example`](../../deploy/nginx/rivet-http-log.conf.example)은 Nginx `http` 블록 안에 한 번만 포함하고, [`deploy/nginx/rivet-proxy.conf.example`](../../deploy/nginx/rivet-proxy.conf.example)은 기존 HTTPS `server` 블록 안에 포함한다. 인증서, `server_name`, HTTP에서 HTTPS로의 리다이렉트와 보안 헤더 책임은 서버의 기존 설정에서 관리한다.

- 기본 내부 포트는 웹 `3000`, API `4000`이다. `WEB_PORT`나 `API_PORT`가 다르면 두 `proxy_pass` 대상도 함께 바꾼다.
- `/api/v1/events`는 버퍼·캐시·압축을 끄고 읽기 제한을 75초로 둔다.
- CSV 스트림은 일반 API와 분리해 버퍼를 끄고 300초 제한을 둔다.
- `client_max_body_size 30m`는 multipart 오버헤드를 허용하기 위한 프록시 상한이다. 실제 파일 25MB 상한은 API가 다시 강제한다.
- `FILE_STORAGE_ROOT`를 Nginx 정적 경로로 노출하지 않는다.
- API와 웹이 보내는 `X-Rivet-Environment`, `X-Rivet-Release-Id`는 Nginx 로그용 비민감 메타데이터다. Nginx는 두 헤더를 외부 응답에서 제거하고 접근 로그의 `environment`, `releaseId`에만 사용한다.
- Next.js의 `X-Powered-By`와 내부 locale rewrite 주소를 담는 `X-Middleware-Rewrite`도 외부 응답에서 제거한다.
- 접근 로그의 `path`는 Nginx `$uri`를 미리 정의한 범주형 template으로 바꾼 값이며, 알 수 없는 경로는 `UNMATCHED_ROUTE`로 남긴다. `duration`은 초 단위 `$request_time`이다. IP, raw path, query, 요청·응답 본문, Cookie와 Authorization 헤더는 형식에 넣지 않는다.
- 앱 응답을 받지 못한 Nginx 자체 오류는 `environment=production`, `releaseId=unknown`으로 남긴다. 해당 시각의 배포 기록과 PM2 상태로 대상 릴리스를 확인한다.

운영 설정에 반영하기 전과 변경 후에 각각 문법을 검사한다. `rivet-http-log.conf.example`을 그대로 `server` 블록에 넣으면 `map`과 `log_format` 문맥 오류가 나므로 실제 `http` include 경로를 먼저 확인한다. 실제 파일 경로는 서버 설정에 맞춘다.

```sh
sudo nginx -t
sudo nginx -s reload
```

설정이 바뀌지 않은 애플리케이션 배포에서는 Nginx를 reload하지 않는다.

## 4. 수동 배포

점검 시간, 대상 `RELEASE_ID`, 새 릴리스 디렉터리와 직전 정상 디렉터리를 먼저 기록한다. 비호환 Outbox `schemaVersion` 변경이 있으면 신·구 payload를 함께 읽는 워커를 API보다 먼저 배포하는 별도 호환 순서를 사용한다.

일반적인 호환 배포는 다음 순서다.

```sh
cd "$RELEASE_DIR"
pnpm db:migrate:deploy

pm2 startOrRestart "$RELEASE_DIR/ecosystem.config.cjs" --only rivet-api --update-env
pm2 describe rivet-api
curl --fail --silent --show-error "$RIVET_ORIGIN/api/v1/health/ready"

pm2 startOrRestart "$RELEASE_DIR/ecosystem.config.cjs" --only rivet-worker --update-env
pm2 describe rivet-worker

pm2 startOrRestart "$RELEASE_DIR/ecosystem.config.cjs" --only rivet-web --update-env
pm2 describe rivet-web
curl --fail --silent --show-error --location "$RIVET_ORIGIN/" >/dev/null
```

`ecosystem.config.cjs`는 선언 순서를 API, 워커, 웹으로 유지하지만 한 번에 세 앱을 시작하지 않는다. API readiness를 확인한 뒤 워커, 그 뒤 웹을 순서대로 반영한다. 처음 설치한 서버에서는 모든 확인이 끝난 뒤에만 `pm2 startup`의 운영체제별 안내를 적용하고 `pm2 save`를 실행한다.

### 배포 후 점검

공개 상태 엔드포인트를 Nginx 경유로 확인한다.

```sh
curl --fail --silent --show-error "$RIVET_ORIGIN/api/v1/health/live"
curl --fail --silent --show-error "$RIVET_ORIGIN/api/v1/health/ready"
```

운영 사용자 데이터와 분리된 점검 계정·워크스페이스로 다음을 확인한다.

1. 로그인·로그아웃과 현재 워크스페이스 목록
2. 점검용 이슈 생성·수정·휴지통 이동 또는 사전에 정한 비파괴 변경
3. 알림 생성과 읽지 않은 수
4. 브라우저의 SSE heartbeat, 강제 재연결과 REST 재조회 수렴
5. 작은 파일 업로드·조회·연결 해제
6. PM2 워커 로그의 polling·정기 작업 상태와 새 반복 오류 부재

구조화 요청 로그에서 같은 `RELEASE_ID`, query 없는 경로, 상태와 처리 시간을 확인한다. 인증된 API 요청 한 건은 내부 `workspaceId`, `membershipId`가 있어야 하고 이메일, IP, Cookie, 토큰과 업무 본문은 없어야 한다. Nginx 로그의 `duration`은 초, API Pino 로그의 `duration`은 밀리초 단위다.

쿠키나 토큰을 명령행에 붙여 SSE를 점검하지 않는다. 로그인된 브라우저의 개발자 도구로 확인한다. Outbox 상태는 비밀을 명령 인자로 넘기지 않는 `PGSERVICE` 또는 PostgreSQL 환경 설정으로 집계한다.

```sh
psql -X -v ON_ERROR_STOP=1 -c \
  "SELECT count(*) AS pending, min(created_at) AS oldest_pending FROM outbox_events WHERE processed_at IS NULL AND canceled_at IS NULL;"
psql -X -v ON_ERROR_STOP=1 -c \
  "SELECT count(*) AS failed FROM outbox_events WHERE processed_at IS NULL AND canceled_at IS NULL AND attempt_count >= 7;"
```

readiness 실패, 반복 재시작, 지속 5xx, 격리·저장·파일·listener 실패, 증가하는 Outbox 지연 또는 민감 로그가 하나라도 있으면 성공으로 기록하지 않는다. 원인을 확인할 때까지 `pm2 save`도 갱신하지 않는다.

### 운영 지표 점검 경계

- API 요청 수, 상태 비율과 지연은 구조화 API·Nginx 요청 로그에서 집계한다.
- Nginx 5xx와 413 업로드 거부도 `rivet-access.log`의 `status`로 집계한다.
- PM2 프로세스 상태, 반복 재시작 횟수와 메모리 사용량은 애플리케이션 로그 필드가 아니다. 운영 서버의 `pm2 describe`와 서버 모니터링에서 확인하고 배포 기록에는 판정과 시각만 남긴다.
- 서버·PostgreSQL·파일 저장소 디스크 사용량과 DB 연결 상태도 외부 운영 점검 대상이다. 저장소 내부 단위 테스트나 정적 설정 검사만으로 정상 판정하지 않는다.

### 외부 관측 수신 점검

먼저 운영과 분리된 PostHog 프로젝트와 Slack 시험 채널에서 실행하고, 실제 운영 대상으로 한 번 더 확인한다. 시험용 외부 환경 파일에는 시험 프로젝트의 `POSTHOG_API_KEY`와 시험 채널의 `SLACK_ALERT_WEBHOOK_URL`만 넣고 `NODE_ENV`, `RELEASE_ID`와 운영 비밀은 넣지 않는다. 전송 전에 각 공급자 콘솔에서 키·Webhook의 프로젝트와 채널 이름을 직접 확인하되 비밀 값 자체는 출력하거나 배포 기록에 남기지 않는다.

```sh
export RIVET_OBSERVABILITY_TEST_ENV_FILE=/secure/path/rivet-observability-test.env
test -r "$RIVET_OBSERVABILITY_TEST_ENV_FILE"
(
  set -a
  . "$RIVET_OBSERVABILITY_TEST_ENV_FILE"
  set +a
  export RIVET_EXTERNAL_SMOKE_TARGET=test
  pnpm observability:smoke -- --dry-run
  RIVET_EXTERNAL_SMOKE_CONFIRM=SEND:test pnpm observability:smoke
)

export RIVET_EXTERNAL_SMOKE_TARGET=production
pnpm observability:smoke -- --dry-run
RIVET_EXTERNAL_SMOKE_CONFIRM=SEND:production pnpm observability:smoke
unset RIVET_EXTERNAL_SMOKE_TARGET
```

시험용 명령은 subshell에서 실행하므로 앞에서 불러온 운영 환경을 덮어쓰지 않는다. `--dry-run`은 외부 호출 없이 payload 종류와 대상 이름만 검증한다. 실제 전송 확인 값은 `SEND:{target}`과 정확히 일치해야 한다. 명령은 허용 제품 이벤트 `search_performed`, [PostHog 수동 Error Tracking 계약](https://posthog.com/docs/error-tracking/installation/manual)의 `$exception_list`를 포함한 정제 예외 `$exception`, Slack `긴급`·`높음` 시험 경고를 각각 한 번 전송하고 비밀이나 공급자 응답 본문 대신 대상, `smokeId`, HTTP 상태와 성공 여부만 출력한다.

출력된 `smokeId`로 다음을 직접 확인한다.

1. PostHog에서 distinct ID `rivet_external_smoke_{smokeId}`의 `search_performed`와 request ID `req_external_smoke_{smokeId}`의 `ObservabilitySmokeTest`가 Error Tracking 예외로 `environment=production`, 대상 `RELEASE_ID`와 함께 수신됐다.
2. 제품 이벤트에는 합성 ID와 허용 속성만 있고 이메일·업무 본문·토큰·원본 파일명이 없다.
3. Slack 시험 채널에 같은 `smokeId`의 `긴급`·`높음` 메시지가 각각 한 건 도착했다.
4. PostHog Product Analytics 요금제가 [공식 정책](https://posthog.com/product-analytics-explorer/pricing)상 최소 1년의 데이터 보관을 제공한다. 현재 Free는 1년, 그 외 플랜은 7년이다.

네 항목의 수신 시각, 프로젝트·채널 이름, `RELEASE_ID`, `smokeId`와 확인자를 배포 기록에 남긴다. 시험 프로젝트의 합성 이벤트는 운영 제품 지표에 합산하지 않는다.

### Resend 실제 수신 점검

운영 발신 도메인과 분리된 개발 Resend 키·검증된 발신 도메인을 사용하고, 같은 테스트 메일함의 서로 다른 plus alias처럼 실제 수신 가능한 등록 계정용 주소와 미가입 초대용 주소를 준비한다. `onboarding@resend.dev`는 [Resend 계정 주소 한 곳에만 보낼 수 있으므로](https://resend.com/docs/knowledge-base/403-error-resend-dev-domain) 두 실제 주소를 쓰는 이 점검에는 사용할 수 없다. Resend Domains에서 개발 도메인이 `Verified`인지 확인하고 그 도메인의 발신 주소를 `RESEND_FROM`에 넣는다. 두 수신 주소만 `RESEND_ALLOWED_RECIPIENTS`에 넣고 `NODE_ENV=development`의 API·워커·웹을 기동한다.

1. 등록 계정용 주소로 가입해 인증 메일을 받고, 링크를 열어 이메일 인증과 로그인을 완료한다.
2. 같은 계정에서 비밀번호 재설정을 요청해 새 메일의 링크로 재설정하고, 이전 세션과 먼저 받은 재설정 링크가 무효인지 확인한다.
3. 관리자로 미가입 초대용 주소를 초대해 메일을 받고, 링크에서 가입·인증·로그인을 진행한 뒤 원래 링크 재진입 없이 초대로 자동 복귀해 수락한다. 별도 브라우저 로그인에서도 귀속된 초대가 복구되는지 함께 확인한다.
4. 세 메일의 From 주소·발신 도메인과 `WEB_ORIGIN`을 확인하고, 토큰이 query가 아니라 `#token=` fragment에 있으며 페이지 진입 직후 주소에서 제거되는지 확인한다.
5. Worker 구조화 로그와 Resend 대시보드에서 같은 Outbox ID의 중복 발송이 없고 각 전달에 provider message ID가 생겼는지 확인한다. 토큰 원문·메일 본문·수신 주소는 운영 기록이나 로그로 복사하지 않는다.

인증·재설정·초대별 수신 시각, 성공 여부, 발신 도메인, 환경과 확인자만 배포 기록에 남긴다. 허용 목록 밖 주소를 시험하지 않고 점검 뒤 개발 키를 운영 키로 재사용하지 않는다.

## 5. 애플리케이션 되돌리기

아래 조건을 모두 확인한 경우에만 직전 앱으로 되돌린다.

- 적용된 마이그레이션이 추가형이고 직전 앱이 새 schema를 읽을 수 있다.
- 새 API가 쓴 데이터와 Outbox payload를 직전 워커가 처리할 수 있다.
- 파일 저장 형식과 경로가 바뀌지 않았다.

조건을 만족하지 않으면 일반 되돌리기를 중단하고 전진 교정 migration 또는 격리된 백업 복구를 선택한다. 조건을 만족하면 외부 쓰기 차단과 worker claim 중단을 확인한 뒤 직전 릴리스 설정을 같은 순서로 적용한다.

```sh
pm2 stop rivet-web
pm2 stop rivet-api
pm2 stop rivet-worker

export RELEASE_ID="$PREVIOUS_RELEASE_ID"
test -n "$RELEASE_ID"
pm2 startOrRestart "$PREVIOUS_RELEASE_DIR/ecosystem.config.cjs" --only rivet-api --update-env
curl --fail --silent --show-error "$RIVET_ORIGIN/api/v1/health/ready"
pm2 startOrRestart "$PREVIOUS_RELEASE_DIR/ecosystem.config.cjs" --only rivet-worker --update-env
pm2 startOrRestart "$PREVIOUS_RELEASE_DIR/ecosystem.config.cjs" --only rivet-web --update-env
```

`RELEASE_ID`도 직전 릴리스 값으로 주입해야 한다. 이후 배포 후 점검과 Outbox 집계를 다시 수행하고, 성공한 경우에만 외부 쓰기를 열고 `pm2 save`를 갱신한다. 데이터베이스 schema는 임의로 되돌리지 않는다.

## 6. PostgreSQL·파일 저장소 백업

전체 백업은 짧은 점검 시간에 외부 쓰기를 차단하고 API·워커를 정상 종료한 뒤 수행한다. PostgreSQL 인증은 제한된 `.pg_service.conf`와 `.pgpass` 또는 동등한 비밀 저장소를 사용하고 연결 문자열을 명령행에 넣지 않는다.

```sh
export PGSERVICE=rivet-production
export BACKUP_DIR=/secure/backup/rivet/backup-id
umask 077
mkdir -p "$BACKUP_DIR"

pm2 stop rivet-web
pm2 stop rivet-api
pm2 stop rivet-worker

pg_dump --format=custom --file="$BACKUP_DIR/database.dump"
tar --create --file="$BACKUP_DIR/files.tar" --directory="$FILE_STORAGE_ROOT" .
pg_restore --list "$BACKUP_DIR/database.dump" >/dev/null
tar --list --file="$BACKUP_DIR/files.tar" >/dev/null
test -s "$BACKUP_DIR/database.dump"
test -s "$BACKUP_DIR/files.tar"
(
  cd "$BACKUP_DIR"
  sha256sum database.dump files.tar >SHA256SUMS
)
```

두 백업의 시작·종료 시각, `RELEASE_ID`, migration 목록과 checksum을 기록하고 운영 서버와 다른 제한된 저장 위치로 복제한다. 검증이 끝나면 API, 워커, 웹을 순서대로 시작해 smoke test를 수행한다. 종료 코드와 파일 크기만으로 백업 성공을 확정하지 않고 아래 복구 훈련까지 통과해야 한다.

## 7. 격리 복구 훈련

운영 DB·파일 경로를 대상으로 실행하지 않는다. 네트워크가 격리된 빈 PostgreSQL 데이터베이스와 빈 파일 루트를 만들고, 운영과 다른 `PGSERVICE`, `DATABASE_URL`, `FILE_STORAGE_ROOT`, `WEB_ORIGIN`을 사용한다.

```sh
export RESTORE_PGSERVICE=rivet-restore
export RESTORE_FILE_STORAGE_ROOT=/isolated/rivet-restore-files
test -d "$RESTORE_FILE_STORAGE_ROOT"
test -z "$(find "$RESTORE_FILE_STORAGE_ROOT" -mindepth 1 -print -quit)"
test "$(psql -X -v ON_ERROR_STOP=1 --dbname="service=$RESTORE_PGSERVICE" -Atc \
  "SELECT count(*) FROM pg_tables WHERE schemaname = 'public';")" = 0
(
  cd "$BACKUP_DIR"
  sha256sum --check SHA256SUMS
)

pg_restore --exit-on-error --single-transaction \
  --dbname="service=$RESTORE_PGSERVICE" "$BACKUP_DIR/database.dump"
tar --extract --file="$BACKUP_DIR/files.tar" --directory="$RESTORE_FILE_STORAGE_ROOT"

PGSERVICE="$RESTORE_PGSERVICE" FILE_STORAGE_ROOT="$RESTORE_FILE_STORAGE_ROOT" \
  sh scripts/check-file-storage-consistency.sh
```

정합성 스크립트는 `files.storage_key`와 `objects/`의 일반 파일을 읽기 전용으로 비교한다. 누락·고아·심볼릭 링크·디렉터리를 보고할 뿐 삭제하거나 DB를 수정하지 않는다. 불일치가 있으면 자동 정리를 실행하지 않고 복구 기준 시각과 백업 쌍을 다시 확인한다.

복원한 schema와 호환되는 릴리스를 격리 환경으로 시작한 뒤 migration 이력, 워크스페이스 격리, 로그인, 이슈, Outbox, 알림, SSE와 파일 흐름을 점검한다. 복구 시간, 데이터 손실 범위, 필요한 권한, 실패 단계와 확인자를 기록해야 훈련 성공으로 본다.

## 8. 완료 기록

비공개 베타 승인 기록에는 다음을 남긴다.

- 릴리스 커밋·`RELEASE_ID`와 빌드 시각
- 전체 출시 게이트 결과와 migration 목록·검토자
- PM2 세 프로세스 상태, Nginx 문법 검사와 smoke 결과
- 배포 또는 되돌리기 시작·종료 시각과 판단자
- PostgreSQL·파일 백업 쌍의 시각·checksum·외부 보관 위치
- 격리 복구 훈련의 소요 시간, 정합성 검사와 승인자
- 열려 있는 출시 차단 결함과 사용자 영향

실제 운영 배포, 백업·복구 훈련과 DNS·TLS·비밀 점검이 실행되기 전에는 문서와 로컬 설정 검증만으로 비공개 베타 출시 완료로 판정하지 않는다.
