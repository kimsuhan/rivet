# MVP 개발 실행 계획서

> 상태: 확정 v0.6
>
> 적용 범위: Rivet 비공개 MVP 베타의 작업 분해, 구현 순서와 출시 판정

## 1. 문서 목적

이 문서는 확정된 제품·화면·기술 명세를 실제 개발 작업과 검증 가능한 마일스톤으로 전환한다. 기능 정의를 다시 설명하는 문서가 아니라, 어떤 순서로 백엔드 계약을 준비하고 웹에 전달하며 어떤 조건에서 작업과 마일스톤을 완료할지를 정하는 실행 기준이다.

일정과 담당자 이름은 이 문서에 고정하지 않는다. 작업 단위가 실제 이슈로 분해된 뒤 팀 가용 인원과 측정된 처리량을 기준으로 정한다.

### 2.2 해석 원칙

- PRD의 `Must`는 MVP 출시 필수 범위다.
- `Should`는 기본적으로 구현하되 일정 위험이 확인되면 제품 책임자의 명시적 승인과 영향 기록이 있어야 제외할 수 있다.
- `Could`와 PRD 비목표는 MVP 작업으로 생성하지 않는다.
- 이 문서의 마일스톤 번호는 의존 순서이며 스프린트, 주차 또는 배포 날짜가 아니다.
- 상세 동작과 수치는 기준 문서를 따르며 이 문서에 같은 계약을 복사해 별도 정본을 만들지 않는다.
- 마일스톤은 기술 계층별 완료가 아니라 사용자가 끝까지 검증할 수 있는 수직 결과로 종료한다.

## 3. 개발 운영 모델

### 3.1 실제 개발 트랙

Rivet MVP 자체의 개발 트랙은 다음 두 개다. Rivet 제품 안의 프로젝트 역할에는 `앱 프론트`가 존재하지만 Rivet MVP의 네이티브 앱은 개발하지 않는다.

| 트랙 | 소유 범위 | 주요 산출물 |
| --- | --- | --- |
| 백엔드 | `apps/api`, `apps/worker`, `packages/database`, OpenAPI 생산 계약 | migration, API·워커 코드, OpenAPI, 생성 클라이언트, 통합 테스트, 작업 전달 |
| 웹 프론트 | `apps/web`, 생성 API 클라이언트 소비, 브라우저 상태와 UI | 화면·상태 처리, 접근성, 컴포넌트 테스트, E2E, 사용자 피드백 |

공통 린트·타입 설정과 저장소 도구는 변경을 시작한 트랙이 소유하되 다른 소비자의 검증까지 통과시킨다.

### 3.2 기능 작업 분리

- 하나의 제품 기능은 유형 선택 없는 `이슈 만들기`로 시작하고 이슈 아래 사용자가 선택한 역할의 최소 실행 작업을 둔다.
- 프로젝트 역할은 선택 가능한 범위만 정한다. 이슈마다 처음 작업할 역할을 선택하거나 분석을 위해 비워 둘 수 있고, 동시에 선택한 역할은 자동 선후 관계 없이 병렬로 시작한다.
- 웹 작업이 API에 의존하면 최초 작업 전달 대상 여부로 준비 상태를 계산한다.
- `API 작업 대기`, `프론트 작업 대기` 같은 교차 팀 상태를 공통 워크플로에 추가하지 않는다.
- 각 팀은 자신의 상태 흐름을 사용하고 백엔드·프론트 협업은 append-only 작업 전달로 표현한다.
- 공통 UI 셸, 디자인 토큰과 API가 필요 없는 화면 뼈대는 백엔드와 병렬로 진행할 수 있다.
- 작업의 수동 선행·후행 관계는 MVP에서 제공하지 않는다. 마일스톤과 우선순위, 전달 준비 상태로 실행 순서를 판단한다.

### 3.3 백엔드에서 웹으로 전달하는 조건

다음 항목을 모두 만족한 시점을 `웹 작업 가능`으로 본다.

1. 변경된 migration과 도메인·API 통합 테스트가 통과했다.
2. 다른 워크스페이스 ID와 권한 없는 역할의 실패 사례가 검증됐다.
3. OpenAPI와 `packages/api-client` 생성 결과가 최신이다.
4. 성공 요청·응답과 주요 오류 응답 예시가 있다.
5. 인증·권한 조건, CSRF와 workspace 경계가 설명돼 있다.
6. 로컬 또는 개발 환경에서 실제 호출할 수 있다.
7. 호환성 영향, migration 여부와 프론트 주의사항이 작업 전달에 기록돼 있다.

계약이 바뀌면 기존 작업 전달을 수정하지 않고 추가 전달을 남긴다. 웹은 백엔드 DTO나 Prisma 모델을 추측해 먼저 구현하지 않는다.

### 4.2 제외 범위

- 네이티브 모바일 앱과 모바일 보드·관리 화면
- 한 계정의 다중 워크스페이스 생성·참여와 전환
- 프로젝트 마일스톤, 타임라인, 상태 업데이트, 프로젝트 문서와 프로젝트 간 의존성
- 사이클, 승인 단계·외부 고객 접수가 있는 별도 트리아지 시스템, 저장된 보기, 이슈 템플릿과 수동 이슈 순서 저장
- GitHub·Slack·Sentry 제품 연동, 공개 API, OAuth 앱과 외부 웹훅
- 이메일 업무 알림, 모바일 푸시, 댓글 스레드, 이모지 반응과 협업 편집
- Redis·BullMQ·외부 메시지 브로커·WebSocket·마이크로서비스
- Kubernetes, 자동 CI/CD, 블루그린·카나리 배포와 자동 롤백

운영 경고를 Slack Incoming Webhook으로 보내는 기능은 제품 Slack 연동이 아니라 운영 관측 범위로만 취급한다.

## 5. 작업 준비와 완료 기준

### 5.1 실행 작업 준비 조건

개별 작업은 다음 정보가 있어야 시작할 수 있다.

- 연결된 PRD 요구사항 ID와 사용자 흐름 또는 화면 ID
- 사용자에게 보이는 완료 결과
- 담당 트랙과 전달 기반 준비 조건
- 변경할 API·데이터·화면 범위
- 성공, 실패, 권한과 충돌 인수 조건
- 필수 테스트 계층
- 범위에서 제외하는 동작

요구사항이 여러 독립 결과를 포함하면 작업을 나누고, 단순히 파일이나 계층이 다르다는 이유만으로 사용자 결과를 과도하게 쪼개지 않는다.

### 5.2 실행 작업 완료 조건

- 인수 조건과 권한·workspace 경계가 구현됐다.
- 관련 단위·통합·컴포넌트 테스트가 통과했다.
- 오류, 로딩, 빈 상태와 재시도 동작이 구현됐다.
- API 변경은 OpenAPI, 생성 클라이언트와 작업 전달이 최신이다.
- 데이터 변경은 migration SQL과 기존 데이터 영향을 검토했다.
- 로그와 분석에 민감 정보나 업무 본문이 포함되지 않는다.
- 변경 범위의 lint, typecheck와 build가 통과했다.
- 사용자 흐름이 바뀌면 대표 E2E 또는 브라우저 검증이 통과했다.

### 5.3 마일스톤 완료 조건

- 해당 마일스톤의 대표 사용자 흐름이 웹에서 끝까지 동작한다.
- 백엔드와 웹의 개별 완료 조건이 모두 충족됐다.
- 마일스톤에서 추가한 데이터의 다른 workspace 노출 방지 테스트가 있다.
- 실패 시 성공처럼 보이거나 데이터를 유실하는 경로가 없다.
- 알려진 미완료 항목은 `Should` 범위와 승인된 제외만 남아 있다.

## 7. 마일스톤 상세

### 7.1 M0 개발 기반

**목표:** 모든 후속 기능이 같은 명령, 계약과 로컬 환경에서 개발되도록 기반을 만든다.

**저장소·공통**

- Node.js 24, pnpm workspace와 Turborepo를 구성한다.
- `apps/web`, `apps/api`, `apps/worker`, `packages/database`, `packages/api-client`, `packages/config`를 초기화한다.
- 각 workspace의 `AGENTS.md`와 개발 지침 경계를 유지한다.
- ESLint Flat Config, Prettier, 공통 TypeScript 설정과 루트 검증 명령을 구성한다.
- 환경 변수 시작 검증, Pino 구조화 로그와 request/job ID를 구성한다.

**데이터·백엔드**

- Prisma multi-file schema, migration과 생성 클라이언트 경로를 구성한다.
- 로컬 `rivet.public` 테스트 DB와 격리된 파일 임시 경로를 테스트에 연결한다.
- API 공통 오류, ValidationPipe, OpenAPI JSON·Swagger와 Orval 생성 경로를 구성한다.
- API liveness·readiness와 워커 시작·종료 골격을 구성한다.
- 최소 Outbox 저장·polling 구조는 계정 이메일이 추가될 수 있는 형태로 준비하되 미래 이벤트를 미리 만들지 않는다.

**웹**

- Next.js App Router, next-intl `ko`, Tailwind, shadcn/ui와 디자인 토큰을 구성한다.
- 앱 셸, 오류 경계, TanStack Query, 생성 Fetch mutator와 라우팅 뼈대를 구성한다.
- API가 필요 없는 공통 로딩·빈 상태·오류 컴포넌트를 검증한다.

**통과 기준**

- `pnpm lint`, `pnpm typecheck`, `pnpm test`, `pnpm test:integration`, `pnpm api:contract:check`, `pnpm build`가 실행 가능하다.
- 웹·API·워커가 각각 시작되고 API readiness와 로컬 PostgreSQL 연결을 확인할 수 있다.
- OpenAPI 변경이 생성 클라이언트 diff로 이어지는 최소 계약 검사가 통과한다.

### 7.5 M4 프로젝트와 작업 전달

**목표:** 프로젝트를 역할별 팀으로 구성하고 이슈별 시작 작업과 백엔드의 프론트 전달을 관리한다.

**백엔드 선행 작업**

- 프로젝트, 역할별 팀, 이슈, 점진적으로 생성되는 팀 작업과 실제 작업 기준 진행률 계산을 구현한다.
- 기본 이슈와 사용자가 선택한 역할의 최초 팀 작업을 하나의 트랜잭션에서 생성하고, 팀 작업 없는 이슈의 선택형 `작업 시작` API를 제공한다.
- 프로젝트 상태·보관과 사용 중 역할별 팀 변경 제한을 구현한다.
- 프로젝트 역할과 팀 작업의 역할·팀·프로젝트 일치 제약을 구현한다.
- 팀 작업 관계를 제거하고 최초 전달 대상에 따른 `API 전달 대기`·`작업 가능` 파생 상태를 구현한다.
- 공통 Lexical Markdown 계약을 기반으로 최초·추가 작업 전달을 구현한다.
- 프로젝트에 웹·앱 역할이 있는 백엔드 작업은 후행 작업 존재 여부와 무관하게 최초 전달 없이 완료하지 못하게 한다.
- `destinationRoles`별 프론트 작업 생성 또는 재사용과 추가 전달 무생성을 구현한다.
- 최초 전달, 백엔드 완료, 대상 프론트 작업의 준비 상태, 이슈 진행률, 활동과 Outbox 이벤트를 원자적으로 기록한다.

**웹 작업**

- PROJ-01~03, 일반 이슈 상세와 분석·병렬·순차·전달·완료 상태의 작업 흐름을 구현한다.
- 프로젝트의 백엔드·웹·앱 역할별 팀을 선택하거나 비우고 최소 한 역할을 검증한다.
- 전역·프로젝트·팀 화면의 업무 생성은 유형 선택 없는 `이슈 만들기`로 통일하고 팀 작업은 이슈의 역할 선택·작업 시작·전달에서만 만든다.
- 이슈 생성과 팀 작업 없는 이슈의 `작업 시작`에서 프로젝트 역할의 부분집합을 복수 선택하고, 선택하지 않은 생성도 정상 처리한다.
- 이슈 아래에 실제 팀 작업 진행률과 전달 기반 준비 상태를 표시하고 팀 작업이 없으면 분석 안내를 제공한다.
- 백엔드 작업의 `전달하고 완료`, 대상 역할 선택, 결과 후행 작업 링크와 추가 전달 표시를 구현한다.
- 상위 이슈 작업 흐름과 전달받은 프론트 작업에서 같은 원본 전달의 최초 내용과 추가 이력을 표시한다.

**통과 기준**

- UF-04~06을 웹에서 완료할 수 있다.
- 프로젝트 역할과 다른 팀의 작업, 상위 이슈 없는 작업과 팀 작업 관계가 생성되지 않는다.
- 이슈와 최초 팀 작업, 전달·완료·대상 프론트 작업·활동·Outbox가 각 경계에서 함께 커밋되거나 함께 롤백된다.
- 기존 후행 작업은 중복 없이 모두 재사용되고 같은 완료 요청 재시도와 추가 전달이 새 팀 작업을 만들지 않는다.
- 프론트 작업 준비 상태는 전달 대상으로 계산하며 팀 워크플로 상태와 분리한다.
- OpenAPI, 생성 클라이언트와 실제 작업 전달 템플릿이 일치한다.

### 7.6 M5 협업과 파일

**목표:** 이슈의 상세 맥락과 업무 파일을 안전하게 한곳에 남긴다.

**백엔드 선행 작업**

- 댓글·멘션·파일 연결·활동·Outbox를 각 저장 트랜잭션에 포함한다.

- 일반 파일 다중 첨부, 진행·실패·재시도·제거와 다운로드를 구현한다.

- PROFILE-01과 이슈 상세의 댓글·활동·첨부 영역을 완성한다.

**통과 기준**

- 미리보기와 저장 후 렌더링 결과가 같고 스크립트·위험 링크가 실행되지 않는다.
- 업로드 일부 실패가 성공 파일을 잃게 하지 않고 완료되지 않은 본문 업로드가 저장되지 않는다.
- 연결된 파일은 연결 해제 또는 휴지통 정책 전에는 정리 대상이 되지 않는다.

### 7.7 M6 알림·검색·실시간

**목표:** 사용자가 자신에게 필요한 변경을 놓치지 않고 다시 찾아 최신 상태로 수렴한다.

**백엔드 선행 작업**

- 이벤트별 알림 후보, 자기 자신·비활성 멤버 제외와 수신자 중복 제거를 구현한다.
- 최초 전달의 재사용 작업은 담당자·자동 구독자, 새 미할당 작업은 대상 팀 활성 멤버를 사건 시점 후보로 저장한다.
- 알림 목록·읽음·모두 읽음과 읽지 않은 개수 API를 구현한다.
- 이슈·프로젝트 전역 검색과 workspace 격리·안정 정렬을 구현한다.
- PostgreSQL `LISTEN/NOTIFY`, 인증된 SSE와 최소 식별 신호를 구현한다.
- 워커 재처리에도 알림이 중복되지 않고 지원하지 않는 payload가 영구 실패로 격리되게 한다.

**웹 작업**

- WORK-02 알림함, 읽지 않은 전역 배지와 관련 댓글·전달 앵커 이동을 구현한다.
- SEARCH-01 전역 검색과 키보드 탐색을 구현한다.
- SSE 신호를 Orval Query Key 무효화와 재조회에 연결한다.
- 연결 끊김을 표시하고 재연결 후 현재 화면 데이터를 다시 검증한다.
- 중복 이벤트가 낙관적 UI와 충돌하지 않도록 최신 서버 상태에 수렴시킨다.

**통과 기준**

- 멘션·구독·할당·완료·작업 전달 알림이 작성자를 제외하고 수신자별 한 건만 생성되며 Outbox 재처리에도 중복되지 않는다.
- 다른 사용자의 알림과 다른 workspace 검색 결과·SSE 신호가 노출되지 않는다.
- 알림 목록과 전역 읽지 않은 수가 항상 일치한다.
- SSE를 강제로 끊은 뒤 재연결 REST 조회로 최신 상태에 수렴한다.
- 검색 결과 없음, 연결 끊김과 재시도 흐름을 웹에서 확인할 수 있다.

### 7.8 M7 삭제·운영·베타 출시

**목표:** 데이터 복구와 운영 진단이 가능한 상태로 비공개 MVP 베타를 시작한다.

**백엔드·워커 작업**

- 이슈·비어 있는 프로젝트 휴지통 이동, 관리자 복구와 30일 예약 영구 삭제를 구현한다.
- 미연결 24시간·임시 1시간 파일 정리와 실제 참조 재검증을 구현한다.
- 세션·토큰·속도 제한·이메일·내보내기 기록 보존 작업을 구현한다.
- 관리자 이슈·프로젝트 CSV 내보내기와 수식 주입 방어를 구현한다.
- readiness, 구조화 로그, PostHog, 최소 지표와 운영 Slack Incoming Webhook 경고를 구성한다.

**웹·운영 작업**

- SET-05 CSV 내보내기와 SET-06 휴지통·복구 화면을 구현한다.
- 공통 권한 없음, 충돌, 네트워크 실패와 복구 상태를 전체 핵심 화면에서 점검한다.
- 제한적 모바일 웹, 키보드 탐색과 접근성 기준을 핵심 흐름에서 검증한다.
- PM2·Nginx·SSE·업로드 설정과 수동 배포·되돌리기를 격리 환경에서 실행한다.
- PostgreSQL과 파일 저장소를 함께 백업하고 복구하는 훈련을 수행한다.

**통과 기준**

- 복구한 리소스와 연결 파일이 영구 삭제되지 않고 정리 작업 재실행이 멱등하다.
- 다른 workspace 데이터가 휴지통·검색·CSV·파일에서 노출되지 않는다.
- 루트 운영 출시 게이트와 Playwright 핵심 흐름이 모두 통과한다.
- API·워커가 같은 DB와 파일 저장소를 사용하고 배포 후 readiness와 worker 처리를 확인한다.
- 백업 복구 뒤 데이터베이스 metadata와 파일 바이너리가 일치한다.
- 출시 차단 결함이 하나도 열려 있지 않다.

### 7.9 M8 전역 이슈 접수·분류와 배정 보완

**목표:** 워크스페이스 전체 이슈를 프로젝트 경계와 무관하게 한곳에서 검토하고 팀 작업 시작·담당자 배정·완료 확인까지 처리한다.

**백엔드 선행 작업**

- `GET /issues`에 다섯 `workQueue`, 세부 필터, 진행률 정렬과 안정 커서를 제공한다.
- `workflowSummary`, 현재 역할 팀·현재 사용자 팀 역할·현재 사용자 담당 작업, 만든 사람과 `totalCount`를 페이지 단위 고정 쿼리 수로 반환한다.
- `/issues/{id}/team-works`의 `initialRoles`, 역할별 담당자가 있는 `roleAssignments`와 `requireCurrentUserTeamMembership` 검증을 제공한다.
- `/issues/{id}/claim`과 `/issues/{id}/assign-team-works`를 구현하고 작업·역할·팀·멤버십·버전·이슈를 잠금 뒤 재검증한다.
- 작업 생성·재사용·담당자·구독·활동과 `TEAM_WORK_ASSIGNED` Outbox를 각 요청의 한 트랜잭션에서 처리한다.
- OpenAPI와 생성 API 클라이언트를 갱신하고 API·PostgreSQL·Worker 단위·통합 테스트를 추가한다.

**웹 작업**

- `/issues` 전역 목록과 기존 `/issues/:displayId`, `/my-issues` 직접 경로를 연결하고 사용자 명칭을 `이슈`, `내 작업`으로 분리한다.
- 데스크톱·모바일 주 탐색에서 `이슈 → 내 작업 → 알림함 → 프로젝트` 상대 순서를 유지한다. 모바일 하단의 다섯 번째 항목은 팀으로 두고 검색은 상단 전역 동작으로 이동한다.
- 빠른 필터, 세부 필터, 정렬과 검색을 URL에서 복원하고 현재 결과 수와 정확한 빈 상태를 표시한다.
- 행에 현재 역할·팀, 담당 필요, 진행률, 만든 사람과 다음 행동 한 개를 표시하고 상태·우선순위·라벨 빠른 변경을 기존 선택기로 구현한다.
- 작업 시작, 우리 팀에서 시작, 내가 맡기, 담당자 지정과 이슈 완료를 데스크톱 모달·모바일 시트에서 제공하고 충돌 시 선택값과 최신 후보를 유지한다.
- SSE 재조회 뒤 현재 조건에서 빠진 행만 제거하고 필터·정렬·현재 페이지·스크롤을 유지한다.

**통과 기준**

- 다섯 빠른 필터와 세부 필터 조합이 확정 조건과 일치하고 진행률·상태·우선순위 동률의 다중 페이지에 중복·누락이 없다.
- 역할별 작업 시작, claim과 일괄 배정이 전부 커밋되거나 전부 롤백되며 다른 워크스페이스·비활성 멤버·보관 팀과 오래된 버전이 거부된다.
- 배정 뒤 전역 이슈, 내 작업과 팀 목록이 같은 팀 작업·담당자를 표시하고 자신의 행동 알림이나 같은 사건의 중복 알림이 없다.
- 데스크톱·모바일에서 새 이슈 생성부터 검토 필요, 작업 시작·배정, 모든 팀 작업 완료, 완료 확인과 이슈 완료까지 Chromium E2E를 통과한다.
- API 계약 검사, 단위·통합·웹 테스트, 타입 검사, 린트, 빌드, 문서 링크와 diff 검증이 모두 통과한다.

### 7.10 M9 이슈 콘텐츠와 팀 실행 구조 통합

**목표:** 이슈와 팀 작업이 같은 문서처럼 중복되던 구조를 제거하고, 작업자가 내 작업에서 들어와도 이슈 맥락과 자신의 실행 상태를 한 화면에서 이해하고 처리하게 한다.

**백엔드 선행 작업**

- `issues`를 콘텐츠 리소스로 정리하고 `team_works`, `api_handoff_targets`를 유지하며 `team_work_relations`를 새 마이그레이션으로 제거한다.
- `IssueType`, `FEATURE`, `TEAM_TASK`, `issues`의 팀 작업 전용 컬럼과 구형 관계를 바로 제거한다. 실데이터가 없으므로 이중 쓰기, 데이터 보존용 호환 DTO와 deprecated 필드를 만들지 않는다.
- 기존 마이그레이션은 수정하지 않고 개발·테스트 데이터베이스는 필요하면 초기화해 전체 마이그레이션을 재검증한다.
- `/issues`와 `/team-works` 계약을 분리하고 상위 이슈 없는 팀 작업 생성, 팀 작업 제목·Markdown 설명·우선순위·라벨·프로젝트·첨부파일 계약을 제거한다.
- 댓글·활동·첨부파일은 이슈에, 작업 전달은 원본·대상 팀 작업 관계에 연결한다. 댓글과 활동의 `teamWorkId`는 선택 문맥으로만 사용한다.
- OpenAPI, `packages/api-client`, Worker 이벤트와 알림 앵커를 새 계약으로 동시 전환한다.

**웹 작업**

- 이슈 상세와 팀 작업 상세의 콘텐츠 레이아웃을 하나의 통합 상세 본문으로 합친다. 이슈 중심 진입은 `/issues/{issueIdentifier}?tab=work&work={teamWorkIdentifier}`, 내 작업 진입은 `/my-issues/{teamWorkIdentifier}?tab=work`를 사용하고 활성 내비게이션·복귀 링크·상단 위계만 구분한다.
- 팀 작업 표시 ID 딥 링크는 통합 상세로 이동시키고 별도 팀 작업 콘텐츠 편집기·첨부파일·댓글 화면을 제거한다.
- 내 작업에서 팀 작업을 열면 `내 작업` 활성 상태를 유지하고, 팀 목록·검색·알림에서 열면 이슈 중심 문맥을 유지한다. 어느 경로에서도 상위 이슈 설명·첨부·대화와 선택 작업의 상태·담당자·작업 노트·최신 전달은 같은 데이터와 본문 컴포넌트로 표시한다.
- 이슈 목록·프로젝트 상세·통합 상세가 같은 이슈 상태·진행률·전달 기반 준비 상태를 사용하게 한다.
- 팀 화면의 직접 팀 작업 생성은 `이슈 만들기`로 교체한다.

**통과 기준**

- 데이터베이스, Prisma, DTO, OpenAPI와 생성 클라이언트에 구형 이슈 유형과 팀 작업 콘텐츠 컬럼이 남지 않는다.
- 모든 팀 작업은 정확히 하나의 이슈를 가지며 상위 이슈 없는 생성 경로가 없다.
- 팀 작업 화면에서 제목·설명·첨부파일·우선순위·라벨·프로젝트를 별도로 저장하거나 편집하지 않는다.
- 이슈 설명·첨부·댓글 변경이 전역 이슈, 프로젝트, 내 작업, 팀 목록과 통합 상세에 같은 원본으로 반영된다.
- 최초 전달은 후행 팀 작업과 명시적 대상 관계로 연결되고 알림은 이슈 통합 상세의 올바른 작업·전달 앵커를 연다.
- API·PostgreSQL·Worker·웹 단위·통합 테스트, OpenAPI 계약 검사, 전체 Chromium E2E, 타입 검사, 린트와 빌드가 통과한다.

## 8. 백로그 작성 기준

### 8.1 작업 계층

| 계층 | 용도 | 예시 |
| --- | --- | --- |
| 마일스톤 | 검증 가능한 제품 결과 | `M4 프로젝트와 작업 전달` |
| 이슈 | 하나의 사용자 결과와 요구사항 묶음 | `프로젝트 역할별 담당 팀 설정` |
| 팀 작업 | 한 팀이 독립적으로 완료·검증할 실행 단위 | `[API] 프로젝트 생성 계약`, `[WEB] 프로젝트 생성 화면` |

이슈는 진행률과 맥락을 묶고 실제 담당자·상태는 점진적으로 생성되는 팀 작업에 둔다. 백엔드와 웹 작업을 하나의 실행 작업에 함께 넣지 않는다.

### 8.2 이슈 설명과 팀 작업 메모

목표, 요구사항, 범위, 제외 범위, 인수 조건과 공통 자료는 이슈 설명에 한 번만 작성한다. 팀 작업은 상태·담당자·팀·역할과 선택 Markdown `작업 노트`만 가지며, 노트는 해당 팀의 구현 범위나 주의사항을 최대 10,000자로 기록한다. 본문 이미지·일반 첨부파일·멘션과 이슈 설명 복사본은 팀 작업에 만들지 않는다.

백엔드의 API 변경 요약과 링크처럼 프론트로 전달할 정식 내용은 작업 노트가 아니라 추가 전용 `작업 전달`에 기록한다.

- 제목 접두어는 실제 팀 또는 역할이 드러나는 `[API]`, `[WORKER]`, `[DB]`, `[WEB]`을 사용한다.
- 요구사항 ID는 링크 가능한 형태로 적고 같은 요구사항을 여러 작업이 담당하면 최종 완료 작업을 명시한다.
- 백엔드 작업 전달은 API 변경이 있을 때만 형식적으로 쓰는 것이 아니라 후행 웹 작업이 구현할 수 있는 계약을 제공해야 한다.
- 완료하지 않은 검증과 승인되지 않은 범위 제외를 작업 전달로 숨기지 않는다.

- 불안정 테스트의 재실행 성공으로 첫 실패를 무시하지 않는다.

## 10. 출시 판정

### 10.1 내부 프로토타입 판정

M3 완료 후 다음을 확인한다.

- 단일 워크스페이스와 팀에서 이슈 생성·수정·목록·보드·내 작업이 동작한다.
- 내부 사용자가 별도 안내 없이 실제 일주일 업무를 기록하고 매일 다시 열어 관리할 수 있다.
- 저장 성공 뒤 데이터 유실, 표시 ID 중복과 화면 간 지속적인 불일치가 없다.

내부 사용에서 발견한 출시 차단 수준 결함은 M4 기능 추가 전에 해결한다.

### 10.2 비공개 MVP 베타 판정

다음을 모두 만족해야 한다.

1. M0~M9의 통과 기준을 만족한다.
2. `pnpm lint`, `pnpm typecheck`, `pnpm test`, `pnpm test:integration`, `pnpm api:contract:check`, `pnpm test:e2e`, `pnpm build`가 통과한다.
3. migration SQL, 수동 배포와 애플리케이션 되돌리기를 검토·실행했다.
4. PostgreSQL과 파일 저장소 백업 복구를 실제로 검증했다.
5. 운영 환경 변수, Resend, PM2, Nginx, SSE, 업로드와 Slack 운영 경고를 점검했다.
6. 열려 있는 출시 차단 결함이 없다.

베타 시작 뒤 1~10명 팀이 4주 연속 실제 업무에 사용하며 PRD의 관찰 지표와 정성 피드백을 수집한다. 임의의 업계 수치 대신 이 베타 기준선과 기존 도구로 돌아간 이유를 함께 판단한다.

### 10.3 출시 차단 결함

- 다른 workspace의 데이터, 파일, 알림, 검색 결과 또는 CSV 노출
- 관리자 권한, 이메일 인증, CSRF, 세션과 일회용 토큰 우회
- 저장 성공 뒤 데이터 유실, 표시 ID 중복과 migration 불일치
- 프로젝트 역할·팀·소속 이슈 불일치 또는 전달 대상 역할 불일치
- 필요한 작업 전달 없이 백엔드 작업 완료
- 이슈 생성이나 최초 전달 실패 뒤 일부 팀 작업·관계·활동·Outbox만 남음
- 전역 이슈 빠른 필터 오분류, 안정 커서 중복·누락이나 페이지 행마다 증가하는 팀 작업 조회
- 작업 시작·claim·일괄 배정 실패 뒤 일부 팀 작업·담당자·구독·활동·Outbox만 남거나 오래된 화면이 새 담당자를 덮어씀
- 동일 최초 완료 요청 또는 Outbox 재처리로 전달·후행 작업·관계·알림이 중복됨
- Outbox 재처리로 알림·메일·삭제 중복
- 연결 파일의 고아 파일 오판 삭제 또는 복구 후 metadata 불일치
- API·워커의 DB·파일 저장소 불일치
- 백업 복구 절차 부재와 운영자가 치명적 실패를 탐지할 수 없는 상태
- 비밀번호·세션·토큰·비밀 키 또는 업무 본문의 로그·분석 기록

## 12. 범위 변경과 실행 점검

- 새 요구가 들어오면 기존 `Must` 완료와 출시 차단 위험에 미치는 영향을 먼저 기록한다.
- MVP 비목표를 추가하려면 대체로 제외할 범위와 출시 일정 영향을 제품 책임자가 승인해야 한다.
- `Should` 제외는 요구사항 ID, 사용자 영향, 우회 방법과 재검토 조건을 남긴다.
- 기술적 편의를 이유로 사용자 흐름을 여러 임시 계약으로 분기하지 않는다.
- 마일스톤 종료 시 요구사항 추적표, 백엔드 전달, 테스트 결과와 알려진 결함을 함께 검토한다.
