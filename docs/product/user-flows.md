# 사용자 흐름

> 상태: 현행
> 적용 범위: 웹에서 제공하는 주요 사용자 흐름과 공통 상호작용
> 정본 책임: 화면을 넘나드는 사용 목적과 실패 처리
> 갱신 조건: 진입 경로, 주요 행동 순서 또는 공통 오류 처리 변경

실제 라우트는 [`apps/web/app`](../../apps/web/app/), 화면 구현은
[`apps/web/features`](../../apps/web/features/)를 정본으로 한다. 이 문서는 화면별 필드와
버튼을 복제하지 않는다.

## 계정 시작

1. 사용자가 가입하고 이메일을 확인한다.
2. 워크스페이스를 만들거나 초대를 수락한다.
3. 워크스페이스 생성자는 첫 팀과 워크플로를 준비한다.
4. 로그인 뒤 마지막으로 선택한 워크스페이스의 업무 화면으로 이동한다.

초대 이메일이 기존 계정, 미확인 계정 또는 신규 사용자에게 도착해도 인증 뒤 같은 초대
수락 흐름으로 복귀해야 한다. 만료·취소·이미 수락한 초대는 서로 구분해 다음 행동을 안내한다.

## 팀과 멤버 관리

1. 관리자는 워크스페이스 멤버를 초대하고 전체 역할을 관리한다.
2. 관리자는 팀을 만들고 한 명 이상의 팀장을 지정한다.
3. 관리자 또는 해당 팀장은 기존 멤버를 팀에 추가하거나 이메일로 팀원을 초대한다.
4. 팀별 워크플로 상태, 순서와 기본값을 관리한다.

미완료 작업을 맡은 멤버 제거, 사용 중인 상태 삭제와 진행 중 작업이 있는 팀 보관은 영향을
먼저 보여 주고 필요한 재할당·대체·완료 경로를 제공한다.

## 프로젝트와 이슈

1. 프로젝트를 만들고 리드, 기간과 참여 팀을 정한다.
2. 전역 이슈 또는 프로젝트 안에서 이슈를 만든다.
3. 템플릿을 선택할 수 있지만 적용 뒤의 입력은 사용자가 자유롭게 수정한다.
4. 이슈에 하나 이상의 참여 팀 작업을 시작하고 담당자를 지정한다.
5. 각 팀은 자신의 워크플로에서 작업을 진행한다.

이슈 본문, 라벨, 첨부파일과 댓글은 모든 팀 작업이 공유한다. 사용자가 내 작업이나 팀 목록에서
진입해도 별도 팀 작업 문서가 아니라 같은 이슈 상세에서 해당 팀 작업을 선택해 본다.

## 작업 실행과 전달

1. 사용자는 내 작업, 팀 보드 또는 이슈 상세에서 팀 작업을 연다.
2. 상태와 담당자를 바꾸고 필요한 작업 노트·댓글·첨부파일을 남긴다.
3. 완료할 때 현재 작업만 완료하거나 다른 참여 팀에 전달한다.
4. 전달을 선택하면 대상 팀과 전달 내용을 확인한 뒤 원본 완료와 대상 작업 생성을 함께 실행한다.
5. 이후 추가 전달은 기존 이력을 바꾸지 않고 후속 기록으로 남긴다.

상태 충돌, 참여 팀 변경 또는 담당 권한 변경이 있으면 오래된 화면의 값으로 덮어쓰지 않는다.
최신 값을 다시 읽고 사용자가 작성한 입력은 가능한 한 유지한다.

## 배포

1. 팀 작업의 배포 적용 여부와 필요한 선행·동시 배포 작업을 확인한다.
2. 업무 완료와 배포 조건이 충족된 작업만 배포 가능으로 표시한다.
3. 개별 팀 작업 또는 프로젝트 범위에서 준비된 작업을 배포 완료 처리한다.
4. 완료 뒤 작업이 바뀌면 재배포 필요 상태로 돌아간다.

배포 상태는 일반 워크플로 상태와 별도다. 이슈 목록의 라벨이나 완료 상태에서 배포 여부를
추론하지 않고 배포 화면에서 명시적으로 확인한다.

## 찾기와 개인화

- 전역 이슈는 여러 프로젝트의 업무를 접수·분류하는 목록이다.
- 내 작업은 현재 사용자에게 할당된 미완료 팀 작업을 실행하는 목록이다.
- 팀 목록·보드는 해당 팀의 워크플로와 담당 업무를 보여 준다.
- 프로젝트 상세는 목표, 참여 팀과 연결된 이슈의 전체 흐름을 보여 준다.
- 검색, 필터, 다중 정렬과 표시 구성은 개인 저장된 보기로 보관할 수 있다.
- 상세에서 목록으로 돌아오면 가능한 범위에서 검색·필터·정렬과 스크롤 문맥을 복원한다.

## 알림

1. 사용자는 인앱 알림함에서 할당, 멘션, 댓글, 완료와 전달 사건을 확인한다.
2. 알림을 열면 정확한 이슈 또는 팀 작업 문맥으로 이동한다.
3. 사용자가 설정에서 Web Push를 활성화하면 앱을 열지 않은 동안 같은 사건을 받을 수 있다.
4. 알림 읽음 상태와 원본 업무 상태는 서로 독립적으로 관리한다.

## 데이터 가져오기와 내보내기

- CSV 가져오기는 파일 선택, 구조 검사, 값 매핑, 저장 전 검증, 최종 확인과 결과 순서로 진행한다.
- 실패하면 어떤 단계에서 무엇을 고쳐야 하는지 보여 주며 부분 저장하지 않는다.
- 내보내기는 파일에 업무 내용이 포함될 수 있음을 실행 전에 안내한다.
- 가져오기·내보내기 파일, 토큰과 업무 본문을 일반 로그나 제품 분석에 남기지 않는다.

## 공통 화면 상태

- 모든 비동기 화면은 로딩, 빈 상태, 오류, 권한 없음과 충돌 상태를 제공한다.
- 사용할 수 없는 행동은 이유와 해결 경로를 설명한다.
- 저장 실패 시 입력을 보존하고 같은 동작을 안전하게 다시 시도할 수 있게 한다.
- 권한 없는 리소스는 다른 워크스페이스의 존재 여부를 드러내지 않는다.
- 모바일은 데스크톱을 축소하지 않고 조회와 핵심 변경 중심의 한 열 흐름을 제공한다.
- 키보드 포커스, 접근 가능한 이름, 포커스 복귀와 색상 외 상태 표현을 유지한다.
