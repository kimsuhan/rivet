# 핵심 용어와 정책

> 상태: 현행
> 적용 범위: 제품 전반의 데이터 소유권, 권한과 상태 판단
> 정본 책임: 코드 여러 계층이 공유하는 제품 불변식
> 갱신 조건: 용어, 소유권, 권한 경계 또는 상태 계산 방식 변경

## 핵심 용어

| 용어 | 의미와 원본 |
| --- | --- |
| 워크스페이스 | 사용자, 팀, 프로젝트와 업무 데이터를 격리하는 최상위 경계 |
| 멤버십 | 한 사용자의 현재 워크스페이스 소속과 `ADMIN` 또는 `MEMBER` 역할 |
| 팀 | 멤버, 팀장, 키와 워크플로를 소유하는 실행 조직 |
| 팀장 | 지정된 팀의 정보·멤버·초대·워크플로만 관리하는 팀 범위 역할 |
| 프로젝트 | 목표, 기간, 리드와 실제 참여 팀을 묶는 업무 범위 |
| 프로젝트 참여 팀 | 프로젝트와 실제 팀을 연결하는 관계. 팀 정보와 워크플로를 복제하지 않음 |
| 이슈 | 제목, 설명, 우선순위, 라벨, 댓글과 첨부파일을 소유하는 업무 내용의 원본 |
| 팀 작업 | 한 팀이 이슈를 실행하기 위한 상태·담당자·배포 정보 |
| 작업 전달 | 원본 팀 작업에서 다른 참여 팀의 작업으로 전달한 내용과 연결 이력 |
| 워크플로 상태 | 팀이 관리하는 표시 상태. 시스템 범주가 진행률과 종료 여부 계산에 사용됨 |
| 배포 | 완료된 팀 작업의 배포 가능 조건과 실제 배포 완료 기록 |

Prisma 모델명과 필드는
[`packages/database/prisma/models`](../../packages/database/prisma/models/)를 정본으로 한다.

## 콘텐츠와 실행의 분리

- 이슈가 업무 콘텐츠의 유일한 원본이다.
- 팀 작업은 이슈 본문을 복제하지 않고 팀, 상태, 담당자, 전달과 배포 같은 실행 정보만 가진다.
- 목록과 상세는 같은 이슈를 보여 주며 진입 경로에 따라 선택된 팀 작업과 복귀 문맥만 달라진다.
- 이슈 상태와 진행률은 연결된 팀 작업의 시스템 범주에서 계산한다. 팀 이름이나 종류로 계산하지 않는다.
- 취소된 팀 작업은 진행률의 완료 대상에서 제외한다.

## 팀과 권한

- 워크스페이스 관리자는 전체 멤버, 팀장 지정, 공용 설정과 관리 기능을 담당한다.
- 팀장은 자신이 이끄는 팀의 이름·설명, 멤버, 이메일 초대와 워크플로를 관리할 수 있다.
- 팀장은 다른 팀, 워크스페이스 전체 역할, 멤버 비활성화와 프로젝트 참여 구성을 임의로 바꿀 수 없다.
- 팀에는 한 명 이상의 팀장을 둘 수 있으며 팀장 지정·해제는 워크스페이스 관리자만 수행한다.
- 사용 중인 워크플로 상태는 참조를 정리하거나 대체 상태를 선택하지 않고 삭제할 수 없다.
- 권한이 없는 리소스의 존재 여부를 불필요하게 노출하지 않는다.

## 프로젝트와 참여 팀

- 프로젝트는 참여 팀 없이 생성할 수 있지만 팀 작업을 시작하려면 활성 참여 팀이 필요하다.
- 같은 프로젝트와 팀 조합은 하나의 활성 참여 관계만 가진다.
- 팀 이름, 멤버와 워크플로의 원본은 `Team`이며 프로젝트 참여 관계가 이를 복제하지 않는다.
- 진행 중인 작업이 있는 참여 팀은 즉시 제외할 수 없다.
- 완료·취소된 작업만 남은 참여 관계는 비활성화하고, 다시 참여할 때 기존 이력을 유지한다.
- 모든 참여 팀의 작업은 기본적으로 병렬이다. 선후 관계는 명시적인 작업 전달이나 배포 의존성으로만 만든다.

## 작업 전달과 완료

- 완료는 현재 작업만 완료하거나 다른 활성 참여 팀에 전달한 뒤 완료할 수 있다.
- 전달 대상에 활성 팀 작업이 있으면 재사용하고, 없으면 해당 팀의 기본 워크플로 상태로 생성한다.
- 소스 작업 자신, 다른 프로젝트 또는 비활성 참여 팀으로 전달할 수 없다.
- 원본 완료, 전달 기록, 대상 팀 작업, 활동과 Outbox는 하나의 트랜잭션으로 반영한다.
- 전달이나 완료 충돌 시 사용자의 입력을 보존하고 최신 상태를 다시 확인하게 한다.

## 목록과 저장된 보기

- 전역 이슈는 업무 내용, 내 작업은 현재 사용자에게 할당된 팀 작업을 중심으로 보여 준다.
- 검색·필터·정렬처럼 공유하거나 복원할 상태는 URL에 둔다.
- 저장된 보기는 멤버십 개인 소유이며 다른 사용자와 공유하지 않는다.
- 다중 정렬은 결정적인 마지막 키를 포함해 페이지 이동 중 중복·누락을 막는다.

## 템플릿과 가져오기

- 이슈 템플릿은 생성 시점 입력값이며 생성된 이슈와 계속 동기화하지 않는다.
- 템플릿 버전, 보관 상태와 적용 대상을 서버에서 다시 검증한다.
- 템플릿이 오래되거나 사용할 수 없으면 작성 중인 입력을 유지하고 재선택을 요구한다.
- CSV 가져오기는 검사, 값 매핑, 최종 검증과 실행을 분리한다.
- 가져오기는 전체 성공 또는 전체 롤백이며 부분 프로젝트·이슈를 남기지 않는다.
- 같은 원본 행을 중복 실행하지 않고, 다른 워크스페이스나 비활성 대상을 매핑할 수 없다.

## 알림과 이메일

- 인앱 알림이 제품 알림의 원본이다.
- Web Push는 사용자가 명시적으로 활성화한 구독으로 인앱 알림을 전달하는 채널이며 별도 수신자 모델이 아니다.
- 브라우저 알림 권한은 사용자 동작에서만 요청한다.
- 이메일은 계정 확인, 비밀번호 재설정과 초대 같은 인증·계정 흐름에 사용한다.
- 제품 업무 알림을 이메일로 복제하지 않는다.

## 삭제와 보존

- 이슈와 프로젝트는 휴지통으로 이동하고 30일 뒤 영구 삭제를 예약한다.
- 복원하면 예약된 삭제 이벤트를 무효화하고 원래 워크스페이스 경계를 유지한다.
- 참조 중인 파일은 정리 작업이 삭제하지 않는다.
- 사용자 피드백, 인증 토큰, 세션, 전달 기록과 운영 감사 데이터는 각각 코드에 정의된 보존 조건을 따른다.
- 정확한 보존 기간과 정리 쿼리의 정본은 API·워커 코드이며 문서에 테이블별 목록을 복제하지 않는다.
