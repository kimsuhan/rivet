## 1. 문서 목적

이 문서는 [DESIGN.md](../../DESIGN.md)의 상위 디자인 방향과 와이어프레임 명세를 Rivet의 실제 제품 UI에 적용할 수 있는 디자인 토큰과 컴포넌트 규칙으로 구체화한다. 색상, 한국어 타이포그래피, 간격, 밀도, 상태 표현, 반응형, 모션과 접근성 기준을 정의해 디자인과 프론트엔드 구현이 같은 기준을 사용하도록 한다. Linear에서 가져온 시각적 근거와 원시 값은 [Linear 시각 디자인 분석](../reference/001.%20Linear%20시각%20디자인%20분석.md)에서 확인한다.

이 문서의 수치와 의미 토큰은 MVP UI의 기본 구현 계약이다. 특정 화면에 별도 규칙이 없다면 이 문서를 따르며, 새로운 시각 값을 임의로 추가하기 전에 기존 토큰의 조합으로 해결할 수 있는지 먼저 확인한다.

### 1.1 범위

- MVP 다크 모드 제품 UI의 원시 색상과 의미 색상
- 한국어 중심의 서체, 크기, 굵기와 행간
- 4px 기반 간격, 밀도, 모서리, 경계선과 깊이
- 데스크톱·모바일 레이아웃과 터치 기준
- 버튼, 입력, 선택기, 탐색, 목록, 보드, 배지와 오버레이
- 이슈 계층, 상태, 우선순위, 팀 작업 주요 행동과 작업 전달 표현
- Lexical WYSIWYG 편집기, Markdown 본문, 이미지 업로드, 첨부파일, 댓글, 활동과 피드백 상태
- 프로필 사진과 사용자 표시
- 포커스, 키보드, 명암비와 모션 접근성
- 한국어 UI 문구의 표기 원칙

### 1.2 이 문서에서 확정하지 않는 범위

- 구체적인 React 컴포넌트 파일 구조와 라이브러리 버전
- 최종 로고, 앱 아이콘과 브랜드 일러스트
- 마케팅·소개 페이지의 대형 타이포그래피와 레이아웃
- 라이트 모드와 사용자 지정 테마 — MVP 이후 범위
- 화면별 정보 구조와 기능 동작 — 선행 와이어프레임과 화면 명세를 따름
- 차트, 고급 분석 시각화와 저장된 보기 전용 컴포넌트

### 1.3 해석 우선순위

1. 제품 요구사항 정의서의 제품 정책과 인수 조건
2. 사용자 흐름 및 화면 명세서의 동작과 상태
3. 와이어프레임 명세서의 영역 구조와 반응형 규칙
4. `DESIGN.md`의 상위 디자인 방향
5. 이 문서의 시각 토큰과 컴포넌트 규칙
6. Linear 시각 디자인 분석의 벤치마킹 원본 값

상위 문서와 충돌하면 상위 문서를 따른다. `DESIGN.md`는 제품의 상위 방향, 이 문서는 정확한 구현 정본이다. 참고 자료의 마케팅 전용 값과 이 문서가 다르면 한국어 업무 화면에 맞게 조정한 이 문서의 값을 우선한다.

### 1.4 프론트엔드 구현 기준

- Tailwind CSS는 이 문서의 의미 토큰을 CSS 변수와 유틸리티에 연결하는 방식으로 사용한다. 화면 코드에서 원시 색상 값을 임의로 사용하지 않는다.
- shadcn/ui 컴포넌트 소스는 `apps/web/components/ui`에 두고 Rivet의 토큰, 밀도와 상태 규칙에 맞게 조정한다. MVP의 UI 소비자는 웹 앱 하나이므로 별도 `packages/ui`를 만들지 않는다.
- 제품 기능 컴포넌트는 shadcn/ui의 원시 구성 요소를 조합하되 이 문서의 의미와 접근성 계약을 유지한다.
- 제품 아이콘은 `lucide-react`로 통일하며 같은 의미에 여러 아이콘을 섞지 않는다.
- 이슈 설명, 작업 노트, 작업 전달과 댓글은 Lexical과 `@lexical/react`, `@lexical/markdown`을 사용한다. 화면은 WYSIWYG 편집과 미리보기를 제공하고 서버에는 Markdown 문자열만 저장한다.

## 2. 디자인 방향

### 2.1 핵심 원칙

1. **업무 기록이 주인공이다.** 장식보다 이슈 제목, 상태, 책임, 전달과 최근 변화를 먼저 읽게 한다.
2. **어둡지만 답답하지 않게 만든다.** 검은 면을 경계선으로 잘게 나누기보다 표면 단계와 간격으로 영역을 구분한다.
3. **촘촘하지만 조작 가능해야 한다.** 데스크톱에서는 32~44px 높이의 밀도 높은 행과 컨트롤을 사용하고, 모바일에서는 최소 44px 터치 영역을 보장한다.
4. **팀, 상태와 행동을 섞지 않는다.** 현재 상태는 아이콘·이름으로, 다음 행동은 동사형 버튼으로 구분하며 색상만으로 의미를 전달하지 않는다.
5. **강조색은 행동과 현재 맥락에만 쓴다.** 라벤더는 주요 행동, 선택, 포커스와 링크에 제한하고 넓은 배경 장식으로 사용하지 않는다.
6. **한국어를 원문으로 설계한다.** 영문 UI를 번역한 듯한 자간, 대문자 라벨과 불필요한 명사형 문구를 피한다.

### 2.2 Rivet의 시각적 서명: 전달 흐름

이슈와 역할별 팀 작업, 최초·추가 작업 전달은 Rivet의 핵심 흐름이다. 전달 카드는 얇은 좌측 Accent와 유형 텍스트로 구분하되 강한 배경색이나 데이터 모델별 중첩 카드로 표현하지 않는다. 실제로 생성된 백엔드·프론트 작업은 병렬 작업으로 같은 위계에 두고, 백엔드 뒤에 이어지는 프론트 작업은 전달 완료 뒤 실제 행으로 표시한다.

## 3. 토큰 구조와 사용 규칙

토큰은 원시 값과 의미 값을 분리한다.

| 계층 | 예시 | 규칙 |
| --- | --- | --- |
| 원시 토큰 | `palette.neutral.950`, `palette.lavender.500` | 실제 색상값을 보관하며 컴포넌트에서 직접 사용하지 않음 |
| 의미 토큰 | `color.bg.canvas`, `color.text.primary` | 제품에서 맡는 역할을 표현하며 컴포넌트가 사용 |
| 컴포넌트 토큰 | `button.primary.bg`, `issue.row.height` | 공통 의미 토큰과 크기 토큰의 조합으로 정의 |

구현체의 CSS 변수나 테마 키도 같은 의미 구조를 유지한다. 예를 들어 `color.bg.canvas`는 `--color-bg-canvas`로 대응할 수 있다. 화면 이름이나 특정 상태 이름을 토큰명에 넣지 않는다.

## 4. 색상 시스템

MVP는 다크 모드만 제공한다. 운영체제의 테마 설정과 관계없이 아래 다크 토큰을 사용하며, 라이트 모드 전환기나 미완성 라이트 토큰은 노출하지 않는다.

### 4.1 원시 팔레트

[Linear 시각 디자인 분석](../reference/001.%20Linear%20시각%20디자인%20분석.md)에서 추출한 어두운 표면과 라벤더를 Rivet의 용도에 맞게 조정해 사용한다.

| 토큰 | 값 | 용도 |
| --- | --- | --- |
| `palette.neutral.1000` | `#010102` | 최하위 캔버스 |
| `palette.neutral.950` | `#0F1011` | 1단계 표면 |
| `palette.neutral.900` | `#141516` | 2단계 표면 |
| `palette.neutral.850` | `#18191A` | 3단계 표면 |
| `palette.neutral.800` | `#191A1B` | 4단계 표면 |
| `palette.neutral.700` | `#23252A` | 기본 경계선 |
| `palette.neutral.600` | `#34343A` | 강한 경계선 |
| `palette.neutral.500` | `#62666D` | 비활성 텍스트·아이콘 |
| `palette.neutral.400` | `#8A8F98` | 보조 텍스트 |
| `palette.neutral.200` | `#D0D6E0` | 두 번째 본문 |
| `palette.neutral.50` | `#F7F8F8` | 기본 본문 |
| `palette.lavender.400` | `#828FFF` | hover·강조 아이콘 |
| `palette.lavender.500` | `#5E6AD2` | 기본 강조색 |
| `palette.lavender.600` | `#515CC2` | pressed |
| `palette.green.400` | `#45C46B` | 성공·완료 표시 |
| `palette.yellow.400` | `#E0B85A` | 경고 |
| `palette.orange.400` | `#E58A4A` | 따뜻한 라벨과 보조 강조 |
| `palette.red.400` | `#EF6A70` | 오류·긴급·파괴적 동작 |
| `palette.blue.400` | `#72A7F2` | 정보 |

라벤더 이외의 색은 작은 아이콘, 상태점, 배지 테두리와 피드백 영역에서만 사용한다. 카드 전체나 큰 화면 영역의 채움색으로 사용하지 않는다.

참고 자료의 성공색은 마케팅 화면보다 작은 제품 UI에서도 충분한 명암을 확보하도록 밝기를 높였다. 같은 방식으로 의미 색상은 벤치마킹 원본의 분위기보다 판독성과 상태 구분을 우선한다.

### 4.2 배경과 경계선 의미 토큰

| 토큰 | 값 | 적용 |
| --- | --- | --- |
| `color.bg.canvas` | `#010102` | 앱 전체와 목록 배경 |
| `color.bg.surface-1` | `#0F1011` | 주 탐색, 행 hover, 인증 카드 |
| `color.bg.surface-2` | `#141516` | 속성 패널, 선택된 탭, 카드 |
| `color.bg.surface-3` | `#18191A` | 드롭다운, 팝오버, 모달 |
| `color.bg.surface-4` | `#191A1B` | 중첩된 선택 영역과 강한 hover |
| `color.bg.overlay` | `rgba(0, 0, 0, 0.64)` | 모달 뒤 스크림 |
| `color.bg.selected` | `rgba(94, 106, 210, 0.16)` | 선택 행·메뉴·현재 결과 |
| `color.bg.accent-subtle` | `rgba(94, 106, 210, 0.12)` | 라벤더 보조 강조 |
| `color.border.default` | `#23252A` | 일반 구분선과 컨테이너 |
| `color.border.strong` | `#34343A` | 중첩 영역과 명시적 구분 |
| `color.border.accent` | `#5E6AD2` | 선택과 포커스 보조 |

표면 단계는 한 번에 한 단계씩 올린다. 예를 들어 캔버스 위의 속성 패널은 `surface-2`, 그 위에 열린 선택기는 `surface-3`을 사용한다. 같은 표면이 겹치면 경계선을 추가하고 더 밝은 임의 색상을 만들지 않는다.

### 4.3 텍스트와 아이콘 의미 토큰

| 토큰 | 값 | 적용 |
| --- | --- | --- |
| `color.text.primary` | `#F7F8F8` | 제목, 입력값, 주요 본문 |
| `color.text.secondary` | `#D0D6E0` | 본문, 보조 값, 미선택 메뉴 |
| `color.text.muted` | `#8A8F98` | 메타데이터, 설명, placeholder |
| `color.text.disabled` | `#62666D` | 비활성 컨트롤과 사용할 수 없는 값 |
| `color.text.on-accent` | `#FFFFFF` | 라벤더 버튼 위 텍스트 |
| `color.text.link` | `#9AA4FF` | 본문 링크와 API 명세 링크 |
| `color.icon.primary` | `#D0D6E0` | 의미 있는 기본 아이콘 |
| `color.icon.muted` | `#8A8F98` | 장식이 아닌 보조 아이콘 |

`color.text.disabled`는 작은 본문 정보에 사용하지 않는다. 명암비가 낮은 값은 비활성 상태처럼 조작할 수 없다는 사실이 이미 다른 방식으로 전달되는 경우에만 사용한다.

### 4.4 피드백 의미 토큰

| 의미 | 전경 | 배경 | 경계 | 사용 예 |
| --- | --- | --- | --- | --- |
| 정보 | `#9AA4FF` | `rgba(94, 106, 210, 0.12)` | `rgba(130, 143, 255, 0.32)` | 연결 끊김 복구, 도움말 |
| 성공 | `#62D783` | `rgba(69, 196, 107, 0.12)` | `rgba(69, 196, 107, 0.32)` | 저장 완료, 전달 완료 |
| 경고 | `#E8C675` | `rgba(224, 184, 90, 0.12)` | `rgba(224, 184, 90, 0.32)` | 영향 안내, 만료 임박 |
| 오류 | `#F38A8E` | `rgba(239, 106, 112, 0.12)` | `rgba(239, 106, 112, 0.36)` | 저장 실패, 입력 오류 |
| 위험 | `#F38A8E` | `rgba(239, 106, 112, 0.16)` | `rgba(239, 106, 112, 0.40)` | 삭제·비활성화 확인 |

피드백 영역은 아이콘, 짧은 제목, 설명과 해결 동작을 함께 제공한다. 성공 토스트를 제외한 오류와 충돌은 해당 필드 또는 목록 위치에 남긴다.

### 4.5 워크플로 상태

사용자에게는 한국어 상태 이름을 표시하고 내부 시스템 범주 이름은 노출하지 않는다. 상태 색은 시스템 범주가 정하며 사용자가 상태 이름을 바꿔도 의미 색은 유지한다.

| 시스템 범주 | 기본 상태 예 | 아이콘·점 색 | 보조 표현 |
| --- | --- | --- | --- |
| Backlog | 미분류, 보류 | `color.text.muted` | 빈 원 또는 보류 아이콘 |
| Unstarted | 할 일 | `color.text.primary` | 빈 원 |
| Started | 진행 중, 검토 | `color.feedback.info` | 진행 원 또는 상태 이름 |
| Completed | 완료 | `color.feedback.success` | 체크 아이콘과 완료 텍스트 |
| Canceled | 취소 | `color.text.muted` | 취소선 원과 취소 텍스트 |

`진행 중`과 `검토`처럼 같은 시스템 범주의 상태는 같은 의미 색을 사용할 수 있다. 반드시 상태 이름을 함께 표시해 둘을 구분한다.

이슈의 고정 상태도 팀 작업과 같은 워크플로 상태 아이콘 체계를 공유한다. `접수됨`은 회색 점선 원, `할 일`은 빈 원, `진행 중`과 `완료 확인`은 각각 1/3·2/3 파이, `완료`는 초록색 체크 원, `취소`는 회색 취소 원을 목록·상세·필터에서 일관되게 사용한다. `일시 중지`만 경고색 `CirclePause`로 구분한다. 팀별 사용자 정의 상태는 시스템 범주와 범주 내 순서에 따른 파이 진행도를 사용하고 사용자 정의 한국어 이름을 함께 표시한다.

프로젝트 상태는 다음 표현을 사용한다.

| 프로젝트 상태 | 색 | 표현 |
| --- | --- | --- |
| 계획 | `color.text.muted` | 상태점 + `계획` |
| 진행 중 | `palette.lavender.400` | 상태점 + `진행 중` |
| 완료 | `palette.green.400` | 체크 + `완료` |
| 취소 | `color.text.disabled` | 취소 아이콘 + `취소` |

### 4.6 우선순위

| 우선순위 | 의미 색 | 필수 표현 |
| --- | --- | --- |
| 없음 | `color.text.muted` | `Minus` + `없음` |
| 낮음 | `color.text.muted` | `SignalLow` + `낮음` |
| 보통 | `color.text.muted` | `SignalMedium` + `보통` |
| 높음 | `color.feedback.warning` | `SignalHigh` + `높음` |
| 긴급 | `palette.red.400` | `CircleAlert` + `긴급` |

우선순위 색은 아이콘에만 적용하고 텍스트는 `color.text.secondary`를 유지한다.

### 4.7 라벨 색

MVP 라벨 선택 팔레트는 라벤더, 파랑, 청록, 초록, 노랑, 주황, 빨강, 회색의 8개 계열로 제한한다.

| 라벨 계열 | 기준색 |
| --- | --- |
| 라벤더 | `#9A8CF2` |
| 파랑 | `#72A7F2` |
| 청록 | `#4BC7C7` |
| 초록 | `#45C46B` |
| 노랑 | `#E0B85A` |
| 주황 | `#E58A4A` |
| 빨강 | `#EF6A70` |
| 회색 | `#8A8F98` |

- 라벨 칩은 투명한 중립 배경과 기본 outline 경계를 사용하고 선택 색은 작은 색점에만 적용한다.
- 라벨 텍스트는 선택 색 자체가 아니라 `color.text.primary`를 사용한다.
- 같은 이름의 라벨은 화면마다 같은 색을 유지한다.
- 오류, 성공과 같은 시스템 피드백을 라벨 색만으로 추론하지 않는다.

## 5. 한국어 타이포그래피

### 5.1 서체 구성

| 역할 | 기본 스택 | 용도 |
| --- | --- | --- |
| 제품 산세리프 | `"Pretendard Variable", Pretendard, Inter, -apple-system, BlinkMacSystemFont, "Segoe UI", sans-serif` | 모든 한국어·영문 제품 UI |
| 고정폭 | `"Geist Mono", "SFMono-Regular", Consolas, "Liberation Mono", monospace` | 표시 ID, 단축키, 코드와 API 예시 |

MVP의 기본 제품 서체는 Pretendard Variable로 확정한다. Linear 전용 서체는 사용하지 않는다. Pretendard를 한국어와 영문 공통으로 사용해 혼합 문장에서 글자 높이와 굵기 차이를 줄인다. 폰트 로딩에 실패해도 시스템 산세리프로 기능과 레이아웃이 유지되어야 한다.

### 5.2 타입 스케일

| 토큰 | 크기 / 행간 | 굵기 | 자간 | 용도 |
| --- | --- | --- | --- | --- |
| `type.title-xl` | 28 / 38px | 600 | `-0.012em` | 인증·초기 설정의 화면 제목 |
| `type.title-lg` | 24 / 34px | 600 | `-0.010em` | 프로젝트·이슈 주요 제목 |
| `type.title-md` | 20 / 28px | 600 | `-0.008em` | 본문 헤더와 모달 제목 |
| `type.title-sm` | 16 / 24px | 600 | `-0.004em` | 카드·섹션 제목 |
| `type.body-lg` | 16 / 25px | 400 | `0` | 설명, Markdown 본문 |
| `type.body` | 14 / 21px | 400 | `0` | 기본 UI와 목록 제목 |
| `type.body-strong` | 14 / 21px | 500 | `0` | 선택 값과 강조 본문 |
| `type.body-sm` | 13 / 19px | 400 | `0` | 보조 정보와 활동 기록 |
| `type.label` | 12 / 17px | 500 | `0` | 필드 라벨, 배지, 탭 보조 정보 |
| `type.caption` | 12 / 17px | 400 | `0` | 시각, 개수, 도움말 |
| `type.button` | 14 / 20px | 500 | `0` | 버튼과 메뉴 항목 |
| `type.mono` | 12 / 18px | 500 | `0` | 표시 ID, 단축키와 짧은 코드 |

한국어는 20px 미만에서 음수 자간을 사용하지 않는다. 영문 대형 제목의 강한 음수 자간을 한국어에 적용하지 않으며, 한 줄 제목을 맞추기 위해 글자 폭을 강제로 줄이지 않는다.

### 5.3 사용 규칙

- 본문 화면 제목은 최대 `type.title-lg`를 사용한다. 마케팅용 40px 이상의 제목은 업무 셸에 사용하지 않는다.
- 이슈 제목은 상세에서 `title-lg`, 목록에서 `body` 또는 `body-strong`을 사용한다.
- 설명과 작업 전달 Markdown은 `body-lg`를 사용하고 본문 한 줄 길이를 데스크톱 기준 76자 안팎으로 제한한다.
- 버튼, 탭과 입력에서 600 이상 굵기를 사용하지 않는다.
- 숫자 정렬이 필요한 진행률, 일시와 개수에는 tabular numbers를 적용한다.
- 표시 ID는 고정폭 서체를 사용할 수 있지만 제목 전체나 상태 이름에는 사용하지 않는다.

## 6. 간격과 밀도

### 6.1 간격 토큰

| 토큰 | 값 | 대표 용도 |
| --- | ---: | --- |
| `space.0` | 0px | 간격 제거 |
| `space.0.5` | 2px | 아이콘 내부 보정 |
| `space.1` | 4px | 아이콘과 짧은 텍스트 |
| `space.1.5` | 6px | 촘촘한 배지 내부 |
| `space.2` | 8px | 컨트롤 내부, 인접 속성 |
| `space.3` | 12px | 입력 그룹, 행 내부 |
| `space.4` | 16px | 카드 내부, 섹션의 작은 간격 |
| `space.5` | 20px | 모바일 본문 좌우 여백 |
| `space.6` | 24px | 패널 내부, 본문 섹션 |
| `space.8` | 32px | 큰 섹션 구분 |
| `space.10` | 40px | 인증 카드 상하 여백 |
| `space.12` | 48px | 독립 화면의 큰 구분 |

기본 단위는 4px이다. 2px과 6px은 광학 보정과 작은 배지 내부에서만 사용한다. 업무 화면에 참고 자료의 마케팅 섹션 간격 96px을 적용하지 않는다.

### 6.2 기본 밀도

MVP 데스크톱의 기본 밀도는 일반 버튼·입력 36px, 목록 행 최소 44px로 확정한다.

| 요소 | 데스크톱 | 모바일·터치 |
| --- | ---: | ---: |
| 주 탐색 항목 | 32px | 44px |
| 일반 버튼·입력 | 36px | 44px |
| 작은 아이콘 버튼 | 시각 28px, hit area 32px | 44px |
| 필터 칩·배지 | 28px | 36~44px hit area |
| Compact 속성·필터 | 시각 28~32px, hit area 40px | 시각 28~32px, hit area 44px |
| Comfortable 속성 | 시각 34~38px, hit area 44px | 시각 34~38px, hit area 44px |
| 목록 행 | 최소 44px | 최소 72px 카드형 행 |
| 보드 카드 | 최소 88px | 미지원 |
| 설정 목록 행 | 최소 48px | 미지원 |

데스크톱 밀도는 사용자가 키보드와 마우스로 많은 작업을 빠르게 처리하는 것을 기준으로 한다. 화면을 촘촘하게 만들기 위해 글자 크기나 클릭 영역을 줄이지 않는다.

## 7. 모서리, 경계선과 깊이

### 7.1 모서리 토큰

| 토큰 | 값 | 적용 |
| --- | ---: | --- |
| `radius.xs` | 4px | 상태 배지, 단축키 키캡 |
| `radius.sm` | 6px | 필터 칩, 작은 메뉴 항목 |
| `radius.md` | 8px | 버튼, 입력, 선택기 호출부 |
| `radius.lg` | 12px | 카드, 팝오버, 모달 |
| `radius.xl` | 16px | 인증 카드와 큰 모바일 시트 |
| `radius.full` | 9999px | 아바타, 상태점, 실제 pill 배지 |

일반 버튼을 pill 형태로 만들지 않는다. 하나의 화면에서 중첩될수록 모서리를 계속 크게 만들지 않고 바깥 컨테이너만 더 큰 값을 사용한다.

### 7.2 경계선과 그림자

- 기본 경계선은 1px `color.border.default`다.
- 현재 선택과 키보드 포커스는 2px 라벤더 링으로 표시하고 레이아웃 크기에 포함하지 않는다.
- 카드와 패널은 배경 단계와 경계선으로 구분하며 일반 상태에 그림자를 사용하지 않는다.
- 드롭다운과 모달처럼 화면 위에 떠야 하는 오버레이만 `0 12px 32px rgba(0, 0, 0, 0.48)` 그림자를 사용할 수 있다.
- 여러 그림자, 안쪽 발광, 넓은 라벤더 글로우와 그라디언트는 사용하지 않는다.

### 7.3 레이어 토큰

| 토큰 | 값 | 적용 |
| --- | ---: | --- |
| `layer.base` | 0 | 일반 콘텐츠 |
| `layer.sticky` | 100 | 고정 헤더, 속성 패널과 모바일 탐색 |
| `layer.dropdown` | 300 | 선택기, 메뉴와 팝오버 |
| `layer.scrim` | 400 | 모달 배경 |
| `layer.modal` | 500 | 모달과 전체 화면 검색 |
| `layer.toast` | 600 | 전역 피드백 |
| `layer.tooltip` | 700 | 짧은 도움말 |

중첩 맥락 안에서 같은 토큰을 사용하고 화면별로 더 큰 숫자를 만들지 않는다. 모달 안 선택기는 모달보다 높은 전역 숫자를 추가하지 않고 모달 내부 레이어에서 해결한다.

## 8. 레이아웃과 반응형

### 8.1 중단점

| 구간 | 너비 | 적용 |
| --- | --- | --- |
| Mobile | 0~767px | 한 열, 고정 하단 탐색, 20px 본문 여백 |
| Mobile wide | 768~1023px | 한 열 모바일 규칙, 최대 720px 읽기 폭 |
| Desktop compact | 1024~1279px | 축소 가능한 주 탐색, 필터 일부 통합 |
| Desktop | 1280~1439px | 기본 240px 주 탐색과 전체 업무 화면 |
| Desktop wide | 1440px 이상 | 본문 확장, 읽기 영역은 최대 폭 유지 |

태블릿 전용 업무 레이아웃은 만들지 않는다. 1023px 이하에서 지원하지 않는 관리·보드·프로젝트 편집은 데스크톱 안내 화면을 제공한다.

### 8.2 셸과 콘텐츠 폭

| 영역 | 기준 |
| --- | --- |
| 펼친 주 탐색 | 240px |
| 축소 주 탐색 | 56px |
| 데스크톱 본문 헤더 | 최소 56px 높이 |
| 모바일 상단 바 | 52px |
| 모바일 하단 탐색 | 64px + safe area |
| 이슈 상세 우측 속성 패널 | 320px, 최소 288px |
| 폼 읽기 폭 | 최대 720px |
| Markdown 읽기 폭 | 최대 760px |
| 일반 모달 | 480~560px |
| 전역 이슈 만들기 모달 | 720px, 최대 화면 너비 `calc(100vw - 48px)` |
| 검색 오버레이 | 640px |
| 전역 이슈 목록 본문·도구 | 최대 1,440~1,600px, 앱 셸 안 가운데 정렬 |

넓은 화면에서도 Markdown 본문과 폼을 끝없이 늘리지 않는다. 일반 목록과 보드는 남은 너비를 사용하되 텍스트 열에 최소·최대 폭을 둔다. 전역 이슈 목록은 초광폭에서 별도 최대 폭을 적용해 제목부터 행동까지의 시선 이동 거리를 제한하고, 1,280~1,440px에서는 사용 가능한 본문 폭을 유지한다.

### 8.3 레이아웃 간격

- 데스크톱 본문 좌우 여백: compact 20px, 기본 24px, wide 32px
- 모바일 본문 좌우 여백: 20px, 360px 이하에서 16px
- 본문 헤더와 첫 콘텐츠 간격: 16px
- 큰 화면 섹션 간격: 32px
- 이슈 상세 주 본문과 속성 패널 사이: 1px 경계선, 추가 카드 외곽선 없음

## 9. 아이콘과 그래픽

- 제품 아이콘 라이브러리는 `lucide-react`를 사용한다.
- 아이콘은 둥근 끝을 가진 1.75px 전후의 단색 선형 스타일로 통일한다.
- 기본 크기는 16px, 주 탐색과 모바일 하단 탐색은 20px, 빈 상태 대표 아이콘은 최대 32px이다.
- 같은 의미에는 항상 같은 아이콘을 사용한다. 예를 들어 상태 변경, 담당자, 우선순위와 작업 전달의 아이콘을 화면마다 바꾸지 않는다.
- 아이콘만 있는 버튼에는 접근 가능한 한국어 이름과 tooltip을 제공한다.
- 장식용 이모지, 수제 SVG와 문자 기호를 제품 아이콘 대신 사용하지 않는다.
- 아바타 이미지가 없으면 이름의 첫 한 글자를 사용하고 임의의 캐릭터 일러스트를 생성하지 않는다.

`lucide-react`에 적절한 아이콘이 없는 경우에만 디자인 검토 후 제품 고유 SVG를 추가한다. 한 화면에서 다른 아이콘 라이브러리를 섞지 않는다.

## 10. 공통 상호작용 상태

모든 조작 가능한 컴포넌트는 다음 상태를 제공한다.

| 상태 | 시각·동작 계약 |
| --- | --- |
| 기본 | 현재 값과 조작 가능성을 텍스트 또는 형태로 표시 |
| hover | 배경을 한 단계 올리거나 `surface-1`을 사용, 레이아웃 이동 없음 |
| active | 배경을 한 단계 더 진하게 하고 1px 안쪽 반응 허용 |
| focus-visible | 2px `color.border.accent` 링 + 2px 여백, hover와 별개 |
| selected | `color.bg.selected`, 선택 아이콘 또는 `aria-selected` 상태 |
| open | hover보다 한 단계 분명한 표면, chevron 방향 또는 `aria-expanded` 전달 |
| disabled | 투명도만 낮추지 않고 이유를 인접 설명 또는 tooltip으로 제공 |
| loading | 현재 값·크기·포커스 가능성을 유지하고 진행 표시와 중복 입력 방지 |
| error | 해당 위치에 오류 문구와 해결 동작, 색상만 사용하지 않음 |

마우스 클릭에는 항상 포커스 링을 강제하지 않되 키보드 탐색에서는 숨기지 않는다.

### 11.1 버튼

| 변형 | 배경 | 텍스트·아이콘 | 사용 |
| --- | --- | --- | --- |
| Primary | `palette.lavender.500` | 흰색 | 화면 또는 모달의 한 개 주요 행동 |
| Secondary | `color.bg.surface-2` + 기본 경계 | primary | 취소가 아닌 보조 행동 |
| Tertiary | 투명 | secondary | 행 안 동작, 더보기 전 단계 |
| Danger | 오류 subtle 배경 + 오류 경계 | 오류 전경 | 삭제·비활성화의 최종 확인 |
| Ghost icon | 투명 | muted → primary | 닫기, 더보기, 접기 |

- 기본 높이는 36px, 작은 버튼은 32px, 인증 등 강조 동작은 40px이다.
- 모바일에서 직접 누르는 버튼은 최소 44px hit area를 가진다.
- 좌우 padding은 기본 14px, 아이콘과 텍스트 간격은 6px이다.
- 한 영역에 Primary 버튼은 원칙적으로 하나만 둔다.
- 로딩 중에는 원래 라벨을 유지하고 앞에 작은 진행 표시를 둔다.
- `확인`, `제출` 대신 `이슈 만들기`, `작업 시작`, `완료`, `프론트에 전달 후 완료`처럼 결과를 쓴다.

### 11.2 입력과 폼

| 요소 | 규칙 |
| --- | --- |
| 텍스트 입력 | 36px 높이, 8px 12px padding, `surface-1` 또는 상위 표면보다 한 단계 낮은 배경 |
| 여러 줄 입력 | 최소 96px, 세로 크기 조정 허용, 본문 타입 사용 |
| 라벨 | 입력 위 6px, `type.label`, 필수 여부는 문구와 보조 표시로 제공 |
| 도움말 | 입력 아래 6px, `type.caption`, muted |
| 오류 | 입력 경계와 오류 문구를 함께 표시, 입력값 유지 |
| 읽기 전용 | 일반 입력처럼 보이지 않게 값과 이유를 텍스트 행으로 표시 |

placeholder를 라벨 대신 사용하지 않는다. 저장 실패 시 전체 폼을 초기화하지 않고 첫 오류 필드로 포커스를 이동한다. 자동 저장 속성은 값 가까이에 저장 중·실패 상태를 표시한다.

### 11.3 선택기와 메뉴

- 호출부 바로 아래에 열고 공간이 없을 때만 위로 전환한다.
- 기본 너비는 호출부 이상, 최소 240px, 최대 360px이다.
- 검색이 필요한 선택기는 8개 이상의 후보 또는 사용자·프로젝트 선택에만 검색 입력을 제공한다.
- 메뉴 행은 데스크톱 36px, 모바일 전체 화면 선택기는 44px 이상이다.
- 현재 값은 체크와 선택 배경으로 함께 표시한다.
- 저장 실패 시 이전 값으로 복구하고 선택기를 강제로 닫지 않는다.
- Escape는 현재 선택기만 닫고 포커스를 호출부로 돌린다.

일반 폼과 생성·편집 모달의 Select 규격은 유지한다. 이슈 관련 목록·보드·프로젝트 작업 행에는 `inline compact`, 통합 이슈 상세의 이슈·현재 팀 작업 속성에는 `property comfortable` 표현을 적용한다.

이슈의 자동 상태는 Select가 아니라 아이콘·한국어 이름의 읽기 표현을 사용한다. hover·focus에서 편집 표면이나 chevron을 표시하지 않으며 `일시 중지`, `재개`, `취소`, `다시 열기`, `이슈 완료`는 별도의 이름 있는 행동으로 제공한다. 아래 선택기 규칙은 팀 작업 상태, 우선순위, 담당자와 라벨처럼 실제 편집 가능한 속성에 적용한다.

- 기본 배경·경계는 투명하고 hover에서 `surface-1`, 열린 상태에서 그보다 한 단계 분명한 표면을 사용한다.
- 보이는 높이는 28~32px, radius는 8px, 아이콘은 16px, 텍스트는 13px을 사용한다. 실제 hit area는 데스크톱 40px, 모바일 44px 이상이며 focus-visible에 2px 링을 표시한다.
- 전역 이슈 데스크톱 행에서 우선순위는 이슈 왼쪽의 좁은 독립 열, 이슈 상태는 이슈 다음의 독립 열에 둔다. 헤더와 모든 행은 같은 Grid 템플릿을 공유해 값이 바뀌어도 다른 열의 시작점을 이동시키지 않는다.
- 이슈 상태 아이콘은 팀 작업과 같은 워크플로 상태 아이콘 체계를 사용하며 `진행 중`과 `완료 확인`은 각각 1/3·2/3 파이로 구분한다. `일시 중지`는 경고색 `CirclePause`를 사용한다. 팀 작업은 시스템 범주 아이콘과 사용자 정의 이름을 사용한다.
- 팀 작업 상태 선택기의 트리거와 메뉴는 시스템 범주명이 아니라 실제 워크플로 상태명인 `미분류`, `할 일`, `진행 중`, `검토`, `완료`, `보류`, `취소`를 표시한다. 같은 범주의 상태를 같은 이름으로 반복하지 않는다.
- 우선순위 아이콘은 `없음 Minus`, `낮음 SignalLow`, `보통 SignalMedium`, `높음 SignalHigh`, `긴급 CircleAlert`를 사용한다.
- 트리거와 메뉴 항목은 같은 아이콘·한국어 이름을 사용하고 색상은 보조 수단으로만 쓴다. 아이콘만 남기는 중간 폭에서도 tooltip과 접근 가능한 전체 이름을 제공한다.
- 행 라벨 메뉴는 Portal로 렌더링하고 완전히 불투명한 `popover` 표면, 1px 이하의 약한 경계, 작은 shadow, 4~8px 간격과 8px 이상 viewport collision padding을 사용한다. 행 또는 뒤쪽 선택기가 레이어 위로 그려지지 않아야 하며 Escape와 키보드 선택으로 닫으면 호출부로 포커스를 돌린다. 바깥의 다른 조작 대상을 선택한 경우에는 해당 대상의 포커스를 유지한다.
- `property comfortable`은 같은 아이콘·이름·메뉴 순서와 상태 표면을 유지하되 텍스트 14px, 시각 높이 34~38px, 실제 hit area 44px을 사용한다. 전체 값 열을 채우는 입력 배경·두꺼운 경계를 두지 않고 상태·우선순위 트리거의 padding·radius를 같게 한다.
- 저장을 시작한 Select 트리거는 현재 값을 유지하고 포커스 복귀 대상에서 제거하지 않는다. `aria-disabled`와 이벤트 차단으로 재실행을 막고 `aria-busy` 또는 가까운 상태 문구로 진행을 전달한다.

### 11.4 탭, 필터와 배지

| 요소 | 형태 | 규칙 |
| --- | --- | --- |
| 기본 탭 | 밑줄 또는 선택 배경 | 전체·진행·백로그처럼 같은 수준의 보기 전환 |
| Segmented control | `surface-1` 안 선택된 `surface-3` | 목록·보드처럼 상호 배타적인 두 보기 |
| 필터 칩 | 시각 28~32px, radius 6~8px | 일반 칩은 28px·6px, Compact 활성 칩은 32px·8px, 필드 이름과 선택값, 해제 동작 |
| 상태 배지 | 작은 상태 아이콘 + 이름 | 배경보다 텍스트 판독 우선 |
| 역할 배지 | 역할 이름 + 팀 이름 | 역할과 팀을 서로 추론하지 않음 |
| 팀 작업 주요 행동 | `작업 시작`, `완료`, `완료됨` | 상태 배지와 분리하고 같은 위치에서 전환 |
| 빠른 필터 | 한 줄 탭 또는 선택 배경 | 전체·검토 필요·담당 필요·진행 중·완료 확인·완료 중 한 개만 선택하고 각 필터의 전체 결과 수를 텍스트와 보조 기술에 전달 |

배지는 한 행에서 최대 두 줄로 늘어나지 않는다. 공간이 부족하면 우선순위가 낮은 라벨을 `+2`처럼 축약하되 상태, 현재 주요 행동과 역할은 숨기지 않는다.

전역 이슈 빠른 필터 수치는 현재 검색·세부 필터를 반영하되 선택된 빠른 필터 조건만 제외한 서버 전체 집계를 사용한다. 0건도 선택할 수 있고 muted 텍스트로 강조만 낮추며, 선택 항목은 색상 외에 밑줄·선택 배경 또는 굵기와 `aria-selected`를 함께 사용한다.

전역 이슈 도구의 세부 필터·정렬은 28~32px의 compact 시각 높이와 13px 텍스트, 14~16px 아이콘, 8px radius를 공유한다. 기본 경계와 배경은 투명하게 두고 hover에는 약한 표면, open에는 한 단계 더 분명한 표면, focus-visible에는 2px 링을 표시한다. 활성 필터 칩만 기본 상태에서도 표면을 유지한다. 실제 hit area는 데스크톱 40px·모바일 44px 이상으로 유지하되 활성 칩의 표면은 32px 안에 둔다. 활성 필터 수는 `세부 필터 2`처럼 이름과 함께 전달하며 정렬 방향을 아이콘만 표시할 때도 tooltip과 접근 가능한 이름에 전체 방향을 남긴다.

내 작업·팀 목록·보드의 필터와 정렬도 같은 Compact 규칙을 사용한다. 필터 메뉴는 Portal의 불투명 팝오버로 열고 Escape나 키보드 선택으로 닫은 뒤 호출부로 포커스를 복원한다. 바깥의 다른 조작 대상을 선택한 경우에는 해당 대상의 포커스를 유지한다. 활성 필터만 이름·값·개별 제거가 있는 칩으로 강조하고 전체 초기화를 함께 제공한다.

내 작업은 작업 코드·이슈 표시 ID·제목·프로젝트 이름 검색, 상태·프로젝트 필터와 정렬만 기본 도구로 제공한다. 기본 정렬 이름은 `실행 순서`이며 `STARTED → UNSTARTED → BACKLOG`, 상위 이슈 우선순위, 워크플로 표시 순서, 최근 수정과 불변 ID 순서를 사용한다. `최근 수정`은 선택 가능한 정렬 기준으로만 제공하고 기본 행 열에는 표시하지 않는다.

이슈 헤더와 프로젝트 기본 목록에는 별도 유형 배지를 사용하지 않는다. 팀 작업은 팀·역할 맥락이 필요한 목록과 통합 상세의 작업 영역에서만 구분한다.

이슈 상세의 `업무`, `활동` 탭은 제목과 본문 사이에 line 변형으로 표시한다. 두 항목은 모바일에서도 한 줄 너비 안에서 같은 크기의 터치 영역을 사용하고 가로 스크롤에 의존하지 않는다. 현재 탭은 텍스트 색뿐 아니라 밑줄 또는 선택 배경과 `aria-selected`로 구분한다. 전달 정보는 `업무` 탭의 작업 내용보다 강한 표면이나 색을 사용하지 않는다.

### 11.5 아바타와 사용자 표시

- 크기: inline 20px, 목록 24px, 선택기 28px, 프로필 40px
- 이미지가 없으면 이름의 첫 글자와 중립 표면을 사용한다.
- 프로필 설정 미리보기는 96px을 사용하고 실제 제품 표시에는 원형 `object-fit: cover`를 적용한다.
- 새 사진 업로드 중에는 기존 사진을 유지하고 미리보기 위에 반투명 진행 레이어를 표시한다. 실패 시 기존 사진으로 복귀한다.
- 담당자 선택에서는 이름을 기본, 이메일과 멤버 상태를 보조로 표시한다.
- 아바타만으로 사용자를 구분해야 하는 상황을 만들지 않는다.
- 비활성 멤버는 이름과 `비활성` 상태를 함께 표시한다.

### 11.6 프로젝트 로고

- 크기: 인라인·사이드바 16px, 선택기·메타 20~24px, 목록 카드 32px, 상세 제목·폼 미리보기 40px
- 이미지는 정사각형 `object-fit: cover`와 4~8px 모서리를 사용하며 사용자 아바타처럼 원형으로 표시하지 않는다.
- 로고가 없거나 이미지를 불러오지 못하면 중립 표면의 폴더 아이콘을 표시한다. 프로젝트 이름은 항상 함께 제공해 로고만으로 구분하게 하지 않는다.
- 선택한 JPG·PNG·WebP는 가운데를 기준으로 정사각형 최적화한다. 업로드 중·실패 상태는 로고 입력 가까이에 표시하고 저장 중인 다른 필드 값을 유지한다.

## 12. 업무 컴포넌트

### 12.1 이슈와 팀 작업 행

| 속성 | 규칙 |
| --- | --- |
| 기본 높이 | 44px, 보조 맥락이 있으면 56~64px |
| 시작점 | 표시 ID → 제목 |
| 후속 열 | 화면 목적에 따라 프로젝트, 팀·역할, 상태, 담당자, 우선순위와 행동을 선택 |
| hover | 행 전체 `surface-1`, 편집 가능한 셀은 독립 hover |
| selected | `color.bg.selected`와 선택 상태 제공 |
| keyboard | 행 열기와 속성 셀 편집의 포커스를 분리 |

제목은 한 줄 말줄임을 기본으로 하되 tooltip에서 전체 제목을 확인할 수 있다. 표시 ID는 줄어들지 않고 제목이 남은 공간을 사용한다.

내 작업과 팀 목록은 전역 이슈 목록과 같은 행 surface·구분선·Compact 속성 트리거를 사용하되 화면 책임에 맞는 정보만 선택한다. 내 작업은 `우선순위`, `작업`, `프로젝트`, `상태`, 이름 없는 우측 행동의 다섯 영역을 사용한다. 우선순위는 상위 이슈의 읽기 표현, 작업은 가장 강한 이슈 제목과 팀 작업 표시 ID·이슈 표시 ID·역할 보조 줄, 프로젝트는 이름과 최대 두 개의 상위 이슈 라벨·나머지 `+N`을 표시한다. 실제 팀 이름은 내 작업 기본 행에서 반복하지 않는다.

내 작업은 개인 실행 큐이므로 담당자, 실제 팀 이름과 최근 수정 시각을 기본 행에서 제외한다. 라벨은 상위 이슈가 소유한 읽기 전용 값으로 프로젝트 아래에만 표시한다. 팀 작업 상태는 Compact 인라인 트리거로 편집하고 상위 이슈 우선순위는 같은 밀도의 읽기 표현으로 표시한다. 우측 행동은 열 헤더 없이 `작업 시작` 또는 `완료` 한 개만 조건부로 표시하며 실행 가능한 행동이 없을 때 비활성 버튼을 남기지 않는다. `COMPLETED`, `CANCELED` 작업은 내 작업 목록에 표시하지 않는다.

팀 목록은 여러 담당자의 작업을 운영하므로 표시 ID·상위 이슈 제목, 프로젝트·라벨, 역할, 상태, 담당자, 우선순위와 최근 수정을 계속 제공할 수 있다. 라벨이 없으면 편집기를 상시 노출하지 않고 두 목록 모두 행 전체 링크와 내부 컨트롤을 중첩하지 않는다.

내 작업 행이 여는 상세는 목록과 같은 개인 실행 문맥을 유지한다. `/my-issues/{teamWorkIdentifier}`에서는 `내 작업` 내비게이션과 복귀 링크를 활성화하고 팀 작업 표시 ID·상태·현재 행동을 상단 실행 영역에서 먼저 읽게 한다. 상위 이슈 제목은 작업 목적을 설명하는 가장 강한 콘텐츠 제목으로 두며 이슈 표시 ID·프로젝트·역할·라벨은 보조 맥락으로 배치한다. `/issues/{issueIdentifier}`에서는 `이슈` 내비게이션과 이슈 표시 ID·제목이 상단 기준이 된다. `/projects/{projectId}/issues/{issueIdentifier}`에서는 `프로젝트`와 해당 프로젝트 항목을 활성화하고 프로젝트 이름을 복귀 링크로 표시한다. 개인 저장된 보기에서 연 상세는 상위 메뉴와 함께 해당 하위 보기를 활성화하되 새로운 강조 표면을 추가하지 않고 기존 점·텍스트 상태를 유지한다.

세 상세 진입 모드는 같은 `업무`·`활동` 탭, 설명·첨부파일·댓글과 Comfortable 속성 컴포넌트를 재사용한다. 내 작업 상세만을 위한 카드, 중첩 패널이나 별도 색상 체계를 추가하지 않고 활성 내비게이션, 뒤로 가기 문구와 상단 정보 위계만 진입 맥락에 따라 바꾼다.

전역 이슈 행은 `우선순위`, `이슈`, `상태`, `현재 작업`, `진행`, `업데이트`, `다음 행동` 일곱 영역과 72~84px 기본 높이를 사용한다. 우선순위는 이슈 왼쪽의 좁은 독립 열에 두고, 이슈 셀은 표시 ID·제목을 첫 줄, 프로젝트·최대 두 개 라벨·나머지 `+N`을 같은 보조 줄로 묶는다. 상태는 이슈 다음의 독립 읽기 전용 열에 둔다. 라벨이 없으면 빈 `라벨` 선택기를 평상시에 노출하지 않고 hover·focus 보조 동작이나 더보기에서 편집한다. 만든 사람은 기본 열에서 제거한다.

제목은 `body-strong`과 primary 텍스트로 가장 먼저 읽히게 한다. 자동 계산된 이슈 상태는 아이콘·이름만 보이는 투명 읽기 표현, 우선순위는 아이콘·한국어 이름을 함께 보이는 같은 밀도의 인라인 선택기로 표시한다. 우선순위와 상태 열은 독립된 고정 폭을 사용하며 우선순위 chevron은 셀의 같은 끝점에 정렬한다. 현재 작업은 역할·실제 팀·진행 상태·담당 필요 수를 제목과 보조 정보 최대 두 줄로 표시한다. 병렬 역할이 많으면 `{첫 역할} 외 N개`로 줄이고 전체 값은 tooltip이나 상세에서 확인한다. 유형 배지와 `차단됨`을 단독 배지로 사용하지 않는다. 진행은 숫자, 백분율과 낮은 명암의 짧은 4px 막대를 가까이 두고 업데이트는 상대 시각만 표시한다.

행 오른쪽에는 현재 조건에 따른 주요 행동 한 개만 두고 나머지는 더보기로 이동한다. 주요 행동은 30~32px의 시각 표면과 13px 텍스트를 쓰되 실제 hit area는 별도로 유지한다. 검토·담당·완료 확인처럼 결정이 필요한 동작만 낮은 채도의 Secondary 또는 조건부 Primary, 단순 이동은 Ghost 또는 행 선택, 더보기는 28~32px의 배경 없는 Ghost icon을 사용한다. 가능한 현재 사용자 역할이 하나면 보이는 문구를 줄일 수 있지만 tooltip·접근 가능한 이름에 실제 역할을 남기고, 여러 역할이면 `우리 팀에서 시작`을 사용한다. 행 hover·focus에서는 행동 대비를 한 단계 높이되 모든 행에 Primary를 반복하지 않는다.

전역 이슈 목록은 표 헤더 아래 한 번과 행 사이의 낮은 명암 구분선만 사용한다. 전체 행 hover는 `surface-1` 한 단계로 표시하며 선택기·도구·더보기에 상시 경계와 강한 배경을 함께 쓰지 않는다. 실제 레이어 경계가 필요한 라벨 팝오버에만 약한 경계와 shadow를 함께 사용한다.

전역 이슈 행의 주요 행동·더보기·우선순위 조작은 데스크톱에서도 실제 hit area를 최소 40px로 유지하고 모바일에서는 44px로 늘린다. 이슈 상태는 조작 대상으로 가장하지 않는다. 현재 조건에서 실행할 수 없는 행동은 비활성 버튼만 남기지 않고 그 자리에 이유 또는 해결 경로를 텍스트로 표시한다.

행 빈 영역을 덮는 독립 상세 링크와 내부 컨트롤을 같은 포커스 순서에 두되 링크 안에 버튼이나 선택기를 중첩하지 않는다. 상태·우선순위·라벨·주요 행동·더보기·진행 링크의 클릭과 Enter·Space는 행 이동으로 전파하지 않는다.

모바일 전역 이슈 카드는 최소 72px을 유지하되 표시 ID·제목, 프로젝트·라벨, 읽기 전용 상태·편집 가능한 우선순위, 현재 작업·담당 필요, 진행률, 업데이트, 주요 행동 순으로 읽힌다. 데스크톱 열을 축소해 가로 스크롤시키지 않고 메타 정보를 세로로 재배치하며 직접 조작하는 버튼과 선택기는 최소 44px hit area를 가진다.

### 12.2 이슈 카드와 보드 열

- 보드 열 너비는 280px, compact에서 264px까지 줄일 수 있다.
- 보드 열 사이 간격은 12px, 열 헤더는 40px 이상이다.
- 카드 padding은 12px, 카드 간 간격은 8px이다.
- 첫 줄에는 표시 ID와 상태, 둘째 줄에는 최대 두 줄 제목을 둔다.
- 담당자, 우선순위, 역할, 라벨과 현재 주요 행동은 하단 메타 영역에 배치한다.
- 상태·담당자·우선순위와 라벨 메뉴는 목록과 같은 Compact 아이콘·이름·순서를 사용한다. 좁은 카드에서는 우선순위 이름을 숨길 수 있지만 tooltip과 접근 가능한 전체 이름은 유지한다.
- drag 중인 카드는 2px 라벤더 경계와 약한 오버레이 그림자만 사용한다.
- 드래그할 수 없어도 상태 선택기로 같은 변경을 할 수 있어야 한다.

### 12.3 이슈와 작업 흐름

- 이슈 행은 굵은 제목, 현재 역할·상태와 실제 팀 작업 완료 수를 표시한다.
- 팀 작업이 없으면 진행률 숫자보다 분석 안내와 `작업 시작`을 먼저 표시한다.
- 순서 관계가 없는 미완료 작업은 `현재 작업` 아래 같은 대비와 카드 위계로 배치하고 세로 연결선을 사용하지 않는다.
- 팀 작업 간 연결선은 사용하지 않고, 백엔드 전달은 최신 전달 Accent와 `새 전달` 표기로 구분한다.
- 팀 작업은 부모 이슈 안에서 20px 들여쓰되 병렬·순차 여부를 텍스트와 구조로 함께 표현한다.
- 역할별 순서는 `백엔드`, `웹 프론트`, `앱 프론트`이며 프로젝트에 없는 역할은 생략한다.
- 작업 흐름을 접어도 현재 역할, 완료 수, 전체 수와 진행률은 남기고 펼침 상태를 아이콘 방향, `aria-expanded`와 자식 영역 존재로 함께 표현한다.
- 전역 이슈의 작업 요약은 소속 팀 작업 전체를 펼치지 않고 현재 역할·팀, 미할당 수, 완료·취소 수와 취소 제외 진행률만 표시한다.

### 12.4 진행률

- 기본 높이 4px, 큰 프로젝트 요약에서 6px을 사용한다.
- 트랙은 `surface-4`, 완료 영역은 라벤더를 기본으로 하고 100%일 때만 성공색을 사용할 수 있다.
- `완료 3 / 7 · 43%`처럼 완료 수, 전체 수와 비율을 함께 제공한다.
- 목록·프로젝트 행의 짧은 진행률은 같은 Progress primitive와 4px 트랙을 사용하고 `aria-valuenow`와 전체 한국어 요약을 제공한다.
- 프로젝트 상태를 진행률 색이나 값으로 추론하지 않는다.
- 분모가 0인 이슈는 숫자만 있는 `0/0 완료` 대신 `아직 시작된 팀 작업이 없습니다`, 분석 안내와 `작업 시작`을 표시한다. 프로젝트 요약처럼 행동이 없는 집계 맥락만 빈 트랙과 `작업 없음 · 0%`를 사용할 수 있다.

### 12.5 팀 작업 주요 행동과 작업 전달

- 수동 작업 순서, 선행·후행 관계와 관계 그래프는 표시하지 않는다.
- 팀 작업 실행 영역에는 `시작 전 → 작업 시작`, `진행 중 → 완료`, `완료 → 완료됨`을 한 개의 주요 행동 위치에서 전환해 표시한다.
- 기본 `백로그`에 담당자가 없으면 주요 버튼 대신 `담당자를 선택해 주세요` 안내를 제공한다. 담당자를 선택하면 `시작 전`으로 전환되고 `작업 시작`이 나타난다.
- 예외 상태 변경은 보조 `상태 변경` 메뉴로 분리하고 실제 워크플로 상태명을 표시한다.
- 최초 전달과 추가 전달은 같은 카드 구조를 사용하고 상단의 유형·작성자·시각으로 구분한다.
- 작업 전달 카드는 `surface-2`, 1px 기본 경계, 12px radius와 16px padding을 사용한다.
- 작업 전달은 업무 탭에서 전체 이력을 최신순 카드로 기본 제공하고, 각 카드의 원문을 별도 펼치기 조작 없이 바로 표시한다.
- API 명세 링크는 본문 링크색과 외부 링크 아이콘을 사용한다.
- 기존 전달에는 편집 아이콘을 표시하지 않는다.
- 대상 역할 선택은 웹·앱이 모두 있을 때 체크 목록으로 표시하고, 한 역할만 있으면 선택기가 아닌 읽기 전용 값으로 표시한다. 최소 한 역할 오류는 목록 가까이에 둔다.
- 백엔드 완료 모달은 `이 작업만 완료`와 `프론트에 전달 후 완료`를 먼저 선택하게 하고 후자를 선택한 경우에만 전달 입력을 펼친다.
- 모달 하단은 선택에 따라 `완료` 또는 `프론트에 전달 후 완료` 한 개만 Primary로 두고 취소는 Secondary로 둔다.

### 12.6 댓글과 활동

- 댓글은 `업무`, 시스템 이력은 `활동` 탭에 분리하고 같은 세로 시간축에 섞지 않는다.
- 댓글 작성자·시각·수정 여부는 한 줄, 본문은 그 아래에 둔다.
- 댓글 작성기는 실제 댓글 목록 바로 아래에 두고 댓글이 없어도 과도한 빈 공간을 만들지 않는다.
- 활동은 작업 카드보다 낮은 표면 대비와 작은 간격을 사용하고 아이콘, 행위자, 변경 내용과 시각을 한 줄 중심으로 표시한다.
- 작업 전달 활동은 원문 카드 대신 한 줄 요약과 `전달 내용 보기`를 제공한다.
- 삭제된 댓글은 같은 높이의 빈 카드가 아니라 원래 위치에 한 줄 상태로 남긴다.

### 12.7 첨부파일과 업로드

**파일 업로드 항목**

- 최소 구성은 파일 아이콘 또는 40px 이미지 썸네일, 파일명, 크기, 상태와 제거 동작이다.
- 업로드 중에는 파일 행 전체를 막지 않고 파일명 아래 2px 진행 막대와 `업로드 중` 텍스트를 표시한다.
- 실패 상태는 오류 아이콘·이유·`다시 시도`·`제거`를 같은 행에 두고 성공한 다른 파일은 유지한다.
- 25MB 초과 파일은 업로드 행을 만들지 않고 파일 선택 영역 가까이에 제한을 안내한다.
- 이미지 최적화 중에는 `이미지 최적화 중`을 업로드와 구분해 표시한다.

**완료된 첨부파일 항목**

- 파일명, 형식, 크기, 업로더와 업로드 시각을 표시하고 행 전체 열기와 명시적인 다운로드 동작을 제공한다.
- 검증된 이미지는 48px 정사각 썸네일을 사용하고 그 밖의 파일은 `lucide-react` 파일 형식 아이콘을 사용한다.
- 긴 파일명은 중간 생략을 사용하되 tooltip 또는 접근 가능한 이름에는 전체 값을 유지한다.
- 제거는 행의 더보기 메뉴에 두고 확인 없이 연결을 해제하되 짧은 되돌리기 알림을 제공한다.

### 12.8 상세 화면 정보 위계

- `업무` 탭은 현재 행동·상태·담당자, 최신 작업 전달, 설명·첨부파일, 작업 노트와 댓글 순서로 읽히게 한다.
- `업무` 탭의 작업 전달 영역은 최초·추가 전달의 전체 이력을 최신순 카드로 표시한다. 카드 안에서 현재 작업 기준 `보냄`, `받음`, `다른 작업`을 구분하며 빈 관계 섹션이나 별도 전체 보기 상태를 만들지 않는다.
- 우측 속성 패널의 편집 속성 아래에는 경계선으로 구분한 `정보` 그룹을 두고 만든 사람, 만든 날짜와 마지막 수정 시각을 기본으로 펼친다.
- 우측 속성은 `80~96px 이름 + 나머지 값`의 동일한 grid와 일정한 행 간격을 사용한다. 상태·담당자·우선순위는 Comfortable 트리거, 프로젝트·팀·역할은 chevron과 입력 경계가 없는 읽기 전용 값으로 표시한다.
- 라벨은 쉼표 문자열 입력 대신 공통 색점 칩을 최대 두 개 표시하고 나머지는 `+N`으로 요약한다. 값이 없으면 `라벨 없음`과 작은 추가 동작을 제공한다.
- `정보` 그룹은 속성과 사이의 구분선 한 번만 사용하고 값 크기·굵기를 편집 속성보다 높이지 않는다.
- 팀 작업의 상위 이슈는 제목 주변의 짧은 breadcrumb와 `연결` 탭에서 접근하며 본문에 큰 상위 이슈 카드를 반복하지 않는다.
- 같은 제목, 상태, 전달 원문과 읽기 전용 정보를 여러 탭에서 같은 위계로 반복하지 않는다.
- 주요 작업이 넓은 상하 여백 때문에 첫 화면 아래로 밀리지 않도록 탭 패널의 첫 섹션은 추가 상단 여백 없이 시작한다.

## 13. 탐색과 오버레이

### 13.1 데스크톱 주 탐색

- 주 메뉴는 `개인` 구역의 `알림함`, `내 작업`과 `워크스페이스` 구역의 `이슈`, `프로젝트`로 나눈다. 팀 탐색은 이 묶음 아래에 둔다.
- 펼친 너비 240px, 축소 너비 56px, `surface-1` 배경과 오른쪽 1px 경계선을 사용한다.
- 현재 워크스페이스 메뉴, 빠른 동작, 주 메뉴, 팀과 사용자 영역을 12~16px 간격의 묶음으로 나눈다.
- 항목 높이는 32px, 좌우 padding 8px, 현재 위치는 선택 배경과 2px 왼쪽 라벤더 표시를 함께 사용한다.
- 현재 워크스페이스 메뉴는 전환 기능 없이 이름과 식별자만 보여 준다. 권한에 맞는 설정 진입점은 사이드바 하단에 항상 노출하고 설정 화면에서는 현재 위치를 표시한다.
- 팀 탐색은 소속 활성 팀만 바로 표시하고 `모든 팀 보기`에서 활성 팀을 `내 팀`과 `다른 팀`으로 구분한다.
- 축소 상태에서도 tooltip 없이 의미를 추론하기 어려운 팀은 키를 표시한다.
- hover만으로 일시 확장하지 않는다.

### 13.2 모바일 탐색

- 하단 탐색은 `내 작업`, `알림함`, `이슈`, `프로젝트`, `팀` 다섯 항목을 균등 분할하고 아이콘과 한국어 이름을 함께 표시한다. 검색은 상단 앱 바의 전역 동작으로 이동한다.
- 상단 앱 바 오른쪽의 28px 프로필 사진 또는 첫 글자 아바타는 사용자 메뉴 호출부로 사용한다.
- 현재 항목은 아이콘·텍스트 색과 보조 기술 상태로 구분하며 pill 배경을 반복하지 않는다.
- 읽지 않은 알림 수는 최대 `99+`로 표시하고 현재 항목 표시와 겹치지 않는다.
- safe area를 포함하고 본문 마지막 내용이 탐색 뒤에 가려지지 않게 한다.

### 13.3 모달, 팝오버와 시트

| 유형 | 규칙 |
| --- | --- |
| 확인 모달 | 480px, 제목·영향·해결 경로·하단 동작 |
| 폼 모달 | 560px, 긴 경우 본문만 스크롤 |
| 이슈 만들기 | 720px, 설명 편집기 기본 펼침, 하단 동작 고정 |
| 검색 오버레이 | 640px, 입력 고정, 결과만 스크롤 |
| 작업 시작·담당자 지정 | 560px, 역할과 팀·담당자·현재 작업을 행 단위로 표시하고 하단 동작 고정 |
| 내가 맡기·우리 팀에서 시작 | 가능한 역할이나 후보가 여러 개일 때만 선택 목록 표시 |
| 이슈 완료 | 완료 역할·팀 작업과 제외된 취소 작업 요약 |
| 템플릿 덮어쓰기 | 실제로 바뀌는 필드 이름만 요약하고 취소를 안전한 기본 동작으로 제공 |
| 모바일 시트 | 화면 아래에서 열고 최대 높이 88vh, 드래그 없이 닫기 제공 |

- 모달을 열면 배경 스크롤을 잠그고 첫 유효 입력 또는 제목에 포커스한다.
- 닫을 때 포커스를 호출부로 복원한다.
- 바깥 클릭으로 닫을 때 입력 손실 가능성이 있으면 확인을 거친다.
- 모달을 겹쳐 두 개 이상 열지 않고 선택기는 현재 모달 내부 레이어로 처리한다.
- 템플릿 선택은 적용 성공 전까지 이슈 입력을 바꾸지 않는다. 보관·권한·version 오류는 destructive 색만으로 구분하지 않고 입력 보존과 재선택 동작을 가까운 인라인 안내로 제공한다.
- 템플릿 적용 요청 중에는 덮어쓰기 확인 동작과 설명·우선순위·라벨·프로젝트·최초 역할 입력을 잠그고 진행 상태를 표시한다. 성공 응답만 입력에 반영하며 일반 실패는 확인창을 닫고 기존 입력과 재시도 동작을 보존한다.

## 14. Lexical 편집기와 Markdown 본문

### 14.1 편집기

- 제목은 한 줄 요약이고 설명이 실제 작업 맥락이므로 이슈 생성 모달의 설명 편집기는 기본으로 펼친다.
- `편집`과 `미리보기` 탭을 제공하고 Markdown 원문 편집과 데스크톱 동시 분할 편집은 MVP에 넣지 않는다.
- 편집 탭은 Lexical 기반 WYSIWYG 화면을 사용하고 Markdown 단축 입력을 지원한다.
- 편집 영역 최소 높이는 생성 180px, 상세 240px, 댓글 96px이다.
- 툴바는 shadcn/ui의 버튼·토글·툴팁과 `lucide-react` 아이콘으로 구성하고 굵게, 기울임, 취소선, 링크, 코드, 목록과 인용의 핵심 동작만 제공한다.
- 이미지 선택 동작을 툴바에 추가하고 Command/Ctrl+V로 붙여넣은 이미지도 같은 업로드 항목을 사용한다.
- 붙여넣은 URL은 링크로 만들 수 있으나 자동 임베드는 제공하지 않는다.
- 표, 임의 HTML 입력과 원격 이미지 자동 가져오기는 제공하지 않는다.
- 업로드 중 이미지는 본문 삽입 위치에 최대 높이 240px 미리보기, 진행 표시와 파일명을 표시한다.
- 업로드 실패 이미지는 깨진 이미지 아이콘 대신 미리보기·실패 문구·다시 시도·제거를 유지한다.
- 저장 실패 시 입력 내용과 현재 탭을 유지한다.

### 14.2 렌더링

- 기본 본문은 `type.body-lg`, 최대 읽기 폭 760px을 사용한다.
- 미리보기와 저장 후 조회는 Lexical 상태를 Markdown으로 변환한 뒤 같은 렌더링 규칙으로 표시한다.
- 제목 간격은 위 24px, 아래 8px이며 첫 제목의 위 간격은 제거한다.
- 문단과 목록 간격은 12px, 목록 들여쓰기는 20px이다.
- inline code는 `surface-2`, 고정폭 12px, 4px radius를 사용한다.
- code block은 `surface-1`, 12px padding, 가로 스크롤과 복사 동작을 제공한다.
- 링크는 색상과 밑줄 또는 hover 밑줄을 함께 사용한다.
- 본문 이미지는 읽기 폭 안에서 최대 너비 100%, 원본 비율과 8px radius를 유지한다. 대체 텍스트가 없으면 장식 이미지로 단정하지 않고 작성 시 입력을 안내한다.
- 파일 접근 권한이 없거나 실제 파일이 누락되면 빈 `img` 대신 `이미지를 불러올 수 없습니다` 상태와 파일명을 표시한다.

## 15. 피드백과 시스템 상태

### 15.1 Toast

- 짧은 성공, 되돌릴 수 있는 완료와 전역 알림에만 사용한다.
- 데스크톱 오른쪽 아래, 모바일 하단 탐색 위에 최대 3개까지 쌓는다.
- 기본 유지 시간은 4초이며 중요한 오류는 자동으로 사라지지 않는다.
- 필드 오류, 권한 부족과 충돌을 toast 하나로 대체하지 않는다.

### 15.2 상태 막대와 인라인 피드백

- 연결 끊김은 본문 헤더 아래 전체 너비 상태 막대로 표시한다.
- 저장 중은 변경한 속성 옆 12~14px 진행 표시를 사용한다.
- 저장 실패는 원래 값 복원, 오류 문구와 `다시 시도`를 같은 위치에 둔다.
- 파일 최적화·업로드 상태는 해당 파일 또는 이미지 위치에 표시하고 전체 화면 로딩으로 대체하지 않는다.
- 충돌은 서버 최신 값과 내 변경을 나란히 보여 주고 `내 변경 다시 적용`을 제공한다.

### 15.3 로딩, 빈 상태와 오류 화면

- 최초 로딩 skeleton은 최종 레이아웃과 같은 높이로 만들어 화면 이동을 막는다.
- 추가 로딩은 기존 결과를 유지하고 하단에 표시한다.
- 빈 상태는 현재 콘텐츠 영역의 가로 중앙을 기본 정렬로 사용하고 제목, 이유 한 줄과 한 개 행동을 한 덩어리로 구성한다. 조밀한 상세 패널의 인라인 빈 상태만 화면에서 명시적으로 시작선 정렬을 선택한다.
- 하위 작업이 없는 이슈의 작업 흐름 빈 상태는 `아직 시작된 팀 작업이 없습니다`, `분석이 끝났다면 작업을 시작할 팀을 선택해 주세요.`, `작업 시작`을 사용한다.
- 필터 결과 없음은 생성보다 `필터 초기화`를 우선한다.
- 전역 이슈 빈 상태는 각각 `아직 등록된 이슈가 없습니다`, `검토를 기다리는 이슈가 없습니다`, `담당자가 필요한 작업이 없습니다`, `조건에 맞는 이슈가 없습니다`와 한 개의 복구 행동을 사용한다.
- 작업 시작·내가 맡기·담당자 지정 실패는 선택값을 지우거나 오버레이를 닫지 않고 최신 담당자·후보와 첫 오류를 입력 가까이에 표시한다.
- 권한 없음과 찾을 수 없음은 큰 일러스트 없이 32px 이하 아이콘, 설명과 안전한 이동 동작을 제공한다.

## 16. 모션

| 토큰 | 시간 | 사용 |
| --- | ---: | --- |
| `motion.instant` | 80ms | 버튼 pressed, 상태점 변화 |
| `motion.fast` | 120ms | hover, 선택 배경, tooltip |
| `motion.default` | 160ms | 메뉴, 팝오버, 탭 전환 |
| `motion.overlay` | 220ms | 모달과 모바일 시트 |

- 기본 easing은 `cubic-bezier(0.2, 0, 0, 1)`을 사용한다.
- hover, 색상과 투명도만 바뀌는 요소는 120ms를 넘지 않는다.
- 목록 정렬·필터 결과를 과도하게 이동시키는 애니메이션은 사용하지 않는다.
- 실시간 변경은 1회 600ms 이하의 은은한 배경 강조 후 원래 상태로 돌아온다.
- `prefers-reduced-motion`에서는 위치 이동과 확대를 제거하고 즉시 전환 또는 짧은 opacity 변화만 사용한다.

## 17. 접근성

### 17.1 명암과 색상

- 일반 텍스트는 WCAG AA 4.5:1, 18px 이상 또는 굵은 큰 텍스트는 3:1 이상을 목표로 한다.
- 포커스, 컴포넌트 경계와 상태 아이콘은 인접 색과 3:1 이상을 목표로 한다.
- 상태, 우선순위, 읽지 않음, 오류와 현재 위치를 색상만으로 구분하지 않는다.
- `color.text.disabled`는 정보를 읽어야 하는 본문에 사용하지 않는다.

### 17.2 키보드와 포커스

- 모든 동작은 Tab과 Shift+Tab으로 도달할 수 있어야 한다.
- 상세 탭은 좌우 방향키 이동을 지원하고 `tablist`, `tab`, `tabpanel`과 연결된 ID를 사용한다.
- 탭 전환 뒤 패널 제목 또는 첫 의미 있는 요소로 읽기 맥락을 제공하되 URL 직접 진입에서는 포커스를 강제로 이동하지 않는다.
- 목록 행과 내부 속성 편집기는 별도 포커스 대상을 제공한다.
- 전역 이슈 행 전체 열기와 내부 주요 행동·우선순위·라벨 선택기는 독립 포커스 대상을 제공하고 Enter·Space 이벤트가 서로 전파되지 않게 한다. 이슈 상태는 읽기 전용 이름을 제공한다.
- 전역 이슈 진행률은 완료 수·전체 수·백분율 텍스트와 `aria-valuenow` 또는 동등한 값을 제공하고, 200% 확대에서도 제목과 주요 행동을 사용할 수 있어야 한다.
- C, /, Escape, Command/Ctrl+Enter 등 전역 단축키는 텍스트 입력 중 실행하지 않는다.
- 모달·선택기에는 포커스 트랩과 호출부 복원을 적용한다.
- 목록 필터·라벨 팝오버와 속성 Select는 Escape와 키보드 선택 후 호출부로 포커스를 복원하며 저장 전환 중에도 포커스 대상이 사라지지 않아야 한다. 바깥의 다른 조작 대상을 선택한 경우에는 그 대상의 포커스를 유지한다.
- 목록 필터·라벨 팝오버 안에서는 위·아래 방향키로 활성 옵션을 순환하고 Home·End로 처음과 끝에 이동한다. Enter와 Space는 현재 옵션을 선택하며 비활성 옵션을 건너뛴다.
- 드래그 동작에는 상태 선택기나 순서 버튼 같은 키보드 대안을 제공한다.

### 17.3 보조 기술 이름

- 아이콘 버튼은 보이는 tooltip과 동일한 한국어 접근성 이름을 사용한다.
- 저장 중, 읽지 않음, 시작 역할 선택, 펼침, 현재 주요 행동과 오류 상태를 속성으로 전달한다.
- 상태·우선순위의 접근 가능한 이름에는 표시 ID가 필요한 목록 맥락과 전체 한국어 현재 값을 포함하고, 아이콘만 보이는 중간 폭에서도 같은 이름을 유지한다.
- 빠른 필터 선택, 현재 결과 수, 검토 필요, 담당 필요와 진행률을 텍스트와 접근성 속성으로 전달한다.
- 동적으로 추가되는 toast와 저장 결과는 중요도에 맞는 live region으로 알린다.
- 표시 ID만으로 링크 이름을 만들지 않고 제목을 함께 제공한다.

## 18. 한국어 UI 문구

### 18.1 기본 원칙

- 메뉴, 상태, 오류, 도움말과 버튼은 한국어를 기본으로 한다.
- `API`, `CSV`, `Markdown`처럼 팀이 실제로 쓰는 고유 기술 용어는 억지로 번역하지 않는다.
- 문장은 능동형과 현재형을 사용하고 존댓말 어미를 화면 안에서 일관되게 유지한다.
- 버튼은 명사보다 결과가 드러나는 동사를 쓴다.
- 오류는 사과보다 원인과 해결 방법을 쓴다.
- 빈 상태는 감성 문구보다 현재 조건과 다음 행동을 알려 준다.

### 18.2 문구 예시

| 상황 | 사용 | 피함 |
| --- | --- | --- |
| 프로젝트 저장 | `프로젝트 만들기` | `확인`, `제출` |
| 기본 업무 생성 | `이슈 만들기` | `기능 이슈 만들기`, 유형 선택 뒤 `만들기` |
| 개인 업무 화면 | `내 작업` | `내 이슈` |
| 전역 검토 화면 | `이슈` | `FEATURE`, `전역 기능 이슈` |
| 시작 역할 | `처음 작업할 팀 (선택)` | `프로젝트 역할에 따라 자동 생성` |
| 팀 작업 없는 이슈 | `작업 시작` | `첫 작업 시작`, 숫자만 있는 `0/0 완료` |
| 팀의 미할당 작업 생성 | `우리 팀에서 시작` | 클라이언트가 팀을 직접 지정하는 문구 |
| 자신에게 할당 | `내가 맡기` | `강제 배정` |
| 미할당 작업 일괄 처리 | `담당자 지정` | 행마다 여러 저장 버튼 |
| 팀 작업 시작 | `작업 시작` | 상태 메뉴에서만 `진행 중` 선택 |
| 팀 작업 완료 | `완료`, 완료 뒤 `완료됨` | 상태 메뉴에서만 `완료` 선택 |
| 전달 이력 | `최초 전달`, `추가 전달` | `연결 정보` |
| 변경 저장 | `변경 저장` | `적용하기`를 모든 곳에 반복 |
| 저장 실패 | `저장하지 못했습니다. 다시 시도해 주세요.` | `문제가 발생했습니다.` |
| 권한 부족 | `이 설정을 변경할 권한이 없습니다.` | `Access denied` |
| 필터 결과 없음 | `조건에 맞는 작업이 없습니다.` | `아직 아무것도 없어요!` |
| 백엔드 완료 방식 | `이 작업만 완료`, `프론트에 전달 후 완료` | 프로젝트 역할만 보고 전달 강제 |

날짜는 화면 밀도에 따라 `2026. 7. 10.` 또는 상대 시각 `5분 전`을 사용하고, tooltip이나 상세 정보에서 절대 시각을 확인할 수 있게 한다.

## 20. 구현과 검수 계약

### 20.1 구현 순서

1. 원시·의미 토큰과 한국어 서체
2. 버튼, 입력, 아이콘, 선택기와 포커스 상태
3. 앱 셸, 주 탐색과 반응형 구조
4. 이슈 행·카드, 상태·우선순위·역할·작업 대기 배지
5. 모달, 검색, Markdown, 파일 업로드와 피드백 상태
6. 프로젝트 계층, 작업 연결선, 댓글·활동, 프로필과 설정 컴포넌트

### 20.2 컴포넌트 검수 상태

각 컴포넌트는 최소 다음 조합을 검수한다.

- 기본, hover, focus-visible, open, active, saving, error, disabled
- 값 없음, 긴 한국어, 긴 영문·URL, 많은 라벨
- 로딩, 저장 성공, 저장 실패와 충돌
- 초광폭·데스크톱 1280px, compact 1024px, 모바일 390px
- 전역 이슈·내 작업·팀 목록·보드·프로젝트 행·이슈 중심 상세·내 작업 진입 상세에서 같은 데이터 상태 비교
- `/issues/{issueIdentifier}`, `/projects/{projectId}/issues/{issueIdentifier}`, `/my-issues/{teamWorkIdentifier}`에서 활성 내비게이션·뒤로 가기·상단 위계는 다르고 공유 본문·상태 값은 같은지 확인
- 키보드 전용 탐색과 200% 확대
- 라벨을 가린 상태에서도 이해 가능한지 확인

### 20.3 금지 규칙

- 화면별 임의 hex, px와 z-index 추가
- 라벤더로 큰 카드나 페이지 영역 채우기
- 상태와 우선순위를 색상만으로 표현하기
- 어두운 배경 위에 여러 단계의 강한 그림자 사용
- 모든 버튼·탭·배지를 pill 형태로 만들기
- 한국어 작은 글자에 강한 음수 자간 적용
- 오류를 toast 하나로만 알리기
- 모바일 미지원 기능을 축소된 데스크톱 화면으로 억지 제공하기
