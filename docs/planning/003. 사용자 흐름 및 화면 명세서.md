## 1. 문서 목적

이 문서는 PRD에서 확정한 요구사항을 사용자가 실제로 이동하고 조작하는 흐름과 화면 단위 명세로 전환한다. 각 화면의 목적, 진입 조건, 핵심 정보, 주요 행동, 유효성 검사, 빈 상태와 실패 상태를 정의해 와이어프레임, 디자인 시스템, 프론트엔드 구현과 API 설계가 같은 제품 구조를 기준으로 진행되게 한다.

이 문서는 시각 디자인을 확정하지 않는다. 저장소 루트의 [DESIGN.md](../../DESIGN.md)는 Rivet의 상위 디자인 방향, 후속 와이어프레임과 디자인 시스템은 영역 구조와 정확한 시각 값을 정의한다. Linear 벤치마킹 원본은 [Linear 시각 디자인 분석](../reference/001.%20Linear%20시각%20디자인%20분석.md)에 참고 자료로 보존하며 현재 제품 기준으로 직접 사용하지 않는다.

### 1.1 명세 범위

- 가입, 인증, 로그인과 초기 워크스페이스 설정
- 내 이슈, 팀 이슈, 목록과 상태 보드
- 프로젝트 생성, 역할별 팀 선택과 프로젝트 상세
- 기능 이슈, 팀 작업, 하위 작업과 차단 관계
- 백엔드 작업 전달과 웹·앱 작업 차단 해제
- Lexical 기반 설명·댓글·작업 전달 편집, 이미지 붙여넣기, Markdown 미리보기와 활동 기록
- 이슈 첨부파일과 사용자 프로필 사진
- 전역 검색, 인앱 알림함과 빠른 조작
- 멤버, 팀, 워크플로, 라벨, 내보내기와 휴지통 설정
- 데스크톱과 제한적 모바일 웹 동작

### 1.2 해석 원칙

- PRD와 충돌하면 PRD를 우선한다.
- PRD에서 정하지 않은 핵심 UX 세부사항은 이 문서의 확정된 UX 결정으로 명시한다.
- 기능 이슈는 결과물의 범위, 팀 작업은 실제 실행 책임을 표현한다.
- 팀 이름을 상태에 넣지 않고 팀, 프로젝트 역할, 상태와 차단 관계를 분리해 보여 준다.
- 빠른 생성과 속성 변경을 우선하고 부가 설정은 설정 화면으로 분리한다.

## 2. UX 원칙

### 2.1 한국어 우선

사용자에게 보이는 메뉴, 상태, 오류와 도움말은 한국어를 기본으로 한다. 시스템 범주와 내부 이벤트 이름은 화면에 직접 노출하지 않는다.

### 2.2 한 화면에서 한 가지 중심 작업

- 내 이슈는 내가 처리할 작업을 찾고 상태를 바꾸는 데 집중한다.
- 팀 이슈는 한 팀의 작업 집합을 목록 또는 보드로 운영하는 데 집중한다.
- 프로젝트는 여러 팀이 함께 만드는 결과물과 진행률을 확인하는 데 집중한다.
- 이슈 상세는 내용, 관계, 대화와 변경 기록을 한곳에서 이해하는 데 집중한다.

### 2.3 속도와 맥락 보존

- 모든 주요 화면에서 같은 이슈 생성 창과 전역 검색을 연다.
- 목록의 상태, 담당자와 우선순위는 상세 화면을 열지 않고 변경한다.
- 상세 화면에서 이전 화면으로 돌아오면 필터, 정렬, 선택 탭과 스크롤 위치를 복원한다.
- 저장 가능한 변경은 먼저 화면에 반영하되 저장 중임을 구분하고 실패하면 이전 값으로 되돌린다.

### 2.4 관계의 명시성

- 차단됨은 상태 선택지가 아니라 별도 배지와 관계 영역으로 표시한다.
- 프로젝트 역할과 실제 담당 팀을 함께 표시한다.
- 같은 팀이 여러 프로젝트 역할을 맡으면 역할 배지로 작업의 책임을 구분한다.
- 완료된 차단 관계는 삭제하지 않고 차단 해제됨으로 남긴다.

### 2.5 안전한 파괴 작업

- 삭제, 보관, 멤버 비활성화처럼 영향이 큰 동작은 결과를 먼저 설명한다.
- 해결해야 할 연결 관계가 있으면 확인창만 띄우지 않고 차단 사유와 이동할 화면을 제공한다.
- 30일 휴지통 복구 가능 여부와 영구 삭제 예정일을 명확히 표시한다.

## 3. 사용자와 권한

| 사용자 | 할 수 있는 일 | 제한되는 일 |
| --- | --- | --- |
| 비로그인 사용자 | 가입, 이메일 인증, 로그인, 비밀번호 재설정, 초대 수락 시작 | 워크스페이스 업무 데이터 접근 |
| 멤버 | 공개된 팀·이슈·프로젝트 조회, 이슈·프로젝트 생성과 수정, 댓글 작성 | 멤버·팀·워크플로·라벨 관리, 내보내기, 휴지통 복구 |
| 관리자 | 멤버의 모든 동작과 워크스페이스 운영 설정 | 다른 워크스페이스 데이터 접근 |
| 비활성 멤버 | 로그인 화면 접근과 계정 안내 확인 | 로그인 후 신규 업무 활동 |

하나의 계정은 최대 하나의 워크스페이스에만 참여한다. 워크스페이스에 참여한 뒤에는 새 워크스페이스를 만들거나 다른 워크스페이스의 초대를 수락할 수 없다.

권한이 없는 기능은 기본적으로 숨긴다. 직접 주소로 접근하면 권한 부족 화면을 표시하고 접근 가능한 이전 화면으로 이동할 수 있게 한다. 존재 여부 자체가 민감한 다른 워크스페이스 리소스는 찾을 수 없음과 같은 응답으로 처리한다.

### 4.3 로그인 후 기본 진입

- 정상 로그인: 내 이슈
- 처음 로그인했고 워크스페이스가 없음: 워크스페이스 생성
- 초대 링크로 가입 또는 로그인: 초대 수락 후 해당 워크스페이스의 내 이슈
- 워크스페이스에 참여한 사용자: 해당 워크스페이스로 바로 진입
- 세션이 만료된 사용자: 로그인 후 원래 접근하려던 내부 주소로 복귀

## 5. 공통 애플리케이션 구조

### 5.1 데스크톱 셸

| 영역 | 내용 | 동작 |
| --- | --- | --- |
| 워크스페이스 헤더 | 워크스페이스 이름, 사용자 메뉴 | 워크스페이스 전환 없이 프로필 사진 변경·삭제와 로그아웃 제공 |
| 주 탐색 | 내 이슈, 알림함, 프로젝트 | 현재 위치 강조, 알림함 읽지 않음 개수 표시 |
| 팀 탐색 | 활성 팀 목록 | 팀 선택 시 마지막으로 사용한 목록 또는 보드로 이동 |
| 전역 동작 | 이슈 만들기, 검색 | 모든 업무 화면에서 고정된 위치와 단축키 제공 |
| 본문 헤더 | 화면 제목, 보기 전환, 필터, 정렬, 화면별 주요 동작 | 스크롤 중에도 핵심 조작에 접근 가능 |
| 본문 | 목록, 보드, 프로젝트 또는 상세 | 점진적 로딩과 화면별 빈 상태 제공 |

관리자에게만 설정 진입점을 표시한다. 보관된 팀은 팀 탐색에 표시하지 않는다.

### 5.2 전역 이슈 만들기

전역 생성 창은 기본적으로 팀 작업을 생성한다. 사용자는 유형을 기능 이슈로 바꿀 수 있다.

#### 팀 작업 입력

| 필드 | 필수 | 동작 |
| --- | :---: | --- |
| 제목 | 필수 | 창을 열면 바로 포커스 |
| 팀 | 필수 | 팀 화면에서는 현재 팀, 다른 화면에서는 마지막 선택 팀을 기본값으로 사용 |
| 상태 | 필수 | 선택한 팀의 미분류 상태를 기본값으로 사용 |
| 담당자 | 선택 | 선택한 팀의 활성 멤버만 표시 |
| 우선순위 | 선택 | 기본값 없음 |
| 프로젝트 | 선택 | 활성 프로젝트만 표시 |
| 프로젝트 역할 | 조건부 필수 | 프로젝트를 선택하면 해당 프로젝트에서 활성화한 역할만 표시하고 역할 선택 시 팀을 자동 결정 |
| 상위 기능 이슈 | 선택 | 프로젝트를 선택했을 때 같은 프로젝트의 기능 이슈만 표시 |
| 라벨 | 선택 | 활성 라벨 여러 개 선택 |
| 설명 | 선택 | Lexical WYSIWYG 편집과 Markdown 미리보기를 기본으로 펼쳐서 제공 |
| 첨부파일 | 선택 | 여러 파일을 선택하며 파일당 최대 25MB와 업로드 상태 표시 |

상위 기능 이슈를 먼저 고르면 프로젝트와 역할을 상위 이슈에서 제한한다. 생성된 팀 작업의 팀은 이후 변경할 수 없으므로 저장 전에 안내한다.

#### 기능 이슈 입력

| 필드 | 필수 | 동작 |
| --- | :---: | --- |
| 제목 | 필수 | 첫 입력 위치 |
| 프로젝트 | 필수 | 활성 프로젝트만 표시 |
| 상태 | 필수 | 워크스페이스 고정 상태의 미분류를 기본값으로 사용 |
| 우선순위 | 선택 | 기본값 없음 |
| 라벨 | 선택 | 활성 라벨 여러 개 선택 |
| 설명 | 선택 | Lexical WYSIWYG 편집과 Markdown 미리보기 |
| 첨부파일 | 선택 | 여러 파일을 선택하며 파일당 최대 25MB와 업로드 상태 표시 |

기능 이슈에는 팀과 담당자 입력을 표시하지 않는다.

#### 저장 동작

- 저장 중 버튼을 중복 실행할 수 없게 한다.
- 성공하면 생성된 표시 ID를 알리고 상세 열기 동작을 제공한다.
- 필수값 누락은 해당 필드 가까이 설명하고 첫 오류 필드로 포커스를 이동한다.
- 프로젝트 역할과 팀이 충돌하면 역할을 기준으로 팀을 다시 결정하고 이유를 안내한다.
- 본문 이미지와 첨부파일 업로드가 모두 완료된 뒤 이슈 저장을 실행한다. 업로드 실패 파일이 있으면 입력을 유지하고 다시 시도하거나 제거하게 한다.

### 5.3 전역 검색

- 검색 창은 현재 화면 위에 겹쳐 열고 입력란에 즉시 포커스한다.
- 검색어를 입력하기 전에는 검색 안내만 있는 빈 상태를 표시하고 최근 열어본 이슈는 제공하지 않는다.
- 두 글자 이상 입력하면 표시 ID와 제목을 검색한다.
- 정확한 표시 ID 일치를 가장 위에 표시한다.
- 결과는 기능 이슈와 팀 작업 유형, 표시 ID, 제목, 프로젝트, 팀 또는 역할, 상태를 구분해 보여 준다.
- 위·아래 방향키로 결과를 이동하고 Enter로 상세를 연다.
- 결과가 없으면 검색 가능한 범위가 ID와 제목임을 안내한다.
- 다른 워크스페이스 데이터와 휴지통의 이슈는 표시하지 않는다.

### 5.4 공통 속성 선택기

상태, 담당자, 우선순위, 라벨과 프로젝트 선택기는 목록과 상세에서 같은 이름, 순서와 의미를 사용한다. 선택 직후 낙관적으로 반영하고 다음 상태를 표시한다.

- 저장 중: 해당 속성에 작은 진행 표시
- 저장 성공: 별도 성공 알림 없이 값 유지
- 저장 실패: 이전 값으로 복구하고 다시 시도 제공
- 충돌: 서버의 최신 값을 보여 주고 내 변경을 다시 적용할지 선택

## 6. 핵심 사용자 흐름

### 6.1 UF-01 회원가입과 첫 설정

| 단계 | 사용자 행동 | 화면·시스템 반응 |
| ---: | --- | --- |
| 1 | 표시 이름, 이메일, 비밀번호와 비밀번호 확인 입력 | 표시 이름의 필수·길이 오류, 이메일 중복과 비밀번호 정책 오류를 해당 필드에 표시 |
| 2 | 회원가입 실행 | 인증 메일 발송 후 이메일 인증 안내로 이동 |
| 3 | 메일의 인증 링크 실행 | 유효한 링크면 인증 완료, 만료·사용 완료 링크면 재발송 안내 |
| 4 | 로그인 | 워크스페이스가 없으면 워크스페이스 생성으로 이동 |
| 5 | 워크스페이스 이름과 슬러그 입력 | 슬러그 사용 가능 여부를 확인하고 생성자는 관리자가 됨 |
| 6 | 기본 팀 이름과 키 입력 | 한국어 기본 워크플로를 함께 생성 |
| 7 | 동료 이메일 입력 또는 건너뛰기 | 초대 발송 결과를 주소별로 표시 |
| 8 | 첫 팀 작업 만들기 | 성공 후 내 이슈 또는 생성한 이슈 상세로 이동 |

초기 멤버 초대는 1인 팀을 고려해 건너뛸 수 있다. 기본 팀 생성은 건너뛸 수 없다.

### 6.2 UF-02 초대 수락

| 조건 | 흐름 |
| --- | --- |
| 가입하지 않은 이메일 | 초대 확인 → 회원가입 → 이메일 인증 → 초대 수락 → 내 이슈 |
| 가입했지만 로그아웃 | 초대 확인 → 로그인 → 초대 이메일과 계정 이메일 확인 → 수락 → 내 이슈 |
| 로그인한 동일 이메일 | 초대 확인 → 수락 → 내 이슈 |

| 다른 워크스페이스에 참여 중인 계정 | 초대 수락을 막고 MVP의 1계정 1워크스페이스 제한을 안내 |
| 만료·취소된 초대 | 사용할 수 없음을 안내하고 관리자에게 재초대를 요청하도록 표시 |

### 6.3 UF-03 일상 팀 작업 생성과 완료

1. 사용자가 현재 팀 화면에서 이슈 만들기를 연다.
2. 팀 작업이 기본 유형이고 현재 팀과 미분류 상태가 자동 선택된다.
3. 제목을 입력하고 필요하면 담당자, 우선순위, 프로젝트와 라벨을 지정한다.
4. 프로젝트를 지정하면 프로젝트 역할을 선택하고 팀이 역할의 담당 팀으로 자동 결정된다.
5. 저장하면 목록 또는 보드에 즉시 나타나며 서버 실패 시 제거하고 재시도를 제공한다.
6. 사용자는 목록에서 상태, 담당자와 우선순위를 바로 바꾼다.
7. 상세에서 설명과 댓글로 맥락을 추가한다.
8. 완료 상태로 바꾸면 기본 미완료 보기에서 제외된다.

다른 작업을 기다리는 경우 보류로 바꾸지 않고 차단 관계를 추가한다.

### 6.4 UF-04 프로젝트 생성

1. 프로젝트 목록에서 프로젝트 만들기를 선택한다.
2. 이름을 입력하고 필요하면 설명, 리드, 시작일과 목표일을 입력한다.
3. 백엔드, 웹 프론트, 앱 프론트 역할별로 담당 팀을 선택하거나 비워 둔다.
4. 최소 한 역할에 활성 팀을 선택해야 저장할 수 있다.
5. 같은 팀을 여러 역할에 선택할 수 있으며 각 역할은 별도로 표시된다.
6. 저장하면 계획 상태로 프로젝트가 생성되고 프로젝트 상세로 이동한다.
7. 상세에서 첫 기능 이슈를 만들거나 단독 팀 작업을 추가한다.

목표일이 시작일보다 빠르면 저장을 막고 두 날짜 필드에 관계 오류를 표시한다.

### 6.5 UF-05 기능 이슈를 역할별 작업으로 분해

1. 프로젝트 상세에서 기능 이슈 만들기를 선택한다.
2. 제목과 상태를 입력하고 저장한다.
3. 기능 이슈 행에서 팀 작업 추가를 선택한다.
4. 프로젝트에 선택된 역할 중 하나를 고른다.
5. 역할에 연결된 팀이 자동 결정되고 해당 팀의 상태와 담당자를 선택한다.
6. 백엔드, 웹 프론트, 앱 프론트 작업을 필요한 만큼 추가한다.
7. 기능 이슈 아래에 역할, 팀, 상태, 담당자, 차단 여부와 진행률을 계층으로 표시한다.
8. 취소된 하위 작업을 제외한 완료 비율을 기능 이슈 진행률로 표시한다.

기능 이슈의 상태는 하위 작업 진행률과 별도로 사용자가 변경한다.

### 6.6 UF-06 백엔드 작업 전달과 차단 해제

1. 사용자가 백엔드 팀 작업과 후행 웹·앱 작업 사이에 차단 관계를 만든다.
2. 후행 작업에는 현재 상태 옆에 차단됨 배지를 표시한다.
3. 백엔드 작업 상세의 작업 전달 영역에서 변경 내용을 작성할 수 있다.
4. 후행 프론트 작업이 미완료인 상태에서 백엔드 작업을 완료하려고 한다.
5. 작업 전달이 없으면 완료를 중단하고 필수 전달 작성 창을 연다.
6. 사용자가 템플릿을 채워 저장하면 백엔드 작업을 완료한다.
7. 후행 작업은 기존 상태를 유지하며 차단 해제됨으로 바뀐다.
8. 실제 후행 작업 담당자와 자동 구독자에게 전달 알림을 한 건씩 보낸다.
9. 완료 후 변경이 생기면 추가 전달을 작성하고 기존 전달은 수정하지 않는다.

작업 전달 템플릿은 다음 항목을 제공한다.

- 변경 요약
- API 명세 링크
- 사용 가능 환경
- 추가·변경 API
- 요청·응답 변경
- 오류·권한
- 프론트 주의사항

후행 프론트 작업이 없으면 작업 전달 없이 백엔드 작업을 완료할 수 있다.

### 6.7 UF-07 내 이슈 처리

1. 로그인 후 내 이슈로 진입한다.
2. 여러 팀에서 자신에게 할당된 미완료 팀 작업을 한 목록에서 확인한다.
3. 상태, 팀, 프로젝트와 우선순위로 필터링하고 정렬한다.
4. 차단된 작업은 상태와 별도로 차단됨 배지를 확인한다.
5. 목록에서 상태와 우선순위를 바꾸거나 상세를 연다.
6. 담당자를 다른 사람으로 변경하면 현재 목록에서 제거된다.
7. 완료 또는 취소된 작업은 기본 목록에서 제거된다.

### 6.8 UF-08 댓글, 멘션과 알림

1. 사용자가 이슈 상세의 Lexical WYSIWYG 편집기에서 댓글을 작성하고 필요하면 이미지를 선택하거나 Command/Ctrl+V로 붙여넣는다.
2. 이미지는 업로드 중 미리보기를 표시하고 성공하면 인증된 파일 참조로 바뀐다.
3. 저장될 Markdown 결과를 미리 보고 활성 멤버를 멘션해 댓글을 저장한다.
4. 멘션 대상과 자동 구독 대상 중 같은 사용자는 한 건의 알림만 받는다.
5. 수신자가 알림함에서 알림을 선택한다.
6. 이슈 상세의 관련 댓글 또는 작업 전달 위치로 이동한다.
7. 알림은 읽음 처리되고 전역 읽지 않음 개수가 갱신된다.

자신이 만든 이벤트에 대해서는 알림을 받지 않는다.

### 6.9 UF-09 삭제와 복구

1. 사용자가 이슈 또는 비어 있는 프로젝트의 더보기 메뉴에서 삭제를 선택한다.
2. 삭제 가능 조건을 확인한다.
3. 연결 관계 때문에 삭제할 수 없으면 정리할 관계와 이동 동작을 보여 준다.
4. 확인 후 일반 목록과 검색에서 제거하고 휴지통으로 이동한다.
5. 관리자는 휴지통에서 삭제일과 영구 삭제 예정일을 확인한다.
6. 복구하면 원래 연결을 유지한 채 일반 화면에 다시 나타난다.
7. 30일이 지나면 자동 영구 삭제되어 복구할 수 없다.

### 6.10 UF-10 동시 수정과 재연결

1. 사용자가 속성을 바꾸면 화면에 먼저 반영하고 저장 중으로 표시한다.
2. 서버 저장에 성공하면 진행 표시를 제거한다.
3. 실패하면 이전 값으로 복구하고 오류와 다시 시도를 보여 준다.
4. 다른 사용자가 같은 이슈를 변경하면 열린 목록과 상세에 반영한다.
5. 오래된 화면의 저장이 최신 변경과 충돌하면 자동 덮어쓰지 않는다.
6. 최신 값을 먼저 보여 주고 사용자가 자신의 변경을 다시 적용하게 한다.
7. 실시간 연결이 끊기면 연결 상태를 알리고 재연결 후 현재 화면 데이터를 다시 확인한다.

### 6.11 UF-11 파일 첨부와 프로필 사진

**이슈 첨부파일**

1. 사용자가 이슈 생성 또는 상세의 첨부 영역에서 하나 이상의 파일을 선택한다.
2. 파일마다 이름, 형식과 크기를 표시하고 25MB를 넘으면 업로드 전에 거부한다.
3. JPEG·PNG·WebP 이미지가 2MB를 넘거나 긴 변이 2,560px을 넘으면 브라우저에서 비율을 유지해 축소·압축한다.
4. 업로드 중 파일별 진행 상태를 표시하고 성공한 파일은 열기·다운로드, 실패한 파일은 다시 시도·제거 동작을 제공한다.
5. 사용자가 첨부를 제거하면 화면에서 즉시 사라지고 연결이 해제된 실제 파일은 서버 정리 정책에 따라 제거된다.

**프로필 사진**

1. 사용자가 사용자 메뉴에서 `프로필 설정`을 연다.
2. JPEG·PNG·WebP 이미지를 선택하면 브라우저에서 긴 변을 최대 512px로 축소·압축하고 미리보기를 표시한다.
3. 저장하면 담당자, 멘션, 댓글과 활동의 사용자 표시에 새 사진이 반영된다.
4. 사진을 삭제하면 표시 이름 첫 글자 아바타로 돌아간다.

## 8. 화면별 명세

### 8.1 AUTH-01 회원가입

**주요 구성**

- 표시 이름
- 이메일
- 비밀번호
- 비밀번호 확인
- 회원가입 버튼
- 로그인 이동

**상태와 오류**

- 표시 이름의 필수·길이 오류, 이메일 형식 오류, 중복 이메일과 비밀번호 정책 미충족을 필드별로 표시한다.
- 제출 중에는 중복 제출을 막는다.
- 성공하면 인증 메일을 보낸 주소를 가린 형태로 보여 주고 이메일 인증 안내로 이동한다.
- 계정 존재 여부를 악용하지 않도록 오류 문구의 보안 수준은 기술 설계에서 확정한다.

### 8.2 AUTH-02 이메일 인증

**인증 전 안내**

- 인증 메일을 보낸 주소
- 메일을 확인하라는 안내
- 제한 시간을 둔 인증 메일 재발송
- 이메일 주소가 잘못된 경우 회원가입으로 돌아가기

**링크 실행 결과**

| 결과 | 화면 동작 |
| --- | --- |
| 성공 | 인증 완료 안내와 로그인 버튼 |
| 만료 | 링크 만료 안내와 재발송 |
| 이미 사용 | 이미 인증되었으면 로그인 안내 |
| 유효하지 않음 | 사용할 수 없는 링크 안내 |

### 8.3 AUTH-03 로그인

- 이메일, 비밀번호, 로그인 버튼과 비밀번호 찾기를 제공한다.
- 인증되지 않은 계정이면 업무 화면에 들어가지 않고 인증 메일 재발송을 안내한다.
- 비활성 멤버는 로그인할 수 없다는 안내와 워크스페이스 관리자 문의 경로를 보여 준다.
- 성공하면 기본적으로 내 이슈로 이동하되 보존된 내부 복귀 주소가 있으면 그 주소를 우선한다.

### 8.4 AUTH-04 비밀번호 재설정

- 이메일 입력 단계는 요청 성공 여부와 관계없이 일관된 완료 안내를 사용한다.
- 새 비밀번호 입력 단계는 새 비밀번호와 확인 필드를 제공한다.
- 만료되었거나 이미 사용한 링크는 새 링크 요청으로 연결한다.
- 성공하면 기존 재설정 링크가 무효가 되었음을 알리고 로그인으로 이동한다.

### 8.5 AUTH-05 초대 수락

- 워크스페이스 이름, 초대 이메일과 초대한 사람을 표시한다.
- 로그인 계정 이메일과 초대 이메일이 다르면 계정 전환을 요구한다.
- 수락 성공 후 해당 워크스페이스의 내 이슈로 이동한다.
- 초대가 중복 수락된 경우 새 멤버십을 만들지 않는다.
- 이미 다른 워크스페이스에 참여한 계정은 초대를 수락할 수 없으며 현재 워크스페이스로 돌아가는 동작을 제공한다.

### 8.6 ONB-01 워크스페이스 생성

| 필드 | 규칙 |
| --- | --- |
| 워크스페이스 이름 | 필수, 사용자에게 보이는 이름 |
| 슬러그 | 필수, 고유성 확인, 주소에 사용할 수 있는 형식만 허용 |

저장 성공 후 기본 팀 생성으로 이동한다. 브라우저를 닫았다가 돌아와도 완료된 단계부터 이어갈 수 있어야 한다.

워크스페이스 생성은 멤버십이 없는 계정에만 표시한다. 기존 워크스페이스에 참여한 계정에는 추가 생성 동작과 워크스페이스 전환 메뉴를 제공하지 않는다.

### 8.7 ONB-02 기본 팀 생성

| 필드 | 규칙 |
| --- | --- |
| 팀 이름 | 필수, 활성 팀 이름과 중복 불가 |
| 팀 키 | 필수, 영문 대문자 2~5자, 표시 ID 예시를 함께 제공 |
| 팀 멤버 | 생성자를 기본 선택, 추가 멤버는 초대 후 설정 가능 |

저장하면 미분류, 할 일, 진행 중, 검토, 완료, 보류, 취소 상태를 자동 생성한다. 팀 키가 첫 이슈 생성 후 바뀌지 않는다는 점을 생성 전에 안내한다.

### 8.8 ONB-03 초기 멤버 초대

- 이메일을 여러 줄 또는 반복 입력으로 추가할 수 있다.
- 중복 이메일은 입력 단계에서 합친다.
- 주소별 성공, 이미 멤버, 이미 초대됨과 실패 결과를 구분한다.
- 건너뛰기와 초대 후 계속 동작을 제공한다.
- 완료 후 첫 이슈 만들기를 강조하고 내 이슈로 이동할 수 있게 한다.

### 8.9 WORK-01 내 이슈

**헤더**

- 제목과 현재 결과 수
- 필터: 상태, 팀, 프로젝트, 우선순위
- 정렬: 생성일, 최근 수정일, 상태, 우선순위와 방향
- 필터 초기화

**목록 행**

- 표시 ID
- 제목
- 팀
- 프로젝트와 프로젝트 역할
- 상태
- 차단됨 배지
- 담당자
- 우선순위
- 라벨
- 최근 수정일

기본 결과는 자신에게 할당된 Completed·Canceled 이외의 팀 작업이다. 기능 이슈는 담당자를 가질 수 없으므로 내 이슈에 나타나지 않는다.

**빈 상태**

- 할당된 작업이 없음: 현재 할당된 미완료 작업이 없음을 표시
- 필터 결과 없음: 필터 초기화 제공
- 처음 사용: 팀 작업 만들기 제공

### 8.10 TEAM-01 팀 이슈 목록

**헤더**

- 팀 이름과 팀 키
- 전체, 진행, 백로그 탭
- 목록·보드 전환
- 필터, 정렬과 팀 작업 만들기

**목록**

- 상태별 또는 정렬 결과에 따라 행을 표시한다.
- 상태, 담당자와 우선순위는 행에서 바로 변경한다.
- 팀은 현재 화면의 팀으로 고정되어 행에서 편집하지 않는다.
- 완료·취소 항목의 포함 여부는 선택한 탭과 필터로 결정한다.
- 기능 이슈는 팀에 속하지 않으므로 팀 이슈 목록에 직접 표시하지 않는다. 프로젝트와 상위 기능 이슈 정보만 팀 작업의 맥락으로 표시한다.

**탭 규칙**

| 탭 | 포함 범주 |
| --- | --- |
| 전체 | 삭제되지 않은 모든 팀 작업 |
| 진행 | Unstarted, Started |
| 백로그 | Backlog |

Canceled와 Completed는 전체 탭 또는 명시적 상태 필터에서 확인한다.

### 8.11 TEAM-02 팀 상태 보드

- 선택한 팀의 상태를 설정 순서대로 열로 표시한다.
- 카드에는 표시 ID, 제목, 담당자, 우선순위, 프로젝트 역할, 라벨과 차단 여부를 표시한다.
- 카드를 다른 열로 옮기면 상태만 변경한다.
- 수동 순서 저장은 하지 않으며 각 열 안의 카드는 현재 정렬 기준을 따른다.
- 드래그 실패 시 원래 열로 되돌리고 오류를 표시한다.
- 모바일에서는 보드 최적화를 제공하지 않고 팀 이슈 목록으로 안내한다.

### 8.12 PROJ-01 프로젝트 목록

**헤더**

- 프로젝트 만들기
- 상태 필터
- 보관된 프로젝트 포함 선택

**프로젝트 항목**

- 이름과 설명 요약
- 상태
- 리드
- 백엔드, 웹 프론트, 앱 프론트 역할별 팀
- 시작일과 목표일
- 완료한 팀 작업 수와 전체 대상 팀 작업 수
- 진행률

진행률 분모가 0이면 0%로 표시한다. 프로젝트 상태와 진행률이 다를 때 둘 다 그대로 보여 주며 자동으로 맞추지 않는다.

**빈 상태**

- 프로젝트가 없음: 프로젝트의 용도와 프로젝트 만들기 제공
- 필터 결과 없음: 필터 초기화 제공
- 활성 프로젝트가 없고 보관만 있음: 보관된 프로젝트 보기 제공

### 8.13 PROJ-02 프로젝트 생성·편집

프로젝트 목록에서 여는 단일 폼으로 구성한다.

**기본 정보**

- 이름: 필수
- 설명: 선택, 여러 줄 일반 텍스트
- 상태: 생성 시 계획
- 리드: 활성 멤버 한 명 또는 없음
- 시작일, 목표일: 선택

**역할별 팀**

| 역할 | 입력 |
| --- | --- |
| 백엔드 담당 팀 | 활성 팀 한 개 또는 선택 안 함 |
| 웹 프론트 담당 팀 | 활성 팀 한 개 또는 선택 안 함 |
| 앱 프론트 담당 팀 | 활성 팀 한 개 또는 선택 안 함 |

최소 한 역할이 필요하다. 같은 팀을 여러 역할에 선택할 수 있다. 선택하지 않은 역할은 프로젝트 상세의 생성 동작, 진행률과 전달 알림 대상에서 제외한다.

**편집 제한**

- 역할에 연결된 팀 작업이 없으면 팀 변경 또는 역할 해제가 가능하다.
- 연결된 작업이 있으면 저장을 막고 영향을 받는 작업 목록을 제공한다.
- 기존 작업을 제거하거나 새 팀의 작업으로 다시 만든 뒤 변경하도록 안내한다.
- 프로젝트에 연결된 이슈가 있으면 삭제를 제공하지 않고 보관을 안내한다.

### 8.14 PROJ-03 프로젝트 상세

**상단 요약**

- 프로젝트 이름, 상태, 리드
- 시작일과 목표일
- 역할별 팀
- 완료 작업 수와 진행률
- 편집, 보관과 조건부 삭제

**본문 기본 보기**

프로젝트의 결과물과 팀 간 작업 순서를 한 번에 파악할 수 있도록 기능 이슈 중심 계층을 기본 보기로 사용한다.

- 기능 이슈를 상위 행으로 표시한다.
- 각 기능 이슈 아래에 역할별 팀 작업을 계층으로 표시한다.
- 상위 기능 이슈가 없는 프로젝트 팀 작업은 별도의 단독 작업 영역에 표시한다.
- 기능 이슈 행에는 상태와 하위 작업 진행률을 표시한다.
- 팀 작업 행에는 역할, 팀, 상태, 담당자, 우선순위와 차단 여부를 표시한다.
- 역할 필터로 백엔드, 웹 프론트와 앱 프론트 작업을 좁힐 수 있다.

**주요 동작**

- 기능 이슈 만들기
- 기능 이슈 아래 팀 작업 만들기
- 단독 팀 작업 만들기
- 기존 팀 작업을 프로젝트에 추가하거나 제거
- 프로젝트 상태, 리드와 일정 변경
- 이슈 상세 열기

기존 단독 팀 작업을 프로젝트에 추가할 때는 작업의 팀과 일치하는 프로젝트 역할이 있어야 한다. 같은 팀이 여러 역할을 맡으면 추가 시 역할을 선택한다. 프로젝트에서 작업 제거는 이슈 삭제가 아니다. 기능 이슈의 하위 작업은 부모와 프로젝트 연결을 함께 유지해야 하므로 개별적으로 다른 프로젝트로 옮기지 않는다.

### 8.15 ISSUE-01 이슈 상세

MVP에서는 데스크톱과 모바일 모두 독립 화면으로 연다. 이전 목록의 필터, 정렬과 스크롤 위치를 복원해 맥락 이탈을 줄인다.

**공통 상단**

- 표시 ID와 이슈 유형
- 제목
- 상태
- 우선순위
- 프로젝트와 역할
- 라벨
- 더보기: 삭제 등 조건부 동작

**기능 이슈 변형**

- 프로젝트 필수
- 팀과 담당자 미표시
- 워크스페이스 고정 상태 사용
- 하위 팀 작업 목록과 진행률 표시
- 프로젝트에서 활성화한 역할별 팀 작업 추가

**팀 작업 변형**

- 팀 필수, 읽기 전용
- 담당자는 해당 팀 활성 멤버 중 선택
- 팀별 워크플로 상태 사용
- 상위 기능 이슈가 있으면 같은 프로젝트 맥락 표시
- 차단함, 차단됨과 차단 해제됨 관계 표시
- 백엔드 역할 작업이면 작업 전달 영역 표시

**설명**

- Lexical WYSIWYG 편집과 Markdown 미리보기 전환
- 이미지 선택과 Command/Ctrl+V 붙여넣기, 파일별 업로드 진행·실패·재시도
- 저장과 취소
- 저장 전 다른 곳으로 이동하면 작성 내용 유실 경고
- 위험한 HTML과 링크 속성 제거

**관계**

- 상위 또는 하위 작업
- 차단함과 차단됨
- 새 관계 검색은 같은 워크스페이스 이슈만 대상
- 자기 자신과 순환 차단 관계는 선택 단계 또는 저장 단계에서 거부
- 완료된 선행 관계는 차단 해제됨으로 유지

**작업 전달**

- 백엔드 프로젝트 역할의 팀 작업에만 표시
- 최초 전달과 추가 전달을 시간순으로 구분
- 최초 전달 전에는 템플릿으로 작성
- 완료 후에는 기존 전달 편집 대신 추가 전달만 제공
- API 명세 링크를 눈에 띄게 표시하되 외부 링크임을 구분

**댓글과 활동**

- 설명 아래에 댓글과 활동을 시간순으로 함께 표시한다.
- 댓글은 작성자, 작성 시각과 수정 여부를 표시한다.
- 활동은 행위자, 시각, 변경 필드와 전후 값을 표시한다.
- 댓글 입력은 Lexical WYSIWYG 편집과 Markdown 미리보기를 제공한다.
- 댓글 이미지도 설명과 같은 업로드·미리보기·저장 규칙을 사용한다.
- 삭제된 댓글은 본문 대신 삭제된 댓글임을 표시한다.
- 알림에서 진입하면 관련 댓글 또는 전달 위치로 이동하고 강조한다.

**첨부파일**

- 일반 첨부파일은 설명과 분리된 영역에 파일명, 형식, 크기, 업로더와 업로드 시각을 표시한다.
- 여러 파일을 추가할 수 있고 파일 하나는 최대 25MB다.
- 검증된 이미지는 화면에서 열 수 있고 그 밖의 파일은 다운로드한다.
- 업로드 중에는 이탈 확인을 제공하고 실패 파일은 입력을 유지한 채 다시 시도하거나 제거한다.
- 같은 워크스페이스의 활성 멤버만 조회·다운로드할 수 있다.

**삭제 제한**

- 하위 작업이 있는 기능 이슈는 삭제 불가
- 다른 작업을 차단 중인 팀 작업은 삭제 불가
- 제한 사유와 정리할 관계를 표시
- 삭제 가능하면 30일 휴지통 이동을 확인

- 이슈 표시 ID와 제목
- 변경 요약
- 발생 시각

**알림 유형**

- 팀 작업 할당
- 설명 또는 댓글 멘션
- 구독 이슈의 새 댓글
- 구독 이슈의 완료 또는 취소
- 작업 전달
- 추가 전달

**동작**

- 항목 선택 시 관련 이슈 위치로 이동하고 읽음 처리
- 개별 읽음·읽지 않음 전환
- 모두 읽음 처리
- 전역 배지와 읽지 않은 개수 동기화

**빈 상태**

- 알림이 없음: 새 알림이 생길 조건을 간단히 안내
- 읽지 않은 알림이 없음: 모든 알림 보기 제공

### 8.17 SEARCH-01 전역 검색

전역 검색은 별도 영구 페이지보다 빠른 검색 창을 기본으로 한다. 직접 주소가 필요하면 현재 검색어를 쿼리 문자열로 보존할 수 있다.

- 빈 입력은 검색 가능한 범위를 안내하는 빈 상태로 두고 최근 열어본 이슈는 표시하지 않는다.
- 두 글자부터 제목 부분 일치 결과를 표시한다.
- 표시 ID는 한 글자 입력부터 정확 일치를 시도할 수 있다.
- 키보드로 결과 선택과 닫기가 가능하다.
- 결과를 연 뒤 뒤로 가면 검색어와 선택 위치를 복원한다.

### 8.18 SET-01 멤버

- 활성, 초대 대기와 비활성 멤버를 구분한다.
- 활성·비활성 멤버는 표시 이름을 기본값으로, 이메일을 보조 정보로 표시한다.
- 이메일 초대, 재초대와 초대 상태 확인을 제공한다.
- 활성 멤버를 팀에 추가하는 경로를 제공한다.
- 멤버에게 할당된 미완료 팀 작업이 있으면 비활성화를 막고 작업 목록, 재할당과 담당 해제 경로를 제공한다.
- 미완료 팀 작업이 모두 정리된 뒤에만 비활성화를 실행하고 해당 멤버의 로그인 세션을 종료한다.
- 비활성 멤버의 과거 이슈와 댓글 작성자 표시는 유지한다.

### 8.19 SET-02 팀

- 활성 팀과 보관된 팀을 구분한다.
- 팀 이름, 키, 멤버 수와 미완료 작업 수를 표시한다.
- 팀 생성 시 이름, 영문 대문자 키와 초기 멤버를 입력한다.
- 이슈가 생성된 팀의 키는 읽기 전용으로 표시하고 이유를 안내한다.
- 미완료 작업 담당자를 팀에서 제거하려 하면 해당 작업 목록과 재할당 경로를 제공한다.
- 미완료 팀 작업이 있으면 팀 보관을 막고 작업 수를 표시한다.

### 8.20 SET-03 워크플로

- 선택한 팀의 상태를 시스템 범주별로 나누어 표시한다.
- 상태 이름과 표시 순서를 변경한다.
- 시스템 범주는 읽기 전용이며 사용자에게는 의미 설명을 제공한다.
- 사용 중인 상태를 삭제하면 대체 상태 선택을 필수로 요구한다.
- 상태 순서 변경은 워크플로 열 순서이며 이슈의 수동 정렬 기능이 아니다.

### 8.21 SET-04 라벨

- 활성 라벨과 보관된 라벨을 구분한다.
- 이름과 색상을 생성·수정한다.
- 활성 라벨 이름 중복을 막는다.
- 사용 중인 라벨은 보관을 기본 동작으로 제공한다.
- 보관된 라벨은 기존 이슈에 남고 새 선택 목록에서는 제외됨을 안내한다.

### 8.22 SET-05 데이터 내보내기

- 관리자만 접근한다.
- 이슈 CSV와 프로젝트 CSV의 포함 필드를 미리 설명한다.
- 내보내기 실행 시 생성 중, 완료와 실패 상태를 표시한다.
- 완료 파일에 설명 또는 작업 전달처럼 민감한 업무 내용이 포함될 수 있음을 실행 전에 안내한다.
- CSV 가져오기는 제공하지 않는다.

### 8.23 SET-06 휴지통

- 이슈와 프로젝트를 유형별로 필터링한다.
- 이름 또는 제목, 표시 ID, 삭제한 사용자, 삭제일과 영구 삭제 예정일을 표시한다.
- 관리자만 복구할 수 있다.
- 복구 전 원래 팀이나 프로젝트가 보관된 경우의 처리 결과를 안내한다.
- 영구 삭제를 즉시 실행하는 기능은 MVP에서 제공하지 않고 보존 기간 만료를 따른다.

### 8.24 PROFILE-01 프로필 설정

- 사용자 메뉴에서 모달로 열며 관리자 여부와 관계없이 모든 활성 멤버가 접근한다.
- 현재 프로필 사진 또는 표시 이름 첫 글자 아바타를 보여 준다.
- `사진 변경`, `사진 삭제`, `닫기`를 제공하며 표시 이름·이메일 변경은 MVP에 포함하지 않는다.
- 허용 형식과 최대 25MB를 선택 전에 안내하고 프론트 최적화 결과가 25MB를 넘으면 저장하지 않는다.
- 업로드 실패 시 기존 사진을 유지하고 다시 시도할 수 있게 한다.

## 9. 공통 구성 요소와 동작

### 9.1 Lexical 편집기

- `편집`과 `미리보기` 두 모드를 제공하며 Markdown 원문 편집 모드는 제공하지 않는다.
- 편집 모드는 Lexical 기반 WYSIWYG UI를 사용하고 `@lexical/markdown`의 단축 입력과 변환 규칙을 적용한다.
- 서버와 API의 정본은 Markdown 문자열이며 Lexical JSON 편집 상태는 저장하거나 전송하지 않는다.
- 설명, 댓글과 작업 전달은 같은 Markdown 변환·렌더링 규칙을 사용하고 미리보기는 현재 편집 내용을 Markdown으로 변환해 표시한다.
- MVP 서식은 제목, 굵게, 기울임, 취소선, 링크, 인용, 순서·비순서 목록, 인라인 코드와 코드 블록으로 제한한다.
- 사용자 정의 이미지 노드와 Markdown 변환 규칙을 사용해 업로드된 이미지를 인증된 파일 주소로 저장한다.
- 파일 선택과 Command/Ctrl+V를 지원한다. 클립보드에 실제 이미지 파일이 없고 원격 HTML 또는 URL만 있으면 서버가 대신 가져오지 않는다.
- JPEG·PNG·WebP 이미지는 2MB 초과 또는 긴 변 2,560px 초과일 때 브라우저에서 비율을 유지해 최대 2,560px, WebP 품질 82%를 기준으로 최적화한다. 변환 결과가 더 크면 원본을 사용한다.
- 업로드 중에는 미리보기와 진행 상태를 표시하고, 실패 시 다시 시도·제거를 제공하며 완료되지 않은 파일이 있으면 저장을 막는다.
- 멘션은 사용자 정의 노드와 변환 규칙으로 처리하며 자동완성에는 활성 워크스페이스 멤버만 표시한다.
- 작업 전달은 기본 템플릿을 삽입하며 템플릿 제목만 남은 내용은 유효한 전달로 인정하지 않는다.
- 스크립트, 허용하지 않은 HTML, 위험한 링크 속성과 임의 임베드를 제거한다.

### 9.2 이슈 행과 카드

필수 표현은 표시 ID, 제목, 상태다. 화면 목적에 따라 팀, 역할, 담당자, 우선순위, 라벨, 프로젝트와 차단 배지를 추가한다. 색상만으로 상태나 우선순위를 구분하지 않고 이름 또는 아이콘과 함께 표시한다.

### 9.3 필터

- 같은 필드의 여러 값은 OR로 적용한다.
- 다른 필드끼리는 AND로 적용한다.
- 활성 필터는 화면 헤더에 요약하고 한 번에 초기화할 수 있다.
- 목록과 보드를 전환해도 유지한다.
- 저장된 보기는 MVP에서 제공하지 않는다.

### 9.4 정렬

- 이슈 화면은 생성일, 최근 수정일, 상태와 우선순위만 제공한다.
- 오름차순과 내림차순을 선택한다.
- 보드 열 안에서도 선택한 정렬 기준을 따른다.
- 사용자가 드래그해 임의 순서를 저장하는 기능은 제공하지 않는다.

### 9.5 진행률

- 기능 이슈: 취소된 하위 팀 작업을 제외한 완료 작업 수 ÷ 대상 하위 팀 작업 수
- 프로젝트: 취소된 연결 팀 작업을 제외한 완료 작업 수 ÷ 대상 연결 팀 작업 수
- 대상 작업이 없으면 0%
- 기능 이슈 자체를 프로젝트 진행률에 별도 항목으로 더하지 않음
- 상태와 진행률이 일치하지 않아도 자동 변경하지 않음

### 9.6 확인창

삭제, 보관, 비활성화와 관계 해제는 다음을 표시한다.

- 대상 이름과 유형
- 즉시 발생하는 변화
- 복구 가능 여부
- 작업을 막는 연결 관계
- 취소와 명확한 실행 버튼

| 세션 만료 | 현재 주소와 작성 중 내용 보존을 시도 | 로그인 후 복귀 |

Lexical 편집기에서 작성 중인 긴 내용은 네트워크 오류만으로 즉시 사라지지 않게 현재 브라우저 세션에서 보존한다. 장기 오프라인 저장과 여러 기기 초안 동기화는 제공하지 않는다.

## 11. 모바일 웹

### 11.1 지원

- 내 이슈와 팀 이슈 목록 조회
- 이슈와 프로젝트 상세 조회
- 이슈 첨부파일 조회·업로드·다운로드
- 사용자 메뉴의 프로필 사진 변경·삭제
- 댓글 작성
- 상태, 담당자와 우선순위 변경
- 알림함 확인과 읽음 처리
- 전역 검색

### 11.2 제한

- 팀 상태 보드는 목록으로 대체한다.
- 팀, 워크플로, 멤버와 워크스페이스 설정은 데스크톱 사용을 안내한다.
- 프로젝트 생성·역할별 팀 편집은 지원하지 않고 데스크톱 사용을 안내한다.
- 차단 관계 생성과 작업 전달 작성·추가 전달은 지원하지 않고 조회만 제공한다.

### 11.3 모바일 탐색

좁은 화면에서는 다음 5개 항목을 고정 하단 탐색으로 제공한다.

| 항목 | 동작 |
| --- | --- |
| 내 이슈 | 현재 사용자에게 할당된 미완료 팀 작업으로 이동 |
| 알림함 | 알림 목록으로 이동하고 읽지 않은 개수를 함께 표시 |
| 팀 | 마지막으로 방문한 팀 이슈 목록으로 이동하고 방문 기록이 없으면 팀을 선택 |
| 프로젝트 | 프로젝트 목록으로 이동 |
| 검색 | 전역 검색을 열고 입력란에 포커스 |

각 항목은 아이콘과 한국어 이름을 함께 표시하고 현재 위치를 구분한다.

모바일 상단 앱 바의 프로필 사진 또는 첫 글자 아바타를 선택하면 사용자 메뉴를 열고 프로필 설정과 로그아웃을 제공한다.

## 12. 키보드와 접근성

### 12.1 키보드 동작

아래는 MVP 기본안이며 실제 구현한 동작만 단축키 도움말에 표시한다.

| 키 | 동작 |
| --- | --- |
| C | 이슈 만들기 |
| / | 전역 검색 |
| Escape | 현재 팝오버 또는 창 닫기 |
| Command/Ctrl + Enter | 생성 또는 댓글 저장 |
| Shift + S | 선택한 이슈의 상태 변경 |
| Shift + A | 선택한 팀 작업의 담당자 변경 |
| Shift + P | 선택한 이슈의 우선순위 변경 |
| Shift + L | 선택한 이슈의 라벨 변경 |
| ? | 단축키 도움말 |

텍스트 입력과 Markdown 편집 중에는 단일 문자 단축키를 실행하지 않는다.

### 12.2 접근성 기준

- 모든 입력은 연결된 이름과 오류 설명을 가진다.
- 팝오버와 모달을 열면 첫 조작 요소로 포커스를 이동한다.
- 모달을 닫으면 열었던 버튼 또는 합리적인 다음 위치로 포커스를 돌린다.
- 현재 선택, 읽지 않음, 차단 여부와 저장 중 상태를 보조 기술에도 전달한다.
- 상태, 우선순위와 진행률은 색상만으로 표현하지 않는다.
- 아이콘 단독 버튼에는 한국어 접근 가능 이름을 제공한다.
- 보드 드래그와 같은 동작은 키보드로 상태를 변경할 대체 경로를 제공한다.

## 16. 확정된 UX 결정

PRD를 변경하지 않고 화면을 구체화하기 위해 다음 결정을 확정했다.

| 번호 | 결정 | 상태 | 결정 영향 |
| ---: | --- | --- | --- |
| Q1 | 이슈 상세를 데스크톱과 모바일에서 독립 화면으로 연다. | 확정 | 목록 맥락 유지 방식과 구현 복잡도 |
| Q2 | 프로젝트 상세의 기본 구조는 기능 이슈 중심 계층이고 역할은 필터와 배지로 표현한다. | 확정 | 여러 팀의 진행 상황을 읽는 방식 |
| Q3 | 초기 설정에서 멤버 초대는 건너뛸 수 있다. | 확정 | 1인 팀 시작 흐름 |
| Q4 | 전역 검색의 빈 입력은 최근 이슈 없이 빈 상태로 시작한다. | 확정 | 최근 기록 저장 범위와 검색 시작 경험 |
| Q5 | 모바일 주 탐색은 내 이슈, 알림함, 팀, 프로젝트와 검색으로 구성한 하단 탐색을 사용한다. | 확정 | 모바일 정보 구조 |
| Q6 | MVP, Alpha와 Beta에서는 하나의 계정이 하나의 워크스페이스만 생성하거나 참여한다. | 확정 | 가입·초대·로그인과 데이터 경계 |
| Q7 | 프로필 사진은 별도 설정 화면 대신 사용자 메뉴의 프로필 설정 모달에서 관리한다. | 확정 | 모든 멤버의 접근성과 MVP 화면 수 |
| Q8 | 편집기 이미지는 파일 선택과 Command/Ctrl+V를 지원하고 업로드 완료 전에는 본문을 저장하지 않는다. | 확정 | 고아 파일, 실패 복구와 Markdown 정본 |

확정된 흐름과 화면 구조를 기준으로 와이어프레임과 디자인 시스템 문서를 작성한다.
