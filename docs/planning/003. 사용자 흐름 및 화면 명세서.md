## 1. 문서 목적

이 문서는 PRD에서 확정한 요구사항을 사용자가 실제로 이동하고 조작하는 흐름과 화면 단위 명세로 전환한다. 각 화면의 목적, 진입 조건, 핵심 정보, 주요 행동, 유효성 검사, 빈 상태와 실패 상태를 정의해 와이어프레임, 디자인 시스템, 프론트엔드 구현과 API 설계가 같은 제품 구조를 기준으로 진행되게 한다.

이 문서는 시각 디자인을 확정하지 않는다. 저장소 루트의 [DESIGN.md](../../DESIGN.md)는 Rivet의 상위 디자인 방향이며, 정확한 레이아웃과 시각 값은 이 문서의 범위에 포함하지 않는다. Linear 벤치마킹 원본은 [Linear 시각 디자인 분석](../reference/001.%20Linear%20시각%20디자인%20분석.md)에 참고 자료로 보존하며 현재 제품 기준으로 직접 사용하지 않는다.

### 1.1 명세 범위

- 가입, 인증, 로그인과 초기 워크스페이스 설정
- 전역 이슈, 내 작업, 팀 작업 목록과 상태 보드
- 프로젝트 생성, 역할별 팀 선택과 프로젝트 상세
- 이슈 콘텐츠, 팀 실행 슬롯과 선택적 작업 전달
- 백엔드 작업 전달과 웹·앱 작업 가능 전환
- Lexical 기반 설명·댓글·작업 전달 편집, 이미지 붙여넣기, Markdown 미리보기와 활동 기록
- 이슈 첨부파일과 사용자 프로필 사진
- 전역 검색, 인앱 알림함과 빠른 조작
- 멤버, 팀, 워크플로, 라벨, CSV 내보내기·가져오기와 휴지통 설정
- 데스크톱과 제한적 모바일 웹 동작

### 1.2 해석 원칙

- PRD와 충돌하면 PRD를 우선한다.
- PRD에서 정하지 않은 핵심 UX 세부사항은 관련 사용자 흐름과 화면 동작에 직접 반영한다.
- 화면의 기본 용어인 이슈는 결과물의 범위, 팀 작업은 실제 실행 책임을 표현한다. 사용자에게 이슈 유형을 선택하거나 내부 분류를 노출하지 않는다.
- 팀 이름을 상태에 넣지 않고 팀, 프로젝트 역할, 팀 워크플로 상태와 현재 주요 행동을 분리해 보여 준다.
- 빠른 생성과 속성 변경을 우선하고 부가 설정은 설정 화면으로 분리한다.

## 2. UX 원칙

### 2.1 한국어 우선

사용자에게 보이는 메뉴, 상태, 오류와 도움말은 한국어를 기본으로 한다. 시스템 범주와 내부 이벤트 이름은 화면에 직접 노출하지 않는다.

### 2.2 한 화면에서 한 가지 중심 작업

- 내 작업은 내가 처리할 팀 작업을 찾고 상태를 바꾸는 데 집중한다. 전역 이슈는 접수·검토·분류와 배정에 집중한다.
- 팀 작업 화면은 한 팀의 실행 레코드를 목록 또는 보드로 운영하는 데 집중한다.
- 프로젝트는 여러 팀이 함께 만드는 결과물과 진행률을 확인하는 데 집중한다.
- 이슈 상세는 `업무`, `전달`, `활동`을 분리해 지금 처리할 내용, 전달 이력과 변경 기록을 각각 이해하는 데 집중한다.

### 2.3 속도와 맥락 보존

- 모든 주요 화면에서 같은 이슈 생성 창과 전역 검색을 연다.
- 목록의 상태, 담당자와 우선순위는 상세 화면을 열지 않고 변경한다.
- 상세 화면에서 이전 화면으로 돌아오면 필터, 정렬, 선택 탭과 스크롤 위치를 복원한다.
- 저장 가능한 변경은 먼저 화면에 반영하되 저장 중임을 구분하고 실패하면 이전 값으로 되돌린다.

### 2.4 관계의 명시성

- 실제로 생성된 프론트 작업은 백엔드와 함께 시작했더라도 즉시 작업 가능하며, 백엔드 뒤에 이어지는 프론트 작업은 전달 완료 시점에 생성하거나 재사용한다.
- 프로젝트 역할과 실제 담당 팀을 함께 표시한다.
- 같은 팀이 여러 프로젝트 역할을 맡으면 역할 배지로 작업의 책임을 구분한다.

### 2.5 안전한 파괴 작업

- 삭제, 보관, 멤버 비활성화처럼 영향이 큰 동작은 결과를 먼저 설명한다.
- 해결해야 할 연결 관계가 있으면 확인창만 띄우지 않고 차단 사유와 이동할 화면을 제공한다.
- 30일 휴지통 복구 가능 여부와 영구 삭제 예정일을 명확히 표시한다.

## 3. 사용자와 권한

| 사용자 | 할 수 있는 일 | 제한되는 일 |
| --- | --- | --- |
| 비로그인 사용자 | 가입, 이메일 인증, 로그인, 비밀번호 재설정, 초대 수락 시작 | 워크스페이스 업무 데이터 접근 |
| 멤버 | 공개된 팀·이슈·프로젝트 조회, 이슈·프로젝트 생성과 수정, 댓글 작성 | 멤버·팀·워크플로·라벨 관리, 내보내기, 휴지통 복구 |
| 관리자 | 멤버의 모든 동작과 워크스페이스 운영 설정 | 다른 워크스페이스 데이터 접근 |
| 비활성 멤버 | 로그인 화면 접근과 계정 안내 확인 | 로그인 후 신규 업무 활동 |

하나의 계정은 최대 하나의 워크스페이스에만 참여한다. 워크스페이스에 참여한 뒤에는 새 워크스페이스를 만들거나 다른 워크스페이스의 초대를 수락할 수 없다.

권한이 없는 기능은 기본적으로 숨긴다. 직접 주소로 접근하면 권한 부족 화면을 표시하고 접근 가능한 이전 화면으로 이동할 수 있게 한다. 존재 여부 자체가 민감한 다른 워크스페이스 리소스는 찾을 수 없음과 같은 응답으로 처리한다.

### 4.3 로그인 후 기본 진입

- 정상 로그인: 내 작업
- 처음 로그인했고 워크스페이스가 없음: 워크스페이스 생성
- 초대 링크로 가입 또는 로그인: 초대 수락 후 해당 워크스페이스의 내 작업
- 워크스페이스에 참여한 사용자: 해당 워크스페이스로 바로 진입
- 세션이 만료된 사용자: 로그인 후 원래 접근하려던 내부 주소로 복귀

## 5. 공통 애플리케이션 구조

### 5.1 데스크톱 셸

| 영역 | 내용 | 동작 |
| --- | --- | --- |
| 워크스페이스 헤더 | 워크스페이스 이름, 사용자 메뉴 | 워크스페이스 전환 없이 프로필 사진 변경·삭제와 로그아웃 제공 |
| 주 탐색 | 이슈, 내 작업, 알림함, 프로젝트 | 이 순서를 유지하고 현재 위치 강조, 알림함 읽지 않음 개수 표시 |
| 팀 탐색 | 활성 팀 목록 | 팀 선택 시 마지막으로 사용한 목록 또는 보드로 이동 |
| 전역 동작 | 이슈 만들기, 검색 | 모든 업무 화면에서 고정된 위치와 단축키 제공 |
| 본문 헤더 | 화면 제목, 보기 전환, 필터, 정렬, 화면별 주요 동작 | 스크롤 중에도 핵심 조작에 접근 가능 |
| 본문 | 목록, 보드, 프로젝트 또는 상세 | 점진적 로딩과 화면별 빈 상태 제공 |

관리자에게만 설정 진입점을 표시한다. 보관된 팀은 팀 탐색에 표시하지 않는다.

### 5.2 전역 이슈 만들기

전역 생성 창의 제목과 주요 버튼은 `이슈 만들기`이며 유형 선택기를 표시하지 않는다. 모든 팀 작업은 생성된 이슈의 역할 선택·작업 시작·전달 흐름에서만 만든다. 팀 화면과 프로젝트의 보조 메뉴에도 독립 팀 작업 생성은 제공하지 않는다.

| 필드 | 필수 | 동작 |
| --- | :---: | --- |
| 제목 | 필수 | 창을 열면 바로 포커스 |
| 프로젝트 | 필수 | 활성 프로젝트만 표시 |
| 처음 작업할 팀 | 선택 | 프로젝트에 설정된 백엔드·웹·앱 역할을 복수 선택 |
| 상태 | 자동 | 선택기를 표시하지 않고 최초 팀 작업 유무에 따라 서버가 `접수됨` 또는 `할 일`로 저장 |
| 우선순위 | 선택 | 기본값 없음 |
| 라벨 | 선택 | 활성 라벨 여러 개 선택 |
| 설명 | 선택 | Lexical WYSIWYG 편집과 Markdown 미리보기를 기본으로 펼쳐서 제공 |
| 첨부파일 | 선택 | 여러 파일을 선택하며 파일당 최대 25MB와 업로드 상태 표시 |

시작 역할 선택은 실제 팀 ID를 받지 않으며 팀과 시작 전 상태는 서버가 프로젝트 설정에서 결정한다. 담당자, 이슈 상태와 팀별 워크플로 상태는 입력받지 않는다. 프로젝트 상세에서 열면 프로젝트는 현재 값으로 고정한다.

시작 역할은 선택하지 않아도 되고 여러 역할을 함께 선택할 수 있다. 저장하면 이슈와 선택 역할의 최초 팀 작업을 하나의 트랜잭션에서 만든다. 선택하지 않으면 이슈만 `접수됨`으로 만들며 팀 작업 사이의 수동 작업 순서는 제공하지 않는다. 생성 작업은 상위 이슈 ID, 역할 담당 팀, 역할, 시작 상태, 선택 담당자와 선택적인 Markdown 작업 노트만 저장한다. 제목·설명·첨부파일·우선순위·라벨·프로젝트는 상위 이슈에서 읽는다.

#### 저장 동작

- 저장 중 버튼을 중복 실행할 수 없게 한다.
- 성공하면 `이슈를 만들었습니다` 한 건만 알리고 새 이슈 상세의 기본 `업무` 탭으로 이동한다.
- 이슈나 선택한 최초 팀 작업 하나라도 만들지 못하면 전체를 롤백하고 입력을 유지한다.
- 필수값 누락은 해당 필드 가까이 설명하고 첫 오류 필드로 포커스를 이동한다.
- 본문 이미지와 첨부파일 업로드가 모두 완료된 뒤 이슈 저장을 실행한다. 업로드 실패 파일이 있으면 입력을 유지하고 다시 시도하거나 제거하게 한다.

### 5.3 전역 검색

- 검색 창은 현재 화면 위에 겹쳐 열고 입력란에 즉시 포커스한다.
- 검색어를 입력하기 전에는 검색 안내만 있는 빈 상태를 표시하고 최근 열어본 이슈는 제공하지 않는다.
- 두 글자 이상 입력하면 표시 ID와 제목을 검색한다.
- 정확한 표시 ID 일치를 가장 위에 표시한다.
- 결과는 이슈와 팀 작업, 표시 ID, 이슈 제목, 프로젝트, 팀 또는 역할, 상태를 구분해 보여 준다. 팀 작업 결과에도 상위 이슈 제목을 함께 표시한다.
- 위·아래 방향키로 결과를 이동하고 Enter로 상세를 연다.
- 결과가 없으면 검색 가능한 범위가 ID와 제목임을 안내한다.
- 다른 워크스페이스 데이터와 휴지통의 이슈는 표시하지 않는다.

### 5.4 공통 속성 선택기

상태, 담당자, 우선순위, 라벨과 프로젝트 선택기는 목록과 상세에서 같은 이름, 순서와 의미를 사용한다. 선택 직후 낙관적으로 반영하고 다음 상태를 표시한다.

- 저장 중: 해당 속성에 작은 진행 표시
- 저장 성공: 별도 성공 알림 없이 값 유지
- 저장 실패: 이전 값으로 복구하고 다시 시도 제공
- 충돌: 서버의 최신 값을 보여 주고 내 변경을 다시 적용할지 선택

## 6. 핵심 사용자 흐름

### 6.1 UF-01 회원가입과 첫 설정

| 단계 | 사용자 행동 | 화면·시스템 반응 |
| ---: | --- | --- |
| 1 | 표시 이름, 이메일, 비밀번호와 비밀번호 확인 입력 | 표시 이름의 필수·길이 오류, 이메일 중복과 비밀번호 정책 오류를 해당 필드에 표시 |
| 2 | 회원가입 실행 | 인증 메일 발송 후 이메일 인증 안내로 이동 |
| 3 | 메일의 인증 링크 실행 | 유효한 링크면 인증 완료, 만료·사용 완료 링크면 재발송 안내 |
| 4 | 로그인 | 워크스페이스가 없으면 워크스페이스 생성으로 이동 |
| 5 | 워크스페이스 이름과 슬러그 입력 | 웹에서 형식과 주소를 미리 확인하고 생성 요청으로 중복을 최종 판정하며 생성자는 관리자가 됨 |
| 6 | 기본 팀 이름과 키 입력 | 한국어 기본 워크플로를 함께 생성 |
| 7 | 동료 이메일 입력 또는 건너뛰기 | 초대 발송 결과를 주소별로 표시 |
| 8 | 첫 이슈 만들기 | 필요하면 처음 작업할 팀을 선택하고 생성한 이슈 상세로 이동 |

초기 멤버 초대는 1인 팀을 고려해 건너뛸 수 있다. 기본 팀 생성은 건너뛸 수 없다.

### 6.2 UF-02 초대 수락

| 조건 | 흐름 |
| --- | --- |
| 가입하지 않은 이메일 | 초대 확인 → 회원가입 → 이메일 인증 → 로그인 → 원래 초대 메일 링크 재진입 → 초대 수락 → 내 작업 |
| 가입했지만 로그아웃 | 초대 확인 → 로그인 → 원래 초대 메일 링크 재진입 → 초대 이메일과 계정 이메일 확인 → 수락 → 내 작업 |
| 로그인한 동일 이메일 | 초대 확인 → 수락 → 내 작업 |
| 이미 수락한 초대 | 이미 참여 중임을 안내하고 워크스페이스로 이동 |
| 다른 워크스페이스에 참여 중인 계정 | 초대 수락을 막고 MVP의 1계정 1워크스페이스 제한을 안내 |
| 만료·취소된 초대 | 사용할 수 없음을 안내하고 관리자에게 재초대를 요청하도록 표시 |

초대 링크는 `/invite#token=...` 형식을 사용한다. 화면은 토큰을 메모리로 읽은 뒤 주소에서 즉시 제거하고 브라우저 저장소나 쿠키에 보관하지 않는다. 인증이 필요한 사용자는 가입·인증·로그인 후 원래 초대 메일의 링크를 다시 연다.

### 6.3 UF-03 일상 이슈 시작과 팀 작업 완료

1. 사용자가 팀 화면에서 `이슈 만들기`를 연다.
2. 프로젝트를 선택하면 현재 팀이 담당하는 프로젝트 역할을 시작 역할 후보로 우선 표시한다.
3. 이슈 제목·설명·우선순위·라벨·첨부파일과 필요하면 시작 역할을 입력한다.
4. 서버는 이슈와 선택 역할의 최소 팀 작업을 한 트랜잭션에서 생성한다.
5. 담당자 없는 새 팀 작업은 기본 `백로그`, 담당자가 지정된 새 팀 작업은 `시작 전`으로 표시한다.
6. 기본 `백로그` 작업에 담당자를 선택하면 담당자와 `시작 전` 전이를 한 번에 저장한다. 이후 담당자 해제·교체는 현재 상태를 자동으로 되돌리지 않는다.
7. 행을 열면 상위 이슈 상세에서 현재 팀 작업이 선택되어 이슈 설명·자료와 실행 속성을 함께 확인한다.
8. `시작 전`의 주요 버튼 `작업 시작`을 누르면 `진행 중`으로 바뀌고 같은 버튼 자리는 `완료`로 전환된다.
9. 팀 작업을 완료하면 같은 자리에 `완료됨`을 표시하고 기본 미완료 보기에서 제외하며 상위 이슈 상태와 진행률을 다시 계산한다.

이슈 생성이나 `작업 시작`에서 백엔드와 프론트를 함께 선택하면 모든 팀 작업을 병렬로 만들고 즉시 작업 가능하게 한다. 백엔드 뒤에 프론트가 이어지는 순차 흐름은 백엔드만 먼저 시작한 뒤 완료 과정에서 전달 대상을 선택해 만든다.

### 6.4 UF-04 프로젝트 생성

1. 프로젝트 목록에서 프로젝트 만들기를 선택한다.
2. 이름을 입력하고 필요하면 설명, 리드, 시작일과 목표일을 입력한다.
3. 백엔드, 웹 프론트, 앱 프론트 역할별로 담당 팀을 선택하거나 비워 둔다.
4. 최소 한 역할에 활성 팀을 선택해야 저장할 수 있다.
5. 같은 팀을 여러 역할에 선택할 수 있으며 각 역할은 별도로 표시된다.
6. 저장하면 계획 상태로 프로젝트가 생성되고 프로젝트 상세로 이동한다.
7. 상세에서 첫 이슈를 만들고 필요한 팀 작업은 해당 이슈의 작업 흐름에서 시작한다.

목표일이 시작일보다 빠르면 저장을 막고 두 날짜 필드에 관계 오류를 표시한다.

### 6.5 UF-05 이슈와 최초 작업 흐름 시작

1. 프로젝트 상세에서 `이슈 만들기`를 선택한다.
2. 제목을 입력하고 필요하면 `처음 작업할 팀`에서 프로젝트 역할을 복수 선택한다. 이슈 상태는 입력하지 않는다.
3. 서버는 이슈와 선택 역할의 최초 팀 작업을 한 트랜잭션에서 생성한다.
4. 역할을 선택하지 않으면 이슈만 생성하고 설명·댓글·첨부파일로 분석을 진행한다.
5. 여러 역할을 선택하면 병렬 `현재 작업`으로 표시하고 백엔드·프론트 모두 즉시 작업 가능하게 한다.
6. 팀 작업 없는 이슈에서 `작업 시작`을 선택하면 프로젝트 역할을 복수 선택하고 없는 역할 작업만 원자적으로 만든다.
7. 이슈 상세의 `업무`에서 현재 작업과 최신 전달을 확인하고 `전달`에서 전체 이력을 확인한다.
8. 자동 흐름 밖 작업이 필요할 때만 더보기의 `팀 작업 추가`를 사용한다.

실제 생성된 미취소 팀 작업만 진행률에 포함하며 이슈 상태는 소속 팀 작업을 서버가 요약해 갱신한다. 팀 작업이 없으면 `접수됨`이며 `0/0 완료`를 강조하지 않고 `아직 시작된 팀 작업이 없습니다`, 분석 안내와 `작업 시작`을 표시한다.

### 6.6 UF-06 백엔드 작업 완료와 선택적 프론트 전달

1. `진행 중` 백엔드 작업에서 주요 버튼 `완료`를 선택한다.
2. 완료 창에서 `이 작업만 완료`와 `프론트에 전달 후 완료` 중 하나를 선택한다.
3. `이 작업만 완료`는 전달 입력 없이 백엔드 작업만 완료한다. 프로젝트에 프론트 역할이 있어도 이 선택을 막지 않는다.
4. `프론트에 전달 후 완료`를 선택한 경우에만 프로젝트의 웹·앱 대상 역할과 전달 입력을 펼친다.
5. 대상 역할이 하나면 고정하고 둘이면 모두 기본 선택하되 최소 하나를 남겨야 한다.
6. 저장하면 최초 전달, 백엔드 완료, 선택 역할의 프론트 작업 생성 또는 재사용, 활동과 Outbox를 한 트랜잭션에서 처리한다.
7. 같은 역할의 미완료 기존 작업이 있으면 재사용하고, 없으면 역할 팀의 기본 `Backlog` 상태·미할당 작업을 생성한다.
8. 최초 전달 대상 프론트 작업은 `api_handoff_targets`에 명시하고 즉시 작업 가능하게 한다.
9. 새 미할당 작업은 대상 팀 활성 멤버, 재사용 작업은 담당자와 자동 구독자에게 작성자를 제외한 알림을 수신자별 한 건 보낸다.
10. 성공하면 상위 이슈의 `업무` 탭에서 전달 원문과 갱신된 진행률을 즉시 표시한다.
11. 프론트 작업의 `업무` 탭에 있는 `받은 전달`에서도 최초 전달과 추가 전달을 확인한다.
12. 완료 후 변경이 생기면 추가 전달을 작성하고 기존 전달은 수정하지 않으며 새 팀 작업을 만들지 않는다.

작업 전달 편집기에는 멘션 선택기를 제공하지 않으며 멘션 참조가 포함된 본문은 저장하지 않는다.

작업 전달 템플릿은 다음 항목을 제공한다.

- 변경 요약
- API 명세 링크
- 사용 가능 환경
- 추가·변경 API
- 요청·응답 변경
- 오류·권한
- 프론트 주의사항

백엔드 전용·프론트 전용·병렬 작업은 모두 전달 없이 완료할 수 있다. 범위 불일치나 완료·취소된 기존 작업만 있는 충돌, 오래된 버전과 저장 실패는 입력한 완료 방식·본문·대상 역할을 유지하고 최신 이슈와 프로젝트 역할을 다시 조회한 뒤 사용자가 명시적으로 재실행하게 한다.

### 6.7 UF-07 내 작업 처리

1. 로그인 후 내 작업으로 진입한다.
2. 여러 팀에서 자신에게 할당된 미완료 팀 작업을 한 목록에서 확인한다.
3. 작업 코드, 이슈 표시 ID·제목과 프로젝트 이름으로 검색하고 상태·프로젝트를 필터링한다.
4. 기본 `실행 순서` 또는 우선순위·생성·최근 수정 기준으로 정렬한다.
5. 각 작업의 우선순위, 이슈 제목과 코드·역할, 프로젝트와 라벨, 실제 상태명과 현재 가능한 `작업 시작` 또는 `완료` 행동을 확인한다.
6. 목록에서 상태를 변경하거나 `작업 시작`·`완료`를 실행하고, 예외 상태는 보조 상태 메뉴에서 실제 워크플로 이름으로 변경한다.
7. 내 작업 상세로 들어가면 사이드바의 `내 작업` 활성 상태와 `내 작업` 복귀 동선을 유지한 채 해당 팀 작업의 상태·주요 행동을 먼저 확인한다.
8. 같은 화면에서 상위 이슈의 설명·첨부파일·댓글과 관련 전달을 공유 원본으로 확인한다.
9. 상세나 다른 화면에서 담당자가 변경되면 목록을 갱신하고 자신에게 할당되지 않은 행을 제거한다.
10. 완료 또는 취소된 작업은 내 작업 결과에서 제거한다.

### 6.8 UF-08 댓글, 멘션과 알림

1. 사용자가 이슈 상세의 `업무` 탭에 있는 Lexical WYSIWYG 편집기에서 댓글을 작성하고 필요하면 이미지를 선택하거나 Command/Ctrl+V로 붙여넣는다.
2. 이미지는 업로드 중 미리보기를 표시하고 성공하면 인증된 파일 참조로 바뀐다.
3. 저장될 Markdown 결과를 미리 보고 활성 멤버를 멘션해 댓글을 저장한다.
4. 멘션 대상과 자동 구독 대상 중 같은 사용자는 한 건의 알림만 받는다.
5. 수신자가 알림함에서 알림을 선택한다.
6. 댓글 알림은 `업무` 탭의 관련 댓글, 전달 알림은 전달받은 프론트 작업의 `업무`에서 정확한 전달을 확장해 연다.
7. 알림은 읽음 처리되고 전역 읽지 않음 개수가 갱신된다.

자신이 만든 이벤트에 대해서는 알림을 받지 않는다.

### 6.9 UF-09 삭제와 복구

1. 사용자가 이슈 또는 비어 있는 프로젝트의 더보기 메뉴에서 삭제를 선택한다.
2. 삭제 가능 조건을 확인한다.
3. 연결 관계 때문에 삭제할 수 없으면 정리할 관계와 이동 동작을 보여 준다.
4. 확인 후 일반 목록과 검색에서 제거하고 휴지통으로 이동한다.
5. 관리자는 휴지통에서 삭제일과 영구 삭제 예정일을 확인한다.
6. 복구하면 원래 연결을 유지한 채 일반 화면에 다시 나타난다.
7. 30일이 지나면 자동 영구 삭제되어 복구할 수 없다.

### 6.10 UF-10 동시 수정과 재연결

1. 사용자가 속성을 바꾸면 화면에 먼저 반영하고 저장 중으로 표시한다.
2. 서버 저장에 성공하면 진행 표시를 제거한다.
3. 실패하면 이전 값으로 복구하고 오류와 다시 시도를 보여 준다.
4. 다른 사용자가 같은 이슈를 변경하면 열린 목록과 상세에 반영한다.
5. 오래된 화면의 저장이 최신 변경과 충돌하면 자동 덮어쓰지 않는다.
6. 최신 값을 먼저 보여 주고 사용자가 자신의 변경을 다시 적용하게 한다.
7. 실시간 연결이 끊기면 연결 상태를 알리고 재연결 후 현재 화면 데이터를 다시 확인한다.
8. 생성 뒤 이슈 프로젝트 변경이 거부되면 같은 요청을 반복하지 않고 최신 상세를 다시 읽어 목록과 상세를 서버 값으로 복구한다. 팀 작업에는 프로젝트 편집기를 제공하지 않는다. 최신 조회가 실패하면 변경 재전송 없이 최신값 조회만 다시 제공한다.

### 6.11 UF-11 파일 첨부와 프로필 사진

**이슈 첨부파일**

1. 사용자가 이슈 생성 또는 상세의 첨부 영역에서 하나 이상의 파일을 선택한다.
2. 파일마다 이름, 형식과 크기를 표시하고 25MB를 넘으면 업로드 전에 거부한다.
3. JPEG·PNG·WebP 이미지가 2MB를 넘거나 긴 변이 2,560px을 넘으면 브라우저에서 비율을 유지해 축소·압축한다.
4. 업로드 중 파일별 진행 상태를 표시하고 성공한 파일은 열기·다운로드, 실패한 파일은 다시 시도·제거 동작을 제공한다.
5. 사용자가 첨부를 제거하면 화면에서 즉시 사라지고 연결이 해제된 실제 파일은 서버 정리 정책에 따라 제거된다.

**프로필 사진**

1. 사용자가 사용자 메뉴에서 `프로필 설정`을 연다.
2. JPEG·PNG·WebP 이미지를 선택하면 브라우저에서 긴 변을 최대 512px로 축소·압축하고 미리보기를 표시한다.
3. 저장하면 담당자, 멘션, 댓글과 활동의 사용자 표시에 새 사진이 반영된다.
4. 사진을 삭제하면 표시 이름 첫 글자 아바타로 돌아간다.

### 6.12 UF-12 전역 이슈 검토와 작업 배정

1. 사용자가 주 탐색의 `이슈`를 선택해 `/issues`로 이동한다.
2. 삭제되지 않은 이슈를 전체 또는 검토 필요, 담당 필요, 진행 중, 완료 확인, 완료 빠른 필터로 좁힌다.
3. 각 빠른 필터에서 현재 검색·세부 필터에 맞는 전체 결과 수를 확인한다. 이 집계는 현재 선택한 빠른 필터만 제외하며 현재 페이지 행 수가 아니라 서버의 전체 결과를 사용한다.
4. 프로젝트·이슈 상태·우선순위·현재 작업 역할·담당자 없음·라벨·만든 사람·생성일·최근 수정일 필터와 정렬을 조합하고 URL에서 같은 보기를 복원한다. 활성 세부 필터는 도구 아래 칩으로 확인하고 개별 제거하거나 전체 초기화할 수 있다.
5. 팀 작업이 없으면 `작업 시작`에서 하나 이상의 프로젝트 역할과 선택 담당자를 지정한다. 기존 미완료 역할 작업은 재사용하고 없는 작업만 만든다.
6. 현재 사용자가 활성 멤버인 역할 팀에 새 작업이 필요하면 `우리 팀에서 시작`으로 미할당 작업을 만들고, 맡을 수 있는 미할당 작업은 `내가 맡기`로 자신에게 할당한다.
7. 여러 미할당 작업은 `담당자 지정`에서 역할별 후보를 선택하고 전체 검증이 성공할 때만 한 번에 저장한다.
8. 모든 대상 작업이 완료돼 상위 이슈가 `완료 확인`으로 자동 전환되면 완료 역할·작업과 제외된 취소 작업을 확인하고 명시적 `이슈 완료`로 최종 인수한다.
9. 저장 또는 버전 충돌이 발생하면 입력값을 유지한 채 최신 담당자·후보를 표시하며, SSE 재조회 뒤 현재 조건에서 빠진 행만 제거하고 필터·정렬·스크롤과 페이지 상태를 유지한다.

한 행의 주요 행동은 `작업 시작 → 담당자 지정 → 우리 팀에서 시작 → 내 작업 열기 → 이슈 완료 → 상세 보기` 순으로 하나만 표시한다. 서버는 우리 팀 시작과 내가 맡기에서 프로젝트 역할과 현재 팀 멤버십을 다시 확인한다.

## 8. 화면별 명세

### 8.1 AUTH-01 회원가입

**주요 구성**

- 표시 이름
- 이메일
- 비밀번호
- 비밀번호 확인
- 회원가입 버튼
- 로그인 이동

**상태와 오류**

- 표시 이름의 필수·길이 오류, 이메일 형식 오류와 비밀번호 정책 미충족을 필드별로 표시한다.
- 제출 중에는 중복 제출을 막는다.
- 유효한 요청을 접수하면 이메일 주소를 가린 형태로 보여 주고, 신규 계정이면 인증 메일을 확인하며 기존 계정이면 로그인 또는 비밀번호 재설정을 사용하라는 공통 안내로 이동한다.
- 기존 이메일 여부를 필드 오류, 상태 코드와 응답 시간 차이로 구분하지 않는다.

### 8.2 AUTH-02 이메일 인증

**인증 전 안내**

- 인증 메일을 보낸 주소
- 메일을 확인하라는 안내
- 제한 시간을 둔 인증 메일 재발송
- 이메일 주소가 잘못된 경우 회원가입으로 돌아가기

**링크 실행 결과**

| 결과 | 화면 동작 |
| --- | --- |
| 성공 | 인증 완료 안내와 로그인 버튼 |
| 만료 | 링크 만료 안내와 재발송 |
| 이미 사용 | 이미 인증되었으면 로그인 안내 |
| 유효하지 않음 | 사용할 수 없는 링크 안내 |

### 8.3 AUTH-03 로그인

- 이메일, 비밀번호, 로그인 버튼과 비밀번호 찾기를 제공한다.
- 인증되지 않은 계정이면 업무 화면에 들어가지 않고 인증 메일 재발송을 안내한다.
- 비활성 멤버는 로그인할 수 없다는 안내와 워크스페이스 관리자 문의 경로를 보여 준다.
- 성공하면 기본적으로 내 작업으로 이동하되 보존된 내부 복귀 주소가 있으면 그 주소를 우선한다.

### 8.4 AUTH-04 비밀번호 재설정

- 이메일 입력 단계는 요청 성공 여부와 관계없이 일관된 완료 안내를 사용한다.
- 새 비밀번호 입력 단계는 새 비밀번호와 확인 필드를 제공한다.
- 만료되었거나 이미 사용한 링크는 새 링크 요청으로 연결한다.
- 성공하면 기존 재설정 링크가 무효가 되었음을 알리고 로그인으로 이동한다.

### 8.5 AUTH-05 초대 수락

- 워크스페이스 이름, 초대 이메일과 초대한 사람을 표시한다.
- 로그인 계정 이메일과 초대 이메일이 다르면 계정 전환을 요구한다.
- 수락 성공 후 해당 워크스페이스의 내 작업으로 이동한다.
- 초대가 중복 수락된 경우 새 멤버십을 만들지 않는다.
- 이미 다른 워크스페이스에 참여한 계정은 초대를 수락할 수 없으며 현재 워크스페이스로 돌아가는 동작을 제공한다.
- 초대 URL은 `/invite#token=...`만 사용한다. 토큰은 메모리로 읽은 직후 `history.replaceState`로 제거하며 `localStorage`, `sessionStorage`와 쿠키에 저장하지 않는다.
- 가입·이메일 인증·로그인이 필요한 사용자는 그 절차를 마친 뒤 원래 초대 메일의 링크를 다시 연다.

### 8.6 ONB-01 워크스페이스 생성

| 필드 | 규칙 |
| --- | --- |
| 워크스페이스 이름 | 필수, 사용자에게 보이는 이름 |
| 슬러그 | 필수, 주소에 사용할 수 있는 형식만 허용하고 입력과 함께 주소 미리보기 제공 |

저장 성공 후 기본 팀 생성으로 이동한다. 브라우저를 닫았다가 돌아와도 완료된 단계부터 이어갈 수 있어야 한다.

별도 슬러그 사용 가능 조회 API는 제공하지 않는다. 생성 요청의 데이터베이스 유일 제약이 중복을 최종 판정하며 `409 WORKSPACE_SLUG_IN_USE`는 입력값을 유지한 채 슬러그 필드 오류로 표시한다.

워크스페이스 생성은 멤버십이 없는 계정에만 표시한다. 기존 워크스페이스에 참여한 계정에는 추가 생성 동작과 워크스페이스 전환 메뉴를 제공하지 않는다.

### 8.7 ONB-02 기본 팀 생성

| 필드 | 규칙 |
| --- | --- |
| 팀 이름 | 필수, 활성 팀 이름과 중복 불가 |
| 팀 키 | 필수, 영문 대문자 2~5자, 표시 ID 예시를 함께 제공 |
| 팀 멤버 | 생성자를 기본 선택, 추가 멤버는 초대 후 설정 가능 |

저장하면 미분류, 할 일, 진행 중, 검토, 완료, 보류, 취소 상태를 자동 생성한다. 팀 키가 첫 이슈 생성 후 바뀌지 않는다는 점을 생성 전에 안내한다.

### 8.8 ONB-03 초기 멤버 초대

- 이메일을 여러 줄 또는 반복 입력으로 추가할 수 있다.
- 중복 이메일은 입력 단계에서 합친다.
- 주소별 성공, 이미 멤버, 이미 초대됨과 실패 결과를 구분한다.
- 건너뛰기와 초대 후 계속 동작을 제공한다.
- 완료 후 첫 이슈 만들기를 강조하고 내 작업으로 이동할 수 있게 한다.

### 8.9 ISSUE-LIST-01 이슈

**헤더**

- 첫 행에 제목 `이슈`, 현재 결과 수와 `이슈 만들기`
- 둘째 도구 행에 데스크톱 320~420px 표시 ID·제목 검색, 빠른 필터, 세부 필터와 정렬
- 공간이 부족하면 검색·빠른 필터와 세부 필터·정렬 묶음을 자연스럽게 나눠 배치
- 활성 세부 필터가 있을 때만 이름·값·개별 제거가 있는 칩과 전체 초기화를 도구 아래에 표시
- 목록 제목 아래의 보기 탭에서 `전체`, 개인 저장 보기와 `새 보기`를 전환한다. 모바일에서는 같은 기능을 compact 선택기로 제공하고, 데스크톱 확장 사이드바에서는 `이슈`·`내 작업` 하위 메뉴로 개인 보기를 노출한다.
- 현재 검색·필터·정렬·표시 옵션을 개인 저장된 보기로 이름 붙여 저장하고 다시 열 수 있다. 선택한 보기의 구성을 바꾸면 `변경됨`을 표시하고 초기화·새 보기로 저장·변경 저장을 제공한다. 저장된 보기는 소유자만 보고, 선택하면 해당 구성으로 URL을 정규화한다.

| 진행 중 | 이슈 상태가 `IN_PROGRESS` |
| 완료 확인 | 이슈 상태가 `REVIEW` |
| 완료 | 상위 이슈 상태가 `DONE` |

취소되지 않은 팀 작업이 하나도 없으면 이슈는 `접수됨`으로 다시 계산되어 검토 필요에 포함된다. 완료된 팀 작업이 하나 이상 있으면 나머지 미완료 작업의 상태와 관계없이 `진행 중`, 모든 유효 작업이 완료되면 `완료 확인`으로 계산한다. `전체`가 기본이며 빠른 필터는 URL에서 단일 선택으로 복원한다. 각 필터에는 현재 검색과 세부 필터를 반영하되 선택된 빠른 필터 조건만 제외한 전체 결과 수를 표시한다. 0건인 필터도 선택할 수 있고 선택 상태는 색상 외에 밑줄·배경 또는 굵기로 구분한다.

**세부 필터와 정렬**

- 프로젝트, 이슈 상태, 우선순위, 현재 작업 역할, 담당자 없음, 라벨, 만든 사람, 생성일 범위와 최근 수정일 범위
- 같은 필터 종류 안에서는 OR, 다른 종류 사이는 AND
- 최근 수정일, 생성일, 상태, 우선순위, 진행률 정렬과 방향
- 기본은 최근 수정일 내림차순이며 같은 값은 불변 ID로 안정 정렬

**목록 행**

- 전역 이슈 본문과 도구는 데스크톱에서 약 1,440~1,600px 최대 폭 안에 가운데 정렬하고 1,280~1,440px에서는 사용 가능한 폭을 유지
- `이슈`, `상태`, `현재 작업`, `진행`, `업데이트`, `다음 행동` 여섯 영역으로 구성
- 이슈 영역은 첫 줄 표시 ID·제목, 다음 줄 프로젝트, 최대 두 개의 라벨과 나머지 `+N`을 표시하며 라벨이 없으면 빈 선택기를 노출하지 않음
- 상태 영역은 제목보다 가벼운 읽기 전용 이슈 상태와 편집 가능한 우선순위 선택기를 사용
- 현재 작업 영역은 역할·실제 팀·진행 상태·담당 필요 수를 한 묶음으로 표시하고 병렬 역할은 높이가 무한히 늘지 않게 요약
- 진행 영역은 완료 수/취소 제외 전체 수, 백분율과 짧은 진행 막대를 가까이 표시
- 업데이트 영역은 간결한 상대 시각을 표시하고 만든 사람은 기본 열에서 제외
- 다음 행동은 한 개만 표시하며 단순 `상세 보기`는 행의 빈 영역 선택으로 제공 가능
- `분석 중 · 시작 팀 없음`, `백엔드 진행 중`, `백엔드 · 웹 프론트 병렬 진행`, `웹 프론트 담당 필요`, `API-12 완료 대기`, `모든 팀 작업 완료`처럼 실제 맥락을 표시하고 `차단됨`만 단독으로 사용하지 않음
- 결정이 필요한 작업 시작·담당자 지정·이슈 완료만 조건부로 강조하고 이동은 Ghost 또는 행 선택을 사용한다. 가능한 우리 팀 역할이 하나면 `백엔드 작업 시작`처럼 역할을 문구에 표시하고 여러 역할이면 `우리 팀에서 시작`을 유지한다.
- 행의 빈 영역 선택은 상세로 이동하지만 상태·우선순위·라벨·주요 행동·더보기·진행 링크 조작은 이동을 실행하지 않는다. 행 열기와 내부 조작은 각각 키보드 포커스를 제공한다.

**작업 시작과 배정**

- 작업 시작은 프로젝트 역할, 연결 팀, 담당자 선택과 기존 미완료 역할 작업 여부를 보여 주며 하나 이상의 역할을 요구한다.
- 우리 팀에서 시작은 가능한 역할이 하나면 확인 후 미할당 작업을 만들고 여러 역할이면 선택한다.
- 내가 맡기는 후보 한 건을 바로 할당하고 여러 건이면 대상 작업을 선택하며 후보가 없으면 역할 작업을 생성해 자신에게 할당한다.
- 담당자 지정은 미할당 미완료 작업만 기본 대상으로 삼고 역할별 활성 팀 멤버 후보와 작업 버전을 함께 저장한다.
- 한 건이라도 충돌하면 전체 저장을 취소하고 입력값, 최신 담당자와 최신 후보를 유지한다.
- 이슈 완료는 `완료 확인` 상태에서 완료 역할·팀 작업과 제외된 취소 작업을 먼저 요약하며 팀 작업 없는 이슈에는 제공하지 않는다.

| 워크스페이스에 이슈 없음 | `아직 등록된 이슈가 없습니다` | 첫 이슈 만들기 |

### 8.10 WORK-01 내 작업

**헤더**

- 제목 `내 작업`과 현재 미완료 결과 수, 설명 `내가 맡은 미완료 작업`
- 보조 동작 `이슈 만들기`
- 검색: 작업 코드, 이슈 표시 ID·제목, 프로젝트 이름
- 필터: 상태, 프로젝트
- 정렬: 실행 순서, 우선순위, 생성일, 최근 수정일과 방향
- 필터 초기화
- 제목 아래의 보기 탭과 데스크톱 확장 사이드바의 `내 작업` 하위 메뉴에서 개인 저장 보기를 열고, 모바일에서는 compact 선택기를 사용한다.
- 현재 구성을 개인 저장된 보기로 저장·열기·이름 변경·삭제·기본 지정한다. 선택한 보기의 구성을 바꾸면 `변경됨`을 표시하고 초기화·새 보기로 저장·변경 저장을 제공한다. 보관된 프로젝트나 권한을 잃은 값은 조용히 제거하지 않고 수정 안내를 표시한다.

**목록 행**

- `우선순위`: 상위 이슈 우선순위의 Compact 아이콘·한국어 이름 읽기 표현
- `작업`: 이슈 제목과 `팀 작업 코드 · 이슈 코드 · 역할` 보조 줄
- `프로젝트`: 프로젝트 이름과 최대 두 개의 상위 이슈 라벨, 나머지 `+N`
- `상태`: 팀 워크플로 상태의 Compact 인라인 편집
- 이름 없는 우측 행동 영역: 현재 가능한 `작업 시작` 또는 `완료` 한 개

내 작업은 개인 실행 큐이므로 담당자, 실제 팀 이름과 최근 수정 시각을 기본 행에 표시하지 않는다. 담당자는 항상 현재 사용자이고, 최근 수정은 정렬 기준으로만 사용한다. 역할은 작업 보조 줄에서 구분하며 라벨은 상위 이슈가 소유한 읽기 전용 값으로 프로젝트 아래에 표시한다. 주요 행동은 데이터 열처럼 이름을 붙이지 않으며 현재 상태에 실행 가능한 행동이 없으면 빈 Primary 버튼이나 대체 문구를 만들지 않는다.

기본 결과는 자신에게 할당된 `BACKLOG`, `UNSTARTED`, `STARTED` 범주의 미완료 팀 작업이다. `COMPLETED`, `CANCELED` 범주는 내 작업에서 조회하지 않는다. 행을 열면 `/my-issues/{teamWorkIdentifier}?tab=work`로 이동해 내 작업 내비게이션 문맥을 유지한다.

기본 `실행 순서`는 `STARTED → UNSTARTED → BACKLOG` 범주, 상위 이슈 우선순위 `URGENT → HIGH → NORMAL → LOW → NONE`, 같은 범주의 팀 워크플로 표시 순서, 최근 수정 내림차순과 불변 ID 내림차순으로 안정 정렬한다. 사용자가 선택한 정렬, 필터와 스크롤은 URL과 뒤로 가기에서 복원한다.

**빈 상태**

- 할당된 작업이 없음: 현재 할당된 미완료 작업이 없음을 표시
- 필터 결과 없음: 필터 초기화 제공
- 처음 사용: 이슈 만들기 제공

### 8.10.1 WORK-DETAIL-01 내 작업 상세

내 작업 상세는 별도의 팀 작업 콘텐츠 문서를 만들지 않고, 선택한 팀 작업과 상위 이슈를 결합한 기존 통합 상세 콘텐츠를 `내 작업` 진입 문맥으로 표시한다. 경로의 팀 작업 표시 ID로 상위 이슈를 해석하며 설명·본문 이미지·첨부파일·댓글·전달·활동은 이슈 상세와 같은 원본과 컴포넌트를 사용한다.

**진입과 내비게이션**

- 사이드바 활성 항목: `내 작업`
- 화면 안 뒤로 가기: `내 작업`으로 표시하고 목록의 필터·정렬·스크롤을 복원
- 직접 주소 또는 새로고침 진입: 안전한 기본 복귀 경로로 `/my-issues` 제공
- 경로가 선택 작업을 이미 나타내므로 `work` 쿼리를 중복하지 않음

**상단과 실행 문맥**

- 팀 작업 표시 ID와 상위 이슈 제목을 주 식별 정보로 표시
- 현재 팀 워크플로 상태와 가능한 `작업 시작` 또는 `완료`를 첫 실행 영역에 표시

- 담당자는 현재 사용자일 때 반복 강조하지 않고, 재배정 여부를 판단하는 데 필요한 위치에서만 표시
- 같은 이슈의 다른 팀 작업은 관련 작업 전환기로 제공하되 현재 작업보다 시각적 위계를 낮춤

`업무`, `전달`, `활동` 탭과 업무 본문은 ISSUE-01의 통합 상세를 재사용한다. 내 작업 상세만을 위한 설명·첨부파일·댓글 저장소, 별도 API 응답 모델 또는 복제된 상세 컴포넌트를 만들지 않는다. 다른 사용자에게 재배정되거나 완료·취소되어 개인 실행 큐에서 제외된 뒤에도 활성 멤버 권한으로 현재 상태를 확인할 수 있으며, `내 작업에서 제외된 작업입니다` 안내와 `/my-issues` 복귀 동선을 제공한다.

최소 한 역할이 필요하다. 같은 팀을 여러 역할에 선택할 수 있다. 선택하지 않은 역할은 프로젝트 상세의 생성 동작, 진행률과 전달 알림 대상에서 제외한다.

**편집 제한**

- 역할에 연결된 팀 작업이 없으면 팀 변경 또는 역할 해제가 가능하다.
- 연결된 작업이 있으면 저장을 막고 영향을 받는 작업 목록을 제공한다.
- 기존 작업을 제거하거나 새 팀의 작업으로 다시 만든 뒤 변경하도록 안내한다.
- 프로젝트에 연결된 이슈가 있으면 삭제를 제공하지 않고 보관을 안내한다.

### 8.15 PROJ-03 프로젝트 상세

**상단 요약**

- 프로젝트 이름, 상태, 리드
- 시작일과 목표일
- 역할별 팀
- 완료 작업 수와 진행률
- 편집, 보관과 조건부 삭제

**본문 기본 보기**

프로젝트의 결과물과 팀 간 전달 흐름을 한 번에 파악할 수 있도록 이슈 중심 계층을 기본 보기로 사용한다.

- 이슈를 상위 행으로 표시한다.
- 각 이슈 아래에 완료·현재 팀 작업과 생성 전 예상 역할을 작업 흐름으로 표시한다.
- 이슈 행에는 현재 역할, 현재 팀 작업 상태와 진행률을 표시한다.
- 팀 작업 행에는 역할, 팀, 상태, 담당자와 현재 주요 행동을 표시하고 우선순위·라벨은 상위 이슈 행에서만 표시한다.
- 역할 필터로 백엔드, 웹 프론트와 앱 프론트 작업을 좁힐 수 있다.

**주요 동작**

- 이슈 만들기
- 이슈 작업 흐름의 팀 작업 추가
- 프로젝트 상태, 리드와 일정 변경
- 이슈 상세 열기

팀 작업은 상위 이슈 프로젝트를 상속하므로 프로젝트에 개별 추가·제거하거나 다른 프로젝트로 이동할 수 없다. 팀 작업을 추가할 때는 이슈 프로젝트에 설정된 역할과 실제 담당 팀을 사용한다.

### 8.16 ISSUE-01 이슈 상세

MVP에서는 데스크톱과 모바일 모두 독립 화면으로 연다. 이전 목록의 필터, 정렬과 스크롤 위치를 복원해 맥락 이탈을 줄인다.

ISSUE-01은 이슈 목록·프로젝트·팀·알림에서 들어오는 이슈 중심 진입 모드다. `/issues/{issueIdentifier}`에서는 사이드바의 `이슈`를 활성화하고 이슈 표시 ID·제목을 상단의 기준으로 삼는다. 내 작업 행에서 들어오는 WORK-DETAIL-01은 별도 경로와 내비게이션 셸을 사용하지만 아래 통합 탭·본문·데이터 원본을 공유한다.

**공통 상단**

- 이슈 표시 ID와 제목
- `업무`, `전달`, `활동` 탭. 기본 선택은 `업무`
- 현재 선택된 팀 작업의 팀·역할과 작업 전환기
- 더보기: 삭제 등 조건부 동작

이슈 헤더는 표시 ID, 제목, 프로젝트, 우선순위, 진행률과 자동 계산 상태만 표시한다. 선택된 팀 작업 본문은 팀·역할, 담당자, 현재 상태에 맞는 주요 행동, 보조 상태 메뉴와 Markdown 작업 노트만 한 번 표시한다.

탭은 `?tab=work`, `?tab=handoffs`, `?tab=activity`로 직접 열 수 있다. 잘못된 값은 `업무`로 복구하고 탭 전환은 현재 상세의 편집 속성, 이전 화면의 필터와 스크롤 복원 맥락을 바꾸지 않는다. 전달 딥 링크는 `?tab=work&work={teamWorkIdentifier}&handoff={handoffId}`로 관련 팀 작업을 선택하고 해당 전달을 확장한다.

**통합 업무 탭**

- 팀 작업이 선택되면 현재 행동, 팀 상태·담당자, 최신 보낸 또는 받은 전달과 작업 노트를 표시한다.
- 이슈 설명·본문 이미지·일반 첨부파일·댓글은 모든 팀 작업 문맥에서 같은 원본을 표시한다.
- 팀 작업이 선택되지 않으면 현재 작업 요약, 설명, 첨부파일, 댓글 순서로 표시한다.
- 팀 작업이 없으면 현재 작업 요약에서 `작업 시작`을 제공한다.
- 팀 작업 상태가 `시작 전`이면 `작업 시작`, `진행 중`이면 `완료`, 완료 뒤에는 `완료됨`을 같은 주요 행동 위치에 제공한다.
- 백엔드 작업의 `완료`는 `이 작업만 완료`와 `프론트에 전달 후 완료`를 선택하는 완료 창을 열고, 후자를 고른 경우에만 전달 편집기를 펼친다.
- 전달받은 프론트 작업은 관련 최초·추가 전달을 이슈 설명 전에 표시한다.

**통합 전달·활동 탭**

- `전달`에는 최초 전달과 추가 전달의 전체 Markdown 이력을 시간순으로 표시하고 카드 안에서 접고 펼친다.
- `활동`에는 이슈와 모든 팀 작업의 시스템 이력을 합쳐 표시하며 각 항목에 관련 팀 작업 표시 ID를 선택적으로 포함한다.
- 팀 작업별 별도 댓글·첨부파일·활동 타임라인을 만들지 않는다.

**설명**

- 설명은 이슈에만 저장하고 팀 작업을 전환해도 같은 원본을 표시한다.
- Lexical WYSIWYG 편집과 Markdown 미리보기 전환
- 이미지 선택과 Command/Ctrl+V 붙여넣기, 파일별 업로드 진행·실패·재시도
- 저장과 취소
- 저장 전 다른 곳으로 이동하면 작성 내용 유실 경고
- 위험한 HTML과 링크 속성 제거

**업무·전달 탭의 작업 전달**

- 백엔드 작업의 `업무`에는 전달 주요 동작과 최신 전체 내용, `전달`에는 전체 이력을 표시
- 전달받은 프론트 작업의 `업무`와 이슈 `전달`에는 같은 원본의 조회 영역을 표시
- 최초 전달과 추가 전달을 시간순으로 구분
- 평소 업무 탭에는 완료 버튼만 제공하고 `프론트에 전달 후 완료`를 선택한 경우에만 템플릿·대상 역할 선택을 제공
- 완료 후에는 기존 전달 편집 대신 추가 전달만 제공
- 성공 결과에 생성·재사용한 후행 작업 링크를 표시
- API 명세 링크를 눈에 띄게 표시하되 외부 링크임을 구분
- 멘션 선택기를 표시하지 않고 멘션 참조가 포함된 본문은 저장 오류로 표시
- 완료 요청이 충돌하면 완료 방식과 작성 본문을 유지하고 최신 이슈·상태 선택지를 다시 읽은 뒤 재실행을 안내

**업무 탭의 댓글**

- 실제 댓글만 작성 시각 순으로 표시하고 시스템 활동을 댓글 사이에 넣지 않는다.
- 댓글은 이슈에 저장하며 선택된 팀 작업에서 작성하면 선택적인 팀 작업 문맥을 함께 기록한다.
- 댓글은 작성자, 작성 시각과 수정 여부를 표시한다.
- 댓글 입력은 실제 댓글 목록 바로 아래에서 Lexical WYSIWYG 편집과 Markdown 미리보기를 제공한다.
- 댓글 이미지도 설명과 같은 업로드·미리보기·저장 규칙을 사용한다.
- 삭제된 댓글은 본문 대신 삭제된 댓글임을 표시한다.

**활동 탭**

- 생성, 상태·담당자·우선순위·라벨 변경, 작업 시작, 작업 노트 변경, 최초·추가 전달과 휴지통 변경 같은 시스템 활동만 시간순으로 표시한다.
- 댓글과 댓글 작성기를 표시하지 않는다.
- 작업 전달 원문은 활동에 반복하지 않고 `백엔드 작업을 웹 프론트에 전달했습니다` 같은 요약만 표시한다.
- 전달 원문은 `업무`의 최신 전달 또는 `전달` 탭에서 연다.
- 같은 저장으로 발생한 이벤트는 저장 데이터를 변경하지 않고 화면에서 대표 활동으로 묶을 수 있다.
- 알림에서 진입하면 관련 댓글 또는 우선순위에 따른 전달 위치로 이동하고 강조한다.

**첨부파일**

- 일반 첨부파일은 이슈에만 연결하고 팀 작업 전용 일반 첨부파일을 제공하지 않는다.
- 일반 첨부파일은 설명과 분리된 영역에 파일명, 형식, 크기, 업로더와 업로드 시각을 표시한다.
- 여러 파일을 추가할 수 있고 파일 하나는 최대 25MB다.
- 검증된 이미지는 화면에서 열 수 있고 그 밖의 파일은 다운로드한다.
- 업로드 중에는 이탈 확인을 제공하고 실패 파일은 입력을 유지한 채 다시 시도하거나 제거한다.
- 같은 워크스페이스의 활성 멤버만 조회·다운로드할 수 있다.

**삭제 제한**

- 이슈 삭제는 팀 작업·전달·댓글·파일에 미치는 영향을 함께 설명한다.
- 개별 팀 작업 제거는 연결된 전달 대상과 알림의 삭제 영향을 함께 설명한다.
- 삭제 가능한 이슈는 팀 작업을 포함해 30일 휴지통 이동을 확인한다.

### 8.18 SEARCH-01 전역 검색

전역 검색은 별도 영구 페이지보다 빠른 검색 창을 기본으로 한다. 직접 주소가 필요하면 현재 검색어를 쿼리 문자열로 보존할 수 있다.

- 빈 입력은 검색 가능한 범위를 안내하는 빈 상태로 두고 최근 열어본 이슈는 표시하지 않는다.
- 두 글자부터 제목 부분 일치 결과를 표시한다.
- 표시 ID는 한 글자 입력부터 정확 일치를 시도할 수 있다.
- 키보드로 결과 선택과 닫기가 가능하다.
- 결과를 연 뒤 뒤로 가면 검색어와 선택 위치를 복원한다.

### 8.19 SET-01 멤버

- 활성, 초대 대기와 비활성 멤버를 구분한다.
- 활성·비활성 멤버는 표시 이름을 기본값으로, 이메일을 보조 정보로 표시한다.
- 이메일 초대, 재초대와 초대 상태 확인을 제공한다.
- 대기·만료 초대는 같은 기록에서 재발송하고, 수락·취소 이력의 재초대는 원본 상태를 유지한 채 새 대기 기록으로 표시한다.
- 수락 이력에서 재초대한 링크를 기존 활성 멤버가 확인해도 멤버십을 중복 생성하지 않는다.
- 활성 멤버를 팀에 추가하는 경로를 제공한다.
- 멤버에게 할당된 미완료 팀 작업이 있으면 비활성화를 막고 작업 목록, 재할당과 담당 해제 경로를 제공한다.
- 미완료 팀 작업이 모두 정리된 뒤에만 비활성화를 실행하고 해당 멤버의 로그인 세션을 종료한다.
- 비활성 멤버의 과거 이슈와 댓글 작성자 표시는 유지한다.

### 8.20 SET-02 팀

- 활성 팀과 보관된 팀을 구분한다.
- 팀 이름, 키, 멤버 수와 미완료 작업 수를 표시한다.
- 팀 생성 시 이름, 영문 대문자 키와 초기 멤버를 입력한다.
- 이슈가 생성된 팀의 키는 읽기 전용으로 표시하고 이유를 안내한다.
- 미완료 작업 담당자를 팀에서 제거하려 하면 해당 작업 목록과 재할당 경로를 제공한다.
- 미완료 팀 작업이 있으면 팀 보관을 막고 작업 수를 표시한다.

### 8.21 SET-03 워크플로

- 선택한 팀의 상태를 시스템 범주별로 나누어 표시한다.
- 상태 이름과 표시 순서를 변경한다.
- 시스템 범주는 읽기 전용이며 사용자에게는 의미 설명을 제공한다.
- 사용 중인 상태를 삭제하면 대체 상태 선택을 필수로 요구한다.
- 상태 순서 변경은 워크플로 열 순서이며 이슈의 수동 정렬 기능이 아니다.

### 8.22 SET-04 라벨

- 활성 라벨과 보관된 라벨을 구분한다.
- 이름과 색상을 생성·수정한다.
- 활성 라벨 이름 중복을 막는다.
- 사용 중인 라벨은 보관을 기본 동작으로 제공한다.
- 보관된 라벨은 기존 이슈에 남고 새 선택 목록에서는 제외됨을 안내한다.

### 8.23 SET-05 데이터 내보내기

- 관리자만 접근한다.
- 이슈 CSV와 프로젝트 CSV의 포함 필드를 미리 설명한다.
- 내보내기 실행 시 생성 중, 완료와 실패 상태를 표시한다.
- 완료 파일에 설명 또는 작업 전달처럼 민감한 업무 내용이 포함될 수 있음을 실행 전에 안내한다.
- CSV 가져오기는 별도 `SET-07`에서 검증과 저장 흐름을 제공한다.

### 8.24 SET-06 휴지통

- 이슈와 프로젝트를 유형별로 필터링한다.
- 이름 또는 제목, 표시 ID, 삭제한 사용자, 삭제일과 영구 삭제 예정일을 표시한다.
- 관리자만 복구할 수 있다.
- 복구 전 원래 팀이나 프로젝트가 보관된 경우의 처리 결과를 안내한다.
- 영구 삭제를 즉시 실행하는 기능은 MVP에서 제공하지 않고 보존 기간 만료를 따른다.

### 8.25 SET-07 CSV 가져오기

- 관리자만 접근하며 파일 선택, 컬럼 확인, 값 매핑, 검증, 최종 확인, 결과의 6단계를 순서대로 제공한다.
- UTF-8 `.csv`, 5MB, 데이터 10,000행, 50컬럼 제한과 필수 `source key`, 제목, 팀, 상태, 프로젝트 컬럼을 파일 선택 단계에서 안내한다.
- 팀·상태·멤버·프로젝트·우선순위·라벨의 원본 값을 현재 워크스페이스의 활성 대상에 연결한다. 보관·비활성 대상은 후보에서 제외하고 알 수 없는 값은 저장 전에 오류로 표시한다.
- 검증 결과는 생성할 프로젝트·이슈·연결 수, 제외 행 수와 행 번호·필드·정제 오류를 표시한다. CSV 원문 값은 오류 문장에 반복하지 않는다.
- 이름으로 식별한 미지원 컬럼은 필드 선택 후보에서 제외하고, 그 밖에도 Rivet 필드에 연결하지 않은 모든 CSV 컬럼을 저장 제외 예정으로 계속 표시한다.
- 댓글, 과거 활동, 알림, 작업 전달 이력, 첨부파일과 외부 자동화 규칙은 미지원임을 파일 선택과 최종 확인에서 명시한다. 가져온 이슈·팀 작업은 담당자가 있어도 생성·할당·멘션 알림을 보내지 않고 실행 결과에서만 완료를 알린다.
- `=`, `+`, `-`, `@`, 탭·캐리지 리턴으로 시작하는 셀은 일반 텍스트 의도와 관계없이 formula injection 오류로 막으며 사용자가 첫 문자를 바꾼 파일로 다시 검증하게 한다.
- 최종 저장은 전체 성공 또는 전체 롤백이며 실행 중 버튼을 잠근다. 파일·매핑 대상·중복 상태가 바뀌면 저장하지 않고 값 매핑 또는 검증 단계로 돌려보낸다.
- 성공 결과에서 생성된 프로젝트와 이슈를 열 수 있고 새 가져오기를 명시적으로 시작할 수 있다. 최근 실행은 현재 워크스페이스 결과만 표시한다.
- 원본 파일과 업무 본문 사본은 실행 기록, 로그와 분석에 남기지 않는다. 매핑한 설명은 생성된 이슈의 정본 본문으로만 저장한다.

### 8.26 PROFILE-01 프로필 설정

- 사용자 메뉴에서 모달로 열며 관리자 여부와 관계없이 모든 활성 멤버가 접근한다.
- 현재 프로필 사진 또는 표시 이름 첫 글자 아바타를 보여 준다.
- `사진 변경`, `사진 삭제`, `닫기`를 제공하며 표시 이름·이메일 변경은 MVP에 포함하지 않는다.
- 허용 형식과 최대 25MB를 선택 전에 안내하고 프론트 최적화 결과가 25MB를 넘으면 저장하지 않는다.
- 업로드 실패 시 기존 사진을 유지하고 다시 시도할 수 있게 한다.

## 9. 공통 구성 요소와 동작

### 9.1 Lexical 편집기

- `편집`과 `미리보기` 두 모드를 제공하며 Markdown 원문 편집 모드는 제공하지 않는다.
- 편집 모드는 Lexical 기반 WYSIWYG UI를 사용하고 `@lexical/markdown`의 단축 입력과 변환 규칙을 적용한다.
- 서버와 API의 정본은 Markdown 문자열이며 Lexical JSON 편집 상태는 저장하거나 전송하지 않는다.
- 이슈 설명, 댓글과 작업 전달은 같은 Markdown 변환·렌더링 규칙을 사용하고 미리보기는 현재 편집 내용을 Markdown으로 변환해 표시한다. 팀 작업 메모는 평문 입력을 사용한다.
- MVP 서식은 제목, 굵게, 기울임, 취소선, 링크, 인용, 순서·비순서 목록, 인라인 코드와 코드 블록으로 제한한다.
- 사용자 정의 이미지 노드와 Markdown 변환 규칙을 사용해 업로드된 이미지를 인증된 파일 주소로 저장한다.
- 파일 선택과 Command/Ctrl+V를 지원한다. 클립보드에 실제 이미지 파일이 없고 원격 HTML 또는 URL만 있으면 서버가 대신 가져오지 않는다.
- JPEG·PNG·WebP 이미지는 2MB 초과 또는 긴 변 2,560px 초과일 때 브라우저에서 비율을 유지해 최대 2,560px, WebP 품질 82%를 기준으로 최적화한다. 변환 결과가 더 크면 원본을 사용한다.
- 업로드 중에는 미리보기와 진행 상태를 표시하고, 실패 시 다시 시도·제거를 제공하며 완료되지 않은 파일이 있으면 저장을 막는다.
- 멘션은 이슈 설명과 댓글에서만 사용자 정의 노드와 변환 규칙으로 처리하며 자동완성에는 활성 워크스페이스 멤버만 표시한다.
- 작업 전달 편집기는 멘션 선택기를 표시하지 않고 멘션 참조가 포함된 본문의 저장을 막는다.
- 작업 전달은 기본 템플릿을 삽입하며 템플릿 제목만 남은 내용은 유효한 전달로 인정하지 않는다.
- 스크립트, 허용하지 않은 HTML, 위험한 링크 속성과 임의 임베드를 제거한다.

### 9.2 이슈 행과 카드

필수 표현은 표시 ID, 제목, 상태다. 화면 목적에 따라 팀, 역할, 담당자, 우선순위, 라벨, 프로젝트와 작업 대기 배지를 추가한다. 색상만으로 상태나 우선순위를 구분하지 않고 이름 또는 아이콘과 함께 표시한다.

### 9.3 필터

- 같은 필드의 여러 값은 OR로 적용한다.
- 다른 필드끼리는 AND로 적용한다.
- 활성 필터는 화면 헤더에 요약하고 한 번에 초기화할 수 있다.
- 목록과 보드를 전환해도 유지한다.
- 저장된 보기는 MVP에서 제공하지 않는다.

### 9.4 정렬

- 내 작업은 기본 실행 순서와 상위 이슈 우선순위, 생성일, 최근 수정일을 제공한다. 팀 목록은 생성일, 최근 수정일, 팀 상태와 상위 이슈 우선순위를 제공하고 전역 이슈 목록은 진행률을 추가한다.
- 오름차순과 내림차순을 선택한다.
- 내 작업의 실행 순서는 방향을 따로 선택하지 않고 `STARTED → UNSTARTED → BACKLOG`, 우선순위, 워크플로 표시 순서, 최근 수정과 불변 ID 순으로 고정한다.
- 보드 열 안에서도 선택한 정렬 기준을 따른다.
- 사용자가 드래그해 임의 순서를 저장하는 기능은 제공하지 않는다.

### 9.5 진행률

- 이슈: 취소된 직계 팀 작업을 제외한 완료 작업 수 ÷ 대상 팀 작업 수
- 프로젝트: 취소된 연결 팀 작업을 제외한 완료 작업 수 ÷ 대상 연결 팀 작업 수
- 대상 작업이 없는 이슈는 숫자만 있는 `0/0 완료` 대신 `아직 시작된 팀 작업이 없습니다`와 `작업 시작`을 표시
- 이슈 자체를 프로젝트 진행률에 별도 항목으로 더하지 않음
- `접수됨`, `할 일`, `진행 중`, `완료 확인`은 소속 팀 작업 변경과 같은 트랜잭션에서 자동 갱신하고 `일시 중지`, `취소`, `완료` 상태에서는 명시적 재개·다시 열기 전까지 자동 전이를 멈춤

### 9.6 확인창

삭제, 보관, 비활성화와 관계 해제는 다음을 표시한다.

- 대상 이름과 유형
- 즉시 발생하는 변화
- 복구 가능 여부
- 작업을 막는 연결 관계
- 취소와 명확한 실행 버튼

## 11. 모바일 웹

### 11.1 지원

- 전역 이슈와 내 작업, 팀 작업 목록 조회
- 전역 이슈의 작업 시작, 우리 팀에서 시작, 내가 맡기와 담당자 지정
- 전역 이슈 표를 제목, 프로젝트, 현재 작업, 담당 필요, 진행률과 주요 행동 중심의 카드로 재배치하고 필터는 하단 시트 또는 전체 화면 패널로 제공
- 이슈와 프로젝트 상세 조회
- 이슈 상세의 `업무`, `전달`, `활동` 탭 전환과 직접 진입
- 이슈 첨부파일 조회·업로드·다운로드
- 사용자 메뉴의 프로필 사진 변경·삭제
- 댓글 작성
- 상태, 담당자와 우선순위 변경
- 알림함 확인과 읽음 처리
- 전역 검색

### 11.2 제한

- 팀 상태 보드는 목록으로 대체한다.
- 팀, 워크플로, 멤버와 워크스페이스 설정은 데스크톱 사용을 안내한다.
- 프로젝트 생성·역할별 팀 편집은 지원하지 않고 데스크톱 사용을 안내한다.
- 작업 전달 작성은 지원하지 않지만 최신 전달과 전체 이력은 조회할 수 있다.

### 11.3 모바일 탐색

좁은 화면에서는 다음 5개 항목을 고정 하단 탐색으로 제공한다. 주 탐색 네 항목은 데스크톱과 같은 상대 순서를 유지하고 팀을 마지막에 둔다. 검색은 상단 앱 바의 전역 동작으로 이동한다.

| 항목 | 동작 |
| --- | --- |
| 이슈 | 워크스페이스 전체 이슈 목록으로 이동 |
| 내 작업 | 현재 사용자에게 할당된 미완료 팀 작업으로 이동 |
| 알림함 | 알림 목록으로 이동하고 읽지 않은 개수를 함께 표시 |
| 프로젝트 | 프로젝트 목록으로 이동 |
| 팀 | 마지막으로 방문한 팀 작업 목록으로 이동하고 방문 기록이 없으면 팀을 선택 |

각 항목은 아이콘과 한국어 이름을 함께 표시하고 현재 위치를 구분한다.

모바일 상단 앱 바의 프로필 사진 또는 첫 글자 아바타를 선택하면 사용자 메뉴를 열고 프로필 설정과 로그아웃을 제공한다.

## 12. 키보드와 접근성

### 12.1 키보드 동작

아래는 MVP 기본안이며 실제 구현한 동작만 단축키 도움말에 표시한다.

| 키 | 동작 |
| --- | --- |
| C | 이슈 만들기 |
| / | 전역 검색 |
| Escape | 현재 팝오버 또는 창 닫기 |
| Command/Ctrl + Enter | 생성 또는 댓글 저장 |
| Shift + S | 선택한 팀 작업의 상태 변경. 이슈에서는 일반 상태 선택기를 열지 않음 |
| Shift + A | 선택한 팀 작업의 담당자 변경 |
| Shift + P | 선택한 이슈의 우선순위 변경 |
| Shift + L | 선택한 이슈의 라벨 변경 |
| ? | 단축키 도움말 |

이슈 상세 탭에 포커스가 있으면 좌우 방향키로 `업무`, `전달`, `활동`을 이동한다.

텍스트 입력과 Markdown 편집 중에는 단일 문자 단축키를 실행하지 않는다.

### 12.2 접근성 기준

- 모든 입력은 연결된 이름과 오류 설명을 가진다.
- 팝오버와 모달을 열면 첫 조작 요소로 포커스를 이동한다.
- 모달을 닫으면 열었던 버튼 또는 합리적인 다음 위치로 포커스를 돌린다.
- 이슈 상세 탭은 `tablist`, `tab`, `tabpanel` 관계와 `aria-selected`를 제공하고 탭과 패널의 접근 가능한 이름을 연결한다.
- 현재 선택, 읽지 않음, 현재 주요 행동과 저장 중 상태를 보조 기술에도 전달한다.
- 전역 이슈 빠른 필터의 선택 상태와 현재 결과 수를 보조 기술에 전달하고 색상만으로 검토·담당 필요 상태를 구분하지 않는다.
- 전역 이슈 행 전체 열기와 내부 선택기·주요 버튼은 독립된 키보드 대상을 사용하며 내부 동작이 행 이동을 실행하지 않는다.
- 작업 시작·내가 맡기·담당자 지정 창은 제목, 역할, 담당자, 오류, 주요 동작 순으로 읽고 저장 실패 시 입력값을 유지한 채 첫 오류로 포커스를 이동한다.
- 탭 전환 뒤 패널 제목 또는 첫 의미 있는 요소로 읽기 맥락을 제공하되 URL 직접 진입에서는 포커스를 빼앗지 않는다.
- 자동 변경된 이슈 상태는 아이콘과 한국어 이름으로 표시하고 원인이 된 팀 작업 행동 뒤 상태 변경을 보조 기술과 가까운 피드백으로 알린다.
- 상태, 우선순위와 진행률은 색상만으로 표현하지 않는다.
- 아이콘 단독 버튼에는 한국어 접근 가능 이름을 제공한다.
- 보드 드래그와 같은 동작은 키보드로 상태를 변경할 대체 경로를 제공한다.

## 13. 분석 이벤트 연결

화면 구현은 PRD의 분석 이벤트를 발생시키되 업무 본문을 속성에 포함하지 않는다.

| --- | --- |

| 이슈 생성 성공 | issue_created |
| 상태·담당자·우선순위 등 변경 성공 | issue_property_changed |
| 팀 작업 완료 | issue_completed |
| 최초 또는 추가 작업 전달 저장 | api_handoff_created |
