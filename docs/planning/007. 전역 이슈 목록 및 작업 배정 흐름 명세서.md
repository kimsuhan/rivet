## 1. 문서 목적

이 문서는 워크스페이스의 모든 제품·기능 단위 이슈를 프로젝트 경계와 무관하게 한곳에서 접수, 검토, 분류하고 실제 팀 작업으로 전환하는 전역 이슈 화면을 정의한다.

현재 `내 작업`은 사용자에게 할당된 팀 작업을 보여 주고 전역 `이슈`는 제품·기능 단위 이슈를 보여 준다. 두 목록의 책임이 다른 만큼 상세 진입에서도 활성 내비게이션과 복귀 동선을 구분한다. 다만 내 작업 상세는 별도 콘텐츠 문서가 아니라 상위 이슈의 공유 상세 본문에서 해당 팀 작업이 선택된 개인 실행 모드다.

전역 `이슈` 화면은 프로젝트 상세를 대체하지 않는다. 프로젝트 상세는 한 프로젝트의 목표, 역할과 전체 작업 구조를 보여 주고, 전역 이슈 화면은 여러 프로젝트의 이슈를 업무 상태에 따라 찾고 다음 행동을 실행하는 접수·분류 업무함으로 사용한다.

### 1.1 해결할 문제

- 워크스페이스의 전체 이슈 목록으로 바로 이동할 수 없다.
- 새 이슈를 검토하고 시작 팀을 선택하려면 프로젝트 상세를 개별적으로 열어야 한다.
- 팀 작업이 없거나 담당자가 없는 이슈를 한곳에서 확인할 수 없다.
- PM이나 멤버가 이슈를 팀 작업으로 전환하고 담당자를 지정하는 흐름이 흩어져 있다.
- `내 이슈`라는 이름이 실제로는 할당된 팀 작업만 포함해 데이터 의미와 맞지 않는다.
- 모든 팀 작업이 끝났지만 상위 이슈가 완료되지 않은 항목을 놓치기 쉽다.

### 1.2 범위

- 사이드바의 `이슈`와 `내 작업` 내비게이션
- 워크스페이스 전체 이슈 목록과 안정된 직접 경로
- 검토 필요, 담당 필요, 진행 중, 완료 확인과 완료 빠른 필터
- 프로젝트, 상태, 우선순위, 역할, 라벨, 생성자와 시각 필터
- 이슈별 현재 작업, 담당 필요 여부, 진행률과 다음 행동 요약
- 작업 시작, 우리 팀에서 시작, 내가 맡기와 담당자 지정
- 목록에서 자동 이슈 상태 확인과 우선순위·라벨 빠른 변경
- 기존 이슈 생성, 팀 작업, 프로젝트와 상세 화면 연결
- API 조회·배정 계약, 트랜잭션, 실시간 갱신과 오류 복구
- 데스크톱·모바일 목록과 접근성 기준

### 1.3 제외 범위

- 외부 고객이 워크스페이스 계정 없이 요청하는 고객 포털
- 고객 조직, 요청자 회사와 CRM 데이터를 관리하는 기능
- 별도의 PM, 고객 또는 팀장 권한 역할 추가
- 승인 단계, 자동 배정, 라운드 로빈과 규칙 기반 자동화
- 칸반 보드, 로드맵, 타임라인과 간트 보기
- 저장된 사용자 필터와 공유 보기
- 이슈에 직접 담당자나 팀을 추가하는 데이터 모델 변경
- 프로젝트 상세와 팀 목록의 기존 역할 대체

실제 외부 고객 접수는 별도의 향후 기능으로 다룬다. MVP 보완에서는 활성 워크스페이스 멤버가 고객 요청을 대신 등록하거나 내부 PM 역할로 접수하는 흐름을 사용한다.

## 2. 제품 개념과 사용자 용어

### 2.1 이슈와 작업의 구분

| 사용자 용어 | 내부 값 | 의미 |
| --- | --- | --- |
| 이슈 | `Issue` | 무엇을 만들거나 해결할지 나타내는 콘텐츠 정본 |
| 팀 작업 | `TeamWork` | 특정 팀과 담당자가 이슈를 실행하는 최소 상태 레코드 |
| 내 작업 | 사용자에게 할당된 `TeamWork` | 현재 사용자가 직접 처리할 미완료 작업 |
| 작업 시작 | 이슈를 하나 이상의 팀 작업으로 전환 | 프로젝트 역할을 선택하고 필요하면 담당자를 지정 |

팀 작업 목록에서는 실행 항목을 구분하기 위해 `팀 작업`을 사용하지만 제목·프로젝트·우선순위·라벨은 상위 이슈에서 읽는다.

### 2.2 화면별 책임

| 화면 | 책임 |
| --- | --- |
| 이슈 | 전체 기능 요청의 접수, 검토, 분류, 작업 시작과 완료 확인 |
| 내 작업 | 자신에게 할당된 미완료 팀 작업 처리 |
| 내 작업 상세 | 선택한 팀 작업을 처리하면서 상위 이슈의 공유 콘텐츠 확인 |
| 팀 | 한 팀의 팀 작업 목록과 상태 운영 |
| 프로젝트 | 한 프로젝트의 목표, 역할, 이슈 계층과 전체 진행 확인 |
| 이슈 상세 | 이슈 콘텐츠와 선택된 팀 작업의 상태·담당자·메모·연결·전달 확인 |

## 3. 내비게이션과 경로

### 3.1 사이드바

주 탐색은 다음 순서를 사용한다.

1. `이슈`
2. `내 작업`
3. `알림함`
4. `프로젝트`

팀 탐색은 기존처럼 주 탐색 아래에 둔다. 전역 `이슈 만들기`와 검색은 현재 위치를 유지한다.

기존 `내 이슈` 문구를 `내 작업`으로 변경한다. 경로는 하위 호환을 위해 `/my-issues`를 유지할 수 있으며 사용자에게 보이는 이름만 바꾼다.

### 3.2 화면 경로

| 화면 | 권장 경로 | 접근 |
| --- | --- | --- |
| 전역 이슈 목록 | `/issues` | 활성 멤버 |
| 이슈 상세 | `/issues/:issueDisplayId?work=:teamWorkDisplayId` | 활성 멤버 |
| 내 작업 | `/my-issues` | 활성 멤버 |
| 내 작업 상세 | `/my-issues/:teamWorkDisplayId?tab=:tab` | 활성 멤버 |

기존 `/issues/:teamWorkDisplayId`는 상위 이슈의 이슈 중심 정본 주소와 선택된 팀 작업 문맥으로 이동한다. 내 작업 행은 `/my-issues/:teamWorkDisplayId`를 생성하고 경로 자체로 작업을 선택한다. 두 상세 경로는 같은 이슈 콘텐츠와 팀 작업 데이터를 사용하며 내 작업 전용 상세 리소스를 만들지 않는다. 필터, 정렬과 커서는 URL 쿼리에 반영해 새로고침, 직접 링크와 브라우저 뒤로 가기를 지원한다. 개인 저장된 보기를 선택한 목록은 같은 membership의 브라우저 세션에서 다른 목록을 거쳐 다시 열어도 해당 정규화된 `view` URL을 복원한다.

### 3.3 로그인 후 기본 진입

MVP의 기본 로그인 진입은 `내 작업`을 유지한다. 자신에게 할당된 실행 작업을 먼저 처리하는 동작을 바꾸지 않는다. PM이나 분류 담당자는 사이드바의 `이슈`로 이동해 전체 접수 업무를 처리한다.

## 4. 사용자와 권한

MVP에는 별도 PM, 고객 또는 팀장 권한이 없다. 다음 동작은 기존 활성 멤버 권한 안에서 제공한다.

| 동작 | 활성 멤버 | 관리자 |
| --- | :---: | :---: |
| 전체 이슈 조회 | 가능 | 가능 |
| 이슈 생성·수정 | 가능 | 가능 |
| 작업 시작 | 가능 | 가능 |
| 자신이 속한 팀에서 시작 | 가능 | 가능 |
| 자신에게 할당 | 가능 | 가능 |
| 팀 작업 담당자 지정 | 가능 | 가능 |

담당자는 해당 팀의 활성 멤버만 선택할 수 있다. 사용자가 팀장 역할을 수행하더라도 MVP에서는 별도 권한으로 구분하지 않는다. 향후 팀장 전용 권한이 도입되면 권한 명세와 함께 제한한다.

다른 워크스페이스, 휴지통 이슈와 접근할 수 없는 리소스는 목록, 필터 후보와 배정 선택기에 노출하지 않는다.

## 5. 전역 이슈 목록 화면

### 5.1 화면 목적

전역 이슈 목록은 다음 질문에 빠르게 답해야 한다.

- 새로 들어왔지만 아직 작업을 시작하지 않은 이슈는 무엇인가?
- 팀 작업은 생겼지만 담당자가 없는 이슈는 무엇인가?
- 현재 어느 팀이 어떤 이슈를 진행하고 있는가?
- 모든 팀 작업이 끝나 최종 완료가 필요한 이슈는 무엇인가?
- 특정 프로젝트, 상태나 우선순위의 이슈는 무엇인가?

### 5.2 화면 헤더

- 첫 행: 제목 `이슈`, 제목 가까이의 현재 결과 수, 주요 동작 `이슈 만들기`
- 둘째 도구 행: 데스크톱 320~420px 표시 ID·제목 검색, 빠른 필터, 세부 필터, 정렬 기준과 방향
- 공간이 부족하면 검색·빠른 필터와 세부 필터·정렬 묶음을 두 줄로 자연스럽게 감싼다.
- 긴 화면 설명은 기본 본문에서 제거하고 필요할 때만 짧은 도움말로 제공한다.
- 보기: MVP에서는 목록만 제공
- Alpha에서는 제목 아래의 보기 탭에서 `전체`, 개인 저장 보기와 `새 보기`를 전환하고, 데스크톱 확장 사이드바에서는 `이슈`·`내 작업` 하위 메뉴로 개인 보기를 바로 연다. 모바일은 같은 기능을 compact 선택기로 제공한다.
- 현재 검색, 빠른·세부 필터, 정렬과 표시 옵션을 개인 저장된 보기로 저장한다. 선택한 보기의 구성을 바꾸면 `변경됨` 상태에서 초기화·새 보기로 저장·변경 저장을 선택한다. 공유·구독·권한별 보기와 중첩 OR 필터는 제공하지 않는다.

세부 필터와 정렬은 목록 전용 compact 도구 표현을 사용한다. 기본 상태에서는 상시 경계와 강한 배경을 표시하지 않는다. hover에는 약한 표면, open에는 한 단계 더 분명한 표면, focus-visible에는 2px 링을 표시한다. 활성 세부 필터가 있으면 `세부 필터 2`처럼 건수를 함께 표시한다. 전역 이슈의 다중 정렬 팝오버는 조건 번호, 필드, 방향, 위·아래 이동과 제거를 한 행에 제공하고 최대 세 조건을 키보드로 조작할 수 있게 한다. 표시 밀도는 현재 상태를 전달하는 단일 전환 버튼으로 제공한다.

이 Compact 도구 표현은 내 작업·팀 목록·보드에도 같은 높이·radius·상태로 적용한다. 필터 메뉴는 불투명한 Portal 팝오버로 열고 Escape와 키보드 선택으로 닫으면 호출부로 포커스를 돌린다. 바깥의 다른 조작 대상을 선택한 경우에는 그 대상의 포커스를 유지한다. 내 작업의 제목과 결과 수도 전역 이슈처럼 같은 헤더 행에 둔다.

공통 시각 표현은 두 목록의 정보 구조를 같게 만든다는 뜻이 아니다. 전역 이슈는 접수·분류를 위해 진행률·현재 작업·다음 행동을 표시하고, 내 작업은 개인 실행 큐로서 우선순위, 이슈 제목과 팀 작업 코드·이슈 코드·역할, 프로젝트와 이슈 라벨, 상태와 현재 실행 행동만 표시한다. 내 작업에는 현재 사용자 담당자, 실제 팀 이름과 최근 수정 시각을 기본 열로 반복하지 않는다.

목록과 도구 영역은 전역 이슈 화면 안에서 약 1,440~1,600px 최대 폭을 사용하고 앱 셸 안에 가운데 정렬한다. 1,280~1,440px에서는 사용 가능한 본문 폭을 유지하며 초광폭에서도 마지막 행동 열이 화면 끝으로 멀어지지 않게 열 사이 거리를 제한한다. 다른 화면의 전역 폭은 바꾸지 않는다.

### 5.3 빠른 필터

다음 빠른 필터를 제공한다.

| 빠른 필터 | 포함 조건 |
| --- | --- |
| 전체 | 삭제되지 않은 모든 이슈 |
| 검토 필요 | 이슈 상태가 `UNSORTED`이고 삭제되지 않은 유효 팀 작업이 없음 |
| 담당 필요 | 완료·취소가 아닌 팀 작업 중 담당자가 없는 작업이 하나 이상 있음 |
| 진행 중 | 이슈 상태가 `IN_PROGRESS` |
| 완료 확인 | 이슈 상태가 `REVIEW` |
| 완료 | 이슈 상태가 `DONE` |

`전체`가 기본값이다. `검토 필요`, `담당 필요`, `진행 중`은 업무 관점의 파생 조건이므로 서로 겹칠 수 있다. 빠른 필터는 기본 목록을 바꾸는 단일 선택이며, 세부 조건이 겹치는 이슈는 각각의 해당 필터에서 조회될 수 있다.

취소되지 않은 팀 작업이 하나도 없는 이슈는 자동으로 `UNSORTED`가 되어 `검토 필요`에 포함된다. 완료된 팀 작업이 하나 이상 있고 미완료 작업이 남으면 `IN_PROGRESS`, 유효 작업이 모두 완료되면 `REVIEW`로 계산한다.

각 빠른 필터에는 현재 검색과 세부 필터를 적용하되 선택된 `workQueue` 조건만 제외한 전체 결과 수를 표시한다. 현재 페이지의 행 수가 아니라 서버가 계산한 페이지네이션 전체 결과를 사용한다. 0건 필터도 선택할 수 있으며 시각적 강조만 낮춘다. 선택 상태는 색상뿐 아니라 밑줄·선택 배경 또는 굵기와 접근성 상태로 구분한다.

### 5.4 세부 필터

- 프로젝트
- 이슈 상태
- 우선순위
- 현재 작업 역할: 백엔드, 웹 프론트, 앱 프론트
- 담당자 없음
- 라벨
- 만든 사람
- 생성일과 최근 수정일 범위

빠른 필터와 세부 필터는 함께 적용한다. 같은 종류 안에서는 OR, 종류 사이에서는 AND로 결합한다. 필터 초기화는 빠른 필터를 `전체`, 정렬을 기본값으로 복구한다.

활성 세부 필터가 있으면 도구 아래에 필터 이름과 값, 개별 제거 동작이 있는 칩을 표시하고 전체 초기화를 제공한다. 활성 세부 필터가 없으면 빈 칩 영역을 차지하지 않는다.

### 5.5 정렬

- 최근 수정일
- 생성일
- 상태
- 우선순위
- 진행률

사용자는 고유 필드를 최대 세 개까지 우선순위 순서로 설정한다. 기본 정렬은 `최근 수정일 내림차순`이며 마지막 동률 해소에는 불변 `id`를 마지막 명시 조건과 같은 방향으로 적용한다. 진행률은 취소 작업을 제외한 표시용 반올림 백분율이다. 개인 저장된 보기에는 조건 배열과 순서를 함께 저장한다.

## 6. 목록 행과 현재 작업 요약

### 6.1 기본 정보

각 목록 행은 다음 일곱 영역으로 표시한다.

1. `우선순위`: 아이콘·한국어 이름의 compact 편집 트리거
2. `이슈`: 첫 줄 표시 ID·제목, 다음 줄 프로젝트, 최대 두 개 라벨과 나머지 `+N`
3. `상태`: 자동 계산된 이슈 상태의 아이콘·한국어 이름 읽기 표현
4. `현재 작업`: 현재 역할·실제 팀·진행 상태·담당 필요 개수·선행 작업 대기 원인
5. `진행`: 완료 작업 수/취소 제외 전체 대상 작업 수, 백분율과 짧은 진행 막대
6. `업데이트`: 최근 수정 상대 시각
7. `다음 행동`: 현재 조건에 따른 주요 행동 한 개

제목은 행에서 가장 강한 텍스트로 표시하고 프로젝트와 최대 두 개 라벨·`+N`은 같은 이슈 셀의 보조 줄에 둔다. 라벨이 없으면 빈 라벨 선택기를 기본 노출하지 않고 라벨 편집은 hover·focus 보조 동작이나 더보기에서 제공한다. 데스크톱 기본 행은 72~84px 안에서 유지한다. 편집 가능한 우선순위는 이슈 왼쪽의 좁은 독립 열에서, 자동 계산된 이슈 상태는 이슈 다음의 독립 열에서 제목보다 가벼운 아이콘·한국어 이름으로 표시한다. 헤더와 모든 행은 같은 Grid 템플릿을 공유해 상태·우선순위 값 길이가 다른 열의 시작점을 이동시키지 않는다. 이슈 상태에는 일반 선택기나 chevron을 표시하지 않는다. 만든 사람은 기본 열에서 제거하고 상세나 선택 가능한 보조 정보에서 확인한다.

우선순위 인라인 트리거는 평상시 배경과 경계가 투명하다. hover에는 약한 표면, open에는 한 단계 더 분명한 표면, focus-visible에는 2px 링을 표시한다. 이슈 상태는 같은 높이와 아이콘 크기의 읽기 표현을 사용하되 hover·open 표면을 만들지 않는다. 보이는 높이는 28~32px, 아이콘은 14~16px, 텍스트는 13~14px을 사용하고 편집 가능한 트리거의 실제 hit area는 데스크톱 40px·모바일 44px 이상을 유지한다. 색상만으로 값을 구분하지 않으며 중간 폭에서 우선순위 이름을 숨겨도 tooltip과 접근 가능한 이름에 전체 값을 유지한다.

전역 이슈 상태는 팀 작업과 같은 워크플로 상태 아이콘 체계를 사용한다. `접수됨`은 점선 원, `할 일`은 빈 원, `진행 중`과 `완료 확인`은 각각 1/3·2/3 파이, `완료`는 체크 원, `취소`는 취소 원으로 표시하고 `일시 중지`만 경고색 `CirclePause`를 사용한다. 우선순위는 `없음 Minus`, `낮음 SignalLow`, `보통 SignalMedium`, `높음 SignalHigh`, `긴급 CircleAlert` 순서다. 프로젝트 행과 상세가 같은 이슈 상태 매핑을 재사용하고 내 작업·팀 목록·보드는 팀별 워크플로 이름과 시스템 범주 아이콘을 사용한다.

팀 작업 전체를 목록 행 안에 펼치지 않는다. 현재 진행과 사용자의 다음 행동을 판단하는 최소 정보만 보여 주며 전체 작업 흐름은 이슈 상세의 `연결` 탭에서 확인한다.

### 6.2 현재 작업

현재 작업은 삭제되지 않고 완료·취소가 아닌 소속 팀 작업이다. 역할과 실제 팀, 진행 상태를 제목과 보조 정보 최대 두 줄로 표시한다. 병렬 작업이 두 개면 역할을 함께 표시하고 세 개 이상이면 `{첫 역할} 외 N개 병렬 진행`으로 줄이되 전체 역할·팀은 tooltip이나 상세에서 확인한다. 담당 필요 수는 별도 박스를 추가하지 않고 같은 보조 줄의 경고 텍스트로 제한한다.

예시:

- `분석 중 · 시작 팀 없음`
- `백엔드 진행 중`
- `백엔드 · 웹 프론트 병렬 진행`
- `웹 프론트 담당 필요`
- `모든 팀 작업 완료`

작업 순서로 대기 중이면 `API-12 완료 대기`처럼 실제 선행 작업을 표시한다. 내부 `차단됨` 표현만 단독으로 사용하지 않는다.

### 6.3 다음 행동 우선순위

목록 행의 주요 동작은 다음 우선순위로 하나만 표시한다.

1. 팀 작업이 없음: `작업 시작`
2. 미할당 미완료 팀 작업이 있음: `담당자 지정`
3. 현재 사용자가 맡을 수 있는 역할 작업이 없음: 조건을 충족하면 `우리 팀에서 시작`
4. 현재 사용자에게 할당된 작업이 있음: `내 작업 열기`
5. 모든 대상 팀 작업이 완료되고 이슈가 미완료: `이슈 완료`
6. 그 외: `상세 보기`

보조 동작은 더보기 메뉴에 둔다. 여러 주요 버튼을 한 행에 나열하지 않는다.

검토·담당·완료 확인처럼 실제 결정이 필요한 행동만 낮은 채도의 Secondary 또는 조건부 Primary, 단순 이동은 Ghost 또는 행 선택, 더보기는 상시 배경·경계가 없는 아이콘 버튼을 사용한다. 주요 행동의 보이는 높이는 30~32px, 더보기는 28~32px로 정돈하되 실제 hit area는 데스크톱 40px·모바일 44px 이상을 유지한다. 가능한 우리 팀 역할이 하나면 보이는 문구를 `작업 시작`처럼 줄일 수 있지만 tooltip과 접근 가능한 이름에는 `{역할} 작업 시작`을 유지한다. 여러 역할이면 `우리 팀에서 시작`을 유지해 선택 창에서 역할과 팀을 명확히 표시한다.

현재 조건에서 주요 행동을 실행할 수 없으면 비활성 버튼만 표시하지 않고 필요한 역할·팀·권한 또는 선행 작업 같은 이유를 그 자리에 설명한다.

행의 빈 영역을 선택하면 이슈 상세로 이동한다. 상태·우선순위·라벨 선택, 다음 행동, 더보기와 진행 관련 링크는 행 이동을 실행하지 않는다. 행 상세 링크와 내부 컨트롤은 중첩하지 않고 각각 키보드로 접근할 수 있게 한다.

## 7. 접수·분류와 작업 배정 흐름

### 7.1 작업 시작

팀 작업이 없는 이슈에서 `작업 시작`을 선택하면 프로젝트 역할을 표시한다.

역할별 입력:

- 역할과 연결된 팀
- 담당자 선택, 선택하지 않음 허용
- 이미 존재하는 미완료 역할 작업 여부

하나 이상의 역할을 선택해야 저장할 수 있다. 선택한 역할별로 팀 작업을 생성하거나 기존 미완료 작업을 재사용하고, 선택한 담당자를 같은 트랜잭션에서 반영한다.

담당자를 선택하지 않은 새 작업은 팀의 기본 `BACKLOG` 상태로 만들고, 담당자를 선택한 새 작업은 `UNSTARTED` 범주의 첫 상태로 만든다. 기본 `BACKLOG` 상태의 기존 작업을 재사용하면서 담당자를 지정하면 담당자와 `UNSTARTED` 전이를 같은 트랜잭션에서 저장한다.

여러 역할을 동시에 선택하면 백엔드·프론트 구분 없이 병렬로 시작하며 모두 즉시 작업 가능하다. 팀 작업 사이에 작업 순서나 전달 대기 관계를 자동 생성하지 않는다.

### 7.2 우리 팀에서 시작

현재 사용자가 속한 활성 팀이 이슈 프로젝트의 역할에 연결돼 있고 해당 역할의 미완료 작업이 없으면 `우리 팀에서 시작`을 제공할 수 있다.

- 가능한 역할이 하나면 해당 역할을 바로 확인한다.
- 가능한 역할이 여러 개면 역할을 선택한다.
- 팀 작업을 기본 `BACKLOG` 상태·담당자 없음으로 생성한다.
- 생성 뒤 해당 역할은 `담당 필요`로 표시하고 담당자가 정해지면 `시작 전`으로 전환한다.

사용자가 해당 역할 팀의 현재 멤버가 아니면 동작을 제공하지 않는다. 프로젝트 역할의 팀을 클라이언트 입력으로 신뢰하지 않고 서버가 다시 확인한다.

### 7.3 내가 맡기

현재 사용자가 속한 프로젝트 역할에 대해 다음 규칙을 적용한다.

- 미할당 미완료 팀 작업이 한 건 있으면 자신에게 할당한다.
- 미할당 작업이 여러 건이면 선택 창을 연다.
- 해당 역할 작업이 없으면 `UNSTARTED` 상태의 팀 작업을 생성하고 자신에게 할당한다.
- 이미 다른 담당자가 있으면 덮어쓰지 않고 상세 또는 담당자 변경 흐름을 사용한다.

생성 또는 재사용, 담당자 검증과 할당은 한 트랜잭션에서 처리한다.

`내가 맡기`는 행의 고정 주요 동작으로 항상 노출하지 않는다. 현재 사용자가 맡을 수 있고 더 높은 우선순위의 다음 행동이 없을 때 또는 행의 더보기 메뉴에서 제공한다.

### 7.4 담당자 지정

담당자가 없는 미완료 팀 작업을 역할별로 표시한다. 담당자 후보는 해당 작업 팀의 활성 멤버만 사용한다.

- 한 이슈의 여러 역할 작업을 한 창에서 지정할 수 있다.
- 기존 담당자가 있는 작업은 기본적으로 변경 대상에서 제외한다.
- 오래된 화면에서 담당자가 이미 지정됐으면 덮어쓰지 않고 최신값을 표시한다.
- 일부 작업만 저장하는 부분 성공을 허용하지 않는다.
- 기본 `BACKLOG` 상태의 작업에 담당자를 지정하면 해당 팀의 첫 `UNSTARTED` 상태로 함께 전환한다.
- 이미 `UNSTARTED`, `STARTED`, `COMPLETED`, `CANCELED`이거나 기본 상태가 아닌 수동 `BACKLOG` 작업은 담당자 변경만으로 현재 상태를 바꾸지 않는다.

### 7.5 이슈 완료

자동 계산된 이슈 상태가 `REVIEW`이면 `이슈 완료`를 제공한다.

실행 전 완료된 역할과 제외된 취소 작업을 요약한다. 저장하면 명시적 완료 행동으로 `DONE`을 적용한다. 팀 작업이 하나도 없는 `UNSORTED` 이슈에는 이 동작을 제공하지 않는다.

## 8. 목록 빠른 편집

목록에서 다음 속성을 변경할 수 있다.

- 우선순위
- 라벨

프로젝트는 이슈 생성 뒤 변경하지 않는 기존 정책을 유지한다. 이슈 상태는 읽기 전용이며 `일시 중지`, `재개`, `취소`, `다시 열기`, `이슈 완료`는 현재 조건에 맞는 주요 행동 또는 더보기 메뉴에서 제공한다. 팀 작업 상태와 담당자는 현재 작업 요약 또는 배정 창에서 변경하며 이슈 행에서 팀 작업 상태를 바꾸지 않는다.

빠른 편집은 낙관적으로 반영하되 저장 중을 표시하고 실패하면 이전 값으로 복구한다. 버전 충돌은 최신 이슈를 다시 읽고 같은 요청을 자동 반복하지 않는다.

저장 중에는 현재 값과 트리거의 포커스 가능성을 유지하고 `aria-busy` 또는 가까운 상태 문구로 진행을 전달한다. 재실행은 `aria-disabled`와 입력 차단으로 막으며 실패 시 행 가까이에 오류와 다시 시도를 표시한다.

라벨 편집 아이콘은 행 hover·focus·열림 상태에서만 표시한다. 라벨 팝오버는 행 stacking context 밖의 Portal과 완전히 불투명한 표면을 사용하고 약한 경계·shadow·viewport collision padding을 적용한다. 아래 행이나 우선순위 선택기가 팝오버 위로 그려지면 안 되며 Escape로 닫은 뒤 포커스는 호출 아이콘으로 돌아간다.

## 9. 화면 상태와 오류 복구

### 9.1 빈 상태

| 상태 | 제목 | 주요 동작 |
| --- | --- | --- |
| 워크스페이스에 이슈 없음 | `아직 등록된 이슈가 없습니다` | `첫 이슈 만들기` |
| 검토 필요 없음 | `검토를 기다리는 이슈가 없습니다` | `전체 이슈 보기` |
| 담당 필요 없음 | `담당자가 필요한 작업이 없습니다` | `전체 이슈 보기` |
| 필터 결과 없음 | `조건에 맞는 이슈가 없습니다` | `필터 초기화` |

빈 상태에 프로젝트 만들기를 주요 동작으로 제공하지 않는다. 전역 이슈 목록의 목적에 맞게 이슈 생성 또는 필터 복구를 제공한다.

### 9.2 로딩

- 첫 페이지는 목록 골격을 표시한다.
- 다음 페이지는 현재 결과를 유지한 채 하단에 로딩 상태를 표시한다.
- 필터 후보 하나의 실패로 전체 목록을 가리지 않는다.
- SSE 갱신 중 현재 필터, 정렬과 스크롤 위치를 유지한다.

### 9.3 오류

- 목록 실패: 전체 화면 오류와 다시 시도
- 빠른 편집 실패: 행 단위 복구와 다시 시도
- 작업 시작·배정 실패: 선택값을 유지하고 영향받은 역할 가까이 오류 표시
- 버전 충돌: 최신 이슈·팀 작업을 다시 읽고 사용자가 재실행
- 저장된 보기 충돌·삭제 실패: 현재 목록 URL과 스크롤을 유지하고 최신 보기 목록을 다시 읽어 이름·version 또는 보관된 필터 값을 수정하게 한다.
- 프로젝트 역할 변경: 사용할 수 없는 선택만 제거하고 현재 후보를 설명

## 10. API 계약

### 10.1 목록 조회

`GET /issues`는 이슈만 반환하므로 유형 필터 없이 다음 파생 필터를 사용한다.

```ts
type IssueWorkQueue =
  | "REVIEW_REQUIRED"
  | "ASSIGNMENT_REQUIRED"
  | "IN_PROGRESS"
  | "COMPLETION_REQUIRED"
  | "COMPLETED";
```

쿼리 파라미터:

- `workQueue`
- 기존 `projectId`, `status`, `priority`, `labelId`
- `activeProjectRole`
- `createdByMembershipId`
- `createdFrom`, `createdTo`, `updatedFrom`, `updatedTo`
- 기존 검색, 다중 정렬(`sorts`), 단일 정렬 호환, 커서와 제한

한 요청에서 `workQueue`는 하나만 받는다. 세부 필터와 함께 적용한다.

### 10.2 목록 응답

이슈 목록 항목에 다음 작업 요약을 제공한다.

```ts
type IssueWorkflowSummary = {
  teamWorkCount: number;
  completedCount: number;
  canceledCount: number;
  unassignedCount: number;
  activeRoles: ProjectRole[];
  waitingOn: Array<{
    issueId: string;
    identifier: string;
    title: string;
  }>;
  allTargetTasksCompleted: boolean;
};

type IssueWorkQueueCounts = {
  ALL: number;
  REVIEW_REQUIRED: number;
  ASSIGNMENT_REQUIRED: number;
  IN_PROGRESS: number;
  COMPLETION_REQUIRED: number;
  COMPLETED: number;
};
```

`IssueSummary`에 `workflowSummary`와 `workQueueCounts`를 제공한다. 현재 검색과 세부 필터는 유지하고 요청의 `workQueue`만 제외한 동일 조건으로 여섯 건수를 계산한다. 서버는 목록 페이지의 이슈 집합을 기준으로 팀 작업과 대기 관계를 일괄 집계해 행별 추가 조회를 만들지 않는다.

### 10.3 작업 시작과 담당자 지정

기존 `POST /issues/{issueId}/start`는 `initialRoles` 계약을 유지한다. 전역 이슈 화면에서 역할별 담당자를 함께 지정할 수 있도록 다음 선택 필드를 추가한다.

```ts
type InitialRoleAssignment = {
  projectRole: ProjectRole;
  assigneeMembershipId: string | null;
};
```

요청은 기존 `initialRoles` 또는 새 `roleAssignments` 중 하나만 사용한다.

```json
{
  "roleAssignments": [
    {
      "projectRole": "BACKEND",
      "assigneeMembershipId": "멤버십 UUID"
    },
    {
      "projectRole": "WEB_FRONTEND",
      "assigneeMembershipId": null
    }
  ]
}
```

`roleAssignments`의 역할은 중복될 수 없고 프로젝트에 설정된 역할이어야 한다. 담당자는 해당 역할 팀의 활성 멤버여야 한다. 생성·재사용한 팀 작업, 담당자 적용, 활동과 Outbox를 한 트랜잭션에서 커밋한다.

담당자가 `null`인 역할은 기본 `BACKLOG`, 담당자가 있는 역할은 첫 `UNSTARTED` 상태를 사용한다. 여러 역할을 함께 요청하면 모두 병렬 작업이며 프론트 역할도 백엔드 전달을 기다리지 않는다.

기존 `initialRoles` 요청은 담당자 없이 시작하는 현재 동작을 유지한다.

### 10.4 내가 맡기

`POST /issues/{issueId}/claim`을 제공한다.

```json
{
  "projectRole": "WEB_FRONTEND",
  "teamWorkId": null
}
```

- `teamWorkId`가 있으면 해당 이슈의 미할당 미완료 팀 작업인지 확인한다.
- 생략하면 해당 역할의 미할당 미완료 작업이 한 건일 때 재사용하고, 없으면 새 작업을 생성한다.
- 후보가 여러 건이면 `409 CLAIM_TARGET_REQUIRED`와 후보 작업을 반환한다.
- 현재 사용자가 해당 역할 팀의 활성 멤버가 아니면 `403 TEAM_MEMBERSHIP_REQUIRED`다.
- 다른 담당자가 생겼으면 `409 ISSUE_ASSIGNMENT_CONFLICT`다.

성공 응답은 상위 이슈 요약, 생성 또는 할당한 팀 작업과 갱신된 `workflowSummary`를 반환한다.

기본 `BACKLOG` 작업을 자신에게 할당하면 첫 `UNSTARTED` 상태 전이도 같은 트랜잭션에서 반영한다.

### 10.5 기존 작업 담당자 지정

팀 작업 수정 API의 `assigneeMembershipId` 계약을 사용한다. 한 이슈의 여러 미할당 작업을 한 번에 지정하는 UI는 역할별 변경을 순차 요청하지 않고 서버의 일괄 배정 엔드포인트를 사용한다.

```text
POST /issues/{issueId}/assign-team-works
```

```json
{
  "assignments": [
    {
      "teamWorkId": "팀 작업 UUID",
      "assigneeMembershipId": "멤버십 UUID",
      "version": 3
    }
  ]
}
```

모든 작업과 담당자를 검증한 뒤 한 트랜잭션에서 반영한다. 한 건이라도 충돌하면 전체를 롤백하고 최신 후보를 반환한다.

각 대상이 기본 `BACKLOG` 상태이면 담당자 지정과 첫 `UNSTARTED` 전이를 함께 저장한다. 담당자 해제·교체와 기본 상태가 아닌 작업은 현재 상태를 유지한다.

## 11. 데이터·트랜잭션과 권한 규칙

- 이슈에 팀이나 담당자 컬럼을 추가하지 않는다.
- 팀·역할·워크플로 상태·담당자와 작업 메모는 `team_works`에만 저장한다.
- 이슈의 자동 상태는 직계 팀 작업 변경과 같은 트랜잭션에서 저장하고, 빠른 필터는 저장된 이슈 상태와 현재 담당자 조건으로 계산한다.
- 완료·취소 판정은 기존 상태 범주를 사용한다.
- 작업 시작, claim과 일괄 배정은 상위 이슈와 영향받는 팀 작업을 잠그고 최신 상태를 다시 확인한다.
- 새 팀 작업의 팀은 프로젝트 역할 연결에서 서버가 결정한다.
- 활동과 알림 Outbox는 업무 변경과 같은 트랜잭션에 저장한다.
- 다른 워크스페이스 ID와 비활성·보관된 팀·멤버는 요청값으로 사용할 수 없다.
- 이슈와 팀 작업 표시 ID 형식, 진행률과 제품 동작을 유지하되 작업 순서와 전달은 새 팀 작업 ID를 참조한다.

## 12. 알림과 실시간 갱신

- 새 담당자가 지정되면 `TEAM_WORK_ASSIGNED` 알림 규칙을 사용한다.
- 자신에게 할당하는 사용자는 자신의 행동 알림을 받지 않는다.
- 담당자 없는 팀 작업 생성은 기존 팀 전달 알림 정책이 적용되는 경우에만 알리고, 목록 화면 추가만으로 새 전역 알림 유형을 만들지 않는다.
- 이슈·팀 작업·프로젝트 역할 변경 신호를 받으면 현재 목록 캐시를 무효화한다.
- 갱신 뒤 현재 빠른 필터 조건에서 제외된 행은 자연스럽게 목록에서 제거하되 사용자의 필터와 스크롤을 초기화하지 않는다.
- 열린 배정 창에서 후보가 바뀌면 저장 전 최신값을 다시 확인한다.

## 13. 모바일과 접근성

### 13.1 모바일

- 사이드바의 `이슈`와 `내 작업`을 모바일 탐색에서도 같은 이름으로 제공한다.
- 목록 행은 표를 가로로 축소하지 않고 카드로 재배치한다.
- 카드는 표시 ID·제목, 프로젝트·라벨, 상태·우선순위, 현재 작업·담당 필요, 진행률, 업데이트, 주요 행동 순으로 읽힌다.
- 이슈 상태는 카드에서도 같은 아이콘·이름의 읽기 표현을 사용하고 우선순위만 compact 인라인 트리거로 제공하며 큰 폼 Select 표면으로 바꾸지 않는다.
- 빠른 필터는 각 전체 건수를 포함한 가로 스크롤 탭 또는 접근 가능한 선택기로 제공한다.
- 필터는 하단 시트 또는 전체 화면 패널로 연다.
- 작업 시작, claim과 담당자 지정은 모바일에서도 제공한다.
- 작업 흐름 상세 편집처럼 기존에 데스크톱으로 제한한 동작은 현재 정책을 유지한다.

### 13.2 접근성

- 데스크톱 목록은 의미 있는 표 또는 목록 구조를 사용한다.
- 표를 사용하면 열 헤더와 데이터 셀의 관계를 연결하고, 목록을 사용하면 행의 제목·보조 정보·행동이 하나의 항목으로 읽히게 한다.
- 행 전체 클릭과 내부 선택기·버튼의 키보드 동작을 충돌시키지 않는다.
- 빠른 필터의 선택 상태와 결과 수를 보조 기술에 전달한다.
- 이슈 상태의 읽기 표현과 우선순위 선택기는 표시 ID와 속성 이름을 포함한 명확한 접근 가능 이름을 가진다.
- 상태·우선순위의 아이콘 단독 축약은 tooltip과 현재 한국어 값을 포함한 접근 가능한 이름을 제공하고 편집 가능한 트리거의 focus-visible에서는 2px 링을 표시한다.
- 라벨 팝오버는 아래 행과 시각적·입력적으로 겹치지 않으며 Escape로 닫으면 호출부에 포커스를 돌린다.
- Compact 필터도 같은 Popover 키보드 계약을 사용하고, 상세의 Comfortable 속성 트리거는 같은 아이콘·이름·메뉴 순서를 유지한 채 34~38px 시각 높이와 44px 조작 영역을 사용한다.
- 진행률은 완료 수·전체 수·백분율 텍스트와 `aria-valuenow` 또는 동등한 값을 함께 제공한다.
- `검토 필요`, `담당 필요`와 대기 상태를 색상만으로 표시하지 않는다.
- 목록 갱신 결과를 과도하게 읽지 않고 저장 성공·실패만 필요한 범위에서 알린다.
- 작업 시작과 배정 창은 제목, 역할, 담당자, 오류와 주요 동작 순으로 읽힌다.
- 저장 실패 시 입력값을 유지하고 첫 오류로 포커스를 이동한다.
- 200% 확대와 모바일 폭에서 주요 동작이 잘리거나 가로 스크롤에만 의존하지 않는다.

## 14. 성능과 페이지네이션

- 기본 페이지 크기와 상한은 기존 이슈 목록 계약을 따른다.
- 빠른 필터와 하위 작업 집계는 데이터베이스에서 페이지 단위로 처리한다.
- 대기열별 건수는 선택된 `workQueue`를 제외한 동일한 검색·세부 필터로 전체 결과를 집계하며 현재 페이지 행을 세지 않는다.
- 이슈별 팀 작업을 추가 조회하는 N+1 패턴을 만들지 않는다.
- 목록 커서는 필터·정렬 조건과 함께 검증하고 다른 조건에 재사용하지 않는다.
- SSE는 변경 신호로만 사용하고 목록 데이터는 REST로 다시 조회한다.
- 필터와 정렬 변경 시 이전 요청을 취소하거나 오래된 응답을 무시한다.

## 15. 수용 기준

### 15.1 내비게이션

- 사용자는 프로젝트를 열지 않고 사이드바에서 전체 이슈 목록으로 이동할 수 있다.
- `이슈`는 콘텐츠 정본, `내 작업`은 자신에게 할당된 `TeamWork`를 의미한다.
- 기존 `/my-issues` 직접 링크는 유지되고 화면 이름만 `내 작업`으로 표시된다.
- 내 작업 행을 열면 `/my-issues/:teamWorkDisplayId`에서 사이드바의 `내 작업`과 `내 작업` 복귀 링크가 유지된다.
- 이슈 상세와 내 작업 상세는 상단 정보 위계만 진입 목적에 맞게 구분하고 설명·첨부파일·댓글·전달·활동과 데이터 원본을 공유한다.
- 이슈 목록 필터와 정렬은 URL과 브라우저 뒤로 가기에서 복원된다.

### 15.2 목록과 필터

- 전역 목록에는 삭제되지 않은 이슈만 표시된다.
- 검토 필요, 담당 필요, 진행 중, 완료 확인과 완료 조건이 서버 결과와 일치한다.
- 프로젝트, 상태, 우선순위, 역할, 라벨과 생성자 필터를 조합할 수 있다.
- 목록 행에서 현재 작업 역할, 담당 필요, 진행률과 다음 행동을 확인할 수 있다.
- 우선순위·제목·프로젝트·라벨·상태를 각각의 정해진 열에서 읽고 만든 사람 없이 일곱 영역 안에서 현재 작업과 다음 행동을 우선 판단할 수 있다.
- 이슈 상태는 Select가 아닌 읽기 전용 아이콘·이름으로, 우선순위는 compact 편집 트리거로 각각 독립된 고정 시작점에서 읽힌다.
- 라벨 팝오버는 불투명한 Portal 레이어로 표시되어 아래 행 컨트롤이 비치거나 위로 겹치지 않는다.
- 세부 필터·정렬·주요 행동·더보기는 compact 위계를 사용하고 행의 상시 경계·표면 반복을 만들지 않는다.
- 내 작업·팀 목록·보드와 프로젝트 작업 행은 같은 상태·우선순위 표현을 사용한다. 내 작업의 라벨은 상위 이슈가 소유한 읽기 전용 값으로 프로젝트 아래에만 표시하고, 담당자·실제 팀 이름은 화면 책임에 필요한 팀 목록·보드·프로젝트 행에서만 표시한다. 상세 속성은 더 넉넉한 Comfortable 밀도를 적용한다.
- 각 빠른 필터의 건수는 검색·세부 필터를 반영한 전체 결과와 일치하며 0건 필터도 선택할 수 있다.
- 페이지가 바뀌어도 같은 정렬 값의 이슈가 중복되거나 누락되지 않는다.
- 전역 이슈에서 최대 세 정렬 조건을 추가·재배치·제거하고 저장된 보기로 같은 순서를 복원할 수 있다.

### 15.3 작업 시작과 배정

- 팀 작업이 없는 이슈에서 역할과 선택 담당자를 함께 지정할 수 있다.
- 여러 역할 작업과 담당자는 전부 커밋되거나 전부 롤백된다.
- 우리 팀에서 시작은 현재 사용자가 실제로 속한 프로젝트 역할에만 제공된다.
- 내가 맡기는 기존 미할당 작업을 재사용하고 없으면 새 작업을 생성한다.
- 이미 다른 담당자가 있는 작업을 오래된 화면에서 덮어쓰지 않는다.
- 여러 미할당 작업의 담당자 지정은 부분 성공하지 않는다.
- 배정 뒤 전역 이슈, 내 작업과 팀 목록이 같은 결과를 표시한다.
- 팀 작업 생성·상태 변경·삭제·복구 뒤 상위 이슈 상태가 같은 응답과 SSE 재조회에서 자동 전이 결과로 수렴한다.

### 15.4 구조와 품질

- 이슈 생성, 작업 시작, 프로젝트와 팀 목록은 분리된 `issues`·`team_works` 모델로 동작한다.
- `IssueType`, `FEATURE`, `TEAM_TASK` 계약과 상위 이슈 없는 단독 팀 작업은 제거한다.
- 목록 추가로 새로운 중복 알림이 생기지 않는다.
- 데스크톱과 모바일에서 필터, 주요 동작과 빈 상태를 사용할 수 있다.
- 키보드만으로 목록, 빠른 필터, 작업 시작과 배정을 완료할 수 있다.
- 전역 이슈·내 작업·팀 목록·보드·프로젝트 행·이슈 중심 상세·내 작업 진입 상세를 같은 데이터 상태로 비교했을 때 상태·우선순위 아이콘, 라벨 칩, 버튼 강조와 표면 규칙이 일치한다.
- 내 작업 행을 열면 `/my-issues/:teamWorkDisplayId`에서 해당 작업이 선택되고 `내 작업` 활성 상태를 유지한 채 이슈 설명·첨부파일·댓글과 실행 속성을 함께 확인한다.
- 팀 목록의 행을 열면 이슈 중심 상세에서 해당 팀 작업이 선택된다.
- 두 상세 진입 경로는 공유 본문·조회 계약을 재사용하고 별도 TeamWork 콘텐츠 페이지를 중복 구현하지 않는다.
- 1024px·모바일·실제 200% 확대에서도 목록과 속성 패널의 주요 조작이 잘리거나 가로 스크롤에만 의존하지 않는다.
