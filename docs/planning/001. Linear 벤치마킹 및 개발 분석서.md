# Linear 벤치마킹 및 개발 범위 분석

## 한눈에 보는 결론

Linear를 복제할 때 핵심은 화면 모양이 아니라 **이슈를 빠르게 만들고, 분류하고, 담당하고, 실행하고, 완료하는 끊김 없는 작업 루프**다. 현재 Linear는 이슈 트래커를 넘어 고객 요청, 로드맵, 릴리스, 분석, 코드 리뷰, AI 에이전트까지 아우르는 제품 개발 시스템이다. 이 범위를 첫 제품에 모두 담으면 구현 범위와 운영 위험이 크게 늘어난다.

개발 우선순위는 다음과 같이 잡는 편이 현실적이다.

1. **P0, 반드시 만든다:** 워크스페이스·팀, `admin/member` 기본 역할, 이슈, 팀별 워크플로, 담당자·우선순위·라벨, 댓글·활동 기록, 목록·보드, 필터·검색, 개인 작업함, 최소 알림.
2. **P1, 제품성이 확인된 뒤 만든다:** 프로젝트, 저장된 보기, 사이클, 하위 이슈·관계, 템플릿, GitHub 연동, 트리아지, 운영 안전장치인 CSV 내보내기.
3. **P2 이후, 고객 요구가 생길 때 만든다:** 비공개 팀·게스트·세밀한 권한, Slack·지원 도구 연동, API·웹훅, 마일스톤·프로젝트 업데이트, 전환 장벽을 낮추는 CSV 가져오기와 Issue sync.
4. **초기에는 제외한다:** Pulse, Issue SLAs, Initiatives, Insights·대시보드, Customer Requests·Asks, Releases, 다단계 서브팀, SAML·SCIM·감사 로그·데이터 리전, Agent platform·MCP access·Linear Agent·Code Intelligence·Coding Sessions·Diffs.

가장 큰 숨은 난이도는 기능 CRUD가 아니다. **권한이 반영된 실시간 동기화, 여러 화면에 걸친 동일 데이터의 일관성, 수동 정렬, 알림 중복 제거, 팀 이동 시 상태·식별자 재매핑, 외부 연동의 멱등성**이 실제 개발비를 만든다. 1인 또는 소규모 팀이라면 Linear가 공식적으로 설명하는 클라이언트 캐시 기반 실시간 동기화와 제한적 오프라인 재시도 구조를 처음부터 그대로 재현할 필요는 없다. 서버를 진실의 원천으로 두고 낙관적 UI와 제한적인 실시간 갱신을 적용해 체감 속도부터 맞추는 편이 낫다.

## 1. 조사 범위와 사실 구분

이 보고서는 2026-07-10에 확인한 Linear 공식 제품 문서, 공식 가격표, 공식 개발자 문서, 공식 보안 문서와 공식 변경 내역을 우선 근거로 사용했다. 기능의 존재와 동작은 **확인된 사실**, 복제 제품의 우선순위와 구현 전략은 **분석·추론**, 실제 개발 기간과 시장 반응은 **불확실성**으로 구분한다.

- **확인된 사실:** 현재 제품에 존재한다고 공식 문서에서 확인되는 기능, 역할, 제약, 요금제 구분.
- **분석·추론:** 해당 기능이 핵심 루프에 얼마나 기여하는지, MVP에서 복제하거나 제거해야 하는지에 대한 판단.
- **불확실성:** Linear 내부 구현의 세부, 실제 기능별 사용률, 복제 제품의 고객군·기술 스택·품질 목표가 정해지지 않아 달라질 수 있는 범위.

공식 가격표는 기능 범위를 파악하는 신호로만 사용했다. 2026-07-10 기준 Free는 무료, Basic은 연간 결제 기준 사용자당 월 10달러, Business는 사용자당 월 16달러, Enterprise는 별도 견적이다. 다만 가격표의 고객 수 표현은 공급사 주장이지 독립적으로 검증된 채택률 근거가 아니므로 제품 우선순위의 근거로 사용하지 않았다.

## 2. Linear의 제품 정체성

### 2.1 확인된 사실

Linear의 현재 기능 표면은 크게 다음 흐름으로 묶인다.

- **Intake:** 이슈 생성, 트리아지, 고객 요청, Slack·이메일·웹 폼 입력.
- **Plan:** 프로젝트, 사이클, Initiatives, 타임라인, 마일스톤, 프로젝트 업데이트.
- **Build:** 팀별 워크플로, 담당자·우선순위, GitHub·GitLab 연동, 댓글·문서, 에이전트.
- **Review and ship:** 코드 리뷰, 릴리스·배포 상태 연결.
- **Monitor:** 검색, 보기, 알림, Insights, 대시보드, 고객 영향 추적.

Linear가 공식적으로 강조하는 제품 원칙은 범용 데이터베이스처럼 무엇이든 구성하게 하는 것이 아니라, 소프트웨어를 만드는 사람을 위해 목적에 맞게 설계하고 불필요한 운영 작업을 줄이는 것이다. 우선순위가 `없음·낮음·중간·높음·긴급`으로 제한되고, 이슈 담당자가 한 명이며, 이슈가 반드시 한 팀에 속하고, 사이클이 반복 일정으로 자동 생성되는 것도 이 관점과 일치한다.

### 2.2 분석·추론

Linear의 실제 제품 정체성은 “기능이 많은 프로젝트 관리 도구”보다 **결정 비용과 조작 비용이 낮은, 의견이 강한 실행 시스템**에 가깝다. 복제 제품은 다음 세 속성을 우선 보존해야 한다.

1. **입력 속도:** 어느 화면에서든 이슈를 바로 만들고 핵심 속성을 키보드로 붙인다.
2. **상태의 가시성:** 이슈 하나의 상태 변경이 목록, 보드, 개인 작업, 프로젝트, 알림에 같은 의미로 반영된다.
3. **의견 있는 기본값:** 선택지를 무한히 늘리는 대신 팀, 담당자 한 명, 제한된 우선순위, 정해진 상태 범주를 제공한다.

화면의 미니멀한 외형만 복제하고 이 세 가지를 놓치면 “Linear처럼 보이지만 느리고 믿을 수 없는 칸반 앱”이 된다.

## 3. 핵심 사용자 루프

Linear의 기능은 세 가지 주기로 이해하면 범위를 자르기 쉽다.

### 3.1 일상 실행 루프

`이슈 생성 → 팀·상태·담당자·우선순위 지정 → 댓글과 자료로 명확화 → 작업 중 상태 변경 → 완료 → 구독자 알림`

이 루프가 복제 제품의 P0다. 사용자는 이슈를 빠르게 만들고, 자신에게 필요한 작업을 찾고, 상태를 바꾸고, 협업 내용을 남기고, 완료 여부를 신뢰할 수 있어야 한다.

### 3.2 계획·정리 루프

`새 요청 수집 → 트리아지 → 백로그 정리 → 프로젝트·사이클에 배치 → 진행·범위 확인 → 미완료 작업 이월`

사이클과 프로젝트는 핵심 이슈 루프를 보강하지만, 초기에는 둘을 모두 완성할 필요가 없다. 특정 고객군이 스프린트형 운영을 강하게 요구하면 사이클을, 여러 이슈를 기능 단위로 묶는 요구가 강하면 프로젝트를 먼저 택한다. 일반적인 소프트웨어 팀용 제품이라면 프로젝트가 사이클보다 범용성이 높다.

### 3.3 조직 정렬 루프

`고객·현업 요청 → 프로젝트 후보 → Initiative·로드맵 → 프로젝트 업데이트 → 분석·대시보드 → 릴리스·고객 영향 확인`

이는 중대형 조직에서 가치가 커지는 루프다. 작은 팀의 첫 버전에는 데이터와 운영 규칙이 부족해 기능을 만들어도 빈 화면이 되기 쉽다. 따라서 초기 제품에서는 제외하는 것이 안전하다.

## 4. 주요·핵심 기능

### 4.1 P0: 제품 성립에 필요한 기능

### 4.2 P1: 사용이 반복될 때 필요한 기능

- **프로젝트:** 여러 팀 또는 여러 이슈를 하나의 결과물로 묶고, 리드·상태·시작일·목표일·설명을 둔다.
- **저장된 보기:** 필터·정렬·그룹·레이아웃 구성을 이름 붙여 재사용하고 공유한다.
- **사이클:** 팀별 반복 기간, 현재·다음 사이클, 미완료 이슈 이월, 기본 진행률을 제공한다.
- **하위 이슈와 관계:** 큰 이슈 분해, `blocked by / blocking / related / duplicate`를 표현한다.
- **템플릿:** 반복되는 이슈 형식과 기본 속성을 미리 채운다.
- **트리아지:** 외부 또는 타 팀에서 온 요청을 일반 워크플로에 넣기 전 수락·거절·중복·보류한다.
- **GitHub 연동:** 이슈 ID가 포함된 브랜치·PR을 연결하고 PR 상태에 따라 이슈 상태를 옮긴다.
- **CSV 내보내기:** 운영 안전장치로 기본 이슈 데이터를 사용자가 회수할 수 있게 한다.

이 기능들은 핵심 루프를 확장하지만, P0 데이터 모델이 안정된 뒤 붙여야 한다. 특히 저장된 보기는 단순 화면 저장이 아니라 권한이 반영된 쿼리 정의이고, GitHub 연동은 웹훅 재전송·순서 뒤바뀜·중복 이벤트를 처리해야 한다.

### 4.3 P2 이후: 이관·외부 협업 요구가 생길 때 필요한 기능

- **CSV 가져오기:** 기존 도구에서 옮겨오는 전환 장벽을 낮춘다. 필드·사용자·상태 매핑과 실패 복구가 필요하므로 내보내기와 별도 단계로 둔다.
- **비공개 팀·게스트·세밀한 권한:** 외부 협업이나 민감한 팀 데이터가 실제 고객 요구가 될 때 추가한다.
- **Issue sync:** Jira·GitHub Issues 같은 다른 이슈 시스템과 단방향 또는 양방향으로 객체와 상태를 맞춘다. 일회성 가져오기와 달리 필드 매핑, 충돌, 권한, 동기화 오류의 지속적인 운영이 필요하다.

## 5. 서브 기능

다음은 유용하지만 제품의 첫 성공 조건은 아닌 기능이다.

- 이슈 기한·추정치, 반복 이슈, 리마인더, 즐겨찾기, 임시 초안.
- 댓글 스레드·해결, 이모지 반응, 설명 버전 복원.
- 프로젝트 마일스톤, 의존성, 건강도 업데이트, 타임라인.
- 커스텀 보기 구독, Slack·이메일·데스크톱·모바일 알림 채널.
- 프로젝트·이슈 문서, 협업 편집, 인라인 댓글.
- 팀 은퇴·자동 닫기·자동 아카이브, 라벨 보관·병합.
- Slack·Microsoft Teams·Sentry·GitLab·지원 도구 연동.
- 공개 API, OAuth 앱, 웹훅, 데이터 웨어하우스 동기화.

첨부파일은 실제 개발 업무의 맥락 보존에 필요하므로 이슈 일반 첨부, 편집기 이미지와 프로필 사진을 MVP에 포함한다.

### 5.1 2026년 기능 표면 중 초기 분류

- **Pulse — 서브 기능, 초기 제외:** 공식 문서상 프로젝트·Initiative 업데이트를 모아 읽고 일간·주간 요약을 Inbox로 받는 피드다. 프로젝트 업데이트 체계가 먼저 있어야 가치가 생기므로 조직 정렬 단계에서 추가한다.
- **Issue sync — P2 서브 기능, 초기 제외:** Jira와는 신규 이슈·프로젝트 및 일부 속성을, GitHub Issues와는 신규 이슈와 일부 속성을 단방향 또는 양방향으로 맞추는 전환·공존 기능이다. 핵심 이슈 루프가 아니라 이관·외부 시스템 공존을 위한 기능으로 본다.
- **Issue SLAs — 특수 운영 서브 기능, 초기 제외:** Business·Enterprise에서 조건에 맞는 이슈에 처리 기한을 자동 적용하고 위험·위반 상태와 알림을 제공한다. 지원·장애·시간 민감 업무가 핵심 고객 요구일 때 P2 이후 검토한다.
- **Agent platform — AI 확장 플랫폼, 초기 제외:** 공식 개발자 문서상 에이전트가 앱 사용자로 설치되어 멘션·위임·댓글·프로젝트 작업에 참여하는 구조이며 관련 API는 Developer Preview다. 별도 인증, 권한, 세션 수명주기와 외부 실행 신뢰가 필요하므로 P4 범위다.
- **MCP access — AI·외부 도구 연결, 초기 제외:** 호환 AI 클라이언트가 인증을 거쳐 이슈·프로젝트·댓글을 조회·생성·수정하게 하는 원격 인터페이스다. 안정된 API와 권한 모델이 선행되어야 하므로 P4 범위다.

이 다섯 기능은 2026-07-10 공식 가격표에 별도 제품 항목으로 제시되지만, 복제 제품의 P0 핵심으로 분류할 근거는 없다. 각각 상위 데이터와 운영 체계가 자리 잡은 뒤 가치를 내는 파생 기능이다.

이 중 **기한, 추정치, 반복 이슈**는 작아 보이지만 시간대, 스케줄러, 누락 복구, 통계 정의가 따라온다. **문서 협업 편집**은 별도 제품 수준의 동시 편집·버전 관리 문제를 만든다. UI에 버튼 하나를 더하는 비용으로 보면 안 된다.

### 7.2 식별자 설계

공식 문서상 Linear 이슈는 팀 식별자와 고유 번호로 표시되고, 다른 팀으로 옮기면 새 표시 ID를 받지만 이전 ID와 URL은 계속 현재 이슈로 연결된다. 이를 흉내 내려면 `issue.id` 같은 불변 내부 ID와 `TEAM-123` 같은 가변 표시 ID를 분리하고, 이전 표시 ID를 별도 별칭 테이블에 보관해야 한다.

MVP에서 팀 이동을 지원하지 않는다면 이 복잡도를 미룰 수 있다. 팀 이동이 필요하다면 상태, 팀 라벨, 사이클, 프로젝트 적합성을 어떻게 재매핑할지 먼저 규칙을 정해야 한다.

### 7.3 상태 모델

Linear 이슈 상태는 팀별로 이름과 순서를 바꿀 수 있지만 시스템 범주는 `Backlog, Unstarted, Started, Completed, Canceled`로 고정되고 중복에는 예약된 `Duplicate` 상태가 있다. 트리아지는 일반 워크플로 밖의 특별한 수신함으로 동작한다.

복제 제품도 다음 두 층을 나누는 편이 좋다.

- `WorkflowState`: “Todo”, “In Review”, “Ready to Merge”처럼 팀이 보는 상태.
- `StateCategory`: 미착수·진행·완료 같은 시스템 의미. 검색, 기본 보기, 진행률, 자동화가 이 범주를 사용.

허용할 상태 전이를 모두 강제할 필요는 없다. Linear형 제품의 장점은 빠른 직접 조작이므로, 초기에는 어떤 상태로도 이동할 수 있게 하고 변경 기록과 알림을 남기는 편이 낫다. 자동화가 생긴 뒤에만 전이 조건을 추가한다.

프로젝트 상태는 이슈 상태와 별개로 두어야 한다. 공식 문서상 프로젝트 상태는 이슈가 모두 끝나도 자동 완료되지 않는다. 이 구분은 “작업 완료율”과 “프로젝트 의사결정”을 섞지 않게 한다.

### 7.4 권한 구조

현재 Linear에는 워크스페이스 Owner·Admin·Member·Guest, 팀 Owner가 있고, 공개·제한·비공개 팀과 Enterprise의 개별 비공개 이슈 공유가 조합된다. 게스트는 초대된 팀의 이슈·프로젝트·문서를 사용할 수 있지만 Initiatives나 Customer Requests 같은 워크스페이스 전역 기능에는 접근하지 못한다. 여러 팀 프로젝트에서는 프로젝트 껍데기를 보더라도 허용된 팀 이슈만 보이는 식의 부분 공개가 생긴다.

이 구조를 그대로 복제하면 거의 모든 쿼리, 검색, 알림, 웹훅, 첨부 링크에 행 단위 권한 검사가 필요하다. MVP 권한은 다음처럼 줄이는 것이 낫다.

1. P0 워크스페이스 역할은 `admin/member`만 둔다.
2. P0의 팀 데이터는 모든 워크스페이스 멤버가 보는 공개 범위로 통일한다.
3. 프로젝트는 연결된 이슈보다 넓은 정보를 노출하지 않는다.
4. private 팀은 P2, 게스트·제한 팀·개별 이슈 공유·서브팀 상속은 그 이후로 보류한다.
5. private 범위를 도입할 때 검색 색인과 알림 생성에도 같은 권한 정책을 재사용한다.

권한 로직을 UI에서 숨기는 것으로 처리하면 안 된다. 서버 쿼리, 검색 결과, 직접 링크, 파일 다운로드, 이벤트 전달에서 같은 정책을 적용해야 한다.

## 8. 검색·알림·동기화·통합

### 8.1 검색

Linear 검색은 이슈 ID, 제목, 설명, 댓글을 포함하고 프로젝트와 문서도 검색한다. 빠른 이슈 검색과 현재 보기 안의 제목 필터는 전체 검색과 별도 경험으로 제공된다.

MVP는 두 단계가 적절하다.

- **1단계:** 정확한 이슈 ID와 제목 prefix·부분 문자열 검색. 데이터베이스 검색으로도 충분하다.
- **2단계:** 설명·댓글을 포함한 전체 텍스트 색인, 관련도·상태 가중치, 오탈자 처리.

전용 검색 엔진은 데이터량과 지연 문제가 확인된 뒤 도입한다. 가장 중요한 것은 검색 정확도보다 권한 누락이 없는 것이다.

### 8.2 알림

공식 문서상 사용자는 이슈를 만들거나 담당되거나 멘션되면 자동 구독되고, Inbox는 핵심 이벤트를 모으며 데스크톱·모바일·Slack은 실시간, 이메일은 즉시 또는 digest로 전달될 수 있다.

복제 제품에서는 `도메인 이벤트 → 알림 결정 → Inbox 레코드 → 채널 전달`을 분리해야 한다. 예를 들어 댓글 생성 자체와 “누가 이 댓글 알림을 받아야 하는가”는 다른 책임이다. 초기 알림 규칙은 다음 네 가지면 충분하다.

- 나에게 이슈가 할당됨.
- 설명·댓글에서 내가 멘션됨.
- 내가 구독한 이슈에 다른 사용자가 댓글을 남김.
- 내가 구독한 이슈가 완료·취소됨.

자신의 행동은 알리지 않고, 같은 이벤트가 여러 규칙에 걸려도 한 알림으로 합치며, 권한을 잃으면 미리 생성된 알림에서도 내용을 볼 수 없게 해야 한다.

### 8.3 실시간·오프라인 동기화

Linear는 공식적으로 변경 사항을 실시간 동기화하고, 전송 실패 시 로컬에 저장했다가 연결 복구 후 재시도한다고 설명한다. 공식 기술 글은 별도의 실시간 동기화 엔진이 제품의 핵심 기반임을 보여준다. 다만 내부 충돌 해결·데이터 분할의 최신 세부는 공개 자료만으로 완전히 확인할 수 없다.

복제 제품의 초기 전략은 다음과 같다.

- 서버 데이터베이스를 최종 진실의 원천으로 둔다.
- 사용자의 직접 변경은 낙관적으로 즉시 화면에 반영한다.
- 실패하면 되돌리고 재시도 선택지를 제공한다.
- 워크스페이스 또는 팀 단위 채널로 변경 이벤트를 푸시한다.
- 엔터티 `version` 또는 `updated_at`으로 오래된 저장을 감지한다.
- 설명처럼 충돌 가능성이 큰 긴 텍스트에는 마지막 쓰기 승리를 명시하거나 충돌 UI를 둔다.
- 완전한 오프라인 큐와 CRDT·OT 기반 협업 편집은 초기 범위에서 제외한다.

리스트·보드·상세 화면이 각각 별도 데이터를 들고 있으면 쉽게 불일치한다. 클라이언트에 정규화된 엔터티 캐시를 두고 모든 화면이 같은 이슈 레코드를 참조하도록 하는 편이 중요하다.

### 8.4 외부 통합

Linear의 공식 개발자 문서는 공개 GraphQL API와 웹훅을 제공하고, 폴링 대신 웹훅 사용을 권한다. GitHub·GitLab 연동은 PR·커밋과 이슈를 연결하고 리뷰·병합 이벤트로 상태를 자동화한다.

처음에는 GitHub 하나만 지원하는 것이 좋다. 구현 시 다음을 피할 수 없다.

- 설치·저장소·팀 매핑과 권한 철회.
- 외부 PR과 내부 이슈의 다대다 연결.
- 웹훅 서명 검증, 재전송, 중복, 순서 뒤바뀜.
- 자동화가 만든 상태 변경을 다시 외부로 보내는 순환 방지.
- 외부 객체 삭제·비공개 전환 시 링크 처리.
- 연결 실패와 마지막 동기화 상태를 관리자가 확인할 운영 UI.

API·웹훅을 초기에 공개하면 내부 모델을 오래 유지해야 한다. 첫 고객에게 꼭 필요한 경우가 아니면 내부 API를 안정화한 뒤 공개하는 편이 낫다.

### 9.1 특히 먼저 설계할 것

- **수동 정렬 키:** 정수 순번 대신 항목 사이에 삽입 가능한 정렬 키를 사용하고 재균형 전략을 둔다.
- **이벤트 기록:** 변경 전후 값을 활동 이벤트로 남겨 알림·감사·디버깅에 재사용한다.
- **삭제 정책:** soft delete, 휴지통 보존 기간, 첨부 삭제 시점을 분리한다.
- **백그라운드 작업:** 이메일, 검색 색인, 웹훅, 반복 이슈 생성은 요청 처리와 분리하고 재시도·실패 큐를 둔다.
- **권한 함수의 단일화:** 목록 조회, 상세 조회, 검색, 이벤트, 첨부가 공통 권한 규칙을 사용하게 한다.
- **시간의 의미:** 워크스페이스·팀 시간대와 UTC 저장 원칙을 초기에 정한다.

## 10. 그대로 복제할 것과 변경할 것

### 10.1 그대로 보존할 제품 원칙

- 이슈는 반드시 한 팀에 속한다.
- 담당자는 한 명이다.
- 우선순위 선택지는 적고 고정한다.
- 팀별 상태 이름과 순서는 사용자화하되 시스템 범주는 고정한다.
- 목록과 보드는 같은 데이터의 다른 표현이다.
- 전역 생성, 인라인 편집, 핵심 단축키를 제공한다.
- 개인 작업, 검색, Inbox를 항상 가까운 1차 탐색으로 둔다.
- 프로젝트 완료와 이슈 완료율을 분리한다.
- 활동 기록을 사용자에게 보여 준다.

### 10.2 의도적으로 단순화할 것

- Linear가 설명하는 클라이언트 캐시 기반 실시간 동기화와 제한적 오프라인 재시도 전체를 복제하지 않고, 서버 중심 낙관적 UI로 시작한다.
- Rich text 범위를 Markdown 기본 서식, 링크, 코드, 체크리스트 정도로 제한한다.
- 초기 보드는 상태 기준 한 종류만 지원한다.
- 필터는 AND 중심의 구조화 조건만 지원하고, 고급 중첩·자연어 해석은 미룬다.
- 팀 계층은 없애고 워크스페이스 아래 1단계 팀만 둔다.
- 프로젝트는 이슈 묶음과 상태·리드·목표일까지만 제공한다.
- 통합은 GitHub 하나부터 시작한다.
- 알림은 인앱부터 시작하고 이메일 digest·Slack·푸시는 나중에 붙인다.
- 인증은 이메일 링크 또는 한 가지 소셜 로그인으로 시작하고 SAML·SCIM은 제외한다.
- API는 내부용으로 유지하고 공개 계약은 제품 모델이 안정된 뒤 만든다.

### 10.3 복제하지 말아야 할 것

브랜드명, 로고, 아이콘 세트, 고유한 카피, 화면의 직접 복제는 상용화 과정에서 검토해야 할 위험 항목이다. 기능적 개념을 참고하더라도 독자적인 이름과 시각 언어를 설계하는 편이 안전하다. 이는 법률 판단이 아니라 제품 위험 목록이며, 실제 적용은 관할과 구체적 구현에 따라 달라지므로 필요하면 전문가 검토를 받아야 한다.

## 11. 1인·소규모 팀의 단계별 범위 — 검증되지 않은 계획용 시나리오

아래 숫자는 공개 벤치마크나 실제 이 프로젝트의 작업 측정에서 나온 일정이 아니라 **범위 비교를 위한 검증되지 않은 계획용 추정치**다. 경험 있는 풀스택 개발자 1명, 웹 전용, 관리형 데이터베이스·인증·파일 저장소 사용, 기존 디자인 시스템 활용을 가정한다. 제품 디자인 탐색, 정식 접근성 검증, 다국어, 외부 보안 심사, 부하 시험, 고도화된 운영 자동화, 네이티브 앱은 기간에서 제외했다. 이 항목이나 데이터 이관·동기화 품질 목표가 추가되면 숫자를 일정 약속으로 사용할 수 없으며, 각 단계의 통과 기준을 기간보다 우선해야 한다.

### 단계 0: 핵심 체감 프로토타입 — 계획용 추정 약 2~4주

- 단일 워크스페이스와 단일 팀.
- 이슈 생성·상세·수정.
- 고정 상태, 담당자, 우선순위.
- 리스트와 상태 보드.
- 빠른 생성 단축키와 인라인 속성 변경.
- 더미 또는 단순 사용자 계정.

**통과 기준:** 한 사람이 별도 문서 없이 실제 일주일의 작업을 기록하고 매일 다시 열 수 있다.

### 단계 1: 팀이 실제로 쓰는 MVP — 계획용 추정 추가 약 6~10주

- 멤버 초대, 여러 팀, 팀별 워크플로.
- 댓글·멘션·활동 기록.
- 라벨, 프로젝트, 기본 필터·정렬.
- `My issues`, ID·제목 검색.
- 인앱 Inbox와 읽음 처리.
- 낙관적 UI, 기본 실시간 갱신.
- 권한은 `admin/member`, 공개 팀 중심.
- CSV 내보내기, 백업·오류 모니터링.

**통과 기준:** 3~10명이 기존 이슈 트래커 대신 4주 이상 사용하며, 데이터 불일치와 알림 누락 때문에 원래 도구로 돌아가지 않는다.

### 단계 2: 소프트웨어 팀에 맞는 제품 — 계획용 추정 추가 약 6~10주

- 저장된 보기, 하위 이슈, 관계, 템플릿.
- 사이클 또는 트리아지 중 고객 요구가 강한 것부터.
- GitHub 연결과 PR 상태 자동화.
- 기본 이메일 알림.
- private 팀, 감사 가능한 관리자 작업.
- CSV 가져오기와 이전 도구 마이그레이션 보조.

**통과 기준:** 신규 팀이 개발자의 직접 도움 없이 가입·이관·기본 설정을 끝내고, PR과 이슈 상태를 이중으로 관리하지 않는다.

### 단계 3: 선택적 확장 — 기간 산정 제외, 수개월 이상 가능

- Slack·Sentry·지원 도구 연동.
- 프로젝트 마일스톤·업데이트·타임라인.
- 공개 API·OAuth·웹훅.
- 게스트·세밀한 공유.
- 고객 요청 또는 기본 분석 중 실제 구매 사유가 되는 영역.

### 단계 4: Enterprise 또는 AI 제품 — 별도 제품선으로 판단

- SAML·SCIM, 감사 로그, 데이터 보존·리전, IP 제한.
- Insights·대시보드·웨어하우스 동기화.
- Releases와 다중 코드·배포 통합.
- AI Agent, 코드 인텔리전스, 코딩 세션, 코드 리뷰.

이 단계는 “다음 기능”이 아니라 보안 운영, 권한 모델, 비용 통제, 실행 격리, 외부 시스템 신뢰성을 포함하는 별도 사업 범위다.

## 12. 추천 MVP 화면 구조

초기 화면은 다음 7개면 충분하다.

1. **My Issues:** 내 미완료 이슈, 우선순위·상태 그룹.
2. **Inbox:** 할당·멘션·댓글 알림.
3. **Team Issues:** 전체·Active·Backlog 탭, 리스트·보드 전환.
4. **Projects:** 프로젝트 목록과 프로젝트별 이슈.
5. **Issue Detail:** 제목·설명·속성·댓글·활동.
6. **Search:** ID·제목 중심 전역 검색.
7. **Settings:** 멤버, 팀, 상태, 라벨.

전역 `Create issue`와 빠른 검색은 어느 화면에서도 접근 가능해야 한다. 별도의 홈 대시보드는 만들지 않아도 된다. 사용자가 매일 돌아오는 홈은 My Issues 또는 최근 사용한 팀 보기면 충분하다.

## 13. 우선 검증해야 할 제품 가설

개발 전에 다음 가설을 확인하면 기능 과잉을 줄일 수 있다.

- 고객은 Linear의 전체 기능이 아니라 속도와 간결함 때문에 대안을 원하는가.
- 대상은 1~10명 팀인가, 10~50명 팀인가. 후자부터 권한·이관·통합 요구가 급격히 커진다.
- 프로젝트와 사이클 중 실제 계획 단위는 무엇인가.
- GitHub 상태 자동화가 구매 이유인가, 단순 편의 기능인가.
- 기존 도구에서 반드시 이관해야 하는 데이터는 이슈·댓글·첨부 중 어디까지인가.
- 웹만으로 충분한가, 모바일 알림과 빠른 캡처가 필수인가.
- “Linear 복제”의 차별점이 더 저렴함인지, 특정 산업 워크플로인지, 자체 호스팅인지, AI인지 명확한가.

기능을 복제하는 것만으로는 전환 이유가 약하다. 가장 현실적인 차별화는 **특정 고객군에 맞춘 더 좁은 제품**, **자체 호스팅·데이터 통제**, **한국어·국내 도구 연동**, **개발 이슈와 특정 업무 도메인의 연결** 중 하나를 택하는 것이다. 이 부분은 사용자군이 정해지지 않아 현재로서는 추론이며 추가 검증이 필요하다.

## 14. 불확실성과 한계

- 공개 문서는 제품 동작을 확인하는 데 적합하지만 기능별 실제 사용률은 제공하지 않는다. 따라서 “많이 쓰이는 기능”이 아니라 핵심 루프 기여도와 구현 의존성으로 우선순위를 정했다.
- Linear의 최신 내부 데이터 모델, 동기화 프로토콜, 충돌 해결 방식은 공개 자료만으로 완전히 확인할 수 없다.
- 개발 기간은 기술 스택, 디자인 완성도, 테스트·운영 수준, 가져오기 범위에 따라 크게 달라질 수 있다.
- 현재 Linear 제품 표면은 AI·코드 리뷰·릴리스 방향으로 빠르게 확장 중이다. 기능명·요금제 포함 동적 정보는 2026-07-10 기준이며 이후 달라질 수 있다.
- 경쟁 분석, 사용자 인터뷰, 실제 Linear 워크스페이스 사용 관찰은 이번 범위에 포함하지 않았다. 다음 조사에서는 대상 고객 5~10명과 함께 핵심 작업 10개를 실제로 수행해 조작 흐름과 전환 이유를 검증하는 것이 좋다.
