# Beta 임시 계획서

> 상태: 협의 중 초안 — B3까지만 정리

## 1. 문서 목적

이 문서는 Closed Beta의 제품 방향과 개발 범위를 대화로 확정하기 위한 임시 작업 문서다. 현재는 B3까지만 기록하며, 정식 Beta 제품 범위 및 실행 계획의 정본으로 사용하지 않는다.

## 2. Beta 기본 방향

- 대상은 Alpha와 동일하게 1~10명 규모의 초대된 소프트웨어 개발 팀으로 유지한다.
- private 팀과 게스트는 추가하지 않고, 현재 공개 팀 모델 안에서 팀 단위 관리 권한을 세분화한다.
- 이슈는 업무 내용의 원본으로 유지하고, 이슈 안의 팀 작업을 실제 실행 단위로 사용한다.
- Alpha 이후 별도로 진행하는 사용성·디자인·회귀 개선은 Beta 기능 범위와 구분해 전환 안정화 작업으로 다룬다.

## 3. B1 이슈 목록 구성

### 3.1 목표

소규모 팀이 이슈 목록을 자신의 업무 기준에 맞게 구성하고, 자주 사용하는 정렬 구성을 반복해서 사용할 수 있게 한다.

### 3.2 다중 정렬

- 현재 한 가지 조건만 적용하는 정렬을 여러 조건의 우선순위를 지정할 수 있는 다중 정렬로 확장한다.
- 예를 들어 `우선순위 높은 순 → 상태 순 → 최근 수정 순`처럼 앞선 조건의 값이 같을 때 다음 조건을 적용한다.
- 개인 저장된 보기에 다중 정렬 조건과 적용 순서를 함께 저장하고 다시 열었을 때 동일하게 복원한다.
- 전역 이슈 목록에서 `우선순위`, `상태`, `최근 수정일`, `생성일`, `진행률` 중 고유한 조건을 최대 세 개까지 순서대로 적용한다.
- 기본 정렬은 `최근 수정일 내림차순`이고 마지막 고정 기준은 불변 `id`이며, 방향은 마지막 명시 조건을 따른다.
- 진행률은 취소된 팀 작업을 제외한 화면 표시용 반올림 백분율을 사용한다.
- 기존 단일 정렬 URL과 저장된 보기는 한 조건의 다중 정렬로 전환해 호환한다.

### 3.3 제외 및 보류

- 작업 주기, 속도 추정, 번다운 차트와 자동 계획은 Beta 범위에 포함하지 않는다.
- 하위 이슈와 일반 이슈 관계는 추가하지 않는다. 이슈 안의 팀 작업이 이미 실행 작업을 나누는 역할을 담당한다.
- 별도 접수함에서 담당 팀·담당자·우선순위·상태를 분류하는 고급 트리아지는 추가하지 않는다. 현재 전역 이슈 목록, 배정, 상태 변경과 저장된 보기로 우선 대응한다.
- 수동 드래그 정렬보다 다중 정렬을 우선한다. 수동 정렬은 실제 사용에서 반복 필요가 확인되면 다시 검토한다.

## 4. B2 팀 단위 관리 권한

### 4.1 목표

워크스페이스 관리자에게 집중된 팀 관리 업무를 각 팀의 책임자에게 나누되, 해당 팀을 벗어나는 워크스페이스 관리 권한까지 부여하지 않는다.

### 4.2 팀장 역할

- 각 팀에 한 명 이상의 팀장을 지정할 수 있다.
- 팀장 지정과 해제는 워크스페이스 관리자만 할 수 있다.
- 팀장은 워크스페이스 전체 관리자가 아니라 지정된 팀 범위에서만 관리 권한을 가진다.
- 팀장은 자신이 맡은 팀의 이름과 설명을 수정할 수 있다.
- 팀장은 기존 워크스페이스 멤버를 자신이 맡은 팀에 추가하거나 팀에서 제외할 수 있다.
- 팀장은 이메일로 새 팀원을 초대할 수 있다. 초대를 수락한 사용자는 워크스페이스 멤버가 되면서 초대한 팀에 합류한다.
- 팀장은 자신이 맡은 팀에서 사용하는 상태값을 생성·수정하고 순서를 변경하거나 사용을 중지할 수 있다.
- 사용 중인 상태값은 바로 삭제할 수 없다. 해당 상태값을 사용하는 이슈를 다른 상태로 이전한 뒤 정리한다.
- 팀장은 다른 팀의 정보·팀원·상태값과 워크스페이스 전체 설정을 변경할 수 없다.
- 팀장은 워크스페이스 멤버를 워크스페이스에서 제외하거나 전체 역할을 변경할 수 없다. 이 권한은 기존 워크스페이스 관리자에게 유지한다.

### 4.3 권한 범위 원칙

- B2는 모든 권한을 임의로 조합하는 사용자 정의 역할이나 권한 편집기를 만드는 범위가 아니다.
- 기존 `admin/member` 권한 위에 팀 범위의 `팀장` 역할을 추가하는 방식으로 시작한다.

### 4.4 제외

- private 팀과 팀별 비공개 데이터는 추가하지 않는다.
- 게스트 역할과 외부 사용자의 제한 참여는 추가하지 않는다.
- 이슈 단위 공유나 이슈별 사용자 정의 권한은 추가하지 않는다.

## 5. B3 GitHub 팀 작업 연동

### 5.1 목표

GitHub Pull Request와 Rivet 팀 작업을 연결하고 PR의 작성·리뷰·검사·병합 흐름을 팀 작업 상태에 반영한다. 상위 이슈 상태는 직접 변경하지 않고 기존 계약대로 연결된 팀 작업 상태와 진행률에서 계산한다.

### 5.2 GitHub App과 저장소 연결

- 외부 Beta 팀의 GitHub 조직에도 설치할 수 있도록 Rivet GitHub App을 `Any account` 설치가 가능한 공개 앱으로 등록한다. GitHub Marketplace에는 등록하지 않는다.
- 워크스페이스 관리자가 GitHub App을 설치·연결 해제하고 접근할 저장소를 선택한다.
- GitHub App은 `Metadata`, `Pull requests`, `Checks`, `Commit statuses` 읽기 권한만 요청한다.
- 소스 코드, GitHub Issues, 댓글, Actions와 Pull Request 쓰기 권한은 요청하지 않는다.
- 하나의 GitHub 저장소를 여러 Rivet 팀에 연결할 수 있고, 하나의 Rivet 팀도 여러 저장소를 사용할 수 있다.
- 팀장은 자신이 맡은 팀에 허용된 저장소와 PR 상태별 팀 작업 상태 매핑을 설정할 수 있다. 워크스페이스 관리자도 모든 팀의 설정을 관리할 수 있다.

### 5.3 PR 연결

- PR은 상위 이슈가 아니라 `API-123`, `WEB-123` 같은 팀 작업 표시 ID에 연결한다.
- 팀 작업 화면에서 표시 ID와 제목을 포함한 Git 브랜치 이름을 복사할 수 있다.
- 브랜치 이름이나 PR 제목에 팀 작업 표시 ID가 있으면 해당 PR을 자동 연결한다.
- 이미 열린 PR은 팀 작업 화면에 GitHub PR URL을 입력해 수동으로 연결할 수 있다.
- 한 팀 작업에 여러 PR을 연결할 수 있다. 연결에서 제거하지 않은 모든 PR이 병합돼야 해당 팀 작업의 병합 후 처리를 실행한다.
- 병합되지 않고 닫힌 PR이 하나라도 있으면 자동 완료하지 않고 연결 확인이 필요한 상태로 표시한다. 사용자가 PR을 다시 열거나 연결에서 제거해야 자동화를 계속한다.
- PR 댓글, commit 메시지와 magic word를 이용한 연결은 B3에 포함하지 않는다.

### 5.4 PR 표시 정보

연결된 팀 작업과 통합 이슈 상세에서 다음 정보를 표시한다.

- 저장소 이름과 PR 번호
- PR 제목, 작성자와 GitHub 이동 링크
- Draft, 열림, 리뷰 대기, 변경 요청, 승인, 검사 실패, 병합 대기, 병합과 미병합 닫힘 상태
- 마지막 GitHub 동기화 시각과 동기화 실패 여부

소스 코드, diff, PR 본문과 댓글 원문은 Rivet에 저장하거나 표시하지 않는다.

### 5.5 상태 자동화

팀장은 각 GitHub 단계에 같은 팀의 기존 워크플로 상태를 연결한다. Rivet이 상태값을 자동 생성하지 않으며 필요한 경우 B2 권한으로 `수정 필요`, `병합 대기` 같은 상태값을 먼저 만든다.

| GitHub 단계 | 연결할 팀 작업 상태 | 시스템 범주 |
| --- | --- | --- |
| Draft 또는 일반 PR 생성 | `진행 중` | `STARTED` |
| 리뷰 준비 또는 리뷰 요청 | `검토` 또는 `리뷰 대기` | `STARTED` |
| 변경 요청 | `수정 필요` | `STARTED` |
| 필요한 리뷰 승인과 필수 검사 통과, GitHub 병합 가능 | `병합 대기` | `STARTED` |
| 모든 연결 PR 병합 | 병합 후 처리 실행 | 완료 방식에 따라 결정 |

- `병합 대기`는 GitHub의 최신 PR 상태를 다시 조회해 필요한 리뷰, 필수 검사와 병합 가능 여부가 모두 충족됐을 때만 적용한다.
- 여러 PR이 연결된 경우 아직 통과하지 못한 PR을 기준으로 팀 작업 단계를 계산하고, 모든 PR이 병합됐을 때만 병합 후 처리를 실행한다.
- 사용자가 팀 작업을 수동으로 일시 중지·취소·완료하거나 GitHub 매핑 밖의 상태로 옮기면 GitHub 상태 자동화를 중지한다. 사용자가 명시적으로 `GitHub 자동화 재개`를 선택하기 전에는 후속 Webhook이 상태를 덮어쓰지 않는다.
- GitHub에 의한 상태 변경은 PR 번호와 저장소, 변경 전후 상태와 시각을 이슈 활동에 남긴다. PR 제목·본문과 댓글은 활동 payload에 복제하지 않는다.

### 5.6 병합 후 처리 예약

- Web·App 팀 작업은 연결된 모든 PR이 병합되면 `이 작업만 완료` 방식으로 자동 완료한다.
- 프로젝트에 활성 프론트 역할이 없는 백엔드 팀 작업도 연결된 모든 PR이 병합되면 `이 작업만 완료` 방식으로 자동 완료한다.
- 프로젝트에 활성 프론트 역할이 있는 백엔드 팀 작업은 PR이 리뷰 대기 또는 병합 대기에 들어갔을 때 담당자가 `병합 후 처리`를 예약할 수 있다.
- 예약에서 `이 작업만 완료` 또는 `프론트에 전달 후 완료`를 선택한다.
- `프론트에 전달 후 완료`는 대상 Web·App 역할과 전달 내용을 병합 전에 입력하며, PR이 병합되기 전까지 수정하거나 취소할 수 있다.
- 연결된 모든 PR이 병합되면 팀 작업 version, 현재 프로젝트 역할, 대상 팀과 PR 연결 상태를 다시 검증한다. 검증을 통과하면 기존 완료 계약을 사용해 팀 작업 상태, 작업 전달, 대상 프론트 작업, 이슈 진행률, 활동과 Outbox를 한 트랜잭션으로 반영한다.
- 병합 후 처리 예약 없이 백엔드 PR이 병합되면 팀 작업을 자동 완료하지 않는다. 기존 `검토` 상태를 유지하고 `병합됨 · 완료 방식 선택 필요` 안내를 표시한다.
- 예약 내용이 현재 팀 작업이나 프로젝트 역할과 충돌하면 자동 완료하지 않고 입력을 보존한 채 다시 확인하도록 안내한다.
- 사용자가 PR 병합 전에 팀 작업을 직접 완료하면 예약을 종료하며 이후 GitHub 이벤트가 완료 상태를 덮어쓰지 않는다.

### 5.7 Webhook과 동기화

- GitHub App은 `pull_request`, `pull_request_review`, `check_suite`, `status`, `installation`과 `installation_repositories` Webhook을 사용한다.
- 수신 API는 원본 요청 본문과 Webhook secret으로 `X-Hub-Signature-256`을 검증한다.
- `X-GitHub-Delivery`를 멱등 키로 사용해 같은 전달과 재전달을 한 번만 처리한다.
- 수신 API는 유효한 이벤트를 Webhook Inbox에 접수하고 빠르게 `2xx`를 반환한다. 실제 상태 계산과 변경은 worker가 비동기로 처리한다.
- Inbox에는 전달 ID, 이벤트와 action, installation·repository·PR 식별자, 외부 시각과 처리 결과만 저장한다. PR 본문, 댓글, diff와 소스 코드는 저장하지 않는다.
- Webhook은 처리 계기만 제공한다. worker는 설치 access token으로 GitHub API에서 연결된 PR의 최신 상태를 다시 조회한 뒤 팀 작업 상태를 계산한다.
- Webhook 누락·순서 역전과 장애 복구를 위해 사용자가 실행하는 `다시 동기화`와 열린 PR의 주기적 상태 대조를 함께 제공한다.
- App 제거 또는 저장소 접근 해제 이벤트를 받으면 새 동기화를 중지하고 기존 연결에는 접근 해제 상태를 표시한다. Rivet 팀 작업과 활동 이력은 삭제하지 않는다.

### 5.8 제외

- GitHub Issues 생성·가져오기와 양방향 동기화
- PR 댓글, 리뷰 댓글과 Rivet 댓글의 동기화
- Rivet 안에서 diff 조회, 리뷰 승인, 변경 요청과 PR 병합
- commit 단위 연결과 상태 자동화
- GitHub 사용자를 Rivet 담당자로 자동 배정
- 배포·릴리스 상태와 대상 브랜치별 자동화
- GitHub 외 Git 공급자 연동

## 6. 다음 협의 범위

B4 이후 범위는 후속 대화에서 순서대로 확정한다. 협의 중 애매한 부분은 후속 결정으로 남기지 않고 해당 범위를 기록하기 전에 바로 확인한다. 합의가 끝나기 전에는 이 문서를 정식 Beta 계획이나 개발 착수 기준으로 사용하지 않는다.
