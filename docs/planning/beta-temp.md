# Beta 임시 계획서

> 상태: 협의 중 초안 — B4와 Beta 운영 기반 과제까지 정리

## 1. 문서 목적

이 문서는 Closed Beta의 제품 방향과 개발 범위를 대화로 확정하기 위한 임시 작업 문서다. 현재는 B4와 Beta 전환에 필요한 문서 정리 원칙까지 기록하며, 정식 Beta 제품 범위 및 실행 계획의 정본으로 사용하지 않는다.

## 2. Beta 기본 방향

- 대상은 Alpha와 동일하게 1~10명 규모의 초대된 소규모 조직으로 유지하되, 프로젝트 참여 팀을 소프트웨어 개발 역할로 제한하지 않는다.
- private 팀과 게스트는 추가하지 않고, 현재 공개 팀 모델 안에서 팀 단위 관리 권한을 세분화한다.
- 이슈는 업무 내용의 원본으로 유지하고, 이슈 안의 팀 작업을 실제 실행 단위로 사용한다.
- 프로젝트는 실제 참여 팀을 동적으로 구성하고, 팀 작업의 시작·전달·완료는 고정된 역할 종류가 아니라 참여 팀을 기준으로 처리한다.
- Alpha 이후 별도로 진행하는 사용성·디자인·회귀 개선은 Beta 기능 범위와 구분해 전환 안정화 작업으로 다룬다.

## 3. B1 이슈 목록 구성

### 3.1 목표

소규모 팀이 이슈 목록을 자신의 업무 기준에 맞게 구성하고, 자주 사용하는 정렬 구성을 반복해서 사용할 수 있게 한다.

### 3.2 다중 정렬

- 현재 한 가지 조건만 적용하는 정렬을 여러 조건의 우선순위를 지정할 수 있는 다중 정렬로 확장한다.
- 예를 들어 `우선순위 높은 순 → 상태 순 → 최근 수정 순`처럼 앞선 조건의 값이 같을 때 다음 조건을 적용한다.
- 개인 저장된 보기에 다중 정렬 조건과 적용 순서를 함께 저장하고 다시 열었을 때 동일하게 복원한다.
- 전역 이슈 목록에서 `우선순위`, `상태`, `최근 수정일`, `생성일`, `진행률` 중 고유한 조건을 최대 세 개까지 순서대로 적용한다.
- 기본 정렬은 `최근 수정일 내림차순`이고 마지막 고정 기준은 불변 `id`이며, 방향은 마지막 명시 조건을 따른다.
- 진행률은 취소된 팀 작업을 제외한 화면 표시용 반올림 백분율을 사용한다.
- 기존 단일 정렬 URL과 저장된 보기는 한 조건의 다중 정렬로 전환해 호환한다.

### 3.3 제외 및 보류

- 작업 주기, 속도 추정, 번다운 차트와 자동 계획은 Beta 범위에 포함하지 않는다.
- 하위 이슈와 일반 이슈 관계는 추가하지 않는다. 이슈 안의 팀 작업이 이미 실행 작업을 나누는 역할을 담당한다.
- 별도 접수함에서 담당 팀·담당자·우선순위·상태를 분류하는 고급 트리아지는 추가하지 않는다. 현재 전역 이슈 목록, 배정, 상태 변경과 저장된 보기로 우선 대응한다.
- 수동 드래그 정렬보다 다중 정렬을 우선한다. 수동 정렬은 실제 사용에서 반복 필요가 확인되면 다시 검토한다.

## 4. B2 팀 단위 관리 권한

### 4.1 목표

워크스페이스 관리자에게 집중된 팀 관리 업무를 각 팀의 책임자에게 나누되, 해당 팀을 벗어나는 워크스페이스 관리 권한까지 부여하지 않는다.

### 4.2 팀장 역할

- 각 팀에 한 명 이상의 팀장을 지정할 수 있다.
- 팀장 지정과 해제는 워크스페이스 관리자만 할 수 있다.
- 팀장은 워크스페이스 전체 관리자가 아니라 지정된 팀 범위에서만 관리 권한을 가진다.
- 팀장은 자신이 맡은 팀의 이름과 설명을 수정할 수 있다.
- 팀장은 기존 워크스페이스 멤버를 자신이 맡은 팀에 추가하거나 팀에서 제외할 수 있다.
- 팀장은 이메일로 새 팀원을 초대할 수 있다. 초대를 수락한 사용자는 워크스페이스 멤버가 되면서 초대한 팀에 합류한다.
- 팀장은 자신이 맡은 팀에서 사용하는 상태값을 생성·수정하고 순서를 변경하거나 사용을 중지할 수 있다.
- 사용 중인 상태값은 바로 삭제할 수 없다. 해당 상태값을 사용하는 이슈를 다른 상태로 이전한 뒤 정리한다.
- 팀장은 다른 팀의 정보·팀원·상태값과 워크스페이스 전체 설정을 변경할 수 없다.
- 팀장은 워크스페이스 멤버를 워크스페이스에서 제외하거나 전체 역할을 변경할 수 없다. 이 권한은 기존 워크스페이스 관리자에게 유지한다.

### 4.3 권한 범위 원칙

- B2는 모든 권한을 임의로 조합하는 사용자 정의 역할이나 권한 편집기를 만드는 범위가 아니다.
- 기존 `admin/member` 권한 위에 팀 범위의 `팀장` 역할을 추가하는 방식으로 시작한다.

### 4.4 제외

- private 팀과 팀별 비공개 데이터는 추가하지 않는다.
- 게스트 역할과 외부 사용자의 제한 참여는 추가하지 않는다.
- 이슈 단위 공유나 이슈별 사용자 정의 권한은 추가하지 않는다.

## 5. B3 동적 프로젝트 참여 팀

> 구현 상태: **완료 (2026-07-19)**. `ProjectTeam` expand/backfill, 동적 TeamWork·템플릿·CSV·전달·알림·삭제 worker, OpenAPI·생성 클라이언트와 Web 흐름을 전환했다. 구 역할 표·enum·호환 컬럼 제거는 구버전 프로세스 종료와 데이터 검증 뒤 별도 contract migration에서 수행한다.

### 5.1 목표

프로젝트에 참여하는 팀을 `BACKEND`, `WEB_FRONTEND`, `APP_FRONTEND` 같은 고정 역할로 제한하지 않는다. 기획, 디자인, 사업, 운영과 개발 등 워크스페이스의 어떤 팀도 프로젝트에 참여하고 같은 TeamWork 계약으로 협업할 수 있게 한다.

새 역할 enum을 계속 추가하거나 자유 입력 역할명을 만드는 대신, 실제 팀 자체를 프로젝트의 책임 단위로 사용한다. 팀 이름과 키가 담당 영역을 설명하며 Beta에서는 같은 팀을 한 프로젝트에 한 번만 연결한다.

### 5.2 프로젝트 참여 팀 모델

- 신규 제품 계약은 프로젝트와 실제 팀을 연결하는 `ProjectTeam` 참여 관계를 사용한다. 기존 `ProjectRoleTeam`은 expand 배포 호환 범위에서만 유지한다.
- 하나의 프로젝트와 팀 조합에는 하나의 활성 참여 관계만 존재한다. 같은 팀이 여러 종류의 일을 맡으면 하나의 참여 관계 아래 여러 TeamWork를 사용한다.
- 프로젝트는 참여 팀 없이 먼저 만들 수 있다. 실제 팀 작업을 시작하려면 한 개 이상의 활성 참여 팀이 있어야 한다.
- 프로젝트 참여 관계가 팀의 이름, 설명, 멤버와 워크플로를 복제하지 않는다. 해당 정보의 원본은 기존 `Team`과 B2의 팀 관리 기능으로 유지한다.
- 참여 관계는 삭제보다 비활성화를 사용해 완료된 TeamWork, 작업 전달, 활동과 알림의 과거 연결을 보존한다.
- 팀 종류나 역할 분류가 준비 상태, 권한, 전달 대상과 완료 방식을 암묵적으로 결정하지 않는다.

### 5.3 참여 팀 관리와 권한

- 워크스페이스 관리자와 프로젝트 리드는 프로젝트 설정에서 활성 워크스페이스 팀을 추가하거나 제외할 수 있다.
- B2 팀장은 자신이 맡은 팀의 멤버와 워크플로를 관리하지만, 프로젝트 참여 구성을 임의로 변경하지는 않는다.
- 진행 중이거나 일시 중지된 TeamWork가 있는 참여 팀은 프로젝트에서 바로 제외할 수 없다. 해당 작업을 완료하거나 취소한 뒤 참여 관계를 비활성화한다.
- 완료·취소된 TeamWork만 남아 있는 팀은 참여를 비활성화할 수 있고, 이후 같은 팀을 다시 추가하면 기존 이력을 유지한 채 참여를 재개한다.
- 비활성 또는 삭제 예약 상태의 팀은 새 프로젝트에 추가하거나 새 TeamWork의 대상으로 선택할 수 없다.

### 5.4 TeamWork 시작과 표시

- TeamWork는 `projectRole` 대신 해당 프로젝트의 참여 관계와 실제 팀을 참조한다. 워크플로 상태와 담당자는 그 팀에 속한 값만 선택할 수 있다.
- 이슈 생성과 `팀 작업 시작` 입력의 `initialRoles`, `roleAssignments`는 참여 팀 식별자를 사용하는 `initialTeams`, `teamAssignments` 계약으로 교체한다.
- 여러 참여 팀의 작업을 한 번에 시작하는 동작, 기존 활성 작업 재사용, 담당자 검증과 트랜잭션 원자성은 현재 계약을 유지한다.
- 이슈 템플릿의 `initialRole`은 프로젝트의 실제 참여 팀을 가리키는 기본 팀으로 교체한다. 프로젝트 변경이나 참여 비활성화로 유효하지 않으면 자동 적용하지 않고 사용자가 다시 선택하도록 안내한다.
- CSV 가져오기, 목록 필터, 요약, 활동 payload, 알림과 검색도 역할명이 아니라 실제 팀 식별자와 표시 정보를 사용한다.
- 화면에는 `Web`, `App`, `API` 같은 시스템 역할명 대신 팀 이름, 팀 키와 TeamWork 표시 ID를 일관되게 표시한다.

### 5.5 일반화된 팀 작업 전달과 완료

- 모든 참여 팀의 TeamWork는 기본적으로 서로 병렬이며 즉시 시작할 수 있다. 특정 팀 종류를 이유로 다른 팀 완료를 기다리는 준비 상태를 자동 부여하지 않는다.
- 완료 방식은 `이 작업만 완료`와 `다른 팀에 전달 후 완료`로 일반화한다.
- 전달 대상은 현재 프로젝트의 활성 참여 팀 중 하나 이상으로 선택한다. 대상 팀에 사용할 수 있는 TeamWork가 있으면 재사용하고, 없으면 해당 팀의 기본 워크플로 상태로 생성한다.
- 명시적으로 생성된 전달만 선후 관계를 만든다. 기획→디자인→개발이나 API→Web 같은 고정 순서를 시스템에 내장하지 않는다.
- 전달 내용, 리소스, 대상 TeamWork 연결과 활동 이력은 현재 작업 전달 계약을 유지한다. 같은 소스 작업 자신을 대상으로 선택하거나 비활성 참여 팀으로 전달하는 요청은 거부한다.
- 상위 이슈 상태와 진행률은 계속 연결된 TeamWork의 시스템 범주에서 계산하며 참여 팀의 종류는 계산에 사용하지 않는다.

### 5.6 기존 데이터와 API 전환

- 기존 프로젝트의 역할별 팀 설정에서 고유한 프로젝트·팀 조합을 추출해 `ProjectTeam` 참여 관계를 생성한다. 한 팀이 여러 기존 역할을 맡은 프로젝트는 역할별 작업·전달 의미를 잃을 수 있으므로 preflight에서 중단하고, 역할마다 서로 다른 팀을 지정하거나 중복 역할 배정을 제거한 뒤 다시 적용한다.
- 기존 TeamWork는 workspace·프로젝트·`teamId`가 모두 일치하는 참여 관계에 연결한다. 전환 기간의 `projectRole`은 구버전 쓰기 호환에만 사용하고 신규 API·Web·생성 클라이언트에는 노출하지 않으며, 모든 프로세스 전환과 데이터 검증 뒤 제거한다.
- 프로젝트 설정, 이슈 생성·시작·완료, 템플릿, CSV 가져오기, 작업 전달, 알림, 목록·상세 조회와 삭제 worker를 같은 계약 변경 범위에서 갱신한다.
- Prisma migration, OpenAPI와 생성 API 클라이언트는 같은 변경에서 갱신하고 기존 역할 기반 요청에는 명시적인 전환 오류를 반환한다. 역할 기반 계약을 장기 이중 운영하지 않는다.
- 마이그레이션 후 프로젝트별 참여 팀 수, TeamWork 연결 수, 담당 팀·워크플로 일치와 전달 대상 연결을 검사해 누락된 과거 작업이 없음을 확인한다.

### 5.7 제외

- 프로젝트별 자유 입력 역할명, 역할 분류와 역할 템플릿
- 참여 승인·요청과 부서 간 결재 흐름
- 프로젝트별로 복제한 팀 멤버와 워크플로
- 팀 작업 사이의 범용 선후 관계 그래프와 자동 일정 계산
- private 프로젝트 참여 팀과 팀별 비공개 이슈

## 6. B4 GitHub 팀 작업 연동

> 착수 상태: **보류 — B2 팀 단위 관리 권한 완료 후 진행**. 2026-07-20 점검에서 B2 팀장 관계와 팀 범위 권한을 구현하지 않은 채 B4 계획으로 넘어간 사실을 확인했다. B4의 팀별 저장소·상태 매핑은 B2 팀장 권한을 전제로 하므로 임시 관리자 전용 기능으로 우회하지 않는다. 상세 착수 게이트와 실행 순서는 [Beta B4 GitHub 팀 작업 연동 구현 계획서](../delivery/008.%20Beta%20B4%20GitHub%20팀%20작업%20연동%20구현%20계획서.md)를 따른다.

### 6.1 목표

GitHub Pull Request와 Rivet 팀 작업을 연결하고 PR의 작성·리뷰·검사·병합 흐름을 팀 작업 상태에 반영한다. GitHub 연동 여부와 자동화 동작은 B3의 프로젝트 참여 팀 종류와 무관하다. 상위 이슈 상태는 직접 변경하지 않고 기존 계약대로 연결된 팀 작업 상태와 진행률에서 계산한다.

### 6.2 GitHub App과 저장소 연결

- 외부 Beta 팀의 GitHub 조직에도 설치할 수 있도록 Rivet GitHub App을 `Any account` 설치가 가능한 공개 앱으로 등록한다. GitHub Marketplace에는 등록하지 않는다.
- 워크스페이스 관리자가 GitHub App을 설치·연결 해제하고 접근할 저장소를 선택한다.
- GitHub App은 `Metadata`, `Pull requests`, `Checks`, `Commit statuses` 읽기 권한만 요청한다.
- 소스 코드, GitHub Issues, 댓글, Actions와 Pull Request 쓰기 권한은 요청하지 않는다.
- 하나의 GitHub 저장소를 여러 Rivet 팀에 연결할 수 있고, 하나의 Rivet 팀도 여러 저장소를 사용할 수 있다.
- 팀장은 자신이 맡은 팀에 허용된 저장소와 PR 상태별 팀 작업 상태 매핑을 설정할 수 있다. 워크스페이스 관리자도 모든 팀의 설정을 관리할 수 있다.

### 6.3 PR 연결

- PR은 상위 이슈가 아니라 `DESIGN-123`, `OPS-123` 같은 실제 팀의 TeamWork 표시 ID에 연결한다.
- 팀 작업 화면에서 표시 ID와 제목을 포함한 Git 브랜치 이름을 복사할 수 있다.
- 브랜치 이름이나 PR 제목에 팀 작업 표시 ID가 있으면 해당 PR을 자동 연결한다.
- 이미 열린 PR은 팀 작업 화면에 GitHub PR URL을 입력해 수동으로 연결할 수 있다.
- 한 팀 작업에 여러 PR을 연결할 수 있다. 연결에서 제거하지 않은 모든 PR이 병합돼야 해당 팀 작업의 병합 후 처리를 실행한다.
- 병합되지 않고 닫힌 PR이 하나라도 있으면 자동 완료하지 않고 연결 확인이 필요한 상태로 표시한다. 사용자가 PR을 다시 열거나 연결에서 제거해야 자동화를 계속한다.
- PR 댓글, commit 메시지와 magic word를 이용한 연결은 B4에 포함하지 않는다.

### 6.4 PR 표시 정보

연결된 팀 작업과 통합 이슈 상세에서 다음 정보를 표시한다.

- 저장소 이름과 PR 번호
- PR 제목, 작성자와 GitHub 이동 링크
- Draft, 열림, 리뷰 대기, 변경 요청, 승인, 검사 실패, 병합 대기, 병합과 미병합 닫힘 상태
- 마지막 GitHub 동기화 시각과 동기화 실패 여부

소스 코드, diff, PR 본문과 댓글 원문은 Rivet에 저장하거나 표시하지 않는다.

### 6.5 상태 자동화

팀장은 각 GitHub 단계에 같은 팀의 기존 워크플로 상태를 연결한다. Rivet이 상태값을 자동 생성하지 않으며 필요한 경우 B2 권한으로 `수정 필요`, `병합 대기` 같은 상태값을 먼저 만든다.

| GitHub 단계 | 연결할 팀 작업 상태 | 시스템 범주 |
| --- | --- | --- |
| Draft 또는 일반 PR 생성 | `진행 중` | `STARTED` |
| 리뷰 준비 또는 리뷰 요청 | `검토` 또는 `리뷰 대기` | `STARTED` |
| 변경 요청 | `수정 필요` | `STARTED` |
| 필요한 리뷰 승인과 필수 검사 통과, GitHub 병합 가능 | `병합 대기` | `STARTED` |
| 모든 연결 PR 병합 | 병합 후 처리 실행 | 완료 방식에 따라 결정 |

- `병합 대기`는 GitHub의 최신 PR 상태를 다시 조회해 필요한 리뷰, 필수 검사와 병합 가능 여부가 모두 충족됐을 때만 적용한다.
- 여러 PR이 연결된 경우 아직 통과하지 못한 PR을 기준으로 팀 작업 단계를 계산하고, 모든 PR이 병합됐을 때만 병합 후 처리를 실행한다.
- 사용자가 팀 작업을 수동으로 일시 중지·취소·완료하거나 GitHub 매핑 밖의 상태로 옮기면 GitHub 상태 자동화를 중지한다. 사용자가 명시적으로 `GitHub 자동화 재개`를 선택하기 전에는 후속 Webhook이 상태를 덮어쓰지 않는다.
- GitHub에 의한 상태 변경은 PR 번호와 저장소, 변경 전후 상태와 시각을 이슈 활동에 남긴다. PR 제목·본문과 댓글은 활동 payload에 복제하지 않는다.

### 6.6 병합 후 처리 예약

- GitHub를 사용하는 모든 참여 팀에 같은 병합 후 처리 계약을 적용하며 팀 이름이나 종류에서 완료 방식을 추론하지 않는다.
- 담당자는 PR이 리뷰 대기 또는 병합 대기에 들어갔을 때 `병합 후 처리`를 예약할 수 있다.
- 예약에서 `이 작업만 완료` 또는 `다른 팀에 전달 후 완료`를 선택한다.
- `다른 팀에 전달 후 완료`는 현재 프로젝트의 활성 참여 팀과 전달 내용을 병합 전에 입력하며, PR이 병합되기 전까지 수정하거나 취소할 수 있다.
- 연결된 모든 PR이 병합되면 팀 작업 version, 프로젝트 참여 상태, 대상 팀과 PR 연결 상태를 다시 검증한다. 검증을 통과하면 B3의 일반화된 완료 계약을 사용해 팀 작업 상태, 작업 전달, 대상 팀 작업, 이슈 진행률, 활동과 Outbox를 한 트랜잭션으로 반영한다.
- 병합 후 처리 예약이 없으면 팀 작업을 자동 완료하지 않는다. 기존 상태를 유지하고 `병합됨 · 완료 방식 선택 필요` 안내를 표시한다.
- 예약 내용이 현재 팀 작업이나 프로젝트 참여 구성과 충돌하면 자동 완료하지 않고 입력을 보존한 채 다시 확인하도록 안내한다.
- 사용자가 PR 병합 전에 팀 작업을 직접 완료하면 예약을 종료하며 이후 GitHub 이벤트가 완료 상태를 덮어쓰지 않는다.

### 6.7 Webhook과 동기화

- GitHub App은 `pull_request`, `pull_request_review`, `check_suite`, `status`, `installation`과 `installation_repositories` Webhook을 사용한다.
- 수신 API는 원본 요청 본문과 Webhook secret으로 `X-Hub-Signature-256`을 검증한다.
- `X-GitHub-Delivery`를 멱등 키로 사용해 같은 전달과 재전달을 한 번만 처리한다.
- 수신 API는 유효한 이벤트를 Webhook Inbox에 접수하고 빠르게 `2xx`를 반환한다. 실제 상태 계산과 변경은 worker가 비동기로 처리한다.
- Inbox에는 전달 ID, 이벤트와 action, installation·repository·PR 식별자, 외부 시각과 처리 결과만 저장한다. PR 본문, 댓글, diff와 소스 코드는 저장하지 않는다.
- Webhook은 처리 계기만 제공한다. worker는 설치 access token으로 GitHub API에서 연결된 PR의 최신 상태를 다시 조회한 뒤 팀 작업 상태를 계산한다.
- Webhook 누락·순서 역전과 장애 복구를 위해 사용자가 실행하는 `다시 동기화`와 열린 PR의 주기적 상태 대조를 함께 제공한다.
- App 제거 또는 저장소 접근 해제 이벤트를 받으면 새 동기화를 중지하고 기존 연결에는 접근 해제 상태를 표시한다. Rivet 팀 작업과 활동 이력은 삭제하지 않는다.

### 6.8 제외

- GitHub Issues 생성·가져오기와 양방향 동기화
- PR 댓글, 리뷰 댓글과 Rivet 댓글의 동기화
- Rivet 안에서 diff 조회, 리뷰 승인, 변경 요청과 PR 병합
- commit 단위 연결과 상태 자동화
- GitHub 사용자를 Rivet 담당자로 자동 배정
- 배포·릴리스 상태와 대상 브랜치별 자동화
- GitHub 외 Git 공급자 연동

## 7. Beta 운영 기반 과제 — 문서 체계 정리

### 7.1 목표

`docs/`를 완료된 계획과 작업 기록의 보관소가 아니라 현재 Rivet을 운영·개발·유지보수할 때 바로 사용하는 정본으로 재구성한다. 과거 계획과 구현 과정은 Git 이력과 Pull Request에서 확인하고, 현재 계약과 중복되거나 효력을 잃은 문서는 기본 문서 트리에서 제거한다.

### 7.2 유지할 문서 범위

- `docs/index.md`: 현재 문서의 진입점, 읽는 순서와 정본 위치
- 제품 문서: 현재 Beta 범위, 핵심 용어, 사용자 흐름과 제품 정책
- 아키텍처 문서: 코드·스키마·OpenAPI만으로 설명되지 않는 서비스 경계, 데이터 일관성, 보안과 삭제 정책
- 개발 문서: 실제로 적용하는 코딩, API, 데이터베이스, 테스트, UI와 문서 작성 기준
- 운영 문서: 환경 구성, 배포·업데이트, 백업·복원, 장애 대응, 보안 운영과 외부 연동 절차
- 장기간 유지할 의사결정 문서: 현재 구조를 이해하는 데 필요한 결정과 변경 조건. 단순 실행 계획이나 완료 보고서는 포함하지 않는다.

### 7.3 정리 원칙과 절차

1. `docs/index.md`의 모든 문서를 `유지`, `통합 후 삭제`, `즉시 삭제`로 분류하고 실제 코드·스키마·운영 절차와 내용이 일치하는지 확인한다.
2. 완료된 MVP·Alpha 계획과 실행 보고서, 일회성 벤치마크·참고자료, 오래된 와이어프레임, 개인 실행 프롬프트와 중복 명세에서 아직 유효한 규칙만 현재 정본으로 옮긴다.
3. 옮긴 뒤 원본은 별도 `docs/archive`를 만들지 않고 삭제한다. 변경 이력과 과거 문맥은 Git과 관련 Pull Request로 보존하며, `docs/index.md`의 기존 완료 문서 미삭제 원칙도 이 기준으로 교체한다.
4. API 필드, Prisma 모델과 화면별 세부 목록처럼 코드에서 생성하거나 검증할 수 있는 내용은 문서에 중복하지 않고 정본 위치를 링크한다.
5. 남은 문서에는 상태, 적용 범위와 갱신 조건을 명확히 하고 `docs/index.md`에서 한 번만 찾아갈 수 있게 중복 색인을 제거한다.
6. 이 임시 계획서는 정식 Beta 제품 계획과 실행 문서가 확정되면 유효한 내용을 옮긴 뒤 삭제한다.

### 7.4 완료 기준

- `docs/index.md`에서 연결된 모든 문서가 현재 운영 또는 유지보수 시점의 명확한 사용 목적을 가진다.
- 삭제 대상 문서에만 존재하던 유효한 제품 정책, 운영 명령과 아키텍처 결정이 남은 정본에 반영돼 있다.
- 문서의 내부 링크, 파일 경로와 소유 범위를 검사하고 코드·OpenAPI·Prisma 계약과 충돌하는 설명이 없다.
- 저장소 기여자가 기능 구현, 배포, 장애 대응과 제품 정책 확인에 필요한 문서를 색인에서 바로 찾을 수 있다.
- 정식 Beta 계획을 기준으로 `docs/index.md`의 현재 단계와 읽는 순서를 갱신한다.

## 8. 구현 순서와 선행 조건

1. B1 다중 정렬은 다른 Beta 기능과 독립적으로 완료한다.
2. B2 팀장 권한을 먼저 제공해 동적 참여 팀의 멤버와 워크플로 관리 주체를 확정한다. 현재 B2가 미구현 상태임을 확인했으므로 다음 구현은 B2로 돌아간다.
3. B3에서 데이터 모델과 API를 동적 참여 팀 기준으로 전환한 뒤 이슈 생성, TeamWork, 전달·완료와 화면을 함께 마이그레이션한다.
4. B4 GitHub 연동은 B2 팀장 권한과 B3 동적 참여 팀이 모두 완료되고 검증된 뒤 구현한다. 기획·디자인·사업·운영처럼 PR을 사용하지 않는 팀은 GitHub 설정 없이 동일한 TeamWork 기능을 사용한다.
5. 문서 인벤토리와 정본 후보 확인은 기능 구현과 병행할 수 있다. 실제 삭제는 유효한 계약을 정식 Beta 문서로 옮기고 관련 기능 검증이 끝난 뒤 수행한다.

## 9. 다음 협의 범위

B5 이후 제품 범위는 후속 대화에서 순서대로 확정한다. 협의 중 애매한 부분은 후속 결정으로 남기지 않고 해당 범위를 기록하기 전에 바로 확인한다. 합의가 끝나기 전에는 이 문서를 정식 Beta 계획이나 개발 착수 기준으로 사용하지 않는다.
