## 1. 문서 목적

이 문서는 구현된 전역 이슈 목록, 내 작업 목록과 통합 상세 화면을 실제 데스크톱 화면으로 검토하고 완료한 UI·UX 보완의 기준을 보존한다. 기존 제품 모델, 경로, 데이터 소유권과 상태 전이 계약은 유지하면서 다음 문제를 해결했다.

- 상태, 진행률과 다음 행동이 서로 다른 사실처럼 보이는 신뢰 문제
- 초광폭 목록과 상세에서 관련 정보와 행동이 지나치게 멀어지는 문제
- 작업 실행 정보와 이슈 설명·첨부파일·댓글이 같은 위계에서 경쟁하는 문제
- 전달의 빈 항목과 유사한 기록이 실제 변경 내용을 가리는 문제
- 활동 기록이 무엇을 어떻게 변경했는지 설명하지 못하는 문제
- 긴 댓글 작성기와 작은 도구가 읽기와 조작을 방해하는 문제

감사 근거는 2026-07-15에 제공된 데스크톱 화면이다. 이 문서는 화면에서 확인할 수 없는 키보드, 포커스, 반응형, 로딩, 오류와 실시간 갱신 동작을 구현 검증 대상으로 명시하며, 확인하지 않은 동작을 이미 충족한 것으로 간주하지 않는다.

## 2. 유지한 제품 계약

완료된 개선은 다음 계약을 바꾸지 않았다.

- 이슈는 제목·설명·첨부파일·댓글의 유일한 콘텐츠 원본이고 팀 작업은 실행 슬롯이다.
- 이슈 상세와 내 작업 상세는 같은 통합 상세 본문을 사용하고 진입 경로와 복귀 문맥만 구분한다.
- 상세의 기본 탭은 `업무`, `활동`이며 URL 직접 진입, 새로고침과 뒤로 가기에서 선택 상태를 복원한다. 전달 전체 이력은 업무의 작업 전달 영역에 기본 표시한다.
- 내 작업은 현재 사용자에게 할당된 미완료 팀 작업의 개인 실행 큐다.
- 내 작업의 데스크톱 열은 `우선순위 | 작업 | 프로젝트 | 상태 | 이름 없는 행동 영역`을 유지한다.
- 내 작업의 `BACKLOG` 사용자 정의 상태처럼 실행 가능한 주요 행동이 없는 행에는 비활성 버튼이나 자리 표시 문구를 추가하지 않는다.
- 전역 이슈 목록은 접수·분류 관점의 일곱 영역과 행마다 한 개의 주요 행동을 유지한다.
- 작업 전달은 append-only이며 최초·추가 전달의 시간순 원문을 보존한다.
- 활동은 댓글과 전달 원문을 복제하지 않고 관련 `업무` 또는 `전달` 위치로 이동할 수 있게 한다.
- 새로운 색상, 아이콘 체계, 중첩 카드와 별도 상세 리소스를 만들지 않는다.

## 3. 개선 원칙과 우선순위

### 3.1 신뢰를 먼저 고친다

상태, 진행률, 완료 수와 주요 행동은 같은 서버 사실과 도메인 규칙에서 파생해야 한다. 예를 들어 `완료` 이슈에 `0%` 진행률이 함께 보이거나 완료된 작업에 `작업 시작`이 보이는 상태를 허용하지 않는다.

- 진행률은 취소되지 않은 직계 팀 작업의 완료 수와 전체 수를 사용한다.
- 분모가 0이면 목록의 행동 가능 문맥에서 숫자 `0%`를 우선하지 않고 `아직 시작된 팀 작업이 없습니다`와 필요한 행동을 표시한다.
- API 응답 자체가 상태와 진행률의 불변식을 위반하면 클라이언트에서 임의로 값을 보정해 숨기지 않는다. 원인을 확인하고 계산·갱신 경계를 고친다.
- 낙관적 상태 변경 뒤에는 상태, 진행률, 목록 포함 여부와 주요 행동을 한 번에 일관되게 갱신한다.
- 저장 실패는 이전 값으로 복구하고 해당 위치에서 다시 시도할 수 있게 한다.

### 3.2 작업자가 지금 판단할 정보를 먼저 둔다

상세 화면의 첫 시선은 다음 질문에 답해야 한다.

1. 지금 선택한 이슈와 팀 작업은 무엇인가?
2. 현재 상태와 담당자는 누구인가?
3. 지금 실행할 행동은 무엇인가?
4. 가장 최근에 받은 또는 보낸 전달은 무엇인가?
5. 작업을 이해하는 데 필요한 이슈 맥락은 무엇인가?

설명, 첨부파일과 댓글의 원본 소유권은 유지하되 현재 작업의 실행 정보보다 강한 첫 화면을 만들지 않는다.

### 3.3 반복보다 요약을 우선한다

- 전달의 값이 없는 필드는 항목마다 `해당 없음`을 반복하지 않는다.
- 연속 활동은 개별 기록을 잃지 않는 범위에서 행위자·시각·대상 기준으로 시각적으로 묶는다.
- 같은 의미의 상태, 작업 코드와 제목을 헤더·작업 목록·본문에 반복하지 않는다.
- 보조 정보는 숨기는 대신 의미 있는 한 줄 요약과 명시적인 펼침 동작을 제공한다.

### 3.4 행동은 대상 가까이에 둔다

`편집`, `추가 전달`처럼 특정 섹션에 속한 행동은 초광폭 화면의 반대편 끝이 아니라 해당 섹션 제목 안에 둔다. 한 영역에는 Primary 버튼을 하나만 사용하고 단순 이동과 항목별 보조 행동은 Ghost 또는 더보기 메뉴 위계로 낮춘다.

## 4. 화면별 개선 기준

### 4.1 전역 이슈 목록

- 목록과 도구는 기존 1,440~1,600px 최대 폭을 실제 레이아웃에 적용해 초광폭에서도 `이슈`와 `다음 행동`의 거리가 과도하게 벌어지지 않게 한다.
- 헤더와 행은 같은 Grid 템플릿을 공유하고 긴 제목·프로젝트·라벨은 `min-width: 0`과 생략 처리로 열을 밀어내지 않는다.
- 이슈 상태, 완료 수, 진행률과 다음 행동의 불변식을 테스트로 고정한다.
- `담당자 지정`, `작업 시작`, `이슈 완료`처럼 결정이 필요한 행동과 `내 작업 열기`, `상세 보기` 같은 단순 이동의 버튼 위계를 구분한다.
- 사이드바와 본문 헤더의 `이슈 만들기`를 모두 유지하면 같은 생성 창과 단축키를 사용하고, 한 화면에서 둘 다 강한 Primary로 경쟁하지 않게 한다.
- 행 이동 영역과 우선순위·라벨·주요 행동·더보기 컨트롤은 중첩 링크를 만들지 않고 각각 키보드로 접근할 수 있게 한다.

### 4.2 내 작업 목록

- `우선순위 | 작업 | 프로젝트 | 상태 | 이름 없는 행동 영역` 구조와 프로젝트 아래 읽기 전용 라벨을 유지한다.
- 제목과 결과 수는 같은 행에 두고 결과 수는 제목의 일부임을 알 수 있는 작은 배지 또는 인접 텍스트로 표시한다.
- 검색, 상태·프로젝트 필터와 정렬을 `필터`와 `정렬` 의미 그룹으로 읽히게 배치한다.
- 아이콘만 표시하는 세부 필터, 정렬 방향과 초기화에는 한국어 tooltip과 접근 가능한 이름을 제공한다.
- 기본 정렬 이름 `실행 순서`는 유지하되 팝오버나 도움말에서 `진행 중 → 시작 전 → 백로그, 우선순위 순`의 의미를 설명한다.
- `작업 시작`은 현재 실행을 여는 Primary가 될 수 있고 `완료`는 같은 행에서 과도하게 반복 강조되지 않도록 Secondary 위계를 기본으로 한다.
- 실행 가능한 행동이 없는 `BACKLOG` 행은 행동 영역을 비워 두되 행 이동과 상태 편집 가능성이 hover, focus-visible과 접근 가능한 이름에서 분명해야 한다.
- 상태 변경으로 현재 조건에서 제외된 행은 성공 피드백과 함께 제거하고, 사용자가 실수한 경우 되돌릴 수 있게 한다.

### 4.3 통합 상세 헤더와 작업 선택

- 헤더는 이슈 제목을 가장 강한 제목으로 유지하고 이슈 표시 ID, 프로젝트, 우선순위, 진행률, 역할과 라벨은 의미가 구분되는 보조 정보로 표시한다.
- `없음 0% (0/2)`처럼 라벨 없는 값 나열을 사용하지 않는다. `우선순위 없음`, `작업 0/2 완료 · 0%`처럼 의미를 함께 전달한다.
- 내 작업 경로에서는 팀 작업 표시 ID, 상태와 현재 주요 행동을 상단 실행 영역에서 먼저 보여 준다.
- 왼쪽 팀 작업 목록의 선택 상태와 본문 제목이 같은 정보를 중복해 경쟁하지 않게 한다. 목록은 전환 문맥, 본문은 선택 작업의 상세 속성을 담당한다.
- 상태, 담당자와 현재 주요 행동은 같은 실행 영역에 두고 화면 양 끝으로 분리하지 않는다.
- 상세 본문은 Markdown 한 줄 길이 약 76자를 유지하는 읽기 폭을 사용하되 섹션 행동이 화면 끝으로 밀려나지 않게 컨테이너 폭을 제한한다.

### 4.4 업무 탭

업무 탭은 다음 순서로 읽히게 한다.

1. 선택 작업의 상태, 담당자, 역할과 주요 행동
2. 이슈의 전체 작업 전달 카드
3. 값이 있을 때만 작업 노트
4. 이슈 설명과 본문 이미지
5. 이슈 첨부파일
6. 이슈 댓글과 댓글 작성기

- 작업 노트가 없으면 큰 빈 영역을 만들지 않고 섹션 제목 가까이에 `작업 노트 추가`를 제공한다.
- 이슈 설명, 첨부파일, 댓글과 댓글 작성기는 하나의 중첩된 대형 카드가 아니라 같은 깊이의 형제 섹션으로 읽히게 한다.
- 이슈 설명은 작업 실행 영역보다 강한 표면을 사용하지 않는다. 긴 설명은 전체 내용을 잃지 않는 범위에서 읽기 폭과 섹션 간격으로 구분한다.
- 첨부파일의 파일명, 형식·크기, 업로더·시각과 명시적 다운로드를 유지하고 제거는 더보기 또는 낮은 위험 위계와 되돌리기를 사용한다.
- 댓글 목록의 `더 보기`에는 가능하면 남은 개수 또는 불러올 범위를 함께 표시한다.

### 4.5 댓글 작성기

- 빈 작성기의 데스크톱 기본 본문 높이는 약 120~160px로 시작하고 입력 내용에 따라 확장한다.
- 글자 수는 항상 강하게 노출하지 않고 제한에 가까워질 때 판독 우선순위를 높인다.
- 툴바는 기존 편집 기능을 유지하되 각 아이콘의 실제 hit area, tooltip, 접근 가능한 한국어 이름과 키보드 순서를 보장한다.
- 비활성 `댓글 남기기` 버튼은 텍스트 입력이 필요하다는 이유를 스크린리더가 이해할 수 있게 연결한다.
- 미리보기 전환, 이미지 삽입과 멤버 멘션 뒤에도 편집기 포커스와 선택 영역을 예측 가능하게 유지한다.

### 4.6 업무 탭의 작업 전달 전체 이력

- 시간순 append-only 기록은 유지하고 각 전달 요약 행에 최초·추가 구분, 작성자, 시각, 발신 작업, 수신 작업과 변경 항목 수 또는 짧은 요약을 표시한다.
- 전문은 한 번에 하나만 펼치는 것을 기본으로 하되 직접 링크나 사용자가 선택한 펼침 상태를 우선한다.
- 전달 본문은 별도 펼치기 조작 없이 카드에 바로 표시하며 카드별 질문 동작은 식별 가능한 더보기 메뉴에서 제공한다.
- 값이 있는 필드만 전문에 표시한다. 모든 선택 필드가 비어 있으면 `입력된 변경사항이 없습니다` 한 줄로 요약한다.
- `댓글로 질문`은 질문 대상 전달의 식별자를 댓글 문맥에 유지하고 댓글 작성 위치로 이동한 뒤 포커스를 복원한다.
- 전체 전달은 업무 탭에서 최신순 카드로 즉시 확인하고 각 카드의 원문만 개별로 펼친다. 전달 원문은 중복 저장하지 않는다.

### 4.7 활동 탭

- 각 이벤트는 아이콘, 행위자, 실제 변경 내용, 관련 팀 작업 표시 ID와 시각을 한 줄 중심으로 표시한다.
- `이슈 속성을 변경했습니다` 대신 가능한 경우 `우선순위: 보통 → 긴급`, `상태: 할 일 → 진행 중`처럼 변경 필드와 전후 값을 표시한다.
- 활동 계약에 전후 값이 없으면 화면에서 추측해 만들지 않는다. 현재 이벤트 payload를 확인하고 필요한 최소 계약을 별도 검토한다.
- 같은 행위자가 짧은 시간에 만든 연속 이벤트는 날짜·행위자·대상 기준으로 시각적으로 묶을 수 있지만 개별 이벤트의 시간순, 식별자와 접근 가능한 읽기 순서를 잃지 않는다.
- 댓글과 전달은 원문을 반복하지 않고 한 줄 요약과 해당 `업무` 또는 `전달` 위치로 이동하는 동작을 제공한다.
- 긴 활동에서는 날짜 구분을 제공하고 이벤트 유형 필터는 기본 판독성 개선 뒤 필요성을 검증해 추가한다.

## 5. 반응형과 접근성 기준

- 데스크톱 표를 단순 축소하거나 가로 스크롤에만 의존하지 않고 기존 모바일 정보 순서로 재배치한다.
- 직접 조작하는 컨트롤은 데스크톱 40px, 모바일 44px 이상의 실제 hit area를 가진다.
- 보조 텍스트와 비활성 텍스트를 구분하고 작은 본문에 `color.text.disabled`를 사용하지 않는다.
- 상태, 우선순위와 진행률은 색상만으로 구분하지 않고 아이콘, 이름과 숫자 요약을 함께 제공한다.
- 탭은 `aria-selected`, 전달 펼침은 `aria-expanded`, 진행률은 `aria-valuenow`와 한국어 요약을 제공한다.
- 상태 변경, 행 제거, 댓글 등록과 추가 로딩 결과를 적절한 live region으로 알리되 같은 내용을 중복 낭독하지 않는다.
- 키보드 포커스는 필터 팝오버, 상태 선택기, 댓글 미리보기와 `댓글로 질문` 이동 뒤 예측 가능한 위치로 복원한다.
- 진행 상태 아이콘에 반복 애니메이션이 있으면 `prefers-reduced-motion`에서 정적인 표현을 제공한다.
- 200% 확대와 제한적 모바일 폭에서 탭, 주요 행동, 작성기 툴바와 전달 전문이 잘리거나 겹치지 않아야 한다.

## 6. 구현 순서

### 6.1 상태와 진행률 정합성

- 상태·진행률·완료 수 계산과 갱신 경계를 확인한다.
- 완료, 완료 확인, 팀 작업 없음과 낙관적 변경 회귀 테스트를 추가한다.
- 화면에서 모순되는 값이 만들어지는 원인을 먼저 고친다.

### 6.2 목록 밀도와 행동 위계

- 전역 이슈와 내 작업의 최대 폭, Grid 정렬, 필터·정렬 그룹과 행동 버튼 위계를 정리한다.
- 현재 5열·7영역 계약, URL 복원과 행 내부 컨트롤 이벤트 경계를 유지한다.

### 6.3 업무 탭과 댓글 밀도

- 상단 실행 정보, 최신 전달, 작업 노트와 이슈 원본 영역의 위계를 정리한다.
- 이슈 설명·첨부파일·댓글을 같은 깊이로 분리하고 빈 작성기의 기본 높이를 줄인다.

### 6.4 전달과 활동 판독성

- 전달 요약 메타, 빈 필드 축약과 펼침 컨트롤 위치를 개선한다.
- 활동 이벤트가 가진 실제 변경 상세를 표시하고 부족한 payload는 추측하지 않고 계약 변경 필요성을 판단한다.

### 6.5 브라우저 검증

- 이슈 목록 → 이슈 상세 → 팀 작업 선택 → 전달 → 활동 → 목록 복귀 흐름을 확인한다.
- 내 작업 → 작업 상세 → 상태 변경 → 댓글 작성 → 목록 복귀 흐름을 확인한다.
- 데스크톱, 제한적 모바일, 200% 확대, 키보드, 포커스, 로딩, 빈 상태, 오류와 실시간 갱신을 확인한다.

## 7. 완료 기준

- 완료 상태, 진행률, 완료 수와 주요 행동이 서로 모순되지 않는다.
- 전역 이슈와 내 작업의 기존 정보 구조, 경로와 필터·정렬 복원이 유지된다.
- 상세 첫 화면에서 선택 작업, 상태, 담당자, 주요 행동과 최신 전달을 찾을 수 있다.
- 작업 노트나 전달 필드가 비어 있을 때 반복되는 큰 빈 영역을 만들지 않는다.
- 댓글 작성기는 비어 있을 때 본문보다 강한 높이를 차지하지 않는다.
- 전달 목록에서 각 기록의 작성자, 시각, 방향과 요약을 펼치기 전에 구분할 수 있다.
- 활동 목록에서 가능한 이벤트는 변경 필드와 전후 값을 확인할 수 있다.
- 직접 링크, 새로고침, 뒤로 가기와 스크롤 복원이 회귀하지 않는다.
- 데스크톱·모바일·200% 확대와 키보드 검증에서 핵심 행동이 잘리거나 포커스를 잃지 않는다.
- 관련 웹 테스트, lint, typecheck, build, API 계약 검사와 실제 Chromium 흐름 검증이 통과한다.
