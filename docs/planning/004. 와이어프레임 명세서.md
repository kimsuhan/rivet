## 1. 문서 목적

이 문서는 사용자 흐름 및 화면 명세서에서 확정한 화면을 저충실도 레이아웃으로 구체화한다. 화면마다 정보의 위치, 조작 순서, 영역 간 우선순위, 반응형 변화와 주요 상태를 정의해 UI 디자인과 프론트엔드 구현이 같은 구조에서 시작하도록 한다.

이 문서는 [DESIGN.md](../../DESIGN.md)를 Rivet의 상위 디자인 지침으로 사용한다. 어두운 캔버스, 단계형 표면, 절제된 라벤더 강조색, 가는 경계선, 촘촘한 제품 UI와 작업 연결선 원칙을 와이어프레임의 시각 방향에 반영한다. 표시한 구조 크기는 화면 관계를 검증하기 위한 기준이며, 최종 디자인 토큰과 한국어 타이포그래피 값은 이 문서의 범위에 포함하지 않는다.

### 1.1 범위

- 인증, 초대 수락과 초기 설정
- 공통 데스크톱 셸과 모바일 하단 탐색
- 전역 이슈 만들기, 검색과 공통 선택기
- 전역 이슈, 내 작업, 팀 목록과 팀 상태 보드
- 프로젝트 목록, 생성·편집과 상세
- 이슈 콘텐츠와 선택된 팀 작업을 합친 통합 상세
- 상태 주요 행동, 선택적 작업 전달, 댓글과 활동
- 이슈 첨부파일, 편집기 이미지 업로드와 프로필 사진
- 알림함
- 멤버, 팀, 워크플로, 라벨, 내보내기와 휴지통 설정
- 로딩, 빈 상태, 오류, 권한과 충돌 상태

### 1.2 이 문서에서 확정하지 않는 범위

- 최종 디자인 토큰과 실제 색상·타이포그래피 값
- 실제 API 요청·응답 형식
- 컴포넌트 라이브러리 선정
- 애니메이션의 시간과 이징 값
- MVP에서 제외된 저장된 보기, 타임라인, 외부 연동과 다중 워크스페이스

### 1.3 해석 우선순위

1. 제품 요구사항 정의서
2. 사용자 흐름 및 화면 명세서
3. `DESIGN.md`의 상위 디자인 방향
4. 이 문서의 레이아웃과 반응형 기준

상위 문서와 충돌하면 상위 문서를 따른다. 이 문서의 기준이 변경되면 관련 화면 명세와 색인을 함께 갱신한다.

### 1.4 DESIGN.md 적용 기준

`DESIGN.md`는 Rivet의 상위 디자인 방향을 정의한다. 다음 원칙을 와이어프레임의 기본값으로 적용하며, 정확한 디자인 토큰 수치는 이 문서에서 확정하지 않는다. 배경이 된 Linear 마케팅 분석은 [참고 자료](../reference/001.%20Linear%20시각%20디자인%20분석.md)일 뿐 제품 화면에 직접 적용하지 않는다.

| 기준 | 와이어프레임 적용 | 적용하지 않거나 조정할 부분 |
| --- | --- | --- |
| 캔버스 | `#010102`에 가까운 어두운 바탕을 기본 화면으로 사용 | 순수 검정 배경은 사용하지 않음 |
| 표면 단계 | `surface-1`부터 `surface-4`까지 단계형 표면과 가는 경계선으로 탐색, 본문, 모달과 선택기의 깊이를 구분 | 장식 목적의 그림자와 그라디언트는 사용하지 않음 |
| 강조색 | `#5e6ad2` 계열 라벤더를 주요 행동, 포커스, 선택 상태와 링크 강조에 제한적으로 사용 | 넓은 배경이나 장식 카드 채움으로 사용하지 않음 |
| 텍스트 | 밝은 본문, muted·subtle·tertiary의 단계로 제목, 보조 정보와 비활성 정보를 구분 | 한국어는 과도한 음수 자간을 적용하지 않음 |
| 간격 | 4px 단위를 기반으로 8·12·16·24·32px 흐름을 사용 | 마케팅 섹션 간격 96px은 업무 화면에 그대로 적용하지 않음 |
| 모서리 | 입력·버튼 8px, 일반 패널 12px을 출발점으로 사용 | 모든 요소를 pill 형태로 만들지 않음 |
| 밀도 | 제품 정보가 주인공이 되도록 탐색과 장식을 최소화 | 마케팅용 대형 카드·스크린샷 프레임은 업무 화면에 반복하지 않음 |
| 깊이 | 표면 밝기 차이와 1px 경계선으로 계층 표현 | 드롭 섀도에 의존하지 않음 |

#### 화면 영역별 표면 기준

| 화면 영역 | 기본 표면 |
| --- | --- |
| 앱 캔버스와 목록 배경 | canvas |
| 주 탐색과 목록 hover | surface-1 |
| 속성 패널, 카드와 선택된 탭 | surface-2 |
| 드롭다운, 팝오버와 모달 | surface-3 |
| 현재 포커스 | 라벤더 focus ring + 해당 표면 유지 |

성공, 오류, 경고, 우선순위와 여러 상태는 제품 업무 UI에서 추가 의미 색상이 필요하다. `DESIGN.md`의 절제된 강조색 원칙을 유지하도록 의미 색상은 작은 아이콘, 배지와 상태 표시 안에서만 사용하고 텍스트를 함께 제공한다. 구체적인 의미 색상 값은 디자인 시스템 명세를 따른다.

## 2. 와이어프레임 원칙

### 2.1 업무 맥락을 먼저 보여 준다

- 이슈 제목보다 먼저 상태를 찾게 하지 않는다. 제목과 표시 ID가 행과 상세의 시작점이다.
- 팀 작업에는 팀, 프로젝트 역할, 워크플로 상태와 현재 주요 행동을 분리해 표시한다.
- 이슈 자체에는 팀과 담당자를 표시하지 않는다.
- 프로젝트 상세는 이슈와 역할별 팀 작업의 부모·자식 관계를 기본 구조로 사용한다.

### 2.2 목록에서 자주 바꾸고 상세에서 깊게 다룬다

- 목록과 보드에서 상태, 담당자와 우선순위를 바로 변경한다.
- 설명, 관계, 작업 전달, 댓글과 활동은 이슈 상세에서 관리한다.
- 상세 화면은 독립 경로를 사용하고 돌아갈 때 이전 필터, 정렬과 스크롤을 복원한다.

### 2.3 팀과 상태를 섞지 않는다

- `API 작업 중`, `웹 작업 대기`처럼 팀 단계를 상태 이름으로 만들지 않는다.
- 실제 진행 상태는 팀별 워크플로로 표현하며 수동 `BLOCKS` 관계는 만들지 않는다.
- 동시에 시작한 역할 작업은 병렬로 표시하고 백엔드 뒤에 이어지는 프론트 작업은 전달 완료 뒤 실제 작업으로 표시한다.

### 2.4 생성보다 이해가 어려운 동작을 더 설명한다

- 빠른 생성에는 최소 필드만 먼저 보여 준다.
- 팀을 바꿀 수 없는 이유, 역할 변경 제한, 삭제 제한과 작업 전달 필수 조건은 동작 위치에서 설명한다.
- 확인창은 단순 경고 대신 영향, 복구 가능 여부와 해결 경로를 제공한다.

## 3. 화면 기준과 반응형 구조

### 3.1 기준 화면

| 구분 | 검증 기준 | 구조 |
| --- | --- | --- |
| 데스크톱 | 너비 1280px 이상 | 고정 주 탐색 + 본문 |
| 작은 데스크톱 | 너비 1024~1279px | 축소 가능한 주 탐색 + 본문 |
| 모바일 | 너비 1023px 이하 | 상단 제목 영역 + 본문 + 고정 하단 탐색 |

태블릿 전용 레이아웃은 만들지 않는다. 1023px 이하에서는 모바일 규칙을 적용한다. 관리 화면, 팀 보드, 프로젝트 편집처럼 모바일에서 지원하지 않는 기능은 화면을 억지로 축소하지 않고 데스크톱 사용 안내를 표시한다.

### 3.2 데스크톱 기준 영역

| 영역 | 기준 | 내용 |
| --- | --- | --- |
| 주 탐색 | 기본 240px, 축소 시 아이콘 중심 | 워크스페이스 이름, 주요 메뉴, 팀 목록, 만들기, 검색, 사용자 메뉴 |
| 본문 헤더 | 본문 상단 고정 가능 | 화면 제목, 보기 전환, 필터, 정렬, 주요 동작 |
| 본문 | 남은 너비 전체 | 목록, 보드, 프로젝트 또는 상세 |
| 우측 속성 패널 | 이슈 상세에서 약 320px | 상태, 담당자, 우선순위, 프로젝트, 역할과 라벨 |

이슈 상세는 주 본문과 우측 속성 패널을 분리해 긴 설명과 협업 기록을 읽는 동안에도 핵심 속성에 접근할 수 있게 한다.

### 3.3 모바일 기준 영역

| 영역 | 내용 | 규칙 |
| --- | --- | --- |
| 상단 바 | 뒤로 가기, 화면 제목, 화면별 보조 동작 | 한 줄 유지, 긴 제목은 본문에서 전체 표시 |
| 본문 | 현재 화면의 핵심 내용 | 한 열, 가로 스크롤을 기본 해결책으로 사용하지 않음 |
| 하단 탐색 | 이슈, 내 작업, 알림함, 프로젝트, 팀 | 주 네 항목의 상대 순서를 유지하고 아이콘과 한국어 이름을 함께 표시. 검색은 상단 전역 동작 사용 |

모바일에서 지원하지 않는 동작은 비활성 버튼으로 남겨 두지 않는다. 필요한 경우 `데스크톱에서 설정할 수 있습니다` 안내와 돌아가기 동작만 제공한다.

## 4. 공통 애플리케이션 셸

### 4.1 DESKTOP-SHELL

| 순서 | 영역 | 표시 내용 | 주요 동작 |
| ---: | --- | --- | --- |
| 1 | 워크스페이스 헤더 | 워크스페이스 이름 | 전환 메뉴 없음 |
| 2 | 빠른 동작 | `이슈 만들기`, `검색` | C, / 단축키 병기 가능 |
| 3 | 주 탐색 | 이슈, 내 작업, 알림함, 프로젝트 | 이 순서를 유지하고 현재 위치와 읽지 않은 알림 수 표시 |
| 4 | 팀 탐색 | 활성 팀 이름과 키 | 마지막 사용한 목록 또는 보드로 이동 |
| 5 | 하단 영역 | 관리자 설정, 프로필 사진·이름을 포함한 사용자 메뉴 | 권한이 없으면 설정 진입점은 숨기되 프로필 설정과 로그아웃은 모든 멤버에게 제공 |
| 6 | 본문 | 화면 헤더와 화면별 콘텐츠 | 주 탐색과 독립 스크롤 |

보관된 팀과 워크스페이스 전환기는 표시하지 않는다. 데스크톱 주 탐색은 팀 이름과 메뉴 이름이 보이는 펼친 상태를 기본으로 하고 사용자가 아이콘 중심의 축소 상태로 전환할 수 있게 한다. 축소해도 팀은 키 또는 구분 가능한 짧은 표시로 접근할 수 있어야 하며, 마우스를 올릴 때만 일시적으로 확장하는 방식은 사용하지 않는다.

### 4.2 MOBILE-SHELL

| 하단 항목 | 기본 이동 | 표시 상태 |
| --- | --- | --- |
| 이슈 | `/issues` | 현재 위치 |
| 내 작업 | `/my-issues` | 현재 위치 |
| 알림함 | `/inbox` | 읽지 않은 개수 |
| 프로젝트 | `/projects` | 현재 위치 |
| 팀 | 마지막 방문 팀 목록 | 방문 기록이 없으면 팀 선택 |

이슈 상세처럼 하단 항목에 직접 대응하지 않는 화면에서는 진입 출처의 항목을 유지한다. 예를 들어 프로젝트에서 이슈를 열면 프로젝트 항목을 선택 상태로 유지한다.

모바일 상단 앱 바에는 전역 검색 동작을 두고 오른쪽에는 28px 프로필 사진 또는 첫 글자 아바타를 둔다. 아바타 선택 시 프로필 설정과 로그아웃이 있는 사용자 메뉴를 연다.

### 4.3 본문 헤더 규칙

본문 헤더는 왼쪽에서 오른쪽으로 다음 순서를 사용한다.

1. 화면 제목과 결과 수 또는 보조 설명
2. 현재 보기 또는 탭
3. 필터와 정렬
4. 화면의 주요 생성 동작

작은 데스크톱에서 공간이 부족하면 필터와 정렬을 `보기 옵션` 메뉴로 합치고 주요 생성 동작은 유지한다. 모바일에서는 제목과 한 개의 핵심 동작만 상단에 두고 나머지는 더보기 또는 본문 필터 시트로 이동한다.

## 5. 공통 오버레이와 선택기

### 5.1 GLOBAL-01 전역 이슈 만들기

데스크톱에서는 화면 중앙의 모달, 모바일에서는 MVP 미지원 안내를 사용한다.

| 영역 | 내용 |
| --- | --- |
| 상단 | `이슈 만들기`, 닫기. 유형 전환 없음 |
| 첫 입력 | 제목 |
| 필수 맥락 | 프로젝트. 이슈 상태는 서버가 자동 계산하므로 입력하지 않음 |
| 시작 역할 | `처음 작업할 팀 (선택)`, 프로젝트의 백엔드·웹·앱 역할 복수 선택 |
| 선택 속성 | 우선순위, 라벨 |
| 설명 영역 | 펼쳐진 Lexical 편집·미리보기 |
| 첨부 영역 | 파일 선택, 파일별 이름·크기·업로드 상태 |
| 하단 | 취소, `이슈 만들기` |

#### 배치 규칙

- 제목은 모달 상단 전체 너비를 사용하고 열리자마자 포커스한다.
- 프로젝트 상세에서 열면 프로젝트 값을 읽기 전용으로 고정하고 전역에서는 활성 프로젝트를 필수 선택한다.
- 프로젝트는 제목 바로 아래에 배치하고 시작 역할은 그다음 전체 너비에 둔다. 이슈 상태, 담당자, 실제 팀과 팀별 상태 입력은 표시하지 않는다.
- 시작 역할에는 프로젝트에 설정된 역할만 표시하고 `아직 작업 범위가 정해지지 않았다면 선택하지 않아도 됩니다.`를 함께 제공한다.
- 각 역할은 키보드로 독립 전환할 수 있고 선택 텍스트와 접근성 상태를 함께 제공한다.
- 제목은 한 줄 요약, 설명은 실제 작업 맥락을 기록하는 영역이므로 설명 편집기를 기본으로 펼친다.
- 설명 편집기는 Lexical WYSIWYG 편집과 Markdown 미리보기 전환을 항상 제공하며, Markdown 원문 편집 화면은 노출하지 않는다. 설명을 입력하지 않아도 이슈를 저장할 수 있다.
- 모달 높이가 화면을 넘으면 제목과 속성 영역을 포함한 본문만 스크롤하고 하단의 취소·`이슈 만들기` 동작은 고정한다.
- 저장 실패 시 모달을 닫지 않고 첫 오류 필드로 포커스를 이동한다.
- 본문 이미지와 일반 첨부파일 업로드가 끝나기 전에는 만들기를 실행하지 않는다. 실패한 파일에는 다시 시도와 제거를 함께 제공한다.

### 5.2 GLOBAL-02 전역 검색

데스크톱에서는 상단 중앙의 넓은 검색 오버레이, 모바일에서는 화면 전체 검색 레이어를 사용한다.

| 상태 | 본문 |
| --- | --- |
| 빈 입력 | `표시 ID 또는 제목으로 검색하세요` 안내만 표시 |
| 한 글자 | 표시 ID 정확 일치만 시도, 제목 검색 조건 안내 |
| 두 글자 이상 | 결과 목록 표시 |
| 결과 없음 | 검색 범위가 ID와 제목임을 안내 |
| 로딩 | 기존 검색어 유지, 결과 영역에 로딩 표시 |

결과 한 행은 유형, 표시 ID, 제목, 프로젝트, 팀 또는 역할과 상태 순서로 읽힌다. 키보드 선택 행은 배경 외에도 보조 기술 상태로 구분한다. 결과를 열었다가 돌아오면 검색어와 선택 위치를 복원한다.

### 5.3 GLOBAL-03 공통 속성 선택기

상태, 담당자, 우선순위, 라벨, 프로젝트 선택기는 호출한 값 바로 아래에 열린다. 선택기 상단에는 검색이 필요한 항목에만 검색 입력을 제공하고, 본문에는 현재 선택과 가능한 값을 표시한다.

이슈 관련 속성은 두 밀도를 사용한다. 목록 행·보드 카드·프로젝트 작업 행의 `Compact`는 28~32px 시각 표면과 데스크톱 40px·모바일 44px 이상의 조작 영역을 분리한다. 통합 이슈 상세의 `Comfortable`은 34~38px 시각 표면과 44px 조작 영역을 사용한다. 두 밀도 모두 기본 배경과 경계를 투명하게 두고 같은 상태 체계를 사용한다. hover에는 약한 표면, open에는 한 단계 더 분명한 표면, focus-visible에는 2px 포커스 링을 표시한다.

상태와 우선순위는 목록·상세·메뉴·보드에서 같은 아이콘, 한국어 이름과 순서를 사용한다. 라벨은 색점이 있는 칩을 최대 두 개까지 표시하고 나머지는 `+N`으로 요약한다. 선택 팝오버는 Portal의 불투명 표면을 사용한다. Escape와 키보드 선택으로 닫으면 호출부로 포커스를 돌리고, 바깥의 다른 조작 대상을 선택해 닫은 경우에는 그 대상의 포커스를 유지한다.

| 상태 | 호출부 표현 |
| --- | --- |
| 저장 중 | 값 옆 진행 표시, 재선택 방지 |
| 저장 성공 | 선택 값 유지 |
| 저장 실패 | 이전 값 복원, 다시 시도 |
| 충돌 | 최신 값 표시, 내 변경 다시 적용 동작 |

### 5.4 GLOBAL-04 확인·차단 모달

| 차단 사유 | 연결된 작업 또는 필수 입력 누락 |

동작을 실행할 수 없으면 위험 동작 버튼을 비활성화하는 데 그치지 않고 해결 경로를 기본 행동으로 제공한다.

### 5.5 GLOBAL-05 프로필 설정

사용자 메뉴에서 데스크톱은 480px 모달, 모바일은 화면 아래 시트로 연다.

| 순서 | 영역 | 내용 |
| ---: | --- | --- |
| 1 | 현재 사용자 | 40px 프로필 사진 또는 표시 이름 첫 글자, 표시 이름, 이메일 |
| 2 | 사진 선택 | JPEG·PNG·WebP, 파일당 최대 25MB 안내와 파일 선택 |
| 3 | 미리보기 | 프론트에서 최대 512px로 최적화된 결과와 업로드 상태 |

업로드 중에는 기존 사진을 유지하고 새 사진 위에 진행 상태를 표시한다. 실패하면 기존 사진을 바꾸지 않고 다시 시도와 파일 다시 선택을 제공한다. 사진을 삭제하면 즉시 표시 이름 첫 글자 아바타로 돌아간다. 표시 이름과 이메일은 이 모달에서 편집하지 않는다.

## 6. 인증과 초기 설정

인증과 초기 설정 화면은 업무 셸을 사용하지 않는다. 화면 중앙의 단일 카드와 간단한 제품 이름만 표시한다. 데스크톱 카드의 읽기 폭을 제한하고 모바일에서는 화면 여백을 제외한 전체 너비를 사용한다.

### 6.1 AUTH-01 회원가입

| 순서 | 영역 | 내용 |
| ---: | --- | --- |
| 1 | 제목 | `계정 만들기`와 이메일 인증 안내 |
| 2 | 입력 | 표시 이름, 이메일, 비밀번호, 비밀번호 확인 |
| 3 | 오류 | 각 입력 바로 아래의 필드 오류 |
| 4 | 주요 동작 | `회원가입` |
| 5 | 보조 동작 | `이미 계정이 있나요? 로그인` |

제출 중에는 입력값을 유지하고 버튼에서 진행 상태를 표시한다. 전체 오류가 필요하면 버튼 위에 두되 필드 오류를 대체하지 않는다.

### 6.2 AUTH-02 이메일 인증

동일 카드에서 상태에 따라 본문과 주요 동작만 바꾼다.

| 상태 | 주요 내용 | 주요 동작 |
| --- | --- | --- |
| 발송됨 | 가린 이메일, 메일 확인 안내 | 재발송 대기 또는 재발송 |
| 성공 | 인증 완료 | 로그인 |
| 만료 | 링크 만료 | 인증 메일 재발송 |
| 이미 사용 | 이미 인증됨 | 로그인 |
| 유효하지 않음 | 사용할 수 없는 링크 | 회원가입 또는 로그인 |

### 6.3 AUTH-03 로그인

회원가입과 같은 카드 폭과 입력 간격을 사용한다. 입력 순서는 이메일, 비밀번호이며 `비밀번호를 잊으셨나요?`는 비밀번호 입력 가까이에 둔다. 인증되지 않은 계정과 비활성 멤버 안내는 로그인 버튼 위의 상태 영역에 표시한다.

### 6.4 AUTH-04 비밀번호 재설정

요청 단계와 새 비밀번호 입력 단계를 별도 경로에서 같은 카드 구조로 제공한다. 요청 완료 화면은 계정 존재 여부를 드러내지 않는 동일 문구를 사용한다. 링크 오류는 카드 상단 상태 영역에서 새 링크 요청으로 연결한다.

### 6.5 AUTH-05 초대 수락

| 순서 | 영역 | 내용 |
| ---: | --- | --- |
| 1 | 초대 요약 | 워크스페이스 이름, 초대한 사람 |
| 2 | 계정 정보 | 초대 이메일과 현재 로그인 이메일 |
| 3 | 상태 설명 | 로그인 필요, 이메일 불일치, 만료 또는 다른 워크스페이스 참여 |
| 4 | 주요 동작 | 가입, 로그인, 수락 또는 현재 워크스페이스로 이동 |

이메일 불일치와 1계정 1워크스페이스 제한에서는 수락 버튼을 표시하지 않는다.

### 6.6 ONB-01~03 초기 설정 공통

초기 설정은 상단에 `워크스페이스 → 기본 팀 → 동료 초대` 진행 단계를 표시한다. 완료한 단계는 다시 입력하도록 강제하지 않고, 새로고침하거나 재진입하면 미완료 단계부터 연다.

| 화면 | 본문 | 하단 동작 |
| --- | --- | --- |
| ONB-01 워크스페이스 | 이름, 슬러그, 주소 미리보기, 슬러그 확인 상태 | 워크스페이스 만들기 |
| ONB-02 기본 팀 | 팀 이름, 팀 키, 표시 ID 예시, 생성자 멤버 | 팀 만들기 |
| ONB-03 동료 초대 | 이메일 반복 입력, 주소별 발송 결과 | 건너뛰기, 초대 후 계속 |

ONB-03 완료 화면에는 `첫 이슈 만들기`를 주요 동작, `내 작업으로 이동`을 보조 동작으로 제공한다. 별도 워크스페이스 추가 또는 전환 동작은 어디에도 표시하지 않는다.

## 7. 업무 화면

### 7.1 ISSUE-LIST-01 전역 이슈 목록

#### 데스크톱 구조

| 순서 | 영역 | 내용 |
| ---: | --- | --- |
| 1 | 본문 헤더 | 한 행에 `이슈`, 현재 결과 수, `이슈 만들기` |
| 2 | 통합 도구 영역 | 320~420px 검색, 건수 포함 빠른 필터, 세부 필터, 정렬 기준·방향. 부족한 폭에서는 두 줄로 자연스럽게 감쌈 |
| 3 | 활성 필터 | 세부 필터가 있을 때만 이름·값·개별 제거 칩과 전체 초기화 |
| 4 | 열 헤더 | 우선순위, 이슈, 상태, 현재 작업, 진행, 업데이트, 다음 행동 |
| 5 | 이슈 목록 | 삭제되지 않은 이슈와 행마다 한 개의 주요 행동 |
| 6 | 추가 로딩 | 기존 결과를 유지한 하단 로딩 |

본문과 도구는 앱 셸 안에서 약 1,440~1,600px 최대 폭으로 가운데 정렬한다. 1,280~1,440px에서는 사용 가능한 폭을 유지하고, 초광폭에서도 마지막 행동 열이 화면 끝으로 멀어지지 않게 실제 열 사이 거리를 제한한다.

데스크톱 이슈 행은 보조 정보가 있어도 72~84px 안에서 유지한다. 우선순위 셀은 이슈 왼쪽의 좁은 독립 열에서 아이콘·한국어 이름의 compact 편집 트리거로 표시하고, 이슈 셀은 표시 ID와 가장 강한 제목을 첫 줄, 프로젝트·최대 두 개 라벨·`+N`을 같은 보조 줄에 묶는다. 라벨이 없으면 빈 라벨 선택기를 평상시에 표시하지 않는다. 상태 셀은 자동 계산된 상태를 아이콘·한국어 이름의 읽기 표현으로 표시한다. 헤더와 모든 행은 같은 Grid 템플릿을 공유해 상태·우선순위의 값 길이가 이슈·현재 작업 등 다른 열의 시작점을 움직이지 않게 한다. 현재 작업 셀은 제목과 역할·실제 팀·담당 필요를 최대 두 줄로 요약한다. 진행 셀은 완료 수/전체 수·백분율과 짧은 막대를 가까이 두며 업데이트 셀에는 상대 시각만 둔다. 만든 사람은 기본 열에서 제거하고 상세나 보조 정보에서 확인한다.

이슈 상태와 우선순위는 기본 상태에서 투명한 읽기 표현을 사용한다. 이슈 상태는 소속 팀 작업으로 계산된 읽기 값이므로 hover·open 표면이나 일반 선택 메뉴를 제공하지 않는다. 우선순위와 팀 작업 상태는 hover에 약한 표면, open에 한 단계 더 분명한 표면, focus-visible에 2px 링을 표시해 편집 가능성과 읽기 값을 구분한다. 실제 hit area는 데스크톱 40px, 모바일 44px 이상을 유지하되 보이는 컨트롤은 28~32px 밀도로 정돈한다. 편집 가능한 상태와 우선순위 메뉴는 트리거와 같은 아이콘·이름을 사용하고, 중간 폭에서 우선순위 이름을 축약해도 접근 가능한 전체 이름을 유지한다.

팀 작업 전체를 펼치지 않으며 `분석 중 · 시작 팀 없음`, 병렬 진행 역할, 담당 필요 또는 `모든 팀 작업 완료`를 표시한다. 행의 독립된 상세 링크가 빈 영역 선택과 키보드 열기를 담당하고 읽기 전용 이슈 상태, 우선순위·라벨 선택기, 주요 버튼·더보기·진행 링크는 별도 의미와 이벤트 경계를 가진다.

주요 행동은 `작업 시작 → 담당자 지정 → 우리 팀에서 시작 → 내 작업 열기 → 이슈 완료 → 상세 보기` 순으로 하나만 표시한다. 나머지 동작은 더보기 메뉴에 둔다.

결정이 필요한 작업 시작·담당자 지정·이슈 완료만 낮은 채도의 Secondary 또는 조건부 Primary를 사용하고, 단순 이동은 Ghost나 행 선택, 더보기는 배경 없는 작은 아이콘 버튼을 사용한다. 행동의 보이는 높이는 30~32px, 실제 hit area는 데스크톱 40px·모바일 44px 이상으로 분리한다. 시작 가능한 현재 사용자 역할이 하나면 보이는 문구를 `작업 시작`처럼 줄일 수 있지만 tooltip과 접근 가능한 이름에는 `웹 프론트 작업 시작`처럼 실제 대상을 남긴다. 여러 역할이면 `우리 팀에서 시작`을 유지해 선택창에서 역할과 팀을 분명히 한다.

라벨 편집 아이콘은 행 hover·focus·열림 상태에서만 보인다. 팝오버는 행 stacking context 밖의 Portal에 완전히 불투명한 표면, 약한 1px 경계와 작은 shadow로 표시하고 호출부와 4~8px 떨어뜨린다. viewport 가장자리에서 충돌을 피하며 Escape와 키보드 선택으로 닫으면 호출 아이콘으로 포커스를 돌리고, 바깥의 다른 조작 대상을 선택한 경우에는 그 대상의 포커스를 유지한다. 표 헤더 아래와 행 사이의 낮은 명암 구분선만 남기고 상태·우선순위·더보기에는 상시 테두리를 반복하지 않는다.

#### 배정 오버레이

- `작업 시작`: 역할별 연결 팀, 담당자 또는 담당자 없음, 기존 미완료 역할 작업 여부를 표로 표시하고 하나 이상의 역할을 요구한다.
- `우리 팀에서 시작`: 가능한 역할이 하나면 확인창, 여러 개면 역할 선택창을 사용하고 생성 결과가 미할당임을 알린다.
- `내가 맡기`: 후보 한 건은 확인, 여러 건은 표시 ID·제목·역할·팀으로 대상 작업을 선택한다.
- `담당자 지정`: 미할당 미완료 작업만 기본 선택하고 역할별 활성 팀 멤버 후보를 제공한다. 충돌 시 창을 닫지 않고 입력값, 최신 담당자와 후보를 함께 표시한다.
- `이슈 완료`: 자동 계산된 `완료 확인` 상태에서 완료 역할·팀 작업과 제외된 취소 작업을 요약하고 명시적 인수로 완료한다.

오버레이는 모달 안에 또 다른 모달을 겹치지 않는다. 담당자 선택기는 현재 모달 레이어에서 열고, 저장 실패 시 첫 오류로 포커스를 옮기며 닫을 때 호출 버튼으로 돌린다.

#### 모바일 구조

- 상단 첫 행: `이슈`, 결과 수와 이슈 만들기
- 도구: 검색, 필터와 정렬을 감싸서 배치하고 활성 세부 필터 칩은 값이 있을 때만 표시
- 빠른 필터: 현재 검색·세부 필터 기준 전체 건수를 포함한 한 줄 스크롤 탭 또는 짧은 선택 메뉴
- 필터: 하단 시트 또는 전체 화면 패널
- 카드 읽기 순서: 표시 ID·제목 → 프로젝트·라벨 → 상태·우선순위 → 현재 작업·담당 필요 → 진행률 → 업데이트 → 한 개의 주요 행동
- 이슈 상태는 데스크톱과 같은 아이콘·이름의 읽기 표현을 사용하고, 우선순위와 팀 작업 상태만 compact 인라인 트리거를 사용하며 큰 폼 Select 표면으로 바꾸지 않음
- 작업 시작·우리 팀에서 시작·내가 맡기·담당자 지정: 전체 화면 시트에서 데스크톱과 같은 검증·충돌 복구 제공

표를 가로로 축소하거나 가로 스크롤에만 의존하지 않는다.

긴 설명과 큰 상단 여백은 두지 않는다. 제목과 결과 수를 같은 행에 두고 `이슈 만들기`, Compact 필터와 정렬을 전역 이슈 목록과 같은 콘텐츠 폭·상단 간격으로 배치한다. 활성 필터는 이름과 값이 있는 칩으로만 강조하고 개별 제거와 전체 초기화를 제공한다.

작업 셀에는 이슈 제목을 가장 강하게 표시하고 둘째 줄에 팀 작업 표시 ID, 상위 이슈 표시 ID와 역할을 둔다. 프로젝트는 이름과 최대 두 개의 상위 이슈 라벨·나머지 `+N`을 독립 열에 표시한다. 실제 팀 이름은 반복하지 않는다. 팀 작업 상태는 Compact 인라인 트리거, 상위 이슈 우선순위는 목록 왼쪽의 같은 밀도 읽기 표현을 사용한다. 담당자와 최근 수정 시각은 기본 행에서 제외한다.

행동 영역에는 헤더 이름을 두지 않고 `UNSTARTED`의 `작업 시작`, `STARTED`의 `완료`처럼 현재 가능한 행동 한 개만 표시한다. 실행 가능한 행동이 없는 `BACKLOG` 사용자 정의 상태에서는 비활성 버튼이나 빈 상태 문구를 만들지 않는다. 행 전체를 선택하면 내 작업 상세로 이동하고 상태와 주요 행동은 행 이동보다 자신의 조작을 우선한다. 완료·취소 상태로 변경되거나 다른 화면에서 담당자가 바뀌어 현재 조건에서 빠지는 행은 저장 성공 또는 SSE 재조회 뒤 목록에서 제거한다.

기본 정렬은 `실행 순서`다. `STARTED → UNSTARTED → BACKLOG`, 상위 이슈 우선순위, 같은 범주의 워크플로 표시 순서, 최근 수정 내림차순과 불변 ID 내림차순으로 정렬한다. 프로젝트 이름·상태 필터, 정렬과 스크롤은 URL과 뒤로 가기에서 복원한다.

| 3 | 프로젝트 목록 | 프로젝트별 요약 행 또는 넓은 카드 |

프로젝트 항목의 정보 순서는 다음과 같다.

1. 이름, 설명 요약과 상태
2. 리드, 시작일과 목표일
3. 백엔드·웹 프론트·앱 프론트 역할과 선택된 팀
4. 완료 작업 수와 전체 작업 수
5. 진행률

선택하지 않은 역할은 `없음`을 반복하기보다 항목에서 생략한다. 프로젝트 상태와 진행률은 한 영역에 함께 표시하지만 어느 하나를 다른 값으로 추론하지 않는다.

#### 모바일 구조

한 프로젝트를 세로 카드로 표시한다. 첫 줄에 이름과 상태, 둘째 영역에 역할별 팀, 마지막 영역에 일정과 진행률을 둔다. 프로젝트 만들기와 편집 동작은 표시하지 않고 상세 조회만 제공한다.

### 7.6 PROJ-02 프로젝트 생성·편집

데스크톱 전용 독립 화면이다. 본문 폭을 제한해 폼의 왼쪽 정렬을 유지한다.

| 순서 | 섹션 | 구성 |
| ---: | --- | --- |
| 1 | 화면 헤더 | `프로젝트 만들기` 또는 프로젝트 이름, 취소 |
| 2 | 기본 정보 | 이름, 설명, 리드 |
| 3 | 일정 | 시작일, 목표일 |
| 4 | 역할별 팀 | 백엔드, 웹 프론트, 앱 프론트 선택기 |
| 5 | 제한·오류 | 날짜 관계, 최소 한 역할, 연결 작업 영향 |
| 6 | 하단 고정 동작 | 취소, 만들기 또는 변경 저장 |

역할별 팀은 세 개의 같은 형태의 행으로 표시한다.

| 행의 왼쪽 | 행의 오른쪽 |
| --- | --- |
| 역할 이름과 설명 | 활성 팀 선택 또는 `선택 안 함` |

같은 팀을 여러 역할에 선택해도 오류로 표시하지 않는다. 최소 한 역할이 선택되지 않으면 저장 시 역할 섹션 전체에 오류를 표시하고 첫 역할 선택기로 포커스를 이동한다. 편집 중 연결된 팀 작업 때문에 역할을 바꿀 수 없으면 해당 행에 작업 수와 `영향받는 작업 보기`를 표시한다.

모바일 접근 시 프로젝트 목록으로 돌아가는 동작과 데스크톱 사용 안내를 표시한다.

### 7.7 PROJ-03 프로젝트 상세

#### 데스크톱 상단

| 영역 | 내용 |
| --- | --- |
| 제목 행 | 프로젝트 이름, 상태, 더보기 |
| 메타 행 | 리드, 시작일, 목표일 |
| 역할 행 | 역할 배지 + 담당 팀 |
| 진행 행 | 완료 작업 수, 전체 작업 수, 진행률 |
| 동작 행 | 이슈 만들기, 보조 메뉴, 편집 |

#### 데스크톱 본문

| 순서 | 영역 | 내용 |
| ---: | --- | --- |
| 1 | 보기 도구 | 역할 필터, 상태 필터, 정렬 |
| 2 | 이슈 목록 | 이슈 상위 행과 분석·병렬·순차·전달 작업 흐름 |

프로젝트에 처음 진입하면 모든 이슈의 작업 흐름을 펼쳐 현재 역할을 바로 보여 준다. 사용자는 이슈별로 흐름을 접거나 다시 펼칠 수 있으며, 이슈 상세에서 프로젝트로 돌아오면 기존 펼침 상태를 복원한다.

이슈 상위 행은 제목, 자동 계산된 읽기 전용 상태, 현재 역할·상태와 실제 팀 작업 진행률을 표시한다. 수동 `팀 작업 추가`는 더보기 메뉴에 둔다. 펼친 영역은 다음 순서로 읽힌다.

1. 팀 작업이 없으면 분석 안내와 `작업 시작`
2. 완료된 작업과 전달 완료 이력
3. 복수의 `현재 작업` 역할 배지와 팀
4. 표시 ID, 제목, 상태, 담당자와 현재 주요 행동
5. 최신 전달과 전체 전달 이력 연결

이슈 상태는 다른 목록과 같은 Compact 아이콘·이름의 읽기 표현을 사용한다. 팀 작업 상태·담당자는 실행 레코드, 우선순위·라벨은 상위 이슈 속성으로 같은 시각 밀도에서 표현한다. 편집 가능한 값만 hover·focus 표면과 메뉴를 제공한다.

역할 필터는 이슈 자체를 숨기지 않는다. 선택한 역할의 현재·완료 작업이나 예상 단계가 없는 이슈는 빈 부모 행을 남기지 않고 결과에서 제외하되, 필터 적용 상태를 분명히 표시한다.

#### 모바일 구조

- 상단: 프로젝트 이름, 상태, 진행률
- 요약: 리드, 일정, 역할별 팀
- 필터: 역할과 상태
- 이슈: 접을 수 있는 세로 카드
- 작업 흐름: 부모 카드 안의 분석 상태, 병렬 현재 작업과 실제 순서·전달 연결

모바일에서는 프로젝트 편집, 이슈 만들기, 팀 작업 추가와 프로젝트 연결 변경을 제공하지 않는다. 이슈 행을 선택해 상세 조회와 지원되는 속성 변경으로 이동할 수 있다.

### 7.8 ISSUE-01 통합 이슈 상세

이슈 콘텐츠와 팀 실행 문맥은 데스크톱과 모바일에서 하나의 공유 상세 본문으로 연다. `/issues/{issueIdentifier}`는 `이슈`가 활성화된 이슈 중심 진입 셸을 사용하고, `/my-issues/{teamWorkIdentifier}`는 7.2.1의 `내 작업` 진입 셸을 사용한다. 두 진입 모드가 설명·첨부파일·댓글·전달·활동 또는 상세 컴포넌트를 복제해서는 안 된다.

#### 데스크톱 구조

| 영역 | 내용 |
| --- | --- |
| 상단 이동 | 이전 화면 이름과 뒤로 가기, 이슈 표시 ID, 더보기 |
| 제목 | 편집 가능한 이슈 제목과 저장 상태 |
| 실행 문맥 | 선택된 팀 작업의 팀·역할·상태·담당자·현재 주요 행동과 다른 작업 전환기 |
| 상세 탭 | `업무`, `전달`, `활동`. 기본 선택은 `업무` |
| 왼쪽 탭 패널 | 공통 이슈 콘텐츠, 선택된 팀 실행 정보, 관계 또는 시스템 이력 |
| 오른쪽 속성 패널 | 이슈 속성과 선택된 팀 작업 속성을 출처별 그룹으로 표시 |

이슈 헤더는 표시 ID, 제목, 프로젝트, 우선순위, 진행률과 자동 계산 상태만 표시한다. 선택된 팀 작업 본문은 팀·역할, 담당자, 현재 상태에 맞는 주요 행동, 보조 상태 메뉴와 작업 노트만 한 번 표시한다. 표시 ID와 상태를 속성 행·배지·제목으로 반복하지 않는다.

#### 모바일 구조

1. 뒤로 가기, 이슈 표시 ID, 더보기
2. 이슈 제목
3. 현재 팀 작업 선택기 또는 이슈 전체 문맥
4. `업무`, `전달`, `활동` 탭
5. 현재 행동과 선택한 탭 패널
6. 이슈·팀 작업 속성 drawer

모바일에서도 이슈 설명·첨부파일·댓글과 선택된 팀 작업 상태·담당자를 같은 화면에서 확인한다. 팀 작업 전환 시 이슈 콘텐츠를 다시 로딩 화면으로 바꾸지 않는다.

#### 탭과 직접 진입

- 이슈 중심 정본 경로는 `/issues/{issueIdentifier}?tab={tab}&work={teamWorkIdentifier}`다.
- 내 작업 진입 정본 경로는 `/my-issues/{teamWorkIdentifier}?tab={tab}`이며 경로 자체가 현재 팀 작업을 선택한다.
- 내 작업 목록은 내 작업 진입 경로를, 이슈·프로젝트·팀·알림은 출발 화면의 책임에 맞는 이슈 중심 경로를 생성한다.
- 기존 `/issues/{teamWorkIdentifier}`는 상위 이슈 정본 경로로 이동한다.
- 잘못된 팀 작업 값은 이슈 전체 문맥으로 복구하고 같은 이슈에 속하지 않은 값은 오류로 처리한다.
- 탭·팀 작업 전환은 이전 목록의 필터·스크롤과 이슈 콘텐츠 편집 상태를 불필요하게 초기화하지 않는다.
- 댓글 anchor는 `업무`, 전달 anchor는 관련 팀 작업을 선택한 `업무`를 열고 해당 전달을 확장한다.

#### 업무 탭

1. 선택된 팀 작업의 현재 행동·상태·담당자
2. 최신 보낸 전달 또는 받은 전달 전체
3. 이슈 설명과 공통 첨부파일
4. 선택된 팀 작업의 작업 노트
5. 이슈 공통 댓글과 작성기

팀 작업이 선택되지 않으면 1·2번 대신 현재 작업 요약과 `작업 시작`을 표시한다. 이슈 설명·첨부파일·댓글은 어떤 팀 작업을 선택해도 같은 원본을 유지한다. 작업 노트는 Compact Lexical 편집기로 Markdown을 저장하며 파일·본문 이미지·멘션 도구를 표시하지 않는다.

팀 작업 실행 영역은 상태·담당자 아래에 한 개의 주요 버튼을 둔다. `시작 전`에서는 `작업 시작`, `진행 중`에서는 `완료`, 완료 뒤에는 같은 위치의 비활성 `완료됨`으로 바뀐다. 예외 상태 변경은 `상태 변경` 보조 메뉴에서 실제 워크플로 상태명을 표시한다.

#### 전달 탭

이슈 전체 전달 이력을 시간순으로 표시한다. 카드에는 최초 전달 또는 추가 전달, 작성자·시각, 원본 백엔드 작업, 대상 프론트 작업, 새 전달 상태와 렌더링된 전체 Markdown을 표시한다. 카드는 같은 화면에서 접고 펼치며, 별도 백엔드 작업 상세나 관계 탭으로 이동하지 않는다.

#### 작업 전달

선택된 백엔드 작업은 업무 탭에 `보낸 전달`과 전달 작성, 전달 탭에 전체 이력을 표시한다. 전달받은 프론트 작업은 업무 탭에 `받은 전달`의 전체 원문, 전달 탭에 전체 이력을 표시한다. 전달 본문과 파일은 팀 작업 콘텐츠로 복제하지 않는다.

| 상태 | 영역 구성 |
| --- | --- |
| 백엔드 완료에서 전달 선택 | 대상 역할, 작업 전달 설명, `프론트에 전달 후 완료` |
| 최초 전달 있음 | 작성자·시각, 변경 요약, API 명세 링크, 나머지 Markdown 본문 |
| 완료 후 변경 | 기존 전달 목록, `추가 전달 작성` |
| 전달받은 프론트 작업 | 원본 백엔드 작업, 최초 전달과 추가 전달 전체 내용 |
| 모바일 | 팀 작업 선택기와 전달 카드의 접기·펼치기를 제공 |

#### 백엔드 작업 완료 방식

백엔드 작업의 `완료`를 선택하면 프로젝트 역할과 무관하게 완료 모달을 연다. 첫 화면에서 완료 방식을 고르고 전달을 선택한 경우에만 입력 영역을 펼친다.

1. `이 작업만 완료`, `프론트에 전달 후 완료` 선택
2. 전달 선택 시 변경 요약, API 명세 링크, 사용 가능 환경, API 변경, 요청·응답, 오류·권한, 프론트 주의사항 템플릿
3. 전달 선택 시 프로젝트의 웹·앱 대상 역할 선택. 한 역할이면 읽기 전용, 두 역할이면 모두 기본 선택하고 최소 한 개 유지
4. 취소, 선택에 따라 `완료` 또는 `프론트에 전달 후 완료`

취소하면 기존 상태를 유지한다. 저장이 실패하면 모달, 완료 방식, 입력 내용과 대상 역할을 유지하고 최신 역할·작업 범위를 다시 조회한다. 전달 성공이면 생성하거나 재사용한 프론트 작업 링크와 갱신된 상위 진행률을 표시한다.

#### 활동 탭

이슈와 모든 팀 작업의 상태·담당자·우선순위·라벨, 작업 노트, 최초·추가 전달과 휴지통 변경을 하나의 압축 타임라인으로 표시한다. 관련 팀 작업 표시 ID를 선택적으로 포함하고 댓글·전달 원문을 반복하지 않는다.

### 7.9 설명, 첨부파일, 댓글과 활동

#### Lexical 설명 편집기

설명 편집기는 이슈 콘텐츠에만 사용한다. 팀 작업을 전환해도 같은 이슈 설명을 표시하며 팀 작업 작업 메모에는 이 편집기를 사용하지 않는다.

| 상태 | 구성 |
| --- | --- |
| 조회 | 렌더링된 Markdown 본문, 편집 |
| 편집 | 편집·미리보기 탭, 핵심 서식 툴바, WYSIWYG 본문, 저장, 취소 |
| 이미지 업로드 중 | 삽입 위치 미리보기, 파일명, 진행 상태, 저장 비활성 |
| 이미지 업로드 실패 | 미리보기 유지, 실패 이유, 다시 시도, 제거 |
| 빈 설명 | 설명 목적 안내, 설명 추가 |
| 저장 실패 | 입력 유지, 다시 시도 |

이미지 도구는 툴바의 이미지 선택과 Command/Ctrl+V 붙여넣기를 함께 제공한다. 클립보드 이미지가 크면 최적화 중 상태를 먼저 표시하고 업로드를 시작한다. 원격 웹 이미지 URL만 붙여넣은 경우 자동으로 가져오지 않는다.

#### 이슈 첨부파일

| 상태 | 구성 |
| --- | --- |
| 비어 있음 | 파일을 첨부해 작업 자료를 공유할 수 있다는 한 줄 안내, 파일 선택 |
| 업로드 중 | 파일명, 크기, 진행률, 취소 또는 제거 |
| 완료 | 파일 아이콘 또는 이미지 썸네일, 파일명, 형식, 크기, 업로더·시각, 열기·다운로드·제거 |
| 실패 | 파일명, 실패 이유, 다시 시도, 제거 |

- 파일 하나는 최대 25MB이며 여러 파일은 업로드 순서대로 표시한다. 별도 수동 정렬은 제공하지 않는다.
- 검증된 이미지만 썸네일과 화면 열기를 제공하고 그 밖의 형식은 파일 아이콘과 다운로드를 사용한다.
- 긴 파일명은 중간을 생략하되 전체 이름을 확인할 수 있게 한다.
- 모바일에서는 한 열 파일 목록으로 바꾸고 업로드·다운로드·제거를 동일하게 제공한다.
- 일반 첨부는 이슈에만 연결하며 팀 작업 전용 첨부 영역을 만들지 않는다.

#### 업무 탭의 댓글

- 실제 댓글만 작성 시각 순으로 표시하고 시스템 활동을 사이에 넣지 않는다.
- 댓글은 이슈에 저장하고 선택된 팀 작업에서 작성한 경우 팀 작업 문맥 배지를 선택적으로 표시한다.
- 댓글은 작성자, 시각, 수정 여부와 본문을 표시한다.
- 삭제된 댓글은 원래 위치에 `삭제된 댓글입니다`를 남긴다.
- 알림에서 진입한 대상은 상단 간격을 확보해 스크롤하고 일시적으로 강조한다.
- 댓글 입력기는 실제 댓글 목록 바로 아래에 두고 Lexical 편집·미리보기, 멘션 자동완성과 저장을 제공한다.

#### 활동 탭의 이력

- 활동은 아이콘, 행위자, 변경 내용과 시각을 한 줄 중심으로 표시한다.
- 같은 저장에서 생긴 전달·완료·후행 작업·관계 이벤트는 화면에서 하나의 대표 활동으로 묶을 수 있다.
- 작업 전달은 요약과 `전달 내용 보기`만 표시하고 Markdown 원문 카드를 반복하지 않는다.

### 7.10 WORK-02 알림함

#### 데스크톱 구조

| 순서 | 영역 | 내용 |
| ---: | --- | --- |
| 1 | 본문 헤더 | `알림함`, 읽지 않은 수, 모두 읽음 |
| 2 | 필터 | 읽지 않음, 모든 알림 |
| 3 | 목록 | 읽음 표시, 유형, 행위자, 이슈, 변경 요약, 발생 시각 |

읽지 않은 항목은 색상 외에 표시점과 보조 기술 상태로 구분한다. 항목 전체 선택 시 `업무` 탭의 관련 댓글로 이동하거나, 전달 알림이면 전달받은 프론트 작업의 `업무`, 상위 이슈의 `연결`, 원본 백엔드 작업의 `연결` 순서로 사용 가능한 전달 앵커를 열고 읽음 처리한다. 개별 읽음 전환은 항목의 보조 동작으로 제공한다.

#### 모바일 구조

상단에 알림함과 모두 읽음을 두고, 필터는 두 개의 탭으로 표시한다. 각 알림은 유형과 읽음 표시, 이슈 제목, 변경 요약, 행위자·시각 순서의 카드형 행으로 사용한다.

### 7.11 SEARCH-01 검색 결과

검색은 5.2의 전역 검색 오버레이를 사용한다. 별도 전체 페이지를 만들지 않는다. 모바일 상단 검색을 선택하면 전체 화면 레이어로 열고 닫으면 이전 화면으로 돌아간다.

## 8. 설정 화면

설정은 데스크톱 전용이다. 업무 주 탐색은 유지하고 본문 안에 설정 보조 탐색을 둔다.

### 8.1 설정 공통 구조

설정에 진입해도 기존 업무 주 탐색과 사용자가 선택한 펼침·축소 상태를 유지한다. 설정 메뉴는 주 탐색을 교체하지 않고 본문 안의 보조 탐색으로 제공한다.

| 영역 | 내용 |
| --- | --- |
| 설정 보조 탐색 | 멤버, 팀, 워크플로, 라벨, 데이터 내보내기, 휴지통 |
| 설정 헤더 | 화면 제목, 설명, 화면별 주요 동작 |
| 설정 본문 | 목록 또는 폼 |

관리자가 아닌 사용자는 설정 진입점을 보지 않는다. 직접 주소 접근은 권한 부족 화면을 표시한다. 모바일 접근은 설정 항목 목록 대신 데스크톱 사용 안내와 업무 화면 복귀를 제공한다.

### 8.2 SET-01 멤버

| 순서 | 영역 | 내용 |
| ---: | --- | --- |
| 1 | 헤더 | 멤버, 이메일 초대 |
| 2 | 탭 | 활성, 초대 대기, 비활성 |
| 3 | 목록 | 이름·이메일, 역할, 팀, 상태, 최근 활동, 더보기 |

초대 대기 행에는 재초대와 취소, 활성 멤버 행에는 팀 추가와 조건부 비활성화를 제공한다. 할당된 미완료 팀 작업이 있으면 비활성화 실행 버튼 대신 작업 수, 작업 목록, 재할당과 담당 해제 경로를 표시한다. 모든 미완료 작업을 정리한 뒤에만 비활성화 확인을 제공한다.

### 8.3 SET-02 팀

| 순서 | 영역 | 내용 |
| ---: | --- | --- |
| 1 | 헤더 | 팀, 팀 만들기 |
| 2 | 탭 | 활성, 보관됨 |
| 3 | 목록 | 이름, 키, 멤버 수, 미완료 작업 수, 더보기 |

팀 생성·편집은 본문 위의 모달을 사용한다. 팀 키가 잠긴 경우 입력 대신 읽기 전용 값과 첫 이슈 생성 후 변경할 수 없다는 설명을 표시한다. 멤버 제거 또는 보관이 막히면 관련 작업 목록을 모달 안에서 열 수 있게 한다.

### 8.4 SET-03 워크플로

상단에서 팀을 선택하고 본문에서 상태를 시스템 범주별 섹션으로 나눈다.

| 상태 행 | 내용 |
| --- | --- |
| 이동 손잡이 | 워크플로 표시 순서 변경 |
| 이름 | 사용자에게 보이는 한국어 상태 이름 |
| 시스템 범주 | 읽기 전용 범주와 설명 |
| 사용 중 수 | 해당 상태의 이슈 수 |
| 더보기 | 이름 변경, 삭제 |

상태 삭제 시 사용 중인 이슈가 있으면 대체 상태 선택 모달을 먼저 연다. 순서 변경은 보드 열 순서일 뿐 이슈의 수동 정렬이 아니라는 도움말을 표시한다.

### 8.5 SET-04 라벨

활성과 보관됨 탭, 라벨 만들기, 라벨 목록으로 구성한다. 목록 행은 색상 표본, 이름, 사용 중 수와 더보기를 표시한다. 라벨 생성·편집 모달은 이름과 색상 입력을 제공하며 중복 오류를 이름 입력 가까이에 표시한다.

### 8.6 SET-05 데이터 내보내기

이슈 CSV와 프로젝트 CSV를 별도 카드로 표시한다. 각 카드에는 포함 필드 요약, 민감한 업무 내용 포함 가능성, 마지막 생성 결과와 `내보내기`를 제공한다.

| 상태 | 표시 |
| --- | --- |
| 생성 전 | 형식과 포함 범위 설명 |
| 생성 중 | 진행 상태, 중복 실행 방지 |
| 완료 | 생성 시각, 파일 받기 |
| 실패 | 실패 요약, 다시 시도 |

CSV 가져오기 동작은 표시하지 않는다.

### 8.7 SET-06 휴지통

| 순서 | 영역 | 내용 |
| ---: | --- | --- |
| 1 | 헤더 | 휴지통, 30일 보존 안내 |
| 2 | 필터 | 이슈, 프로젝트 |
| 3 | 목록 | 이름·표시 ID, 삭제한 사용자, 삭제일, 영구 삭제 예정일, 복구 |

복구를 선택하면 원래 팀이나 프로젝트가 보관된 경우의 결과를 먼저 안내한다. 즉시 영구 삭제 동작은 표시하지 않는다.

## 9. 공통 상태 와이어프레임

### 9.1 로딩

- 최초 로딩은 헤더, 필터와 목록의 위치가 움직이지 않도록 같은 높이의 자리 표시를 사용한다.
- 추가 로딩은 기존 항목을 유지하고 목록 하단에 표시한다.
- 속성 변경은 화면 전체 로딩 대신 변경한 값 가까이에 진행 상태를 표시한다.

### 9.2 빈 상태

빈 상태는 화면 중앙 장식보다 목록이 시작될 위치에 배치한다. 제목, 이유 한 줄과 한 개의 주요 행동으로 구성한다. 필터 결과 없음은 생성 행동 대신 필터 초기화를 우선한다.

### 9.3 오류와 권한

| 상황 | 위치 | 행동 |
| --- | --- | --- |
| 저장 실패 | 변경한 필드 또는 행 | 다시 시도 |
| 목록 로딩 실패 | 목록 시작 위치 | 다시 불러오기 |
| 권한 없음 | 업무 셸 본문 | 접근 가능한 이전 화면 |
| 찾을 수 없음 | 업무 셸 본문 | 내 작업으로 이동 |
| 세션 만료 | 로그인 화면 | 로그인 후 원래 주소 복귀 |
| 연결 끊김 | 본문 상단 상태 막대 | 자동 재연결, 수동 다시 시도 |
| 파일 업로드 실패 | 해당 파일 행 또는 편집기 삽입 위치 | 다시 시도, 제거 |
| 전달 범위 충돌 | 전달 모달과 영향 작업 링크 | 본문·대상 역할을 유지하고 최신 작업 흐름 확인 뒤 다시 실행 |

### 9.4 충돌

충돌은 전역 토스트만으로 알리지 않는다. 충돌한 필드에서 서버의 최신 값, 사용자가 입력한 값과 `내 변경 다시 적용`을 표시한다. 여러 필드가 충돌하면 이슈 상세 상단에 요약하고 각 필드로 이동할 수 있게 한다.

| 컴포넌트 | 최소 표시 | 사용 화면 |
| --- | --- | --- |
| 이슈 행 | 이슈 표시 ID, 제목, 상태 | 전역 이슈, 프로젝트 상세 |
| 팀 작업 행 | 작업 코드, 상위 이슈 표시 ID·제목, 팀 상태 | 내 작업, 팀 목록 |

| 전역 이슈 카드 | 표시 ID, 제목, 프로젝트·라벨, 상태·우선순위, 현재 작업·담당 필요, 진행률, 업데이트, 주요 행동 | 전역 이슈 모바일 |

| 역할 배지 | 역할 이름, 연결 팀 | 프로젝트, 이슈 상세 |
| 팀 작업 주요 행동 | `작업 시작`, `완료`, `완료됨`과 현재 상태의 비색상 구분 | 팀 작업 목록과 통합 상세 |
| 진행률 | 완료 수, 전체 수, 비율 | 이슈, 프로젝트 |
| Compact 속성 트리거 | 공통 아이콘·한국어 이름, 28~32px 시각 표면, 40~44px 조작 영역 | 전역 이슈, 내 작업, 팀 목록·보드, 프로젝트 작업 행 |
| Comfortable 속성 트리거 | Compact와 같은 의미 표현, 34~38px 시각 표면, 44px 조작 영역 | 통합 이슈 상세의 이슈·현재 팀 작업 속성 |
| 라벨 칩 | 색점, 최대 두 개 이름, 나머지 `+N`, 불투명 편집 팝오버 | 이슈 관련 목록·보드·상세 |
| 팀 작업 전환기 | 표시 ID, 팀·역할·상태, 현재 선택 | 통합 이슈 상세 |
| 상세 탭 | `업무`, `전달`, `활동`, 현재 선택과 연결된 패널 | 통합 이슈 상세 |
| 사용자 표시 | 프로필 사진 또는 첫 글자 아바타, 이름, 보조 이메일 또는 상태 | 사용자 메뉴, 담당자 선택, 멤버 설정, 댓글·활동 |
| Lexical 편집기 | WYSIWYG 편집·Markdown 미리보기, 이미지 선택·붙여넣기, 저장·취소 | 설명, 댓글, 작업 전달 |
| 파일 업로드 항목 | 파일명, 크기, 진행·성공·실패 상태, 다시 시도·제거 | 전역 이슈 만들기, 이슈 상세, 프로필 설정 |
| 첨부파일 항목 | 파일명, 형식, 크기, 업로더·시각, 다운로드 | 이슈 상세 |
| 활동 항목 | 행위자, 변경 내용, 시각 | 이슈 상세 |

같은 속성은 목록, 보드와 상세에서 같은 이름과 선택 순서를 사용한다. 상태와 우선순위는 색상만으로 구분하지 않는다.

## 11. 키보드와 접근성 반영

- C는 전역 이슈 만들기, /는 검색을 열고 첫 입력에 포커스한다.
- Escape는 가장 최근에 연 선택기 또는 모달 하나만 닫는다.
- Compact 필터·속성 팝오버와 Comfortable 속성 선택기를 Escape 또는 키보드 선택으로 닫으면 열었던 호출부로 포커스를 복원한다. 바깥의 다른 조작 대상을 선택한 경우에는 그 대상의 포커스를 유지한다.
- Command/Ctrl + Enter는 현재 활성화된 생성 또는 댓글 저장에만 적용한다.
- 상세 탭에 포커스가 있으면 좌우 방향키로 이동하고 활성 탭을 `aria-selected`로 전달한다.
- Command/Ctrl+V로 이미지 파일을 붙여넣으면 편집기 삽입 위치에 업로드 상태를 만들고 완료 사실을 보조 기술에 알린다.
- 텍스트 입력 중에는 단일 문자 단축키를 실행하지 않는다.
- 모달을 닫으면 열었던 버튼으로 포커스를 복원한다.
- 표 형태 목록은 키보드로 행과 편집 가능한 셀을 구분해 이동할 수 있어야 한다.
- 전역 이슈 빠른 필터는 현재 선택과 결과 수를 보조 기술에 전달하고 검토·담당 필요 상태를 색상만으로 표현하지 않는다.
- 배정 시트와 모달은 역할, 담당자, 오류와 저장 동작 순으로 읽히며 실패 시 선택값을 유지하고 첫 오류로 포커스를 이동한다.
- 팀 작업 보드의 상태 변경은 드래그 외에 상태 선택기를 제공한다. 이슈의 자동 상태에는 드래그나 일반 선택기를 제공하지 않는다.
- 읽지 않음, 현재 위치, 저장 중, 시작 역할 선택과 현재 주요 행동을 보조 기술에 전달한다.
- 저장 중인 속성은 현재 값과 포커스 가능성을 유지하되 재실행을 막고 `aria-busy` 또는 가까운 상태 문구로 진행을 전달한다. 실패하면 이전 값을 복구하고 같은 행 가까이에 다시 시도를 제공한다.
- 탭과 패널은 `tablist`, `tab`, `tabpanel` 관계와 고유 ID로 연결하고 전환 뒤 패널 제목 또는 첫 의미 있는 요소로 읽기 맥락을 제공한다.
- URL로 특정 탭에 직접 들어온 경우에는 포커스를 강제로 이동하지 않는다.
- 모바일 하단 탐색은 아이콘과 한국어 이름을 함께 표시한다.

## 12. 화면 간 이동과 맥락 복원

| 출발 | 도착 | 돌아왔을 때 복원 |
| --- | --- | --- |
| 전역 이슈 | 이슈 상세 | 빠른 필터, 세부 필터, 정렬, 현재 페이지, 스크롤 |
| 내 작업 | 내 작업 상세 | 필터, 정렬, 스크롤과 `내 작업` 활성 상태 |
| 팀 목록 | 선택된 팀 작업 문맥의 이슈 상세 | 팀, 탭, 보기, 필터, 정렬, 스크롤 |
| 팀 보드 | 선택된 팀 작업 문맥의 이슈 상세 | 팀, 필터, 정렬, 가로·세로 위치 |
| 프로젝트 상세 | 이슈 상세 | 역할 필터, 펼침 상태, 스크롤 |
| 알림함 | 관련 이슈 위치 | 알림 필터와 스크롤 |
| 검색 | 이슈 상세 | 검색어와 선택 결과 위치 |

브라우저 뒤로 가기와 화면 안 뒤로 가기는 같은 복원 결과를 가져야 한다. 직접 주소로 내 작업 상세에 들어온 경우에는 `내 작업으로 이동`, 이슈 상세에 들어온 경우에는 `이슈 목록으로 이동`을 안전한 기본 경로로 제공한다.

## 13. 구현용 화면 우선순위

와이어프레임과 컴포넌트 구현은 다음 순서로 진행한다.

1. 데스크톱·모바일 공통 셸
2. 인증과 초기 설정
3. 전역 이슈 만들기와 공통 속성 선택기
4. 전역 이슈와 내 작업, 팀 작업 목록 및 배정 오버레이
5. 이슈 상세 공통 구조와 댓글
6. 프로젝트 목록·생성·상세와 이슈 작업 흐름
7. 전달 이력과 전달 편집기
8. 팀 상태 보드
9. 알림함과 전역 검색
10. 프로필 설정, 설정 화면과 공통 오류 상태

이 순서는 제품 출시 우선순위가 아니라 공통 구조를 먼저 검증하기 위한 디자인·프론트엔드 작업 순서다.
