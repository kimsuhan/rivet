## 1. 문서 목적

이 문서는 Linear 벤치마킹 결과를 Rivet의 첫 제품 개발에 사용할 수 있는 요구사항으로 전환한다. MVP에서 반드시 해결할 사용자 문제, 기능 범위, 핵심 정책, 품질 기준과 출시 조건을 정의하며, 상세 기술 구조와 화면별 구현 방식은 이 문서의 범위에 포함하지 않는다.

Rivet MVP는 Linear의 모든 기능을 복제하는 제품이 아니다. **소규모 소프트웨어 팀이 이슈를 빠르게 만들고, 분류하고, 담당하고, 실행하고, 완료하는 일상 작업 루프를 끊김 없이 운영하는 제품**을 목표로 한다.

## 2. 제품 요약

### 2.1 문제 정의

소규모 소프트웨어 팀은 매일 다음 작업을 반복한다.

- 해야 할 일을 빠르게 기록한다.
- 어느 팀의 누가 처리할지 정한다.
- 우선순위와 진행 상태를 공유한다.
- 댓글과 변경 기록으로 맥락을 남긴다.
- 자신에게 필요한 작업과 알림을 다시 찾는다.

이 과정이 여러 화면과 도구에 흩어지거나 변경 결과가 즉시 일치하지 않으면 팀은 이슈 트래커를 신뢰하지 못하고 문서, 메신저, 구두 전달로 돌아간다. 기능 수가 많더라도 생성과 상태 변경이 느리고 데이터가 화면마다 다르면 일상 실행 도구로 정착하기 어렵다.

### 2.2 제품 비전

Rivet은 1~10명 소프트웨어 팀을 위한 빠르고 의견 있는 실행 시스템이다. 사용자가 설정을 설계하는 데 시간을 쓰기보다, 제한된 기본값 안에서 작업을 즉시 기록하고 실행 상태를 신뢰할 수 있게 한다. **기능을 줄여 기존 도구보다 빠르고 단순하게 사용하는 것**을 첫 번째 차별점으로 삼는다.

### 2.3 핵심 가치 제안

1. **빠른 입력:** 어느 주요 화면에서든 최소한의 조작으로 이슈를 만든다.
2. **명확한 책임:** 이슈마다 한 명의 담당자와 제한된 우선순위를 사용한다.
3. **일관된 상태:** 목록, 보드, 상세, 개인 작업함에서 같은 이슈가 같은 상태로 보인다.
4. **낮은 운영 부담:** 복잡한 설정과 자동화 없이도 팀이 바로 사용할 수 있다.

### 2.4 초기 제품 가정

다음은 MVP의 확정된 대상과 아직 사용으로 검증해야 할 가정이다.

- 첫 고객은 1~10명 규모의 소프트웨어 개발 팀이다.
- 제품의 기본 표시 언어는 한국어다.
- MVP에서는 하나의 계정이 하나의 워크스페이스만 생성하거나 참여한다.
- 데스크톱 웹만으로 초기 핵심 업무를 처리할 수 있다.
- 팀은 고급 리포팅보다 빠른 입력, 검색, 상태 공유를 더 중요하게 여긴다.
- 워크스페이스 내부 데이터를 모든 멤버가 볼 수 있는 공개 팀 모델로 시작할 수 있다.
- 팀은 여러 이슈를 결과물 단위로 묶고 진행 상태를 확인할 수 있는 최소 프로젝트 기능이 필요하다.
- 사이클과 GitHub 연동 없이도 핵심 이슈·프로젝트 루프의 가치를 먼저 검증할 수 있다.

## 3. 목표와 비목표

### 3.1 MVP 목표

- 사용자가 별도 안내 없이 워크스페이스와 팀을 만들고 동료를 초대할 수 있다.
- 사용자가 어느 주요 화면에서든 이슈를 빠르게 생성하고 핵심 속성을 변경할 수 있다.
- 팀이 목록과 상태 보드에서 같은 이슈 집합을 일관되게 관리할 수 있다.
- 사용자가 프로젝트의 백엔드·웹 프론트·앱 프론트 역할에 필요한 팀만 연결하고 진행률을 확인할 수 있다.
- 하나의 기능을 팀별 실행 작업으로 나누고 작업 순서와 API 작업 전달을 이해하기 쉽게 관리할 수 있다.
- 사용자가 자신에게 할당된 미완료 작업과 자신이 받아야 할 알림을 놓치지 않는다.
- 댓글, 멘션, 활동 기록을 통해 이슈의 현재 맥락과 변경 이력을 이해할 수 있다.
- 사용자가 이슈에 업무 파일을 첨부하고 설명·댓글·작업 전달에 이미지를 붙여넣어 맥락을 한곳에 남길 수 있다.
- 1~10명 팀이 기존 이슈 트래커를 대체해 4주 이상 실제 업무에 사용할 수 있는 안정성을 확보한다.

### 3.2 MVP 비목표

- Linear의 시각 요소, 브랜드, 카피 또는 화면을 그대로 복제하지 않는다.
- 프로젝트의 마일스톤, 타임라인, 상태 업데이트, 의존성과 문서 기능은 구현하지 않는다.
- 사이클, 승인 단계가 있는 별도 트리아지 시스템과 저장된 보기를 구현하지 않는다. 워크스페이스 내부 이슈의 접수·검토·분류 업무함은 MVP에 포함한다.
- GitHub, Slack, Sentry 등 외부 도구와 연동하지 않는다.
- 비공개 팀, 게스트, 이슈 단위 공유와 세밀한 역할 권한을 제공하지 않는다.
- 한 계정의 다중 워크스페이스 생성·참여와 워크스페이스 전환을 제공하지 않는다.
- 네이티브 모바일 앱과 모바일 푸시를 제공하지 않는다. 업무 이메일 알림은 MVP뿐 아니라 Alpha, Beta와 Public에서도 제공하지 않으며 이메일은 계정 인증·초대·비밀번호 재설정에만 사용한다.
- 완전한 오프라인 편집이나 협업 문서 편집을 지원하지 않는다.
- AI 에이전트, 분석 대시보드, 고객 요청, 릴리스 관리, Enterprise 보안 기능을 제공하지 않는다.
- 공개 API, OAuth 앱, 외부 웹훅을 제품 계약으로 제공하지 않는다.

## 4. 대상 사용자

### 4.1 주요 사용자: 팀 멤버

개발자, 디자이너, 제품 담당자 등 팀의 실행 업무를 직접 처리하는 사용자다.

**주요 요구**

- 생각의 흐름을 끊지 않고 이슈를 생성하고 싶다.
- 지금 자신이 해야 할 일과 팀이 진행 중인 일을 빠르게 보고 싶다.
- 여러 이슈가 어떤 결과물을 위해 진행되는지 프로젝트 단위로 확인하고 싶다.
- 상세 화면을 반복해서 열지 않고 상태, 담당자, 우선순위를 바꾸고 싶다.
- 이슈에 대한 결정과 변경 이유를 나중에 다시 확인하고 싶다.

### 4.2 보조 사용자: 워크스페이스 관리자

워크스페이스를 만들고 멤버, 팀, 워크플로와 라벨을 관리하는 사용자다. 초기에는 별도의 전담 관리자가 아니라 팀 리드나 제품 담당자가 이 역할을 겸한다고 가정한다.

**주요 요구**

- 팀이 업무를 시작하는 데 필요한 설정을 짧은 시간 안에 끝내고 싶다.
- 잘못된 상태나 라벨 변경으로 기존 이슈가 손상되지 않게 관리하고 싶다.
- 문제가 생겼을 때 팀의 데이터를 회수할 수 있어야 한다.

## 5. 핵심 사용자 여정

### 5.1 첫 사용과 팀 구성

일반 가입: `표시 이름·이메일·비밀번호 입력 → 이메일 인증 → 워크스페이스 생성 → 기본 팀 생성 → 선택적 멤버 초대 → 첫 이슈 생성`

초대 가입: `초대 링크 → 초대 진행 상태 생성과 이메일 확인 → 회원가입 또는 로그인 → 자동 초대 복귀 → 초대 수락 → 내 작업`

워크스페이스가 없는 계정만 새 워크스페이스를 만들 수 있다. 유효한 초대 진행 상태가 있는 계정은 워크스페이스 생성보다 초대 수락을 먼저 안내한다. 이미 워크스페이스에 참여한 계정은 다른 워크스페이스를 만들거나 초대를 수락할 수 없으며, 초대를 닫고 현재 워크스페이스로 돌아가거나 다른 계정으로 로그인할 수 있다.

완료 조건은 관리자가 기본 팀의 워크플로를 확인하고 첫 이슈를 생성하는 것이다. 초기 멤버 초대는 건너뛸 수 있다.

### 5.2 일상 실행

`이슈 만들기 → 처음 작업할 팀 선택 또는 분석 → 설명·댓글로 명확화 → 필요한 팀 작업 시작 → 상태 변경 → 완료`

이 흐름은 Rivet의 최우선 사용자 여정이다. 각 단계는 목록, 보드 또는 상세 화면 중 적절한 위치에서 수행할 수 있어야 하며, 변경 결과는 다른 화면에도 같은 의미로 반영되어야 한다.

### 5.3 프로젝트 계획과 진행 확인

`프로젝트 생성 → 역할별 담당 팀·리드·일정 설정 → 이슈 만들기 → 현재·완료·예상 작업 흐름 확인 → 진행률 확인 → 프로젝트 상태 변경`

프로젝트는 여러 기능과 팀 작업을 하나의 결과물로 묶는 최소 계획 단위다. `백엔드`, `웹 프론트`, `앱 프론트` 역할 중 필요한 역할에만 팀을 연결한다. 프로젝트 상태는 팀 작업 완료율과 별개로 사용자가 직접 결정한다.

### 5.4 팀 간 작업 전달

`이슈 만들기에서 백엔드 선택 → 전달 대상 역할 선택 → 전달하고 완료 → 후행 웹·앱 작업 생성 또는 재사용 → 작업 가능 안내·알림 → 웹·앱 작업 시작`

팀 이름을 상태에 넣지 않는다. 작업 상태는 공통 흐름만 표현하고, 실행 책임은 팀, 웹·앱 구분은 프로젝트 역할, 선행·후행 관계는 사용자 화면에서 `작업 순서`로 표현한다.

### 5.5 전역 이슈 검토와 개인 업무 확인

`이슈 진입 → 검토 필요·담당 필요 항목 분류 → 역할별 작업 시작·배정 → 내 작업 진입 → 내 작업 문맥을 유지한 상세에서 선택된 팀 작업 처리`

전역 `이슈`는 삭제되지 않은 모든 이슈의 접수·검토·분류와 팀 작업 배정을 담당한다. `내 작업`은 자신에게 할당된 미완료 팀 작업을 보여 주며, 행을 열면 `내 작업` 활성 상태와 복귀 동선을 유지하는 상세 경로에서 해당 작업을 처리한다. 이 상세는 별도 팀 작업 콘텐츠 문서가 아니라 상위 이슈의 설명·첨부파일·댓글과 선택된 팀 작업 실행 속성을 결합한 공유 통합 상세다.

### 5.6 비동기 협업

`이슈에 댓글·멘션 작성 → 수신자 알림함 알림 → 이슈 맥락 확인 → 응답 또는 상태 변경 → 읽음 처리`

동일한 사건으로 중복 알림이 생기지 않아야 하며, 자신의 행동에 대한 알림은 받지 않는다.

### 5.7 작업 재탐색

`전역 검색 실행 → 표시 ID 또는 제목 입력 → 결과 선택 → 이슈 상세 확인`

정확한 이슈 표시 ID는 가장 높은 우선순위로 검색되어야 한다.

## 6. 정보 구조

MVP의 1차 탐색은 다음 화면으로 구성한다.

| 화면 | 목적 | 주요 동작 |
| --- | --- | --- |
| 이슈 | 워크스페이스 전체 이슈의 접수·검토·분류와 배정 | 빠른 필터, 세부 필터, 작업 시작, 내가 맡기, 담당자 지정, 완료 확인 |
| 내 작업 | 나에게 할당된 미완료 팀 작업 확인 | 정렬, 필터, 인라인 속성 변경, 상세 열기 |
| 알림함 | 할당·멘션·댓글·작업 전달 알림 확인 | 읽음 처리, 관련 이슈 이동 |
| 팀 작업 | 팀의 전체·진행·백로그 작업 관리 | 목록·보드 전환, 필터, 정렬, 이슈 만들기 |
| 프로젝트 | 역할별 팀과 이슈·팀 작업을 결과물 단위로 관리 | 프로젝트 생성, 이슈 만들기, 역할별 팀·상태·리드·일정과 작업 흐름 관리, 진행률 확인 |
| 이슈 상세 | 이슈 콘텐츠와 선택된 팀 작업의 실행 맥락 확인 | 제목·설명·첨부파일, 팀 작업 상태·담당자·메모, 작업 흐름·전달, 댓글·활동 확인 |
| 검색 | 이슈 제목과 이슈·팀 작업 표시 ID 재탐색 | 정확한 표시 ID 또는 이슈 제목 검색, 통합 상세 이동 |
| 사용자 메뉴 | 개인 계정 정보 관리 | 프로필 사진 등록·변경·삭제, 로그아웃 |
| 설정 | 워크스페이스 운영 설정 | 멤버, 팀, 상태, 라벨 관리, 데이터 내보내기 |

전역 `이슈 만들기`와 검색은 모든 주요 화면에서 접근할 수 있어야 한다. 별도 홈 대시보드는 MVP에 만들지 않으며, 로그인 후 기본 진입점은 `/my-issues`의 `내 작업`으로 한다.

## 7. 기능 요구사항

요구사항의 우선순위는 다음과 같다.

- **Must:** MVP 출시 조건이다.
- **Should:** MVP에 포함하는 것을 원칙으로 하지만 일정 위험이 발생하면 제품 책임자 승인 후 후속 배포할 수 있다.
- **Could:** 설계를 막지 않는 범위에서만 구현하며 출시 조건이 아니다.

### 7.1 인증, 워크스페이스와 멤버

| ID | 우선순위 | 요구사항 | 인수 조건 |
| --- | --- | --- | --- |
| ORG-001 | Must | 사용자는 표시 이름, 이메일과 비밀번호로 회원가입할 수 있다. | 공백이 아닌 1~50자의 표시 이름, 유효한 이메일과 비밀번호 정책을 통과한 비밀번호를 입력하면 계정 존재 여부와 관계없이 같은 접수 안내를 표시한다. 신규 이메일만 계정과 인증 메일을 만들고 기존 이메일로 중복 계정을 만들지 않으며, 인증 전 계정은 업무 데이터에 접근할 수 없다. 표시 이름은 중복을 허용한다. |
| ORG-002 | Must | 사용자는 가입 이메일의 링크로 이메일 주소를 인증할 수 있다. | 유효 기간 안의 인증 링크를 한 번만 사용할 수 있고 인증 완료 후 로그인할 수 있다. |
| ORG-003 | Must | 이메일 인증을 마친 사용자는 이메일과 비밀번호로 로그인할 수 있다. | 워크스페이스 멤버십이 있으면 해당 단일 워크스페이스로 진입하고, 없으면 워크스페이스 생성 또는 초대 수락 흐름을 따른다. OAuth 로그인은 제공하지 않는다. |
| ORG-004 | Must | 사용자는 이메일을 통해 비밀번호를 재설정할 수 있다. | 재설정 링크는 만료되고 한 번만 사용할 수 있으며 변경 후 기존 재설정 링크는 무효가 된다. |
| ORG-005 | Must | 워크스페이스에 참여하지 않은 사용자는 이름과 고유 슬러그를 가진 워크스페이스 하나를 만들 수 있다. | 생성자는 자동으로 `admin`이 되고 기본 팀 생성 단계로 이동한다. 워크스페이스 멤버십이 생긴 뒤에는 추가 워크스페이스를 만들 수 없다. |
| ORG-006 | Must | 관리자는 이메일 주소로 멤버를 초대할 수 있다. | 초대 수락 전후 상태가 구분되고, 워크스페이스가 없는 사용자가 수락하면 `member`가 된다. 이미 다른 워크스페이스에 참여한 계정은 초대를 수락할 수 없으며 같은 초대를 반복해도 중복 멤버십이 생기지 않는다. |
| ORG-007 | Must | 권한은 `admin`과 `member` 두 역할로 제한한다. | 멤버·팀·워크플로 설정 변경은 `admin`만 가능하고 일반 업무 작업은 두 역할 모두 가능하다. |
| ORG-008 | Should | 관리자는 멤버를 비활성화할 수 있다. | 미완료 팀 작업이 할당되어 있으면 해당 작업을 재할당하거나 담당 해제하기 전에는 비활성화할 수 없다. 비활성 멤버는 로그인과 신규 활동이 제한되며 과거 이슈·댓글의 작성자 정보는 보존된다. |
| ORG-009 | Must | 사용자는 선택적으로 프로필 사진을 등록, 변경하거나 삭제할 수 있다. | 프로필 사진은 JPEG·PNG·WebP 이미지만 허용하고 프론트에서 긴 변을 최대 512px로 축소·압축한 뒤 업로드한다. 사진이 없으면 표시 이름의 첫 글자를 사용하며 변경 결과는 담당자, 멘션, 댓글과 활동의 사용자 표시에 반영된다. |

### 7.2 팀과 워크플로

| ID | 우선순위 | 요구사항 | 인수 조건 |
| --- | --- | --- | --- |
| TEAM-001 | Must | 관리자는 워크스페이스 아래 여러 팀을 만들 수 있다. | 팀은 고유한 이름과 2~5자의 영문 대문자 키를 가지며 모든 워크스페이스 멤버에게 공개된다. |
| TEAM-002 | Must | 관리자는 팀 이름을 변경할 수 있다. | 이름 변경 후 기존 프로젝트·이슈·활동 연결이 유지된다. 이슈가 한 건이라도 생성된 팀의 키는 변경할 수 없다. |
| TEAM-003 | Must | 관리자는 활성 멤버를 팀에 추가하거나 제거할 수 있다. | 한 사용자는 여러 팀에 속할 수 있다. 미완료 작업의 담당자인 멤버는 해당 작업을 재할당하거나 담당 해제하기 전에는 팀에서 제거할 수 없다. |
| TEAM-004 | Must | 팀 생성 시 한국어 기본 워크플로를 제공한다. | `미분류`, `보류`, `할 일`, `진행 중`, `검토`, `완료`, `취소` 상태가 시스템 범주 순서로 바로 생성되며 `미분류`가 초기 기본값이다. |
| TEAM-005 | Must | 관리자는 팀별 상태를 추가하고 이름·순서·기본값을 관리할 수 있다. | 시스템 범주는 `Backlog`, `Unstarted`, `Started`, `Completed`, `Canceled` 순서로 고정한다. 새 상태는 선택한 범주의 끝에 추가하고 상태 순서는 같은 범주 안에서만 변경한다. 생성 후 범주는 바꿀 수 없다. 모든 범주의 상태를 기본값으로 지정할 수 있으며 팀마다 기본값은 정확히 하나다. 이름·순서·기본값 변경 후 기존 이슈의 연결과 현재 상태는 유지된다. |
| TEAM-006 | Must | 시스템 상태 범주는 `Backlog`, `Unstarted`, `Started`, `Completed`, `Canceled`로 고정한다. | 목록 기본 탭, 완료 여부와 진행률은 사용자에게 보이는 상태 이름이 아니라 시스템 범주를 기준으로 동작한다. |
| TEAM-007 | Should | 사용 중인 상태를 삭제할 때 안전장치를 제공한다. | 해당 상태의 이슈가 있으면 대체 상태를 선택하기 전에는 삭제할 수 없다. |
| TEAM-008 | Must | 관리자는 팀을 삭제하는 대신 보관할 수 있다. | 미완료 팀 작업이 있으면 보관할 수 없다. 보관 후 기존 이력은 유지되지만 새 프로젝트 역할과 이슈의 팀 선택 목록에서는 제외된다. |

MVP의 팀은 모두 워크스페이스에 공개한다. 비공개 팀, 팀 계층, 팀별 관리자와 세밀한 팀 권한은 제공하지 않는다.

### 7.3 이슈

| ID | 우선순위 | 요구사항 | 인수 조건 |
| --- | --- | --- | --- |
| ISSUE-001 | Must | 전역·프로젝트·팀 화면의 기본 생성 동작은 유형 선택 없는 `이슈 만들기`다. | 이슈는 제목과 프로젝트를 필수로 가지고 팀·담당자에 직접 귀속되지 않는다. 팀 작업 없이 생성하면 `접수됨`, 시작 역할을 함께 선택하면 생성된 팀 작업을 반영해 `할 일`로 저장된다. |
| ISSUE-002 | Must | 모든 팀 작업은 정확히 하나의 상위 이슈 아래에서만 생성된다. | 상위 이슈 없는 단독 팀 작업과 직접 팀 작업 생성 UI·API가 없으며 운영·조사 업무도 먼저 이슈를 만든다. |
| ISSUE-003 | Must | 이슈와 팀 작업은 불변 내부 ID와 사람이 읽을 수 있는 표시 ID를 별도로 가진다. | 이슈는 `F-{워크스페이스 순번}`, 팀 작업은 `{TEAM_KEY}-{팀 순번}` 형식을 사용하며 동시 생성에도 중복되지 않는다. |
| ISSUE-004 | Must | 이슈는 제목·설명·첨부파일·프로젝트·우선순위·라벨의 유일한 정본이다. | 팀 작업은 이 값을 복사하지 않고 상위 이슈에서 읽으며 담당자는 팀 작업의 활성 팀 멤버 중 최대 한 명이다. |
| ISSUE-005 | Must | 사용자는 이슈의 제목, 설명, 첨부파일과 속성을 수정할 수 있다. | 저장 성공 후 전역 이슈, 팀 목록, 내 작업, 프로젝트와 통합 상세가 같은 이슈 값을 표시한다. |
| ISSUE-006 | Must | 사용자는 팀 작업의 상태·담당자·선택적인 작업 메모와 이슈의 우선순위·라벨을 즉시 변경할 수 있다. | 작업 메모는 10,000자 이하의 Markdown과 활성 멤버 멘션을 지원하고 이미지·일반 파일은 지원하지 않는다. 저장 실패는 변경한 값만 복구한다. |
| ISSUE-007 | Must | 사용자는 이슈를 완료, 일시 중지, 재개, 취소하거나 다시 열 수 있다. | `완료`는 모든 유효 팀 작업이 완료된 `완료 확인`에서만 가능하고 재개·다시 열기는 현재 팀 작업으로 상태를 다시 계산한다. |
| ISSUE-008 | Should | 사용자는 이슈를 삭제할 수 있다. | 삭제 전 확인을 거쳐 휴지통으로 이동하며 팀 작업·관계·전달·댓글과 파일은 같은 이슈 생명주기를 따른다. 삭제된 이슈는 일반 검색과 목록에 나타나지 않는다. |
| ISSUE-009 | Must | 표시 ID 순번은 안정적으로 증가한다. | 삭제된 번호를 재사용하지 않고 워크스페이스에서 같은 표시 ID가 두 이슈에 발급되지 않는다. |
| ISSUE-010 | Must | MVP에서는 팀 작업의 소속 팀을 변경할 수 없다. | 다른 팀으로 넘길 때는 같은 이슈 아래 해당 역할의 새 팀 작업을 만들고 작업 순서 또는 전달 관계를 연결한다. |
| ISSUE-011 | Must | 이슈는 여러 팀 작업을 실행 슬롯으로 가질 수 있다. | 팀 작업은 프로젝트를 별도로 저장하지 않고 상위 이슈 프로젝트의 역할과 담당 팀만 사용한다. |
| ISSUE-012 | Must | 팀 작업 사이에 `작업 순서`를 설정할 수 있다. | `이 작업 전에 완료` 또는 `이 작업 다음에 시작` 방향을 선택할 수 있고 자기 자신이나 순환 관계는 등록할 수 없다. |
| ISSUE-013 | Must | 미완료 선행 작업이 있는 팀 작업에는 구체적인 대기 원인을 표시한다. | `{표시 ID} 완료 후 시작 가능`처럼 선행 작업을 직접 알리고, 모든 선행 작업이 완료되면 `선행 작업 완료`와 `작업 가능`으로 표시한다. |
| ISSUE-014 | Must | 이슈는 직계 팀 작업의 진행률과 자동 요약 상태를 표시한다. | 유효 팀 작업 없음은 `접수됨`, 시작 전 작업만 있으면 `할 일`, 시작·완료 이력이 있고 미완료 작업이 남으면 `진행 중`, 모두 완료되면 `완료 확인`으로 같은 트랜잭션에서 갱신한다. |
| ISSUE-015 | Must | 사용자는 이슈에 여러 파일을 첨부하고 내려받을 수 있다. | 일반 자료는 이슈, 대화 자료는 댓글, 전달 결과물은 작업 전달에 연결한다. 팀 작업 일반 첨부는 제공하지 않으며 파일 하나는 최대 25MB다. |
| ISSUE-016 | Must | 사용자는 이슈마다 처음 작업할 프로젝트 역할을 선택하거나 선택하지 않을 수 있다. | 선택하지 않으면 이슈만 만들고, 단일·복수 역할을 선택하면 해당 팀 작업을 같은 트랜잭션에서 만든다. 동시에 선택한 작업은 병렬이며 하나라도 실패하면 모두 롤백한다. |
| ISSUE-017 | Must | 이슈 상세는 분석·현재·완료·예상·전달 상태를 구분한 `작업 흐름`을 표시한다. | 실제 생성된 미취소 작업만 진행률에 포함하고 작업이 없으면 분석 안내와 `작업 시작`을 제공한다. |
| ISSUE-018 | Must | 사용자는 `/issues`에서 삭제되지 않은 워크스페이스 전체 이슈를 접수·검토·분류한다. | 전체, 검토 필요, 담당 필요, 진행 중, 완료 확인, 완료 필터와 세부 필터를 URL에 보존한다. |
| ISSUE-019 | Must | 사용자는 팀 작업이 없는 이슈에서 프로젝트 역할별 작업과 선택 담당자를 함께 시작할 수 있다. | 기존 미완료 역할 작업은 재사용하고 없는 작업만 만들며 역할·담당자 전체를 활동·구독·Outbox와 한 트랜잭션에서 커밋한다. 담당자는 역할 팀의 활성 멤버만 허용하고 복수 역할을 자동으로 순서화하지 않는다. |
| ISSUE-020 | Must | 활성 팀 멤버는 연결된 프로젝트 역할에서 우리 팀 작업을 시작하거나 자신이 맡을 수 있다. | 서버가 현재 프로젝트 역할과 팀 멤버십을 다시 확인하며, 미할당 작업이 한 건이면 재사용하고 여러 건이면 선택을 요구하며 없으면 새 작업을 만든다. 다른 담당자를 덮어쓰지 않는다. |
| ISSUE-021 | Must | 사용자는 한 이슈의 여러 미할당 팀 작업에 역할별 담당자를 원자적으로 지정할 수 있다. | 모든 작업의 부모·팀·버전·현재 담당자와 후보를 검증한 뒤 전부 반영하거나 전부 롤백하며 오래된 화면이 새 담당자를 덮어쓰지 않는다. |
| ISSUE-022 | Must | 목록 행은 현재 작업 요약과 다음 행동을 한 개만 제공한다. | `내 작업 열기`는 `/my-issues/{teamWorkIdentifier}`에서 `내 작업` 문맥을 유지하며 해당 작업을 선택하고, `완료 확인` 상태의 이슈만 명시적 완료 행동으로 완료한다. |
| ISSUE-023 | Must | 전역 이슈·프로젝트·내 작업에서 연 상세는 하나의 통합 콘텐츠 본문을 사용한다. | `/issues/{issueIdentifier}?work={teamWorkIdentifier}`, `/projects/{projectId}/issues/{issueIdentifier}?work={teamWorkIdentifier}`, `/my-issues/{teamWorkIdentifier}`는 활성 내비게이션·복귀 동선·상단 위계를 구분하되 이슈 설명·첨부파일·댓글과 선택된 팀 작업 상태·담당자·메모·전달을 같은 원본으로 표시한다. |
| ISSUE-024 | Must | 팀 작업 댓글과 활동은 상위 이슈 협업 기록에 통합한다. | 댓글·활동은 이슈에 저장하고 선택적인 팀 작업 문맥을 포함하며 같은 내용을 별도 팀 작업 타임라인에 복제하지 않는다. |

### 7.5 프로젝트

| ID | 우선순위 | 요구사항 | 인수 조건 |
| --- | --- | --- | --- |
| PROJECT-001 | Must | 사용자는 워크스페이스에 프로젝트를 생성할 수 있다. | 이름은 필수이며 설명, 리드, 시작일과 목표일은 선택적으로 입력할 수 있다. 리드는 활성 멤버 중 최대 한 명이며 목표일은 시작일보다 빠를 수 없다. 생성한 프로젝트는 모든 워크스페이스 멤버에게 공개된다. |
| PROJECT-002 | Must | 프로젝트 생성 시 `백엔드 담당 팀`, `웹 프론트 담당 팀`, `앱 프론트 담당 팀`을 각각 선택하거나 비워 둘 수 있다. | 세 역할 중 최소 하나에는 활성 팀을 선택해야 하며 선택하지 않은 역할은 해당 프로젝트의 작업 생성·진행률·알림 대상에서 제외된다. |
| PROJECT-003 | Must | 같은 팀을 프로젝트의 여러 역할에 연결할 수 있다. | 한 팀이 백엔드와 웹 프론트처럼 여러 역할을 담당해도 각 팀 작업에는 어떤 역할의 작업인지 별도로 저장되고 표시된다. |
| PROJECT-004 | Must | 사용자는 프로젝트 생성 후 역할별 담당 팀을 추가하거나 변경할 수 있다. | 해당 역할에 연결된 팀 작업이 있으면 역할을 비우거나 담당 팀을 변경하기 전에 작업을 제거하거나 새 담당 팀의 작업으로 다시 만들어야 한다. |
| PROJECT-005 | Must | 프로젝트 이슈에서 팀 작업을 시작할 때 설정된 프로젝트 역할을 선택한다. | 역할을 선택하면 담당 팀이 자동으로 결정되며 팀 작업은 프로젝트를 별도로 저장하지 않고 상위 이슈를 통해 상속한다. |
| PROJECT-006 | Must | 이슈는 정확히 한 프로젝트에 속한다. | 모든 팀 작업은 상위 이슈 프로젝트를 사용하며 상위 이슈 없는 팀 작업은 만들 수 없다. |
| PROJECT-007 | Must | 프로젝트는 `계획`, `진행 중`, `완료`, `취소` 중 하나의 상태를 가진다. | 기본 상태는 `계획`이며 사용자가 직접 변경한다. 모든 팀 작업이 완료되어도 프로젝트 상태는 자동으로 바뀌지 않는다. |
| PROJECT-008 | Must | 프로젝트 목록에서 역할별 팀, 상태, 리드, 목표일과 진행률을 확인할 수 있다. | 진행률의 분모는 취소된 작업을 제외한 연결 팀 작업 수, 분자는 그중 완료된 팀 작업 수로 계산하며 분모가 0이면 0%로 표시한다. 이슈 자체는 작업 수로 중복 계산하지 않는다. |
| PROJECT-009 | Must | 프로젝트 상세의 주요 동작은 `이슈 만들기`이며 이슈와 역할별 작업 흐름을 조회·관리할 수 있다. | 현재·완료 작업과 생성 전 예상 역할을 계층으로 표시하고 추가 팀 작업은 상위 이슈의 작업 흐름에서만 만든다. |
| PROJECT-010 | Should | 사용자는 프로젝트를 보관할 수 있다. | 보관된 프로젝트는 기본 목록과 새 이슈의 프로젝트 선택 목록에서 제외되지만 연결된 이슈, 역할별 팀과 프로젝트 기록은 보존된다. |
| PROJECT-011 | Should | 사용자는 비어 있는 프로젝트를 삭제할 수 있다. | 연결된 이슈가 있으면 삭제 대신 보관하도록 안내한다. 삭제한 프로젝트는 휴지통으로 이동한다. |

MVP에서는 프로젝트 마일스톤, 타임라인, 주기적 상태 업데이트, 프로젝트 간 의존성, 프로젝트 문서와 자동 상태 변경을 제공하지 않는다.
프로젝트 역할은 `백엔드`, `웹 프론트`, `앱 프론트`로 고정하며 사용자 정의 역할은 후속 범위로 둔다.

### 7.6 댓글, 멘션과 활동 기록

| ID | 우선순위 | 요구사항 | 인수 조건 |
| --- | --- | --- | --- |
| COLLAB-001 | Must | 사용자는 이슈 설명·작업 메모·작업 전달·댓글을 Lexical 기반 WYSIWYG 편집기로 작성하고 저장 결과를 미리 볼 수 있다. | 편집과 미리보기 전환, Markdown 단축 입력을 제공하며 서버에는 Markdown 문자열로 저장한다. Markdown 원문 편집 모드는 제공하지 않고 저장 결과는 미리보기와 같은 의미로 렌더링된다. 작성한 댓글에는 작성자와 작성 시간이 표시된다. |
| COLLAB-002 | Must | 사용자는 모든 Markdown 편집기에서 `@`를 입력해 활성 멤버를 검색하고 멘션할 수 있다. | 유효한 멤버를 키보드 또는 포인터로 선택해 저장하면 수신자에게 한 건의 `MENTIONED` 알림이 생성되고, 같은 저장 사건의 할당·구독·전달 후보와 겹쳐도 멘션 유형이 우선한다. |
| COLLAB-003 | Should | 댓글 작성자는 자신의 댓글을 수정하거나 삭제할 수 있다. | 수정 여부가 표시되며 삭제된 댓글은 본문 대신 삭제 상태를 남겨 대화 순서를 보존한다. |
| COLLAB-004 | Must | 주요 이슈 변경은 활동 기록으로 남는다. | 생성, 제목, 상태, 담당자, 우선순위, 라벨, 상위 작업과 차단 관계 변경에 대해 행위자, 시각, 변경 전후 값이 표시된다. 작업 전달은 별도 활동 유형으로 기록한다. |
| COLLAB-005 | Must | 설명과 댓글은 스크립트 실행이 불가능하도록 안전하게 표시한다. | 허용하지 않은 HTML과 위험한 링크 속성이 렌더링되지 않는다. |
| COLLAB-006 | Must | 사용자는 설명·댓글·작업 전달 편집기에 이미지를 선택하거나 Command/Ctrl+V로 붙여넣어 삽입할 수 있다. | 실제 형식이 JPEG·PNG·WebP·GIF인 파일만 본문 이미지로 허용하고 SVG는 일반 첨부로만 처리한다. 클립보드에 실제 이미지 파일이 있을 때만 Rivet 저장소에 업로드하고 Markdown에는 인증된 파일 참조를 저장한다. 업로드 중·실패 상태와 다시 시도·제거 동작을 제공하며 완료되지 않은 업로드가 있으면 본문을 저장할 수 없다. |

MVP에서는 댓글 스레드, 이모지 반응, 협업 편집과 설명 버전 복원을 제공하지 않는다.

### 7.7 API 작업 전달

| ID | 우선순위 | 요구사항 | 인수 조건 |
| --- | --- | --- | --- |
| HANDOFF-001 | Must | 백엔드 역할의 팀 작업에는 Lexical 편집기로 `작업 전달`을 작성하고 Markdown 문자열로 저장할 수 있다. | 기본 템플릿은 변경 요약, API 명세 링크, 사용 가능 환경, 추가·변경 API, 요청·응답 변경, 오류·권한, 프론트 주의사항을 포함한다. |
| HANDOFF-002 | Must | 웹 또는 앱 역할이 설정된 프로젝트의 백엔드 작업은 유효한 최초 전달 없이 완료할 수 없다. | 후행 작업의 기존 존재 여부가 아니라 프로젝트의 프론트 역할로 판단하고 최초 완료 동작을 `전달하고 완료`로 제공한다. 백엔드 전용 프로젝트만 전달 없이 완료할 수 있다. |
| HANDOFF-003 | Must | 최초 전달은 선택한 웹·앱 역할의 후행 작업을 생성하거나 기존 미완료 작업을 모두 재사용한다. | 새로 만든 후행 작업에만 백엔드 작업을 선행 작업으로 연결한다. 기존 작업은 병렬로 시작했을 수 있으므로 사용자가 만든 작업 순서만 유지하고, 완료·취소 작업만 있거나 역할·팀·프로젝트 범위가 다르면 자동 수정 없이 충돌을 반환한다. |
| HANDOFF-004 | Must | 최초 전달 완료 시 실제 후행 작업과 본문 멘션을 기준으로 알림을 보낸다. | 재사용 작업은 담당자와 자동 구독자, 새 미할당 작업은 대상 팀의 활성 멤버, 본문 멘션은 활성 워크스페이스 멤버가 후보이며 작성자를 제외하고 수신자별 한 건만 생성한다. 멘션 대상에는 `MENTIONED` 유형을 우선한다. |
| HANDOFF-005 | Must | 완료 후 변경된 API 내용은 기존 전달을 덮어쓰지 않고 `추가 전달`로 남긴다. | 추가 전달마다 작성자와 시각이 기록되고 기존 후행 작업의 담당자와 자동 구독자에게 알리되 새 팀 작업은 만들지 않는다. |
| HANDOFF-006 | Must | 최초 전달은 프로젝트에 설정된 프론트 역할 중 하나 이상을 대상으로 선택할 수 있다. | 역할이 하나면 고정하고 웹·앱이 모두 있으면 둘 다 기본 선택한다. 전달, 백엔드 완료, 후행 작업·관계, 상위 진행률, 활동과 Outbox는 함께 커밋되거나 함께 롤백된다. |

전체 API 명세를 이슈에 복제하지 않는다. Swagger·OpenAPI 등 기준 문서를 연결하고 해당 작업에서 바뀐 내용과 사용 시 주의사항을 기록한다.

### 7.8 목록, 보드, 필터와 정렬

| ID | 우선순위 | 요구사항 | 인수 조건 |
| --- | --- | --- | --- |
| VIEW-001 | Must | 팀 작업 화면은 같은 팀 작업 집합을 목록과 상태 보드로 표현한다. | 레이아웃을 바꿔도 활성 필터가 유지되고 두 레이아웃의 팀 작업 수와 속성이 일치한다. |
| VIEW-002 | Must | 목록은 상태, 담당자, 우선순위와 제목을 기본 표시한다. | 사용자는 상세 화면을 열지 않고 상태, 담당자와 우선순위를 변경할 수 있다. |
| VIEW-003 | Must | 보드는 상태별 열로 팀 작업을 그룹화한다. | 카드를 다른 열로 옮기면 상태가 변경되고 실패 시 원래 열로 돌아간다. |
| VIEW-004 | Must | 사용자는 팀, 프로젝트, 프로젝트 역할, 상태, 차단 여부, 담당자, 우선순위와 라벨로 팀 작업을 필터링할 수 있다. | 같은 필드에서 여러 값을 선택하면 OR, 서로 다른 필드의 조건은 AND로 결합되며 활성 조건을 한 번에 초기화할 수 있다. |
| VIEW-005 | Must | 사용자는 생성일, 최근 수정일, 상태와 우선순위로 정렬할 수 있다. | 선택한 정렬 기준과 방향이 현재 보기의 모든 결과에 일관되게 적용된다. |
| VIEW-006 | Should | 팀 작업 화면에 `전체`, `진행`, `백로그` 기본 탭을 제공한다. | 전체는 삭제되지 않은 팀 작업, 진행은 `Unstarted·Started`, 백로그는 `Backlog` 범주의 팀 작업을 표시한다. |
| VIEW-007 | Must | 전역 이슈 목록은 프로젝트, 이슈 상태, 우선순위, 현재 작업 역할, 담당자 없음, 라벨, 생성자와 생성·수정일 범위를 조합하고 생성일·최근 수정일·상태·우선순위·진행률로 정렬한다. | 같은 종류의 값은 OR, 종류 사이는 AND로 결합하고 기본은 최근 수정일 내림차순이다. 필터·정렬은 URL에서 복원되며 모든 정렬은 불변 ID를 동률 해소 값으로 사용해 여러 페이지에 중복·누락이 없다. |

수동 순서 정렬, 고급 OR 조건, 중첩 필터, 자연어 필터, 사용자 저장 보기와 임의 보드 그룹은 MVP 범위가 아니다. 팀 목록과 내 작업은 생성일, 최근 수정일, 상태와 우선순위를 사용하고 전역 이슈 목록에만 진행률 정렬을 추가한다.

| --- | --- | --- | --- |
| MY-001 | Must | 내 작업은 현재 사용자에게 할당된 미완료 팀 작업을 보여 준다. | `Completed`와 `Canceled` 범주의 팀 작업은 기본 결과에서 제외되고 여러 팀의 결과가 함께 표시되며 기존 `/my-issues` 경로를 유지한다. |

## 8. 핵심 제품 정책

### 8.1 도메인 불변 조건

- 모든 업무 데이터는 정확히 한 워크스페이스에 속한다.
- 하나의 사용자 계정은 최대 하나의 워크스페이스 멤버십만 가진다.
- 이슈는 정확히 한 프로젝트에 속하며 팀과 담당자를 직접 갖지 않는다.
- 팀 작업은 정확히 하나의 상위 이슈, 한 팀, 한 프로젝트 역할과 한 워크플로 상태에 속한다.
- 프로젝트는 `백엔드`, `웹 프론트`, `앱 프론트` 역할 중 최소 하나에 활성 팀을 연결한다.
- 같은 팀이 프로젝트의 여러 역할을 담당할 수 있으며 각 팀 작업은 정확히 한 프로젝트 역할을 가진다.
- 이슈의 `UNSORTED`, `TODO`, `IN_PROGRESS`, `REVIEW`는 삭제되지 않고 취소되지 않은 직계 팀 작업으로 계산해 저장한다. `PAUSED`, `CANCELED`, `DONE`만 사용자의 명시적 결정으로 자동 계산을 덮어쓴다.
- 이슈만 제목·설명·첨부파일·우선순위·라벨·프로젝트를 소유하고 팀 작업은 이를 복제하지 않는다.
- 팀 작업은 선택적인 평문 작업 메모 외 콘텐츠 필드를 갖지 않는다.
- 팀 작업 담당자는 없거나 소속 팀의 활성 멤버 한 명이다.
- 우선순위 값은 `없음·낮음·중간·높음·긴급`으로 제한한다.
- 이슈 표시 ID와 팀 키 기반 작업 표시 ID는 워크스페이스 안에서 유일하다.
- 사용자화된 상태 이름과 시스템 상태 범주를 분리한다.
- 완료 여부, 기본 보기와 통계는 시스템 상태 범주를 기준으로 판단한다.
- 작업 대기 여부는 상태가 아니라 미완료 선행 작업으로부터 계산하며 사용자에게 구체적인 선행 작업을 표시한다.
- 프론트 역할이 있는 프로젝트의 백엔드 작업은 후행 작업 존재 여부와 무관하게 최초 작업 전달 없이 완료할 수 없다.
- 알림 이벤트와 알림 수신자 결정을 분리하고 수신자별 중복을 제거한다.
- 활동 기록은 원본 변경과 함께 생성하며 사후 재구성에만 의존하지 않는다.

### 8.3 기본 이슈 상태

이슈는 다음 고정 상태를 사용한다. `접수됨`, `할 일`, `진행 중`, `완료 확인`은 서버가 직계 팀 작업을 요약해 저장하고 사용자는 직접 선택하지 않는다.

| 내부 값 | 표시 이름 | 시스템 범주 | 기본 용도 |
| --- | --- | --- | --- |
| `UNSORTED` | 접수됨 | Backlog | 아직 실행할 팀 작업이 없는 요청·아이디어·회의 결과 |
| `TODO` | 할 일 | Unstarted | 실행할 팀 작업은 정해졌지만 실제 작업을 시작하지 않음 |
| `IN_PROGRESS` | 진행 중 | Started | 시작 또는 완료한 작업이 있고 미완료 작업이 남음 |
| `REVIEW` | 완료 확인 | Started | 취소되지 않은 모든 팀 작업이 완료되어 최종 인수를 기다림 |
| `DONE` | 완료 | Completed | 완료 확인을 승인함 |
| `PAUSED` | 일시 중지 | Backlog | 이슈 전체를 명시적으로 멈춤 |
| `CANCELED` | 취소 | Canceled | 이슈 전체를 더 진행하지 않음 |

팀 작업은 팀 생성 시 `미분류`, `할 일`, `진행 중`, `검토`, `완료`, `보류`, `취소`를 만들고 관리자가 이름과 순서를 조정할 수 있다. 이슈에서 역할 선택·작업 시작·최초 전달로 생성하는 팀 작업은 해당 팀의 `Unstarted` 범주 중 표시 순서가 가장 빠른 상태로 시작한다. 상위 이슈 없는 단독 팀 작업은 만들지 않는다.

다른 작업을 기다리는 경우에는 `일시 중지`나 팀의 `보류`로 바꾸지 않고 현재 상태와 `{표시 ID} 완료 후 시작 가능` 같은 작업 순서 안내를 함께 사용한다.

### 8.4 프로젝트 상태

| 상태 | 의미 |
| --- | --- |
| 계획 | 실행 전 계획 중인 프로젝트 |
| 진행 중 | 현재 실행 중인 프로젝트 |
| 완료 | 목표를 달성하고 종료한 프로젝트 |
| 취소 | 더 이상 진행하지 않는 프로젝트 |

### 8.5 알림 채널

- 인앱 알림은 알림 수신자, 읽음 상태, 이동 대상과 중복 제거의 유일한 제품 정본이다.
- 이메일은 회원가입·초대·이메일 인증·비밀번호 재설정 같은 계정 흐름에만 사용한다.
- 업무 할당·멘션·댓글·작업 전달을 이메일로 발송하지 않으며 이 정책은 Alpha, Beta와 Public까지 유지한다.
- 개인 실시간 알림 확장은 Web Push를 우선한다. 사용자가 명시적으로 권한을 허용한 브라우저 구독에만 보내고, Push 실패가 인앱 알림의 생성·읽음 상태를 바꾸지 않는다.
- Slack은 반복 요구가 확인될 때 팀·프로젝트 공용 알림부터 제한적으로 검토한다. 개인 DM, 사용자 매핑과 양방향 명령은 별도 범위다.

## 9. 비기능 요구사항

### 9.1 성능

아래 수치는 MVP 베타 환경의 초기 품질 목표이며 실제 트래픽을 측정한 뒤 조정한다.

- 주요 목록은 일반적인 1~10명 팀의 데이터에서 사용자가 기다림을 인지하지 않을 수준으로 동작해야 한다.
- 이슈·프로젝트 생성과 속성 변경 API의 서버 응답 시간은 정상 부하에서 p95 1초 이내를 목표로 한다.
- ID·제목 검색 결과는 정상 부하에서 p95 1초 이내에 반환하는 것을 목표로 한다.
- 낙관적 UI를 적용할 수 있는 변경은 네트워크 응답 전에 시각적으로 반영한다.
- 큰 목록은 페이지네이션 또는 점진적 로딩을 사용하며 전체 데이터를 한 번에 브라우저로 보내지 않는다.
- 전역 이슈의 빠른 필터와 작업 요약은 현재 페이지의 이슈 집합을 한 번에 집계하며 행마다 하위 작업을 추가 조회하지 않는다.
- JPEG·PNG·WebP 이미지가 2MB를 넘거나 긴 변이 2,560px을 넘으면 프론트에서 비율을 유지해 최대 2,560px로 축소하고 WebP 품질 82%를 기준으로 압축한다. 변환 결과가 더 크면 원본을 사용하며 최종 파일이 25MB를 넘으면 업로드를 거부한다.

### 9.2 신뢰성과 데이터 일관성

- 동시 생성에도 이슈 표시 ID가 중복되지 않아야 한다.
- 저장 성공으로 표시한 변경은 새로고침 후에도 유지되어야 한다.
- 이슈·팀 작업·작업 순서·작업 전달은 모든 화면에서 같은 연결 상태를 보여야 한다.
- 루트 이슈와 최초 팀 작업 생성, 최초 전달과 백엔드 완료·후행 작업·관계·활동·Outbox는 각각 원자적으로 처리되어야 한다.
- 역할별 담당자를 포함한 작업 시작, 내가 맡기와 여러 팀 작업의 일괄 배정은 대상 검증, 생성·재사용, 담당자 변경, 활동, 구독과 Outbox를 각각 한 트랜잭션에서 처리해야 한다.
- 알림과 활동 이벤트의 재처리는 멱등해야 한다.
- 실시간 연결이 끊겨도 재연결 후 서버 상태로 수렴해야 한다.
- 데이터베이스 백업과 복구 절차를 출시 전에 검증해야 한다.
- 데이터베이스의 파일 메타데이터와 로컬 업로드 파일을 함께 백업·복구하고 정기 정리 작업을 반복 실행해도 연결된 파일이 삭제되지 않아야 한다.

### 9.3 보안과 개인정보

- 모든 업무 API는 인증과 워크스페이스 경계를 검증해야 한다.
- 다른 워크스페이스의 리소스 ID를 직접 입력해도 데이터가 노출되지 않아야 한다.
- 인증 토큰과 초대 토큰은 만료 및 폐기 정책을 가져야 한다.
- 비밀번호는 복구할 수 없는 강한 해시로 저장하며 이메일 인증·비밀번호 재설정 토큰 원문을 저장하지 않는다.
- 사용자 입력은 저장과 출력 시 적절히 검증하고 안전하게 렌더링해야 한다.
- 첨부파일 원본 이름을 저장 경로로 사용하지 않고, 실제 형식과 크기를 서버에서 다시 검증하며 파일 조회·다운로드에도 인증과 워크스페이스 경계를 적용해야 한다.
- 비밀 값과 인증 정보는 로그에 기록하지 않는다.

### 9.4 접근성과 키보드

- 생성, 검색, 이슈 열기와 핵심 속성 변경은 키보드로 수행할 수 있어야 한다.
- 포커스 위치가 시각적으로 구분되고 모달을 닫은 뒤 합리적인 위치로 복귀해야 한다.
- 아이콘만 있는 동작에는 접근 가능한 이름을 제공한다.
- 상태와 우선순위는 색상만으로 구분하지 않는다.
- 정식 접근성 인증은 MVP 범위가 아니지만 핵심 흐름에서 명백한 키보드 차단과 대비 문제를 출시 결함으로 취급한다.

### 9.5 호환성

- 최신 안정 버전의 Chrome을 1차 지원한다.
- Safari와 Firefox 최신 안정 버전의 핵심 사용자 여정은 출시 전 확인한다.
- 데스크톱 웹을 정식 지원 대상으로 한다.
- 모바일 웹에서는 전역 이슈 목록을 카드로 제공하고 필터 패널, 작업 시작, 우리 팀에서 시작, 내가 맡기와 담당자 지정을 지원한다. 이슈·프로젝트 조회, 댓글 작성과 상태·담당자·우선순위 변경도 유지한다.
- 모바일 보드, 팀 관리와 워크스페이스 설정은 MVP 최적화 대상에서 제외한다.

## 10. 제품 분석과 이벤트 측정

사용자 행동 측정은 제품 가설을 검증하기 위한 최소 범위로 구성한다. 이벤트 이름과 속성에는 이슈 제목, 설명, 댓글 본문과 API 작업 전달 등 민감한 업무 내용을 넣지 않는다. 사용자와 워크스페이스는 내부 식별자로만 구분한다.

- PostHog Cloud로 전송하는 명시적 기능 이벤트와 정제된 예외 메타데이터는 1년 보관한다.
- 자체 서버의 구조화 운영 로그는 30일 보관한다.
- 자동 이벤트 수집과 세션 녹화는 사용하지 않으며 이슈 제목, 설명, 댓글, 작업 전달, 이메일과 토큰을 외부 분석 데이터에 포함하지 않는다.

## 11. 성공 지표와 출시 기준

### 11.1 핵심 성공 조건

MVP의 최종 검증 기준은 **1~10명 규모의 베타 팀이 기존 이슈 트래커 대신 Rivet을 4주 연속 실제 업무에 사용하고, 이슈·프로젝트 데이터 불일치나 작업 전달·알림 누락 때문에 기존 도구로 돌아가지 않는 것**이다.

### 11.2 관찰 지표

초기 베타에서는 표본이 작으므로 임의의 업계 기준을 성공선으로 사용하지 않는다. 다음 지표의 기준선을 수집하고 팀별 인터뷰와 함께 판단한다.

- 초대된 멤버 중 주간 활성 멤버 비율.
- 팀별 주간 생성 이슈 수와 완료 이슈 수.
- 주간 활성 프로젝트 수와 프로젝트별 이슈·팀 작업 수.
- API 작업 완료 후 후행 작업의 전달 확인과 작업 가능 판단까지 걸린 시간.
- 이슈 생성 후 첫 상태 변경 또는 댓글까지 걸린 시간.
- 활성 멤버 중 전역 이슈, 내 작업, 팀 작업, 알림함의 주간 사용 비율.
- 검색 후 결과를 선택한 비율과 결과 없음 비율.
- 생성된 알림 중 읽은 비율과 읽기까지 걸린 시간.
- 저장 실패, 동기화 충돌, 중복 알림과 데이터 불일치 발생 건수.
- 베타 팀의 1주차 및 4주차 정성 만족도와 이탈 사유.

### 11.3 출시 차단 조건

다음 중 하나가 남아 있으면 공개 MVP 출시를 진행하지 않는다.

- 워크스페이스 경계를 넘어 데이터가 노출될 수 있는 결함.
- 저장 성공으로 표시한 이슈 변경의 유실 또는 화면 간 지속적인 불일치.
- 프로젝트와 연결 이슈의 관계 또는 진행률이 화면마다 다르게 표시되는 결함.
- 프로젝트 역할과 담당 팀이 맞지 않는 팀 작업 또는 순환 작업 순서가 생성되는 결함.
- 프론트 역할이 있는 프로젝트의 백엔드 작업이 최초 전달 없이 완료되는 결함.
- 이슈 생성 또는 최초 전달 실패 뒤 일부 팀 작업·관계·활동·Outbox만 남는 결함.
- 작업 시작·내가 맡기·일괄 배정 실패 뒤 일부 팀 작업·담당자·활동·구독·Outbox만 남거나 오래된 화면이 새 담당자를 덮어쓰는 결함.
- 전역 이슈 빠른 필터가 확정 조건과 다르거나 같은 정렬값의 다중 페이지에서 이슈가 중복·누락되는 결함.
- 이슈 표시 ID 중복 발급.
- 핵심 생성·수정 흐름을 막는 재현 가능한 오류.
- 알림 폭주 또는 같은 이벤트의 반복 알림.
- 백업 복구 절차 미검증.
- 다른 워크스페이스의 첨부파일을 조회하거나 내려받을 수 있는 결함.
- 연결된 첨부파일이 정리 작업으로 삭제되거나 데이터베이스 복구 후 파일과 메타데이터가 불일치하는 결함.
- 운영자가 반복 실패와 치명적 오류를 탐지할 수 없는 상태.

## 12. 단계별 출시 범위

### 12.1 내부 프로토타입

- 단일 워크스페이스와 단일 팀.
- 이슈 생성·수정·상세.
- 고정 상태, 담당자와 우선순위.
- 목록과 상태 보드.
- 전역 생성과 핵심 인라인 편집.

**통과 기준:** 내부 사용자가 별도 문서 없이 일주일의 실제 작업을 기록하고 매일 다시 열어 관리할 수 있다.

### 12.2 MVP 완료 범위

- 여러 팀, 멤버 초대와 `admin/member` 권한.
- 동적 팀·팀 멤버 관리, 한국어 기본 워크플로와 라벨.
- 역할별 담당 팀을 선택하는 프로젝트와 기본 진행률.
- 이슈, 선택에 따라 점진적으로 생성되는 팀 작업, 작업 흐름과 작업 순서.
- Lexical WYSIWYG 편집·미리보기와 Markdown 저장, 댓글, 멘션과 활동 기록.
- 이슈 일반 첨부, 편집기 이미지 붙여넣기와 프로필 사진.
- API 작업 전달, 작업 가능 안내와 알림.
- 전역 이슈 접수·분류·작업 배정, 필터, 정렬, 내 작업과 검색.
- 인앱 알림함과 제한적 모바일 웹.
- 낙관적 UI와 제한적 실시간 갱신.
- CSV 내보내기, 백업과 오류 모니터링.

**완료 기준:** 구현 범위와 자동 검증 결과는 개발 완료 보고서에서 판정하고, 실제 사용자 유지·전환 성과는 Closed Alpha와 Closed Beta에서 검증한다.

### 12.3 Closed Alpha

Closed Alpha는 초대된 소규모 소프트웨어 팀이 기존 데이터를 옮겨 실제 업무를 반복하고, 제품 전환과 개인 알림의 실패 지점을 확인하는 단계다. 다음 기능을 우선한다.

1. 전환 장벽을 낮추는 **CSV 가져오기**.
2. 필터·정렬·표시 구성을 반복 사용하는 **개인 저장된 보기**.
3. 기존 인앱 알림을 정본으로 사용하는 **Web Push**.
4. 반복 이슈 입력을 줄이는 **워크스페이스 이슈 템플릿**.
5. 정성 피드백과 최소 제품 이벤트를 연결하는 **Alpha 피드백·계측**.

GitHub와 Slack은 Alpha 사용에서 반복 문제가 확인될 때 하나씩 검토한다. GitHub는 PR 링크와 상태의 읽기 표현부터, Slack은 팀·프로젝트 공용 알림부터 시작하고 처음부터 양방향 자동화를 만들지 않는다.

### 12.4 Closed Beta와 Public

Closed Beta는 Alpha 기능의 안정성과 반복 사용을 더 많은 초대 팀에서 검증하는 단계다. 사이클, 고급 트리아지, 하위 이슈·관계, private 팀과 세밀한 권한은 Alpha 근거가 있을 때만 선택한다.

Public은 공개 가입, 지원·보안 운영과 확장 비용을 감당할 준비가 된 뒤 판단한다. 프로젝트 마일스톤·타임라인, 공개 API·OAuth, 네이티브 앱, Enterprise 보안과 AI 기능은 Public의 기본 조건이 아니며 별도 제품 근거가 필요하다.

한 계정의 다중 워크스페이스 생성·참여와 워크스페이스 전환은 MVP, Alpha와 Beta 범위에 포함하지 않는다. Public 출시 이후 핵심 사용이 안정화된 다음 고도화 기능으로 검토한다.

## 13. 주요 리스크와 대응

| 리스크 | 영향 | MVP 대응 |
| --- | --- | --- |
| 화면마다 이슈 값이 다르게 보임 | 제품 신뢰 상실 | 정규화된 클라이언트 상태, 서버 재검증, 화면 간 일관성 테스트 |
| 실시간 연결 유실 | 다른 사용자의 변경 누락 | 재연결 시 데이터 재조회, 서버를 최종 원천으로 유지 |
| 오래된 저장이 최신 값을 덮음 | 사용자 변경 유실 | 버전 충돌 감지와 명시적 재시도 |
| 알림 중복·폭주 | 알림함 사용 중단 | 이벤트·수신자 기준 멱등 키, 자신의 행동 제외, 규칙 최소화 |
| 상태 삭제로 이슈가 고아가 됨 | 데이터 무결성 훼손 | 대체 상태 지정 전 삭제 차단 |
| 프로젝트 역할과 팀 작업 연결이 불일치함 | 계획 정보의 신뢰 상실 | 역할별 팀 제약, 단일 프로젝트 규칙, 관계 무결성 테스트 |
| 작업 순서가 순환하거나 선행 완료가 반영되지 않음 | 팀 간 작업 정체 | 순환 등록 차단, 완료 시 재계산, 관계 이력 테스트 |
| API 전달 없이 백엔드 작업이 완료됨 | 프론트 작업 누락·재질문 | 프로젝트 프론트 역할을 기준으로 `전달하고 완료`와 대상 역할 선택을 강제 |
| 검색·직접 링크의 워크스페이스 누출 | 심각한 보안 사고 | 모든 조회 경로에 공통 워크스페이스 권한 적용 |
| 빠른 조작보다 설정 기능에 개발 집중 | 핵심 가치 약화 | 생성·인라인 편집·전역 이슈·내 작업을 먼저 사용자 테스트 |
| Linear 기능 추격으로 범위 확대 | 핵심 검증 지연 | 단계별 비목표 유지, 추가 기능은 Alpha 사용 근거와 승인 필요 |
| 차별화가 불명확함 | 전환 동기 부족 | 베타 전에 핵심 고객군과 전환 이유 인터뷰 수행 |

## 14. 범위 변경 원칙

MVP 문서는 완료 기준으로 동결한다. Alpha 이후 새 기능을 추가하려면 다음 질문에 모두 답해야 한다.

1. 핵심 사용자 여정 중 어떤 실패를 해결하는가?
2. 현재 요구사항이나 더 단순한 운영 방식으로 해결할 수 없는가?
3. Alpha 또는 Beta 사용자에게서 반복적으로 확인된 요구인가?
4. 권한, 검색, 알림, 실시간 동기화와 데이터 내보내기에 미치는 영향은 무엇인가?
5. 추가 후에도 현재 단계의 진입·종료 조건을 더 빠르고 정확하게 검증할 수 있는가?

답이 불명확하면 현재 단계에 넣지 않고 후속 검토 항목으로 남긴다.
