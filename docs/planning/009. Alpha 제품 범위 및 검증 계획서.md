## 1. 문서 목적

이 문서는 MVP 기능 개발을 마친 Rivet이 Closed Alpha에서 무엇을 추가하고 무엇을 검증할지 정의한다. Alpha의 목적은 Linear 기능을 더 많이 복제하는 것이 아니라 초대된 소규모 소프트웨어 팀이 기존 데이터를 옮기고, 자주 쓰는 업무 관점을 재사용하고, 중요한 개인 알림을 놓치지 않으며, 제품을 계속 사용할 근거를 확인하는 것이다.

이 문서는 Alpha의 제품 범위와 인수 조건을 정한다. 테이블·엔드포인트·브라우저별 구현 방식과 마이그레이션 순서는 각 기능 구현 전에 기술 명세와 실행 작업에서 구체화한다.

## 2. 제품 단계

| 단계 | 목적 | 종료 판단 |
| --- | --- | --- |
| 내부 MVP | 핵심 이슈·팀 작업·프로젝트·협업·운영 루프 구현 | M0~M9와 MVP 보완 범위의 개발 완료 |
| Closed Alpha | 실제 팀의 데이터 전환, 반복 사용과 개인 알림 검증 | Alpha 필수 기능과 대표 흐름 통과, 출시 차단 결함 없음 |
| Closed Beta | 더 많은 초대 팀에서 안정성·유지·운영 비용 검증 | 팀별 실제 사용 기준선과 이탈 원인 확보, 운영 대응 가능 |
| Public | 공개 가입과 지원·보안 운영을 감당할 수 있는 상태 | 별도 Public 승인 기준 충족 |

Alpha는 MVP를 다시 만드는 단계가 아니다. 기존 이슈·팀 작업·프로젝트·알림 계약을 정본으로 유지하고 전환과 반복 사용에 필요한 기능만 추가한다.

## 3. Alpha 목표

1. 관리자가 개발자의 데이터 직접 수정 없이 기존 업무 데이터를 Rivet으로 옮긴다.
2. 사용자가 매번 필터와 표시 옵션을 다시 구성하지 않고 자신의 업무 관점을 복원한다.
3. 반복되는 이슈 형식과 기본 속성을 템플릿으로 재사용한다.
4. 사용자가 앱을 열어 두지 않아도 중요한 개인 알림을 Web Push로 확인한다.
5. 운영자가 정량 이벤트와 정성 피드백을 함께 보고 다음 제품 결정을 내린다.
6. GitHub와 Slack은 실제 반복 문제가 확인된 경우에만 제한된 첫 단계로 검토한다.

## 4. 대상 사용자와 운영 모델

- 대상은 1~10명 규모의 초대된 소프트웨어 개발 팀이다.
- 한 계정은 하나의 워크스페이스만 생성하거나 참여한다.
- 워크스페이스 내부 데이터는 모든 활성 멤버가 볼 수 있는 공개 팀 모델을 유지한다.
- `admin/member` 역할과 기존 팀 멤버십 권한을 유지한다.
- 데스크톱 웹을 핵심 업무 기준으로 삼고 모바일 웹은 조회·댓글·핵심 상태 변경을 제한적으로 유지한다.
- Alpha 지원과 피드백 대응은 운영자가 직접 수행하되 데이터베이스 수동 수정으로 정상 제품 흐름을 대체하지 않는다.

## 5. 제품 원칙

### 5.1 기존 정본을 확장한다

- 이슈는 제목·설명·첨부파일·댓글의 유일한 콘텐츠 원본이다.
- 팀 작업은 이슈에 속한 실행 슬롯이며 자체 콘텐츠 저장소를 만들지 않는다.
- 인앱 알림은 수신자, 읽음 상태, 이동 대상과 중복 제거의 유일한 알림 정본이다.
- 서버와 PostgreSQL을 최종 데이터 원천으로 유지한다.
- 다른 워크스페이스 데이터는 가져오기, 저장된 보기, 템플릿, Push와 피드백에서도 노출하지 않는다.

### 5.2 이메일 정책

- Resend는 회원가입·초대·이메일 인증·비밀번호 재설정 같은 계정 흐름에만 사용한다.
- 할당·멘션·댓글·작업 전달과 상태 변경을 이메일로 발송하지 않는다.
- 업무 이메일 알림은 Alpha, Beta와 Public 범위에서 제공하지 않는다.
- 향후 자체 메일 서버나 다른 공급자를 검토하더라도 Public 이후의 별도 제품·운영 결정으로 취급한다.

### 5.3 Alpha는 근거를 만들기 위한 범위다

- 필수 기능을 한 번에 사용할 수 있게 만들되 조건부 연동을 동시에 구현하지 않는다.
- 기능 존재보다 실제 전환 성공, 반복 사용과 실패 복구를 완료 기준으로 삼는다.
- 제품 이벤트에는 이슈 제목, 설명, 댓글, 전달 본문, 이메일, 토큰과 파일명을 넣지 않는다.
- 정성 피드백 본문은 제품 분석 공급자로 전송하지 않는다.

## 7. CSV 가져오기

### 7.1 목적

기존 이슈 트래커의 업무를 수작업으로 다시 입력하지 않고 Rivet의 팀·워크플로·멤버·프로젝트 구조에 안전하게 연결한다. CSV 가져오기는 일회성 전환 기능이며 외부 시스템과 지속적으로 동기화하는 Issue sync가 아니다.

### 7.2 요구사항

| ID | 요구사항 | 인수 조건 |
| --- | --- | --- |
| ALPHA-IMPORT-001 | 관리자는 문서화된 CSV 형식을 업로드할 수 있다. | 허용 형식·인코딩·행 수·파일 크기를 업로드 전에 안내하고 비관리자는 실행할 수 없다. |
| ALPHA-IMPORT-002 | 저장 전에 필드와 대상 값을 매핑한다. | 팀, 워크플로 상태, 담당자, 프로젝트, 우선순위와 라벨의 원본 값을 기존 값 또는 새 값에 명시적으로 연결한다. |
| ALPHA-IMPORT-003 | 가져오기 전 검증 결과를 확인한다. | 전체 행 수, 생성·연결·제외 예정 수와 행별 오류를 보여 주며 오류가 있는 동안 저장하지 않는다. |
| ALPHA-IMPORT-004 | 한 번의 실행은 원자적으로 완료된다. | 일부 이슈만 저장된 채 실패하지 않고 전체 성공 또는 전체 롤백된다. |
| ALPHA-IMPORT-005 | 같은 파일의 중복 실행을 안전하게 처리한다. | 실행 식별자와 source key로 중복을 감지하고 사용자가 명시적으로 새 실행을 선택하지 않으면 같은 항목을 다시 만들지 않는다. |
| ALPHA-IMPORT-006 | 결과와 실패를 추적한다. | 관리자만 실행자, 시각, 입력 행 수, 결과 수와 정제된 오류 코드를 볼 수 있고 원본 CSV·업무 본문은 운영 로그에 남기지 않는다. |

### 7.3 최소 데이터 범위

- 프로젝트, 이슈 제목·설명, 팀, 상태, 담당자, 우선순위와 라벨
- 현재 Rivet 프로젝트 역할·팀 구조에 맞는 선택적 프로젝트 연결
- 원본 표시 ID를 충돌 없는 source reference로 보존할 수 있는 감사 정보
- 팀과 상태를 찾을 수 없는 행의 명시적 매핑 또는 제외

댓글·활동·알림·작업 전달 이력, 첨부파일 바이너리와 외부 시스템의 자동화 규칙은 Alpha 가져오기 범위에 포함하지 않는다. 원본 데이터가 이 범위를 포함해도 조용히 버리지 않고 미지원 항목으로 보고한다.

가져오기로 생성한 이슈·팀 작업은 외부 이력을 재현하거나 대량 알림을 보내지 않는다. `ISSUE_IMPORTED` 활동과 관리자 실행 결과는 남기되 일반 이슈 생성·작업 생성 Outbox, 담당자·멘션 알림은 발행하지 않는다.

## 8. 개인 저장된 보기

### 8.1 목적

사용자가 전역 이슈와 내 작업에서 자주 사용하는 검색·필터·정렬·표시 구성을 이름 붙여 다시 열 수 있게 한다.

### 8.2 요구사항

| ID | 요구사항 | 인수 조건 |
| --- | --- | --- |
| ALPHA-VIEW-001 | 사용자는 현재 목록 구성을 개인 보기로 저장한다. | 리소스 종류, 검색, 빠른 필터, 세부 필터, 정렬과 표시 옵션을 정규화된 구조로 저장한다. |
| ALPHA-VIEW-002 | 저장된 보기를 다시 열면 같은 구성이 복원된다. | URL과 화면 상태가 일치하고 존재하지 않거나 권한을 잃은 값은 오류를 숨기지 않고 정리 동작을 제공한다. |
| ALPHA-VIEW-003 | 사용자는 이름을 바꾸고 삭제하거나 기본 보기로 지정한다. | 이름 충돌과 동시 수정은 명시적으로 처리하고 삭제가 이슈·팀 작업 데이터에 영향을 주지 않는다. |
| ALPHA-VIEW-004 | 개인 보기는 소유자에게만 노출된다. | 다른 멤버가 ID를 알아도 조회·수정할 수 없고 workspace 경계를 넘지 않는다. |

Alpha에서는 전역 이슈와 내 작업을 우선 지원한다. 팀 공유 보기, 권한별 공유, 구독과 복잡한 중첩 OR 필터는 포함하지 않는다.

## 9. Web Push

### 9.1 목적

사용자가 Rivet을 열어 두지 않았을 때도 기존 인앱 알림 중 중요한 개인 사건을 확인하고 정확한 업무 위치로 돌아오게 한다.

### 9.2 초기 이벤트

- 내게 팀 작업이 할당됨
- 설명 또는 댓글에서 멘션됨
- 최초 또는 추가 API 작업 전달을 받음
- 구독한 이슈에 다른 사용자가 댓글을 작성하거나 이슈를 완료·취소함

### 9.3 요구사항

| ID | 요구사항 | 인수 조건 |
| --- | --- | --- |
| ALPHA-PUSH-001 | 사용자는 브라우저에서 명시적으로 Push를 켜고 끈다. | 사용자 동작 없이 권한 창을 열지 않고 거절·미지원 상태에서도 인앱 알림이 정상 동작한다. |
| ALPHA-PUSH-002 | 한 사용자는 여러 브라우저 구독을 관리한다. | 브라우저별 등록·해제·최근 성공·실패 상태를 구분하고 테스트 알림을 보낼 수 있다. |
| ALPHA-PUSH-003 | Push는 생성된 인앱 알림을 전달한다. | 별도 수신자 규칙을 만들지 않고 같은 notification ID와 이동 대상을 사용한다. |
| ALPHA-PUSH-004 | 중복과 만료 구독을 안전하게 처리한다. | 같은 알림과 구독 조합을 한 번만 전달하고 영구 실패 구독은 비활성화한다. |
| ALPHA-PUSH-005 | 알림을 누르면 정확한 위치로 이동한다. | 이슈, 팀 작업, 댓글 또는 전달 딥 링크를 열고 권한이 없거나 삭제된 대상은 안전한 안내를 표시한다. |
| ALPHA-PUSH-006 | 민감 정보를 최소화한다. | 잠금 화면 payload에는 업무 본문·파일명·이메일을 넣지 않고 일반화된 사건명과 필요한 최소 식별자만 사용한다. |

브라우저 지원 범위, 서비스 워커 갱신, VAPID 키 보관과 구독 정보 보호는 구현 전에 기술 계약으로 고정한다. 모바일 푸시와 네이티브 앱은 포함하지 않는다.

## 10. 이슈 템플릿

### 10.1 목적

반복되는 기능 개발, 버그와 운영 작업의 설명 구조와 기본 속성을 재사용해 이슈 생성 비용과 누락을 줄인다.

### 10.2 요구사항

| ID | 요구사항 | 인수 조건 |
| --- | --- | --- |
| ALPHA-TEMPLATE-001 | 관리자는 워크스페이스 이슈 템플릿을 생성·수정·보관한다. | 이름과 설명 Markdown을 필수로 하고 보관 템플릿은 신규 선택에서 제외하되 기존 생성 이슈에는 영향을 주지 않는다. |
| ALPHA-TEMPLATE-002 | 템플릿은 안전한 기본 속성을 가진다. | 설명, 우선순위, 라벨, 선택적 프로젝트와 최초 역할을 저장하고 적용 시 현재 권한과 프로젝트 역할을 다시 검증한다. |
| ALPHA-TEMPLATE-003 | 사용자는 생성 화면에서 템플릿을 적용하고 수정한다. | 템플릿 적용 뒤 값을 자유롭게 바꿀 수 있고 원본 템플릿은 변경되지 않는다. |
| ALPHA-TEMPLATE-004 | 템플릿 변경은 이력과 충돌을 안전하게 처리한다. | 오래된 관리 화면이 최신 템플릿을 조용히 덮어쓰지 않고 활동 기록에는 본문 전체를 복제하지 않는다. |

반복 일정에 따른 자동 생성, 템플릿 실행 자동화, 팀별 비공개 템플릿과 외부 공유는 포함하지 않는다.

## 11. Alpha 피드백과 제품 계측

### 11.1 피드백

- 모든 인증된 사용자가 전역 UI에서 피드백을 보낼 수 있다.
- 카테고리, 본문, 현재 경로와 릴리스 ID를 저장한다.
- 이슈·댓글·작업 전달 본문, 첨부파일, 콘솔 로그와 화면 캡처는 자동 첨부하지 않는다.
- 제출자는 접수 성공을 확인하고 중복 제출을 피할 수 있다.
- 관리자는 workspace, 카테고리, 접수 시각과 상태를 기준으로 조회하고 `접수`, `검토 중`, `반영`, `보류` 상태를 관리한다.
- 피드백 본문은 PostHog로 전송하지 않고 Rivet 데이터베이스에서 권한과 보존 정책을 적용한다.

### 11.2 최소 제품 이벤트

- 초대 수락과 첫 로그인
- 첫 CSV 가져오기 시작·검증·완료·실패
- 첫 이슈 생성, 첫 팀 작업 시작과 첫 완료
- 저장된 보기 생성·열기
- 템플릿 적용과 템플릿 기반 이슈 생성
- Push 권한 결과, 테스트 알림과 알림 클릭
- 검색 실행·결과 없음·결과 선택
- 인앱 알림 생성·읽음까지의 시간
- 피드백 제출 사건. 본문은 포함하지 않음

이벤트는 허용 목록과 버전 계약을 사용하고 workspace·membership 식별자는 분석에 필요한 최소 범위로 제한한다.

## 12. 조건부 연동

### 12.1 GitHub

다음 문제가 여러 Alpha 사용자에게 반복될 때 검토한다.

- Rivet 이슈와 PR 링크를 수동으로 양쪽에 반복 기록함
- PR 상태와 이슈 상태가 달라 완료 판단이 지연됨

첫 범위는 GitHub App 설치, 저장소·팀 매핑, 이슈 표시 ID가 포함된 PR 링크와 상태의 읽기 표현이다. 자동 상태 전이, 양방향 댓글, 다중 공급자와 Issue sync는 첫 범위가 아니다.

### 12.2 Slack

다음 문제가 여러 Alpha 사용자에게 반복될 때 검토한다.

- 팀원이 Rivet을 열지 않아 공용 작업 전달이나 프로젝트 상태를 늦게 확인함
- Web Push보다 팀 채널에서 공유해야 할 사건이 명확함

첫 범위는 관리자가 등록한 Slack Incoming Webhook, 팀·프로젝트별 대상 채널, 소수의 공용 사건과 테스트 알림이다. 개인 DM, Slack 사용자 매핑, Slack에서 이슈 생성·수정과 양방향 명령은 첫 범위가 아니다.

GitHub와 Slack은 같은 마일스톤에서 동시에 구현하지 않는다. 반복 문제, 예상 사용자 영향과 운영 비용을 비교해 하나를 선택한다.

## 13. Alpha 비목표

- 업무 이메일 알림과 이메일 digest
- 다중 워크스페이스 생성·참여와 전환
- 사이클, 고급 트리아지와 수동 이슈 순서
- 하위 이슈·일반 관계와 프로젝트 마일스톤·타임라인
- private 팀, 게스트, 이슈 단위 공유와 사용자 정의 역할
- 모바일 푸시와 네이티브 앱
- 공개 API, 개인 액세스 토큰, OAuth 앱과 범용 웹훅
- Slack·GitHub 양방향 자동화와 Issue sync
- AI Agent, MCP, 코드 인텔리전스와 분석 대시보드
- 자동 CI/CD, Kubernetes, 다중 리전과 대규모 분산 인프라

## 14. 관찰 지표

Alpha 표본이 작으므로 임의의 업계 기준을 성공선으로 사용하지 않는다. 팀별 기준선과 정성 피드백을 함께 본다.

- 초대부터 가입·워크스페이스 설정·첫 가져오기 완료까지의 단계별 이탈
- 가져오기 검증 실패율, 주요 매핑 오류와 재시도 횟수
- 저장된 보기의 생성 사용자 수와 반복 열기 횟수
- 템플릿 생성·적용과 템플릿 기반 이슈 생성 비율
- Push 권한 허용·거절·미지원, 전달 성공과 클릭 비율
- 첫 이슈 생성부터 첫 팀 작업 시작·완료까지의 시간
- 인앱 알림 읽음까지의 시간과 Push 사용 전후 변화
- 피드백 카테고리, 반복 문제와 해결 후 재발 여부
- 저장 실패, 중복 가져오기·알림, 권한 오류와 데이터 불일치 건수

## 15. Alpha 진입 조건

- MVP 요구사항과 M0~M9 개발 범위가 완료 상태로 기록돼 있다.
- lint, typecheck, unit, integration, API 계약과 production build가 현재 기준으로 통과한다.
- CSV 가져오기, 저장된 보기, Web Push, 이슈 템플릿과 피드백의 대표 성공·실패·권한 흐름이 자동 검증된다.
- 실제 Chromium에서 데스크톱·compact·모바일 핵심 흐름을 검증한다.
- 다른 workspace 데이터 노출, 부분 저장, 중복 알림과 연결 파일 손실 같은 출시 차단 결함이 없다.
- 운영 환경에서 인증 이메일, 오류 탐지, 백업과 복구 절차를 확인한다.

## 16. Closed Beta 전환 조건

- Alpha 팀이 개발자의 데이터 직접 수정 없이 가입·가져오기·기본 설정을 완료한다.
- 저장된 보기와 템플릿이 한 번 생성된 뒤 실제 반복 업무에서 다시 사용된다.
- Web Push가 인앱 알림과 중복되거나 누락된 것처럼 보이지 않고 딥 링크가 정확한 위치를 연다.
- 가져오기 실패가 부분 데이터와 중복 데이터를 남기지 않는다.
- 모든 심각도 높은 피드백과 출시 차단 결함에 원인, 사용자 영향과 처리 결과가 기록된다.
- 조건부 연동의 구현 여부를 실제 사용 근거로 결정한다.
- Alpha 운영에서 복구할 수 없는 데이터 손실, workspace 경계 침해와 알림 폭주가 없다.

## 17. 범위 변경 원칙

새 기능을 Alpha에 추가하려면 다음을 모두 기록한다.

1. 어떤 Alpha 목표와 반복 문제를 해결하는가.
2. 현재 필수 기능이나 운영 지원으로 해결할 수 없는 이유는 무엇인가.
3. 누가 얼마나 반복적으로 요청하거나 실패했는가.
4. 데이터 모델, 권한, 알림, 실시간, 내보내기와 복구에 미치는 영향은 무엇인가.
5. 기존 Alpha 필수 기능 중 무엇을 늦추는가.
6. 구현하지 않을 때의 우회 방법과 Beta 재검토 조건은 무엇인가.

근거가 불충분하면 Alpha 범위에 넣지 않는다.
