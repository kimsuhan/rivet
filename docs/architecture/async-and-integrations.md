# 비동기 작업과 외부 연동

> 상태: 현행
> 적용 범위: Outbox, Worker, SSE, 알림, 이메일, Web Push와 관측 연동
> 정본 책임: 비동기 신뢰성, 원본 데이터와 전달 채널의 관계
> 갱신 조건: 이벤트, 재시도, 전달 채널, 실시간 또는 외부 공급자 변경

## 기본 원칙

- 제품 상태 변경과 Outbox 이벤트를 같은 데이터베이스 트랜잭션에 기록한다.
- 외부 공급자 호출은 원본 트랜잭션 안에서 실행하지 않는다.
- Worker는 이벤트 ID, lock owner와 lock 만료로 처리권을 관리한다.
- 이벤트 payload는 최소 식별자와 버전만 가지며 처리 시 현재 데이터를 다시 조회한다.
- 외부 전달 실패가 이미 성공한 제품 요청을 실패시키지 않는다.
- 외부 서비스와 작업 종류별 실패를 분류해 재시도, 취소 또는 영구 실패로 끝낸다.

## 재시도와 실패

- 일시 오류는 지수적으로 늘어나는 확정 간격에 jitter를 적용해 재시도한다.
- 최대 시도 횟수를 넘기거나 계약·권한·설정 오류가 확인되면 영구 실패로 기록한다.
- 만료된 토큰, 이미 복원된 리소스처럼 현재 상태에서 무효한 이벤트는 취소한다.
- lock을 잃은 Worker는 완료나 실패 상태를 덮어쓰지 않는다.
- 운영자는 원인을 수정하고 외부 부작용 여부를 확인한 뒤 선택한 이벤트만 재시도한다.
- payload를 데이터베이스에서 즉석 수정하거나 실패 이벤트를 삭제해 해결하지 않는다.

정확한 retry 간격과 분류는
[`apps/worker/src/modules/outbox`](../../apps/worker/src/modules/outbox/)를 정본으로 한다.

## 인앱 알림과 Web Push

- `Notification`이 제품 알림과 읽음 상태의 원본이다.
- Web Push는 알림을 브라우저에 전달한 결과를 별도 기록하는 채널이다.
- 사용자의 명시적 동작에서만 브라우저 권한을 요청하고 세션별 구독을 만든다.
- 세션이 폐기·만료되거나 공급자가 endpoint 만료를 반환하면 구독의 endpoint와 키를 제거한다.
- Push payload는 업무 본문을 담지 않고 사용자가 로그인 후 권한을 다시 확인할 내부 경로만 제공한다.
- Push 실패나 구독 부재가 인앱 알림 생성을 취소하지 않는다.

## 인증 이메일

- 계정 확인, 비밀번호 재설정과 초대 이벤트만 Resend로 전달한다.
- Worker는 전송 전에 토큰과 대상 흐름이 아직 유효한지 다시 확인한다.
- 개발·테스트는 `RESEND_ALLOWED_RECIPIENTS` 밖의 주소로 보내지 않는다.
- 공급자 영구 오류는 무한 재시도하지 않으며 만료된 링크는 새 발급 흐름으로 해결한다.
- 제품 업무 알림을 이메일로 추가하지 않는다.

## SSE

- SSE는 데이터 변경 사실을 알려 브라우저가 REST 데이터를 다시 조회하게 하는 신호다.
- PostgreSQL listener가 준비되지 않으면 신규 SSE 연결을 `503`으로 거부하고 가능한 REST API는 유지한다.
- 연결은 heartbeat로 생존을 확인하고 브라우저는 끊김 뒤 다시 연결한다.
- 과거 이벤트 전체를 재생하는 로그로 사용하지 않는다.
- 재연결 뒤 TanStack Query가 권한 있는 최신 REST 상태로 수렴해야 한다.

## 정기 작업

- Worker는 UTC 기준 정기 시각에 보존 데이터와 파일을 정리한다.
- PostgreSQL advisory lock으로 한 프로세스만 정리하고 작은 batch와 `SKIP LOCKED`를 사용한다.
- 한 정리 단계가 실패해도 독립 단계는 계속 수행하고 실패 단계와 job ID를 경고한다.
- 참조 중인 파일이나 현재 상태가 달라진 삭제 이벤트는 건너뛴다.

## 관측 연동

- PostHog에는 허용 목록의 제품 이벤트와 정제된 예외만 보낸다.
- 동일 이벤트 재처리로 분석 이벤트가 중복되지 않도록 안정적인 event ID를 사용한다.
- Slack은 긴급·높음 수준의 운영 경고 전달에 사용한다.
- 공급자 호출은 짧은 timeout을 사용하고 실패를 정제 코드로 기록한다.
- 공급자 응답 본문, webhook URL, API 키와 사용자 업무 내용은 로그에 남기지 않는다.

외부 연동의 환경·점검 절차는
[보안 운영과 외부 연동](../operations/security-and-integrations.md)을 따른다.
