# 시스템 경계

> 상태: 현행
> 적용 범위: 모노레포 앱·패키지 책임과 동기·비동기 데이터 흐름
> 정본 책임: 코드 구조만으로 드러나지 않는 서비스 경계
> 갱신 조건: 앱·패키지 책임, 주요 데이터 흐름 또는 외부 경계 변경

## 실행 구성

Rivet은 하나의 저장소와 PostgreSQL을 공유하는 세 실행 프로세스로 구성한다.

| 프로세스 | 책임 | 하지 않는 일 |
| --- | --- | --- |
| `apps/web` | 화면, 브라우저 상태, 사용자 상호작용과 API 소비 | 업무 규칙을 별도 쓰기 경로로 구현하지 않음 |
| `apps/api` | 인증된 HTTP 계약, 권한·검증, 트랜잭션과 Outbox 생산 | 이메일·Push 전송과 예약 정리를 직접 수행하지 않음 |
| `apps/worker` | Outbox 소비, 이메일·Web Push, 삭제·보존과 파일 정리 | 사용자 요청용 HTTP 계약을 제공하지 않음 |

운영에서는 세 프로세스가 같은 릴리스 ID, 데이터베이스와 파일 저장소를 사용한다.

## 공유 패키지

| 패키지 | 책임 |
| --- | --- |
| `packages/database` | Prisma schema, migration, 생성 Client와 DB 연결 기반 |
| `packages/api-client` | OpenAPI에서 생성한 Fetch 함수, 타입과 TanStack Query Hook |
| `packages/event-contracts` | API 생산자와 Worker 소비자가 공유하는 버전 이벤트 계약 |
| `packages/config` | TypeScript·ESLint 등 저장소 공통 개발 설정 |

공유 패키지는 앱 내부 서비스나 HTTP DTO를 재수출하지 않는다. 두 실행 경계가 실제로 공유하는
계약만 공개하고, 앱 하나에서만 쓰는 코드는 해당 앱에 둔다.

## 요청 흐름

1. 브라우저는 같은 origin의 웹과 `/api/v1`을 사용한다.
2. Nginx가 웹 요청은 `apps/web`, API·SSE·CSV 요청은 `apps/api`로 전달한다.
3. API는 세션에서 사용자·워크스페이스·멤버십 범위를 확정한다.
4. 서비스가 권한과 입력을 검증하고 필요한 행을 잠근 뒤 하나의 트랜잭션으로 업무 데이터,
   활동과 Outbox를 기록한다.
5. 응답 뒤 웹은 생성 API 클라이언트와 TanStack Query 캐시를 통해 화면을 수렴시킨다.

웹은 Prisma를 import하거나 임의 API URL을 조합하지 않는다. 엔드포인트와 DTO의 정본은
[`apps/api/openapi/openapi.json`](../../apps/api/openapi/openapi.json)이며 변경 시 생성
클라이언트를 같은 작업에서 갱신한다.

## 비동기 흐름

1. API가 원본 상태 변경과 Outbox 이벤트를 같은 트랜잭션에 기록한다.
2. Worker가 처리 가능한 이벤트를 잠금과 임대 방식으로 가져온다.
3. 이벤트 버전을 검증하고 현재 데이터 상태를 다시 확인한다.
4. 외부 부작용이나 정리 작업을 멱등하게 실행한다.
5. 성공, 재시도, 취소 또는 영구 실패를 기록한다.

이메일·Push·정리 작업 실패는 이미 성공한 원본 제품 트랜잭션을 되돌리지 않는다.

## 파일 경계

- API와 Worker는 같은 절대 `FILE_STORAGE_ROOT`를 사용한다.
- 파일 메타데이터는 PostgreSQL, 바이너리는 파일 저장소에 둔다.
- 브라우저는 저장소 경로를 알지 못하고 인증된 API로 업로드·조회한다.
- Nginx가 파일 저장소를 정적 디렉터리로 노출하지 않는다.
- 업로드 임시 파일과 연결 해제 파일은 Worker가 참조 상태를 확인한 뒤 정리한다.

## 외부 서비스 경계

| 서비스 | 용도 | 실패 영향 |
| --- | --- | --- |
| Resend | 계정 확인, 재설정과 초대 이메일 | 해당 비동기 전달만 재시도·실패 |
| Web Push 공급자 | 사용자가 구독한 제품 알림 전달 | 인앱 알림 원본은 유지 |
| PostHog | 허용된 제품 이벤트와 정제된 예외 관측 | 제품 요청과 Outbox 완료를 실패시키지 않음 |
| Slack Incoming Webhook | 높은 수준의 운영 경고 | 제품 트랜잭션을 실패시키지 않음 |

외부 서비스의 키, 응답 본문과 사용자 업무 내용은 공유 이벤트나 일반 로그에 넣지 않는다.

## 확장 판단

현재 구성은 단일 PostgreSQL, 로컬 파일 저장소와 PM2 단일 프로세스 인스턴스를 전제로 한다.
서비스 분리, 별도 큐, 객체 저장소, 다중 인스턴스와 무중단 배포는 실제 용량·가용성·복구
요구가 현재 구성을 반복적으로 초과할 때 검토한다.
