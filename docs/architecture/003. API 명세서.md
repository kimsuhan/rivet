## 1. 문서 목적

이 문서는 Rivet MVP의 웹 애플리케이션과 API 서버 사이의 REST 계약을 정의한다. 프론트엔드가 데이터베이스 구조나 백엔드 내부 DTO를 추측하지 않고 구현할 수 있도록 공통 규약, 리소스별 엔드포인트, 요청·응답 DTO, 오류 코드, 파일 업로드와 동시 수정 방식을 명시한다.

OpenAPI 문서는 이 명세를 구현 가능한 기계 판독 계약으로 옮긴 결과물이어야 한다. 이 문서와 OpenAPI가 다르면 배포 전 반드시 하나로 맞춘다.

### 1.1 범위

- 회원가입, 이메일 인증, 로그인, 비밀번호 재설정과 세션
- 현재 사용자, 워크스페이스, 멤버와 초대
- 팀, 팀 멤버, 워크플로 상태와 라벨
- 프로젝트와 역할별 담당 팀
- 이슈, 팀 실행 작업, 필터·정렬과 전달 기반 준비 상태
- 설명, 댓글, 멘션, 활동과 API 작업 전달
- 파일 업로드, 이슈 첨부파일, 본문 이미지와 프로필 사진
- 알림함, 검색, 휴지통, CSV 내보내기와 SSE 연결
- 권한, 페이지네이션, 충돌과 공통 오류 계약

### 1.2 제외 범위

- Public API, 개인 액세스 토큰과 OAuth
- GraphQL, 웹훅과 외부 서비스 연동 API
- 관리자용 서버 운영·작업 큐 제어 API
- 워커 내부 Outbox payload 전체와 재시도 알고리즘
- 비밀번호 세부 정책, 속도 제한 수치와 쿠키 속성의 최종값
- SSE 이벤트별 수신자 계산과 전체 이벤트 카탈로그
- OpenAPI YAML 원문과 NestJS DTO 구현 코드

## 2. API 기본 계약

### 2.1 기본 경로와 형식

| 항목 | 계약 |
| --- | --- |
| 기본 경로 | `/api/v1` |
| 기본 요청·응답 | `application/json; charset=utf-8` |
| 파일 업로드 | `multipart/form-data` |
| CSV | `text/csv; charset=utf-8` |
| 실시간 연결 | `text/event-stream` |
| 필드명 | 영문 `camelCase` |
| 내부 리소스 ID | UUID 문자열 |
| 시각 | UTC 기준 ISO 8601 문자열, 예: `2026-07-10T03:12:45.000Z` |
| 날짜 | 시간대 없는 `YYYY-MM-DD` 문자열 |

- 운영 웹과 API는 같은 외부 출처를 사용한다. Nginx가 `/`는 웹, `/api/v1`과 인증 파일 경로는 API로 전달하며 운영 브라우저 CORS는 활성화하지 않는다.
- 업무 API는 현재 로그인 사용자의 단일 활성 워크스페이스를 서버에서 결정한다. 클라이언트가 임의의 `workspaceId`를 요청 본문에 넣어 접근 범위를 선택하지 않는다.
- 다른 워크스페이스의 리소스, 삭제된 리소스와 접근할 수 없는 리소스는 존재 여부를 구분하지 않고 `404 RESOURCE_NOT_FOUND`로 응답한다.
- 응답 DTO는 데이터베이스 행이나 Prisma 모델을 그대로 노출하지 않는다.
- 요청에서 생략한 필드는 변경하지 않는다. 명시적 `null`은 해당 필드가 선택값이고 해제가 허용될 때만 사용한다.
- API enum은 안정적인 영문 대문자 코드를 사용하고 한국어 표시는 이 문서의 대응표를 기준으로 프론트에서 표현한다.
- 모든 응답은 `X-Request-ID` 헤더를 포함하며 오류 본문의 `requestId`와 같은 값이다.

### 2.2 성공 응답

단일 리소스는 불필요한 공통 래퍼 없이 객체를 바로 반환한다.

```json
{
  "id": "7cb4a651-5eaa-4af3-bba5-f80cf70d8f45",
  "name": "프론트 웹",
  "key": "WEB",
  "version": 3
}
```

목록은 다음 형식을 사용한다.

```json
{
  "items": [],
  "nextCursor": null
}
```

| 동작 | 상태 코드 |
| --- | ---: |
| 조회·수정·도메인 동작 성공 | `200` |
| 리소스 생성 | `201` |
| 계정 존재 여부를 숨기는 인증 요청 접수 | `202` |
| 응답 본문이 없는 삭제·로그아웃 | `204` |
| CSV·파일 스트리밍 | `200` |

### 2.3 오류 응답

오류는 다음 객체를 직접 반환한다.

```json
{
  "code": "VERSION_CONFLICT",
  "message": "리소스가 다른 요청에서 변경되었습니다.",
  "fieldErrors": {},
  "requestId": "req_01J2V7BRK7PKJY6HPN21TX4ZB5",
  "currentVersion": 8,
  "details": {
    "resourceId": "7cb4a651-5eaa-4af3-bba5-f80cf70d8f45"
  }
}
```

| 필드 | 형식 | 규칙 |
| --- | --- | --- |
| `code` | 문자열 | 프론트 분기에 사용하는 안정적인 코드 |
| `message` | 문자열 | 개발·기본 안내용 한국어 설명. 화면 문구의 유일한 원천으로 사용하지 않음 |
| `fieldErrors` | `Record<string, string[]>` | 필드 오류가 없으면 빈 객체 |
| `requestId` | 문자열 | 모든 오류에 포함하는 운영 추적 ID |
| `currentVersion` | 정수 | `VERSION_CONFLICT`일 때만 포함 |
| `details` | 객체 | 차단 리소스처럼 화면 복구에 필요한 비민감 구조화 정보가 있을 때만 포함 |

인증 정보, 이메일 존재 여부, 다른 워크스페이스의 ID와 업무 본문은 오류에 포함하지 않는다.

### 2.4 공통 상태 코드

| 상태 | 용도 |
| ---: | --- |
| `400` | JSON 구문, 커서 또는 쿼리 형식 오류 |
| `401` | 로그인 세션 없음·만료 |
| `403` | 인증됐지만 동작 권한 없음, 이메일 미인증, CSRF 실패 |
| `404` | 없거나 접근할 수 없는 리소스 |
| `409` | 버전 충돌, 중복, 관계·현재 상태 충돌 |
| `410` | 존재했던 일회용 토큰이 만료됨 |
| `413` | 업로드 파일 25MB 초과 |
| `415` | 해당 사용 위치에서 허용하지 않는 파일 형식 |
| `422` | 형식은 맞지만 필드·도메인 검증 실패 |
| `429` | 요청 속도 제한 초과 |
| `503` | 메타데이터는 있으나 파일 바이너리를 사용할 수 없는 운영 불일치 |

### 2.5 인증과 CSRF

- 세션 식별자는 JavaScript에서 읽을 수 없는 보안 쿠키로 전달한다.
- 인증된 상태 변경 요청은 `X-CSRF-Token` 헤더를 필수로 사용한다.
- `GET /auth/session`과 로그인 성공 응답에서 현재 세션용 `csrfToken`을 제공한다.
- `401 SESSION_REQUIRED`를 받으면 프론트는 현재 주소를 보존하고 로그인으로 이동한다.
- `403 EMAIL_NOT_VERIFIED`와 `403 MEMBERSHIP_INACTIVE`는 로그인 실패 또는 업무 접근 차단 흐름으로 처리한다.
- 구체적인 쿠키 이름, `SameSite`, Origin 검사와 토큰 갱신 규칙은 보안 명세서에서 확정한다.

### 2.6 페이지네이션

| 파라미터 | 기본값 | 규칙 |
| --- | ---: | --- |
| `limit` | `50` | 최소 1, 최대 100 |
| `cursor` | 없음 | 서버가 발급한 불투명 문자열 |

- 모든 정렬은 불변 `id`를 마지막 동률 해소 값으로 사용하며 정렬 방향을 주 정렬 값과 `id`에 일관되게 적용한다.
- 커서는 서버가 정규화한 현재 정렬 기준의 실제 값과 마지막 항목의 `id`를 포함한다. 상태 정렬처럼 값이 여러 개인 경우 전체 정렬 튜플을 포함한다.
- 클라이언트는 커서를 해석하거나 직접 만들지 않는다.
- 커서는 발급받은 워크스페이스에서 같은 정렬 기준·방향·필터 조건의 다음 페이지를 조회할 때만 사용한다. 조건을 바꾸면 기존 커서를 폐기하고 처음부터 조회한다.
- `nextCursor = null`이면 다음 페이지가 없다.
- 임의 페이지 번호와 큰 `offset`은 제공하지 않는다.

### 2.7 필터와 정렬

- 같은 필드의 여러 값은 쉼표로 구분하고 OR로 적용한다. 예: `priority=HIGH,URGENT`.
- 서로 다른 필드는 AND로 적용한다.
- 빈 문자열과 알 수 없는 enum은 `400 INVALID_QUERY`로 거부한다.
- 이슈 정렬은 `createdAt`, `updatedAt`, `status`, `priority`, `progress`를 허용한다. 팀 작업 정렬은 `createdAt`, `updatedAt`, `status`, 상위 이슈의 `priority`를 허용한다.
- `sortDirection`은 `asc` 또는 `desc`다.
- 이슈와 프로젝트의 기본 정렬은 모두 `updatedAt desc`다.
- 이슈 우선순위 오름차순은 `NONE < LOW < MEDIUM < HIGH < URGENT`이고 내림차순은 그 역순이다. 같은 우선순위는 `id`로 안정 정렬한다.
- 이슈 상태 정렬은 `IssueStatus`의 고정 순서와 `id`, 팀 작업 상태 정렬은 `StateCategory → workflowState.position → id` 튜플을 사용한다. `StateCategory` 오름차순은 `BACKLOG < UNSTARTED < STARTED < COMPLETED < CANCELED`다.
- 오름차순과 내림차순은 상태 정렬 튜플 전체와 마지막 `id`에 같은 방향으로 적용한다.
- 프로젝트 목표일 정렬에서 `null`은 정렬 방향과 관계없이 마지막에 둔다.
- 목록과 보드는 같은 이슈 목록 API와 필터 의미를 사용한다. 보드는 팀의 상태별로 같은 API를 조회해 열을 구성하며 수동 순서를 저장하지 않는다.

### 2.8 동시 수정

사용자가 수정하는 주요 리소스의 `PATCH`와 상태 전환 요청에는 마지막으로 조회한 `version`을 포함한다.

```json
{
  "version": 7,
  "title": "이메일 인증 오류 처리"
}
```

- 서버 버전이 다르면 변경하지 않고 `409 VERSION_CONFLICT`와 `currentVersion`을 반환한다.
- 성공하면 응답 리소스의 `version`이 1 증가한다.
- 설명·댓글·작업 전달 Markdown은 자동 병합하거나 마지막 쓰기 우선으로 덮어쓰지 않는다.
- 관계 추가처럼 기존 값을 덮어쓰지 않는 원자 동작도 응답에 영향받은 리소스의 최신 버전을 반환한다.

### 2.9 중복 요청

- MVP에서는 범용 `Idempotency-Key` 저장소를 추가하지 않는다.
- 프론트는 제출 중 버튼을 비활성화하고 동일 요청의 동시 전송을 막는다.
- 서버는 이메일, 초대, 관계, 최초 작업 전달과 알림의 유일 제약으로 중복 결과를 막는다.
- 생성 요청이 응답 전에 끊기면 클라이언트는 목록을 다시 조회해 결과 존재 여부를 확인한 뒤 재시도한다.

## 3. 공통 표현

### 3.1 enum과 한국어 표시

| API enum | 값과 한국어 표시 |
| --- | --- |
| `MembershipRole` | `ADMIN` 관리자, `MEMBER` 멤버 |
| `MembershipStatus` | `ACTIVE` 활성, `INACTIVE` 비활성 |
| `IssueStatus` | `UNSORTED` 접수됨, `TODO` 할 일, `IN_PROGRESS` 진행 중, `REVIEW` 완료 확인, `DONE` 완료, `PAUSED` 일시 중지, `CANCELED` 취소 |
| `IssueStatusAction` | `PAUSE` 일시 중지, `RESUME` 재개, `CANCEL` 취소, `COMPLETE` 완료, `REOPEN` 다시 열기 |
| `StateCategory` | `BACKLOG` 백로그, `UNSTARTED` 시작 전, `STARTED` 진행, `COMPLETED` 완료, `CANCELED` 취소 |
| `IssuePriority` | `NONE` 없음, `LOW` 낮음, `MEDIUM` 중간, `HIGH` 높음, `URGENT` 긴급 |
| `ProjectRole` | `BACKEND` 백엔드, `WEB_FRONTEND` 웹 프론트, `APP_FRONTEND` 앱 프론트 |
| `ProjectStatus` | `PLANNED` 계획, `IN_PROGRESS` 진행 중, `COMPLETED` 완료, `CANCELED` 취소 |
| `HandoffKind` | `INITIAL` 최초 전달, `FOLLOW_UP` 추가 전달 |
| `NotificationType` | `TEAM_WORK_ASSIGNED`, `MENTIONED`, `COMMENT_ADDED`, `ISSUE_COMPLETED`, `ISSUE_CANCELED`, `API_HANDOFF_CREATED`, `API_HANDOFF_FOLLOW_UP_CREATED` |
| `FileScope` | `USER_PROFILE` 사용자 프로필, `WORKSPACE` 워크스페이스 업무 파일 |
| `IssueFileKind` | `ISSUE_ATTACHMENT`, `DESCRIPTION_IMAGE`, `COMMENT_IMAGE`, `HANDOFF_IMAGE` |

알림 유형의 최종 한국어 문장은 현재 이슈·사용자 정보를 조합해 화면에서 만든다. 서버는 번역된 문장 스냅샷을 저장하거나 응답하지 않는다.

### 3.2 사용자와 멤버

```ts
type IssueStatus = "UNSORTED" | "TODO" | "IN_PROGRESS" | "REVIEW" | "DONE" | "PAUSED" | "CANCELED";
type IssueStatusAction = "PAUSE" | "RESUME" | "CANCEL" | "COMPLETE" | "REOPEN";
type StateCategory = "BACKLOG" | "UNSTARTED" | "STARTED" | "COMPLETED" | "CANCELED";
type IssuePriority = "NONE" | "LOW" | "MEDIUM" | "HIGH" | "URGENT";
type ProjectRole = "BACKEND" | "WEB_FRONTEND" | "APP_FRONTEND";
type NotificationType =
  | "TEAM_WORK_ASSIGNED"
  | "MENTIONED"
  | "COMMENT_ADDED"
  | "ISSUE_COMPLETED"
  | "ISSUE_CANCELED"
  | "API_HANDOFF_CREATED"
  | "API_HANDOFF_FOLLOW_UP_CREATED";

type UserSummary = {
  id: string;
  displayName: string;
  avatarFileId: string | null;
};

type MemberSummary = {
  id: string; // workspace membership ID
  user: UserSummary;
  role: "ADMIN" | "MEMBER";
  status: "ACTIVE" | "INACTIVE";
};

type TeamSummary = {
  id: string;
  name: string;
  key: string;
  archived: boolean;
};

type WorkflowStateSummary = {
  id: string;
  name: string;
  category: "BACKLOG" | "UNSTARTED" | "STARTED" | "COMPLETED" | "CANCELED";
  position: number;
  isDefault: boolean;
  version: number;
};

type LabelSummary = {
  id: string;
  name: string;
  color: string;
  archived: boolean;
};

type ProjectSummary = {
  id: string;
  name: string;
  status: "PLANNED" | "IN_PROGRESS" | "COMPLETED" | "CANCELED";
};
```

- 업무 리소스의 작성자·담당자·멘션 대상은 계정 ID가 아니라 멤버십 ID로 참조한다.
- 이메일은 멤버 설정 화면처럼 권한과 목적이 있는 응답에서만 포함한다.
- `avatarFileId`가 `null`이면 프론트는 `displayName`의 첫 글자를 표시한다.

### 3.3 Markdown과 멘션

- 설명은 최대 100,000자, 댓글과 작업 전달은 각각 최대 50,000자다.
- API는 Markdown 문자열만 받고 Lexical JSON과 렌더링 HTML은 받지 않는다.
- 멘션은 이슈 설명과 댓글에서만 `@[표시 이름](rivet-member:{membershipId})` 형식으로 직렬화한다.
- 서버는 설명과 댓글 Markdown을 파싱해 멘션 ID가 현재 워크스페이스의 활성 멤버인지 확인하고 별도 멘션 관계를 동기화한다.
- 작업 전달은 멘션을 지원하지 않으며 멘션 참조가 포함된 본문을 `422 MARKDOWN_INVALID`로 거부한다.
- 본문 이미지는 `![대체 텍스트](/files/{fileId})` 형식만 업무 파일 참조로 인식한다.
- 서버는 본문에 포함된 파일이 업로더 소유의 미연결 파일이거나 이미 같은 본문에 연결된 파일인지 검증한다.
- 외부 이미지 URL과 Data URL은 저장 시 거부한다. 일반 외부 링크는 안전한 `http` 또는 `https` URL만 허용한다.
- 파일 업로드가 끝나지 않은 임시 브라우저 URL은 요청에 포함하지 않는다.

### 3.4 파일 표현

```ts
type FileResource = {
  id: string;
  scope: "USER_PROFILE" | "WORKSPACE";
  originalName: string;
  detectedMimeType: string;
  sizeBytes: number;
  inlineDisplayable: boolean;
  linked: boolean;
  createdAt: string;
};

type IssueAttachment = {
  id: string;
  kind: "ISSUE_ATTACHMENT" | "DESCRIPTION_IMAGE" | "COMMENT_IMAGE" | "HANDOFF_IMAGE";
  file: FileResource;
  uploader: UserSummary;
  createdAt: string;
};
```

- `inlineDisplayable`은 서버가 실제 형식을 확인한 JPEG·PNG·WebP·GIF에만 `true`다.
- 프로필 사진 연결에는 JPEG·PNG·WebP만 허용한다.
- 파일 바이너리 URL은 영구 공개 URL이 아니다. 프론트는 인증된 파일 스트리밍 엔드포인트를 사용한다.

### 4.2 주요 요청·응답

`POST /auth/sign-up`

```json
{
  "displayName": "김민수",
  "email": "minsu@example.com",
  "password": "사용자 입력 비밀번호"
}
```

```json
{
  "accepted": true,
  "emailMasked": "mi***@example.com"
}
```

문법과 비밀번호 정책을 통과한 회원가입은 계정 존재 여부와 관계없이 같은 `202`와 응답 본문을 반환한다. 신규 이메일만 계정과 인증 메일을 만들고, 기존 미인증 계정은 제한 안에서 인증 메일을 다시 보낼 수 있으며, 기존 인증 계정은 아무 계정 변경도 하지 않는다. 인증 메일 재발송과 비밀번호 재설정 요청도 대상 존재 여부를 구분하지 않는 공통 응답을 사용한다.

- `POST /auth/email-verifications/verify`: `{ "token": "..." }` → `200 { "verified": true }`
- `POST /auth/email-verifications/resend`: `{ "email": "minsu@example.com" }` → `202 { "accepted": true, "emailMasked": "mi***@example.com" }`
- `POST /auth/password-resets/request`: `{ "email": "minsu@example.com" }` → 본문 없는 `204`
- `POST /auth/password-resets/confirm`: `{ "token": "...", "password": "새 비밀번호" }` → `200 { "reset": true }`

`POST /auth/login`

```json
{
  "email": "minsu@example.com",
  "password": "사용자 입력 비밀번호"
}
```

성공 응답은 `GET /auth/session`의 인증 상태와 같고 세션 쿠키를 함께 설정한다.

```json
{
  "authenticated": true,
  "csrfToken": "세션용 CSRF 토큰",
  "user": {
    "id": "d8ccff9e-52bc-48f9-bb47-9f996d670f80",
    "displayName": "김민수",
    "avatarFileId": null,
    "email": "minsu@example.com"
  },
  "membership": null,
  "workspace": null,
  "onboardingStep": "CREATE_WORKSPACE"
}
```

`onboardingStep`은 `CREATE_WORKSPACE`, `CREATE_TEAM`, `COMPLETE` 중 하나다. 이메일 미인증 사용자는 업무 세션을 만들지 않고 `EMAIL_NOT_VERIFIED`로 재발송 흐름을 안내한다.

세션이 없을 때 `GET /auth/session`은 오류 대신 `{ "authenticated": false }`를 반환한다. 이를 통해 웹 애플리케이션이 최초 로딩에서 로그인 화면과 업무 화면을 결정할 수 있다.

`POST /auth/invitations/preview`와 `POST /auth/invitations/accept`는 `/invite#token=...` 링크의 fragment를 메모리로 읽고 주소에서 제거한 뒤 토큰을 `{ "token": "..." }` 본문으로 받는다. 미리보기는 워크스페이스 이름, 가린 초대 이메일, 초대한 사용자 표시 이름과 만료 시각만 반환하며 초대 수락은 이메일 인증을 마친 로그인 계정에만 허용한다. 인증이 필요하면 가입·인증·로그인 후 원래 초대 메일의 링크를 다시 열고, 토큰 원문은 브라우저 저장소나 쿠키에 보관하지 않는다.

인증·초대·재설정 토큰은 URL에서 받은 뒤 API 요청 본문으로 전달하며 서버 로그와 분석 이벤트에 남기지 않는다.

## 6. 멤버와 초대 API

### 6.1 엔드포인트

| 메서드 | 경로 | 권한 | 용도 |
| --- | --- | --- | --- |
| `GET` | `/members` | 활성 멤버 | 활성·비활성 멤버 목록 |
| `GET` | `/members/{membershipId}` | 활성 멤버 | 멤버 상세와 팀 요약 |
| `POST` | `/members/{membershipId}/deactivate` | 관리자 | 조건 확인 후 멤버 비활성화 |
| `GET` | `/invitations` | 관리자 | 대기·수락·취소·만료 초대 목록 |
| `POST` | `/invitations` | 관리자 | 이메일 여러 건 초대 |
| `POST` | `/invitations/{invitationId}/resend` | 관리자 | 초대 재발송 또는 종료 이력 재초대 |
| `POST` | `/invitations/{invitationId}/cancel` | 관리자 | 대기 초대 취소 |

`GET /members`는 `status=ACTIVE,INACTIVE`, `teamId`, `query`, 커서를 지원한다. 멘션·담당자 선택기는 `status=ACTIVE`와 필요한 `teamId`를 사용한다.

`GET /invitations`는 `status=PENDING,ACCEPTED,CANCELED,EXPIRED` 쉼표 OR 필터, `limit`과 `createdAt + id` 기반 불투명 커서를 지원하며 응답의 `nextCursor`로 다음 페이지를 조회한다.

`POST /invitations`

```json
{
  "emails": ["web@example.com", "app@example.com"]
}
```

입력은 한 번에 최대 50개이며 정규화 후 중복을 합친다. 주소별 처리 결과를 반환한다.

```json
{
  "items": [
    { "email": "web@example.com", "result": "INVITED", "invitationId": "..." },
    { "email": "app@example.com", "result": "ALREADY_INVITED", "invitationId": "..." }
  ]
}
```

`result`는 `INVITED`, `ALREADY_MEMBER`, `ALREADY_INVITED`, `FAILED`다. 일부 주소 실패가 전체 성공을 롤백하지 않는다.

`POST /invitations/{invitationId}/resend`는 대기 중이거나 만료됐지만 종료되지 않은 초대의 발신자·만료 시각과 토큰을 같은 행에서 갱신한다. 수락·취소된 초대는 원본 이력을 유지하고 같은 이메일의 새 초대 행과 새 토큰을 만든다. 응답은 실제 유효한 최소 `InvitationResource`이며 종료 이력 재초대에서는 요청 경로와 다른 새 `id`를 반환한다. 같은 이메일의 다른 미종료 초대가 이미 있으면 `409 INVITATION_ALREADY_PENDING`으로 거부하고, 새 토큰 발급과 동시에 이전 미사용 토큰은 사용할 수 없게 한다.

종료 이력에서 재발급된 링크를 같은 워크스페이스의 기존 활성 `MEMBER`가 수락하면 새 멤버십을 만들지 않고 새 초대 행과 토큰만 종료하며 기존 멤버십·워크스페이스를 수락 응답으로 반환한다. 일반 `POST /invitations`의 기존 멤버 주소별 결과는 계속 `ALREADY_MEMBER`다. 취소와 수락도 각각 변경된 최소 초대 또는 멤버십·워크스페이스 결과를 반환한다.

멤버 비활성화 요청에는 `version` 대신 현재 미완료 할당을 트랜잭션에서 다시 확인한다. 차단되면 `409 MEMBER_HAS_OPEN_ASSIGNMENTS`와 `details.issues`에 접근 가능한 이슈의 `id`, `identifier`, `title`을 반환한다.

```json
{
  "name": "프론트 웹",
  "key": "WEB",
  "memberIds": ["현재 생성자 멤버십 ID"]

```

성공 응답에는 팀과 생성된 7개 기본 `workflowStates`를 포함한다. 키는 영문 대문자 2~5자이며 첫 팀 작업 번호가 발급된 뒤에는 수정할 수 없다.

팀 멤버 제거가 미완료 할당 때문에 차단되면 `TEAM_MEMBER_HAS_OPEN_ASSIGNMENTS`, 팀 보관이 미완료 작업 때문에 차단되면 `TEAM_HAS_OPEN_ISSUES`와 대상 이슈 요약을 반환한다.

### 7.2 워크플로 엔드포인트

| 메서드 | 경로 | 권한 | 용도 |
| --- | --- | --- | --- |
| `GET` | `/teams/{teamId}/workflow-states` | 활성 멤버 | 팀 상태를 `position` 순서로 조회 |
| `PATCH` | `/workflow-states/{stateId}` | 관리자 | 상태 이름 수정 |
| `PUT` | `/teams/{teamId}/workflow-states/order` | 관리자 | 전체 상태 순서를 원자적으로 교체 |
| `DELETE` | `/workflow-states/{stateId}` | 관리자 | 미사용 상태 삭제 또는 대체 상태로 이동 후 삭제 |

상태의 시스템 `category`는 수정할 수 없다.

```json
{
  "states": [
    { "id": "state-1", "version": 2 },
    { "id": "state-2", "version": 4 }
  ]
}
```

순서 변경 요청에는 현재 팀 상태 전체를 누락 없이 한 번씩 전달한다. 상태 삭제는 사용 중이면 `replacementStateId` 쿼리를 필수로 받고 대상 이슈 이동과 기본 상태 이전을 한 트랜잭션에서 처리한다.

상태 삭제 요청에는 대상 상태의 `version`도 쿼리로 전달한다. 대체 상태는 같은 팀에 속하고 삭제 대상과 달라야 한다.

| `GET` | `/projects/{projectId}` | 활성 멤버 | 프로젝트 상세·진행률 조회 |
| `PATCH` | `/projects/{projectId}` | 활성 멤버 | 기본 정보·상태·역할별 팀 수정 |
| `POST` | `/projects/{projectId}/archive` | 활성 멤버 | 프로젝트 보관 |
| `POST` | `/projects/{projectId}/trash` | 활성 멤버 | 비어 있는 프로젝트를 휴지통으로 이동 |

`GET /projects`는 `status`, `leadMembershipId`, `includeArchived=false`, `sort=updatedAt|targetDate`, `sortDirection`, 커서를 지원한다. 기본 정렬은 `updatedAt desc`이며 모든 정렬은 `id`를 마지막 동률 해소 값으로 사용한다. `targetDate`의 `null`은 `asc`, `desc` 모두 마지막이고 커서는 실제 정렬 값 또는 `null`과 `id`를 포함한다.

프로젝트 보관·휴지통 이동 요청 본문은 `{ "version": 현재버전 }`이다.

## 10. 이슈와 팀 작업 API

### 10.1 리소스 방향

`Issue`는 제목·설명·프로젝트·우선순위·라벨·댓글·첨부파일의 유일한 콘텐츠 리소스다. `TeamWork`는 이슈에 속한 팀 실행 상태이며 팀·역할·워크플로 상태·담당자·짧은 작업 메모만 가진다. 두 리소스를 `type`으로 섞지 않으며 상위 이슈 없는 팀 작업 생성 API도 제공하지 않는다.

### 10.2 엔드포인트

| 메서드 | 경로 | 권한 | 용도 |
| --- | --- | --- | --- |
| `GET` | `/issues` | 활성 멤버 | 전역·프로젝트 이슈 목록 조회 |
| `POST` | `/issues` | 활성 멤버 | 이슈와 선택한 초기 역할의 팀 작업 생성 |
| `GET` | `/issues/{issueRef}` | 활성 멤버 | UUID 또는 `F-*` 표시 ID로 통합 상세 조회 |
| `PATCH` | `/issues/{issueId}` | 활성 멤버 | 이슈 콘텐츠·속성·명시적 상태 행동 수정 |
| `POST` | `/issues/{issueId}/trash` | 활성 멤버 | 이슈와 소속 팀 작업 휴지통 이동 |
| `POST` | `/issues/{issueId}/team-works` | 활성 멤버 | 선택 역할의 팀 작업 시작 또는 재사용 |
| `POST` | `/issues/{issueId}/claim` | 대상 역할 팀의 활성 멤버 | 미할당 작업 재사용 또는 생성 후 자신에게 할당 |
| `POST` | `/issues/{issueId}/assign-team-works` | 활성 멤버 | 여러 미할당 팀 작업 담당자 일괄 지정 |
| `GET` | `/team-works` | 활성 멤버 | 팀 목록·내 작업 공통 조회 |
| `GET` | `/team-works/{teamWorkRef}` | 활성 멤버 | UUID 또는 팀 표시 ID로 팀 작업과 상위 이슈 요약 조회 |

`GET /team-works/{teamWorkRef}`는 별도 상세 화면용이 아니라 표시 ID 해석, 알림과 딥 링크 진입에 사용한다. 웹은 응답의 이슈 표시 ID로 `/issues/{issueIdentifier}?tab=work&work={teamWorkIdentifier}`를 연다.

### 10.3 목록 쿼리

`GET /issues`는 다음 필터를 지원한다.

| 파라미터 | 값 |
| --- | --- |
| `query` | 이슈 표시 ID 또는 제목 검색어 |
| `workQueue` | `REVIEW_REQUIRED`, `ASSIGNMENT_REQUIRED`, `IN_PROGRESS`, `COMPLETION_REQUIRED`, `COMPLETED` 중 하나 |
| `projectId`, `activeProjectRole` | 프로젝트와 현재 미완료 역할 목록 |
| `status`, `priority`, `labelId` | 이슈 속성 목록 |
| `createdByMembershipId` | 이슈 생성자 목록 |
| `createdFrom`, `createdTo`, `updatedFrom`, `updatedTo` | 시각 범위 |
| `sort` | `createdAt`, `updatedAt`, `status`, `priority`, `progress` |
| `sortDirection`, `cursor`, `limit` | 정렬과 공통 페이지네이션 |

`GET /team-works`는 다음 필터를 지원한다.

| 파라미터 | 값 |
| --- | --- |
| `query` | 팀 작업 표시 ID 또는 상위 이슈 제목 검색어 |
| `issueId`, `teamId`, `projectId`, `projectRole` | 소속 범위 |
| `workflowStateId`, `stateCategory` | 팀 실행 상태 |
| `assigneeMembershipId` | 멤버십 UUID 목록 또는 `me` |
| `unassigned` | 담당자 없는 미완료 작업 여부 |
| `sort` | `createdAt`, `updatedAt`, `status`, `priority` |
| `sortDirection`, `cursor`, `limit` | 정렬과 공통 페이지네이션 |

팀의 `전체` 탭은 `teamId`, `진행`은 `stateCategory=UNSTARTED,STARTED`, `백로그`는 `stateCategory=BACKLOG`를 사용한다. 내 작업 기본값은 `assigneeMembershipId=me&stateCategory=BACKLOG,UNSTARTED,STARTED`다. 팀 작업의 `priority` 정렬은 상위 이슈 우선순위로 계산하며 팀 작업에 값을 복제하지 않는다.

기본 정렬은 `updatedAt desc`다. 모든 정렬은 불변 `id`를 마지막 동률 해소 값으로 사용하고 주 정렬 값과 같은 방향을 적용한다. 커서는 서버가 정규화한 정렬 튜플과 `id`를 담는 불투명 값이며 같은 워크스페이스·리소스·필터·정렬에서만 재사용한다.

`workQueue`는 이슈 상태와 소속 팀 작업을 조합해 계산한다. 유효 작업 없음은 `REVIEW_REQUIRED`, 미완료 미할당 작업 존재는 `ASSIGNMENT_REQUIRED`, 이슈 `IN_PROGRESS`는 `IN_PROGRESS`, 이슈 `REVIEW`는 `COMPLETION_REQUIRED`, 이슈 `DONE`은 `COMPLETED`다. 서버의 이슈 자동 상태는 유효 팀 작업 없음 `UNSORTED`, 시작 전만 존재 `TODO`, 일부 시작·완료와 미완료 공존 `IN_PROGRESS`, 전체 완료 `REVIEW`로 계산한다.

### 10.5 이슈 생성과 작업 시작

`POST /issues` 요청은 다음과 같다.

```json
{
  "title": "이메일 가입 흐름",
  "descriptionMarkdown": "가입과 인증 흐름을 구현한다.",
  "priority": "HIGH",
  "projectId": "...",
  "initialRoles": ["BACKEND", "WEB_FRONTEND"],
  "labelIds": ["..."],
  "attachmentFileIds": ["..."]
}
```

- `type`, `teamId`, `workflowStateId`, `assigneeMembershipId`, `parentIssueId`와 팀 작업 제목·설명 필드는 받지 않는다.
- `initialRoles`는 선택 필드다. 생략하거나 빈 배열이면 이슈만 만들고 값이 있으면 현재 프로젝트 역할의 중복 없는 부분집합이어야 한다.
- 선택 역할의 팀 작업은 같은 트랜잭션에서 만들며 팀·시작 상태는 프로젝트 설정에서 서버가 결정한다.
- 팀 작업은 이슈 콘텐츠를 복제하지 않는다. 작업 사이 순서 관계도 자동 생성하지 않는다.
- 설명 이미지와 일반 첨부는 이슈에만 연결한다.

성공 응답은 `{ issue: IssueDetail, createdTeamWorks: TeamWorkSummary[] }`다.

`POST /issues/{issueId}/team-works`는 `initialRoles` 또는 역할별 담당자가 있는 `roleAssignments` 중 하나를 받는다. 역할별 기존 미완료 팀 작업은 재사용하고 없는 역할만 만든다. 복수 역할 중 하나라도 실패하면 전체를 롤백한다.

```json
{
  "roleAssignments": [
    { "projectRole": "BACKEND", "assigneeMembershipId": "멤버십 UUID" },
    { "projectRole": "WEB_FRONTEND", "assigneeMembershipId": null }
  ],
  "requireCurrentUserTeamMembership": false
}
```

`initialRoles`와 `roleAssignments`를 함께 보내면 `422 INITIAL_ROLE_INPUT_CONFLICT`다. 담당자는 프로젝트 역할에서 결정한 팀의 활성 멤버여야 한다. `requireCurrentUserTeamMembership = true`이면 선택한 모든 역할 팀에 현재 사용자가 속하는지 추가 검증한다.

### 10.6 내가 맡기와 일괄 배정

`POST /issues/{issueId}/claim` 요청은 다음과 같다.

```json
{
  "projectRole": "WEB_FRONTEND",
  "teamWorkId": null
}
```

`teamWorkId`가 있으면 같은 이슈·역할·팀의 미할당 미완료 작업인지 확인한다. 생략하면 후보가 한 건일 때 재사용하고 없으면 새 작업을 만든다. 후보가 여러 건이면 `409 CLAIM_TARGET_REQUIRED`, 다른 담당자가 생겼으면 `409 TEAM_WORK_ASSIGNMENT_CONFLICT`다.

`POST /issues/{issueId}/assign-team-works` 요청은 다음과 같다.

```json
{
  "assignments": [
    {
      "teamWorkId": "팀 작업 UUID",
      "assigneeMembershipId": "멤버십 UUID",
      "version": 3
    }
  ]
}
```

모든 작업이 경로 이슈에 속하고 미완료·미할당이며 버전과 담당 후보가 유효해야 한다. 전부 검증한 뒤 한 트랜잭션에서 반영하고 한 건이라도 충돌하면 전체를 롤백한다.

### 10.7 이슈 수정

`PATCH /issues/{issueId}`는 `title`, `descriptionMarkdown`, `priority`, `labelIds`, `statusAction`, `version`만 받는다. 프로젝트는 생성 뒤 변경하지 않는다. `PAUSE`, `RESUME`, `CANCEL`, `REOPEN`, `COMPLETE`만 명시적 상태 행동으로 허용하고 `UNSORTED`, `TODO`, `IN_PROGRESS`, `REVIEW` 직접 지정은 `422 ISSUE_STATUS_DERIVED`로 거부한다.

`COMPLETE`는 현재 상태가 `REVIEW`이고 취소되지 않은 팀 작업이 모두 완료된 경우에만 `DONE`으로 전환한다. 팀 작업 생성·상태 변경·제거·복구와 최초 전달은 상위 이슈를 잠그고 자동 상태를 같은 트랜잭션에서 다시 계산한다. 이슈가 `PAUSED`, `CANCELED`, `DONE`이면 새 작업 시작·재개 요청을 `409 ISSUE_REOPEN_REQUIRED`로 거부한다.

### 10.8 팀 작업 수정과 전달 완료

`PATCH /team-works/{teamWorkId}`는 `workflowStateId`, `assigneeMembershipId`, `workNoteMarkdown`, `version`, 완료 범주로의 전이에서만 쓰는 `completionMode`와 선택 `handoff`를 받는다. 제목·Markdown 설명·우선순위·라벨·프로젝트·팀·이슈·첨부파일 필드는 받을 수 없다.

```json
{
  "version": 4,
  "workflowStateId": "완료 상태 ID",
  "completionMode": "HANDOFF_AND_COMPLETE",
  "workNoteMarkdown": "## 구현 노트\n\n인증 API와 오류 코드 정의까지 완료",
  "handoff": {
    "bodyMarkdown": "## 변경 요약\n...",
    "destinationRoles": ["WEB_FRONTEND", "APP_FRONTEND"]
  }
}
```

- `workNoteMarkdown`은 최대 10,000자의 Markdown이며 멘션·본문 이미지·일반 파일을 지원하지 않는다. 해당 팀의 활성 멤버만 수정할 수 있다.
- 프론트 팀 작업의 `STARTED` 전이는 백엔드 작업이나 API 전달 존재 여부와 무관하게 즉시 허용한다. 병렬로 시작한 백엔드·프론트 작업은 서로의 진행을 기다리지 않는다.
- `workflowStateId`가 완료 범주로 전이하는 요청에는 `completionMode`가 필수다. `COMPLETE_ONLY`는 `handoff`를 허용하지 않고(`422 TEAM_WORK_HANDOFF_NOT_ALLOWED`), `HANDOFF_AND_COMPLETE`는 `handoff`가 필수이며(`422 TEAM_WORK_HANDOFF_REQUIRED`) 프로젝트에 웹·앱 프론트 역할이 하나도 없으면 `422 TEAM_WORK_HANDOFF_NO_FRONTEND_ROLE`이다. 완료가 아닌 전이 요청에 `completionMode` 또는 `handoff`가 포함되면 `422 TEAM_WORK_COMPLETION_MODE_NOT_ALLOWED`다.
- 담당자 지정 요청이 기본(`isDefault`) `BACKLOG` 팀 작업을 대상으로 하면 담당자와 해당 팀의 첫 `UNSTARTED` 상태를 같은 트랜잭션에서 저장한다. 담당자 해제·교체나 수동 `BACKLOG` 작업에는 이 자동 전이를 적용하지 않는다. 같은 규칙을 팀 작업 수정, 내가 맡기, 일괄 담당자 지정과 이슈 생성·작업 시작의 담당자 재사용 경로에 동일하게 적용한다.
- `HANDOFF_AND_COMPLETE`는 최초 전달, 대상 연결, 백엔드 완료, 대상 프론트 작업 생성 또는 재사용, 이슈 자동 상태, 활동과 Outbox를 한 트랜잭션에서 처리한다.
- 성공 응답은 `teamWork`, `issue`, 선택 `handoff`, `downstreamTeamWorks`를 반환한다. `teamWork`(및 목록의 `TeamWorkSummary`)는 API 전달 대기 여부를 나타내는 필드를 갖지 않는다.

## 11. 댓글, 활동과 작업 전달 API

### 11.1 엔드포인트

| 메서드 | 경로 | 권한 | 용도 |
| --- | --- | --- | --- |
| `GET` | `/issues/{issueId}/timeline` | 활성 멤버 | 댓글·활동·작업 전달 통합 타임라인 |
| `POST` | `/issues/{issueId}/comments` | 활성 멤버 | 댓글 작성 |
| `PATCH` | `/comments/{commentId}` | 작성자 | 자신의 댓글 수정 |
| `DELETE` | `/comments/{commentId}` | 작성자 | 자신의 댓글 본문 제거 |
| `POST` | `/team-works/{teamWorkId}/handoffs` | 활성 멤버 | 백엔드 팀 작업의 최초 전달 뒤 추가 전달 작성 |

### 11.2 타임라인

`GET /issues/{issueId}/timeline`은 커서를 지원하며 시간순 표시를 위해 `sortDirection=asc|desc`를 받는다.

```ts
type TimelineItem =
  | { type: "COMMENT"; comment: CommentResource & { teamWorkId: string | null }; createdAt: string }
  | { type: "ACTIVITY"; activity: ActivityResource & { teamWorkId: string | null; teamWorkIdentifier: string | null }; createdAt: string }
  | { type: "HANDOFF"; handoff: HandoffResource & { sourceTeamWorkId: string; targetTeamWorkIds: string[] }; createdAt: string };
```

활동의 `before`와 `after`는 상태·담당자·우선순위처럼 해당 변경을 표시하는 최소 요약만 포함한다. Markdown 본문과 이메일은 활동 응답에 복제하지 않는다.

작업 전달 원문은 `HANDOFF` 항목만 유일한 원본으로 반환하고 같은 내용을 별도 활동으로 복제하지 않는다. 활동은 관련 실행 문맥이 있으면 `teamWorkId`와 표시용 `teamWorkIdentifier`를 함께 반환하며, 웹은 해당 팀 작업이 선택된 정본 통합 상세 주소로 이동시킨다.

### 11.3 댓글

```json
{
  "teamWorkId": "선택 팀 작업 UUID 또는 null",
  "bodyMarkdown": "@[김민수](rivet-member:멤버십-UUID) 확인 부탁드립니다.\n\n![오류 화면](/files/파일-UUID)"
}
```

댓글은 항상 경로의 이슈에 속하며 `teamWorkId`는 선택 실행 문맥일 뿐 별도 스레드를 만들지 않는다. 값이 있으면 같은 이슈의 팀 작업인지 검증한다. 댓글 작성은 멘션, 본문 이미지 연결, 자동 구독, 활동과 Outbox를 한 트랜잭션에서 처리한다. 수정 요청에는 `version`을 추가한다. 삭제 성공은 `204`이며 타임라인에는 작성자·작성 시각과 삭제 상태만 남는다.

댓글 삭제는 `version` 쿼리 파라미터를 받아 삭제 직전의 다른 편집을 조용히 지우지 않는다.

### 11.4 작업 전달

```json
{
  "kind": "FOLLOW_UP",
  "bodyMarkdown": "## 변경 요약\n...\n\n## API 명세 링크\nhttps://..."
}
```

- 백엔드 프로젝트 역할의 팀 작업에만 작성할 수 있다.
- `INITIAL`은 원본 백엔드 팀 작업당 한 건이다. 프로젝트에 프론트 역할이 있으면 백엔드 완료와 함께 `PATCH /team-works/{teamWorkId}`의 인라인 `handoff`로만 생성한다. 템플릿 제목만 남은 본문은 `HANDOFF_CONTENT_REQUIRED`다.
- `FOLLOW_UP`은 최초 전달이 있을 때만 추가할 수 있다.
- 작업 전달은 수정·삭제할 수 없다.
- 전달 본문 이미지도 Markdown 파일 참조와 연결을 같은 트랜잭션에서 저장한다.
- 작업 전달 본문에는 멘션 참조를 허용하지 않으며 포함하면 `422 MARKDOWN_INVALID`로 거부한다. 작업 전달 멘션 관계와 별도 `MENTIONED` 수신자는 만들지 않는다.
- 최초 전달 알림은 재사용 작업의 담당자·이슈 자동 구독자와 새 미할당 작업의 대상 팀 활성 멤버를 합쳐 작성자를 제외하고 중복 제거한다. 추가 전달은 같은 이슈에 이미 존재하는 미완료 웹·앱 작업의 담당자와 자동 구독자만 사용한다.
- `FOLLOW_UP` 요청은 `destinationRoles`를 받지 않으며 새 팀 작업을 자동 생성하지 않는다.
- 상태 완료와 최초 전달을 동시에 처리해야 하면 별도 호출 두 번이 아니라 `PATCH /team-works/{teamWorkId}`의 `handoff`를 사용한다.

## 13. 파일 API

### 13.1 업로드와 파일 조회

| 메서드 | 경로 | 권한 | 용도 |
| --- | --- | --- | --- |
| `POST` | `/files` | 인증 사용자 | 파일 한 개 업로드 |
| `GET` | `/files/{fileId}` | 파일 접근자 | 파일 메타데이터 조회 |
| `GET` | `/files/{fileId}/content` | 파일 접근자 | 허용된 이미지는 인라인, 그 외 다운로드 스트리밍 |
| `GET` | `/files/{fileId}/download` | 파일 접근자 | 항상 다운로드 스트리밍 |
| `DELETE` | `/files/{fileId}` | 업로더 | 아직 연결되지 않은 파일 즉시 제거 |

`POST /files`는 `file`과 `scope` 두 multipart 필드를 받는다. 한 요청에 파일 한 개만 보내 파일별 진행·실패·재시도를 독립적으로 처리한다.

```text
Content-Type: multipart/form-data
file: <binary>
scope: WORKSPACE | USER_PROFILE
```

- 최종 파일 크기는 1바이트 이상 26,214,400바이트 이하이고 제품 표기는 최대 25MB다.
- API는 스트리밍 중 제한을 넘으면 즉시 중단하고 `413 FILE_TOO_LARGE`를 반환한다.
- `USER_PROFILE`은 JPEG·PNG·WebP만 업로드할 수 있다.
- `WORKSPACE`는 일반 첨부를 위해 형식을 제한하지 않지만 실제 형식을 감지해 저장한다.
- 업로드 성공 파일은 미연결 상태이며 현재 업로더만 메타데이터와 바이너리에 접근할 수 있다.
- 성공 응답의 ID를 이슈·본문·프로필 저장 요청에 넣어 연결한다.
- 업로드 후 24시간 동안 연결되지 않으면 워커가 메타데이터와 바이너리를 제거한다.

`GET /files/{fileId}/content`는 JPEG·PNG·WebP·GIF만 `Content-Disposition: inline`으로 제공한다. SVG와 그 밖의 파일은 `attachment`로 강제하고 모든 바이너리 응답에 `X-Content-Type-Options: nosniff`를 사용한다.

Markdown의 `/files/{fileId}`는 웹 애플리케이션이 같은 출처에서 인증된 `/api/v1/files/{fileId}/content`로 전달하는 논리 경로다. 업로드 디렉터리를 Nginx 정적 경로로 공개하지 않는다.

### 13.2 이슈 일반 첨부

| 메서드 | 경로 | 권한 | 용도 |
| --- | --- | --- | --- |
| `GET` | `/issues/{issueId}/attachments` | 활성 멤버 | 일반 첨부 목록 |
| `POST` | `/issues/{issueId}/attachments` | 활성 멤버 | 미연결 파일을 일반 첨부로 연결 |
| `DELETE` | `/issues/{issueId}/attachments/{attachmentId}` | 활성 멤버 | 일반 첨부 연결 해제 |

```json
{
  "fileId": "..."
}
```

첨부 목록은 `createdAt`, `id` 오름차순이며 수동 정렬을 제공하지 않는다. 삭제는 파일을 즉시 지우지 않고 연결을 해제해 24시간 유예 뒤 정리한다. 휴지통에 있는 이슈의 파일 연결은 복구 기간 동안 유지한다.

### 13.3 본문 이미지 연결

- 설명은 이슈 생성·수정, 댓글은 댓글 생성·수정, 작업 전달은 전달 생성 요청에서 Markdown 파일 ID를 추출한다.
- 서버가 연결 종류를 `DESCRIPTION_IMAGE`, `COMMENT_IMAGE`, `HANDOFF_IMAGE`로 결정하므로 클라이언트가 종류를 별도 요청하지 않는다.
- 실제 JPEG·PNG·WebP·GIF가 아닌 파일을 본문 이미지로 참조하면 `415 FILE_TYPE_NOT_INLINE_DISPLAYABLE`이다.
- 같은 파일 ID를 한 본문에서 여러 번 표시할 수 있지만 다른 이슈·댓글·전달에 재사용할 수 없다.
- 본문에서 이미지 참조를 제거하면 저장 트랜잭션에서 연결을 해제한다.
- 원격 HTML이나 이미지 URL을 서버가 가져오는 엔드포인트는 제공하지 않는다.

### 13.4 파일 접근과 누락

| 파일 상태 | 접근 규칙 |
| --- | --- |
| 미연결 | 업로더 본인만 |
| 이슈 연결 | 같은 워크스페이스의 활성 이슈 접근자 |
| 프로필 사진 | 활성 사용자 표시가 필요한 인증 화면 범위 |
| 다른 워크스페이스·권한 없음 | `404 RESOURCE_NOT_FOUND` |
| 메타데이터는 있으나 바이너리 없음 | `503 FILE_UNAVAILABLE` |

`FILE_UNAVAILABLE`일 때 프론트는 참조를 자동 삭제하지 않고 불러올 수 없음 상태를 표시한다.

## 14. 알림 API

| 메서드 | 경로 | 권한 | 용도 |
| --- | --- | --- | --- |
| `GET` | `/notifications` | 활성 멤버 | 현재 사용자의 알림 목록 |
| `GET` | `/notifications/unread-count` | 활성 멤버 | 읽지 않은 개수 |
| `PATCH` | `/notifications/{notificationId}` | 수신자 | 읽음·읽지 않음 변경 |
| `POST` | `/notifications/read-all` | 활성 멤버 | 현재 알림을 모두 읽음 처리 |

`GET /notifications`는 `read=true|false`, `type`, 커서를 지원하며 기본 정렬은 최신순이다.

```ts
type NotificationResource = {
  id: string;
  type: NotificationType;
  actor: UserSummary | null;
  issue: { id: string; identifier: string; title: string };
  teamWork: { id: string; identifier: string; projectRole: ProjectRole } | null;
  commentId: string | null;
  handoffId: string | null;
  readAt: string | null;
  createdAt: string;
};
```

```json
{
  "read": true
}
```

항목 선택은 읽음 변경 성공 여부와 관계없이 접근 가능한 이슈로 이동할 수 있다. 이슈가 영구 삭제됐거나 접근할 수 없으면 안전한 기본 화면으로 이동한다.

작업 전달 알림의 `issue`는 항상 전달이 속한 이슈다. 워커는 실제 후행 작업 중 수신자가 담당자인 작업, 자동 구독자인 작업, 대상 팀의 활성 멤버인 작업 순으로 하나를 골라 `teamWork`에 저장한다. 같은 우선순위 후보가 여러 개면 프로젝트 역할, 표시 식별자와 ID 순으로 안정 정렬한다. 맞는 후행 작업이 없으면 `teamWork = null`이다. 웹은 이슈 통합 상세를 열고 팀 작업이 있으면 `work` 쿼리로 선택한 뒤 같은 `handoffId` 앵커로 이동한다.

## 15. 검색 API

| 메서드 | 경로 | 권한 | 용도 |
| --- | --- | --- | --- |
| `GET` | `/search` | 활성 멤버 | 이슈 콘텐츠와 팀 작업 표시 ID 검색 |

| 파라미터 | 규칙 |
| --- | --- |
| `query` | 공백 정리 후 검색어 |
| `limit` | 기본 20, 최대 50 |
| `cursor` | 다음 결과 커서 |

- 표시 ID 형식이면 이슈와 팀 작업에서 한 글자부터 정확 일치를 시도한다.
- 제목 부분 검색은 이슈 제목에만 두 글자 이상일 때 실행한다.
- 정확한 표시 ID 일치를 제목 일치보다 먼저 반환한다.
- 팀 작업은 별도 제목이 없으므로 상위 이슈 제목과 함께 표시한다.
- 설명, 댓글, 작업 메모, 프로젝트와 작업 전달 본문은 검색하지 않는다.

```ts
type SearchResult =
  | {
      resourceType: "ISSUE";
      matchType: "IDENTIFIER_EXACT" | "TITLE_PARTIAL";
      issue: IssueSummary;
    }
  | {
      resourceType: "TEAM_WORK";
      matchType: "IDENTIFIER_EXACT";
      teamWork: TeamWorkSummary;
    };
```

## 16. 휴지통과 복구 API

| 메서드 | 경로 | 권한 | 용도 |
| --- | --- | --- | --- |
| `GET` | `/trash` | 관리자 | 삭제된 이슈·프로젝트 조회 |
| `POST` | `/trash/issues/{issueId}/restore` | 관리자 | 이슈 복구 |
| `POST` | `/trash/projects/{projectId}/restore` | 관리자 | 프로젝트 복구 |

`GET /trash`는 `resourceType=ISSUE,PROJECT`, `query`, `deletedByMembershipId`, 커서를 지원한다. 각 항목은 `deletedAt`, `purgeAt`, 삭제자와 원래 연결 요약을 포함한다.

복구 요청에는 휴지통에서 조회한 `version`을 포함한다. 원래 팀·프로젝트·상태가 보관된 경우 관계는 유지하며 응답에 `warnings`를 포함할 수 있다. MVP에서는 즉시 영구 삭제 API를 제공하지 않는다.

## 17. CSV 내보내기 API

| 메서드 | 경로 | 권한 | 용도 |
| --- | --- | --- | --- |
| `GET` | `/exports/issues.csv` | 관리자 | 이슈 CSV 스트리밍 |
| `GET` | `/exports/projects.csv` | 관리자 | 프로젝트 CSV 스트리밍 |

- 응답은 UTF-8 BOM을 포함한 CSV로 내려 한국어 스프레드시트 호환성을 확보한다.
- 파일명은 `rivet-issues-YYYYMMDD.csv`, `rivet-projects-YYYYMMDD.csv` 형식이다.
- CSV 수식 실행을 막기 위해 `=`, `+`, `-`, `@`로 시작하는 사용자 입력 셀은 안전하게 이스케이프한다.
- 생성과 다운로드 완료 시각을 감사 기록으로 남긴다.
- MVP 데이터 규모에서는 요청 중 스트리밍하며 비동기 내보내기 API를 만들지 않는다.

이슈 CSV는 이슈 행을 기준으로 표시 ID, 프로젝트, 제목, 설명 Markdown, 이슈 상태, 우선순위, 라벨, 첨부파일 메타데이터·참조, 생성·수정 시각과 소속 팀 작업 요약을 포함한다. 팀 작업 요약에는 표시 ID, 역할, 팀, 워크플로 상태, 담당자와 짧은 작업 메모만 넣고 이슈 콘텐츠를 반복하지 않는다. 프로젝트 CSV에는 이름, 설명, 상태, 리드, 역할별 팀, 시작일, 목표일, 생성·수정 시각을 포함한다. 바이너리 파일은 묶지 않는다.

## 18. 실시간 연결 API

| 메서드 | 경로 | 권한 | 용도 |
| --- | --- | --- | --- |
| `GET` | `/events` | 활성 멤버 | 현재 워크스페이스 변경 SSE 수신 |

```text
event: resource.changed
id: 01J2V7G9P1M6A9RFM9XV4GB4HR
data: {"resourceType":"ISSUE","resourceId":"...","changeType":"UPDATED","version":8}
```

| 필드 | 규칙 |
| --- | --- |
| `id` | 클라이언트 중복 처리용 불변 이벤트 ID |
| `resourceType` | `ISSUE`, `TEAM_WORK`, `PROJECT`, `COMMENT`, `HANDOFF`, `NOTIFICATION`, `MEMBER`, `TEAM`, `WORKFLOW_STATE`, `LABEL`, `FILE` |
| `resourceId` | 변경된 내부 리소스 ID |
| `changeType` | `CREATED`, `UPDATED`, `DELETED`, `RESTORED` |
| `version` | 버전이 있는 리소스의 최신 값, 없으면 `null` |

- SSE에는 업무 본문, 이메일과 전체 리소스 스냅샷을 넣지 않는다.
- 프론트는 이벤트를 받으면 관련 캐시를 무효화하거나 REST API로 최신 데이터를 다시 조회한다.
- 서버는 과거 이벤트 완전 재생을 보장하지 않는다. 재연결 후 현재 화면과 읽지 않은 알림 수를 다시 조회한다.
- `Last-Event-ID`는 중복 억제 참고값으로만 받고 누락 구간 재생 계약으로 사용하지 않는다.
- 세부 이벤트 발행 시점과 알림 수신자 규칙은 실시간·알림 및 비동기 작업 명세서에서 확정한다.

## 21. OpenAPI와 프론트 전달 기준

- NestJS의 요청 DTO와 응답 DTO에서 OpenAPI를 생성한다.
- 각 엔드포인트는 성공 예시와 문서에 정의된 주요 오류 예시를 포함한다.
- enum, nullable, 선택 필드, `IssueStatusAction`, `IssueSummary`, `TeamWorkSummary`를 정확히 표현한다.
- 전역 이슈의 `workQueue`·세부 필터·진행률 정렬, `workflowSummary`, `totalCount`, `workQueueCounts`, 역할별 작업 시작, claim과 일괄 배정 성공·충돌 응답을 OpenAPI와 생성 클라이언트에 함께 반영한다.
- 개발 환경은 OpenAPI JSON과 Swagger UI를 제공하되 운영 환경의 UI 공개 여부는 운영 설정으로 결정한다.
- `packages/api-client`는 OpenAPI에서 타입과 호출 함수를 생성하며 사람이 생성 결과를 직접 수정하지 않는다.
- CI는 OpenAPI 생성 결과가 최신인지 검증한다.
- 이번 변경은 실데이터가 없는 개발 단계의 동시 전환이다. 같은 `/v1`에서 구형 `IssueType`, `FEATURE`, `TEAM_TASK`, 혼합 이슈 DTO, 독립 팀 작업 생성·콘텐츠 필드를 즉시 제거하며 deprecated 호환 필드나 이중 계약을 남기지 않는다.

백엔드가 프론트에 API 작업을 전달할 때는 다음을 함께 제공한다.

1. 변경된 OpenAPI와 생성 클라이언트
2. 호출 가능한 개발 환경과 인증 조건
3. 성공 요청·응답 예시
4. 주요 오류 코드와 화면 처리 기준
5. 기존 계약과의 호환성 영향
6. 관련 작업 전달의 API 명세 링크와 프론트 주의사항
