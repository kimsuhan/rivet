## 1. 문서 목적

이 문서는 Rivet MVP를 실제 사용 가능한 품질로 검증하고 자체 서버에 배포한 뒤, 장애를 탐지하고 복구하기 위한 계약을 정의한다. 프론트엔드, API와 워커의 자동 테스트 범위, PostgreSQL 마이그레이션, PM2·Nginx 구성, 상태 확인, 로그·경고, 백업·복구와 장애 대응 절차를 하나의 출시 기준으로 묶는다.

Rivet의 운영 인프라는 이 저장소가 자동으로 생성하거나 변경하지 않는다. 운영자는 이미 보유한 클라우드 서버와 자체 PostgreSQL을 사용해 직접 빌드·배포하며, 이 문서는 그 과정에서 지켜야 할 순서와 확인 기준만 제공한다.

### 1.1 범위

- 도메인·데이터베이스·API·워커·프론트엔드·브라우저 테스트 전략
- 백엔드 API 작업을 프론트엔드에 전달하기 전 검증 기준
- 출시 전 품질 게이트와 출시 차단 결함
- pnpm workspace와 Turborepo 기반 설치·검증·빌드 순서
- Prisma 마이그레이션 검토·적용·실패 복구 원칙
- `apps/web`, `apps/api`, `apps/worker`의 PM2 실행·종료 기준
- Nginx의 같은 출처 라우팅, TLS, 업로드와 SSE 프록시 기준
- 환경 변수, 비밀 값 주입·교체·유출 대응
- 상태 확인, 배포 후 점검과 애플리케이션 되돌리기
- PostgreSQL·로컬 파일 저장소 백업, 복구와 정합성 확인
- 구조화 로그, PostHog, 운영 지표, 경고와 장애 대응 절차

### 1.2 제외 범위

- 클라우드 서버, 네트워크, PostgreSQL과 도메인의 생성·구매·프로비저닝
- 실제 서버 주소, 도메인, 포트, 인증서 경로와 비밀 값
- Terraform, Docker Swarm, Kubernetes와 공급자별 인프라 코드
- 자동 배포 파이프라인과 무중단·블루그린·카나리 배포 시스템
- 별도 모니터링 대시보드, 로그 수집 클러스터와 당직 관리 시스템
- 대규모 부하·다중 리전·재해 복구 센터 설계
- Public 이후의 다중 인스턴스 자동 확장과 객체 저장소 운영

MVP에서는 수동 실행 가능한 검증·배포 명령과 짧은 운영 절차를 우선한다. 실제 반복 비용이나 장애 위험이 확인되기 전에는 배포 플랫폼과 새로운 운영 서비스를 추가하지 않는다.

### 1.3 결정 상태

| 상태 | 의미 |
| --- | --- |
| 확정 | 선행 문서에서 이미 결정됐거나 MVP 출시 안전상 변경하지 않는 기준 |
| 권장 | 현재 자체 서버 구성에 가장 단순하게 맞는 기본안이며 검토 후 확정할 항목 |
| 미정 | 운영 방식 또는 팀 선택이 필요한 항목 |

### 1.4 공식 참고 기준

- Prisma 운영 마이그레이션은 [`prisma migrate deploy`](https://docs.prisma.io/docs/cli/migrate/deploy)의 운영 환경 동작을 기준으로 한다.
- PM2 프로세스 선언과 종료는 [Ecosystem File](https://pm2.keymetrics.io/docs/usage/application-declaration/)과 [Graceful Start/Shutdown](https://pm2.keymetrics.io/docs/usage/signals-clean-restart/)을 기준으로 한다.
- Nginx 프록시 제한 시간과 버퍼링은 [`ngx_http_proxy_module`](https://nginx.org/en/docs/http/ngx_http_proxy_module.html)을 기준으로 한다.
- 브라우저 흐름 검증은 Playwright의 [Web server](https://playwright.dev/docs/test-webserver)와 [Projects](https://playwright.dev/docs/test-projects) 구성을 기준으로 한다.
- 운영 경고는 Slack의 [Incoming Webhooks](https://api.slack.com/messaging/webhooks) 전송 방식을 기준으로 한다.

버전별 명령과 옵션은 구현 시작 시 고정한 패키지 버전의 공식 문서를 다시 확인한다.

## 2. 운영 목표와 기본 원칙

### 2.1 운영 목표

1. 출시 전에 워크스페이스 격리, 핵심 업무 저장과 파일 접근을 자동 검증한다.
2. 배포 결과가 잘못되면 데이터베이스를 임의로 되돌리지 않고 직전 애플리케이션 버전으로 안전하게 복귀한다.
3. API, 워커와 실시간 연결의 실패를 사용자 신고보다 먼저 확인할 수 있게 한다.
4. 데이터베이스 메타데이터와 로컬 파일 바이너리를 함께 복구할 수 있게 한다.
5. 한 명의 운영자가 문서만 보고 배포·점검·기본 장애 대응을 수행할 수 있게 한다.

### 2.2 기본 원칙

- 자동 테스트는 성공 흐름보다 데이터 격리, 권한, 중복 처리와 실패 복구를 우선한다.
- 외부 공급자와 운영 데이터에 의존하지 않는 테스트를 기본으로 한다.
- 개발과 운영은 같은 빌드 과정을 사용하되 데이터, 비밀 값과 외부 발송 키를 분리한다.
- 운영 마이그레이션은 저장소에 커밋된 이력만 적용하며 운영에서 스키마를 즉석 수정하지 않는다.
- `web`, `api`, `worker`는 각각 배포하더라도 데이터 계약과 이벤트 호환성을 하나의 릴리스로 검증한다.
- 운영 로그에는 진단에 필요한 내부 식별자만 남기고 업무 본문과 인증 정보를 남기지 않는다.
- 복구 절차는 문서 존재가 아니라 실제 복구 연습 성공으로 검증한다.

## 3. 환경과 릴리스 식별

### 3.1 환경 구분

| 환경 | 목적 | 데이터 | 외부 연동 |
| --- | --- | --- | --- |
| 로컬 | 개인 개발, 단위·통합·브라우저 테스트 | 생성 가능한 합성 데이터 | Resend·PostHog 기본 비활성 또는 가짜 구현 |
| 개발 | 프론트·백엔드 통합, API 전달과 수동 인수 | 테스트 계정과 공유 개발 데이터 | 개발 전용 Resend 키·발신 주소, 허용 목록 적용 |
| 운영 | 초대된 MVP 사용자 제공 | 실제 업무 데이터 | 운영 전용 Resend 키와 PostHog 프로젝트 |

- 환경마다 별도 `DATABASE_URL`, `FILE_STORAGE_ROOT`, Resend 키와 HMAC 키를 사용한다.
- 개발 데이터를 운영으로 복사하거나 운영 데이터를 로컬·개발 환경으로 내려받지 않는다.
- 자동 테스트는 전용 PostgreSQL 데이터베이스와 임시 파일 루트를 사용한다.
- 운영에서만 PostHog 제품 이벤트를 전송하며 자동 수집과 세션 녹화는 사용하지 않는다.

### 3.2 릴리스 식별자

각 빌드는 동일한 `RELEASE_ID`를 `web`, `api`, `worker`에 주입한다. 값은 Git 커밋 SHA 또는 운영자가 추적 가능한 고유 버전을 사용한다.

릴리스 식별자는 다음 위치에서 확인할 수 있어야 한다.

- API·워커 시작 로그와 모든 구조화 오류 로그
- PostHog의 정제된 예외 속성
- 운영 상태 확인 응답의 비민감 버전 필드 또는 운영자 전용 확인 수단
- 배포 기록의 빌드 시각, 마이그레이션 목록과 프로세스 재시작 결과

브라우저에 Git 저장소 주소, 서버 경로와 빌드 환경 변수를 통째로 노출하지 않는다.

## 4. 테스트 도구와 저장소 구성

### 4.1 확정 도구

| 대상 | 권장 도구 | 용도 | 상태 |
| --- | --- | --- | --- |
| API·워커 | Jest + Nest Testing | 도메인 서비스, 모듈과 워커 처리기 검증 | 확정 |
| HTTP API | Supertest | 실제 Nest 애플리케이션의 요청·응답, 쿠키와 보안 헤더 검증 | 확정 |
| 웹 | Vitest + Testing Library | 순수 상태 로직과 사용자 상호작용이 중요한 컴포넌트 검증 | 확정 |
| 전체 사용자 흐름 | Playwright | 브라우저, 웹·API 연결과 핵심 흐름 검증 | 확정 |
| 데이터베이스 | 실제 PostgreSQL 테스트 DB | Prisma 쿼리, 제약과 트랜잭션 검증 | 확정 |

프레임워크의 일반적인 도구를 그대로 사용한다. 테스트용 범용 추상화 계층, 별도 DSL과 자체 테스트 러너는 만들지 않는다.

### 4.2 테스트 위치

```text
apps/
  web/
    **/*.test.ts(x)
    e2e/**/*.spec.ts
  api/
    src/**/*.spec.ts
    test/**/*.e2e-spec.ts
  worker/
    src/**/*.spec.ts
    test/**/*.integration-spec.ts
packages/
  database/
    test/**/*.integration-spec.ts
```

- 작은 단위 테스트는 대상 코드 가까이에 둔다.
- PostgreSQL, HTTP 서버나 파일시스템을 사용하는 검증은 `test/` 아래 통합 테스트로 구분한다.
- 브라우저 E2E는 `apps/web/e2e` 한 곳에서 웹·API·워커가 연결된 흐름을 검증한다.
- 같은 정책을 API·워커·E2E에서 모두 반복하지 않는다. 가장 낮은 계층에서 경계값을 충분히 검증하고 E2E는 대표 흐름만 남긴다.

### 4.3 테스트 대역 원칙

- PostgreSQL 제약, Prisma 트랜잭션, 잠금과 `LISTEN/NOTIFY`는 실제 PostgreSQL로 검증한다.
- 단위·통합 테스트의 Resend 호출은 얇은 메일 어댑터의 가짜 구현으로 대체한다.
- 개발 환경에서만 허용 목록 주소를 대상으로 실제 Resend 발송 점검을 수행한다.
- 토큰 만료·재시도 테스트는 가짜 시계 또는 주입된 현재 시각을 사용하고 실제 대기를 만들지 않는다.
- 로컬 파일 테스트는 운영 저장소와 분리된 임시 디렉터리를 사용하고 테스트 종료 시 제거한다.
- PostHog 전송은 자동 테스트에서 가짜 구현으로 대체하고 허용 필드만 전달되는지 검증한다.

### 4.4 커버리지 기준

- 패키지별 커버리지 리포트는 생성하지만 MVP 초기에는 저장소 전체 퍼센트를 출시 게이트로 강제하지 않는다.
- 인증·권한, 워크스페이스 격리, 트랜잭션, Outbox 재시도·멱등성과 파일 접근 경계는 커버리지 수치와 무관하게 필수 테스트로 관리한다.
- 버그 수정에는 같은 결함을 재현하는 회귀 테스트를 포함한다.
- 코드 기반이 안정된 뒤 패키지별 실제 기준선과 위험도를 근거로 threshold 도입을 검토한다.

## 5. 계층별 필수 검증

### 5.1 도메인 단위 테스트

- 시스템 상태 범주와 팀별 사용자 정의 상태 전환
- 프로젝트의 백엔드·웹 프론트·앱 프론트 역할별 팀 선택 규칙
- 이슈별 최초 역할의 생략·빈 배열·단일·복수 선택과 프로젝트 역할 부분집합·중복 검증
- 이슈와 팀 작업의 분리 스키마, 필수 상위 이슈와 콘텐츠 중복 금지
- 팀 작업 관계 테이블과 관계 API가 존재하지 않음
- 프로젝트 프론트 역할 기준의 최초 작업 전달 필수 조건과 `destinationRoles` 검증
- 최초 전달 대상 작업 생성·재사용·준비 상태 전환과 추가 전달 무생성 판단
- 재사용 작업과 새 미할당 작업을 구분한 전달 수신자 계산
- 댓글·멘션·할당·구독이 겹칠 때 알림 우선순위와 중복 제거
- 비밀번호 길이, 토큰 목적·만료와 세션 유휴·절대 만료
- 이미지 최적화 조건과 파일 연결 가능 상태
- 전역 이슈 다섯 업무함 조건, 완료·취소 작업 이력과 진행률·다음 행동 우선순위
- `initialRoles`·`roleAssignments` 상호 배타, 현재 역할 팀 멤버십과 claim 후보 선택 규칙

순수 계산 규칙은 데이터베이스나 Nest 모듈을 띄우지 않고 검증한다.

### 5.2 데이터베이스 통합 테스트

- 같은 팀에서 표시 ID를 동시에 생성해도 중복되지 않는다.
- 한 계정이 둘 이상의 활성 워크스페이스 멤버십을 만들 수 없다.
- 프로젝트 역할별 팀 규칙과 참조 워크스페이스가 데이터베이스에서도 보호된다.
- 루트 이슈와 사용자가 선택한 역할별 최초 팀 작업이 함께 커밋되며 어느 삽입이 실패해도 모두 롤백된다.
- 팀 작업이 없는 이슈에서 선택한 역할의 작업 시작이 한 트랜잭션으로 처리되고, 기존 미완료 역할 작업은 재사용하며 생성 실패 시 일부 작업이 남지 않는다.
- 최초 전달, 백엔드 완료, 대상 프론트 작업 생성·재사용과 준비 상태, 활동과 Outbox가 함께 커밋되거나 함께 롤백되고 전달 직후 진행률이 같은 스냅샷에서 계산된다.
- 이슈 생성 때 병렬로 시작한 작업과 최초 전달 모두 팀 작업 관계를 만들지 않는다.
- 업무 변경이 롤백되면 활동 이벤트와 Outbox도 함께 롤백된다.
- 같은 Outbox 이벤트를 다시 처리해도 알림과 파일 삭제가 중복되지 않는다.
- 잠긴 작업이 워커 종료 뒤 잠금 유효 시간 이후 회수된다.
- 이슈·프로젝트 휴지통, 복구와 30일 뒤 영구 삭제가 참조 순서를 지킨다.
- 파일 연결 해제, 24시간 유예와 실제 참조 재확인이 삭제를 보호한다.
- 속도 제한 버킷의 동시 증가가 상한을 우회하지 않는다.
- 역할별 담당자 시작, 내가 맡기와 일괄 배정이 작업·담당자·구독·활동·Outbox를 전부 커밋하거나 전부 롤백한다.
- 동시 claim은 한 요청만 성공하고 일괄 배정의 한 작업 버전·부모·후보 충돌이 전체 변경을 롤백한다.

### 5.3 API 통합 테스트

- 가입·이메일 인증·로그인·로그아웃·비밀번호 재설정의 성공과 실패 계약
- 세션 쿠키 속성, CSRF와 정확한 Origin 검증
- `admin/member`, 소유자와 활성 멤버십 권한
- UUID, 표시 ID, 검색, 알림, 관계, 휴지통과 CSV의 워크스페이스 격리
- 오래된 `version` 수정의 `409` 충돌과 최신 데이터 보존
- `POST /issues`, `POST /issues/{id}/start`, 최초 전달 `PATCH /issues/{id}`의 성공 응답과 `INITIAL_ROLE_REQUIRED`, `INITIAL_ROLE_NOT_AVAILABLE`, `PROJECT_FRONTEND_ROLE_REQUIRED`, `HANDOFF_DESTINATION_REQUIRED`, `DOWNSTREAM_TASK_SCOPE_CONFLICT`, `DOWNSTREAM_TASK_ALREADY_CLOSED`, `ISSUE_VERSION_CONFLICT` 계약
- 이슈 생성의 `initialRoles` 생략·빈 배열·단일·복수 선택, 중복과 프로젝트 미설정 역할 거부, 상위 이슈 없는 팀 작업 생성 계약 부재
- 상위 이슈와 전달받은 프론트 작업 상세가 같은 `ApiHandoff`의 최초·추가 전달과 준비 상태를 반환함
- 커서 페이지네이션의 중복·누락 없는 안정 정렬
- 25MB 파일 제한, 실제 형식, 경로 조작과 파일 소유자 검증
- Markdown 이미지·일반 첨부·프로필 파일의 접근 권한
- 보안 오류에서 계정·리소스 존재 여부를 과도하게 노출하지 않음
- 요청 본문, 쿠키, 토큰과 원본 파일명이 로그에 남지 않음
- OpenAPI 성공·오류 예시와 실제 응답의 호환성
- 이슈 목록의 다섯 `workQueue`, 팀 작업 목록 필터, 진행률·상태·우선순위 동률의 다중 페이지 안정 커서, 워크스페이스·휴지통 격리
- `workflowSummary`, 현재 사용자 역할·담당 작업, 만든 사람과 `totalCount`의 페이지 단위 응답
- 검색·세부 필터를 반영하고 선택된 `workQueue`만 제외한 `workQueueCounts`가 현재 페이지 크기와 무관하게 전체 결과를 반환하며 다른 워크스페이스·휴지통을 제외함
- `roleAssignments`, `requireCurrentUserTeamMembership`, claim과 `assign-team-works`의 성공·오류·충돌·전체 롤백 계약

### 5.4 워커 통합 테스트

- 빈 큐 1초 조회, 50건 claim, 최대 동시 처리 5건과 5분 잠금 계약
- 최대 7회와 확정된 재시도 간격의 계산

- 사건 당시 수신자 스냅샷과 처리 시 활성 멤버십 재검증
- 전달 알림의 수신자별 이동 대상이 후행 작업의 담당자, 자동 구독자, 대상 팀 활성 멤버 순으로 정해지고 대상이 없으면 상위 이슈, 원본 백엔드 작업 순으로 대체됨

- 동일 전달 Outbox 재처리의 수신자별 알림 중복 방지
- `FAILED` 작업이 자동 조회에서 제외됨
- 정기 작업 advisory lock과 워커 시작 시 누락 작업 따라잡기
- 임시·미연결·저장소 단독 파일 정리와 연결 파일 누락 탐지
- 종료 신호 뒤 새 작업을 claim하지 않고 처리 중 작업을 정리함
- 작업 시작·claim·일괄 배정의 `TEAM_WORK_ASSIGNED`, 행위자 제외와 같은 배정 이벤트 재처리 중복 방지

### 5.5 프론트엔드 테스트

- 한국어 입력, 긴 제목과 오류 메시지가 레이아웃을 깨뜨리지 않음
- Lexical 편집기의 저장 Markdown과 미리보기가 같은 결과를 사용함
- Command/Ctrl+V 이미지 업로드, 진행·실패·재시도와 파일 참조 삽입
- 큰 이미지 리사이징·WebP 변환과 원본 유지 조건
- 이슈 낙관적 수정 성공, 실패 되돌리기와 `409` 충돌 안내
- 로딩, 빈 상태, 권한 없음, 삭제됨과 네트워크 오류 상태
- 백엔드·웹·앱 팀 작업의 독립 상태와 전달 기반 `API 전달 대기`·`작업 가능` 표시

- 이슈 생성과 팀 작업 없는 이슈의 시작 역할 단일·복수 선택, 키보드 조작, 선택 상태의 텍스트·접근성 속성, 검증 실패 뒤 값·포커스 유지
- 작업 흐름의 분석·병렬·순차·전달·완료 상태, 기존 이슈의 `작업 시작`과 행동 없는 `0/0 완료` 미노출

- 업무 탭 최신 전달 기본 확장, 전달 탭 전체 이력 접기·펼치기와 전달 딥 링크 앵커

- SSE 중복 신호 억제, 연결 끊김 안내와 재연결 후 REST 재조회
- 키보드 탐색, 포커스 표시, 대화상자와 오류의 기본 접근성
- 사이드바와 모바일의 `이슈 → 내 작업 → 알림함 → 프로젝트` 상대 순서, `/issues/{issueIdentifier}`의 `이슈` 활성 상태와 `/my-issues/{teamWorkIdentifier}`의 `내 작업` 활성 상태·복귀 링크·새로고침·직접 진입
- 이슈 중심 상세와 내 작업 진입 상세의 공유 본문·동일 데이터, 상단 정보 위계 구분과 재배정·완료·취소 뒤 `내 작업에서 제외된 작업입니다` 안내
- 빠른 필터·세부 필터·정렬 URL 복원, 행 작업 요약·한 개의 주요 행동, 작업 시작·우리 팀에서 시작·내가 맡기·담당자 지정·이슈 완료
- 배정 충돌의 선택값 유지, SSE 뒤 현재 조건 행 제거와 필터·정렬·페이지·스크롤 유지, 모바일 카드·필터 패널

shadcn/ui 자체 동작과 브라우저가 보장하는 동작을 복제해 시험하지 않는다. Rivet에서 조합하며 추가한 상태와 도메인 동작만 검증한다.

### 5.7 이슈 중심 작업 흐름 회귀 세트

다음 사례는 가장 낮은 적절한 계층의 자동 테스트와 Chromium 대표 E2E를 조합해 모두 유지한다.

- 시작 역할을 선택하지 않으면 상위 이슈만 생성되고 `0/0 완료` 대신 분석·작업 시작 안내가 표시됨
- 백엔드만, 웹만, 앱만 또는 복수 역할을 선택하면 선택한 역할 작업만 생성됨
- 프로젝트에 없는 역할과 중복 역할 요청이 거부되고 중복 제출이 팀 작업을 중복 생성하지 않음
- 루트 이슈 또는 필요한 최초 팀 작업 삽입 실패 시 전체 트랜잭션 롤백
- 팀 작업이 없는 이슈에서 단일·복수 역할로 작업을 시작하고 기존 미완료 역할 작업은 재사용하며 실패 시 일부 작업이 남지 않음
- 동시에 시작한 역할 작업은 병렬이며 팀 작업 관계를 만들지 않음
- 최초 전달의 선택 역할별 웹·앱 대상 작업 자동 생성 또는 재사용
- 역할별 기존 미완료 대상 작업 재사용과 중복 생성 방지
- 동일 완료 요청 재시도 시 전달·대상 작업·Outbox·알림 중복 방지
- 추가 전달에서 새 팀 작업이 생성되지 않음
- 상위 이슈와 대상 프론트 작업에서 최초·추가 전달을 같은 원본 데이터로 조회함
- 알림이 대상 프론트 작업의 전달 위치를 열고 해당 전달을 확장함
- 새 미할당 대상 작업의 팀 활성 멤버 알림과 작성자 제외
- 전달 실패 시 백엔드 완료나 일부 대상 작업·활동이 남지 않음
- 전달 직후 실제 팀 작업 기준 진행률 계산
- 취소 작업과 생성되지 않은 프로젝트 역할이 진행률 분자·분모에서 제외됨
- 이슈·팀 작업 표시 ID 진입, 새 migration으로 팀 작업 관계를 제거한 개발 데이터 재생성과 미리 만든 대상 작업 재사용
- 전역 생성의 유형 선택기 제거와 프로젝트 주요 동작 `이슈 만들기`
- 최초 전달 전 `API 전달 대기`, 최초 전달 뒤 `작업 가능`, 준비 전 시작 API 거부
- 작업 흐름의 분석·병렬·전달·완료 상태와 전달 중심 사용자 문구
- 시작 역할 선택의 키보드·접근성 상태·오류 포커스, 보조 문구 대비와 모바일 읽기 순서

### 5.8 전역 이슈와 작업 배정 회귀 세트

- 전역 목록이 삭제되지 않은 이슈만 반환하고 다른 워크스페이스·휴지통 이슈를 제외함
- 검토 필요는 하위 작업이 전혀 없을 때만 참이고 완료·취소 작업 이력이 있는 이슈를 다시 포함하지 않음
- 담당 필요와 진행 중의 중첩, 완료 확인과 완료 조건이 상위·하위 상태 조합과 일치함
- 빠른 필터와 프로젝트·상태·우선순위·현재 역할·담당자 없음·라벨·생성자·생성일·수정일 필터가 OR/AND 규칙으로 결합됨
- 진행률·상태·우선순위가 같은 이슈의 불변 ID 동률 해소와 다중 페이지 중복·누락 부재
- 페이지 크기와 행 수를 늘려도 하위 작업·역할·멤버십·대기 관계 집계 쿼리 수가 고정되고 `workflowSummary`가 일치함
- 빠른 필터별 건수는 검색·세부 필터를 반영하고 현재 선택 `workQueue`만 제외하며 0건과 완료·취소 조합, 다른 워크스페이스·휴지통, 다중 페이지 전체 결과에서 정확함
- `initialRoles`와 `roleAssignments` 상호 배타, 역할별 담당자 활성 팀 멤버 검증과 복수 역할 전체 롤백
- 우리 팀에서 시작의 현재 팀 멤버십 서버 재검증과 미할당 작업 생성
- claim의 기존 미할당 작업 재사용, 작업 없음 시 생성, 다중 후보 선택 요구, 다른 담당자 충돌과 동시 요청 한 건 성공
- 일괄 배정의 부모·팀·버전·후보 검증, 한 건 실패 시 담당자·구독·활동·Outbox 전체 롤백
- 작업 창이 100건을 넘는 하위 작업과 활성 팀 멤버의 커서 페이지를 모두 읽어 완료 요약·claim·일괄 배정 후보를 누락하지 않음
- 작업 시작·배정 실패 뒤 선택 후보가 비활성화되면 기존 이름과 최신 후보를 설명하고 사용할 수 없는 ID를 재전송하거나 일부 선택만 저장하지 않음
- 완료 창을 연 뒤 하위 작업이 재개되면 빠른 완료의 서버 트랜잭션 재검증이 `ISSUE_COMPLETION_NOT_READY`로 거부하고, 모든 대상 작업 완료 뒤에는 같은 계약이 성공함
- `TEAM_WORK_ASSIGNED` 사용, 자신의 행동 알림 제외와 같은 배정 사건의 수신자별 중복 부재
- 데스크톱은 제목·결과 수가 한 헤더에 있고 검색·빠른 필터·세부 필터·정렬이 통합 도구 영역에 있으며, 여섯 열과 제한된 최대 폭 안에서 만든 사람 없이 제목·현재 작업·다음 행동 위계를 유지함
- 프로젝트·최대 두 개 라벨과 `+N`, 가벼운 상태·우선순위 빠른 편집, 진행 숫자·막대, 상대 업데이트와 역할이 하나일 때 구체적인 시작 문구가 표시됨
- 행 빈 영역 상세 이동과 내부 상태·우선순위·라벨·주요 행동·더보기 이벤트가 분리되고 빠른 필터 선택·건수, 진행률 값, 모바일 카드 읽기 순서와 200% 확대가 접근 가능함
- 모바일 카드와 데스크톱 행에서 한 개의 주요 행동, 키보드 상태와 URL·SSE 맥락 복원

### 5.9 CSV 가져오기 회귀 세트

- UTF-8과 BOM, CRLF, 따옴표·쉼표·줄바꿈 셀의 정상 파싱
- 빈 파일, UTF-16·잘못된 UTF-8·NUL, 빈·중복 헤더, 잘못된 행 컬럼 수와 `.csv`·MIME 제한
- 정확히 5MB·10,000행·50컬럼 파싱 경계 성공과 하나 초과 실패, 10,000행 execute의 실제 PostgreSQL 원자 저장 성공
- 선행 공백을 포함한 `=`, `+`, `-`, `@`, 탭·캐리지 리턴 formula injection 값 거부
- 필수 컬럼, 팀별 상태·멤버, 프로젝트·우선순위·라벨의 알 수 없는 값과 보관·비활성·다른 워크스페이스 대상 거부
- 파일 내부 source key 중복, 기존 source key 제외, 완료 파일 중복과 같은 실행 ID 동시 요청 한 번만 성공
- 완료 파일 중복 허용을 명시한 재실행에서 기존 source key 행만 제외되고 새 이슈가 생기지 않음
- 같은 source key의 서로 다른 실행을 워크스페이스 잠금으로 직렬화해 이슈 한 건만 생성
- 검증 뒤 팀·상태·프로젝트·라벨 version이 바뀌면 execute가 재검증을 요구하고 일부 데이터가 생기지 않음
- 설명의 멘션·파일 참조 거부, 이름으로 식별한 미지원 컬럼의 필드 매핑 거부와 모든 미매핑 컬럼의 화면 안내
- 저장 중 PostgreSQL 오류 뒤 프로젝트·라벨·이슈·팀 작업·연결·source row·표시 번호 카운터가 모두 원상 복구됨
- Member의 모든 가져오기 API `403`, 다른 워크스페이스 관리자의 실행 결과 `404`
- 오류·로그·분석 이벤트에 원본 파일, 원본 행, 제목·설명·이메일·mapping JSON이 없음
- 데스크톱 브라우저에서 파일 선택부터 결과 링크까지 6단계 완료, 오류 preview의 행·필드 접근 가능 이름과 모바일의 데스크톱 안내

## 6. 테스트 데이터와 격리

### 6.1 데이터 원칙

- 실행마다 고유한 이메일·워크스페이스·팀 접두어를 사용한다.
- 실제 사용자 데이터를 복사하지 않고 합성 데이터만 사용한다.
- 다른 워크스페이스 접근 차단을 위해 최소 두 개의 격리된 워크스페이스를 만든다.
- 고정 시드에는 시스템 상태 범주처럼 모든 테스트가 공유해도 되는 최소 데이터만 둔다.
- 테스트 순서에 따라 성공하는 공유 상태를 만들지 않는다.

### 6.2 PostgreSQL과 파일 초기화

- 자동 테스트 전용 데이터베이스만 초기화할 수 있다.
- `NODE_ENV=test`와 허용된 DB 이름·호스트 검증을 통과하지 않으면 초기화 명령을 거부한다.
- 통합 테스트는 마이그레이션 이력을 처음부터 적용한 데이터베이스에서 실행한다.
- MVP에서는 데이터베이스 하나를 초기화해 통합 테스트를 직렬 실행한다. 병렬화는 실행 시간이 실제 병목이 된 뒤 추가한다.
- 마이그레이션 스냅샷을 우회하는 `db push` 결과를 출시 검증으로 인정하지 않는다.
- 각 실행은 고유한 임시 `FILE_STORAGE_ROOT`를 사용하며 저장소 루트 밖을 삭제하지 못하게 한다.
- E2E 인증 링크는 가짜 메일 어댑터가 보관한 테스트 전용 URL을 통해 획득한다.

## 7. 검증 명령과 품질 게이트

### 7.1 루트 명령 계약

| 명령 | 목적 |
| --- | --- |
| `pnpm lint` | 전체 정적 검사 |
| `pnpm typecheck` | 웹·API·워커·공통 패키지 타입 검사 |
| `pnpm test` | 빠른 단위 테스트 |
| `pnpm test:coverage` | 패키지별 커버리지 리포트 생성 |
| `pnpm test:integration` | PostgreSQL·API·워커 통합 테스트 |
| `pnpm test:e2e` | 격리 포트의 웹·API·워커를 연결한 Playwright 핵심 흐름 |
| `pnpm api:contract:check` | OpenAPI 생성 결과와 `packages/api-client` 최신 상태 확인 |
| `pnpm build` | 운영용 `web`, `api`, `worker` 빌드 |
| `pnpm db:migrate:deploy` | 운영·개발의 커밋된 Prisma 마이그레이션 적용 |

구현 단계의 루트 `package.json`이 위 목적의 명령을 제공하고 Turbo가 내부 작업 순서를 관리한다. 실패를 무시하거나 `--force`로 통과시키는 배포 명령은 만들지 않는다.

`pnpm test:e2e`는 기본 개발 포트와 분리된 API `4300`, Web `3300`을 사용한다. 웹 E2E workspace가 직접 선언한 `tsx` 실행기로 Playwright worker의 TypeScript 소스 import를 처리하므로 임시 `NODE_OPTIONS`, 개인 로컬 loader 경로와 기존 프로세스 종료에 의존하지 않는다. 필요하면 `PLAYWRIGHT_API_PORT`, `PLAYWRIGHT_WEB_PORT`로 격리 포트만 대체할 수 있다.

### 7.2 백엔드 작업 전달 게이트

백엔드가 API 작업을 `프론트 작업 가능`으로 전달하기 전에 다음을 통과해야 한다.

1. 변경된 도메인·데이터베이스·API 통합 테스트
2. 다른 워크스페이스 ID와 권한 없는 역할의 실패 검증
3. OpenAPI와 생성 클라이언트 최신 상태
4. 성공 응답과 주요 오류 응답 예시
5. 개발 환경 실제 호출 확인
6. 호환성 영향, 마이그레이션 여부와 프론트 주의사항 기록

### 7.3 운영 출시 게이트

1. 잠금 파일 기준 의존성 설치
2. Prisma 스키마 검증과 클라이언트 생성
3. 린트와 타입 검사
4. 단위·통합 테스트
5. OpenAPI·생성 클라이언트 검증
6. Playwright 핵심 흐름
7. `web`, `api`, `worker` 운영 빌드
8. 마이그레이션 SQL과 배포·되돌리기 검토
9. 백업 상태와 운영 환경 변수 사전 확인

자동 CI/CD는 MVP 범위가 아니므로 같은 명령을 개발자 또는 운영자가 수동 실행할 수 있어야 한다. CI를 도입하더라도 게이트의 의미는 바꾸지 않는다.

### 7.4 불안정 테스트

- 재실행해 통과했다는 이유만으로 첫 실패를 성공 처리하지 않는다.
- 네트워크·시간·공유 상태 의존 원인을 제거하거나 출시 게이트에서 제외하고 추적한다.
- 핵심 보안·데이터 격리·마이그레이션 테스트는 불안정 상태로 예외 처리할 수 없다.
- E2E 자동 재시도는 원인 진단용 결과 수집에만 사용하며 첫 실패를 기록한다.

## 8. 성능과 실패 복구 검증

### 8.1 최소 성능 기준

- 정상 개발 데이터 규모에서 주요 읽기·변경 API p95 1초 이내를 확인한다.
- 이슈 목록, 검색, 타임라인과 알림 목록에 무제한 조회나 명백한 N+1 쿼리가 없는지 확인한다.
- 전역 이슈는 1건과 최대 페이지 크기의 결과에서 집계 쿼리 수가 동일한지 계수하고 `workflowSummary`, 만든 사람, `totalCount`와 `workQueueCounts`가 고정된 수의 조회로 완성되며 N+1이 생기지 않는지 확인한다.
- 25MB 경계 파일 업로드가 서버 메모리에 여러 사본으로 장시간 남지 않는지 확인한다.
- SSE 느린 클라이언트를 종료한 뒤 API 메모리와 연결 수가 회수되는지 확인한다.
- Outbox 50건 처리 중 API 응답이 실사용 불가능하게 지연되지 않는지 확인한다.
- CSV 10,000행 execute가 문서화한 180초 transaction timeout 안에 완료되고 이슈·팀 작업·source row·활동 수가 모두 일치하는지 실제 PostgreSQL에서 확인한다.

대규모 부하 테스트 플랫폼은 도입하지 않는다. 대표 데이터 시드와 짧은 반복 측정으로 병목을 찾고 실제 사용량이 늘 때 별도 도구를 검토한다.

### 8.2 실패 주입

- 트랜잭션 커밋 전 API 프로세스 종료
- 루트 이슈 생성 중 선택한 최초 팀 작업 삽입 실패, 작업 시작 중 두 번째 팀 작업 삽입 실패와 최초 전달 중 두 번째 후행 작업 삽입 실패
- Outbox 외부 호출 뒤 결과 기록 전 워커 종료
- PostgreSQL listener와 SSE 연결 강제 종료
- Resend 일시·영구 오류 응답
- 파일 저장 중 디스크 쓰기 실패와 연결 파일 바이너리 누락
- 마이그레이션 SQL 실패와 애플리케이션 시작 검증 실패

각 실패 뒤 중복 알림·메일·삭제, 저장 데이터 유실과 다른 워크스페이스 노출이 없어야 한다.

| `RESEND_API_KEY` | 워커 | 예 | 환경별 키 분리 |
| `RESEND_FROM` | 워커 | 아니오 | 환경에 맞는 검증된 발신 주소 |
| `RESEND_ALLOWED_RECIPIENTS` | 개발 워커 | 운영 통제 | 개발 실제 발송 허용 목록, 운영에서는 사용하지 않음 |
| `ONE_TIME_TOKEN_HMAC_KEY` | API·워커 | 예 | 최소 32바이트 무작위 값 |
| `CSRF_HMAC_KEY` | API | 예 | 최소 32바이트 무작위 값 |
| `RATE_LIMIT_HMAC_KEY` | API·워커 | 예 | 최소 32바이트 무작위 값 |
| `POSTHOG_API_KEY` | 허용된 분석 모듈 | 예 | 운영 프로젝트만 활성화 |
| `SLACK_ALERT_WEBHOOK_URL` | API·워커·운영 점검 | 예 | 운영 경고 전용 Slack 채널의 Incoming Webhook URL |
| `API_PORT`, `WEB_PORT` | API·웹 | 아니오 | Nginx 뒤의 로컬 바인딩 포트 |

실제 코드에서 추가 설정이 필요하면 소유 앱, 기본값 허용 여부와 시작 검증을 함께 문서화한다. 같은 의미의 설정 키를 앱마다 다르게 만들지 않는다.

### 9.2 시작 검증

- 필수 설정이 없거나 형식이 잘못되면 애플리케이션 시작을 실패시킨다.
- 운영 `WEB_ORIGIN`이 HTTPS가 아니거나 운영 보안 쿠키를 끄는 설정은 거부한다.
- API와 워커는 `FILE_STORAGE_ROOT`가 존재하고 필요한 최소 읽기·쓰기 권한이 있는지 확인한다.
- 코드가 요구하는 마이그레이션이 적용되지 않았다면 준비 완료 상태가 되지 않는다.
- 시작 오류에는 설정 키 이름만 남기고 값은 로그에 남기지 않는다.

### 9.3 비밀 값 교체

| 비밀 | 교체 절차의 핵심 |
| --- | --- |
| Resend 키 | 새 키 주입, 워커 재시작, 허용 주소 시험 발송, 기존 키 폐기 |
| 일회용 토큰 HMAC | 기존 미사용 링크 폐기 공지, 새 키 반영, 필요 링크 재발급 |
| CSRF HMAC | API 재시작, 클라이언트의 새 CSRF 토큰 재조회 확인 |
| 속도 제한 HMAC | 제한 버킷 연속성이 끊기므로 계획된 점검 시간에 교체 |
| PostgreSQL 자격 증명 | 새 계정·권한 확인, API·워커 반영, 기존 자격 증명 폐기 |

- 비밀은 저장소, 빌드 산출물, PM2 Ecosystem 파일의 커밋 내용과 배포 로그에 직접 기록하지 않는다.
- 유출이 의심되면 관련 비밀을 교체하고 노출 로그·산출물을 격리한 뒤 영향 기간의 인증·관리자 행위를 확인한다.
- 세션 토큰이나 일회용 토큰 원문이 노출되면 해당 세션·토큰을 데이터베이스에서 폐기한다.

## 10. 빌드와 릴리스 산출물

### 10.1 빌드 순서

1. Node.js 24 LTS와 저장소에 고정된 pnpm 버전 확인
2. `pnpm install --frozen-lockfile`
3. Prisma 스키마 검증과 클라이언트 생성
4. OpenAPI 생성과 `packages/api-client` 갱신 확인
5. `pnpm lint`, `pnpm typecheck`, `pnpm test`
6. `pnpm test:integration`, `pnpm test:e2e`
7. `pnpm build`

Turbo는 패키지 의존 관계에 따라 `packages/database`, API 계약과 앱 빌드 순서를 관리한다. 원격 캐시는 MVP 필수 요소가 아니다.

### 10.2 산출물 기준

- 운영 서버는 TypeScript 소스 실행 도구가 아니라 빌드된 JavaScript 또는 프레임워크 운영 산출물을 실행한다.
- `web`, `api`, `worker` 산출물은 같은 `RELEASE_ID`, 잠금 파일과 Prisma 스키마에서 만들어야 한다.
- Prisma 마이그레이션과 PM2 Ecosystem 파일은 릴리스 산출물과 함께 추적한다.
- 운영에 필요한 production 의존성만 포함하고 테스트 결과, 테스트 업로드와 로컬 환경 파일은 포함하지 않는다.
- 직전 정상 릴리스 산출물은 애플리케이션 되돌리기를 위해 최소 한 개 보존한다.

## 11. Prisma 마이그레이션 운영

### 11.1 작성과 검토

- 스키마 변경은 개발 환경에서 새 Prisma 마이그레이션 파일로 생성하고 코드 리뷰한다.
- 생성 SQL의 잠금 범위, 기존 행 처리, `NOT NULL`, 유일 제약, 인덱스와 데이터 손실 가능성을 직접 확인한다.
- 운영 데이터 수정이 필요하면 재실행 가능한 별도 단계로 작성하고 처리 건수와 실패 기준을 남긴다.
- 이미 운영에 적용한 마이그레이션 파일을 수정하거나 삭제하지 않는다.
- 운영에서 `prisma migrate dev`, `db push`와 수동 DDL을 사용하지 않는다.

### 11.2 호환 가능한 변경 순서

1. 기존 코드가 계속 동작하는 nullable 필드·테이블·인덱스를 추가한다.
2. 새 코드가 기존·신규 형태를 함께 읽을 수 있게 배포한다.
3. 필요한 데이터를 채우고 새 쓰기로 전환한다.
4. 모든 앱 전환과 검증이 끝난 후 별도 릴리스에서 기존 필드·계약을 제거한다.

한 번의 MVP 배포를 위해 위험한 대형 잠금이나 비호환 변경을 강행하지 않는다.

이슈·팀 작업 분리 개편은 아직 실데이터가 없는 개발 단계의 명시적 예외다. 기존 마이그레이션 파일은 수정하지 않되 새 마이그레이션에서 구형 팀 작업 컬럼·enum·관계 테이블을 바로 제거하고 개발·테스트 데이터베이스를 초기화할 수 있다. API·Worker·Web·OpenAPI·생성 클라이언트는 한 변경에서 함께 전환하며 구형 DTO, 이중 쓰기와 deprecated 호환 필드를 남기지 않는다. 이 예외는 운영 데이터가 생긴 뒤의 일반 변경에 재사용하지 않는다.

### 11.3 운영 적용

1. 대상 릴리스와 마이그레이션 목록을 확인한다.
2. 최근 PostgreSQL·파일 백업 상태를 확인한다.
3. 새 쓰기를 잠시 중단할 필요가 있는지 SQL을 검토한다.
4. 운영 `DATABASE_URL`을 확인한 뒤 `pnpm db:migrate:deploy`를 한 번 실행한다.
5. 적용 결과와 Prisma 마이그레이션 이력을 확인한다.
6. 성공한 경우에만 애플리케이션 프로세스 배포를 계속한다.

`migrate deploy`는 커밋된 미적용 마이그레이션을 적용하는 운영 명령으로 사용한다. 스키마 드리프트 탐지와 생성 클라이언트 빌드는 별도 출시 게이트에서 수행한다.

### 11.4 실패와 되돌리기

- 마이그레이션 실패 즉시 애플리케이션 재시작을 중단하고 실패한 SQL과 적용 범위를 확인한다.
- 부분 적용 여부를 추측해 마이그레이션 테이블을 직접 수정하지 않는다.
- 데이터 손상이 없고 원인을 수정할 수 있으면 새 교정 마이그레이션으로 전진 복구한다.
- 애플리케이션 되돌리기는 새 스키마와 호환되는 직전 버전만 허용한다.
- 운영 스키마를 자동 `down` SQL로 되돌리는 것을 기본 복구 전략으로 사용하지 않는다.
- 데이터 손상 가능성이 있으면 쓰기를 중단하고 검증된 백업 복구 절차로 전환한다.

## 12. PM2 프로세스 운영

### 12.1 프로세스 선언

저장소 루트의 하나의 PM2 Ecosystem 파일에서 다음 세 프로세스를 선언한다. 실제 포트, 서버 경로와 비밀 값은 파일에 하드코딩하지 않는다.

| 이름 | 실행 단위 | 모드·인스턴스 | 준비 완료 조건 |
| --- | --- | --- | --- |
| `rivet-web` | `apps/web` 운영 서버 | fork, 1개 | 로컬 HTTP 응답 가능 |
| `rivet-api` | `apps/api` NestJS 서버 | fork, 1개 | 설정·DB·파일 루트·listener 확인 후 HTTP 수신 |
| `rivet-worker` | `apps/worker` NestJS 프로세스 | fork, 1개 | 설정·DB·파일 루트 확인 후 polling 시작 가능 |

- 운영에서 `watch`를 사용하지 않는다.
- 세 앱 모두 자동 재시작과 시작 실패 반복 제한을 설정한다.
- 메모리 재시작 한도는 서버 실제 메모리를 확인해 앱별로 지정하고 임의의 공통값을 고정하지 않는다.
- 서버 재부팅 후 프로세스 목록이 복원되도록 운영 서버에서 PM2 startup과 save 절차를 완료한다.

### 12.2 시작 순서

일반적인 호환 배포는 다음 순서를 사용한다.

1. 환경 변수와 공유 `FILE_STORAGE_ROOT` 확인
2. Prisma 마이그레이션 적용
3. `rivet-api` 시작 또는 재시작
4. API 준비 상태와 PostgreSQL listener 확인
5. `rivet-worker` 시작 또는 재시작
6. Outbox 처리와 정기 작업 상태 확인
7. `rivet-web` 시작 또는 재시작
8. Nginx 경유 상태 확인과 핵심 smoke test

Outbox payload의 `schemaVersion`이 바뀌는 릴리스는 예외로 새 워커가 기존·신규 payload를 모두 읽게 먼저 배포한 뒤 API가 새 버전을 발행하게 한다. 구버전 처리 지원은 대기·실패 이벤트가 없음을 확인한 이후 릴리스에서 제거한다.

### 12.3 정상 종료

| 프로세스 | 종료 시 동작 | 최대 대기 권장값 |
| --- | --- | ---: |
| API | 새 요청 수신 중단, SSE 종료, HTTP·listener·DB 연결 정리 | 30초 |
| 워커 | 새 claim 중단, 진행 중 작업 종료 시도, DB 연결 정리 | 30초 |
| 웹 | 새 요청 수신 중단, 진행 요청 정리 | 10초 |

- NestJS shutdown hook과 PM2 종료 신호를 연결한다.
- 제한 시간 뒤 강제 종료된 워커 작업은 5분 잠금 만료 후 회수되며 처리기는 멱등해야 한다.
- 종료 중 이메일·알림·삭제 결과를 확인하지 못했다면 성공으로 추정하지 않고 재처리 가능 상태를 유지한다.
- PM2는 애플리케이션 준비 신호를 기다린 뒤 새 프로세스를 정상으로 판단하게 한다.

### 12.4 프로세스 오류

- 짧은 시간에 반복 재시작하면 자동 재시작만 계속하지 않고 오류 상태로 남겨 경고한다.
- 시작 로그에는 프로세스명, 환경, 릴리스와 정제된 실패 코드가 있어야 한다.
- API와 워커의 Prisma 클라이언트 버전 또는 설정이 다르면 출시를 중단한다.
- 워커가 내려가도 API 업무 저장은 가능하지만 Outbox 지연 경고가 발생해야 한다.

## 13. Nginx 운영 기준

### 13.1 같은 출처 라우팅

| 외부 경로 | 대상 | 추가 기준 |
| --- | --- | --- |
| `/` | `rivet-web` | Next.js 운영 응답과 정적 자산 |
| `/api/v1/events` | `rivet-api` | SSE 전용 버퍼링·캐시·압축 해제 |
| `/api/v1/*` | `rivet-api` | JSON API, 업로드와 인증된 파일 조회 |

- 외부 HTTPS 출처 하나만 사용하고 운영 CORS를 활성화하지 않는다.
- HTTP 요청은 HTTPS로 영구 리다이렉트한다.
- API·웹 내부 포트와 `FILE_STORAGE_ROOT`를 공용 네트워크나 정적 경로로 직접 공개하지 않는다.
- 클라이언트가 보낸 전달 헤더를 그대로 신뢰하지 않고 Nginx가 `Host`, 프로토콜과 실제 연결 주소를 덮어쓴다.
- 애플리케이션은 신뢰 프록시를 Nginx 한 홉으로 제한한다.

### 13.2 업로드와 일반 프록시

- Nginx `client_max_body_size`는 multipart 오버헤드를 포함해 30MB로 설정한다.
- API는 Nginx 통과 여부와 관계없이 실제 파일 25MB 제한과 형식을 다시 검증한다.
- 파일 응답은 API 권한 확인 뒤 제공하고 위험 형식은 강제 다운로드한다.
- 일반 API 제한 시간은 유한값을 사용하되 CSV 스트리밍과 SSE는 별도 위치에서 설정한다.
- 보안 헤더는 Nginx와 앱 중 책임 위치를 하나로 정해 중복·충돌하지 않게 한다.

### 13.3 SSE

SSE 위치는 일반 API보다 먼저 일치하도록 분리하고 다음 의미를 유지한다.

```nginx
location = /api/v1/events {
    proxy_http_version 1.1;
    proxy_set_header Connection "";
    proxy_buffering off;
    proxy_cache off;
    gzip off;
    proxy_read_timeout 75s;
}
```

실제 `proxy_pass`, 공통 전달 헤더와 TLS 설정은 운영 서버 구성에 맞춰 추가한다.

- API도 SSE 응답에 `X-Accel-Buffering: no`, `Cache-Control: no-cache`와 올바른 `Content-Type`을 보낸다.
- 20초 heartbeat가 있으므로 Nginx 읽기 제한은 이를 충분히 넘는 75초를 기본값으로 한다.
- 연결 종료를 숨기는 무제한 캐시나 응답 압축을 사용하지 않는다.
- 배포 뒤 브라우저에서 heartbeat와 재연결 후 REST 재조회를 확인한다.

## 14. 상태 확인과 준비 상태

### 14.1 API 운영 엔드포인트

| 경로 | 성공 조건 | 실패 동작 |
| --- | --- | --- |
| `GET /api/v1/health/live` | Node.js 프로세스가 요청 처리 가능 | 프로세스 오류 시 연결 실패 또는 `503` |
| `GET /api/v1/health/ready` | 필수 설정, DB, 파일 루트와 listener 정상 | 하나라도 불가하면 `503` |

- 두 엔드포인트는 세션 없이 호출할 수 있지만 민감한 설정, DB 주소, 경로와 예외 내용을 반환하지 않는다.
- 성공은 `{ "status": "ok" }`, 실패는 `{ "status": "unavailable" }` 수준으로 제한한다.
- `Cache-Control: no-store`를 적용한다.
- liveness는 외부 의존성을 조회하지 않고 이벤트 루프가 응답 가능한지만 확인한다.
- readiness의 DB 확인은 짧은 제한 시간의 가벼운 쿼리를 사용한다.
- 파일 루트는 시작 시 읽기·쓰기 검증하고 readiness에서는 최근 검증 상태와 접근 가능 여부를 확인한다.

### 14.2 웹과 워커

- 웹은 Nginx를 거친 로그인 화면과 정적 자산 응답으로 확인한다.
- 워커는 HTTP 엔드포인트를 만들지 않는다.
- 워커 준비 상태는 PM2 상태, 시작 준비 신호, 최근 polling과 `maintenance_last_success_at`으로 확인한다.
- 프로세스가 `online`이어도 Outbox가 5분 넘게 지연되거나 정기 작업이 26시간 넘게 성공하지 않으면 비정상 운영 상태다.

## 15. 수동 배포 절차

### 15.1 배포 전

1. 대상 커밋과 `RELEASE_ID`를 확정한다.
2. 출시 게이트 전체 결과를 확인한다.
3. 환경 변수·비밀 값 누락과 서버 디스크 여유 공간을 확인한다.
4. PostgreSQL과 `FILE_STORAGE_ROOT`의 최근 백업 성공을 확인한다.
5. 마이그레이션 SQL, 잠금 가능성과 구버전 호환성을 확인한다.
6. 직전 정상 릴리스 산출물이 남아 있는지 확인한다.
7. 점검 시간과 되돌리기 판단 담당자를 정한다.

### 15.2 배포 실행

MVP 확정안은 짧은 점검 시간을 허용하는 단일 서버 제자리 배포다.

1. 새 릴리스 디렉터리에 검증된 산출물을 준비한다.
2. 필요한 경우 사용자에게 짧은 점검 상태를 안내하고 새 쓰기를 중단한다.
3. `pnpm db:migrate:deploy`를 실행하고 결과를 확인한다.
4. PM2 시작 순서에 따라 API, 워커와 웹을 반영한다.
5. Nginx는 경로·포트 구성이 바뀐 경우에만 설정 검사 후 reload한다.
6. liveness·readiness와 PM2 상태를 확인한다.
7. Nginx를 거친 smoke test를 수행한다.
8. 오류율, Outbox 지연과 listener 상태를 집중 확인한다.
9. 성공 릴리스 시각, 마이그레이션과 확인자를 기록한다.

저장소는 서버 접속, 파일 전송과 PM2 실행을 자동화하지 않는다. 운영자는 이 순서를 자체 서버 환경에 맞게 실행한다.

### 15.3 배포 후 smoke test

| 순서 | 점검 | 성공 기준 |
| ---: | --- | --- |
| 1 | 웹 접속 | HTTPS 로그인 화면과 정적 자산 정상 |
| 2 | API 상태 | liveness·readiness `200` |
| 3 | 인증 | 운영 전용 점검 계정 로그인·로그아웃 성공 |
| 4 | 기본 조회 | 프로젝트·이슈 목록이 현재 워크스페이스만 반환 |
| 5 | 변경 | 점검용 이슈 생성·수정·삭제 또는 역할별 작업 시작·담당자 지정 같은 사전에 정한 비파괴 변경 성공 |
| 6 | 워커 | Outbox가 처리되고 새 `FAILED`가 없음 |
| 7 | 알림 | 점검 이벤트 알림과 읽지 않은 수 일치 |
| 8 | SSE | 연결, heartbeat, 강제 재연결 뒤 최신 상태 수렴 |
| 9 | 파일 | 작은 점검 파일 업로드·조회·연결 해제 성공 |
| 10 | 로그 | 토큰·이메일·업무 본문 없이 릴리스·추적 ID 확인 가능 |

운영 점검 계정과 워크스페이스는 실제 사용자 데이터와 분리한다. 점검 데이터는 확인 뒤 제품 규칙에 따라 삭제한다.

### 15.4 성공과 중단 판단

다음 중 하나라도 발생하면 배포를 성공 처리하지 않는다.

- readiness 실패 또는 반복적인 PM2 재시작
- 새 5xx 오류의 지속 발생
- 로그인·워크스페이스 격리·이슈 저장 실패
- 마이그레이션 불일치 또는 데이터 손상 징후
- 워커가 처리하지 못해 Outbox 지연이 계속 증가함
- PostgreSQL listener 미연결과 SSE 신규 연결 실패
- 파일 업로드·조회 또는 공유 저장소 권한 실패
- 비밀 값, 토큰이나 업무 본문이 로그에 노출됨

## 16. 애플리케이션 되돌리기

### 16.1 되돌리기 가능 조건

- 마이그레이션이 추가형이며 직전 애플리케이션이 새 스키마에서도 동작한다.
- 새 API가 저장한 데이터와 Outbox payload를 직전 워커가 안전하게 처리할 수 있다.
- 파일 저장 형식과 경로가 직전 버전과 호환된다.

### 16.2 절차

1. 새 쓰기와 worker claim을 중단한다.
2. 장애 시각, 릴리스와 영향을 기록한다.
3. 직전 정상 `api`, `worker`, `web` 산출물을 호환 순서로 복원한다.
4. 프로세스 준비 상태와 listener를 확인한다.
5. 핵심 smoke test와 Outbox 대기·실패 상태를 확인한다.
6. 사용자 쓰기를 다시 허용하고 원인 분석과 교정 릴리스를 준비한다.

데이터베이스 스키마는 기본적으로 되돌리지 않는다. 직전 앱이 새 스키마와 호환되지 않거나 데이터 손상이 발생했다면 일반 애플리케이션 되돌리기를 중단하고 전진 교정 마이그레이션 또는 백업 복구 절차를 선택한다.

## 17. 백업과 복구

### 17.1 백업 대상

- PostgreSQL 전체 데이터와 마이그레이션 이력
- `FILE_STORAGE_ROOT`의 최종 파일과 필요한 임시 처리 상태
- 운영에 사용한 릴리스 산출물과 Prisma 마이그레이션
- Nginx·PM2 설정의 비밀 값이 제거된 복구 가능한 사본
- 환경 변수 이름과 발급·교체 절차. 실제 비밀 값은 별도 안전한 저장소에서 관리

PM2 로그, 빌드 캐시와 `tmp`의 오래된 파일은 핵심 복구 대상이 아니다.

### 17.2 백업 정합성

- PostgreSQL 파일 메타데이터와 `FILE_STORAGE_ROOT` 바이너리를 가능한 한 같은 복구 지점으로 보관한다.
- MVP의 정기 전체 백업은 짧은 점검 시간에 API 새 쓰기와 worker 처리를 멈추고 두 저장소를 순서대로 백업하는 방식을 권장한다.
- 무중단 개별 백업을 사용하면 각 백업 시각을 기록하고 복구 후 파일 정합성 검사를 필수 수행한다.
- 백업 파일은 운영 서버와 다른 저장 위치에 두고 접근 권한과 보존 기간을 제한한다.
- 백업 성공은 명령 종료 코드뿐 아니라 생성 파일 크기, 목록과 복원 가능성으로 확인한다.

### 17.3 복구 절차

1. 장애 원인과 복구 기준 시각을 확정하고 사용자 쓰기를 중단한다.
2. 손상된 현재 데이터와 파일을 덮어쓰기 전에 격리한다.
3. 격리된 복구 환경에서 PostgreSQL을 복원한다.
4. 같은 복구 지점의 `FILE_STORAGE_ROOT`를 복원한다.
5. 해당 스키마와 호환되는 애플리케이션 릴리스를 시작한다.
6. 마이그레이션 이력, 행 수와 핵심 관계를 확인한다.
7. 파일 정합성 작업을 탐지 전용으로 실행해 누락·고아 파일을 확인한다.
8. 워크스페이스 격리, 로그인, 이슈, Outbox, 알림과 파일 smoke test를 수행한다.
9. 결과와 데이터 손실 범위를 기록한 뒤 운영 전환을 승인한다.

복구 검증 전 고아 파일 삭제 작업을 실행하지 않는다. 연결 파일 누락은 자동으로 참조를 지우지 않고 복구 판단 대상으로 남긴다.

### 17.4 복구 훈련

- MVP 운영 출시 전에 PostgreSQL과 파일 저장소를 함께 복구하는 전체 훈련을 한 번 성공해야 한다.
- 훈련은 운영 원본을 덮어쓰지 않는 격리 환경에서 수행한다.
- 실제 복구 시간, 누락 단계, 필요한 권한과 수동 판단을 기록해 이 절차를 갱신한다.
- 운영에서는 PostgreSQL과 `FILE_STORAGE_ROOT`의 복구 가능한 백업이 24시간 안에 최소 한 번 성공해야 한다.
- 장애가 발생하면 복구 결정을 내린 시점부터 4시간 안에 검증된 서비스 재개를 목표로 한다.
- 백업이 24시간 동안 성공하지 못하면 즉시 경고하고 다음 백업만 기다리지 않고 원인을 해결한다.
- 큰 스키마 변경이나 파일 저장 방식 변경 뒤에는 같은 전체 복구 훈련을 다시 수행한다.

## 18. 로그와 관측성

### 18.1 구조화 로그

| 앱 | 필수 필드 |
| --- | --- |
| API | timestamp, level, environment, releaseId, requestId, method, path template, status, duration, 내부 workspace·membership ID |
| 워커 | timestamp, level, environment, releaseId, jobId, event type, attempt, duration, result, 정제 오류 코드 |
| 웹 서버 | timestamp, level, environment, releaseId, 요청 경로 템플릿, 상태, 정제 예외 이름 |

- JSON 한 줄 형식을 기본으로 하고 PM2·Nginx 로그 순환을 설정한다.
- 자체 서버의 구조화 운영 로그는 30일 뒤 삭제한다.
- 서버 시각은 UTC로 기록하고 화면 표시와 운영 보고에서 KST로 변환한다.
- 여러 줄 스택은 구조화 필드로 묶되 사용자 입력이 로그 구조를 깨뜨리지 않게 한다.

### 18.2 기록 금지

- 비밀번호, 세션 쿠키, CSRF 토큰과 일회용 토큰
- `Authorization`, `Cookie`, `Set-Cookie` 헤더
- 이메일과 IP 원문, Resend 응답 본문
- 이슈 제목·설명·댓글·작업 전달과 원본 파일명
- 요청·응답 전체 본문, CSV 내용과 파일 바이너리
- `DATABASE_URL`, Resend·PostHog 키와 HMAC 키

오류 분석을 위해 업무 본문이 필요하더라도 자동 수집하지 않고 사용자의 명시적 제공 절차를 사용한다.

### 18.3 PostHog

- 운영 환경의 허용된 제품 이벤트와 처리되지 않은 정제 예외만 전송한다.
- 예외에는 오류 이름, 정제된 스택, 환경, 릴리스와 `requestId` 또는 `jobId`만 포함한다.
- 업무 본문, 이메일, 토큰, 원본 파일명과 공급자 응답을 속성에 넣지 않는다.
- PostHog 전송 실패가 API 요청과 워커 업무 처리 성공을 되돌리지 않는다.
- PostHog 데이터는 선행 보안 문서에 따라 1년 보관한다.

## 19. 지표와 경고

### 19.1 최소 지표

- API 요청 수, p95 지연, 4xx·5xx 비율과 DB 연결 상태
- PM2 프로세스 상태, 반복 재시작과 메모리 사용
- `outbox_pending_count`, `outbox_oldest_pending_seconds`, `outbox_failed_count`
- 이메일 결과, 알림 생성과 워커 처리 시간
- `sse_connections`, `sse_reconnect_total`, `postgres_listener_connected`
- `maintenance_last_success_at`, 정리 삭제 수와 연결 파일 누락 수
- Nginx 5xx, 업로드 거부와 서버·파일 저장소 디스크 사용량
- CSV 가져오기 성공·검증 실패·전체 롤백 수와 처리 행 수. 원본 파일명·source key·업무 본문은 지표 label로 사용하지 않음

### 19.2 경고 조건

| 조건 | 기본 심각도 |
| --- | --- |
| 다른 워크스페이스 데이터·파일 노출 또는 비밀 값 로그 노출 | 긴급 |
| 데이터 손상, 저장 성공 뒤 유실 또는 복구 불가능 징후 | 긴급 |
| API readiness 지속 실패, DB 연결 불가 또는 반복 재시작 | 높음 |
| 새 `FAILED` Outbox 발생 | 높음 |
| 가장 오래된 처리 가능 이벤트가 5분 초과 | 높음 |
| PostgreSQL listener가 30초 넘게 재연결되지 않음 | 높음 |
| 연결된 파일 바이너리 누락 | 높음 |
| 정기 maintenance가 26시간 안에 성공하지 않음 | 보통 |
| Resend 설정 오류 또는 공급자 연속 실패 | 보통, 인증 흐름 전체 차단 시 높음 |
| DB·파일 저장소 디스크 사용량이 운영 임계값 초과 | 높음 |

### 19.3 Slack Incoming Webhook 전달 기준

- 모든 운영 경고의 최종 수신처는 `SLACK_ALERT_WEBHOOK_URL`이 연결된 운영 전용 Slack 채널 하나로 통일한다.
- Webhook URL은 채널에 메시지를 게시할 수 있는 비밀 값이므로 저장소, 로그, 오류 추적과 화면에 노출하지 않는다.
- Slack SDK나 Bot API를 추가하지 않고 런타임의 HTTP 클라이언트로 JSON payload를 POST한다.
- 메시지 첫 줄은 `[Rivet][환경][심각도] 경고 유형` 형식으로 작성한다.
- 본문에는 발생 시각, 환경, `RELEASE_ID`, 정제 오류 코드, `requestId` 또는 `jobId`와 확인 절차만 포함한다.
- 이메일, 토큰, 업무 본문, 원본 파일명, 비밀 값과 공급자 응답 본문은 Slack 메시지에 넣지 않는다.
- Resend 자체 장애도 Slack 경로로 전달해 같은 이메일 공급자에 순환 의존하지 않는다.
- API·워커 프로세스 중단은 서버·오류 추적 측 점검이 같은 Slack Webhook으로 알리게 한다.
- Webhook 호출 실패가 원래 업무 처리 결과를 되돌리지 않으며 정제된 실패를 구조화 로그와 PostHog에 남긴다.
- 운영 출시 전과 Webhook·채널 변경 뒤 긴급·높음 등급의 시험 경고를 각각 한 번 보내 수신을 확인한다.
- 별도 이메일·SMS·모바일 푸시 경고 채널은 MVP에 추가하지 않는다.
- 디스크 경고 임계값은 실제 서버 용량과 평상시 사용량을 확인해 운영 환경에서 지정한다.

## 20. 장애 대응 절차

### 20.1 공통 순서

1. 사용자 영향, 시작 시각, 환경과 `RELEASE_ID`를 확인한다.
2. 추가 손상 가능성이 있으면 새 쓰기 또는 관련 worker 처리기를 중단한다.
3. `requestId`, `jobId`와 내부 ID로 로그 범위를 좁힌다.
4. 최근 배포·마이그레이션·비밀 교체와 외부 공급자 상태를 확인한다.
5. 되돌리기, 전진 수정, 재시도 또는 백업 복구 중 가장 안전한 방법을 선택한다.
6. 복구 뒤 핵심 smoke test와 데이터 정합성을 확인한다.
7. 원인, 영향, 임시 조치와 재발 방지 작업을 기록한다.

### 20.2 `FAILED` Outbox

1. 이벤트 ID, 유형, 시도 횟수와 정제 오류 코드를 조회한다.
2. payload의 업무 본문을 로그로 복사하지 않고 참조 리소스의 현재 상태를 권한 있는 방식으로 확인한다.
3. 설정·코드·외부 공급자·데이터 오류 중 원인을 수정한다.
4. 이미 외부 부작용이 발생했는지 멱등 키와 결과 기록을 확인한다.
5. 운영자 승인 뒤 선택한 이벤트만 재시도 가능 상태로 전환한다.
6. 성공 또는 다시 실패한 결과와 판단 근거를 운영 기록에 남긴다.

MVP에서는 관리자 큐 화면과 전체 일괄 재시도를 만들지 않는다. 데이터베이스에서 이벤트를 삭제하거나 payload를 즉석 수정해 해결하지 않는다.

### 20.3 Resend 장애

- 운영 키, 발신 주소, 도메인 검증과 공급자 응답 분류를 확인한다.
- 일시 오류는 확정된 Outbox 간격으로 재시도한다.
- 설정·권한 같은 영구 오류는 자동 재시도를 끝내고 경고한다.
- 원인 수정 뒤 토큰이 아직 유효한 이벤트만 승인해 재시도한다.
- 토큰이 만료됐다면 기존 이벤트를 고치지 않고 사용자에게 새 링크 발급 흐름을 제공한다.

### 20.4 listener와 SSE 장애

- `postgres_listener_connected`, API 로그와 PostgreSQL 연결 상태를 확인한다.
- listener가 없으면 기존 SSE를 종료하고 신규 SSE는 `503`으로 거부하되 가능한 REST API는 유지한다.
- 연결 복구 뒤 브라우저가 다시 연결하고 REST 재조회로 최신 상태에 수렴하는지 확인한다.
- Nginx 버퍼링, 압축, 읽기 제한과 heartbeat 수신을 함께 확인한다.
- 과거 SSE 이벤트 재생으로 복구하려 하지 않는다.

### 20.5 파일 저장소 장애

- 디스크 용량, 마운트, API·워커 권한과 `FILE_STORAGE_ROOT` 일치 여부를 확인한다.
- 경로 조작이나 저장소 루트 이탈 징후가 있으면 업로드·파일 정리를 즉시 중단한다.
- 메타데이터만 있고 바이너리가 없으면 참조를 자동 삭제하지 않는다.
- 백업 복구 가능 여부를 확인하고 화면에는 파일을 불러올 수 없음 상태를 유지한다.
- 복구 전 고아 파일 삭제를 실행하지 않는다.

### 20.6 보안 사고

- 다른 워크스페이스 노출, 토큰·비밀 노출과 권한 우회는 긴급 사고로 취급한다.
- 관련 엔드포인트 또는 전체 쓰기를 중단하고 노출된 비밀·세션·토큰을 폐기한다.
- 로그와 PostHog에서 노출 범위를 확인하되 민감 정보를 추가 복제하지 않는다.
- 수정 뒤 격리·권한·CSRF·토큰 재사용 자동 테스트를 통과하기 전 운영을 재개하지 않는다.

## 21. 출시 차단 결함

다음 결함이 하나라도 열려 있으면 MVP를 출시하지 않는다.

- 다른 워크스페이스의 데이터, 알림, 검색 결과, 파일 또는 CSV 노출
- 관리자 권한 우회, 이메일 인증 우회와 세션·토큰 재사용
- 표시 ID 중복, 저장 성공 뒤 데이터 유실과 마이그레이션 불일치
- 이슈 생성 또는 최초 전달 실패 뒤 루트·팀 작업·전달·관계·활동·Outbox의 부분 성공
- 전역 이슈 빠른 필터 오분류, 진행률 안정 커서의 중복·누락 또는 페이지 행 수에 따라 증가하는 하위 작업 N+1 조회
- 작업 시작·claim·일괄 배정 실패 뒤 팀 작업·담당자·구독·활동·Outbox 부분 성공이나 오래된 화면의 기존 담당자 덮어쓰기
- 프로젝트에 프론트 역할이 있는데 최초 전달 없이 백엔드 작업이 완료됨
- 같은 이벤트 재처리로 알림·메일·파일 삭제가 중복됨
- 연결된 파일을 고아 파일 정리가 삭제함
- API와 워커가 서로 다른 파일 저장소 또는 데이터베이스를 사용함
- 배포 후 API readiness, worker 처리와 핵심 로그인·이슈 흐름 실패
- 데이터베이스와 파일 저장소의 검증된 복구 절차 부재
- 비밀번호, 쿠키, 토큰, 비밀 키 또는 업무 본문이 로그·분석에 기록됨
- CSV 가져오기 실패 뒤 프로젝트·이슈·라벨·연결·source row·카운터가 일부 남거나 중복 source key가 생성됨
- 다른 워크스페이스의 가져오기 매핑 후보·실행 결과 노출 또는 Member의 가져오기 실행 성공

경미한 시각 오류와 비핵심 브라우저 호환 문제는 사용자 영향과 우회 방법을 기록하고 별도 판단할 수 있다.

## 22. MVP에서 도입하지 않는 구성

- Kubernetes, 컨테이너 오케스트레이션과 서비스 메시
- 블루그린, 카나리와 자동 롤백 배포
- Redis 기반 분산 잠금·큐·세션
- 별도 로그 검색 클러스터와 메트릭 시계열 플랫폼
- 관리자용 Outbox 재처리 UI
- 전체 브라우저·기기 조합 E2E와 대규모 부하 테스트 플랫폼
- 운영 데이터 자동 익명화 복제 환경
- 자동 데이터베이스 down migration

| 현재 구성 | 확장 검토 조건 |
| --- | --- |
| 수동 제자리 배포 | 배포 빈도·중단 시간이 사용자 업무를 반복 방해함 |
| PM2 단일 인스턴스 | 장애·부하로 가용성 목표를 지속적으로 충족하지 못함 |
| 직렬 통합·E2E | 검증 시간이 개발 흐름의 실제 병목이 됨 |
| Slack 운영 채널 하나 | 담당자 증가로 누락·인계 문제가 발생함 |
| 자체 PostgreSQL·로컬 파일 | 서버 분리, 용량 또는 복구 요구를 충족하지 못함 |
