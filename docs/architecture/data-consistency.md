# 데이터 일관성

> 상태: 현행
> 적용 범위: PostgreSQL, Prisma, API 트랜잭션과 Worker 처리
> 정본 책임: 원자성, 계산 상태, 멱등성, 동시성과 migration 원칙
> 갱신 조건: 데이터 불변식, 쓰기 경계, 이벤트 또는 migration 전략 변경

데이터 구조와 제약의 정본은
[`packages/database/prisma`](../../packages/database/prisma/), 실제 쓰기 규칙의 정본은
[`apps/api/src/modules`](../../apps/api/src/modules/)다.

## 워크스페이스 격리

- 업무 데이터는 `workspaceId`를 최상위 격리 키로 사용한다.
- 관계를 연결할 때 ID 하나만 확인하지 않고 같은 워크스페이스에 속하는지 함께 검증한다.
- 복합 unique와 relation은 가능한 범위에서 워크스페이스 경계를 데이터베이스에도 반영한다.
- 사용자가 전달한 워크스페이스 ID를 세션 범위 대신 신뢰하지 않는다.

## 원자적 쓰기

다음 변경은 관련 행, 활동과 Outbox를 하나의 PostgreSQL 트랜잭션으로 처리한다.

- 이슈 생성과 최초 팀 작업 시작
- 여러 팀 작업 시작·배정·claim
- 팀 작업 완료와 다른 팀으로의 전달
- 상태·담당자 변경에 따른 이슈 상태·진행률 갱신
- 휴지통 이동·복원과 영구 삭제 예약
- 가져오기 실행과 source key 기록
- 배포 완료와 관련 의존성 갱신

트랜잭션 실패 뒤 일부 팀 작업, 전달, 활동 또는 이벤트가 남아서는 안 된다.

## 계산 상태

- 이슈가 업무 내용, 팀 작업이 실행 상태의 원본이다.
- 이슈 상태와 진행률은 연결된 팀 작업의 시스템 범주에서 계산한다.
- 표시용 워크플로 이름이나 팀 종류를 계산 규칙에 사용하지 않는다.
- 취소된 팀 작업은 진행률 대상에서 제외한다.
- 배포 상태는 팀 작업 완료와 별도이며 배포 의존성과 완료 기록에서 계산한다.

계산 결과를 저장하는 경우 원본 변경과 같은 트랜잭션에서 갱신하고, 일괄 보정 작업은 같은 계산
함수 또는 명시적으로 검증된 SQL 계약을 사용한다.

## 동시성

- 변경 가능한 주요 리소스는 `version`을 사용해 오래된 쓰기를 감지한다.
- 카운터, 기본값, 참여 팀, 담당자와 상태 전이처럼 경쟁 가능한 쓰기는 필요한 행을 잠근다.
- 충돌을 마지막 쓰기 승리로 숨기지 않고 명시적인 `409` 계약으로 반환한다.
- 재시도 화면은 사용자의 입력을 보존하고 최신 리소스·선택지를 다시 읽는다.
- 목록 커서는 결정적인 마지막 키를 포함해 동률 데이터의 중복과 누락을 막는다.

## 멱등성

- 클라이언트가 재시도할 수 있는 생성·제출은 안정적인 요청 키 또는 source key를 사용한다.
- Worker는 Outbox 이벤트 ID와 외부 전달 기록으로 같은 부작용의 중복 실행을 막는다.
- 완료한 이벤트를 다시 처리해 알림, 이메일, Push와 파일 삭제가 중복돼서는 안 된다.
- 재시도 전에 외부 부작용이 이미 성공했는지 전달 기록과 현재 상태를 확인한다.

## Outbox 계약

- 이벤트 payload는 최소 식별자와 `schemaVersion`만 포함하며 업무 본문을 복제하지 않는다.
- 생산자와 소비자가 동시에 배포되지 않을 수 있으므로 기존 버전을 제자리에서 바꾸지 않는다.
- 호환되지 않는 변경은 새 버전을 추가하고 신·구 생산자·소비자의 배포 순서를 검증한다.
- 알 수 없는 버전과 불필요한 payload 필드는 영구 실패로 분류한다.
- 이벤트 처리 시 원본 리소스의 현재 상태를 다시 확인하고 이미 무효한 작업은 취소한다.

공유 이벤트의 정본은
[`packages/event-contracts`](../../packages/event-contracts/)다.

## Prisma와 migration

- schema와 migration SQL을 같은 변경에서 작성하고 `pnpm db:validate`와 생성 검사를 실행한다.
- 기존 데이터가 있는 필수 컬럼은 expand → backfill → 검증 → contract 순서로 변경한다.
- 앱의 신·구 버전이 공존하는 동안 양쪽이 읽을 수 있는 상태를 유지한다.
- backfill은 결정적이고 재실행 가능해야 하며 불명확한 충돌은 preflight에서 중단한다.
- 운영 migration 전 잠금 범위, 기존 행 처리, 실행 시간, 앱 호환성과 롤백 경계를 사람이 검토한다.
- 적용된 migration 파일을 수정하거나 checksum을 우회하지 않는다.
- 데이터베이스 down migration을 애플리케이션 롤백 수단으로 사용하지 않는다.

프로젝트 참여 관계와 팀 작업 연결은 `ProjectTeam`과 `project_team_id`를 단일 계약으로
사용한다. contract migration은 모든 팀 작업과 템플릿의 프로젝트 참여 팀 연결을 검증한 뒤에만
필수 제약을 적용하며, 불일치가 있으면 데이터를 추정해 덮어쓰지 않고 배포를 중단한다.

## 검증

데이터 변경은 최소한 다음을 확인한다.

- 정상 흐름과 현실적인 충돌·재시도
- 다른 워크스페이스와 비활성·보관 대상 거부
- 실패 뒤 부분 데이터 부재
- PostgreSQL unique·foreign key·lock 동작
- 같은 작업 또는 이벤트의 반복 실행
- migration 전후 행 수, 참조 누락과 계산 상태
