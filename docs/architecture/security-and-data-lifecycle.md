# 보안과 데이터 수명주기

> 상태: 현행
> 적용 범위: 인증, 권한, 워크스페이스 격리, 파일, 로그, 삭제와 보존
> 정본 책임: 여러 기능이 공유하는 보안·데이터 보호 불변식
> 갱신 조건: 인증 방식, 권한 모델, 데이터 노출 또는 보존 정책 변경

## 인증

- 비밀번호는 단방향 해시로 저장하고 원문을 기록하지 않는다.
- 이메일 확인을 마친 활성 사용자와 멤버십만 보호된 제품 기능을 사용한다.
- 인증 세션은 추측할 수 없는 토큰으로 식별하며 서버에는 검증 가능한 형태만 보관한다.
- 운영 HTTPS에서는 `__Host-` 접두사의 Secure·HttpOnly·SameSite=Lax 쿠키를 사용한다.
- 로컬 HTTP는 별도 비보안 이름을 사용하며 운영 쿠키와 섞지 않는다.
- 이메일 확인, 비밀번호 재설정과 초대 토큰은 목적·만료·사용·폐기 상태를 검증하고 재사용을 막는다.
- 인증 시도와 토큰 발급에는 정규화한 식별자의 HMAC 기반 rate limit key를 사용한다.

정확한 쿠키, 세션과 토큰 구현은
[`apps/api/src/modules/auth`](../../apps/api/src/modules/auth/)와
[`apps/api/src/common/auth`](../../apps/api/src/common/auth/)를 정본으로 한다.

## CSRF와 origin

- 쿠키 인증을 사용하는 상태 변경 요청은 CSRF 토큰과 허용 origin을 검증한다.
- 브라우저가 보낸 origin, 운영 `WEB_ORIGIN`과 프록시 구성을 함께 맞춘다.
- CORS나 프록시 완화로 CSRF 검사를 우회하지 않는다.
- 외부 callback이나 서명 webhook을 추가할 때는 일반 세션 요청과 분리된 인증 경계를 설계한다.

## 권한과 격리

- 세션에서 확정한 사용자, 워크스페이스와 멤버십을 모든 업무 쿼리의 기준으로 사용한다.
- 관리자, 일반 멤버와 팀장 권한은 역할 이름이 아니라 실제 작업별 정책으로 검증한다.
- 팀장 권한은 지정된 팀을 벗어나지 않는다.
- 다른 워크스페이스 ID를 알고 있어도 조회·수정·검색·알림·파일·내보내기에 접근할 수 없다.
- 권한 없는 리소스는 필요할 때 `404`로 수렴시켜 존재 여부를 노출하지 않는다.
- 화면의 버튼 숨김은 편의 처리이며 API 권한 검사를 대체하지 않는다.

## 입력과 출력

- DTO와 제품 규칙을 서버에서 다시 검증한다.
- Markdown은 문자열을 원본으로 저장하고 HTML 원문을 실행하지 않는다.
- 렌더링은 허용 목록 기반 sanitize를 거치며 `dangerouslySetInnerHTML`을 사용하지 않는다.
- CSV formula injection, 경로 조작, 잘못된 MIME·크기와 외부 workspace 참조를 거부한다.
- API 오류는 안정적인 코드와 안전한 메시지를 반환하고 내부 SQL·스택·공급자 응답을 노출하지 않는다.

## 파일

- 파일 메타데이터와 바이너리 경로를 분리한다.
- 저장 키는 서버가 생성하며 사용자 파일명을 경로로 사용하지 않는다.
- 파일 조회·다운로드는 사용자, 워크스페이스와 연결 리소스 권한을 다시 확인한다.
- 파일 저장소 루트를 Nginx 정적 경로로 노출하지 않는다.
- 연결된 파일, 프로젝트 로고와 사용자 아바타는 고아 파일 정리에서 제외한다.
- 메타데이터만 있고 바이너리가 없거나 반대인 상태를 자동으로 숨기지 않고 운영 오류로 기록한다.

## 삭제와 복원

- 이슈와 프로젝트는 즉시 물리 삭제하지 않고 휴지통으로 이동한다.
- 휴지통 이동 시 삭제 시각, 실행자와 30일 뒤 `purgeAt`을 기록하고 Outbox로 영구 삭제를 예약한다.
- 복원은 삭제 시각과 예약을 제거하고 삭제 이벤트를 취소한다.
- Worker는 실행 시 현재 삭제 상태와 `purgeAt`을 다시 검증한다.
- 프로젝트 영구 삭제는 연결된 이슈가 남아 있으면 재시도하고 관계 순서를 보존한다.
- 삭제로 연결이 끊긴 파일은 참조를 다시 확인한 뒤 별도 정리한다.

## 보존

- 세션, 일회성 토큰, 이메일 전달, rate limit, 내보내기 감사, 피드백과 Outbox는 Worker가 작은 batch로 정리한다.
- 사용자 피드백은 생성 시 보존 만료 시각을 기록한다.
- 만료되거나 폐기된 세션에 연결된 Push 구독은 endpoint와 키를 제거하고 비활성화한다.
- 보존 정리는 PostgreSQL advisory lock으로 한 Worker만 실행한다.
- 테이블별 기간과 쿼리의 정본은
  [`apps/worker/src/modules/maintenance`](../../apps/worker/src/modules/maintenance/)다.

## 로그와 분석

다음 값은 일반 로그, PostHog, Slack 경고와 이벤트 payload에 기록하지 않는다.

- 비밀번호, 세션·CSRF·초대·재설정 토큰과 비밀 키
- Cookie, Authorization 헤더와 Push endpoint·암호화 키
- 이메일 주소, IP, raw URL query와 사용자 입력 원문
- 이슈·댓글·전달·피드백 본문, 파일명과 파일 내용

로그에는 `requestId`, `jobId`, 내부 리소스 ID, 정제 오류 코드, 환경과 릴리스 ID처럼 운영에
필요한 최소 메타데이터만 남긴다.

## 필수 보안 회귀

- 다른 워크스페이스 접근 거부
- 일반 멤버와 팀장의 권한 경계
- CSRF·origin·쿠키 속성
- 토큰 만료·폐기·재사용과 rate limit
- 파일 경로 이탈과 연결 리소스 접근
- 민감 정보가 로그·분석·이벤트에 포함되지 않음
- 휴지통 복원과 영구 삭제의 경쟁 조건
