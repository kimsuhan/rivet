## 1. 문서 목적

이 문서는 Rivet MVP의 계정 인증, 서버 세션, 일회용 링크, CSRF 방어, `admin/member` 권한과 워크스페이스 데이터 격리를 구현 가능한 보안 기준으로 구체화한다. API와 웹 애플리케이션이 같은 보안 경계를 사용하고, 직접 URL·검색·파일·알림·CSV 같은 우회 경로에서도 다른 워크스페이스의 데이터가 노출되지 않게 하는 것이 핵심 목적이다.

보안 제어는 프론트 화면 노출 여부가 아니라 API 서버와 데이터베이스 조건에서 강제한다. 운영 인프라의 세부 실행 명령은 이 문서의 범위에 포함하지 않는다.

### 1.1 범위

- 표시 이름·이메일·비밀번호 회원가입과 이메일 인증
- 로그인, 로그아웃, 비밀번호 재설정과 서버 관리 세션
- 비밀번호 정책과 Argon2id 저장 기준
- 이메일 인증·초대·비밀번호 재설정 일회용 토큰
- 세션 쿠키, CSRF 토큰, Origin 검증과 CORS
- 인증·메일 요청 속도 제한과 계정 열거 방지
- 활성 멤버십, `admin/member` 권한과 리소스 소유자 권한
- 워크스페이스 격리, IDOR 방어와 파일 접근 제어
- 입력 검증, Markdown·CSV·파일 출력 안전성
- 비밀 값, 로그, 보안 헤더와 보안 검증 기준

### 1.2 제외 범위

- OAuth, 소셜 로그인, SSO와 외부 IdP
- MFA, 패스키와 복구 코드
- 사용자 이메일·표시 이름·비밀번호 변경 화면
- 계정 삭제, 다중 워크스페이스와 워크스페이스 전환
- 역할 변경, 사용자 정의 역할과 팀별 비공개 권한
- 활성 세션·기기 목록과 원격 개별 로그아웃
- CAPTCHA, 위험 기반 인증과 별도 봇 탐지 서비스
- WAF, IDS/IPS, DDoS 방어 서비스와 침투 테스트 절차

MVP는 이메일과 비밀번호의 단일 요인 인증이다. Public 이후 외부 사용자 범위와 계정 가치가 커지기 전에 MFA 또는 패스키 도입을 다시 검토한다.

### 1.3 공식 기준

이 문서는 다음 공식 기준의 MVP 적용 범위를 참고한다.

- [NIST SP 800-63B](https://pages.nist.gov/800-63-4/sp800-63b.html): 비밀번호 길이, 취약 비밀번호 차단, 속도 제한과 세션 기준
- [OWASP Password Storage Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Password_Storage_Cheat_Sheet.html): Argon2id 저장 기준
- [OWASP Authentication Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Authentication_Cheat_Sheet.html): 인증 응답, 로그인 제한과 계정 열거 방지
- [OWASP Session Management Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Session_Management_Cheat_Sheet.html): 세션 ID와 쿠키 속성
- [OWASP CSRF Prevention Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Cross-Site_Request_Forgery_Prevention_Cheat_Sheet.html): CSRF 토큰, 사용자 정의 헤더와 Origin 검증

Rivet은 정부 시스템의 인증 보증 등급을 선언하지 않는다. 위 문서에서 제품 규모와 단일 요인 인증에 적용할 수 있는 구현 원칙만 사용한다.

## 2. 보안 목표와 위협 범위

### 2.1 우선 보호 대상

1. 비밀번호, 세션 식별자와 일회용 토큰
2. 이메일과 사용자 프로필
3. 이슈 제목·설명·댓글·작업 전달과 첨부파일
4. 워크스페이스 멤버십과 관리자 권한
5. CSV 내보내기와 휴지통 데이터
6. Resend, 데이터베이스, PostHog와 토큰 HMAC 비밀 키

### 2.2 MVP에서 방어할 위협

| 위협 | 기본 방어 |
| --- | --- |
| 비밀번호 추측·크리덴셜 스터핑 | 긴 비밀번호, 취약 비밀번호 차단, Argon2id, 계정·네트워크 속도 제한 |
| 계정 존재 여부 수집 | 일반화된 인증·복구 응답, 비슷한 처리 시간, 속도 제한 |
| 세션 탈취·고정 | 고엔트로피 불투명 세션, `Secure`·`HttpOnly`, 로그인 시 새 세션 발급, 서버 폐기 |
| CSRF·로그인 CSRF | 같은 출처 배포, `SameSite=Lax`, CSRF 헤더, Origin·Referer 검증 |
| 다른 워크스페이스 ID 직접 요청 | 활성 멤버십 컨텍스트와 모든 쿼리의 `workspace_id` 조건, 일관된 404 |
| 권한 상승·대량 할당 | 명시적 DTO 허용 필드, 서버 권한 가드, 데이터베이스 제약 |
| XSS·위험한 링크 | Markdown 허용 목록 렌더링, CSP, 위험한 URL·HTML 제거 |
| 악성·위장 파일 | 실제 크기·형식 검사, UUID 경로, 이미지 외 강제 다운로드, `nosniff` |
| CSV 수식 주입 | 위험한 시작 문자의 셀 이스케이프 |
| 토큰·개인정보 로그 유출 | URL 조각 토큰, 요청 본문 미기록, 정제 로그와 비밀 값 마스킹 |

### 2.3 기본 원칙

- 인증과 권한은 모든 요청에서 서버가 다시 확인한다.
- 거부가 기본값이며 명시적으로 허용된 동작만 실행한다.
- 리소스 ID만으로 조회하지 않고 현재 워크스페이스 경계를 함께 조건으로 사용한다.
- 인증 비밀은 원문으로 저장하거나 로그에 남기지 않는다.
- 브라우저 저장소에 세션·일회용 토큰을 보존하지 않는다.
- 보안 오류는 내부 진단 정보와 사용자 응답을 분리한다.
- 암호 알고리즘을 직접 구현하지 않고 검증된 런타임·라이브러리 기능을 사용한다.
- MVP에서 실제 위협을 줄이지 않는 공급자, 인증 계층과 추상화는 추가하지 않는다.

## 3. 인증 상태와 요청 처리 순서

### 3.1 인증 상태

| 상태 | 허용 동작 |
| --- | --- |
| 비로그인 | 가입, 이메일 인증, 로그인, 비밀번호 재설정 요청·확정, 초대 미리보기 |
| 이메일 미인증 | 인증 링크 사용과 제한된 인증 메일 재발송만 허용 |
| 인증됨·멤버십 없음 | 워크스페이스 한 번 생성 또는 유효한 초대 수락 |
| 인증됨·활성 멤버십 | 역할에 맞는 업무 API 사용 |
| 인증됨·비활성 멤버십 | 업무 API와 로그인 거부, 기존 세션은 폐기 |

회원가입만으로 업무 세션을 만들지 않는다. 이메일 인증을 마친 사용자가 로그인에 성공했을 때 서버 세션을 발급한다.

### 3.2 업무 요청 처리 순서

1. Nginx가 HTTPS만 허용하고 신뢰할 프록시 헤더를 다시 설정한다.
2. API가 요청 형식, 본문 크기와 허용 Content-Type을 확인한다.
3. 세션 쿠키 해시로 활성·미만료 세션을 찾는다.
4. 사용자의 이메일 인증 완료 여부를 확인한다.
5. 단일 활성 워크스페이스 멤버십을 조회해 요청 컨텍스트에 넣는다.
6. 상태 변경 요청은 CSRF 토큰과 출처를 검증한다.
7. 엔드포인트 역할과 리소스 소유자 조건을 확인한다.
8. 저장소 쿼리에 `workspace_id`와 삭제·보관 조건을 함께 적용한다.
9. 요청이 참조하는 다른 리소스도 같은 워크스페이스인지 검증한다.
10. 성공 또는 정제된 오류를 `requestId`와 함께 반환한다.

인증 실패, CSRF 실패와 권한 실패 요청은 도메인 서비스를 실행하지 않는다.

## 4. 이메일과 계정 식별

### 4.1 이메일 입력

- 앞뒤 공백을 제거하고 일반적인 이메일 구문과 최대 254자를 검증한다.
- 로그인·유일성 비교에는 소문자 정규화 값을 사용한다.
- Gmail 점·플러스 주소처럼 공급자별 의미를 추측해 변환하지 않는다.
- 표시·메일 발송에는 사용자가 입력한 원본 이메일을 사용한다.
- 인증이 완료되기 전에는 업무 데이터에 접근할 수 없다.
- 이메일 변경은 MVP에 포함하지 않는다.

### 4.2 표시 이름

- 앞뒤 공백을 제거한 1~50자의 문자열만 허용한다.
- 중복을 허용하며 로그인 식별자로 사용하지 않는다.
- 제어 문자와 화면 방향을 속이는 위험한 문자는 거부한다.
- 화면 출력은 텍스트로 이스케이프하고 HTML로 해석하지 않는다.

### 4.3 계정 열거 방지

- 로그인 실패는 계정 없음, 비밀번호 오류를 구분하지 않고 `INVALID_CREDENTIALS`를 반환한다.
- 이메일 미인증·비활성 상태는 올바른 비밀번호 검증이 끝난 뒤에만 해당 사용자에게 안내한다.
- 비밀번호 재설정 요청은 이메일 존재 여부와 관계없이 같은 `204` 응답을 반환한다.
- 회원가입은 이메일 존재 여부와 관계없이 같은 `202`와 접수 본문을 반환하고 중복 이메일 전용 오류 코드를 외부에 제공하지 않는다.
- 인증 메일 재발송도 대상 계정의 존재·인증 상태와 관계없이 같은 성공 응답을 반환한다.
- 계정이 없는 로그인도 고정된 더미 Argon2id 해시를 검증해 실패 처리 시간 차이를 줄인다.
- 완전히 같은 응답 시간을 만들기 위한 인위적 고정 지연은 두지 않고 속도 제한과 비슷한 코드 경로를 함께 사용한다.

## 5. 비밀번호 정책

### 5.1 입력 규칙

MVP 권장안은 다음과 같다.

| 항목 | 기준 |
| --- | --- |
| 최소 길이 | 12개 Unicode 코드 포인트 |
| 최대 길이 | 128개 Unicode 코드 포인트 |
| 허용 문자 | 공백을 포함한 출력 가능한 ASCII와 Unicode |
| 정규화 | 해시 전 Unicode NFC |
| 조합 규칙 | 대문자·소문자·숫자·특수문자 조합 강제 없음 |
| 취약 비밀번호 | 공통·유출 비밀번호와 `rivet`, 이메일 로컬부 등 문맥 단어 차단 |
| 주기 변경 | 없음. 유출·침해가 확인될 때만 재설정 요구 |

- Unicode NFC 정규화 후 길이를 계산하고 같은 값을 해시·검증한다.
- 최소 12자는 MVP 사용성을 고려해 NIST의 단일 요인 15자 권장보다 완화한 제품 결정이다. Public 전에는 MFA·패스키 도입 또는 최소 15자 상향을 다시 검토한다.
- 비밀번호 앞뒤 공백을 임의로 제거하거나 대소문자를 바꾸지 않는다.
- 최대 길이를 넘는 입력은 해시 연산 전에 거부해 자원 고갈을 막는다.
- 비밀번호 확인 값은 프론트 입력 검사용이며 API에는 한 번만 전송한다.
- 붙여넣기, 브라우저 자동 완성과 비밀번호 관리자를 허용한다.
- 비밀번호 표시·숨김 전환을 제공하되 기본값은 숨김이다.
- 보안 질문, 힌트와 부분 문자 입력은 사용하지 않는다.

### 5.2 취약 비밀번호 차단

- 저장소에 포함한 로컬 차단 목록과 제품 문맥 단어를 가입·재설정 시 검사한다.
- 외부 서비스로 비밀번호 원문이나 전체 해시를 보내지 않는다.
- 차단 목록은 전체 문자열 일치에 사용하며 포함된 일반 단어만으로 긴 문구를 거부하지 않는다.
- 차단된 경우 `PASSWORD_TOO_COMMON` 필드 오류와 긴 문구 사용 안내를 반환한다.
- 목록 갱신은 의존성 업데이트와 같은 검토 절차를 거치며 런타임 외부 호출에 의존하지 않는다.

### 5.3 비밀번호 저장

- 해시는 Argon2id를 사용한다.
- 최소 기준은 메모리 19MiB, 반복 2회, 병렬도 1이다.
- 운영 서버에서 로그인 해시 시간이 정상 부하 기준 약 250~500ms 안에 들도록 메모리·반복 횟수를 상향 조정할 수 있다.
- 알고리즘, 버전, salt와 비용 파라미터는 Argon2 인코딩 문자열에 함께 저장한다.
- 각 비밀번호의 salt는 라이브러리가 안전한 난수로 생성한다.
- 별도 pepper는 키 회전과 전체 비밀번호 재설정 부담에 비해 MVP 이득이 제한적이므로 사용하지 않는다.
- 로그인 성공 시 저장 파라미터가 현재 기준보다 낮으면 같은 요청 안에서 최신 파라미터로 다시 해시한다.
- 비교는 Argon2 라이브러리의 검증 함수를 사용하고 직접 비교 루프를 구현하지 않는다.

## 6. 회원가입과 이메일 인증

### 6.1 회원가입

1. 표시 이름, 이메일과 비밀번호를 검증한다.
2. 네트워크와 이메일 대상 속도 제한을 확인한다.
3. 정규화 이메일로 기존 계정을 확인한다.
4. 신규 이메일이면 비밀번호 차단 목록과 Argon2id 해시를 확인하고 사용자, 24시간 이메일 인증 토큰과 Outbox를 트랜잭션으로 저장한다.
5. 기존 이메일이면 비밀번호를 변경하지 않고 더미 Argon2id 계산을 수행한다. 미인증 계정은 제한 안에서 유효한 인증 링크를 다시 발송할 수 있고, 인증된 계정은 변경하지 않는다.
6. 계정 존재 여부와 관계없이 같은 `202`와 가린 입력 이메일을 반환한다.
7. 클라이언트는 신규 계정이면 인증 메일을 확인하고 기존 계정이면 로그인 또는 비밀번호 재설정을 사용하라는 공통 안내로 이동한다.

사용자 행이 만들어진 뒤 메일 발송이 지연·실패해도 계정을 되돌리지 않는다. 명시적인 재발송은 같은 대상의 이전 미사용 인증 토큰을 폐기하고 새 토큰을 발급한다.

### 6.2 이메일 인증

- 이메일 링크는 `https://{WEB_ORIGIN}/verify-email#token={token}` 형식을 사용한다.
- URL 조각은 HTTP 요청과 일반 Referer에 전송되지 않으며 웹 애플리케이션만 읽는다.
- 화면은 토큰을 메모리로 읽은 즉시 주소에서 제거한 뒤 API 요청 본문으로 전송한다.
- 인증 페이지는 토큰 처리 전 제3자 스크립트와 분석 이벤트를 실행하지 않는다.
- 유효한 토큰 사용과 `email_verified_at` 기록은 하나의 트랜잭션에서 처리한다.
- 이미 인증된 계정의 재사용 링크는 새 변경 없이 로그인 안내를 반환할 수 있다.
- 이메일 인증 뒤 자동 로그인하지 않고 사용자가 비밀번호로 로그인하게 한다.

## 7. 로그인과 비밀번호 재설정

### 7.1 로그인

1. JSON 요청, Origin과 속도 제한을 확인한다.
2. 정규화 이메일로 사용자를 찾고 없으면 더미 해시를 사용한다.
3. Argon2id로 전체 비밀번호를 검증한다.
4. 실패하면 계정 존재 여부와 무관하게 같은 오류와 상태 코드를 반환한다.
5. 성공한 경우 이메일 인증과 활성 멤버십 상태를 확인한다.
6. 기존 브라우저 쿠키 값은 재사용하지 않고 새 서버 세션과 쿠키를 발급한다.
7. 계정 단위 실패 카운터를 초기화한다.

로그인 성공 시 세션 ID, CSRF 토큰, 사용자·멤버십·워크스페이스와 온보딩 상태를 API 명세대로 반환한다.

### 7.2 비밀번호 재설정 요청

- 요청 이메일의 존재 여부와 관계없이 같은 상태 코드와 빈 응답을 반환한다.
- 존재하고 인증된 계정에만 30분 토큰과 Outbox를 만든다.
- 같은 대상의 이전 미사용 재설정 토큰은 폐기한다.
- 속도 제한된 요청은 메일을 만들지 않으며 `Retry-After`를 포함한 `429`를 반환한다.
- 응답 시간 차이를 줄이되 메일 발송은 항상 워커에서 비동기로 처리한다.

### 7.3 비밀번호 재설정 확정

1. 토큰의 MAC, 목적, 대상, 미사용·미폐기·만료 상태를 확인한다.
2. 새 비밀번호 정책과 차단 목록을 확인한다.
3. 새 Argon2id 해시, 토큰 사용, 같은 목적의 다른 토큰 폐기와 전체 세션 폐기를 한 트랜잭션에서 처리한다.
4. 세션 쿠키가 있으면 만료 값으로 덮어쓴다.
5. 자동 로그인하지 않고 로그인 화면으로 이동한다.

현재 비밀번호를 입력하는 비밀번호 변경 기능은 MVP 화면에 포함하지 않는다.

## 8. 세션 관리

### 8.1 세션 식별자

- Node.js 암호학적 난수 생성기로 32바이트를 만들고 base64url로 인코딩한다.
- 쿠키에는 의미 없는 원문 식별자만 넣고 사용자 ID, 역할과 만료 시각을 넣지 않는다.
- 데이터베이스에는 `SHA-256(sessionToken)`만 저장한다.
- 원문 식별자는 로그, 오류, 분석 이벤트와 브라우저 저장소에 기록하지 않는다.
- 요청마다 쿠키 해시와 활성·미만료 세션을 함께 조회한다.

### 8.2 세션 쿠키

운영 쿠키는 다음 계약을 사용한다.

```text
Set-Cookie: __Host-rivet_session=<opaque>; Path=/; Secure; HttpOnly; SameSite=Lax
```

| 속성 | 기준 |
| --- | --- |
| 이름 | `__Host-rivet_session` |
| `Secure` | 운영·개발 HTTPS에서 필수 |
| `HttpOnly` | 필수 |
| `SameSite` | `Lax` |
| `Path` | `/` |
| `Domain` | 설정하지 않음 |
| 만료 | 세션의 절대 만료와 같거나 더 짧게 설정 |

- 로컬 HTTP 개발은 이름을 `rivet_session`으로 바꾸고 `Secure`만 끌 수 있다. 운영 설정에서 이 예외를 허용하지 않는다.
- 세션 쿠키는 Next.js 클라이언트 JavaScript에서 읽지 않는다.
- 브라우저 `localStorage`, `sessionStorage`와 IndexedDB에 세션 식별자를 복제하지 않는다.

### 8.3 세션 수명

| 기준 | 값 |
| --- | ---: |
| 유휴 만료 | 마지막 사용 후 7일 |
| 절대 만료 | 생성 후 30일 |
| `lastSeenAt` 쓰기 간격 | 최대 5분에 한 번 |
| 만료·폐기 세션 보존 | 종료 후 30일 |

- 서버 시각을 기준으로 만료를 판단하고 쿠키 만료만 신뢰하지 않는다.
- 유효 요청이 있어도 절대 만료 시각은 연장하지 않는다.
- 매 요청의 데이터베이스 쓰기를 피하기 위해 `lastSeenAt`은 5분 이상 지났을 때만 갱신한다.
- 만료·폐기 세션은 인증에 사용할 수 없고 정기 보존 작업에서 제거한다.

### 8.4 세션 발급과 폐기

- 로그인마다 새 세션을 발급해 기존 쿠키를 세션으로 승격하지 않는다.
- 로그아웃은 현재 세션의 `revokedAt`을 기록하고 쿠키를 즉시 만료시킨다.
- 비밀번호 재설정은 현재 세션을 포함한 사용자의 모든 세션을 폐기한다.
- 멤버 비활성화는 같은 트랜잭션에서 대상 사용자의 모든 세션을 폐기한다.
- 세션이 만료·폐기됐거나 연결 사용자가 없으면 `401 SESSION_REQUIRED`다.
- MFA와 권한 승격이 없으므로 별도 단계 상승 세션은 만들지 않는다.

## 9. 일회용 토큰

### 9.1 토큰 형식

일회용 링크 토큰은 무작위 UUID인 `tokenId`와 서버 비밀 키의 HMAC을 결합한다.

```text
base64url(tokenIdBytes).base64url(HMAC-SHA256(key, purpose + ":" + tokenId))
```

- `tokenId`는 애플리케이션이 생성한 UUID다.
- `ONE_TIME_TOKEN_HMAC_KEY`는 최소 32바이트 무작위 비밀 값이다.
- 데이터베이스에는 전체 토큰의 SHA-256 해시만 저장한다.
- 워커는 Outbox의 토큰 ID와 목적, 같은 HMAC 키로 발송 링크를 다시 만들 수 있다.
- 검증은 목적별 API에서만 수행하고 MAC은 상수 시간 비교를 사용한다.
- 토큰 문자열을 파싱해도 대상 사용자·이메일·워크스페이스 정보가 드러나지 않는다.

### 9.2 수명과 일회 사용

| 목적 | 유효 기간 |
| --- | ---: |
| 이메일 인증 | 24시간 |
| 워크스페이스 초대 | 7일 |
| 비밀번호 재설정 | 30분 |

- 같은 목적과 대상에 새 토큰을 발급하면 이전 미사용 토큰을 폐기한다.
- 사용 시 대상 행을 잠그고 해시, 목적, 대상, 만료, 사용·폐기 상태를 확인한다.
- 업무 변경과 `usedAt` 기록은 한 트랜잭션에서 처리한다.
- 사용·만료·폐기 토큰은 종료 후 30일 보존한 뒤 제거한다.
- HMAC 키가 유출되거나 교체되면 기존 미사용 링크를 모두 폐기하고 필요한 링크를 재발급한다.

### 9.3 토큰 노출 방지

- 이메일 링크의 토큰은 쿼리 문자열이 아니라 URL 조각에 넣는다.
- 토큰 페이지는 `Cache-Control: no-store`와 `Referrer-Policy: no-referrer`를 사용한다.
- 토큰을 읽은 뒤 `history.replaceState`로 주소에서 즉시 제거한다.
- 토큰 원문, 토큰이 포함된 URL, API 요청 본문과 이메일 HTML을 애플리케이션 로그에 남기지 않는다.
- PostHog 자동 수집과 세션 녹화를 사용하지 않으며 토큰 페이지의 URL을 분석 이벤트에 전송하지 않는다.

## 10. CSRF, 출처와 CORS

### 10.1 배포 출처

- 운영 웹은 Nginx가 같은 출처의 `/api/v1`과 `/files` 요청을 API 서버로 전달한다.
- 운영 API는 브라우저용 광범위 CORS를 활성화하지 않는다.
- 로컬·개발에서 출처가 다르면 환경변수의 정확한 허용 출처 목록만 사용한다.
- 자격 증명 요청에 `Access-Control-Allow-Origin: *`를 사용하지 않는다.
- 허용 출처 값은 스킴, 호스트와 포트까지 정확히 비교하고 접미사 일치를 사용하지 않는다.

### 10.2 CSRF 토큰

- 인증된 `POST`, `PUT`, `PATCH`, `DELETE`는 `X-CSRF-Token` 헤더가 필수다.
- 토큰은 `base64url(HMAC-SHA256(CSRF_HMAC_KEY, sessionToken))`으로 세션 원문에 결합한다.
- `GET /auth/session`과 로그인 성공 응답이 현재 토큰을 제공한다.
- 데이터베이스에 CSRF 토큰 컬럼을 추가하지 않는다.
- 서버는 요청 쿠키로 토큰을 다시 계산하고 상수 시간으로 비교한다.
- 세션이 바뀌거나 폐기되면 이전 CSRF 토큰도 사용할 수 없다.
- `GET`, `HEAD`, `OPTIONS`는 상태를 변경하지 않으며 CSRF 헤더를 요구하지 않는다.

### 10.3 Origin 검증

- 모든 상태 변경 요청에서 `Origin`이 현재 웹 출처와 정확히 같은지 확인한다.
- `Origin`이 없는 브라우저 요청은 `Referer`의 origin을 보조 값으로 확인한다.
- 둘 다 없거나 허용 출처와 다르면 `403 CSRF_INVALID`다.
- 프록시의 대상 origin은 사용자 입력 헤더가 아니라 환경 설정의 외부 웹 origin을 기준으로 판단한다.
- 가입·로그인·재설정처럼 세션 전 요청도 JSON Content-Type과 Origin 검증을 적용해 로그인 CSRF를 막는다.
- 파일 업로드는 multipart 요청이므로 인증 세션, CSRF 헤더와 Origin을 모두 확인한다.

`SameSite=Lax`와 Origin 검증은 방어 계층이며 CSRF 토큰 검증을 대체하지 않는다.

## 11. 인증·메일 요청 속도 제한

### 11.1 저장 방식

MVP 권장안은 기존 PostgreSQL에 만료 가능한 작은 속도 제한 버킷을 저장하는 것이다. Redis와 별도 서비스를 추가하지 않는다.

`auth_rate_limit_buckets`는 다음 최소 정보만 가진다.

| 필드 | 용도 |
| --- | --- |
| `scope`, `key_hash`, `window_started_at` | 버킷 복합 식별자 |
| `attempt_count` | 현재 창의 요청·실패 횟수 |
| `blocked_until` | 제한 해제 시각 |
| `expires_at` | 정리 대상 시각 |

- IP, 정규화 이메일과 토큰 ID 원문은 저장하지 않고 `HMAC-SHA256(RATE_LIMIT_HMAC_KEY, scope + ":" + value)`를 키로 사용한다.
- 증가와 제한 판정은 원자적 upsert 또는 행 잠금으로 처리한다.
- 성공한 로그인은 계정 실패 버킷을 제거한다.
- 만료 버킷은 워커가 매일 제거한다.
- HMAC 키는 다른 토큰·CSRF 키와 분리한다.
- 제한 저장소 오류 시 로그인·메일 발송 엔드포인트는 안전하게 실패하고 일반 업무 API까지 중단하지 않는다.

이 모델을 확정하면 도메인 및 데이터 모델 명세서에 운영 보안 테이블과 보존 정책을 함께 반영한다.

### 11.2 기본 제한 수치

| 요청 | 대상 제한 | 네트워크 제한 |
| --- | --- | --- |
| 로그인 실패 | 계정별 15분에 5회 | IP별 15분에 60회 |
| 회원가입 | 이메일별 24시간에 3회 | IP별 1시간에 10회 |
| 인증 메일 재발송 | 이메일별 1시간에 3회 | IP별 1시간에 10회 |
| 비밀번호 재설정 요청 | 이메일별 1시간에 3회 | IP별 1시간에 10회 |
| 토큰 검증 실패 | 토큰 ID별 15분에 5회 | IP별 15분에 30회 |
| 초대 미리보기 실패 | 토큰 ID별 15분에 5회 | IP별 15분에 30회 |

- 하나라도 제한을 넘으면 `429 RATE_LIMITED`와 초 단위 `Retry-After`를 반환한다.
- 영구 계정 잠금은 공격자가 다른 사용자를 차단하는 수단이 될 수 있어 사용하지 않는다.
- 성공 로그인만 계정 실패 횟수를 초기화하며 새 토큰 발급은 검증 실패 횟수를 우회하지 못한다.
- 관리자 초대 발송은 워크스페이스당 24시간 100개 이메일을 상한으로 둔다.
- 실제 비공개 베타 사용량과 공격 로그를 근거로 수치를 낮추거나 높일 수 있으며 변경 이력을 남긴다.

### 11.3 네트워크 주소 신뢰

- Nginx만 API 서버에 직접 연결할 수 있게 방화벽과 바인딩을 제한한다.
- Nginx는 외부의 `X-Forwarded-For`를 그대로 신뢰하지 않고 실제 접속 주소를 기준으로 프록시 헤더를 다시 설정한다.
- API는 설정된 프록시 홉 하나만 신뢰해 클라이언트 IP를 계산한다.
- 주소 전체를 애플리케이션 로그와 속도 제한 테이블에 원문으로 저장하지 않는다.

## 12. 권한과 워크스페이스 격리

### 12.1 역할

| 동작 | Admin | Member |
| --- | :---: | :---: |
| 워크스페이스 이름·슬러그 수정 | 가능 | 불가 |
| 멤버 초대·재발송·취소·비활성화 | 가능 | 불가 |
| 팀·팀 멤버·워크플로 관리 | 가능 | 불가 |
| 라벨 관리 | 가능 | 불가 |
| 프로젝트·이슈·관계·작업 전달 | 가능 | 가능 |
| 댓글 작성 | 가능 | 가능 |
| 자신의 댓글 수정·삭제 | 가능 | 가능 |
| 다른 사용자의 댓글 수정·삭제 | 불가 | 불가 |
| 자신의 프로필 사진 | 가능 | 가능 |
| 이슈 파일 첨부·조회 | 가능 | 가능 |
| 자신의 알림 | 가능 | 가능 |
| 휴지통 조회·복구 | 가능 | 불가 |
| CSV 내보내기 | 가능 | 불가 |

- 초대 수락 멤버는 항상 `MEMBER`다.
- 역할 변경 API는 MVP에 없으며 회원가입으로 워크스페이스를 만든 사용자만 `ADMIN`이 된다.
- 관리자는 자기 자신을 비활성화할 수 없다.
- 프론트에서 숨긴 메뉴와 버튼은 권한 검사를 대신하지 않는다.

### 12.2 워크스페이스 컨텍스트

- 활성 세션의 사용자 ID로 최대 한 개의 활성 멤버십을 조회한다.
- 업무 서비스는 호출 시 명시적 `workspaceId`와 `membershipId` 컨텍스트를 받는다.
- 컨트롤러 요청 본문의 `workspaceId`를 신뢰하지 않는다.
- 모든 워크스페이스 업무 쿼리는 기본 키와 `workspace_id`를 함께 조건으로 사용한다.
- 새 관계를 만들 때 양쪽 리소스와 멤버십의 워크스페이스를 모두 검증한다.
- 직접 표시 ID 조회, 검색, SSE, 알림, 파일, 휴지통과 CSV에도 같은 경계를 적용한다.
- 다른 워크스페이스, 삭제됨과 실제로 없음은 외부에서 모두 `404 RESOURCE_NOT_FOUND`다.

Prisma 저장소 함수는 워크스페이스 업무 리소스 조회에 `workspaceId` 없는 공개 메서드를 만들지 않는다. 내부 ID만으로 조회해야 하는 워커 작업은 이벤트의 워크스페이스와 원본 행을 함께 검증한다.

### 12.3 멤버 비활성화

1. 요청자 관리자 권한과 대상 활성 상태를 확인한다.
2. 대상이 요청자 본인이 아닌지 확인한다.
3. 대상의 미완료 팀 작업을 행 잠금 후 다시 조회한다.
4. 한 건이라도 있으면 변경하지 않고 접근 가능한 작업 요약을 반환한다.
5. 없으면 멤버십 비활성화와 모든 로그인 세션 폐기를 한 트랜잭션에서 처리한다.
6. 과거 이슈·댓글·활동의 작성자 참조는 유지한다.
7. 처리 완료 이벤트는 업무 본문 없이 기록한다.

## 13. 리소스별 추가 권한

### 13.1 파일

| 상태 | 허용 사용자 |
| --- | --- |
| 미연결 파일 | 업로더 본인 |
| 이슈 일반 첨부·본문 이미지 | 대상 이슈에 접근 가능한 같은 워크스페이스의 활성 멤버 |
| 프로필 사진 | 같은 활성 워크스페이스에서 사용자 표시가 필요한 인증 사용자 |
| 휴지통 이슈 파일 | 관리자 휴지통·복구 화면과 보존 관계 안의 접근자 |

- 파일 ID만으로 로컬 경로를 계산하지 않고 메타데이터 접근 권한을 먼저 확인한다.
- 업로드 루트는 Nginx 정적 경로로 노출하지 않는다.
- 다른 사용자의 미연결 파일과 다른 워크스페이스 파일은 `404`로 처리한다.
- 원본 파일명은 다운로드 표시용이며 헤더 삽입 문자를 제거하고 RFC 호환 방식으로 인코딩한다.

### 13.2 댓글과 작업 전달

- 댓글 수정·삭제는 `authorMembershipId`가 현재 멤버십과 같을 때만 가능하다.
- 관리자는 다른 사용자의 댓글을 편집하거나 삭제할 수 없다.
- 작업 전달은 활성 멤버가 작성할 수 있지만 대상이 백엔드 프로젝트 역할 팀 작업인지 서버가 확인한다.
- 최초·추가 전달은 수정·삭제할 수 없다.

### 13.3 알림과 CSV

- 알림은 `recipientMembershipId`가 현재 멤버십과 같은 행만 조회·변경한다.
- SSE는 연결 시 세션·활성 멤버십을 확인하고 해당 워크스페이스 신호만 전달한다.
- CSV는 관리자 권한을 요청 시작과 스트리밍 직전에 확인한다.
- CSV 쿼리도 현재 워크스페이스를 고정하며 클라이언트가 범위를 지정하지 않는다.
- 다운로드 응답은 `Cache-Control: private, no-store`를 사용한다.

## 14. 입력·출력과 브라우저 보안

### 14.1 요청 DTO와 데이터베이스

- 전역 DTO 검증은 알려지지 않은 필드를 제거하지 않고 오류로 거부한다.
- 숫자, enum과 boolean의 암시적 변환을 사용하지 않는다.
- 컨트롤러는 허용된 필드만 도메인 서비스로 전달한다.
- Prisma의 매개변수화된 쿼리를 기본으로 사용한다.
- 원시 SQL은 Prisma의 안전한 태그·바인딩을 사용하고 문자열 보간을 금지한다.
- 정렬 필드와 방향은 서버 허용 목록에서 매핑하며 요청 값을 SQL 식별자로 직접 넣지 않는다.
- JSON, multipart와 CSV 요청·응답 크기 제한을 엔드포인트별로 둔다.

### 14.2 Markdown과 링크

- 서버 저장 정본은 Markdown이며 렌더링 HTML을 저장하지 않는다.
- 렌더러는 허용된 Markdown 노드만 HTML로 바꾸고 원시 HTML은 제거한다.
- 링크는 `http`, `https`와 제품 내부 파일·멘션 형식만 허용한다.
- `javascript:`, `data:` 링크, 이벤트 속성, iframe, object와 임의 embed를 제거한다.
- 외부 링크는 새 창을 사용할 때 `rel="noopener noreferrer"`를 적용한다.
- 외부 이미지 URL은 렌더링·프록시하지 않고 인증된 `/files/{fileId}`만 본문 이미지로 허용한다.

### 14.3 파일과 CSV

- 파일은 스트리밍하며 26,214,400바이트를 넘는 즉시 중단한다.
- 확장자와 브라우저 MIME이 아니라 서버가 감지한 파일 시그니처를 기준으로 인라인 여부를 결정한다.
- JPEG·PNG·WebP·GIF만 인라인으로 제공하고 SVG를 포함한 나머지는 강제 다운로드한다.
- 모든 파일 응답에 `X-Content-Type-Options: nosniff`를 적용한다.
- 저장 키는 UUID 상대 경로이며 `..`, 절대 경로와 사용자 파일명을 포함하지 않는다.
- CSV 사용자 입력 셀이 `=`, `+`, `-`, `@`, 탭 또는 캐리지 리턴으로 시작하면 앞에 작은따옴표를 붙인다.

### 14.4 보안 응답 헤더

| 헤더 | 운영 기준 |
| --- | --- |
| `Content-Security-Policy` | 기본 출처 `self`, `object-src 'none'`, `base-uri 'self'`, `frame-ancestors 'none'`; 필요한 PostHog·이미지 미리보기 출처만 명시 |
| `Strict-Transport-Security` | TLS 안정화 후 `max-age=31536000`; 모든 하위 도메인 확인 전 `includeSubDomains`·`preload` 제외 |
| `X-Content-Type-Options` | `nosniff` |
| `Referrer-Policy` | 기본 `strict-origin-when-cross-origin`, 토큰 화면 `no-referrer` |
| `Permissions-Policy` | 사용하지 않는 카메라·마이크·위치 권한 비활성화 |
| `X-Frame-Options` | `DENY`, CSP `frame-ancestors` 보조 |

CSP는 배포 환경의 실제 Next.js, PostHog 연결과 Lexical 이미지 미리보기를 브라우저에서 검증한 뒤 허용 출처를 늘린다. 임시 해결을 위해 전체 `unsafe-eval`, `*` 또는 모든 data 스킴을 허용하지 않는다.

## 15. 전송 계층과 프록시

- 운영 HTTP 요청은 HTTPS로 영구 리다이렉트한다.
- API와 웹의 외부 통신은 TLS만 사용하며 인증 이후 HTTP로 내려가지 않는다.
- API·워커의 내부 PM2 포트는 공용 네트워크에 직접 노출하지 않는다.
- Nginx가 요청 본문 크기, 연결 제한과 SSE 버퍼링 정책을 담당하고 API가 도메인 제한을 다시 검증한다.
- `Host`, `X-Forwarded-Host`, `X-Forwarded-Proto`를 사용자 입력으로 신뢰하지 않고 Nginx와 외부 origin 설정을 기준으로 링크와 출처를 만든다.
- 이메일 링크는 요청 Host가 아니라 설정된 `WEB_ORIGIN`으로 생성한다.
- 운영에서 TLS와 `Secure` 쿠키를 끄는 설정은 애플리케이션 시작 단계에서 거부한다.

## 16. 비밀 값과 키 운영

### 16.1 비밀 값

| 비밀 | 용도 |
| --- | --- |
| `DATABASE_URL` | PostgreSQL 연결 |
| `RESEND_API_KEY` | 이메일 발송 |
| `ONE_TIME_TOKEN_HMAC_KEY` | 이메일 인증·초대·재설정 링크 MAC |
| `CSRF_HMAC_KEY` | 세션 결합 CSRF 토큰 |
| `RATE_LIMIT_HMAC_KEY` | 이메일·IP·토큰 속도 제한 키 비식별화 |
| `POSTHOG_API_KEY` | 운영 제품 이벤트·정제 오류 전송 |

- HMAC 키는 각각 최소 32바이트 무작위 값이며 서로 재사용하지 않는다.
- 비밀 값은 저장소, 예시 환경 파일의 실제 값, 이미지, 로그와 오류 추적에 넣지 않는다.
- API와 워커는 필요한 비밀만 주입받는다. 웹 브라우저 번들에는 서버 비밀이 포함되지 않는다.
- 키 비교와 MAC 검증은 런타임의 상수 시간 함수를 사용한다.
- 비밀 교체 절차와 담당자는 배포·운영 명세서에 기록한다.

### 16.2 키 교체 영향

| 키 | 교체 영향 |
| --- | --- |
| 일회용 토큰 HMAC | 기존 미사용 이메일 링크 폐기·재발급 |
| CSRF HMAC | 기존 세션은 유지되지만 클라이언트가 새 CSRF 토큰 재조회 |
| 속도 제한 HMAC | 기존 버킷을 더 이상 찾지 못하므로 계획된 점검 시간에만 교체 |
| Resend | 메일 작업 재시도 전에 새 키 반영 |

세션 토큰은 별도 전역 서명 키를 사용하지 않으므로 데이터베이스 해시 조회와 행 폐기로 관리한다.

## 17. 로그, 오류와 분석 데이터

### 17.1 기록 허용

- `requestId`, 환경, 경로 템플릿, 메서드, 상태 코드와 처리 시간
- 내부 사용자·멤버십·워크스페이스 ID
- 인증 성공·실패 유형, 속도 제한 scope와 비식별 key hash 일부
- 세션 발급·폐기, 토큰 발급·사용·폐기 이벤트의 내부 ID
- 관리자 초대·비활성화·CSV 실행의 행위자와 대상 내부 ID
- 보안 검증 실패 코드와 예외 이름

### 17.2 기록 금지

- 비밀번호와 Argon2 입력
- 세션 쿠키, CSRF 토큰과 일회용 토큰
- `Authorization`, `Cookie`, `Set-Cookie` 헤더
- 이메일 원문, IP 원문과 Resend 응답 본문
- 이슈 제목·설명·댓글·작업 전달과 원본 파일명
- 요청·응답 전체 본문과 CSV 내용
- 데이터베이스·Resend·HMAC 비밀 키

인증 실패 로그도 계정 존재 여부를 외부 응답에 반영하지 않는다. 구조화 운영 로그는 30일, PostHog의 명시적 기능 이벤트와 정제된 예외는 1년 보존한다.

## 19. 보안 검증 기준

### 19.1 인증

- 12자 미만, 128자 초과와 차단 목록 비밀번호가 저장되지 않는다.
- Unicode NFC와 공백을 포함한 긴 문구가 가입·로그인에서 일관되게 처리된다.
- 계정이 없는 로그인도 더미 Argon2id 검증 경로를 거친다.
- 이메일 인증 전 업무 로그인과 업무 API 접근이 차단된다.
- 비밀번호 재설정 요청이 계정 존재 여부와 관계없이 같은 외부 응답을 반환한다.
- 재설정 성공 뒤 모든 기존 세션과 다른 재설정 링크가 무효화된다.

### 19.2 세션과 CSRF

- 세션 토큰이 32바이트 CSPRNG 결과이고 데이터베이스에는 해시만 저장된다.
- 운영 쿠키에 `__Host-`, `Secure`, `HttpOnly`, `SameSite=Lax`, `Path=/`가 적용되고 `Domain`이 없다.
- 로그인 전 쿠키 값이 로그인 후 세션으로 재사용되지 않는다.
- 유휴 7일과 절대 30일 만료를 서버가 강제한다.
- 상태 변경 요청의 CSRF 누락·변조·다른 세션 토큰과 외부 Origin이 모두 거부된다.
- 로그인·가입 같은 세션 전 요청도 외부 Origin과 단순 form 요청을 거부한다.

### 19.3 권한과 격리

- Member가 관리자 API를 직접 호출해도 `403`이다.
- 다른 사용자의 댓글·알림·미연결 파일을 직접 ID로 요청해도 접근할 수 없다.
- 다른 워크스페이스의 UUID와 표시 ID는 조회, 검색, 관계, SSE, 알림, 파일, 휴지통과 CSV에서 모두 `404`다.
- 생성·변경 요청의 참조 ID도 같은 워크스페이스인지 검증한다.
- 관리자 자기 비활성화와 미완료 담당 작업이 있는 멤버 비활성화가 롤백된다.
- CSV는 현재 워크스페이스 행만 포함하고 수식 시작 문자를 안전하게 이스케이프한다.

### 19.4 토큰과 속도 제한

- 다른 목적 API에서 토큰을 재사용할 수 없다.
- 같은 토큰의 동시 사용 중 한 요청만 성공한다.
- 재발급하면 이전 미사용 토큰이 즉시 폐기된다.
- 토큰 원문과 URL이 로그·PostHog·Referer에 남지 않는다.
- 계정과 네트워크 제한을 각각 적용하고 동시 요청에도 상한을 넘지 않는다.
- 제한 응답에 남은 대기 시간을 제공하고 영구 계정 잠금을 만들지 않는다.

### 19.5 출시 차단 보안 결함

- 다른 워크스페이스 데이터 또는 파일 노출
- 권한 없는 관리자 동작 성공
- 비밀번호·세션·일회용 토큰 원문 저장 또는 로그 기록
- 이메일 인증 우회, 재설정 토큰 재사용 또는 세션 폐기 실패
- CSRF 방어 없이 쿠키 인증 상태 변경 가능
- 파일 경로 조작, 위험 형식 인라인 실행 또는 CSV 수식 실행 가능

## 20. 구현 순서

1. 비밀번호 정책, Argon2id와 이메일 정규화
2. 일회용 토큰 생성·검증과 Resend Outbox
3. 로그인 세션, 보안 쿠키와 세션 폐기
4. CSRF 토큰, Origin 검증과 같은 출처 프록시
5. 인증·메일 속도 제한 버킷
6. 활성 멤버십 컨텍스트와 Admin·소유자 가드
7. 워크스페이스 격리 저장소 패턴과 파일·검색·CSV 검증
8. 보안 헤더, 정제 로그와 출시 차단 통합 테스트

각 단계는 성공 흐름뿐 아니라 우회 요청과 실패 응답을 자동 검증한 뒤 다음 기능에서 재사용한다.
