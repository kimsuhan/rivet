# 개발 지침 색인

> 상태: 현행
> 적용 범위: Rivet 모노레포의 코드 작성, 리뷰, 리팩터링과 폴더 구성
> 정본 책임: 변경 범위별 필수 개발 지침 진입점
> 갱신 조건: 앱·패키지 추가, 지침 책임 또는 필수 읽기 조건 변경

## 1. 목적

개발 작업에서 변경 범위와 관계있는 규칙만 찾을 수 있도록 개발 지침을 안내한다. 이 색인은 규칙 본문을 반복하지 않으며, 작업자는 아래 표에 따라 필요한 문서만 읽는다.

제품 동작과 아키텍처 계약이 필요하면 [전체 문서 색인](../index.md)에서 관련 문서를 찾고, UI
시각 기준은 [DESIGN.md](../../DESIGN.md)와 [웹 개발 지침](./web.md)을 따른다.

## 2. 읽는 문서

| 작업 범위 | 필수 문서 | 추가 조건 |
| --- | --- | --- |
| 모든 코드 구현·리뷰·리팩터링 | [공통 개발 지침](./common.md) | 항상 읽는다. |
| `apps/web` | [웹 개발 지침](./web.md) | 웹 코드나 브라우저 동작을 변경할 때 읽는다. |
| `apps/api` | [API 개발 지침](./api.md) | API 코드나 HTTP 계약 구현을 변경할 때 읽는다. |
| `apps/worker` | [워커 개발 지침](./worker.md) | 비동기·예약·정리 작업을 변경할 때 읽는다. |
| `packages/database`, Prisma | [데이터베이스 개발 지침](./database.md) | 스키마, migration, 쿼리 또는 트랜잭션을 변경할 때 읽는다. |

- API에서 Prisma 쿼리나 트랜잭션을 변경하면 `api.md`와 `database.md`를 함께 읽는다.
- 워커에서 Outbox 조회·상태 변경이나 데이터 정리를 변경하면 `worker.md`와 `database.md`를 함께 읽는다.
- 웹과 API 계약을 함께 변경하면 `web.md`와 `api.md`를 함께 읽는다.
- 여러 앱을 건드리지 않는 작업에서 관계없는 앱 문서를 미리 읽지 않는다.
- 기획 문서 작성, 단순 질의와 코드 기준이 필요하지 않은 작업에는 이 개발 지침을 적용하지 않는다.

## 3. 기준 우선순위

규칙이 충돌하면 다음 순서를 따른다.

1. 보안·데이터 격리·API 계약을 포함한 프로젝트 명세
2. 이 색인과 작업 범위에 해당하는 개발 지침
3. 저장소에 고정된 ESLint·TypeScript·Prettier 설정
4. Next.js, NestJS, Prisma와 사용하는 라이브러리의 공식 관례

프레임워크 관례를 따른다는 이유로 프로젝트의 워크스페이스 격리, 한국어 UI, API 계약과 파일 보안 기준을 약화할 수 없다.

## 4. 관리 원칙

- 하나의 규칙은 가장 적합한 개발 지침 한 곳에서만 관리한다.
- 공통 규칙을 앱별 문서에 복사하지 않고 [공통 개발 지침](./common.md)을 참조한다.
- 새 앱이나 독립 패키지에 별도 규칙이 실제로 필요할 때만 문서를 추가하고 이 색인에 읽기 조건을 등록한다.
- 제품 기능의 우선순위, 화면별 레이아웃, 엔드포인트별 상세 요청·응답, 운영 서버 비밀 값과 Git 운영 정책은 개발 지침에 중복 작성하지 않는다.
