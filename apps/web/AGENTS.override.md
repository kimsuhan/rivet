# 웹 애플리케이션 작업 지침

## 적용 범위

- 이 지침은 `apps/web` 아래의 모든 파일에 적용한다.

## 필수 문서

- `apps/web` 아래의 코드를 구현, 리뷰 또는 리팩터링하기 전에 다음 문서를 순서대로 읽고 따른다.
  1. [공통 개발 지침](../../docs/development/common.md)
  2. [웹 개발 지침](../../docs/development/web.md)
- 단순 질의나 코드 기준이 필요하지 않은 문서 작업에서는 개발 지침을 읽지 않는다.

## 조건부 문서

- UI 구조, 스타일, 컴포넌트 또는 접근성을 변경할 때는 [DESIGN.md](../../DESIGN.md)와 [디자인 시스템 명세서](../../docs/planning/005.%20디자인%20시스템%20명세서.md)를 추가로 읽는다.
- API 요청·응답 계약을 함께 변경할 때는 [API 개발 지침](../../docs/development/api.md)을 추가로 읽는다.
- 제품 동작이나 화면별 요구사항이 필요할 때만 [문서 색인](../../docs/index.md)에서 관련 기획·기술 설계 문서를 찾아 읽는다.
- 관계없는 API·워커·데이터베이스 문서는 미리 읽지 않는다.

## 책임 경계

- 화면 조합, 사용자 상호작용, 브라우저 상태와 API 소비만 담당한다.
- Prisma, API 내부 DTO와 서버 권한 규칙을 가져오거나 복제하지 않는다.
- 서버 상태는 생성 API 클라이언트와 TanStack Query를 사용하고 임의 endpoint 문자열을 만들지 않는다.
- 기능 코드는 `features/<feature>`, 라우팅과 조합은 `app`에 둔다.

## 완료 조건

- 성공뿐 아니라 로딩, 빈 상태, 오류, 권한 없음과 충돌 상태를 확인한다.
- UI 변경은 데스크톱·모바일, 키보드 조작, 포커스와 접근 가능한 이름을 확인한다.
- 변경 범위의 웹 테스트, lint, typecheck와 build를 실행한다.
- 사용자 흐름이 바뀌면 관련 Playwright 검사나 브라우저 수동 확인을 추가한다.
- API 계약이 바뀌면 생성 클라이언트 재생성과 계약 검사를 확인한다.
