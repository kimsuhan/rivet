# 워커 애플리케이션 작업 지침

## 적용 범위

- 이 지침은 `apps/worker` 아래의 모든 파일에 적용한다.

## 필수 문서

- `apps/worker` 아래의 코드를 구현, 리뷰 또는 리팩터링하기 전에 다음 문서를 순서대로 읽고 따른다.
  1. [공통 개발 지침](../../docs/development/common.md)
  2. [워커 개발 지침](../../docs/development/worker.md)
- 단순 질의나 코드 기준이 필요하지 않은 문서 작업에서는 개발 지침을 읽지 않는다.

## 조건부 문서

- Outbox 조회·상태 변경, 정리 쿼리, 트랜잭션, 스키마 또는 migration을 변경할 때는 [데이터베이스 개발 지침](../../docs/development/database.md)을 추가로 읽는다.
- API의 이벤트 생산 계약이나 공유 payload까지 함께 변경할 때는 [API 개발 지침](../../docs/development/api.md)을 추가로 읽는다.
- 알림, 이메일, 재시도, 예약 작업과 보존 정책의 상세 동작이 필요할 때만 [문서 색인](../../docs/index.md)에서 관련 기술 설계 문서를 찾아 읽는다.
- 관계없는 웹 문서는 미리 읽지 않는다.

## 책임 경계

- Outbox polling, handler 선택과 개별 업무 처리를 분리한다.
- HTTP 제품 API를 제공하거나 `apps/api` 내부 코드를 import하지 않는다.
- handler는 재실행돼도 중복 메일·알림·삭제가 생기지 않도록 멱등하게 작성한다.
- 외부 호출과 긴 작업을 claim 트랜잭션 안에서 실행하지 않는다.
- 재시도 가능 오류와 영구 실패를 구분하고 지원하지 않는 계약을 무한 재시도하지 않는다.

## 완료 조건

- 성공, 중복 실행, 일시 실패, 영구 실패와 종료 신호 경로를 검증한다.
- 로그에 `jobId`, 이벤트 식별 정보와 안전한 metadata만 포함되는지 확인한다.
- 변경 범위의 handler 단위·통합 테스트, lint, typecheck와 build를 실행한다.
- 이벤트 계약이 바뀌면 생산자·소비자와 schema version 호환성을 함께 확인한다.
- Prisma 변경이 있으면 데이터베이스 override의 검증 절차도 따른다.
