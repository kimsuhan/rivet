## 1. 문서의 역할

이 문서는 Rivet이 어떤 인상과 원칙을 가진 제품이어야 하는지 정의하는 상위 디자인 지침이다. 새로운 화면을 설계하거나 기존 UI를 검토할 때 먼저 방향을 맞추는 용도로 사용한다.

정확한 색상값, 글자 크기, 간격과 컴포넌트 variant의 정본은
`apps/web/app/globals.css`와 `apps/web/components/ui`다. 구현·접근성 기준은
[웹 개발 지침](./docs/development/web.md)을 따른다. 이 문서에 코드의 토큰을 복제하지 않는다.

문서만 수정하지 않고 영향을 받는 제품 흐름, 웹 개발 지침과 구현을 함께 갱신한다.

## 2. 제품이 주는 인상

Rivet은 업무 팀이 이슈, 프로젝트와 팀 간 전달을 빠르게 이해하고 처리하는 도구다. 화면은
다음 인상을 유지한다.

- **차분함:** 강한 장식보다 업무 내용과 현재 상태가 먼저 보인다.
- **정밀함:** 작은 간격, 경계선과 상태 변화가 의도적으로 맞물린다.
- **연결감:** 하나의 이슈와 여러 참여 팀의 작업 관계를 한 흐름으로 읽을 수 있다.
- **속도감:** 자주 쓰는 속성은 목록에서 바로 바꾸고 깊은 기록은 상세에서 다룬다.
- **신뢰감:** 저장, 오류, 충돌과 권한 상태가 숨지 않고 발생한 위치에 나타난다.

장식이 적다는 것은 무표정하다는 뜻이 아니다. Rivet의 개성은 여러 색이나 큰 그래픽이 아니라, 어두운 표면 위에 정돈된 정보와 작업 연결선이 만드는 흐름에서 나온다.

## 3. 핵심 디자인 원칙

### 3.1 업무 기록이 주인공이다

- 이슈 제목, 표시 ID, 상태, 담당자와 관계를 장식보다 먼저 읽게 한다.
- 탐색과 컨테이너는 콘텐츠를 돋보이게 하는 배경 역할에 머문다.
- 빈 공간을 장식으로 채우지 않고 정보 묶음과 읽기 순서를 분명히 하는 데 사용한다.

### 3.2 촘촘하지만 조작 가능해야 한다

- 데스크톱은 많은 작업을 빠르게 훑고 편집할 수 있는 밀도를 유지한다.
- 글자나 클릭 영역을 줄여 억지로 정보를 더 넣지 않는다.
- 모바일은 데스크톱을 축소하지 않고 조회와 핵심 변경에 맞는 한 열 구조를 사용한다.

### 3.3 서로 다른 의미를 섞지 않는다

- 팀, 워크플로 상태, 프로젝트 참여 관계와 차단 관계를 서로 추론하지 않는다.
- `차단됨`은 상태가 아니므로 상태 이름이나 상태 색으로 표현하지 않는다.
- 프로젝트 상태와 진행률을 같은 값처럼 보이게 하지 않는다.
- 색상만으로 상태, 우선순위, 오류와 현재 위치를 전달하지 않는다.

### 3.4 생성보다 이해를 돕는다

- 생성 화면은 빠르게 시작할 수 있게 하되 실제 작업 맥락을 적는 설명 영역을 숨기지 않는다.
- 사용할 수 없는 동작은 단순히 비활성화하지 않고 이유와 해결 경로를 알려 준다.
- 오류는 전역 알림으로만 보내지 않고 사용자가 조작한 위치에서 설명한다.

## 4. 시각 방향

### 4.1 어두운 제품 UI

- 현재 제품은 다크 모드를 기본 시각 체계로 사용한다.
- 순수 검정보다 미세하게 색이 들어간 어두운 캔버스를 사용한다.
- 탐색, 본문, 속성 패널과 오버레이는 단계형 표면으로 구분한다.
- 깊이는 표면 밝기 차이와 가는 경계선으로 표현하고 일반 카드에 강한 그림자를 사용하지 않는다.

### 4.2 절제된 라벤더 강조색

- 라벤더는 주요 행동, 현재 선택, 키보드 포커스와 중요한 링크에 사용한다.
- 넓은 카드나 페이지 배경을 라벤더로 채우지 않는다.
- 성공, 경고, 오류와 우선순위 색은 작은 아이콘, 상태점과 피드백 영역에 제한한다.
- 여러 밝은 색을 한 화면의 장식으로 경쟁시키지 않는다.

### 4.3 한국어 중심 타이포그래피

- 제품 기본 서체는 Pretendard Variable이다.
- 한국어는 영문 마케팅 제목처럼 강한 음수 자간을 적용하지 않는다.
- 제목보다 본문과 업무 메타데이터의 장시간 가독성을 우선한다.
- 표시 ID와 코드 외에는 고정폭 서체를 사용하지 않는다.

### 4.4 간격, 모서리와 깊이

- 4px 흐름을 기반으로 간격을 일관되게 조합한다.
- 버튼과 입력은 적당한 모서리를 사용하고 모든 요소를 pill 형태로 만들지 않는다.
- 카드 중첩과 그림자보다 경계선, 간격과 표면 단계로 구조를 만든다.
- 한 화면에 서로 다른 모서리·그림자 스타일을 임의로 추가하지 않는다.

## 5. Rivet의 시각적 서명: 작업 연결선

작업 연결선은 이슈와 참여 팀 작업의 계층, 명시적인 전달·배포 관계를 보여 주는 Rivet의 고유
시각 요소다.

- 이슈의 팀 작업 계층과 명시적인 전달·배포 관계에만 사용한다.
- 선과 함께 들여쓰기, 아이콘, 상태 이름과 관계 설명을 제공한다.
- 현재 선택, 진행과 완료는 디자인 시스템에 정의된 의미 토큰으로 구분한다.
- 단순 목록 구분선, 카드 장식과 탐색 강조에는 사용하지 않는다.
- 연결선이 없어도 보조 기술과 텍스트만으로 같은 관계를 이해할 수 있어야 한다.

작업 연결선의 목적은 꾸미는 것이 아니라 `이슈 → 참여 팀 작업 → 전달·배포` 흐름과 막힌
지점을 빠르게 읽게 하는 것이다.

## 6. 상호작용 원칙

- 상태, 담당자와 우선순위처럼 자주 바꾸는 값은 목록에서 바로 편집한다.
- 설명, 관계, 작업 전달, 댓글과 활동처럼 깊은 맥락은 상세 화면에서 다룬다.
- 저장 중에는 변경한 위치를 유지하고 실패하면 입력을 잃지 않은 채 다시 시도할 수 있게 한다.
- 실시간 충돌은 최신 값과 사용자의 변경을 함께 보여 준다.
- hover에만 의존하는 동작을 만들지 않는다.
- 드래그로 가능한 동작에는 키보드와 선택기 대안을 제공한다.
- 화면을 떠났다가 돌아오면 필터, 정렬, 펼침과 스크롤 맥락을 복원한다.

## 7. 한국어 UI 문구

- 메뉴, 상태, 오류, 도움말과 버튼은 한국어를 기본으로 한다.
- `API`, `CSV`, `Markdown`처럼 팀이 실제로 사용하는 기술 용어는 억지로 번역하지 않는다.
- 버튼은 `확인`, `제출`보다 `프로젝트 만들기`, `변경 저장`, `전달하고 완료`처럼 결과를 쓴다.
- 오류는 사과나 모호한 문구보다 원인과 해결 방법을 알려 준다.
- 빈 상태는 감성적인 표현보다 현재 조건과 다음 행동을 설명한다.
- 같은 행동은 화면이 달라도 같은 이름을 사용한다.

## 8. 접근성과 반응형 기준

- 키보드 포커스를 명확하게 표시하고 포커스 순서를 시각적 읽기 순서와 맞춘다.
- 상태, 우선순위, 읽지 않음과 오류를 색상 외의 텍스트·아이콘으로 함께 전달한다.
- 모바일 터치 영역은 손가락으로 안정적으로 누를 수 있는 크기를 유지한다.
- 모션 감소 설정에서는 위치 이동과 확대를 제거한다.
- 모바일에서 지원하지 않는 기능은 축소된 데스크톱 화면 대신 이유와 이동 경로를 제공한다.
- 긴 한국어, 긴 URL, 많은 라벨과 200% 확대에서도 핵심 행동을 사용할 수 있어야 한다.

## 9. 해야 할 것과 피할 것

### 해야 할 것

- 화면을 만들기 전에 사용자가 가장 먼저 판단해야 할 정보를 정한다.
- 기존 토큰과 컴포넌트 조합을 먼저 사용한다.
- 라벤더를 적게 사용해 주요 행동과 현재 위치의 의미를 보존한다.
- 오류, 빈 상태와 권한 부족에서도 다음 행동을 제공한다.
- 새로운 패턴이 여러 화면에서 반복되면 공통 UI 구현과 웹 개발 지침에 반영한다.

### 피할 것

- 임의의 색상, 간격, 그림자와 z-index를 화면 안에 직접 추가한다.
- 대형 그라디언트, 발광 효과와 장식 카드를 제품 화면의 주인공으로 만든다.
- 모든 정보에 배지와 테두리를 붙여 시각적 소음을 만든다.
- 상태, 팀과 역할을 색상이나 위치만으로 추론하게 한다.
- 모바일 미지원 기능을 비활성 버튼으로 그대로 남긴다.
- Linear의 마케팅 화면을 제품 업무 UI에 그대로 복제한다.

## 10. 변경 원칙

- 방향과 원칙이 바뀌면 이 문서를 갱신한다.
- 정확한 시각 값이나 컴포넌트 계약이 바뀌면 공통 UI 구현과 웹 개발 지침을 갱신한다.
- 화면 구조나 동작이 바뀌면 사용자 흐름 문서와 구현을 갱신한다.
- 두 문서에 같은 토큰 표나 컴포넌트 수치를 복사하지 않는다.
