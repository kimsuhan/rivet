# Rivet 작업 하네스

## 소통과 범위

- 사용자에게 설명하거나 보고할 때는 한국어를 사용한다.
- 작업 시작 전에 변경 범위와 접근 방식을 짧게 알린다.
- 질문, 설명, 리뷰 요청은 읽기 전용으로 처리하고 사용자가 변경을 요청한 경우에만 파일을 수정한다.
- 요청 범위를 벗어난 리팩터링, 리네이밍과 구조 변경을 하지 않는다.

## 문서 탐색

- 제품 동작, 기준 또는 의사결정 배경이 필요할 때만 `docs/index.md`를 먼저 확인하고 관련 문서만 읽는다.
- 코드 구현, 리뷰 또는 리팩터링은 `docs/development/index.md`에서 변경 범위에 지정된 개발 지침만 읽는다.
- 작업 경로에 더 가까운 `AGENTS.md`가 있으면 그 문서의 필수·조건부 읽기 규칙을 적용한다.
- 개발 지침을 찾기 위해 `docs/index.md`나 `docs/development/` 전체를 탐색하지 않는다.
- 단순 질의나 코드 기준이 필요하지 않은 문서 작업에서는 개발 지침을 읽지 않는다.

## 작업 시작

- 관련 파일, 현재 구조와 `git status`를 확인해 기존 사용자 변경을 구분한다.
- 기존 변경은 사용자 소유로 보고 되돌리거나 덮어쓰지 않는다.
- 구현 전에 변경할 앱·패키지, 공개 계약과 검증 범위를 정한다.
- 결과를 materially 바꾸는 선택만 사용자에게 묻고, 범위 안의 세부 사항은 기존 기준에 따라 진행한다.

## 구현

- 기존 코드 스타일과 프로젝트 경계를 우선하며 가장 작은 변경으로 목표를 달성한다.
- 파일 수정은 `apply_patch`를 사용하고 생성 코드와 사용자 관리 비밀 값을 직접 수정하지 않는다.
- TDD 방식(테스트를 먼저 작성)은 사용하지 않는다. 기능을 구현한 후 정상 동작을 검증하는 해피패스 테스트와 실제 발생 가능한 엣지 케이스에서 버그를 방지하는 테스트를 작성한다.
- 사용하지 않는 빈 폴더, placeholder와 미래용 추상화를 만들지 않는다.
- 새로운 `apps/*` 또는 `packages/*` workspace를 만들 때는 같은 변경에서 해당 경로의 `AGENTS.md`도 생성한다.
- 새 workspace의 지침에는 필수 문서, 책임 경계, 금지 사항과 검증 절차를 적고 80줄 이하로 유지한다.
- 새 workspace에 독립 개발 지침이 필요하면 `docs/development/`에 추가하고 색인도 함께 갱신한다.

## 안전

- 사용자가 명시하지 않은 commit, push, branch 변경과 외부 배포를 하지 않는다.
- `git reset --hard`, 무단 파일 삭제와 다른 작업을 훼손하는 명령을 사용하지 않는다.
- 실제 비밀 값, 토큰, 쿠키, 운영 환경 변수와 사용자 데이터를 Git 추적 파일이나 로그에 저장하지 않는다. 사용자가 지정한 Git 제외 로컬 환경 파일만 예외로 한다.
- 실패를 숨기는 우회, 검증 생략과 운영 전용 예외를 추가하지 않는다.

## 검증

- 변경 범위에 맞는 lint, typecheck, test와 build를 실행한다.
- PostgreSQL이 필요한 로컬 테스트는 루트 `.env.test.local`의 `DATABASE_URL`을 사용한다. 해당 파일은 Git에 포함하지 않는다.
- 로컬 `rivet` 데이터베이스의 `public` 스키마는 테스트 전용이므로 초기화, migration과 데이터 변경을 허용하되 다른 데이터베이스나 스키마는 건드리지 않는다.
- 계약·스키마·migration·생성 코드 변경은 해당 생성 및 일관성 검사를 추가한다.
- 문서 변경은 링크, 파일 경로와 `git diff --check`를 확인한다.
- 실행할 수 없는 검증이나 실패한 명령은 원인과 남은 위험을 함께 보고한다.

## 완료 보고

- 결과를 먼저 말하고 주요 변경 파일과 동작을 요약한다.
- 실행한 검증과 결과를 적고 실행하지 않은 검증은 이유를 밝힌다.
- 남은 작업이나 사용자 결정이 실제로 있을 때만 마지막에 제시한다.
